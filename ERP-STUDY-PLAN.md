# ERP + MES 미니 프로젝트 학습 계획서

> **대상**: ERP/MES 도메인 지식이 전무한 백엔드 개발자
> **목표**:
> 1. ERP의 핵심 개념(모듈 구조, 모듈 간 트랜잭션 연계, 마스터/트랜잭션 데이터 분리, 기간계 시스템 특유의 무결성·감사·배치)을 체득
> 2. MES(제조 현장 실행 시스템)의 핵심 개념(작업지시·실적·자재 소모·품질·설비)을 체득
> 3. **두 시스템을 별도 애플리케이션으로 분리하여 REST + Kafka로 연계** — 기간계 실무의 핵심인 **시스템 간 인터페이스(I/F)** 학습
> **학습 방식**: **도메인 이해 중심 학습**. AI가 매 단계마다 도메인을 글로 정리해 브리핑 → 사용자가 충분히 이해 → AI가 그 도메인을 코드로 구현 → AI가 코드 워크스루로 설명 → 사용자는 "도메인이 어떻게 코드가 되는가"를 학습. 즉, **사용자는 코드를 직접 타이핑하지 않고, 도메인과 설계 결정을 이해하는 데 집중한다**.
> **총 기간**: 약 20주 (5개월) — ERP 12주 + MES 연계 8주
> **사용자 역할**: 도메인 이해 / 설계 검토 / 질문 / 시연 확인 / 회고
> **AI 역할**: 도메인 브리핑 작성 / 모델링 제안 / 코드 구현 / 코드 워크스루 / 시연 가이드

---

## 1. 학습 원칙

1. **도메인 이해가 학습의 본체, 코드는 그 결과물**
   각 단계 시작 시 AI가 도메인 브리핑(이 모듈이 현실에서 어떤 업무를 처리하는지, 왜 그렇게 설계되는지)을 글로 작성한다. 사용자는 이 글을 읽고 충분히 이해될 때까지 질문한다. **이해되지 않은 채로 코드 단계로 넘어가지 않는다.**
2. **코드는 AI가 작성, 사용자는 코드 워크스루로 학습**
   사용자는 코드를 직접 타이핑하지 않는다. 대신 AI가 작성한 코드의 핵심 부분을 함께 읽으며 "왜 이렇게 설계했는가", "이 도메인 규칙이 어느 코드로 표현되었는가" 를 추적한다.
3. **모듈 간 연계가 핵심**
   각 모듈은 독립적이지 않다. 항상 **"이 기능이 다른 모듈에 어떤 영향을 주는가"**를 도메인 브리핑과 코드 양쪽에서 함께 다룬다. 예: 출하 → 재고 차감 → 매출 전표 자동 생성.
4. **트랜잭션과 정합성 우선**
   기간계의 본질은 데이터 무결성이다. 동시성, 트랜잭션 경계, 롤백 시나리오를 매 단계 짚는다.
5. **테스트는 업무 규칙의 명세**
   AI가 도메인 단위 테스트를 작성하고, 사용자는 테스트가 표현하는 **업무 규칙**을 읽고 이해한다. 테스트는 "이 시스템이 어떤 업무를 보장하는가"의 살아있는 문서다.
6. **이해 못한 부분은 다시 질문**
   ERP 도메인은 한 번에 이해되지 않는다. 한 번 넘어간 개념도 다음 Phase에서 다시 짚어가며 누적적으로 이해한다.

## 2. 기술 스택

### 필수
| 영역 | 선택 | 선정 이유 |
|------|------|-----------|
| 언어 | **Java 21** | LTS, record/pattern matching이 도메인 모델링에 유리 |
| 프레임워크 | **Spring Boot 3.3.x** | 사용자 지정 |
| DB | **MySQL 8.x** | 사용자 지정 |
| 빌드 | **Gradle (Kotlin DSL)** | Spring 생태계 표준 |

### 데이터/영속성
| 영역 | 선택 | 선정 이유 |
|------|------|-----------|
| ORM | **Spring Data JPA + Hibernate** | 엔티티 관계가 복잡한 ERP 도메인에 적합. 1:N, N:M, 상속 구조를 직관적으로 표현 |
| 동적 쿼리 | **QueryDSL 5.x** | ERP는 다중 필터 조회/리포트가 매우 많음. JPQL 문자열로는 한계 |
| 마이그레이션 | **Flyway** | 스키마 변경 이력을 코드와 함께 버전 관리. 기간계는 스키마 변경이 잦음 |

### 보안/운영
| 영역 | 선택 | 선정 이유 |
|------|------|-----------|
| 인증/인가 | **Spring Security + JWT** | 사용자/권한은 ERP 필수. 모듈별 권한 분리를 학습 |
| 감사 로그 | **Spring Data Envers** 또는 직접 구현 | 누가 언제 어떤 데이터를 바꿨는지 — 기간계 필수 |
| API 문서 | **springdoc-openapi (Swagger UI)** | 화면 만들기 전에는 Swagger UI가 곧 테스트 화면 |
| 검증 | **Bean Validation (Jakarta)** | 입력 검증 표준 |
| 매핑 | **MapStruct** | 엔티티 ↔ DTO 변환을 컴파일 타임에 안전하게 |
| 코드 보일러플레이트 | **Lombok** | 학습 비용 대비 생산성 ↑ |

### 화면 (점진적 도입)
| 단계 | 도구 | 비고 |
|------|------|------|
| Phase 1~5 | **Swagger UI만** | 화면 만드느라 도메인 학습이 늦어지는 걸 방지 |
| Phase 6 이후 | **Thymeleaf + HTMX + Bootstrap** | 백엔드 학습에 집중하면서도 ERP 화면 흐름 체험 |

> Vue/React는 의도적으로 제외. 프론트 학습이 ERP 학습 시간을 잡아먹으면 안 됨.

### 테스트
| 영역 | 선택 |
|------|------|
| 단위 | JUnit 5 + AssertJ |
| 통합 | Spring Boot Test + **Testcontainers (MySQL)** |
| 시나리오 | RestAssured 또는 MockMvc |

### 시스템 간 연계 (MES 도입 시 ⭐)
| 영역 | 선택 | 선정 이유 |
|------|------|-----------|
| 동기 호출 | **Spring 6 RestClient** | ERP → MES 작업지시 같은 즉시 응답 필요 케이스 |
| 비동기 메시징 | **Apache Kafka** | MES → ERP 실적 전송 — 대용량/내구성 보장 |
| 트랜잭션 정합성 | **Transactional Outbox Pattern** | DB 트랜잭션과 메시지 발행의 원자성 보장 (기간계 핵심 패턴) |
| 회복성 | **Resilience4j** | Circuit Breaker / Retry / Timeout — 외부 시스템 장애 격리 |
| API 계약 | **OpenAPI Spec 공유** | ERP-MES 간 API 명세를 한 파일로 관리 |
| 분산 추적 | **Micrometer Tracing + Zipkin** | 두 시스템 걸친 요청 추적 |

### 인프라/개발환경
| 영역 | 선택 |
|------|------|
| 컨테이너 | Docker + docker-compose (MySQL × 2개, Kafka, Redis, Zipkin) |
| 캐시 (후반) | Redis (마스터 데이터 캐싱 학습용) |
| 메시징 | **Kafka** (Phase 11부터) — 그 전에는 Spring Events |
| 배치 | **Spring Batch** (Phase 9) |
| 스케줄링 | Spring Scheduler |

## 3. 프로젝트 도메인 컨텍스트

학습용 가상 회사 **"현우전자"** 를 운영하는 **ERP + MES** 두 개의 시스템을 만든다.

### 회사 설정
- 현우전자는 **노트북·모니터를 제조해서 기업 고객에게 판매**한다
- 노트북 1종 + 모니터 1종으로 시작 (각 BOM 3~4단계)
- 부서: 영업팀 / 구매팀 / 생산팀 / 재무팀 / 인사팀 / **공장(현장 작업자)**

### 두 개의 시스템

| 시스템 | 역할 | 주 사용자 | 분류 |
|--------|------|-----------|------|
| **현우-ERP** (`hwlee-erp`) | 회사 자원 관리 — 영업/구매/재고/회계/인사/생산 계획 | 본사 사무직 | 기간계 |
| **현우-MES** (`hwlee-mes`) | 공장 현장 실행 — 작업지시 실행/실적/품질/설비 | 공장 작업자/관리자 | 현장계 |

### 시스템 간 연계 (이 학습의 핵심)

```
[ERP PP]                              [MES]
생산 계획/지시 발행 ──REST 동기 호출──→ 작업지시 등록
                                          ↓
                                       현장 작업
                                          ↓
실적/자재 소모 수신 ←── Kafka 비동기 ←─ 생산 실적 발생
       ↓
[ERP MM] 자재 차감
[ERP MM] 완제품 입고
[ERP FI] 원가 계산
```

### 왜 이렇게 나누는가
- 실무에서 ERP와 MES는 **거의 항상 별도 시스템**으로 분리됨 (벤더도 다르고, 운영 주체도 다름)
- **시스템 간 인터페이스 개발(I/F)이 기간계 실무의 절반**
- DB 분리 → "다른 시스템과 데이터 정합성을 어떻게 맞추는가" 학습 (Outbox Pattern, 멱등성, 재시도)

이렇게 가상 회사와 두 시스템을 분리해 둬야 "이 기능을 누가, 왜 쓰는가"와 "시스템 간 데이터를 어떻게 주고받는가"를 항상 의식할 수 있다.

## 4. 커리큘럼 (Phase 0 ~ Phase 10)

총 **약 12주 (3개월)** / 주당 학습 시간 **8~10시간 가정** (평일 저녁 + 주말).

각 Phase는 다음 7단계 사이클로 진행한다.

| 단계 | 주체 | 산출물 |
|------|------|--------|
| 1. **도메인 브리핑** | AI 작성 → 사용자 읽기 | "이 모듈이 현실에서 무엇을 하는가, 왜 필요한가, 다른 모듈과 어떻게 엮이는가"를 정리한 글 |
| 2. **이해 확인 / Q&A** | 사용자 질문 → AI 답변 | 사용자가 도메인을 충분히 이해할 때까지 반복. 이해 안 되면 다음 단계로 넘어가지 않음 |
| 3. **모델링 제안** | AI 작성 → 사용자 검토 | ERD, 엔티티 관계도, API 설계, 핵심 비즈니스 규칙 목록 |
| 4. **사용자 승인** | 사용자 결정 | 모델링이 도메인을 잘 반영하는지 검토하고 OK 사인. 수정 의견 있으면 반영 |
| 5. **코드 구현** | AI 작성 | 실제 Spring Boot 코드, 마이그레이션, 테스트 작성 |
| 6. **코드 워크스루** | AI 설명 → 사용자 학습 | 핵심 파일을 함께 읽으며 "도메인 규칙이 어떤 코드로 표현됐는지" 짚어가며 설명 |
| 7. **시연 + 회고** | 사용자 시연 → 학습 일지 | Swagger UI에서 직접 호출해 보고, 새로 알게 된 ERP 용어/개념을 `LEARNING-LOG.md`에 기록 |

> **핵심**: 단계 2(이해 확인)와 단계 6(코드 워크스루)이 학습의 중심이다. 단계 5(코드 구현)는 AI가 빠르게 처리하므로, 사용자는 절약한 시간을 단계 2와 6에 투자한다.

### Phase 1. 마스터 데이터 (1주)
**목표**: ERP의 토대 = 마스터 데이터. 트랜잭션이 참조하는 기준 정보를 먼저 만든다.

| 만들 것 | 설명 |
|---------|------|
| 고객 마스터 (Customer) | 회사명, 사업자번호, 주소, 신용한도, 결제조건 |
| 상품 마스터 (Item) | 상품코드, 명칭, 단위, 표준원가, 표준판매가 |
| 거래처 마스터 (Vendor) | 매입처 정보 |
| 부서/직원 마스터 (Department/Employee) | 부서 트리 + 직원 |

**기능**: CRUD + 페이징 검색 + Soft Delete + 변경 이력 추적

**ERP 학습 포인트**:
- "코드 체계" 직접 설계 (예: `CUST-2026-0001`) → 실제 ERP는 코드가 핵심
- 마스터 데이터는 **함부로 삭제 못 함** (트랜잭션이 참조 중) → Soft Delete 필요성 체감
- JPA Auditing 으로 등록자/수정자/일시 자동 기록

---

### Phase 2. SD — 수주~출하~인보이스 (2주)
**목표**: 영업 모듈로 OTC(Order-to-Cash) 흐름의 절반 구현

| 만들 것 | 설명 |
|---------|------|
| 견적(Quotation) | 고객별 가격 제안 |
| 수주(Sales Order) | 주문 확정 + 헤더/라인 구조 |
| 출하(Delivery) | 출고 지시 |
| 인보이스(Invoice) | 세금계산서 발행 |

**기능**:
- 수주 등록 시 고객 신용한도 체크
- 수주 → 출하 → 인보이스 상태 전이 (State Machine 학습)
- 부분 출하 / 부분 인보이스 (실무 핵심)

**ERP 학습 포인트**:
- 헤더-라인(Header-Detail) 구조 (1주문 N라인)
- 비즈니스 상태(Status) 흐름 관리
- 가격 계산 로직 (단가 × 수량 - 할인 + 세금)

---

### Phase 3. MM — 재고와 입출고 (1.5주)
**목표**: 자재 모듈로 재고 관리

| 만들 것 | 설명 |
|---------|------|
| 창고(Warehouse) 마스터 | 창고/위치 관리 |
| 재고(Stock) | 상품 × 창고 단위 보유량 |
| 입고(Goods Receipt) | 매입 시 재고 증가 |
| 출고(Goods Issue) | 출하 시 재고 감소 |
| 재고이동(Transfer) | 창고 간 이동 |

**기능**:
- 재고 부족 시 출고 거부
- 재고 이력(Stock Movement) 테이블로 추적성 확보

**ERP 학습 포인트**:
- **동시성 처리** — 같은 상품 재고를 동시에 빼려고 할 때
- 비관적 락(`PESSIMISTIC_WRITE`) vs 낙관적 락(`@Version`) 직접 비교 학습
- 재고 평가 방법 개념 학습 (선입선출/평균법) — 구현은 평균법만

---

### Phase 4. SD ↔ MM 연계 (1주) ⭐ ERP의 핵심
**목표**: 모듈을 한 트랜잭션으로 엮는다

- [ ] 출하(Delivery) 확정 시 → 재고 자동 차감
- [ ] 입고(Goods Receipt) 확정 시 → 재고 자동 증가
- [ ] 한 트랜잭션 안에서 실패하면 전체 롤백
- [ ] Spring Application Events 로 모듈 간 결합도 낮추기

**ERP 학습 포인트**:
- "왜 ERP가 단순 CRUD가 아닌가"를 코드로 체감
- 도메인 이벤트 패턴
- 트랜잭션 경계 설계 (`@Transactional` 전파 옵션)

---

### Phase 5. FI — 자동 회계 전표 (2주)

### Phase 12. MES 마스터 + 작업지시 수신 (ERP → MES 동기) (1주)
**목표**: ERP에서 발행한 생산지시가 MES로 들어온다

| 만들 것 | 설명 |
|---------|------|
| MES 설비(Equipment) 마스터 | 생산 라인/설비 정보 |
| MES 작업자(Operator) 마스터 | 현장 작업자 |
| 작업지시(Work Order) 수신 API | ERP가 호출 → MES에 등록 |
| ERP 측 RestClient + 재시도 | MES 일시 장애 시 동작 |

**학습 포인트**:
- 동기 REST 호출의 장단점
- 멱등성(Idempotency) — 같은 작업지시가 두 번 들어와도 한 번만 등록
- ERP의 "생산지시" 와 MES의 "작업지시"는 같은 실체에 대한 두 시스템의 관점

---

### Phase 13. MES 생산 실적 등록 (1.5주)
**목표**: 현장 작업자가 생산한 실적을 MES에 기록

| 만들 것 | 설명 |
|---------|------|
| 생산 실적(Production Performance) | 작업지시별 생산 수량/시간 |
| 자재 투입(Material Consumption) | BOM 기반 자재 소모 기록 |
| 작업 시작/일시정지/완료 상태 전이 | State Machine |
| 실적 입력용 간단 화면 (Thymeleaf) | 현장 작업자 시뮬레이션 |

**학습 포인트**:
- ERP와 다른 도메인 (시간 단위 추적, 작업자, 설비)
- 부분 실적 / 부분 자재 소모 처리

---

### Phase 14. MES → ERP 실적 전송 (Kafka 비동기) ⭐ 핵심 (1.5주)
**목표**: MES 실적이 비동기로 ERP에 흘러가서 ERP의 재고/회계가 반영됨

- [ ] MES에서 **Transactional Outbox** 테이블 신설
- [ ] 실적 저장 트랜잭션과 함께 Outbox에 이벤트 적재
- [ ] Outbox Polling Publisher → Kafka 토픽 발행
- [ ] ERP 측 Kafka Consumer가 수신
- [ ] ERP가 자재 차감 / 완제품 입고 / 원가 계산
- [ ] 실패/중복 처리 (DLQ, 멱등성 키)

**학습 포인트** (이 학습 전체에서 가장 중요한 부분 중 하나):
- "DB 커밋 + 메시지 발행"의 원자성 문제
- Outbox Pattern으로 두 시스템 간 결과적 일관성(Eventual Consistency) 보장
- 메시지 한 건이 양쪽 DB를 다 바꾸는 흐름의 정합성

---

### Phase 15. MES 품질 검사 + 설비 상태 (1.5주)
**목표**: MES의 나머지 핵심 — 품질과 설비

| 만들 것 | 설명 |
|---------|------|
| 품질 검사(Quality Inspection) | 생산 후 합격/불량 판정 |
| 불량 사유(Defect Reason) 마스터 | 불량 코드 체계 |
| 설비 상태(Equipment Status) | 가동/정지/정비 |
| 가동률 계산 | OEE 미니 버전 |
| 불량 발생 시 ERP에 통보 | Kafka 이벤트로 |

**학습 포인트**:
- MES 고유 도메인 (제조업의 본질)
- ERP가 모르는 정보(불량률/가동률)를 MES가 어떻게 관리하고 일부만 ERP로 전달하는가

### Phase 16. 통합 시나리오 + 운영 관점 (1.5주)
**목표**: 두 시스템이 한 흐름으로 동작하는 종합 시나리오 + 운영성 확보

- [ ] **End-to-End 시나리오 한 번에 실행**:
  영업 수주(ERP SD) → 생산 계획(ERP PP) → 작업지시(ERP→MES) → 실적(MES) → 자재 차감/완제품 입고/원가 계산(MES→ERP) → 출하(ERP SD) → 인보이스(ERP)
- [ ] 분산 추적으로 한 거래의 흐름 시각화
- [ ] ERP-MES 데이터 정합성 검증 배치 (불일치 발견 시 알림)
- [ ] 한쪽 시스템 장애 시 다른 쪽 동작 (Circuit Breaker 동작 확인)
- [ ] 통합 README + 아키텍처 다이어그램 정리

**학습 포인트**:
- 단일 시스템 사고 → **분산 시스템 사고**로 전환
- 운영자가 "어디서 끊겼는지" 알 수 있게 만드는 게 분산 시스템의 핵심
- 데이터 정합성을 사후에 검증하고 보정하는 메커니즘 (실무 패턴)

## 6. 진행 방식 — 매일 / 매주 루틴

### 매일 (학습일, 약 2~2.5시간)
1. **10분**: 어제 학습 내용 / 새 용어 노트 다시 읽기
2. **40~60분**: 그날의 도메인 브리핑 읽기 + 질문/답변 (단계 1~2)
3. **20~30분**: AI가 제안한 모델링 검토 + 승인 (단계 3~4)
   - *AI가 코드 구현(단계 5)을 진행하는 동안 사용자는 잠시 쉼*
4. **40~60분**: AI의 코드 워크스루 듣기 + 핵심 파일 직접 읽어 보기 (단계 6)
5. **10~20분**: Swagger UI 시연 + 학습 일지 작성 (단계 7)

> 코드를 "타이핑"하지 않으므로 같은 시간에 더 깊은 도메인 학습이 가능. 다만 **단계 6(코드 워크스루)을 건너뛰면 학습이 무너진다**. 코드를 작성하지 않는 만큼, 작성된 코드를 읽는 시간을 반드시 확보한다.

### 매주 마지막 날
- 그 주에 만든 기능을 **시나리오 시퀀스로** Swagger UI에서 처음~끝까지 직접 실행
  예: 고객 등록 → 상품 등록 → 수주 → 출하 → 인보이스 발행
- 동작이 안 되거나 결과가 의외인 부분이 있으면 그 자리에서 AI에게 질문
- 새 용어/개념/막힌 점을 `LEARNING-LOG.md` 에 기록 (사용자가 직접 작성)

### 매 Phase 종료 시
- AI가 그 Phase의 **도메인 다이어그램**과 ERD를 그려 줌 → 사용자는 그것을 보고 자기 말로 다시 설명해 보기 (이해 점검)
- 사용자가 "이 Phase에서 가장 중요한 한 가지 개념"을 한 문단으로 직접 작성 → 학습 일지에 누적
- README 업데이트 (AI가 초안 → 사용자가 보강)

```
        ├─ common/
        ├─ master/             # 설비, 작업자
        │   ├─ equipment/
        │   └─ operator/
        ├─ workorder/          # 작업지시 수신
        ├─ performance/        # 생산 실적
        ├─ quality/            # 품질 검사
        └─ integration/        # ERP 연계
            ├─ inbound/        # ERP에서 받는 REST API
            ├─ outbox/         # Outbox 테이블 + Polling Publisher
            └─ messaging/      # Kafka Producer
```

### 각 시스템의 코드 컨벤션
모든 도메인 모듈은 **`controller / service / domain / repository / dto`** 5단 구조로 일관되게.

### 주요 결정 포인트
- **DB 분리**: ERP와 MES는 다른 MySQL 데이터베이스 (`erp_db`, `mes_db`) — 같은 인스턴스라도 스키마 분리
- **포트 분리**: ERP 8080, MES 8081
- **공통 모듈은 만들지 않음** (학습 초기에는 의도적으로 코드 중복 허용. 두 시스템의 독립성을 코드로 체감)

## 9. 성공 기준 (자가 진단)

### Part 1 — ERP 12주 종료 시점
- [ ] ERP에서 마스터 데이터와 트랜잭션 데이터의 차이를 예시로 설명할 수 있다
- [ ] OTC(Order-to-Cash) 전체 흐름을 6단계로 말할 수 있다
- [ ] 한 출하 트랜잭션이 SD/MM/FI 3개 모듈에 어떻게 영향을 주는지 코드로 보여줄 수 있다
- [ ] 동시 출하 시 재고 음수가 안 되게 막은 방법을 두 가지 이상 설명할 수 있다
- [ ] 야간 배치가 왜 필요한지, 실시간으로 못 하는 이유를 설명할 수 있다
- [ ] 누가 언제 어떤 데이터를 바꿨는지 조회할 수 있는 화면 또는 API가 있다
- [ ] 회사 분개 한 줄(예: 매출채권 / 매출)이 어느 거래에서 자동 생성되는지 설명할 수 있다

### Part 2 — MES + 연계 20주 종료 시점
- [ ] ERP와 MES의 책임 분담을 5분 안에 설명할 수 있다 (계획 vs 실행, 재무 vs 현장)
- [ ] ERP의 "생산지시"가 MES의 "작업지시"가 되어 다시 ERP의 "재고/원가"로 돌아오는 전 과정을 코드로 추적할 수 있다
- [ ] Transactional Outbox Pattern을 직접 구현하고, 왜 필요한지 설명할 수 있다 (DB 커밋과 메시지 발행의 원자성)
- [ ] 동기 REST 호출과 비동기 Kafka 메시징을 어느 케이스에 어느 것을 쓰는지 판단 기준을 가지고 있다
- [ ] MES가 다운되었을 때 ERP가 어떻게 동작하는지 (Circuit Breaker), 복구 시 어떻게 따라잡는지 설명할 수 있다
- [ ] 같은 메시지가 두 번 들어와도 한 번만 처리되도록 멱등성을 보장하는 방법을 두 가지 이상 안다
- [ ] 두 시스템 간 데이터 정합성이 깨졌을 때 어떻게 발견하고 보정하는지 (정합성 검증 배치) 설명할 수 있다
- [ ] 한 비즈니스 거래가 두 시스템에 걸쳐 어떻게 흐르는지 분산 추적으로 시각화해서 보여줄 수 있다

```bash
# 시작 시 첫 명령

```

### 진행 흐름
- **Part 1 (W1~W12)**: `hwlee-erp` 단일 프로젝트로 ERP 학습
- **Part 2 (W13~W20)**: `hwlee-mes` 별도 프로젝트 신설 + ERP-MES 연계
- 각 Phase마다 7단계 사이클(도메인 브리핑 → 이해 확인 → 모델링 → 승인 → 코드 구현 → 워크스루 → 시연/회고)로 진행

### Phase 0 시작 시 첫 사이클 예시
1. AI가 **"왜 ERP 학습에 Docker, MySQL, Spring Boot 21이 필요한가, 각 도구가 어떤 역할인가"** 도메인 브리핑 작성
2. 사용자가 읽고 이해 안 되는 부분 질문
3. AI가 디렉토리 구조와 docker-compose.yml 설계 제안
4. 사용자 승인
5. AI가 실제 파일들 작성 (`docker-compose.yml`, `build.gradle.kts`, `ErpApplication.java`, 헬스체크 API, 첫 Flyway 마이그레이션, 첫 Testcontainers 테스트)
6. AI가 작성된 파일들을 사용자와 함께 한 줄씩 읽으며 설명
7. 사용자가 직접 `docker compose up` → 헬스체크 호출 → 결과 확인 → `LEARNING-LOG.md`에 첫 학습 기록

준비되시면 **"Phase 0 시작"** 이라고 말씀해 주세요. 1번 단계(도메인 브리핑)부터 바로 진행하겠습니다.
