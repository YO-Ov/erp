## 2026-05-XX (Phase X — 작업 내용)

### 새로 알게 된 것
### 새로 알게 된 것
- **트랜잭션 번호의 일자별 시퀀스** — `Q-20260526-001`, `SO-20260527-001` 형식. 마스터(연도별) 와 트랜잭션(일자별)이 한 `code_sequence` 테이블에서 `period_key` 값으로 분리되어 공존.
- **DRAFT vs CONFIRMED 의 의미 차이** — 영업이 자유롭게 입력하는 단계(DRAFT) vs 회사가 책임지는 단계(CONFIRMED). 신용한도 검증은 CONFIRMED 시점에만 도는 게 실무 패턴. 메시지 "남은 한도 500000, 요청 1000000" 처럼 한 응답에 3개 변수가 들어가 즉시 진단 가능.
- **수주 상태는 메서드가 아니라 파생값** — 견적(`send`/`accept`)은 상태를 직접 바꾸지만, 수주는 `Delivery` / `Invoice` 가 만들어질 때 SO 라인의 `shipped_qty` / `invoiced_qty` 누적값을 보고 `recomputeStatus()` 가 자동 계산. SETTER 가 없다.
- **"보낸 만큼만 청구"** — 출하한 수량 ≥ 청구 수량. 출하 안 한 수량 청구 시 409 "remainingInvoiceable=0, quantity=1". 매출은 출하 시점에 인식되는 회계 원칙이 도메인에 박혀 있음.
- **부가세 정합성** — `subtotal + tax = total`. 6대 × 120만 = 720만 + 부가세 72만 = 792만 — 한 `recomputeTotal()` 메서드가 3개 필드를 동시 갱신.
- **취소의 정합성** — 출하 cancel 시 SO 라인의 `shipped_qty` 자동 차감 + `recomputeStatus()` 로 SHIPPING → CONFIRMED 자동 복귀. 이 모든 게 한 트랜잭션. **출하 실적이 있는 수주는 cancel 불가** — 자식부터 정리하는 패턴 (회계 추적 무너지지 않게).
- **HTTP 상태 코드의 의미 차이** — 400(입력 형식 오류) / 404(자원 없음) / 409(자원은 있는데 현재 상태에서 그 동작 거부, 즉 도메인 규칙 위반). 클라이언트가 UX 를 다르게 처리할 수 있게 만드는 약속.
