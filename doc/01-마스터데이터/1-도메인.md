# 01. 마스터 데이터 — ① 도메인 편 : ERP 의 "명사 사전"

> **이 글의 목적**: ERP 의 모든 거래(출하·입고·전표…)가 **무엇을 딛고 서 있는지** — 그 토대인
> **마스터 데이터**를 업무 관점에서 이해한다. 왜 거래와 따로 관리하는지, 왜 함부로 지우지 않는지.
> 코드(엔티티/CRUD)는 다음 글(② 구현 편)에서.

---

## 1. 두 종류의 데이터 — 명사와 동사

ERP 안의 데이터는 크게 두 부류로 나뉜다.

| | 마스터 데이터 (Master) | 트랜잭션 데이터 (Transaction) |
| --- | --- | --- |
| 정체 | **명사** — 회사가 다루는 *대상* | **동사** — 그 대상들 사이에 *벌어진 사건* |
| 예 | 고객, 품목, 창고, 직원 | 수주, 출하, 입고, 전표, 급여 |
| 성격 | 비교적 **고정** (자주 안 바뀜) | 계속 **쌓인다** (사건마다 새 레코드) |
| 수명 | 오래 산다 (고객은 몇 년째 거래) | 한 건의 사건 (그 출하 한 번) |

> **한 문장으로**: 마스터는 "**누가·무엇을**", 트랜잭션은 "**언제 무슨 일이**".
> "**○○회사**(고객)에 **노트북**(품목)을 **출하**(트랜잭션)했다" — 굵은 둘이 마스터, 출하가 트랜잭션.

ERP 를 배울 때 마스터부터 보는 이유가 이것이다. **명사를 먼저 정의해야 동사로 문장을 쓸 수 있다.**
고객·품목이 등록돼 있어야 "그 고객에게 그 품목을 판다(수주)" 가 성립한다.

---

## 2. 왜 따로 관리하나 — "한 번 정의, everywhere 참조"

노트북이라는 품목을 생각해 보자. 이 노트북은 **수주·출하·입고·재고·생산·매출원가 전표** 등
수십 군데서 등장한다. 만약 거래마다 "노트북, 표준원가 60만원" 을 **매번 새로 적는다면**?

- 가격이 바뀌면 **수십 군데를 다 고쳐야** 한다 → 누락 → 데이터 불일치
- 같은 노트북인데 거래마다 철자·코드가 달라짐 → 집계가 안 됨

그래서 ERP 는 노트북을 **품목 마스터에 딱 한 번** 정의하고, 모든 거래는 그걸 **참조(코드로 가리킴)**한다.

```
[품목 마스터]  노트북  (ITEM-001, 표준원가 60만)
      ▲   ▲   ▲
      │   │   └── 매출원가 전표가 참조
      │   └────── 재고가 참조
      └────────── 수주·출하가 참조
```

> **단일 정의(single source of truth)**: 노트북 정보는 한 곳에만. 바뀌면 거기 한 번만 고치면 끝.
> 이게 마스터를 거래와 분리하는 가장 큰 이유다.

---

## 3. 우리 시스템의 마스터 6종

우리 ERP(`com.hwlee.erp.master` 패키지)는 6개의 마스터를 둔다. 각각이 어떤 거래의 "주어/목적어" 인지가 핵심:

| 마스터 | 무엇 | 주로 어디서 쓰이나 |
| --- | --- | --- |
| **고객 (Customer)** | 우리가 **파는** 상대 | 수주·출하·세금계산서·매출채권 (SD/FI) |
| **거래처 (Vendor)** | 우리가 **사는** 상대 (공급사) | 입고·매입채무 (MM/FI) |
| **품목 (Item)** | 사고파고 만드는 **물건** | 거의 모든 모듈 (SD·MM·PP·FI) |
| **창고 (Warehouse)** | 재고가 **있는 장소** | 재고·입고·출고 (MM) |
| **부서 (Department)** | 조직의 **단위** | 직원 소속, 권한 (HR/보안) |
| **직원 (Employee)** | 일하는 **사람** | 급여·근태, 로그인 계정 연결 (HR/보안) |

이 6종이 등록돼 있어야 02(SD)·03(MM) 이하의 모든 거래가 돌아간다.

### 고객 ↔ 거래처는 왜 나눌까?
둘 다 "거래 상대" 인데 **방향이 반대**다. 고객은 돈이 **들어오는**(매출·채권) 쪽, 거래처는 돈이
**나가는**(매입·채무) 쪽. 회계 계정도 매출채권(1200) vs 매입채무(2100)로 갈린다. 그래서 별도 마스터다.

---

## 4. 마스터가 갖는 공통 성질

개별 마스터는 달라도, 마스터라면 보통 다음을 갖는다.

### (1) 식별 코드 — 사람이 읽는 열쇠
DB 의 내부 id(숫자) 와 별개로, **업무용 코드**를 둔다: 고객 `CUST-001`, 품목 `ITEM-001`,
창고 `WH-001`. 사람이 알아보고, 화면·문서에서 부르고, 중복을 막는(unique) 열쇠다.

### (2) 분류·속성 — 거래의 동작을 바꾸는 메타
단순한 이름표가 아니라, **거래 로직을 분기**시키는 속성을 담는다. 예:
- 품목의 **유형**(완제품 FINISHED / 부품 COMPONENT) → 매입 전표에서 재고 계정이 **제품(1400) vs 원재료(1410)**로 갈림 (05 에서 봤다)
- 고객의 **결제조건·신용한도** → 수주 확정 시 "한도 초과면 막기" 판단에 쓰임 (02 에서)

→ 마스터의 속성 하나가 트랜잭션의 분개·검증을 바꾼다. 마스터는 "정적인 참조표" 그 이상이다.

### (3) 상태 — 지우지 않고 "비활성"
마스터는 함부로 **삭제하지 않는다.** 왜?

> 3년 전 거래한 고객을 DB 에서 지우면, 그 고객을 참조하던 **옛 수주·전표가 미아**가 된다.
> 과거 장부가 "유령 고객" 을 가리키게 되는 것 — 회계·감사에서 치명적.

그래서 "더 이상 거래 안 함" 은 **삭제가 아니라 상태를 비활성(inactive)으로** 바꾼다. 과거 기록은
온전히 보존되고, 새 거래에서만 그 고객을 못 고르게 한다. (참조 무결성 + 감사 추적)

---

## 5. 마스터가 흔들리면 — 전체가 흔들린다

마스터는 토대라서, 여기가 부실하면 위층 거래가 전부 부실해진다. 전형적 사고:

- **중복 등록**: 같은 고객이 "○○회사" / "(주)○○" 로 두 번 등록 → 매출 집계가 둘로 쪼개짐
- **표준 정보 오류**: 품목 표준원가가 틀림 → 매출원가·재고평가가 전부 틀어짐
- **막 지움**: 거래처를 삭제 → 과거 매입 전표의 거래처가 깨짐

그래서 마스터는 **등록 단계의 검증(코드 중복·필수값)** 과 **삭제 대신 비활성** 으로 단단히 지킨다.
ERP 품질의 8할은 마스터 품질이라는 말이 있을 정도다.

---

## 요약

1. 데이터는 **마스터(명사: 고객·품목·창고…)** 와 **트랜잭션(동사: 수주·출하·전표…)** 으로 나뉜다
2. 마스터를 분리하는 이유 = **단일 정의, 모든 거래가 참조** (바뀌면 한 곳만 수정 → 불일치 방지)
3. 우리 시스템 6종: **고객·거래처·품목·창고·부서·직원**. 고객/거래처는 돈의 방향(매출/매입)으로 구분
4. 마스터 공통 성질: **식별 코드** / **거래를 분기시키는 속성**(품목 유형·신용한도) / **삭제 대신 비활성**
5. 마스터가 흔들리면 위층 거래·집계·회계가 전부 흔들린다 → 검증과 비활성으로 보호
