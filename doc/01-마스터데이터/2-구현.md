# 01. 마스터 데이터 — ② 구현 편 : 엔티티·채번·CRUD·비활성

> **이 글의 목적**: ① 도메인 편의 마스터 4성질 — **식별 코드 / 거래를 분기시키는 속성 / 단일 정의 /
> 삭제 대신 비활성** — 이 실제 코드에서 어떻게 표현됐는지 본다. 고객(Customer)·품목(Item) 엔티티,
> 동시성 안전 채번기, CRUD 서비스, 그리고 "지우지 않는 삭제(Soft Delete)" 까지. 패키지는 `com.hwlee.erp.master`.

---

## 1. 공통 토대 — `BaseEntityWithCode`

마스터 6종은 공통 베이스를 상속한다. 도메인 편의 "마스터 공통 성질" 이 그대로 필드로 들어가 있다:

```java
@MappedSuperclass
public abstract class BaseEntityWithCode extends BaseEntity {   // BaseEntity = id + 생성/수정 시각·작성자

    @Column(name = "code", nullable = false, unique = true, length = 30, updatable = false)
    private String code;                       // ① 식별 코드 — unique, updatable=false(한 번 정하면 불변)

    @Enumerated(EnumType.STRING)
    @Column(name = "status", nullable = false, length = 16)
    private MasterStatus status = MasterStatus.ACTIVE;   // ④ 상태 — 기본 ACTIVE

    @Column(name = "deleted_at")
    private LocalDateTime deletedAt;           // ④ Soft Delete 표시 (5절)

    protected void assignCode(String code) {
        if (this.code != null) throw new IllegalStateException("code 는 한 번만 할당");  // 코드 불변 보장
        this.code = code;
    }
    public void changeStatus(MasterStatus newStatus) { ... this.status = newStatus; }
    public boolean isDeleted() { return deletedAt != null; }
}
```

읽을 점:
- `code` 는 `unique` + `updatable = false` — **중복 불가 + 한 번 정하면 안 바뀜**. 도메인 편 "식별 코드 = 열쇠".
- 마스터엔 **`status`(활성/비활성)** 와 **`deletedAt`(삭제 표시)** 가 *둘 다* 있다. 둘은 역할이 다르다 — 5절에서.
- `code` 발급은 엔티티가 직접 안 한다 — `assignCode(code)` 로 **외부에서 받아 주입**한다. 누가 발급? → 2절.

## 2. 식별 코드 채번 — `CodeGenerator` (동시성 안전)

도메인 편의 `CUST-001` 같은 코드는 누가 만드나? `common.code.CodeGenerator` 가 **마스터·트랜잭션 공용**으로 발급한다.

```java
// 형식: <PREFIX>-<기간키>-<0패딩 일련번호>
//   마스터(연 단위):    CUST-2026-0042   (4자리)
//   트랜잭션(일 단위):  SO-20260524-001  (3자리)

@Transactional(propagation = Propagation.REQUIRES_NEW)   // ← 별도 트랜잭션에서 발급
public String nextCode(String prefix) {
    String periodKey = String.valueOf(LocalDate.now(clock).getYear());  // "2026"
    int issued = issue(prefix, periodKey);
    return String.format("%s-%s-%04d", prefix, periodKey, issued);      // CUST-2026-0001
}
```

설계의 핵심 두 가지:

### (a) 왜 별도 트랜잭션(`REQUIRES_NEW`)인가
번호 발급을 **호출자 트랜잭션과 분리**한다. 고객 등록이 나중에 실패해도, 발급된 번호는 이미 커밋돼
"번호 구멍"(결번)이 생길 수 있다. 하지만 그 대신 **번호가 겹치지 않음(동시성 안전)** 을 얻는다 —
결번보다 중복이 훨씬 위험하므로 이 트레이드오프를 택했다.

### (b) 동시에 두 명이 등록하면? — `SELECT … FOR UPDATE`
같은 순간 두 트랜잭션이 `CUST` 다음 번호를 달라고 하면 같은 번호가 나갈 위험이 있다. 그래서:

```java
CodeSequence sequence = repository.findForUpdate(prefix, periodKey)   // 행을 락(FOR UPDATE)으로 점유
        .orElseGet(() -> createOrLockExisting(prefix, periodKey));    // 없으면 INSERT, 경합 시 재조회
return sequence.issueAndIncrement();                                  // 락 잡은 채 +1
```

`code_sequence` 테이블의 `(prefix, periodKey)` 행을 **락으로 잡고** 증가시키므로, 한 번에 한 트랜잭션만
번호를 가져간다. 행이 아직 없으면 INSERT 하되, UNIQUE 위반(다른 트랜잭션이 먼저 만듦)이 나면 다시
락 걸고 조회해 합류한다 — 동시성의 빈틈을 막는다.

> **마스터 vs 트랜잭션 채번 차이**: 마스터는 **연 단위 4자리**(`CUST-2026-0001`), 트랜잭션은 **일 단위 3자리**
> (`SO-20260524-001`). 고객은 한 해에 수백 명, 수주는 하루에 수백 건이라 기간 단위가 다르다.

## 3. 엔티티 — 속성이 곧 "거래 분기 메타"

### 고객 (Customer)

```java
@Entity @Table(name = "customer")
public class Customer extends BaseEntityWithCode implements Auditable {
    private String name;
    private String businessNo;           // 사업자번호 — unique, 수정 불가(외부 식별자)
    private String address;
    private BigDecimal creditLimit;       // ② 신용한도 — 02 수주 확정에서 "한도 초과 막기"에 쓰임
    private PaymentTerms paymentTerms;    // ② 결제조건(enum) — 외상 기일 정책

    public static Customer create(String code, String name, String businessNo,
                                  String address, BigDecimal creditLimit, PaymentTerms paymentTerms) {
        validate(name, businessNo, creditLimit, paymentTerms);   // 생성 시 필수값·범위 검증
        Customer c = new Customer();
        c.assignCode(code);   // 외부(서비스)가 채번해 넘긴 코드 주입
        c.name = name; c.businessNo = businessNo; ...
        return c;
    }
    public void update(String name, String address, BigDecimal creditLimit, PaymentTerms paymentTerms) {
        // ... business_no 는 매개변수에 없다 = 수정 불가(외부 식별자라 불변)
    }
}
```

도메인 편 "거래를 분기시키는 속성" 이 여기 산다: `creditLimit`(신용한도)는 02 에서 **수주 확정 시 한도
초과를 막는** 판단에 직접 쓰인다. 마스터 필드 하나가 트랜잭션의 검증 로직을 바꾼다.

또 **생성자가 `static create(...)` 팩토리** 이고, 안에서 `validate(...)` 를 먼저 부른다 — *불완전·불량한
마스터가 애초에 만들어질 수 없게* 입구에서 막는다(도메인 편 "등록 단계 검증"). `businessNo` 가
`update` 인자에 없는 것도 의도적 — **외부 식별자는 한 번 정하면 못 바꾼다**(코드 주석에 명시).

### 품목 (Item)

```java
@Entity @Table(name = "item")
public class Item extends BaseEntityWithCode {
    private String name;
    private ItemCategory category;        // 분류(제품/부품 등)
    private ItemType itemType = ItemType.FINISHED;  // ② 완제품/부품 — 05 매입분개에서 재고계정 분기!
    private ItemUnit unit;                // 단위(EA 등)
    private BigDecimal standardCost;       // 표준원가 — 재고평가·매출원가 기준
    private BigDecimal standardPrice;      // 표준판매가
}
```

주석이 품목을 *"영업/구매/재고/생산 모두가 참조하는 ERP 의 혈관"* 이라 부른다 — 도메인 편 2절의
"노트북 한 품목이 수십 군데서 참조됨" 이 그대로다. 특히 **`itemType`(FINISHED/COMPONENT)** 은 05 에서
본 그 필드 — 매입 전표에서 재고 계정을 **제품(1400) vs 원재료(1410)** 로 가르는 분기 키다.

> 품목 검증에서 흥미로운 결정 하나: `standardPrice >= standardCost`(원가 이상 판매)는 **검증하지 않는다.**
> 실무에선 손해 보고 파는 경우(재고 떨이 등)도 있어서 — 코드 주석이 그 판단을 남겨 뒀다.

## 4. CRUD 서비스 — 채번·검증·매핑의 조립

`CustomerService` 가 도메인 편의 성질들을 실제 흐름으로 엮는다:

```java
@Service @Transactional(readOnly = true)   // 기본 읽기전용, 쓰기 메서드만 @Transactional 로 열기
public class CustomerService {
    static final String CODE_PREFIX = "CUST";

    @Transactional
    public CustomerResponse create(CustomerCreateRequest req) {
        if (repository.existsByBusinessNo(req.businessNo()))            // ① 중복 방지(사업자번호)
            throw new IllegalStateException("이미 등록된 사업자번호입니다: " + req.businessNo());
        String code = codeGenerator.nextCode(CODE_PREFIX);             // ② 채번 (CUST-2026-XXXX)
        Customer customer = Customer.create(code, req.name(), ...);    // ③ 팩토리(내부 검증)
        return mapper.toResponse(repository.save(customer));           // ④ 저장 + DTO 변환
    }

    public Page<CustomerResponse> search(Specification<Customer> spec, Pageable pageable) {
        return repository.findAll(spec, pageable).map(mapper::toResponse);  // 동적 조건 + 페이징
    }
    // update / delete / findById / findByCode ...
}
```

읽을 점:
- **중복 방지가 두 겹**: 서비스가 `existsByBusinessNo` 로 미리 막고(친절한 에러), DB `unique` 제약이
  최후의 방어선(동시 등록 경합). 도메인 편 "중복 등록 사고" 를 코드·DB 양쪽에서 막는다.
- **DTO 분리**: 외부엔 `CustomerResponse`(응답)·`CustomerCreateRequest`(입력) 로만 노출하고, 엔티티는
  `Mapper` 로 변환. 엔티티 내부 구조가 API 로 새지 않게 한다.
- **`search(Specification, Pageable)`**: 목록은 동적 조건(이름·상태 검색) + 페이징. 마스터는 건수가
  많을 수 있어 전체 로드 대신 페이지 단위로 본다.

## 5. "지우지 않는 삭제" — Soft Delete ⭐

도메인 편 4-(3) "삭제 대신 비활성" 이 코드에선 **두 가지 메커니즘**으로 나뉜다. 헷갈리기 쉬우니 구분하자.

| | `status` (MasterStatus) | `deletedAt` (Soft Delete) |
| --- | --- | --- |
| 의미 | "거래 가능 / 일시 중지(INACTIVE)" — **업무 상태** | "삭제됨" — **존재 자체를 숨김** |
| 바꾸는 법 | `changeStatus(INACTIVE)` (명시적 업무 행위) | `delete()` 호출 → 자동 변환 |
| 조회 시 | 그대로 보임(상태로 필터 가능) | **아예 안 보임**(자동 제외) |

### delete 가 진짜로 지우지 않는 트릭

`Customer` 엔티티엔 두 개의 Hibernate 어노테이션이 붙어 있다:

```java
@SQLDelete(sql = "UPDATE customer SET deleted_at = NOW() WHERE id = ?")  // DELETE → UPDATE 로 가로채기
@SQLRestriction("deleted_at IS NULL")                                   // 모든 조회에 자동으로 이 조건 추가
public class Customer extends BaseEntityWithCode implements Auditable { ... }
```

그래서 서비스가 평범하게 `repository.delete(customer)` 를 불러도:

```java
@Transactional
public void delete(Long id) {
    Customer customer = getOrThrow(id);
    repository.delete(customer);   // @SQLDelete 에 의해 실제로는 UPDATE customer SET deleted_at=NOW()
}
```

- **물리 삭제가 안 일어난다.** 행은 남고 `deleted_at` 에 시각만 찍힌다 → 도메인 편의 "과거 전표가
  미아가 되는 사고" 가 원천 차단. 과거 수주가 가리키던 고객 행은 그대로 존재한다.
- 그런데 **일반 조회에는 안 나온다.** `@SQLRestriction("deleted_at IS NULL")` 이 모든 SELECT 에
  자동으로 붙어, 삭제된 고객은 목록·검색에서 사라진다. "새 거래에선 못 고르게" 가 이걸로 달성된다.

> **정리**: `status=INACTIVE` 는 "잠깐 거래 중단"(되살리기 쉬움), Soft Delete 는 "삭제했지만 과거를 위해
> 데이터는 보존". 둘 다 **물리 삭제를 피해 참조 무결성과 감사 추적을 지킨다** — 도메인 편의 핵심 실천.

### 덤 — 감사 추적(`Auditable`)
`Customer` 는 `implements Auditable` 로 변경 이력을 남긴다. `auditSnapshot()` 이 어떤 필드(code·name·
creditLimit·status…)를 추적할지 명시적으로 고른다. 마스터 변경은 "누가 언제 신용한도를 바꿨나" 가
중요해서다(06 인증/감사에서 깊게).

## 요약

1. 마스터 6종은 `BaseEntityWithCode`(code unique·불변 + status + deletedAt) 공통 베이스를 상속
2. **채번 `CodeGenerator`**: `REQUIRES_NEW`(번호 안 겹치게 별도 트랜잭션) + `SELECT … FOR UPDATE`(동시성).
   마스터=연 4자리(CUST-2026-0001), 트랜잭션=일 3자리(SO-20260524-001)
3. **엔티티 속성 = 거래 분기 메타**: 고객 `creditLimit`(수주 한도검증), 품목 `itemType`(매입 재고계정 분기).
   `static create` 팩토리가 입구에서 검증, 외부 식별자(businessNo)는 수정 불가
4. **CRUD 서비스**: 중복 방지(서비스 + DB unique 이중), DTO 분리, 동적검색+페이징
5. **Soft Delete ⭐**: `@SQLDelete`(DELETE→deleted_at UPDATE) + `@SQLRestriction`(조회 자동 제외) →
   물리 삭제 없이 "과거 보존 + 새 거래에서 숨김". `status`(업무 중지)와는 별개 메커니즘

> 어떻게 흐르는지, 그리고 신용한도(creditLimit)가 수주 확정에서 어떻게 작동하는지.
