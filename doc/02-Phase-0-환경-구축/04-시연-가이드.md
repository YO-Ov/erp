# Phase 0 시연 가이드

> 7단계 사이클의 마지막 — **단계 7: 시연 + 회고**.
> 지금까지 도메인 브리핑/모델링/코드 워크스루를 "읽기"로 학습했다면, 이번에는 직접 손으로 동작시켜 본다.
> 이 가이드는 **위에서 아래로 순서대로** 따라 한다. 각 단계마다 "확인 포인트"를 두어 어디서 막혔는지 즉시 알 수 있게 했다.

---

## 시연의 목표 (왜 이걸 직접 돌려보는가)

코드 워크스루로 "왜 이렇게 짰는지"는 머리로 이해했다. 하지만 기간계 학습에서는 **실제로 한 줄도 안 빼고 끝까지 동작하는 것**을 직접 확인하는 게 중요하다. Phase 1부터는 트랜잭션/동시성/연계가 들어가기 때문에, **"내 환경이 정상이다"** 라는 기준점이 Phase 0에서 박혀 있어야 한다.

이번 시연에서 확인할 6가지:
1. Docker Compose로 MySQL 8 컨테이너가 뜬다
2. Spring Boot 앱이 8080 포트로 부팅된다
3. **Flyway**가 `V1__init.sql`을 자동 적용한다 (= 스키마 버전 관리가 동작한다)
4. **헬스체크 API**가 200 OK와 함께 DB까지 OK라고 응답한다
5. **Swagger UI** 화면에서 API 호출이 가능하다 (= 화면 없이도 API를 직접 시험할 수 있다)
6. **Testcontainers** 기반 통합 테스트가 통과한다 (= CI에서도 DB 없이 자체 테스트 가능)

## 사전 점검 (Pre-flight check)

작업 디렉토리는 항상 ERP 프로젝트 루트 기준이다.

```bash
cd /Users/hwlee/IdeaProjects/my-app/erp
```

도구 버전 확인:

```bash
docker --version          # Docker 20+ 권장
docker compose version    # v2 형식 (compose 명령이 docker 하위)
java --version            # 21 (Temurin/Zulu 등 어느 배포판이든 21이면 됨)
./hwlee-erp/gradlew --version   # Gradle 8.x
```

확인 포인트:
- Java가 **21이 아니면** Spring Boot 3.3은 부팅되지 않는다. `jenv`나 SDKMAN으로 21로 전환.
- Docker Desktop이 **실행 중**인지 확인 (Mac은 메뉴바 아이콘이 살아 있어야 함).
- 8080 / 3307 포트가 이미 사용 중이지 않은지: `lsof -i :8080` / `lsof -i :3307`.

## STEP 1. MySQL 컨테이너 띄우기

```bash
docker compose up -d
```

- `-d`는 detached 모드 (백그라운드 실행).
- 처음 실행이면 mysql:8.0 이미지를 받느라 1~2분 걸린다.

상태 확인:

```bash
docker compose ps
```

기대 출력 (요약):
```
NAME                IMAGE       STATUS                   PORTS
hwlee-erp-mysql     mysql:8.0   Up X seconds (healthy)   0.0.0.0:3307->3306/tcp
```

확인 포인트:
- **STATUS가 `(healthy)`** 가 되어야 다음 단계로 넘어간다. healthy로 바뀌기까지 보통 10~20초.
- `(unhealthy)` 가 뜨면 `docker compose logs mysql`로 원인 확인 (대부분 포트 충돌 또는 볼륨 권한).

부팅 로그를 흘려보고 싶다면:

```bash
docker compose logs -f mysql
```

`ready for connections` 라인이 보이면 OK. `Ctrl+C`로 빠져나오기.

---

## STEP 2. MySQL 접속해서 DB 확인 (선택)

Spring 앱을 띄우기 전, **DB가 비어 있다**는 것을 한번 눈으로 확인한다. 그래야 Flyway가 일을 한 후 어떻게 변했는지 비교가 된다.

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "SHOW TABLES;"
```

기대 출력: `Empty set`.

확인 포인트: 아직 어떤 테이블도 없음. 이게 정상.

---

## STEP 3. Spring Boot 앱 실행

```bash
cd hwlee-erp
./gradlew bootRun
```

- 첫 실행이면 Gradle 의존성 다운로드로 1~2분 걸린다.
- `bootRun`은 로컬 프로파일(`application-local.yml`)을 자동으로 함께 사용한다 (application.yml의 `profiles.active: local`).

부팅 로그에서 **반드시 확인할 4가지**:

```
o.f.c.i.database.base.BaseDatabaseType   : Database: jdbc:mysql://localhost:3307/erp_db (MySQL 8.0)
o.f.core.internal.command.DbMigrate      : Migrating schema `erp_db` to version "1 - init"
o.f.core.internal.command.DbMigrate      : Successfully applied 1 migration to schema `erp_db`
o.s.b.w.embedded.tomcat.TomcatWebServer  : Tomcat started on port 8080 (http)
```

확인 포인트:
1. **Flyway가 3307 MySQL에 붙었다** — docker compose의 MySQL을 잡았다는 증거.
2. **`Migrating schema 'erp_db' to version "1 - init"`** — V1__init.sql을 실행 중.
3. **`Successfully applied 1 migration`** — V1__init.sql 적용 완료. 이 시점에 `schema_init_marker` 테이블이 생기고 1행 insert된다.
4. **`Tomcat started on port 8080`** — HTTP 서버 기동 완료. 이제 API 호출 가능.

> 만약 `ddl-auto: validate`인데 엔티티가 없다고 에러가 나면? — Phase 0에는 아직 엔티티가 없어서 validate가 비교할 대상이 없다 → 에러 없이 통과한다. Phase 1부터는 엔티티가 추가되면서 이 옵션이 "스키마가 엔티티와 일치하는지" 검증 역할을 시작한다.

## STEP 4. 헬스체크 API 호출 (cURL)

새 터미널을 열어서 (앱은 그대로 켜둔 채):

```bash
curl -i http://localhost:8080/api/health
```

기대 응답:

```
HTTP/1.1 200 OK
Content-Type: application/json

{"status":"UP","db":"OK"}
```

확인 포인트:
- **HTTP 200** — 컨트롤러가 정상 응답.
- **`"status":"UP"`** — Spring 컨텍스트가 살아 있다는 의미.
- **`"db":"OK"`** — HealthController가 `DataSource.getConnection()`을 받아서 `SELECT 1`을 성공시킨 결과. 즉, **앱이 DB까지 정확히 연결됐다는 종단 확인**.

만약 `db":"FAIL: ..."` 가 나오면? → docker compose가 죽었거나, 3307이 아닌 다른 포트로 부팅됐을 가능성. STEP 1로 돌아간다.

## STEP 5. Swagger UI에서 호출

브라우저에서:

```
http://localhost:8080/swagger-ui.html
```

확인 포인트:
- Swagger UI 페이지가 뜨고 좌측에 `health-controller` 그룹이 보인다.
- `GET /api/health` 펼치기 → **Try it out** → **Execute**.
- 응답 본문에 `{"status":"UP","db":"OK"}` 가 표시된다.

이게 왜 중요한가: ERP 학습 Phase 1~5는 **프론트엔드 없이 Swagger UI만으로 모든 API를 호출**한다. Phase 0에서 이 화면이 동작한다는 것을 손으로 확인해 두는 게 Phase 1의 출발점이다.

OpenAPI 스펙 자체도 직접 받아 보자:

```bash
curl -s http://localhost:8080/api-docs | head -40
```

— `openapi: 3.x.x`로 시작하는 JSON이 나오면 OK.

## STEP 6. DB에 Flyway 흔적이 남았는지 확인

Spring 앱은 띄워 둔 채로, 다시 새 터미널에서:

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "SHOW TABLES;"
```

기대 출력:
```
+----------------------+
| Tables_in_erp_db     |
+----------------------+
| flyway_schema_history|
| schema_init_marker   |
+----------------------+
```

두 테이블의 내용:

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e \
  "SELECT version, description, success FROM flyway_schema_history;"

docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e \
  "SELECT * FROM schema_init_marker;"
```

기대:
- `flyway_schema_history`에 `version=1, description='init', success=1` 행이 있다.
- `schema_init_marker`에 `'Phase 0 init OK'` 행이 있다.

확인 포인트:
- **이게 Flyway의 본질이다**. 스키마 변경을 SQL 파일로 관리하고, **언제, 누가, 어떤 버전을 적용했는지 DB에 기록**한다. Phase 1부터 마스터 테이블이 추가될 때마다 V2, V3, ... 파일이 쌓이고 이 이력이 누적된다.

## STEP 7. 통합 테스트 실행 (Testcontainers)

Spring 앱은 끄지 않아도 된다. 별도 터미널에서:

```bash
cd /Users/hwlee/IdeaProjects/my-app/erp/hwlee-erp
./gradlew test
```

처음 실행이면 testcontainers/mysql:8.x 이미지를 받느라 1~2분 더 걸린다 (한번만).

기대 출력 끝부분:
```
BUILD SUCCESSFUL in Xs
```

확인 포인트:
- **`HealthIntegrationTest > 헬스체크는_UP을_반환하며_DB도_OK다 PASSED`** 가 나와야 한다.
- 이 테스트는 **docker compose의 MySQL을 쓰지 않는다**. Testcontainers가 별도의 일시적 MySQL 컨테이너를 띄워서 거기에 Flyway까지 돌리고, MockMvc로 `/api/health`를 호출해 응답을 검증한다.
- 즉, **개발자 환경에서 docker compose가 꺼져 있어도, CI 서버에서 Docker만 있으면 테스트가 통과**한다는 의미.

테스트 리포트는 다음 경로에서 HTML로 볼 수 있다:
```
hwlee-erp/build/reports/tests/test/index.html
```

## STEP 8. 정리

시연이 끝났으면 환경을 정리한다 (그대로 둬도 무방하지만, 다음 학습 시 깨끗하게 시작하고 싶다면).

Spring 앱 종료: 앱이 떠 있는 터미널에서 `Ctrl+C`.

컨테이너 정지 (데이터는 보존):
```bash
cd /Users/hwlee/IdeaProjects/my-app/erp
docker compose stop
```

컨테이너 + 데이터 모두 제거 (다음에 Flyway 초기 마이그레이션부터 다시 시연하고 싶을 때):
```bash
docker compose down -v
```

> `-v` 옵션은 볼륨(저장된 MySQL 데이터)까지 삭제한다. **Phase 1부터는 신중**히. Phase 0에서는 데이터가 더미라 자유롭게 지워도 된다.

## 트러블슈팅 (자주 만나는 4가지)

### 1. `Port 3307 is already in use`
- 다른 MySQL 컨테이너가 같은 포트를 잡고 있다. `docker ps`로 확인 후 정리.
- 또는 호스트 MySQL이 3307에 바인딩되어 있을 수 있다 (드물지만 가능).

### 2. `Tomcat started on port 8080`이 안 뜨고 `Port 8080 is already in use`
- 다른 Spring 앱 또는 IntelliJ에서 띄운 앱이 8080을 쓰고 있다. `lsof -i :8080`으로 PID 확인 후 정리.

### 3. Flyway가 `Migration checksum mismatch` 오류
- V1__init.sql을 한번 적용한 후 내용을 수정했을 때 발생. **Phase 0에서는** `docker compose down -v`로 볼륨 날리고 다시 시작해도 된다. **Phase 1부터는** 절대 적용된 마이그레이션을 수정하지 않는다 (운영 DB에선 복구 불가).

### 4. `./gradlew test`가 `Could not find a valid Docker environment` 에러
- Testcontainers가 Docker 데몬을 못 찾는 경우. Docker Desktop이 켜져 있는지 확인.
- Mac에서 Docker 소켓 경로가 비표준이면 `~/.testcontainers.properties`에 `docker.host=unix:///Users/<you>/.docker/run/docker.sock` 같은 설정이 필요할 수 있다.

## 시연 완료 체크리스트

다음 6가지를 직접 확인했으면 Phase 0의 환경 검증은 끝이다.

- [ ] `docker compose ps`에서 MySQL이 `(healthy)`
- [ ] `./gradlew bootRun` 부팅 로그에 Flyway "Successfully applied 1 migration" 출력
- [ ] `curl http://localhost:8080/api/health` 가 `{"status":"UP","db":"OK"}` 응답
- [ ] 브라우저에서 Swagger UI 화면 표시 + `/api/health` 실행 성공
- [ ] DB에 `flyway_schema_history`, `schema_init_marker` 두 테이블 존재
- [ ] `./gradlew test`가 BUILD SUCCESSFUL

## 시연 후 — 무엇이 학습됐는가

이번 시연으로 **체감**으로 알게 된 것:

1. **Spring Boot 앱이 살아 있다 ≠ DB까지 동작한다**. 그래서 헬스체크가 DB까지 찍어 보는 것이다. Phase 1부터는 마스터 데이터가 들어오면서 헬스체크가 더 의미를 갖는다.
2. **Flyway가 스키마를 "코드"로 만든다**. 다음 사람이 이 레포를 받아 `docker compose up` + `./gradlew bootRun`만 하면 같은 스키마가 자동으로 재현된다 — 이게 기간계 운영의 출발점.
3. **Testcontainers는 "Docker만 있으면 어디서든"** 통합 테스트가 돈다. CI/CD에서 별도의 DB 인프라 없이 매 PR마다 실제 MySQL에 대고 테스트할 수 있다는 의미.

Phase 1에서는 이 토대 위에 **고객/상품/거래처/직원** 마스터 테이블이 V2 마이그레이션으로 추가되고, 각각의 CRUD API가 Swagger UI에 채워지기 시작한다.
