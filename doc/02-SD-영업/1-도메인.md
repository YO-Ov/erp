# 02. SD(영업) — ① 도메인 편 : 주문에서 현금까지 (OTC)

> **이 글의 목적**: 01 에서 깔아 둔 **고객·품목**을 딛고, 영업이 어떻게 **견적 → 수주 → 출하 →
> 세금계산서** 로 흘러 매출과 현금이 되는지(OTC, Order-to-Cash)를 업무 관점에서 이해한다.
> 각 단계가 왜 따로 있는지, 신용한도가 어디서 작동하는지. 코드는 ② 구현 편에서.

---

## 1. OTC — 영업의 큰 줄기

SD(Sales & Distribution)는 **물건을 팔아 돈을 받기까지의 전 과정**을 다룬다. 이 흐름을
**OTC(Order-to-Cash, 주문에서 현금까지)** 라 부른다. 우리 시스템은 4단계로 모델링했다:

```
 [견적]  ──▶  [수주]  ──▶  [출하]  ──▶  [세금계산서]  ──▶ (입금)
Quotation   SalesOrder   Delivery     Invoice        Payment(FI)
 "팔까요?"   "삽니다"     "보냈다"     "청구한다"      "받았다"
```

| 단계 | 무슨 일 | 핵심 질문 |
| --- | --- | --- |
| **견적 (Quotation)** | 고객에게 "이 가격에 팝니다" 제안 | 아직 **약속 아님** — 거절·만료될 수 있음 |
| **수주 (SalesOrder)** | 고객이 "그 가격에 사겠다" 주문 확정 | 여기서 **신용한도 점검** ⭐ |
| **출하 (Delivery)** | 창고에서 물건을 실제로 내보냄 | 여기서 **재고 차감 + 매출원가** (04·05) |
| **세금계산서 (Invoice)** | 보낸 물건값을 공식 청구 | 여기서 **매출·부가세·매출채권** (05) |

핵심: **한 거래를 4개의 문서로 쪼갠다.** 왜 한 번에 안 하고 나눌까? → 2절.

---

## 2. 왜 4단계로 쪼개나 — 단계마다 "되돌릴 수 없는 선"이 다르다

각 단계는 **책임과 시점이 다르다.** 합치면 이 구분이 뭉개진다.

- **견적**은 *약속이 아니다.* 고객이 "생각해 볼게요" 하면 그냥 만료(EXPIRED)된다. 견적을 수주와
  같이 묶으면, "제안만 했는데 재고가 잡히고 매출이 잡히는" 사고가 난다.
- **수주**는 *법적 약속*이다. 여기서부터 회사가 "납품하겠다" 는 의무를 진다. 그래서 **신용한도 점검**
  같은 통제가 이 시점에 들어간다.
- **출하**는 *물리적 사건*이다. 실제로 물건이 창고를 떠난다 → 재고가 줄고 원가가 비용이 된다.
  "주문은 받았지만 아직 안 보냄" 과 "보냄" 은 명확히 구분돼야 한다.
- **세금계산서**는 *회계·세무 사건*이다. 매출을 인식하고 부가세를 계상한다. 출하와 분리해야
  "물건은 갔지만 아직 청구 전" 같은 현실 상태를 표현할 수 있다.

> 한 줄로: **쪼갬으로써 "어디까지 진행됐는지" 와 "무엇을 되돌릴 수 있는지" 가 명확해진다.**
> 견적 취소는 쉽지만, 출하 취소는 재고를 되돌려야 하고, 계산서 취소는 회계를 역분개해야 한다.

---

## 3. 각 단계의 상태머신 — 문서의 일생

01 마스터(고정적)와 달리, SD 문서는 **상태가 흐르는 트랜잭션 데이터**다. 각자 정해진 길을 따라 산다.

### 견적 — 제안하고, 받아들여지거나 사라진다
```
DRAFT ──발송──▶ SENT ──수락──▶ ACCEPTED
   │              └──기한경과──▶ EXPIRED
   └──취소──▶ CANCELLED
```
작성(DRAFT)→발송(SENT)→고객 수락(ACCEPTED). `valid_until` 이 지나면 자동으로 만료(EXPIRED).
수락된 견적은 **수주의 출발점**이 된다.

### 수주 — OTC 의 중심축, 가장 긴 일생
```
DRAFT ─확정─▶ CONFIRMED ─▶ SHIPPING ↔ SHIPPED ─▶ INVOICING ↔ INVOICED ─▶ CLOSED
            ↘ CANCELLED (DRAFT·CONFIRMED 에서만)
```
- **확정(CONFIRMED)**: 여기서 **신용한도 점검**(4절). 통과해야 확정된다.
- **SHIPPING/SHIPPED, INVOICING/INVOICED**: 출하·계산서 진행에 따라 상태가 자동으로 따라 움직인다.
- **취소는 DRAFT·CONFIRMED 까지만.** 이미 출하가 시작되면(SHIPPING~) 함부로 못 지운다 —
  물리적으로 물건이 나갔기 때문(2절의 "되돌릴 수 없는 선").
- 출하·청구가 다 끝나면 **CLOSED**(거래 종료).

### 출하 / 세금계산서 — 단순하고 단호하다
```
출하:        DRAFT ─확정─▶ SHIPPED ─▶ CANCELLED
세금계산서:  DRAFT ─발행─▶ ISSUED  ─▶ CANCELLED
```
둘 다 "작성 → 확정(SHIPPED/ISSUED) → 취소" 의 짧은 길. **확정 순간이 곧 사건 발생 시점**이다:
출하 확정 = 재고 차감·매출원가(04·05), 계산서 발행 = 매출·부가세·매출채권(05).

> 상태머신이 중요한 이유: **잘못된 전이를 막는다.** "DRAFT 견적을 바로 수주로", "취소된 출하를
> 다시 발송" 같은 불가능한 행동을 상태가 가로막아 데이터가 엉키지 않게 한다.

---

## 4. 신용한도 — 영업과 재무의 줄다리기 ⭐

OTC 에서 도메인적으로 가장 흥미로운 통제가 **신용한도(credit limit)** 다. 01 에서 본 고객 마스터의
`creditLimit` 필드가 여기서 일한다.

### 문제 상황
영업은 많이 팔고 싶다. 그런데 외상으로 100억을 주문한 고객이 돈을 못 갚으면? 회사가 손해다.
그래서 재무는 **고객별로 "외상 한도"** 를 정해 둔다 — "이 고객에겐 5천만원까지만 외상."

### 통제 지점 = 수주 확정
```
수주 확정 시점:
   미수금(이미 외상으로 깔린 금액) + 이번 주문액  >  신용한도 ?
        예 → 확정 거부 (이 주문은 못 받음)
        아니오 → 확정 진행
```

이 점검을 **수주 확정(CONFIRMED) 시점**에 두는 게 핵심이다:
- 견적 때 막으면 너무 이르다(아직 약속 아님).
- 출하 때 막으면 너무 늦다(이미 창고가 준비 중).
- **수주 확정 = "회사가 의무를 지는 순간"** 이라, 딱 여기서 "이 고객을 믿고 외상 줘도 되나" 를 판단한다.

> 이게 ERP 가 단순 주문관리와 다른 점이다 — 영업의 행동(수주)에 **재무의 통제(한도)**가 자동으로
> 끼어든다. 사람이 매번 "이 고객 외상 괜찮나?" 확인 안 해도 시스템이 막아 준다.

---

## 5. SD 가 다른 모듈을 깨우는 지점 (04·05 예고)

SD 는 혼자 끝나지 않는다. 출하·계산서 확정이 재고(MM)·회계(FI)를 자동으로 깨운다 — 04·05 에서 본 그 연계:

| SD 사건 | 깨어나는 모듈 | 결과 |
| --- | --- | --- |
| **출하 확정** | MM, FI | 재고 차감 + 매출원가 전표(차)매출원가/대)재고) |
| **세금계산서 발행** | FI | 매출 전표(차)매출채권/대)매출+부가세) |

SD 는 "출하했다 / 발행했다" 고 **이벤트만 발행**하고, MM·FI 리스너가 알아서 따라온다(느슨한 결합).
즉 **02 SD 는 04(연계)·05(회계)의 출발점**이다 — 이 글들의 모든 사건이 여기 SD 에서 시작한다.

---

## 요약

1. SD = **OTC(주문에서 현금까지)**: 견적 → 수주 → 출하 → 세금계산서 → (입금)
2. 4단계로 **쪼개는 이유**: 단계마다 책임·시점·"되돌릴 수 없는 선" 이 다르다 (제안/약속/물리/회계)
3. 각 문서는 **상태머신**으로 산다. 잘못된 전이를 막아 데이터가 엉키지 않게 함.
   수주가 가장 길고(확정~출하~청구~종료), 취소는 출하 전까지만
4. **신용한도 ⭐**: 고객 `creditLimit` 가 **수주 확정 시점**에 "미수금+주문액 > 한도" 를 점검해 거부.
   영업 행동에 재무 통제가 자동 개입
5. SD 는 **04·05 의 출발점** — 출하·계산서 확정이 재고·회계를 이벤트로 깨운다
