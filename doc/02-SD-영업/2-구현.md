# 02. SD(영업) — ② 구현 편 : 상태머신·신용한도·이벤트 발행

> **이 글의 목적**: ① 도메인 편의 OTC 4단계(견적→수주→출하→세금계산서)와 두 통제 —
> **상태머신**(잘못된 전이 차단)과 **신용한도**(수주 확정 시 한도 점검) — 가 실제 코드에서
> 어떻게 표현됐는지 본다. 견적의 "1회 소비", 수주 라인이 진실의 원천인 구조, `CreditLimitChecker`,
> 그리고 출하·발행이 04·05 를 깨우는 이벤트 발행 지점까지. 패키지는 `com.hwlee.erp.sd`.

---

## 0. 패키지 한눈에

SD 는 OTC 4단계가 그대로 하위 패키지로 나뉜다. 각 단계가 **엔티티 + 상태 + 서비스**의 한 묶음이다.

```
com.hwlee.erp.sd
 ├─ quotation/   Quotation, QuotationStatus, QuotationService      ← 견적
 ├─ order/       SalesOrder(+Line), SalesOrderStatus, SalesOrderService
 │   ├─ creditcheck/  CreditLimitChecker                           ← 신용한도 점검 ⭐
 │   └─ dto/          CreditStatusResponse …
 ├─ delivery/    Delivery(+Line), DeliveryService
 │   └─ event/        DeliveryShippedEvent, DeliveryCancelledEvent ← 04 접점(MM)
 └─ invoice/     Invoice(+Line), InvoiceService
     └─ event/        InvoiceIssuedEvent                           ← 05 접점(FI)
```

01 마스터(고정적)와 다른 점: SD 엔티티는 모두 **`status` 필드 + 상태 전이 메서드**를 갖는다. 도메인 편의
"문서의 일생" 이 곧 코드의 상태머신이다. 핵심 패턴 하나를 먼저 보자 — **모든 전이 메서드는 "현재 상태가
맞는지" 부터 검사한다.**

## 1. 견적 — 상태머신 메서드가 잘못된 전이를 막는다

`Quotation` 의 전이 메서드들은 전부 같은 모양이다: **"지금 이 상태에서만 이 행동이 가능"** 을 입구에서 강제한다.

```java
public void send() {
    if (status != QuotationStatus.DRAFT)
        throw new IllegalStateException("DRAFT 견적만 발송 가능합니다. 현재: " + status);
    if (lines.isEmpty())
        throw new IllegalStateException("라인이 비어 있는 견적은 발송할 수 없습니다.");
    this.status = QuotationStatus.SENT;
}

public void accept() {
    if (status != QuotationStatus.SENT)
        throw new IllegalStateException("SENT 견적만 수락 처리할 수 있습니다. 현재: " + status);
    this.status = QuotationStatus.ACCEPTED;
}

public void expire() { /* SENT 만 → EXPIRED */ }
public void cancel() { /* ACCEPTED·CANCELLED 가 아니면 → CANCELLED */ }
```

읽을 점:
- 도메인 편 상태도 `DRAFT ─발송─▶ SENT ─수락─▶ ACCEPTED` 가 **그대로 if 가드**로 들어갔다. "DRAFT 견적을
  바로 수락" 같은 불가능한 전이는 `IllegalStateException` 으로 막힌다 — 도메인 편 "잘못된 전이를 막는다".
- `send()` 는 상태뿐 아니라 **라인이 비었는지**도 본다. 빈 견적을 고객에게 보내는 사고를 입구에서 차단.
- 상태 변경은 **오직 엔티티 메서드를 통해서만**. 서비스가 `quotation.setStatus(...)` 로 직접 바꾸지 않는다 —
  전이 규칙이 엔티티 안에 갇혀 있어 우회가 불가능하다.

### 견적의 "1회 소비" — `markConverted` / `revertConversion` ⭐

도메인 편엔 없던, 화면을 돌려보다 발견해 보강한 규칙이다(PROGRESS 의 SD 갭 ①): **한 견적으로는 수주를 딱 한 번만
만든다.** 이를 위해 `ACCEPTED` 와 별도로 `CONVERTED`(수주전환됨) 상태를 두었다.

```java
public enum QuotationStatus {
    DRAFT, SENT, ACCEPTED, CONVERTED, EXPIRED, CANCELLED
}
```

```java
/** 수주로 전환 — ACCEPTED → CONVERTED. 견적당 수주 1건만 허용. */
public void markConverted() {
    if (status != QuotationStatus.ACCEPTED)
        throw new IllegalStateException(
                "수락된(ACCEPTED) 견적만 수주로 전환할 수 있습니다. 현재: " + status
                + " — 이미 수주로 전환된 견적은 재사용할 수 없습니다(분할 수주는 견적 없이 진행).");
    this.status = QuotationStatus.CONVERTED;
}

/** 전환 취소 — CONVERTED → ACCEPTED. 수주가 취소되면 견적을 되살려 재사용 가능하게. */
public void revertConversion() {
    if (status != QuotationStatus.CONVERTED)
        throw new IllegalStateException("CONVERTED 견적만 전환 취소할 수 있습니다. 현재: " + status);
    this.status = QuotationStatus.ACCEPTED;
}
```

왜 필요한가: `CONVERTED` 가 없으면 견적은 수주를 만든 뒤에도 `ACCEPTED` 로 남아, **같은 견적으로 수주를 두 번
세 번** 만들 수 있다(중복 수주 사고). 상태를 한 칸 더 두어 "이미 쓴 견적" 을 표시하고, 다음에 보듯 `markConverted`
가 두 번째 호출을 거부한다. 화면에서도 수주 작성 폼의 견적 드롭다운은 `ACCEPTED` 만 보여주므로, 소비된 견적은
자동으로 목록에서 빠진다.

> 짝을 이루는 `revertConversion` 이 중요하다 — 수주를 **취소**하면 묶였던 견적이 도로 `ACCEPTED` 로 살아나
> 재사용 가능해진다. 상태 전이를 **가역적**으로 설계해 "취소했더니 견적이 영영 죽는" 사고를 막았다.

## 2. 수주 — 라인이 진실의 원천, 헤더 상태는 재계산된다

수주는 OTC 의 중심축이자 가장 긴 일생을 산다(도메인 편 3절). 상태 enum 부터 그 길이가 드러난다:

```java
public enum SalesOrderStatus {
    DRAFT, CONFIRMED, SHIPPING, SHIPPED, INVOICING, INVOICED, CLOSED, CANCELLED
}
```

### 확정·취소 — 여기에도 같은 가드 패턴

```java
public void confirm(LocalDateTime now) {
    if (status != SalesOrderStatus.DRAFT)
        throw new IllegalStateException("DRAFT 수주만 확정 가능합니다. 현재: " + status);
    if (lines.isEmpty())
        throw new IllegalStateException("라인이 비어 있는 수주는 확정할 수 없습니다.");
    this.status = SalesOrderStatus.CONFIRMED;
    this.confirmedAt = now;                       // 확정 시각을 박는다
}

public void cancel() {
    if (status != SalesOrderStatus.DRAFT && status != SalesOrderStatus.CONFIRMED)
        throw new IllegalStateException("DRAFT 또는 CONFIRMED 수주만 취소 가능합니다. 현재: " + status);
    boolean anyShipped = lines.stream().anyMatch(l -> l.getShippedQty().signum() > 0);
    if (anyShipped)
        throw new IllegalStateException("출하 실적이 있는 수주는 취소할 수 없습니다.");
    this.status = SalesOrderStatus.CANCELLED;
}
```

`cancel()` 이 도메인 편의 **"되돌릴 수 없는 선"** 을 코드로 못박는다 — 상태가 `CONFIRMED` 까지여도, **라인 중
하나라도 출하 실적(shipped_qty > 0)이 있으면** 취소를 거부한다. 물리적으로 물건이 나간 수주는 함부로 못 지운다.

### `shipped_qty`·`invoiced_qty` 가 단일 진실, 헤더는 따라간다

도메인 편의 `SHIPPING ↔ SHIPPED`, `INVOICING ↔ INVOICED` 가 어떻게 "자동으로 따라 움직이는지" 가 핵심이다.
비밀은 **라인에 누적 수량을 들고, 헤더 상태는 그 누적에서 재계산**하는 구조다. 라인이 진실의 원천이다.

```java
// SalesOrderLine — 주문/출하/청구 수량을 각각 누적
private BigDecimal orderQty;                  // 주문량 (불변)
private BigDecimal shippedQty = BigDecimal.ZERO;   // 출하 누적
private BigDecimal invoicedQty = BigDecimal.ZERO;  // 청구 누적

public boolean isFullyShipped()  { return shippedQty.compareTo(orderQty) == 0; }
public boolean isFullyInvoiced() { return invoicedQty.compareTo(orderQty) == 0; }
```

```java
// SalesOrder — 출하/청구 사건이 들어오면 라인에 누적하고 헤더를 다시 계산
public void recordShipment(SalesOrderLine line, BigDecimal qty) {
    ensureLineBelongsToThis(line);
    ensureProgressable("출하");
    line.addShippedQty(qty);
    recomputeStatus();        // ← 라인 누적값을 보고 헤더 상태를 다시 정한다
}

public void recordInvoicing(SalesOrderLine line, BigDecimal qty) {
    ensureLineBelongsToThis(line);
    ensureProgressable("청구");
    line.addInvoicedQty(qty);
    recomputeStatus();
}
// cancelShipment / cancelInvoicing 은 subtract… 로 감소시킨 뒤 똑같이 recomputeStatus()
```

`recomputeStatus()` 가 "모든 라인이 완전 출하됐나 / 일부만인가" 를 보고 `SHIPPING`(일부)·`SHIPPED`(전부)·
`INVOICED`·`CLOSED` 를 정한다. 그래서 **출하·청구를 부분적으로(라인별·여러 번) 해도** 헤더 상태가 항상 라인의
실제 누적과 일치한다. 도메인 편의 "↔ 양방향 화살표(부분 진행)" 가 이 누적·재계산 구조에서 나온다.

> 설계 포인트: 헤더 상태를 **직접 세팅하지 않고 항상 라인에서 도출**한다. 출하 취소로 수량이 줄면 헤더가
> `SHIPPED → SHIPPING` 으로 자연히 되돌아간다 — 상태 불일치(헤더는 SHIPPED인데 라인은 일부 취소됨)가 구조적으로
> 불가능하다.

## 3. 신용한도 — `CreditLimitChecker` 는 왜 엔티티가 아니라 서비스에 있나 ⭐

도메인 편 4절의 통제 — **수주 확정 시 "미수금 + 주문액 > 한도" 면 거부** — 가 사는 곳이다. 검증 전체는 짧다:

```java
@Component
@RequiredArgsConstructor
public class CreditLimitChecker {

    private final SalesOrderRepository repository;

    public void check(Customer customer, BigDecimal orderTotal, Long excludeOrderId) {
        if (customer.getStatus() != MasterStatus.ACTIVE)
            throw new IllegalStateException("거래 불가 고객입니다 (status=" + customer.getStatus() + ").");

        BigDecimal used = repository.sumActiveOrderAmountByCustomer(customer.getId(), excludeOrderId);
        if (used == null) used = BigDecimal.ZERO;          // 활성 수주(CONFIRMED~INVOICED) 합계 = 미수금
        BigDecimal remaining = customer.getCreditLimit().subtract(used);

        if (orderTotal.compareTo(remaining) > 0)
            throw new IllegalStateException("신용한도 초과: 남은 한도 " + remaining + ", 요청 " + orderTotal);
    }
}
```

### 왜 `SalesOrder.confirm()` 안에 넣지 않았나

한도 점검을 엔티티 메서드(1·2절의 상태 가드처럼)에 둘 수도 있었다. 그러지 않은 이유 — 클래스 주석이 직접 말한다:

- **외부 조회가 필요하다.** "이 고객의 *다른* 활성 수주 합계" 는 다른 수주들을 DB 에서 긁어야 안다
  (`sumActiveOrderAmountByCustomer`). 엔티티는 자기 자신만 알아야 하고 리포지토리를 참조하면 안 된다 —
  그래서 **리포지토리가 필요한 검증은 서비스 계층**으로 뺐다.
- 그래서 역할이 갈린다: **상태 전이(자기 완결적 규칙)는 엔티티가**, **외부 데이터가 필요한 검증은 서비스(별도
  컴포넌트)가** 담당한다. 1·2절의 `confirm()` 가드와 이 `check()` 는 같은 "확정" 행위의 두 측면이다.

`excludeOrderId` 인자도 눈여겨볼 점: 이미 만들어 둔 수주를 다시 확정할 때 **자기 자신은 합계에서 빼야**
이중 계산이 안 된다. 그래서 "현재 확정하려는 수주 id" 를 넘겨 제외한다.

> 주석에 적힌 **의도적 미구현**: 두 사람이 동시에 확정하면 둘 다 "한도 통과" 를 본 뒤 합산 초과할 수 있다
> (race condition). 이건 03(MM)·동시성 학습에서 락으로 다룬다 — Phase 2 에선 일부러 비워 뒀다.

### 서비스가 엮는 흐름 — create / confirm / cancel

`SalesOrderService` 가 1·2절(엔티티 전이)·3절(한도 점검)·1절(견적 소비)을 한 흐름으로 조립한다:

```java
@Transactional
public SalesOrderResponse create(SalesOrderCreateRequest req) {
    Customer customer  = ...;
    Quotation quotation = resolveQuotation(req.quotationId());
    if (quotation != null) {
        quotation.markConverted();   // ① 견적 1회 소비 — 이미 전환됐으면 여기서 거부
    }
    String number = numberGenerator.nextSalesOrderNumber(req.orderDate());
    SalesOrder order = SalesOrder.draft(number, customer, salesperson, quotation, req.orderDate());
    addLines(order, req.lines());
    return mapper.toResponse(repository.save(order));
}

@Transactional
public SalesOrderResponse confirm(Long id) {
    SalesOrder order = getOrThrow(id);
    creditLimitChecker.check(order.getCustomer(), order.getTotalAmount(), order.getId());  // ② 한도 점검
    order.confirm(LocalDateTime.now(clock));                                                // ③ 상태 전이
    return mapper.toResponse(order);
}

@Transactional
public SalesOrderResponse cancel(Long id) {
    SalesOrder order = getOrThrow(id);
    order.cancel();                                       // ④ 출하 실적 있으면 거부
    Quotation quotation = order.getQuotation();
    if (quotation != null && quotation.getStatus() == QuotationStatus.CONVERTED)
        quotation.revertConversion();                    // ⑤ 묶였던 견적을 ACCEPTED 로 복원
    return mapper.toResponse(order);
}
```

견적 소비(`markConverted`)와 복원(`revertConversion`)이 **create/cancel 의 짝**으로 정확히 마주 보는 게 보인다.
한도 점검은 `confirm` 단 한 곳 — 도메인 편 "통제 지점 = 수주 확정" 그대로다.

### 화면에 한도를 미리 보여주기 — `creditStatus`

검증(거부)만으로는 사용자가 답답하다. 그래서 확정 *전에* 같은 산식을 **조회용**으로 노출한다(PROGRESS SD 갭 ②):

```java
public CreditStatusResponse creditStatus(Long customerId) {
    Customer customer = customerRepository.findById(customerId).orElseThrow(...);
    BigDecimal used = repository.sumActiveOrderAmountByCustomer(customerId, null);  // 검증과 같은 합계
    if (used == null) used = BigDecimal.ZERO;
    BigDecimal remaining = customer.getCreditLimit().subtract(used);
    return new CreditStatusResponse(customerId, customer.getCreditLimit(), used, remaining);
}
// → GET /api/sales-orders/credit-status?customerId=
```

`CreditLimitChecker.check` 와 **동일한 `sumActiveOrderAmountByCustomer`** 를 써서, 화면이 보여주는 "남은 한도" 와
실제 확정 시 적용되는 판정이 어긋나지 않게 했다. 수주 작성 폼은 고객·라인 합계를 반영해 "이 주문 포함 시 남은
한도" 를, 상세 화면은 확정 전 DRAFT 면 "확정 시 남은 한도" 예고와 초과 경고를 띄운다.

## 4. 출하·발행이 04·05 를 깨운다 — 이벤트 발행 지점

도메인 편 5절 "SD 는 04·05 의 출발점" 이 코드에선 **`events.publishEvent(...)` 한 줄**로 나타난다. SD 는 "보냈다 /
발행했다" 고 외치기만 하고, 누가 듣는지 모른다(느슨한 결합).

### 출하 확정 → `DeliveryShippedEvent` (MM 이 재고 차감)

```java
@Transactional
public DeliveryResponse create(DeliveryCreateRequest req) {
    SalesOrder order = salesOrderRepository.findById(req.salesOrderId()).orElseThrow(...);
    Delivery delivery = Delivery.draft(number, order, req.warehouseId(), req.shippedDate());
    for (DeliveryLineRequest lineReq : req.lines()) {
        SalesOrderLine sol = order.findLineById(lineReq.salesOrderLineId());
        delivery.addLine(sol, lineReq.quantity());
    }

    delivery.ship();                                  // 출하 DRAFT → SHIPPED
    for (DeliveryLine dline : delivery.getLines())    // 2절: SO 라인에 출하수량 누적 + 헤더 재계산
        order.recordShipment(dline.getSalesOrderLine(), dline.getQuantity());

    repository.save(delivery);

    // ⭐ Phase 4 — 출하 확정 사건 발행. MM 의 @TransactionalEventListener(BEFORE_COMMIT) 가
    //    같은 트랜잭션 안에서 GoodsIssue 자동 생성 + 재고 차감. 가용 부족이면 여기까지 전부 롤백.
    events.publishEvent(new DeliveryShippedEvent(
            delivery.getId(), delivery.getWarehouseId(), delivery.getShippedDate(),
            toEventLines(delivery)));

    return mapper.toResponse(delivery);
}
```

이벤트 본문이 **ID 만이 아니라 (창고·라인 itemId·수량)까지** 담는 게 의도적이다 — 리스너가 같은 트랜잭션 안에서
즉시 쓰므로 다시 DB 를 조회할 필요가 없다. 이벤트의 소유권은 **발행자(SD)** 에 있고, MM 이 이 타입을 import 해
구독할 뿐이라 의존 방향은 `MM → SD` 단방향이다(04 에서 자세히).

```java
public record DeliveryShippedEvent(
        Long deliveryId, Long warehouseId, LocalDate shippedDate, List<Line> lines) {
    public record Line(Long itemId, BigDecimal quantity) {}
}
```

취소도 대칭이다 — `cancel()` 후 `DeliveryCancelledEvent` 를 발행해 MM 이 재고를 복원하고, 2절의
`order.cancelShipment(...)` 로 SO 라인 출하수량을 되돌린다.

### 인보이스 발행 → `InvoiceIssuedEvent` (FI 가 매출 분개)

세금계산서 쪽은 부가세 계산이 추가된다. `Invoice` 가 라인 합계에서 공급가·세액·총액을 스스로 계산한다:

```java
public static final BigDecimal VAT_RATE = new BigDecimal("0.10");

private void recalculateTotals() {
    this.subtotal    = lines.stream().map(InvoiceLine::getLineTotal)
                            .reduce(BigDecimal.ZERO, BigDecimal::add);
    this.taxAmount   = subtotal.multiply(VAT_RATE).setScale(2, RoundingMode.HALF_UP);  // 부가세 10%
    this.totalAmount = subtotal.add(taxAmount);
}
```

```java
@Transactional
public InvoiceResponse create(InvoiceCreateRequest req) {
    // ... 라인 구성 ...
    invoice.issue();                                   // 인보이스 DRAFT → ISSUED
    for (InvoiceLine iline : invoice.getLines())       // 2절: SO 라인에 청구수량 누적 + 헤더 재계산
        order.recordInvoicing(iline.getSalesOrderLine(), iline.getQuantity());
    Invoice saved = repository.save(invoice);

    // ⭐ Phase 5 — 인보이스 발행 사건. FI 의 SalesAccountingListener 가 BEFORE_COMMIT 에서
    //    매출 자동 분개(차)매출채권 / 대)매출 + 부가세예수금) 생성. 분개 실패면 발행 자체가 롤백.
    events.publishEvent(new InvoiceIssuedEvent(
            saved.getId(), saved.getNumber(), saved.getInvoiceDate(),
            saved.getSubtotal(), saved.getTaxAmount(), saved.getTotalAmount()));

    return mapper.toResponse(saved);
}
```

`DeliveryShippedEvent` 와 **완전히 같은 패턴**이다 — 확정 메서드 호출 → SO 라인 누적 → 이벤트 발행. 차이는 듣는
모듈(MM vs FI)과 본문(재고 차감용 라인 vs 분개용 금액)뿐이다. `subtotal·taxAmount·totalAmount` 를 본문에 실어
FI 가 `차)매출채권 totalAmount / 대)매출 subtotal + 부가세예수금 taxAmount` 로 분개한다.

> 두 이벤트 모두 주석이 **"실패하면 전부 롤백"** 을 강조한다. 리스너가 `BEFORE_COMMIT`(커밋 직전·같은
> 트랜잭션)에서 돌기 때문이다 — 재고가 모자라거나 분개 차/대가 안 맞으면 출하·발행 자체가 없던 일이 된다.
> "느슨한 결합이되 원자성은 보장" — 이 메커니즘의 알맹이는 04 에서 깊게 다룬다.

## 요약

1. SD 엔티티는 모두 **상태 전이 메서드 + 입구 가드**(`if status != … throw`)를 갖는다 — 도메인 편 상태도가
   그대로 if 문이 되어 잘못된 전이를 `IllegalStateException` 으로 차단. 상태 변경은 엔티티 메서드로만.
2. **견적 1회 소비 ⭐**: `CONVERTED` 상태 + `markConverted()`(ACCEPTED→CONVERTED, 두 번째 거부)/
   `revertConversion()`(수주 취소 시 ACCEPTED 복원) 으로 같은 견적의 중복 수주를 막고, 취소는 가역적으로.
3. **수주 라인이 진실의 원천**: `shippedQty`·`invoicedQty` 를 라인에 누적하고 헤더 상태는 `recomputeStatus()`
   로 항상 재도출 — 부분 출하/청구·취소에도 헤더가 라인과 절대 어긋나지 않음.
4. **신용한도 `CreditLimitChecker` ⭐**: "다른 수주 합계" 라는 외부 조회가 필요해 엔티티가 아닌 **서비스
   컴포넌트**에 둠. `confirm` 한 곳에서만 점검(미수금+주문액>한도 거부), `creditStatus` 로 화면에 동일 산식
   예고. 동시성 race 는 03 으로 보류.
5. **이벤트 발행 = 04·05 출발점**: 출하 `ship()`→`DeliveryShippedEvent`(MM 재고차감), 발행 `issue()`(부가세
   10%)→`InvoiceIssuedEvent`(FI 매출분개). 둘 다 "확정→SO라인 누적→발행" 동일 패턴, `BEFORE_COMMIT` 리스너라
   실패 시 원천 행위까지 롤백.

> **다음**: [`03 MM(재고)`] — 위 `DeliveryShippedEvent` 를 받아 재고를 차감하는 쪽, 그리고 신용한도에서 미룬
> **동시성**(같은 재고를 동시에 빼는 race 를 락으로 막기)을 다룬다.
