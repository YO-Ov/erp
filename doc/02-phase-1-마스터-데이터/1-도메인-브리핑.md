> 7단계 사이클의 **1단계: 도메인 브리핑**.

## 1. 마스터 데이터란 무엇인가

### 정의

**마스터 데이터(Master Data)** = 회사가 가진 **기준 정보**. 한 번 등록하면 거의 변하지 않고, 트랜잭션이 발생할 때마다 그것을 *참조*한다.

비유하자면 **회사라는 게임의 "사전(dictionary)"** 이다. 게임을 시작하기 전에 등장인물(고객/직원), 아이템(상품), 장소(창고/부서) 카탈로그가 먼저 정의되어 있어야, 그다음에 "누가, 무엇을, 어디서, 얼마에 거래했다" 라는 사건(트랜잭션)을 기록할 수 있다.

### 마스터 vs 트랜잭션 — 한 표로

| 구분 | 마스터 데이터 | 트랜잭션 데이터 |
| --- | --- | --- |
| 본질 | "있는 것" (Things) | "일어난 것" (Events) |
| 예시 | 고객, 상품, 거래처, 직원, 창고, 계정과목 | 수주, 출하, 인보이스, 입고, 출금, 전표 |
| 발생 빈도 | 드물게 등록/수정 | 매일 수십~수만 건 |
| 수명 | 길다 (수년 ~ 회사 폐업까지) | 짧다 (당일 발생, 이후엔 조회만) |
| 삭제 가능성 | **거의 불가능** (참조 무결성) | 보통은 취소·반품 처리, 물리 삭제는 안 함 |
| 화면 위치 | "기준정보" 또는 "마스터" 메뉴 | "영업", "구매", "회계" 같은 업무 메뉴 |
| 학습 순서 | **먼저** (Phase 1) | 그 다음 (Phase 2부터) |

### 우리가 만드는 4개의 마스터

이번 Phase에서 만들 마스터:

| 마스터 | 누가 쓰나 | 핵심 정보 |
| --- | --- | --- |
| **고객 (Customer)** | 영업팀이 수주에서 참조 | 회사명, 사업자번호, 주소, 신용한도, 결제조건 |
| **상품 (Item)** | 영업/구매/생산/재고 — **모든 모듈이 참조** | 상품코드, 명칭, 단위, 표준원가, 표준판매가 |
| **거래처 (Vendor)** | 구매팀이 입고에서 참조 | 회사명, 사업자번호, 결제조건 |
| **부서/직원 (Department/Employee)** | 인사 + 모든 모듈의 "누가 했는가" 추적 | 부서 트리, 직원 정보, 입사일 |

상품은 가장 광범위하게 참조되는 마스터다. 영업도 사고, 구매도 발주하고, 생산도 만들고, 재고도 관리한다. 그래서 상품 마스터는 ERP의 **혈관**과 같다.

## 2. 왜 마스터를 가장 먼저 만드는가

### 종속성의 방향

```
마스터 데이터 ────참조하는 화살표──→ 트랜잭션 데이터
                  (없음)
```

화살표는 **한 방향뿐**이다. 트랜잭션은 마스터를 참조하지만, 마스터는 트랜잭션을 신경 쓰지 않는다(직접적으로는). 그래서:

- 마스터 없이 트랜잭션을 만들 수 없다 ← *수주 한 건* = "고객이라는 마스터" + "상품이라는 마스터" + "수량/금액"
- 트랜잭션 없이도 마스터는 존재할 수 있다 ← 회사 신규 설립 시점에는 모든 트랜잭션이 0이지만 고객/상품 정보는 이미 있다

### 실무에서 ERP 도입 첫 작업도 "마스터 정비"

회사가 새 ERP를 도입할 때 가장 큰 비용/시간이 드는 게 바로 마스터 데이터 이관이다.

- 기존 시스템 + 엑셀에 흩어진 거래처 5,000개를 **하나의 정제된 마스터**로 모은다
- 중복 제거 (똑같은 회사가 다른 이름으로 3건 등록되어 있던 케이스)
- 코드 체계 통일 (예전에는 회사명 한글, 이제는 `CUST-2026-0001`)
- 이걸 못 하면 ERP는 가동 첫날부터 잘못된 데이터를 쌓기 시작한다

> 학습용 미니 ERP에서도 똑같다. Phase 1에서 마스터를 단단히 깔아둬야 Phase 2의 수주가 무너지지 않는다.

## 3. 마스터 데이터의 5가지 핵심 특성

이번 Phase에서 가장 중요한 부분. 이 다섯 가지가 "마스터 = 그냥 CRUD"가 아닌 이유다.

### 3.1 비즈니스 코드 체계 (Business Code / Natural Key)

#### 무엇인가
사람이 읽을 수 있는 식별자. 예:
- 고객: `CUST-2026-0001`, `CUST-2026-0002`, ...
- 상품: `ITEM-NB-LP15-001` (노트북 15인치 1번째)
- 거래처: `VEND-2026-0001`
- 직원: `EMP-2026-0001`

DB의 PK(BIGINT AUTO_INCREMENT)와는 **별도**로 둔다.

#### 왜 분리하는가
PK는 시스템 내부용 숫자고, 비즈니스 코드는 **시스템 밖에서도 쓰이는 식별자**다.

- 전화 상담: "고객 ID 47번이요" → 안 통한다. "CUST-2026-0042 신원전자 건이요" → 통한다.
- 종이 영수증/세금계산서에 인쇄되는 건 비즈니스 코드
- 시스템을 교체해도(PK는 새 시스템에서 다시 매겨짐) 비즈니스 코드는 보존되어야 함
- 코드 패턴만 봐도 종류를 안다 (`CUST` = 고객, `ITEM` = 상품, `VEND` = 거래처)

#### 학습 포인트
**코드 자동 생성을 어떻게 구현하나?**

옵션 A — DB 시퀀스 기반:
```
SELECT MAX(고객ID 끝 4자리) + 1 → CUST-2026-0043
```
간단하지만 동시성 문제 (두 명이 동시에 등록 시 같은 번호 나올 수 있음).

옵션 B — 별도 시퀀스 테이블 with 락:
```
SELECT FOR UPDATE 로 시퀀스 테이블의 다음 번호를 점유
```
안전하지만 성능 트레이드오프. (안전하지만 성능을 포기)

옵션 C — Snowflake/ULID처럼 분산 가능한 ID:
나중에 시스템이 분산되면 필요. 지금은 과한 설계.

→ Phase 1에서는 **옵션 B**로 학습 (코드 시퀀스 테이블 + 비관적 락). "왜 동시성 문제를 해결해야 하는가"를 코드로 직접 체감.

### 3.2 Soft Delete (논리 삭제)

#### 왜 물리 삭제하면 안 되는가

상상해 보자. 작년에 "신원전자"에 노트북 100대를 팔았다. 매출 1억. 회계 자료에 분개도 끊었고, 세금도 냈다.

올해 영업 담당자가 "신원전자는 거래 끊긴 지 오래야" 라며 **물리 삭제** 했다고 치자.

- 작년 수주 화면을 열면? → 고객 컬럼이 NULL이거나, FK 위반으로 오류
- 5년 후 세무조사: "이 매출 1억의 거래처가 어디인가?" → 답할 수 없음
- 채권이 남아 있던 상태였다면 → "누구한테 돈을 받아야 할지 영원히 미스터리"

이게 ERP가 마스터를 함부로 삭제 못 하는 이유다.

#### 해결: Soft Delete

물리적으로 행을 지우는 게 아니라 **"지워졌다는 표식"** 만 남긴다.

```
customer
  id  | code           | name      | deleted_at
  47  | CUST-2026-0001 | 신원전자  | 2026-05-14 10:23:00   ← 지워진 것으로 간주
  48  | CUST-2026-0002 | 한솔전자  | NULL                   ← 살아있음
```

일반 조회 쿼리는 자동으로 `WHERE deleted_at IS NULL`을 붙여서 마치 지워진 것처럼 보이게 한다. 하지만 과거 트랜잭션은 여전히 ID 47을 가리키고 있으므로 추적 가능.

#### JPA에서의 구현

JPA는 이 패턴을 어노테이션으로 지원:
- `@SQLDelete(sql = "UPDATE customer SET deleted_at = NOW() WHERE id = ?")` — DELETE 호출 시 실제로는 UPDATE
- `@SQLRestriction("deleted_at IS NULL")` — 모든 조회에 자동으로 필터 붙임 (Hibernate 6+ 신문법, 구버전은 `@Where`)

이 두 어노테이션을 엔티티에 붙이면, 서비스 코드에서는 그냥 `repository.delete(customer)`라고 써도 알아서 Soft Delete가 된다.

### 3.3 변경 이력 추적 (Auditing)

#### 누가 언제 만들었나 / 누가 언제 바꿨나

기간계 시스템의 **법적 요구사항**이다. 회계 자료/개인정보는 "누가 언제 무엇을 했는가"를 추적할 수 있어야 한다.

모든 마스터(나중에는 모든 트랜잭션)에 공통으로 들어가는 4개 컬럼:

| 컬럼 | 의미 |
| --- | --- |
| `created_at` | 최초 등록 시각 |
| `created_by` | 최초 등록자 (직원 ID 또는 사용자명) |
| `updated_at` | 마지막 수정 시각 |
| `updated_by` | 마지막 수정자 |

#### JPA Auditing — 어노테이션으로 자동화

매번 직접 `customer.setCreatedAt(LocalDateTime.now())` 호출하면 빠뜨리기 쉽다. JPA가 자동으로 채워준다:

```java
@CreatedDate           // INSERT 시 자동 세팅
private LocalDateTime createdAt;

@LastModifiedDate      // UPDATE 시 자동 갱신
private LocalDateTime updatedAt;

@CreatedBy             // INSERT 시 현재 로그인 사용자로 자동 세팅
private String createdBy;

@LastModifiedBy        // UPDATE 시 자동 갱신
private String updatedBy;
```

이것을 가능하게 하려면:
- `@EnableJpaAuditing` 으로 활성화
- `AuditorAware<String>` 빈으로 "현재 사용자가 누구인가" 알려주는 함수 등록
- Phase 1에서는 아직 인증이 없으므로 `system`이라는 고정 값으로 시작. Phase 6에서 Spring Security가 들어오면 실제 로그인 사용자로 자동 전환.

#### "전후 값까지 모두 보존"은 다음 Phase로 미룸

위 4개는 **"마지막 상태"** 만 안다. "원래 신용한도가 1억이었는데 누가 5천으로 줄였다" 같은 변경 이력까지 보려면 Spring Data Envers 같은 도구가 필요한데, **이건 Phase 6에서 감사 로그와 함께 도입**한다.

Phase 1은 일단 4개 컬럼 기반 Auditing까지만.

### 3.4 페이징 + 다중 필터 검색

마스터는 보통 수천~수만 건. 화면에 한 번에 다 띄울 수 없다.

#### 페이징
- Spring Data JPA의 `Pageable` 인터페이스를 활용
- 응답에 "전체 건수 / 현재 페이지 / 페이지 크기 / 정렬" 메타가 함께 나감
- 프론트(또는 Swagger UI)에서 페이지 번호만 바꿔 호출

#### 다중 필터
사용자가 흔히 쓰는 조회 조건들:
- 고객명 부분 일치 (`name LIKE '%신원%'`)
- 사업자번호 정확 일치
- 등록일 범위 (`created_at BETWEEN ? AND ?`)
- 활성 여부
- 담당자

이런 동적 조건을 깔끔하게 표현하려면 **QueryDSL**이 필요. ERP 학습 계획서에서 QueryDSL을 일찍 도입하는 이유가 이것이다.

Phase 1에서는 단순한 조건만 JPA Method Query / `Specification`으로 처리하고, **QueryDSL의 본격적인 활용은 Phase 2 (수주 조회)부터**.

### 3.5 활성/비활성 상태

Soft Delete만으로는 부족한 케이스가 있다.

> "이 고객은 당분간 거래는 안 하지만, 언젠가 다시 거래할 수도 있다. 삭제까지는 좀 그래."

이런 상황을 표현하기 위해 `status` 필드를 둔다:
- `ACTIVE` — 정상 거래 가능
- `INACTIVE` — 휴면 (조회에는 잡히지만 새 거래 등록은 막음)
- `BLOCKED` — 거래 정지 (신용 불량 등)

`deleted_at IS NULL` 인 행 중에서 `status`로 다시 한 번 분류하는 셈.

> Phase 1에서 4개 마스터 모두에게 `status` 필드를 둘 것인가? 모델링 단계(`2-설계-제안.md`)에서 결정한다.

## 4. 모듈 간 영향 — 이 4개 마스터가 앞으로 어떻게 쓰일지

마스터를 만들 때 가장 중요한 마음가짐: **"내가 만든 이 정보를, 앞으로 누가, 어디서 쓰게 될 것인가"**.

| 마스터 | Phase 2 (SD) | Phase 3 (MM) | Phase 5 (FI) | Phase 6 (감사) | Phase 7 (HR) | Phase 8 (PP) |
| --- | --- | --- | --- | --- | --- | --- |
| 고객 | 수주의 "누구에게" | — | 매출채권의 주체 | "누가 등록"의 대상은 아님 | — | — |
| 상품 | 수주 라인 | 재고/입출고 | 매출 인식의 단가 기준 | — | — | BOM의 노드 |
| 거래처 | — | 입고의 "누구에게서" | 매입채무의 주체 | — | — | — |
| 직원 | 수주 담당자 | — | 전표 작성자 | **"누가"의 본체** | **본체** | 작업자 |

이걸 미리 머리에 그려두면, "지금 만드는 컬럼이 정말 필요한가" 같은 판단이 쉬워진다.

특히 **상품 마스터**가 모든 모듈에 닿는 핵심이다. 그래서 상품을 설계할 때 가장 신중해야 한다.

## 5. JPA가 마스터 데이터에 잘 맞는 이유 (Phase 0 복습 + 확장)

Phase 0에서도 짚었지만, Phase 1에서 더 진하게 체감하게 된다.

1. **객체 그래프** — 고객은 주소 여러 개를 가질 수 있고, 직원은 부서에 속한다. JPA의 `@OneToMany`, `@ManyToOne`으로 도메인 그대로 표현.
2. **영속성 컨텍스트** — 같은 트랜잭션 안에서 같은 엔티티는 한 번만 조회. 마스터 캐싱 효과.
3. **Dirty Checking** — `customer.setName("바뀐 이름")` 만 호출해도 자동으로 UPDATE SQL 생성. 마스터 수정 코드가 단순해짐.
4. **공통 베이스 엔티티** — `BaseEntity`를 만들어 `createdAt/updatedAt` 등을 한 곳에 정의하고, 모든 마스터가 상속. 코드 중복 제거.
5. **Auditing 어노테이션** — 위에서 본 4개 자동화.

## 6. Phase 1에서 미리 알아둘 함정

설계/구현 단계에서 자주 마주칠 것들을 미리 예고:

| 함정 | 설명 | Phase 1에서의 결정 |
| --- | --- | --- |
| **돈은 `BigDecimal`** | float/double은 부동소수점 오차로 회계 정합성을 깨뜨림 | 신용한도, 표준원가, 판매가 모두 `BigDecimal` |
| **VARCHAR 길이** | 한글 4글자 = UTF-8 12바이트. `VARCHAR(10)`이면 짤림 | 회사명/주소 등은 충분히 (예: 200) |
| **인덱스** | 비즈니스 코드, 자주 필터링되는 컬럼은 인덱스 필수 | `code` 컬럼 UNIQUE INDEX |
| **enum 컬럼** | DB에 ENUM 타입을 쓸지, VARCHAR + Java Enum 매핑할지 | **VARCHAR + `@Enumerated(STRING)`** (스키마 변경 자유로움) |
| **주소 / 다국어** | 실무 ERP는 복잡하지만 학습용은 단순화 | 주소 1개 필드, 한국어만 |
| **DTO 변환** | 엔티티를 그대로 API로 노출하면 안 됨 (순환 참조, 민감 정보) | MapStruct로 변환 |
| **트랜잭션 경계** | 단일 마스터 CRUD는 단순. 다음 Phase의 "수주 + 재고 차감" 같은 복합 트랜잭션 대비 | `@Transactional`을 서비스 메서드에 적절히 |
| **마스터 캐싱** | 자주 읽히고 거의 안 변함 → 캐시 이상적 | Phase 1에서는 도입하지 않음 (Redis는 후반) |
