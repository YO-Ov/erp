# 1/9. 공통 베이스 엔티티 — `BaseEntity` / `BaseEntityWithCode` / `MasterStatus`

> Phase 1 코드 워크스루의 첫 편. 모든 마스터(고객/상품/거래처/부서/직원)가 공통으로 들고 가는 횡단 관심사를 어디에 두는가의 이야기.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/common/entity/
├─ BaseEntity.java            (id + Auditing 4컬럼)
├─ BaseEntityWithCode.java    (code + status + deletedAt)
└─ MasterStatus.java          (ACTIVE / INACTIVE / BLOCKED)
```

설계서 §2 "공통 베이스 엔티티 (BaseEntity 계층)" 의 코드 버전.

---

## 🔥 왜 2단 계층인가

```
           BaseEntity                     ← 모든 엔티티 공통
                │
                ▼
        BaseEntityWithCode                ← 마스터만 공통
                │
    ┌───────────┼───────────┬────────────┐
    ▼           ▼           ▼            ▼
 Customer     Item        Vendor      Employee
                                     (Department는 자기참조라 별도)
```

이렇게 나눈 이유는 **"공통이지만 모든 엔티티의 공통은 아닌 것"** 을 분리하기 위해서다.

- `id` + `created_at/by` + `updated_at/by` 는 **모든** 엔티티가 가져야 함 → `BaseEntity`
- `code` + `status` + `deleted_at` 은 **마스터만** 가져야 함 → `BaseEntityWithCode`

만약 Phase 2에서 트랜잭션 엔티티 `SalesOrder` 가 등장한다고 생각해보자. 주문번호 `code` 는 있겠지만 ACTIVE/INACTIVE 같은 마스터 의미의 `status` 는 없다(주문은 PLACED/SHIPPED/DELIVERED... 같이 흐름이 다른 상태). Soft Delete도 마스터와 트랜잭션이 다르게 다뤄진다(트랜잭션은 보통 취소 플래그를 따로 둠).

→ **트랜잭션은 `BaseEntity` 만 상속**하고, 마스터는 `BaseEntityWithCode` 를 상속하면 깔끔하게 분리된다.

---

## 🔥 `BaseEntity` — 한 줄씩

```java
@Getter
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "id")
    private Long id;

    @CreatedDate
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @CreatedBy
    @Column(name = "created_by", nullable = false, updatable = false, length = 64)
    private String createdBy;

    @LastModifiedDate
    @Column(name = "updated_at", nullable = false)
    private LocalDateTime updatedAt;

    @LastModifiedBy
    @Column(name = "updated_by", nullable = false, length = 64)
    private String updatedBy;
}
```

### `@MappedSuperclass` vs `@Entity`

핵심은 이 어노테이션이다.

- `@Entity` → DB에 **자체 테이블**이 생긴다.
- `@MappedSuperclass` → 테이블은 안 생기지만, 자식 엔티티가 상속하면 **이 클래스의 필드가 자식 테이블의 컬럼**으로 들어간다.

즉 `BaseEntity` 자체는 DB에 없고, `Customer` 테이블에 `id/created_at/created_by/updated_at/updated_by` 컬럼이 박힌다. 자식마다 똑같은 컬럼 5개를 적지 않아도 된다는 게 핵심.

### `@EntityListeners(AuditingEntityListener.class)`

JPA가 INSERT/UPDATE 직전에 `AuditingEntityListener` 를 호출해서 `@CreatedDate` / `@CreatedBy` / `@LastModifiedDate` / `@LastModifiedBy` 가 붙은 필드를 **자동으로 채워준다**.

서비스 코드에서 `customer.setCreatedAt(...)` 같은 걸 일일이 호출하지 않아도 되는 이유가 여기 있다. 03편(Auditing 인프라)에서 활성화 설정을 짚는다.

### `updatable = false` 의 의미

`created_at` / `created_by` 에만 `updatable = false` 가 붙어 있다.

→ 한 번 INSERT 되고 나면 그 뒤 UPDATE 쿼리에 이 컬럼은 **포함되지 않는다**. 누군가 코드로 `customer.createdAt = ...` 을 강제로 바꿔도 SQL이 아예 안 날아간다. **생성 정보는 불변이라는 도메인 규칙을 코드 레벨에서 강제**하는 거다.

### Setter가 하나도 없는 이유

Lombok `@Getter` 만 붙고 `@Setter` 가 없다. JPA에 필요한 건 **필드 접근 + 리플렉션**이라 Setter가 굳이 필요 없다. 외부에서 `id`, `createdAt` 등을 함부로 못 바꾸게 막는 효과도 같이 노린다.

## 🔥 `BaseEntityWithCode` — 마스터 전용 3컬럼

```java
@Getter
@MappedSuperclass
public abstract class BaseEntityWithCode extends BaseEntity {

    @Column(name = "code", nullable = false, unique = true, length = 30, updatable = false)
    private String code;

    @Enumerated(EnumType.STRING)
    @Column(name = "status", nullable = false, length = 16)
    private MasterStatus status = MasterStatus.ACTIVE;

    @Column(name = "deleted_at")
    private LocalDateTime deletedAt;

    protected void assignCode(String code) {
        if (this.code != null) {
            throw new IllegalStateException("code 는 한 번만 할당할 수 있다. 기존: " + this.code);
        }
        this.code = code;
    }

    public void changeStatus(MasterStatus newStatus) {
        if (newStatus == null) {
            throw new IllegalArgumentException("status 는 null 일 수 없다.");
        }
        this.status = newStatus;
    }

    public boolean isDeleted() {
        return deletedAt != null;
    }
}
```

### `code` 가 `updatable = false` 인 이유

이게 단순한 보호장치 그 이상이다. ERP에서 **마스터 코드는 거래의 외래 식별자**다.

```
CUST-2026-0001 이라는 코드로 작년에 100건의 매출이 잡혔다.
누군가 이 코드를 CUST-2026-0099 로 바꾸면?
→ 회계 장부 추적이 단절된다.
```

그래서 **코드는 발급되는 순간부터 영구 불변**이라는 도메인 규칙이 있다. `updatable = false` 가 그 규칙의 코드 표현이고, 추가로 `assignCode()` 가 **딱 한 번만** 할당되도록 막는다.

```java
if (this.code != null) {
    throw new IllegalStateException("code 는 한 번만 할당할 수 있다. ...");
}
```

`protected` 라서 외부 서비스에서는 못 부르고, 자식 엔티티(`Customer.assignCode(...)` 같은 헬퍼)에서만 호출 가능하다. 02편의 `CodeGenerator` 와 짝을 이룬다.

### `status` 에 기본값을 필드 초기화로 박은 이유

```java
private MasterStatus status = MasterStatus.ACTIVE;
```

DDL에서 `DEFAULT 'ACTIVE'` 를 박는 방법도 있지만, **자바 객체로만 만들고 아직 저장 안 한 시점에도 `getStatus()` 가 `ACTIVE` 를 돌려줘야 자연스럽다**. 그래서 자바 필드 초기화로 둠. DB 디폴트는 보조 안전망일 뿐이고, 1차 진실은 자바 쪽.

`@Enumerated(EnumType.STRING)` — `ORDINAL` 이면 `ACTIVE=0, INACTIVE=1, BLOCKED=2` 가 DB에 저장되는데, 나중에 enum 순서가 바뀌면 데이터가 통째로 깨진다. **enum은 무조건 STRING** 으로 저장하는 게 ERP의 안전 규칙.

### `deletedAt` 이 null 인지로 살아있음을 판단

```java
public boolean isDeleted() {
    return deletedAt != null;
}
```

설계서 §5 의 Soft Delete 정책. `deletedAt` 컬럼이 채워지면 삭제된 거고, NULL이면 살아있는 거다.

실제 SQL 수준에서 "삭제된 행을 조회 결과에서 빼주는" 마법은 자식 엔티티의 `@SQLDelete` / `@SQLRestriction` 어노테이션이 담당한다. 그건 04편(Customer)에서 짚는다. 여기 베이스에는 컬럼 정의만 둠.

> 왜 베이스에 `@SQLDelete` 를 못 두는가? 어노테이션 인자에 **테이블명**이 들어가야 해서다 (`@SQLDelete(sql = "UPDATE customer SET deleted_at = NOW() WHERE id = ?")`). 베이스는 어느 테이블인지 모르니까 자식에서 선언할 수밖에 없다.

### 비즈니스 메서드 — `changeStatus(...)`

```java
public void changeStatus(MasterStatus newStatus) { ... }
```

Setter를 안 만들고 의미 있는 메서드명을 쓴다. 의도가 코드에 드러나고, null 체크 같은 invariant를 한 곳에 모을 수 있다. "객체가 자기 상태를 책임진다"는 DDD 의 작은 실천.

## 🔥 `MasterStatus` — 왜 3개인가

```java
public enum MasterStatus {
    ACTIVE,     // 정상 거래 가능
    INACTIVE,   // 휴면 — 조회는 되지만 새 거래 등록은 막는다
    BLOCKED     // 거래 정지 (신용 불량, 계약 종료 등)
}
```

도메인 브리핑에서 짚었던 마스터 데이터의 **5번째 특성**(활성 상태) 의 구현체.

| 상태 | 의미 | Soft Delete와의 관계 |
|---|---|---|
| ACTIVE | 모든 행위 가능 | `deleted_at IS NULL` |
| INACTIVE | 조회는 OK, 새 거래 ❌ | `deleted_at IS NULL` |
| BLOCKED | 거래 정지 (신용/계약 문제) | `deleted_at IS NULL` |
| (삭제됨) | 조회 자체 ❌ | `deleted_at IS NOT NULL` |

**`status` 와 `deleted_at` 은 별개의 축**이라는 점이 중요하다.

- 휴면 고객은 "지워진" 게 아니라 "거래 안 하는 중" 이다. 작년 매출 조회할 때는 여전히 보여야 한다.
- BLOCKED는 신용 불량 같은 정책적 차단. 정보는 그대로 두고 신규 거래만 막음.
- Soft Delete (`deleted_at`) 는 "마스터 자체가 잘못 등록됨/회사에서 떠남" 같이 더 강한 의미.

→ enum 하나에 `DELETED` 를 추가하지 않은 이유가 이거다. **삭제는 상태가 아니라 별도 축**이라서 컬럼을 따로 둠.

## 🔥 상속 → DB 컬럼 매핑 한눈에

`Customer` 클래스가 `BaseEntityWithCode` 를 상속하면 `customer` 테이블은 다음 컬럼을 갖게 된다:

```
customer
├─ (BaseEntity 에서)
│   ├─ id              BIGINT PK
│   ├─ created_at      DATETIME
│   ├─ created_by      VARCHAR(64)
│   ├─ updated_at      DATETIME
│   └─ updated_by      VARCHAR(64)
├─ (BaseEntityWithCode 에서)
│   ├─ code            VARCHAR(30) UNIQUE
│   ├─ status          VARCHAR(16)
│   └─ deleted_at      DATETIME NULL
└─ (Customer 자체)
    ├─ name
    ├─ business_no
    ├─ address
    ├─ credit_limit
    └─ payment_terms
```

상속이 곧 컬럼이다. 자식에 따로 정의 안 해도 `@MappedSuperclass` 덕에 자동으로 들어온다. 마이그레이션 V3 (Customer 테이블 생성, 08편)을 보면 이 컬럼들이 그대로 DDL에 나타나 있는 걸 확인할 수 있다.
