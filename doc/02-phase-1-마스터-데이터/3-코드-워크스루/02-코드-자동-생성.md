# 2/9. 코드 자동 생성 — `CodeSequence` / `CodeSequenceRepository` / `CodeGenerator`

> 마스터 코드(`CUST-2026-0001`)를 동시성 안전하게 발급하는 핵심 인프라.
> 도메인 브리핑 §3에서 짚었던 "옵션 B: 시퀀스 테이블 + 비관적 락" 의 실제 구현.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/common/code/
├─ CodeSequence.java            (Entity — 시퀀스 카운터 테이블)
├─ CodeSequenceRepository.java  (FOR UPDATE 락 쿼리)
└─ CodeGenerator.java           (Service — 발급 로직)
```

---

## 🔥 왜 이 인프라가 필요한가 — 30초 복습

```
요구사항: 신규 고객을 등록하면 `CUST-2026-0001`, `CUST-2026-0002`, ... 순차 발급.

순진한 구현:
  int next = SELECT MAX(...) FROM customer WHERE code LIKE 'CUST-2026-%' + 1;
  INSERT INTO customer ... code = 'CUST-2026-' + next

→ 두 트랜잭션이 동시에 0001 을 발급해서 UNIQUE 위반 발생.
  운영 환경에서 매우 자주 발생하는 동시성 버그.
```

**해결책**: 카운터를 별도 테이블 `code_sequence` 로 두고, `SELECT ... FOR UPDATE` 로 그 행을 점유한 뒤에만 증가시킨다. 이렇게 하면 같은 (prefix, year) 를 동시에 발급하려는 다른 트랜잭션은 **DB 수준에서 줄을 서서 기다린다**.

---

## 🔥 `CodeSequence` — 카운터 엔티티

```java
@Entity
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@EntityListeners(AuditingEntityListener.class)
@Table(
        name = "code_sequence",
        uniqueConstraints = @UniqueConstraint(
            name = "uk_code_sequence_prefix_year",
            columnNames = {"prefix", "year"})
)
public class CodeSequence {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "prefix", nullable = false, length = 16)
    private String prefix;

    @Column(name = "year", nullable = false)
    private int year;

    @Column(name = "next_number", nullable = false)
    private int nextNumber;

    @LastModifiedDate
    @Column(name = "updated_at", nullable = false)
    private LocalDateTime updatedAt;
    ...
}
```

### `BaseEntity` 를 상속하지 않는 이유

01편의 `BaseEntity` 에는 `createdBy/createdAt/updatedBy/updatedAt` 4개가 있다. 그런데 `CodeSequence` 는 일부러 안 받고 `updated_at` 만 가져왔다.

→ **이 테이블은 마스터도 트랜잭션도 아닌 "운영용 카운터"** 다. 누가 언제 만들었는지(`createdBy`) 같은 정보가 의미 없다. 매번 자동으로 1씩 증가하는 캐시 같은 존재. "공통 베이스를 상속하는 게 항상 옳은 건 아니다" 라는 작은 교훈.

`updated_at` 만 남긴 건 운영 중 디버깅용 — "마지막 발급이 언제였나" 정도는 알고 싶음.

### `uniqueConstraints = (prefix, year)` 이게 안전망의 핵심

```java
@UniqueConstraint(name = "uk_code_sequence_prefix_year",
                  columnNames = {"prefix", "year"})
```

(`CUST`, 2026) 행은 DB 안에 **단 하나만** 존재해야 한다. 두 트랜잭션이 동시에 신규로 만들려고 INSERT를 쏴도 DB가 한 쪽에 `DataIntegrityViolationException` 을 던져준다. 이게 아래 `CodeGenerator.createOrLockExisting()` 의 catch 분기와 짝을 이룬다.

### `@NoArgsConstructor(access = PROTECTED)`

JPA는 리플렉션으로 객체를 만들 때 기본 생성자가 필요한데, **외부에서 new 로 만드는 건 금지**하고 싶다. PROTECTED 로 막아두고, 대신:

```java
public static CodeSequence initial(String prefix, int year) {
    CodeSequence seq = new CodeSequence();
    seq.prefix = prefix;
    seq.year = year;
    seq.nextNumber = 1;
    return seq;
}
```

**정적 팩토리 메서드** `initial(...)` 로만 만들 수 있게 한다. "처음 만들 때는 1부터 시작한다" 는 도메인 규칙이 메서드 이름과 본문에 명시되어 있다.

### `issueAndIncrement()` — 가장 중요한 한 메서드

```java
int issueAndIncrement() {
    int issued = this.nextNumber;
    this.nextNumber = issued + 1;
    return issued;
}
```

3줄짜리 메서드인데 **얘가 잘못되면 코드 충돌이 난다**. 핵심 약속:

1. 현재 `nextNumber` 를 발급한다 (예: 5).
2. `nextNumber` 를 6으로 올린다.
3. 발급된 5를 반환한다.

가시성도 주의: `int issueAndIncrement()` — **package-private**. 외부 서비스가 직접 부르면 락 없이 호출될 위험이 있으니 같은 패키지의 `CodeGenerator` 만 부를 수 있게 막아 둔다.

> 💡 javadoc에 명시돼 있다: "호출 측에서는 반드시 비관적 락으로 행을 점유한 상태여야 한다." → 도메인 규칙을 코드와 주석 양쪽에서 강제.

## 🔥 `CodeSequenceRepository` — `FOR UPDATE` 한 줄

```java
public interface CodeSequenceRepository extends JpaRepository<CodeSequence, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("select s from CodeSequence s where s.prefix = :prefix and s.year = :year")
    Optional<CodeSequence> findForUpdate(@Param("prefix") String prefix, @Param("year") int year);
}
```

### `@Lock(LockModeType.PESSIMISTIC_WRITE)` 가 곧 `SELECT ... FOR UPDATE`

JPA가 이 어노테이션을 보면 발사되는 실제 SQL은:

```sql
SELECT s.* FROM code_sequence s
 WHERE s.prefix = 'CUST' AND s.year = 2026
   FOR UPDATE
```

`FOR UPDATE` 가 붙으면 **이 행에 쓰기 락이 걸리고, 다른 트랜잭션은 같은 행을 `FOR UPDATE` 로 잡으려 하면 대기**한다 (또는 lock timeout 시 실패).

→ 동시성 안전성의 핵심은 이 한 줄이다. 메서드명 `findForUpdate` 도 의도를 명확히 했다.

### 왜 `Optional` 인가

(prefix, year) 행이 **아직 없을 수도 있다** — 그 해의 첫 발급 요청일 때. 그래서 `Optional` 로 받아 "있으면 점유, 없으면 새로 만들기" 두 분기를 `CodeGenerator` 에서 처리한다.

## 🔥 `CodeGenerator` — 발급 로직 한 줄씩

```java
@Service
@RequiredArgsConstructor
public class CodeGenerator {

    private static final String CODE_FORMAT = "%s-%d-%04d";

    private final CodeSequenceRepository repository;
    private final Clock clock;

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public String nextCode(String prefix) {
        if (prefix == null || prefix.isBlank()) {
            throw new IllegalArgumentException("prefix 는 비어 있을 수 없다.");
        }
        int year = LocalDate.now(clock).getYear();

        CodeSequence sequence = repository.findForUpdate(prefix, year)
                .orElseGet(() -> createOrLockExisting(prefix, year));

        int issued = sequence.issueAndIncrement();
        return String.format(CODE_FORMAT, prefix, year, issued);
    }

    private CodeSequence createOrLockExisting(String prefix, int year) {
        try {
            return repository.saveAndFlush(CodeSequence.initial(prefix, year));
        } catch (DataIntegrityViolationException duplicate) {
            return repository.findForUpdate(prefix, year)
                    .orElseThrow(() -> new IllegalStateException(
                            "code_sequence 초기화 경합 처리 실패: prefix=" + prefix + ", year=" + year,
                            duplicate));
        }
    }
}
```

### `@Transactional(propagation = REQUIRES_NEW)` — 이게 미묘하지만 중요

기본 `@Transactional` 만 붙이면 **호출 측 트랜잭션 안에서 같이 동작**한다(REQUIRED). 그러면 이런 문제가 생긴다:

```
[호출 측 트랜잭션 시작]
  코드 발급 → CUST-2026-0042 받음
  Customer 저장 시도 → 사업자번호 중복으로 실패
[전체 트랜잭션 롤백]
  → 0042 도 같이 롤백됨
  → 다음 사람이 다시 발급하면 또 0042 (정상)
```

이건 사실 자연스러워 보이지만, 큰 문제가 있다: **롤백이 자주 일어나면 같은 번호 발급 요청이 반복**되면서 락 경합이 심해진다. 그리고 다른 트랜잭션이 동시에 코드를 발급받았다면 락이 풀리는 동안 그쪽이 먼저 0042 를 가져갈 수도 있다.

`REQUIRES_NEW` 는 **새 트랜잭션을 열어서 발급하고 곧바로 커밋**한다:

```
[호출 측 트랜잭션 시작]
  [코드 발급 — 새 트랜잭션 시작]
    sequence.issueAndIncrement()
    next_number = 43 으로 UPDATE
  [코드 발급 — 커밋]  ← 여기서 락 풀림 (락 점유 시간 ↓)
  Customer 저장 시도 → 실패
[호출 측 트랜잭션 롤백]
  → 0042 는 이미 커밋됐으니 안 돌아옴
  → "번호 구멍" 발생: 다음 사람은 0043 받음
```

**트레이드오프**: 동시성 안전성을 위해 번호 구멍을 수용한다. ERP 도메인에서 번호 구멍은 흔하다(취소된 주문번호, 삭제된 영수증 등). 동시성 안전성을 못 보장하면 **데이터가 깨진다** — 이쪽이 훨씬 큰 문제이므로 트레이드오프는 명확하다.

→ 회계상 "번호 빠진 게 발견되면 사유서를 남긴다" 정도가 표준 운영 관행이다.

### `Clock` 주입 — 테스트를 위한 작은 투자

```java
int year = LocalDate.now(clock).getYear();
```

`LocalDate.now()` 를 직접 호출하지 않고 `Clock` 을 주입받아 `LocalDate.now(clock)` 으로 쓴다. 왜?

- 운영: `ClockConfig` 가 `Clock.systemDefaultZone()` 빈을 등록 → 일반 시간 사용 (03편)
- 테스트: 고정 Clock(`Clock.fixed(...)`) 으로 교체 → "2026년에 발급된 코드가 정말 `CUST-2026-` 으로 시작하는가?" 같은 검증을 결정적(deterministic)으로 할 수 있음

JDK가 시계마저 주입 가능한 인터페이스로 만들어둔 덕분에, 외부 라이브러리 없이 시간 의존성을 깔끔하게 끊을 수 있다.

### `findForUpdate.orElseGet(...)` — 두 단계 동시성

```java
CodeSequence sequence = repository.findForUpdate(prefix, year)
        .orElseGet(() -> createOrLockExisting(prefix, year));
```

**1단계**: `FOR UPDATE` 로 행이 있는지 본다. 있으면 락을 잡은 채로 받아온다. **여기서 락이 잡히기 때문에**, 다른 트랜잭션이 같은 행을 `FOR UPDATE` 로 잡으려 하면 대기한다.

**2단계**: 행이 없다 → `createOrLockExisting(...)` 로 INSERT 시도.

```java
private CodeSequence createOrLockExisting(String prefix, int year) {
    try {
        return repository.saveAndFlush(CodeSequence.initial(prefix, year));
    } catch (DataIntegrityViolationException duplicate) {
        return repository.findForUpdate(prefix, year)
                .orElseThrow(...);
    }
}
```

이 catch 가 미묘한 경합을 막아준다. 동시에 두 트랜잭션 T1, T2 가 처음 발급을 시도한다고 하자:

```
T1: findForUpdate → 없음 → INSERT 시도 → 성공 → next_number=1 반환
T2: findForUpdate → 없음 (T1 INSERT 가 아직 커밋 전이라 안 보임)
    → INSERT 시도 → UNIQUE 위반 (uk_code_sequence_prefix_year)
    → catch 진입 → findForUpdate 재시도
       → 이제는 T1 이 만든 행이 보이고, FOR UPDATE 로 락 대기
       → T1 커밋 후 락 획득 → next_number=2 반환
```

UNIQUE 제약이 안전망이고, catch 가 그 안전망에 잡힌 케이스를 처리한다. 둘이 짝이 되어야 동시성이 진짜 안전해진다.

### 코드 포맷 — `%s-%d-%04d`

```java
private static final String CODE_FORMAT = "%s-%d-%04d";
...
return String.format(CODE_FORMAT, prefix, year, issued);
```

- `%s` → "CUST"
- `%d` → 2026
- `%04d` → 0042 (네 자리, 빈 자리는 0 패딩)

`%04d` 가 1만 건까지만 커버한다는 점은 의도된 제약. 학습 프로젝트에 1만 건이면 충분하고, 실무에서는 5자리/6자리로 늘리거나 해를 바꾸는 시점에 시퀀스를 리셋한다.

## 🔥 시나리오 — Customer 등록 시 전체 흐름

```
[웹 요청: POST /api/v1/customers]
   │
   ▼
[CustomerController.create(...)]
   │
   ▼
[CustomerService.create(req)]
   │ — 호출 측 트랜잭션 T0 시작
   │
   ├─ codeGenerator.nextCode("CUST")
   │     │ — REQUIRES_NEW → 새 트랜잭션 T1 시작
   │     │
   │     ├─ findForUpdate("CUST", 2026)
   │     │    → SELECT ... WHERE prefix='CUST' AND year=2026 FOR UPDATE
   │     │    → next_number=43 인 행을 락
   │     │
   │     ├─ sequence.issueAndIncrement()
   │     │    → 43 발급, next_number=44 로 변경
   │     │
   │     └─ T1 커밋 (락 해제)
   │
   ├─ customer = Customer.of("CUST-2026-0043", ...);
   ├─ customerRepository.save(customer);
   │
   └─ T0 커밋
```

다른 트랜잭션이 동시에 코드를 발급받으려 하면 위 `T1 커밋` 이전까지 대기한다 → 0044 를 받는다.
