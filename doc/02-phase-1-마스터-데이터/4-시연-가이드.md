# Phase 1 시연 가이드

> 7단계 사이클의 마지막 — **단계 7: 시연 + 회고**.
> 도메인 브리핑 → 모델링 → 코드 워크스루로 "왜 이렇게 짰는가"는 머리로 이해했다.
> 이번에는 직접 Swagger UI / cURL / 통합 테스트로 **마스터 데이터의 5대 특성**이 실제로 동작하는지 손으로 확인한다.

---

## 시연의 목표 (Phase 1이 무엇을 보장하는가)

Phase 0에서는 "환경이 동작한다"만 검증했다. Phase 1에서는 **5가지 마스터 데이터의 핵심 특성**이 실제로 동작하는지 확인한다.

1. **자동 코드 생성** — `CUST-2026-0001`, `ITEM-2026-0001`, ... 형식의 코드가 자동으로 발급된다 (동시 요청에도 중복 없음)
2. **Soft Delete** — `DELETE` 호출해도 행이 사라지지 않고 `deleted_at` 만 채워지며 일반 조회에서 제외된다
3. **JPA Auditing** — 모든 변경에 `created_at/by`, `updated_at/by`가 자동 기록된다
4. **페이징 + 동적 검색** — `?name=...&status=ACTIVE&page=0&size=10` 식 다중 필터 조회
5. **참조 무결성** — Employee → Department FK, 사업자번호 unique, 코드 unique

거기에 ERP 특유의 시드 데이터 흐름(현우전자 + 5개 하위 부서 + 시연용 고객/상품/거래처/직원)이 Flyway V8로 자동 들어오는지도 확인한다.

이번 시연에서 확인할 8가지:
1. Flyway V2~V8 마이그레이션이 모두 적용된다
2. 시드 데이터(고객 2건, 상품 2건, 거래처 1건, 직원 3건, 부서 6건)가 보인다
3. Swagger UI에 5개 그룹(`customer-controller`, `item-controller`, `vendor-controller`, `department-controller`, `employee-controller`)이 표시된다
4. 신규 고객 등록 → `CUST-2026-NNNN` 코드 자동 발급 + `code_sequence` 행이 증가
5. 페이징/검색 (`?name=신원&status=ACTIVE`)이 동작
6. Soft Delete 후 일반 조회에서 사라짐 (DB 행은 그대로 남아 있음)
7. 사업자번호 중복 등록 → `409 Conflict` + ProblemDetail 응답
8. 통합 테스트 전부(18개) + 동시성 테스트(50스레드) 통과

## STEP 1. DB 초기화 후 새로 띄우기 (권장)

Phase 0 시연으로 DB에 `schema_init_marker`만 들어 있는 상태라면 그냥 진행해도 V2~V8가 차례로 적용된다. 하지만 **시드 데이터가 깨끗하게 들어가는 것을 보고 싶다면** 한 번 초기화하는 게 직관적이다.

```bash
# 앱이 떠 있다면 Ctrl+C로 끈 뒤
docker compose down -v        # ⚠️ 볼륨까지 삭제 — Phase 1은 아직 운영 데이터 없음, OK
docker compose up -d
docker compose ps             # (healthy) 까지 대기
```

확인 포인트:
- Phase 1 이후 (Phase 2~) 부터는 `-v` 옵션을 **절대 함부로 쓰지 않는다**. 마스터/트랜잭션 데이터가 다 날아간다.
- 지금은 시드밖에 없으므로 자유롭게 초기화 가능.

## STEP 3. 시드 데이터가 들어왔는지 DB로 확인

새 터미널에서:

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT 'department' AS table_name, COUNT(*) AS rows FROM department
  UNION ALL SELECT 'customer',  COUNT(*) FROM customer
  UNION ALL SELECT 'item',      COUNT(*) FROM item
  UNION ALL SELECT 'vendor',    COUNT(*) FROM vendor
  UNION ALL SELECT 'employee',  COUNT(*) FROM employee
  UNION ALL SELECT 'code_sequence', COUNT(*) FROM code_sequence;"
```

기대 출력:
```
+---------------+------+
| table_name    | rows |
+---------------+------+
| department    |    6 |     ← 현우전자(루트) + 영업/구매/생산/재무/인사 5개
| customer      |    2 |     ← 신원전자 / 한솔테크
| item          |    2 |     ← 노트북 / 모니터
| vendor        |    1 |     ← 동부부품
| employee      |    3 |     ← 김영업 / 이재무 / 박생산
| code_sequence |    4 |     ← CUST / ITEM / VEND / EMP 각 1행
+---------------+------+
```

부서 트리 확인:
```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT d.code, d.name, IFNULL(p.code, '(루트)') AS parent
    FROM department d LEFT JOIN department p ON d.parent_id = p.id
   ORDER BY p.code IS NULL DESC, d.code;"
```

기대:
```
+-----------------+--------+---------+
| code            | name   | parent  |
+-----------------+--------+---------+
| DEPT-HQ         | 현우전자 | (루트)   |
| DEPT-FINANCE    | 재무팀  | DEPT-HQ |
| DEPT-HR         | 인사팀  | DEPT-HQ |
| DEPT-PRODUCTION | 생산팀  | DEPT-HQ |
| DEPT-PURCHASE   | 구매팀  | DEPT-HQ |
| DEPT-SALES      | 영업팀  | DEPT-HQ |
+-----------------+--------+---------+
```

확인 포인트:
- **자기참조 FK** — `department.parent_id` → `department.id` 가 실제로 채워졌는지.
- **`code_sequence` 초기값** — V8에서 시드된 고객/상품/거래처/직원 다음 번호부터 발급되도록 `next_number` 가 3/3/2/4로 설정됨. 이게 다음 STEP에서 검증된다.

## STEP 5. 자동 코드 생성 — 신규 고객 등록

이 시연이 Phase 1의 하이라이트다. **"코드를 클라이언트가 부여하지 않고, 서버가 시퀀스를 잠가서 발급한다"**는 흐름을 직접 본다.

### 5-1. 등록 전 시퀀스 상태 확인

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT prefix, year, next_number FROM code_sequence WHERE prefix='CUST';"
```

기대:
```
+--------+------+-------------+
| prefix | year | next_number |
+--------+------+-------------+
| CUST   | 2026 |           3 |
+--------+------+-------------+
```

(V8 시드가 0001, 0002 두 건을 미리 넣었으므로 다음 번호는 3.)

### 5-2. Swagger UI에서 신규 등록

`POST /api/customers` 펼치기 → **Try it out** → 다음 본문 입력:

```json
{
  "name": "테스트고객사",
  "businessNo": "999-88-77777",
  "address": "서울시 종로구",
  "creditLimit": 5000000.00,
  "paymentTerms": "NET30"
}
```

**Execute** → 응답:

```json
{
  "id": 3,
  "code": "CUST-2026-0003",
  "name": "테스트고객사",
  "businessNo": "999-88-77777",
  "address": "서울시 종로구",
  "creditLimit": 5000000.00,
  "paymentTerms": "NET30",
  "status": "ACTIVE",
  "createdAt": "2026-...",
  "createdBy": "system",
  "updatedAt": "2026-...",
  "updatedBy": "system"
}
```

확인 포인트:
- HTTP **201 Created** + `Location: /api/customers/3` 헤더.
- **`code: "CUST-2026-0003"`** — 클라이언트가 코드를 보내지 않았는데 서버가 자동 발급.
- **`status: "ACTIVE"`** — 기본 상태값이 자동 부여.
- **`createdAt` / `createdBy: "system"`** — JPA Auditing이 채워준 값. 등록자가 `system`인 이유는 Phase 6 인증 도입 전까지는 사용자가 없기 때문 (`AuditorAware`가 임시로 "system" 반환).

### 5-3. 등록 후 시퀀스 상태 재확인

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT prefix, year, next_number FROM code_sequence WHERE prefix='CUST';"
```

기대: `next_number` 가 **3 → 4** 로 증가.

확인 포인트:
- **시퀀스 행이 같은 트랜잭션 안에서 비관적 락(`PESSIMISTIC_WRITE`)으로 잠겨 +1 된다**. 이 락 덕분에 50개 스레드가 동시에 호출해도 코드가 중복되지 않는다 (STEP 9 통합 테스트에서 검증).

## STEP 6. 페이징 + 동적 검색

ERP 화면의 핵심 — "이름에 '신원'이 들어가는 ACTIVE 고객을 페이지당 10건씩".

```bash
curl -s "http://localhost:8080/api/customers?name=신원&status=ACTIVE&page=0&size=10" | jq
```

기대 응답 (요약):
```json
{
  "content": [
    {
      "id": 1,
      "code": "CUST-2026-0001",
      "name": "신원전자",
      ...
    }
  ],
  "pageable": { "pageNumber": 0, "pageSize": 10, ... },
  "totalElements": 1,
  "totalPages": 1,
  "first": true,
  "last": true
}
```

다른 검색 조합도 시도해 보기:

```bash
# 사업자번호 정확 일치
curl -s "http://localhost:8080/api/customers?businessNo=111-22-33333" | jq '.content[].name'

# 상품 카테고리 필터 (enum)
curl -s "http://localhost:8080/api/items?category=NOTEBOOK" | jq '.content[].name'

# 정렬
curl -s "http://localhost:8080/api/customers?sort=createdAt,desc" | jq '.content[].code'
```

확인 포인트:
- **빈 파라미터는 자동으로 무시**된다 (`CustomerSpecifications.nameContains`가 null/blank 체크). `?name=` 만 보내도 에러 없이 전체 조회됨.
- 검색은 `Specification` 으로 조합되어, 여러 조건을 `AND` 로 연결한다. Phase 10에서 QueryDSL로 발전 예정.

## STEP 7. Soft Delete — "삭제했는데 DB에는 남아 있다"

이게 마스터 데이터의 핵심 규칙. 트랜잭션 데이터가 참조 중인 고객을 함부로 지워서는 안 되기 때문에, **물리 삭제가 아니라 `deleted_at` 컬럼만 채운다**.

### 7-1. 시드된 한솔테크(`CUST-2026-0002`)를 삭제

```bash
curl -i -X DELETE "http://localhost:8080/api/customers/2"
```

기대: `HTTP/1.1 204 No Content`.

### 7-2. API에서는 사라진다

```bash
curl -i "http://localhost:8080/api/customers/2"
```

기대:
```
HTTP/1.1 404 Not Found
Content-Type: application/problem+json

{
  "type": "about:blank",
  "title": "Resource Not Found",
  "status": 404,
  "detail": "Customer not found: id=2"
}
```

목록 조회에서도 사라짐:
```bash
curl -s "http://localhost:8080/api/customers?size=100" | jq '.content[].code'
# CUST-2026-0001, CUST-2026-0003 만 보이고 CUST-2026-0002 는 빠짐
```

### 7-3. 하지만 DB에는 그대로 있다

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT id, code, name, deleted_at FROM customer WHERE id=2;"
```

기대:
```
+----+----------------+----------+---------------------+
| id | code           | name     | deleted_at          |
+----+----------------+----------+---------------------+
|  2 | CUST-2026-0002 | 한솔테크 | 2026-05-24 ...     |
+----+----------------+----------+---------------------+
```

확인 포인트:
- **`deleted_at` 이 NULL이 아니면 일반 조회에서 자동 제외**된다. Repository 메서드들이 모두 `findActiveById` 같은 식으로 `deleted_at IS NULL` 조건을 묶어서 검색한다.
- 이게 왜 중요한가: Phase 2부터 수주(Sales Order)가 고객 ID를 참조한다. 고객을 진짜 지웠다면 과거 수주의 고객 정보가 사라져 회계가 무너진다.

### 7-4. (선택) 복구 — DB에서 직접

운영에서는 별도 "복구 API"가 보통은 없다. 학습 환경에서는 직접 SQL로 복구 가능:

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  UPDATE customer SET deleted_at=NULL WHERE id=2;"

curl -s "http://localhost:8080/api/customers/2" | jq '.name'    # → "한솔테크"
```

## STEP 8. 업무 규칙 위반 — 사업자번호 중복

ERP는 **무결성이 본질**이다. 같은 사업자번호로 두 번 등록되면 회계가 깨진다. 그래서 Service 단에서 사전 검증 + DB 단에서 unique 제약 양쪽으로 막는다.

### 8-1. 신원전자와 같은 사업자번호로 등록 시도

```bash
curl -i -X POST "http://localhost:8080/api/customers" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "복제회사",
    "businessNo": "111-22-33333",
    "creditLimit": 0,
    "paymentTerms": "COD"
  }'
```

기대:
```
HTTP/1.1 409 Conflict
Content-Type: application/problem+json

{
  "type": "about:blank",
  "title": "Conflict",
  "status": 409,
  "detail": "이미 등록된 사업자번호: 111-22-33333"
}
```

확인 포인트:
- **`IllegalStateException` → 409 Conflict** 매핑은 `GlobalExceptionHandler`에서 한다. RFC 9457 `ProblemDetail` 형식이라 클라이언트가 일관되게 처리할 수 있다.
- 만약 Service 검증을 뚫고 들어왔어도 DB의 `uk_customer_business_no` 제약이 두 번째 방어선 역할 (이 경우 `DataIntegrityViolationException` → 같은 409).

### 8-2. 검증 실패 — 빈 이름

```bash
curl -i -X POST "http://localhost:8080/api/customers" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "",
    "businessNo": "555-66-77777",
    "creditLimit": 0,
    "paymentTerms": "COD"
  }'
```

기대:
```
HTTP/1.1 400 Bad Request

{
  "title": "Validation Failed",
  "status": 400,
  "detail": "입력값 검증에 실패했습니다.",
  "fieldErrors": {
    "name": "must not be blank"
  }
}
```

확인 포인트:
- **`@NotBlank`** 가 ProblemDetail의 `fieldErrors` 에 그대로 매핑된다. 클라이언트(나중에 Thymeleaf 화면)가 어느 필드가 틀렸는지 정확히 알 수 있다.

## STEP 9. 통합 테스트 + 동시성 검증

Spring 앱은 띄워 둔 채로, 별도 터미널에서:

```bash
cd /Users/hwlee/workspace/erp/hwlee-erp
./gradlew test
```

기대 출력 (요약):
```
HealthIntegrationTest          PASSED
CustomerCrudIntegrationTest    > 5개 테스트 PASSED
ItemCrudIntegrationTest        > 3개 테스트 PASSED
VendorCrudIntegrationTest      > 2개 테스트 PASSED
DepartmentCrudIntegrationTest  > 3개 테스트 PASSED
EmployeeCrudIntegrationTest    > 3개 테스트 PASSED
CodeGeneratorConcurrencyTest   > 코드 생성은 동시 요청에서도 중복되지 않는다 PASSED

BUILD SUCCESSFUL in Xs
```

확인 포인트 — 각 테스트가 검증하는 **업무 규칙**:

| 테스트 | 보장하는 업무 규칙 |
| --- | --- |
| `신규_고객을_생성하면_코드가_자동_발급된다` | 클라이언트가 코드를 보내지 않아도 `CUST-YYYY-NNNN` 형식으로 발급 |
| `사업자번호가_중복되면_생성이_거부된다` | 같은 사업자번호로 두 번 등록 불가 |
| `삭제된_고객은_일반_조회에서_보이지_않는다` | Soft Delete의 행위 명세 |
| `고객을_수정하면_업데이트된_값이_반영된다` | 단, 사업자번호는 수정 불가 (DTO에 필드 없음) |
| `코드_생성은_동시_요청에서도_중복되지_않는다` | **50개 스레드가 동시 호출해도 코드가 unique** — 비관적 락의 효과 |

마지막 동시성 테스트가 핵심:
- 50개 스레드가 동시에 `CodeGenerator.nextCode("TEST")` 를 호출.
- `PESSIMISTIC_WRITE` 락이 없다면 같은 `next_number` 가 여러 스레드에 동시에 발급되어 `Set` 의 size가 50 미만이 되어 실패한다.
- 락이 동작하면 정확히 50개가 모두 unique → 테스트 통과.

## 트러블슈팅

### 1. 시드 데이터가 안 보인다 (`SELECT COUNT(*) FROM customer` → 0)
- Flyway `V8__seed_master_data.sql` 이 적용되지 않은 것. 부팅 로그를 다시 확인하고 "Successfully applied 8 migrations" 가 나왔는지 본다.
- 만약 이전에 Phase 0만 적용된 채로 V8를 추가했다면, 다시 부팅하면 V2~V8가 자동 추가 적용된다.

### 2. `Migration checksum mismatch` 에러
- 이미 적용된 V1~V8 중 하나를 수정한 경우. Phase 1은 학습 단계라 `docker compose down -v` 로 초기화해도 무방.
- **운영에서는 절대 적용된 마이그레이션을 수정하지 않는다**. 항상 새 버전(V9, V10...)으로 추가한다.

### 3. 자동 코드가 `CUST-2026-0001` 부터 다시 발급된다 (시드와 충돌)
- `code_sequence` 행 초기화가 안 됐다는 의미. V8의 INSERT가 빠졌거나, 다른 prefix로 들어갔는지 확인.
- 직접 수정: `UPDATE code_sequence SET next_number = 3 WHERE prefix='CUST';`

### 4. `./gradlew test` 가 동시성 테스트에서 실패 (size < 50)
- `CodeSequence.advance()` 의 비관적 락이 누락된 변경이 들어간 경우. Repository의 `@Lock(LockModeType.PESSIMISTIC_WRITE)` 또는 `SELECT ... FOR UPDATE` 가 살아 있는지 확인.

## 시연 후 — 무엇이 학습됐는가

Phase 1 시연으로 **체감**으로 알게 된 것:

1. **"마스터 데이터를 함부로 못 지운다"는 말의 의미** — DELETE 호출이 통과하지만 DB 행은 그대로 남고, 검색에서만 빠진다. 트랜잭션 데이터(Phase 2 수주)가 이 행을 참조 중이어도 깨지지 않는 이유다.
2. **"코드 발급은 단순 시퀀스가 아니다"** — `CUST-YYYY-NNNN` 같은 사람이 읽는 코드는 prefix/year/순번 3개를 조합해야 한다. 그리고 동시 발급 시 락이 없으면 중복이 발생한다. ERP에서 코드 체계 설계가 왜 중요한 첫 단계인지 체감.
3. **"Flyway가 어제와 오늘을 잇는다"** — Phase 0에서 만든 DB에 V2~V8만 자동 추가 적용된다. 운영 DB도 같은 방식 — 어제 떠 있던 시스템에 새 변경만 누적되며 절대 처음부터 다시 만들지 않는다.
4. **"검증 실패와 업무 규칙 위반은 다르다"** — `@NotBlank` 위반은 400 (Bad Request), 중복 사업자번호는 409 (Conflict). HTTP 상태 코드로 둘을 구분하는 게 ERP 같은 시스템에서 클라이언트 처리를 단순하게 한다.

## 다음 단계 (Phase 2 예고)

Phase 2부터는 **트랜잭션 데이터**가 시작된다.

- **견적(Quotation)** → **수주(Sales Order)** → **출하(Delivery)** → **인보이스(Invoice)** 의 OTC 흐름.
- 각 트랜잭션이 Phase 1의 마스터(고객/상품)를 참조한다. 여기서 처음으로 "왜 마스터를 못 지우는가" 가 실제 코드로 드러난다.
- 수주 등록 시 **고객 신용한도 체크** — 마스터의 `creditLimit` 가 처음으로 업무 규칙에 사용된다.
- **헤더-라인(Header-Detail)** 구조 — 1주문 N라인. JPA OneToMany + cascade 설계.
- **상태(Status) 전이 기계** — DRAFT → CONFIRMED → SHIPPED → INVOICED. 한 단계라도 건너뛰면 안 됨.

Phase 1에서 다진 "코드 자동 발급 / Soft Delete / Auditing / 페이징" 인프라가 Phase 2에서도 그대로 재사용된다 — 마스터에서 한 번 만들어 둔 패턴이 트랜잭션 모듈에서도 빠짐없이 적용되는 게 ERP 학습의 핵심.
