# Phase 2 도메인 브리핑 — SD(영업) / OTC 트랜잭션

> 7단계 사이클의 **1단계: 도메인 브리핑**.
> 코드 한 줄 보기 전에, *"트랜잭션 데이터란 무엇이고, OTC(Order-to-Cash) 흐름은 어떻게 흐르는가"* 를 글로 충분히 이해한다.
> Phase 1에서 깐 마스터(고객/상품)가 이 Phase에서 처음으로 "**참조 당하는** 입장"이 된다. 머리에 그림이 안 그려지면 모델링/구현으로 넘어가지 않는다.

---

## 0. 이 Phase가 끝났을 때 답할 수 있어야 할 것

- "트랜잭션 데이터와 마스터 데이터의 차이를 OTC 예시로 설명할 수 있다"
- "수주에서 **헤더-라인(Header-Detail)** 구조가 왜 필요한가? 한 테이블로 통합하면 무엇이 깨지는가?"
- "수주는 왜 단순 CRUD가 아닌 **상태 전이 기계(State Machine)** 인가? 임의 상태로 점프하면 안 되는 이유는?"
- "**부분 출하 / 부분 인보이스**는 무엇이고, 왜 실무에서 거의 모든 거래가 이걸로 흘러가는가?"
- "수주 1건이 **신용한도**를 어떻게 소진/복원하며, 신용한도 체크를 Service에 두면 안 되는 (혹은 두어도 되는) 케이스는?"
- "트랜잭션 번호(`SO-20260524-001`)는 마스터 코드(`CUST-2026-0001`)와 무엇이 본질적으로 다른가?"

이 6개에 자기 말로 답할 수 있게 되는 게 Phase 2의 목표다.

## 1. 트랜잭션 데이터 — 다시 한번, 마스터와의 대비

Phase 1에서 짚었지만, 이제 트랜잭션 쪽을 본격적으로 만들기 시작하므로 시점을 바꿔서 한 번 더.

### "있는 것" 과 "일어난 것"

| 구분 | 마스터 데이터 (Phase 1) | 트랜잭션 데이터 (Phase 2~) |
| --- | --- | --- |
| 본질 | "있는 것" (Things) | **"일어난 것" (Events)** |
| 시제 | 현재형 — "있다" | **과거형 — "팔았다 / 보냈다 / 청구했다"** |
| 예시 | 고객, 상품, 거래처, 직원 | **수주, 출하, 인보이스, 입금** |
| 식별자 | 코드 (`CUST-2026-0001`) — 계속 같은 값 | **번호 (`SO-20260524-001`) — 발생일 + 일련번호** |
| 수정 | 자유로움 (주소 변경 등) | **확정되면 거의 수정 불가** (취소·반품으로 처리) |
| 삭제 | Soft Delete (덮어두기) | **물리 삭제 절대 X** (감사 추적 핵심) |
| FK 방향 | (받는 쪽) — 트랜잭션이 마스터를 가리킴 | **(주는 쪽)** — 마스터의 코드/PK 를 들고 다님 |
| 수명 | 길다 (회사 폐업까지) | 길다 (회계 보존 기간 = 보통 10년+) |

핵심은 시제다. 마스터는 "고객 신원전자가 **있다**", 트랜잭션은 "신원전자에게 노트북 10대를 **팔았다**". Phase 1이 명사형의 세계였다면 Phase 2부터는 동사형의 세계다.

### 마스터를 처음으로 "써먹는다"

Phase 1에서 "고객/상품을 마스터로 만든다" 의 가치는 사실 **이번 Phase부터 진짜 드러난다**.

- 수주 라인은 `(상품 PK, 수량, 단가)` 3개로 한 줄이 된다. 상품 마스터가 없었다면 `상품명, 단위, 카테고리, 표준원가, ...` 를 매번 라인에 직접 적어 넣어야 한다.
- 고객 신용한도(`creditLimit`)는 Phase 1 시점에서는 "그냥 컬럼 하나"였지만, Phase 2 수주 등록 단계에서 처음으로 **업무 규칙**으로 깨어난다.
- "고객을 물리 삭제하면 안 되는 이유" 가 이번 Phase에서 코드로 실증된다 — 작년에 쓴 수주의 고객이 사라지면 회계가 무너지니까.

## 2. OTC — Order-to-Cash 흐름의 전체 그림

영업(SD) 모듈의 본질은 한 줄로:

> **"고객에게 팔고, 보내고, 청구하고, 받는다"** — Order-to-Cash, 줄여서 **OTC**.

### OTC 6단계 (실무)

```
1. 견적 (Quotation)        — "이 가격에 어떠세요?"  (제안, 법적 효력 약함)
2. 수주 (Sales Order)      — "주문할게요"           (확정, 헤더 + 라인)
3. 출하 (Delivery)         — "보냈습니다"           (창고에서 출고, 재고 차감)
4. 인보이스 (Invoice)      — "결제하세요"           (세금계산서, 매출채권 발생)
5. 수금 (Payment)          — "입금했습니다"         (매출채권 소멸)
6. (회계 마감)             — 매출/세무 신고
```

각 단계는 **별개의 트랜잭션(엔티티)** 이고, 앞 단계를 **참조(FK)** 하며 이어진다. 즉:

```
Customer (Master)
   │
   ▼
Quotation ──→ SalesOrder ──→ Delivery ──→ Invoice ──→ Payment
                                                       │
                                                       ▼ (Phase 5: FI)
                                                  매출채권 / 매출전표
```

각 화살표가 **FK + 상태 전이**의 두 가지 의미를 모두 가진다.

### Phase 2의 범위 (학습 계획서 기준)

| 단계 | Phase 2에서 만드나? | 비고 |
| --- | --- | --- |
| 견적 (Quotation) | **만든다** (간단히) | 학습 목적 — 수주의 모태 |
| 수주 (Sales Order) | **만든다** (핵심) | 헤더-라인 구조 + 상태 전이 + 신용한도 |
| 출하 (Delivery) | **만든다** | 부분 출하 + 상태 전이 |
| 인보이스 (Invoice) | **만든다** | 매출액 계산 + 상태 전이 |
| 수금 (Payment) | **Phase 5(FI)** 에서 | "매출채권 소멸"을 회계 모듈에서 다룸 |
| 재고 차감 | **Phase 4(SD↔MM 연계)** 에서 | 지금은 "출하 처리만 하고 재고는 아직 안 본다" |
| 자동 전표 생성 | **Phase 5(FI)** 에서 |  |

> **Phase 2의 핵심은 "OTC 흐름의 절반"** — 수주~인보이스까지의 데이터 흐름을 만들고, 그 뒤(재고/회계/수금)는 Phase 3, 4, 5에서 단계적으로 붙인다.

### 한 시나리오로 정리 — "신원전자가 노트북 10대를 주문"

학습 내내 머리에 그려둘 한 줄짜리 시나리오:

```
1. (Phase 1 시드)  고객  : 신원전자        (CUST-2026-0001)
                  상품  : 노트북 X 1대 ₩1,500,000  (ITEM-2026-0001)

2. (견적)         "노트북 10대 ₩15,000,000 어떠세요?"
                  → Quotation Q-20260524-001  (DRAFT → SENT)

3. (수주)         "주문할게요"
                  → SalesOrder SO-20260524-001  (DRAFT → CONFIRMED)
                    헤더: 고객=신원전자, 총액=₩15,000,000
                    라인: ITEM-2026-0001 × 10, 단가 ₩1,500,000

                  + 신용한도 체크 (남은 한도 ≥ ₩15,000,000 인가?)

4. (부분 출하)    "오늘 6대 먼저 보냅니다"
                  → Delivery DLV-20260524-001  (SHIPPED)
                    라인: ITEM-2026-0001 × 6  ← SO 라인의 일부만 출하

5. (부분 인보이스) "출하한 6대분 청구"
                  → Invoice INV-20260524-001  (ISSUED)
                    총액 = ₩9,000,000 (6대분)

6. (재출하 + 재청구) — 나머지 4대로 동일 패턴 반복

7. (Phase 5)      입금 처리 → 매출채권 소멸
```

이 시나리오가 **Phase 2 시연 가이드(`4-시연-가이드.md`)** 에서 그대로 재현된다.

## 3. 트랜잭션 데이터의 5가지 핵심 특성

이 다섯 가지가 "수주 = 그냥 CRUD가 아닌 이유" 다.

### 3.1 헤더-라인(Header-Detail) 구조

#### 무엇인가
한 거래(주문 1건)는 **하나의 헤더 + N개의 라인** 으로 이뤄진다.

```
SalesOrder (헤더)                ← 1건
  id, number, customer_id, status, order_date, total_amount, ...
      │
      └── SalesOrderLine (라인)  ← N건 (보통 1~수십)
            id, sales_order_id, line_no, item_id, quantity, unit_price, line_total
```

- 헤더: "누가 / 언제 / 어떤 상태로 주문했는가" — 거래 전체의 메타데이터
- 라인: "무엇을 / 몇 개 / 얼마에" — 상품별 상세 1줄씩

#### 왜 한 테이블로 합치면 안 되나
직관적으로는 "한 줄에 다 적으면 되는 거 아냐?" 싶다. 두 가지 이유로 안 된다.

1. **한 주문에 여러 상품이 들어간다** — 노트북 10대 + 모니터 5대를 같이 주문하면 라인이 2개. 헤더-라인을 합치면 한 주문이 두 행으로 쪼개지고, "주문 1건"이라는 개념이 깨진다.
2. **라인별로 출하/인보이스가 분리된다** — "노트북 10대는 오늘 보내고, 모니터 5대는 다음 주에 보냄"이 가능해야 한다. 라인 단위로 추적되는 상태가 따로 있어야 한다.

> **헤더-라인은 ERP 트랜잭션의 기본형이다**. 수주, 출하, 인보이스, 입고, 회계전표, 급여명세서 — 거의 모든 트랜잭션이 같은 구조. Phase 2에서 이 패턴을 한 번 단단히 익히면 이후 Phase에 그대로 재사용된다.

#### JPA 표현
```java
@Entity
class SalesOrder extends BaseEntityWithCode {
    @OneToMany(mappedBy = "salesOrder", cascade = ALL, orphanRemoval = true)
    private List<SalesOrderLine> lines = new ArrayList<>();
    ...
}

@Entity
class SalesOrderLine extends BaseEntity {
    @ManyToOne(fetch = LAZY)
    @JoinColumn(name = "sales_order_id")
    private SalesOrder salesOrder;

    @ManyToOne(fetch = LAZY)
    @JoinColumn(name = "item_id")
    private Item item;     // ← Phase 1 마스터 참조
    ...
}
```

설계 단계(`2-설계-제안.md`)에서 `cascade` / `orphanRemoval` / `FetchType` 의 트레이드오프를 자세히 다룬다.

### 3.2 상태 전이 (State Machine)

#### 무엇인가
트랜잭션은 시간을 따라 **정해진 상태들 사이를 한 방향으로** 흐른다.

수주의 상태:
```
DRAFT       (작성 중)
   │ confirm()
   ▼
CONFIRMED   (확정 — 신용한도 차감, 출하 대기)
   │ ship() / partialShip()
   ▼
SHIPPING    (출하 중 — 부분 출하가 가능하므로 중간 상태)
   │ allShipped()
   ▼
SHIPPED     (전량 출하 완료)
   │ invoice() / partialInvoice()
   ▼
INVOICING   (청구 중)
   │ allInvoiced()
   ▼
INVOICED    (전량 청구 완료 — 사실상 SD 종결)
   │
   ▼ (Phase 5: FI 수금 시) → CLOSED

(예외)
ANY → CANCELLED    (취소 — 단, CONFIRMED 이후엔 별도 승인 절차)
```

#### 왜 단순 CRUD가 아닌가
업무 규칙이 상태에 묶여 있기 때문이다:

- `DRAFT` 상태에서만 라인을 자유롭게 추가/삭제할 수 있다 (확정 후엔 변경 추적이 필요).
- `CONFIRMED` 가 되어야 비로소 신용한도를 차감한다 (드래프트 단계는 한도 영향 X).
- `SHIPPED` 가 아닌 상태에서 인보이스를 발행하려고 하면 → 거부.
- 임의로 `INVOICED → DRAFT` 로 되돌아가는 건 절대 안 됨 (회계 자료 정합성).

이런 규칙을 if-else로 곳곳에 깔면 빠뜨리기 쉽다. 그래서 **"상태 전이는 도메인 객체의 메서드로만"** 표현한다:

```java
class SalesOrder {
    public void confirm() {
        if (status != DRAFT) throw new IllegalStateException(...);
        if (lines.isEmpty()) throw new IllegalStateException(...);
        // 신용한도 검증은 Service 단에서 (외부 의존이므로)
        this.status = CONFIRMED;
    }
}
```

Service나 Controller에서 `order.setStatus(CONFIRMED)` 같은 직접 변경을 **금지**한다. 모든 상태 변경은 위 같은 메서드를 거치게 강제하면, 한 곳에서만 규칙을 보면 된다.

> **상태 전이 메서드 안에서 검증이 끝난 뒤 status 를 바꾼다** — 이 패턴이 트랜잭션 도메인의 핵심 코드 스타일이다.

#### Phase 2의 상태 전이 학습 목표
- 4개 트랜잭션(견적/수주/출하/인보이스) 각각의 상태 전이 다이어그램을 머리에 그릴 수 있다.
- "왜 DRAFT 와 CONFIRMED 가 분리되어 있는가" 를 신용한도 차감 타이밍으로 설명할 수 있다.
- 부분 출하/인보이스 때문에 중간 상태(`SHIPPING`, `INVOICING`)가 필요한 이유를 설명할 수 있다.

### 3.3 트랜잭션 번호 체계 — 마스터 코드와의 차이

마스터 코드와 트랜잭션 번호는 둘 다 "사람이 읽는 식별자"지만 패턴이 다르다.

| 항목 | 마스터 코드 (Phase 1) | 트랜잭션 번호 (Phase 2) |
| --- | --- | --- |
| 예시 | `CUST-2026-0001` | `SO-20260524-001` |
| 구성 | prefix + **연도** + 순번 | prefix + **발생일자(YYYYMMDD)** + 그날의 순번 |
| 리셋 주기 | 연 단위 (`year + next_number`) | **일 단위** (그날 자정 지나면 001부터 재시작) |
| 의미 | "어느 해 등록된 N번째 고객" | "**언제 일어났던 N번째 거래**" |
| 사람이 보고 알 수 있는 정보 | 등록 연도, 종류 | **거래 발생일**, 종류 — 영업 입장에서 매우 중요 |

#### 왜 트랜잭션은 일자 기반인가
- "어제 들어온 수주를 찾아줘" → 번호 앞 8자리만 보면 됨 (`SO-20260523-*`).
- 회계 마감은 일/월 단위. 번호에 날짜가 박혀 있어야 마감 작업이 수월.
- 인쇄물(주문서/세금계산서)에 "발생일자"가 함께 찍히면 사용자가 안심.

#### Phase 1 인프라의 재사용
Phase 1에서 만든 `code_sequence` 테이블과 `CodeGenerator` 는 prefix + year + next_number 구조였다. Phase 2에서는 이걸 **`prefix + date(YYYYMMDD) + next_number`** 로 확장하거나, 별도 시퀀스 발급기를 추가해야 한다.

> 설계 단계(`2-설계-제안.md`)에서 "기존 `code_sequence` 를 확장할지, 새 `transaction_sequence` 를 둘지" 를 결정한다.

같은 비관적 락(`PESSIMISTIC_WRITE`) 패턴은 그대로 재사용 — 50개 스레드가 동시에 수주를 등록해도 같은 번호가 나오면 안 되므로.

### 3.4 마스터 참조와 무결성 — Phase 1 마스터의 "첫 사용"

수주 1건은 적어도 두 종류의 마스터를 참조한다.

```
SalesOrder.customer_id        → Customer.id   (헤더)
SalesOrderLine.item_id        → Item.id       (라인마다)
SalesOrder.salesperson_id     → Employee.id   (담당자, optional)
```

#### 참조 무결성 = "지워진 마스터를 가리키면 안 된다"

DB 레벨에서 FK 제약으로 보호한다:
```sql
CONSTRAINT fk_so_customer FOREIGN KEY (customer_id) REFERENCES customer(id)
CONSTRAINT fk_sol_item    FOREIGN KEY (item_id)     REFERENCES item(id)
```

> "그러면 Phase 1 Soft Delete 와 충돌하지 않나? — `deleted_at` 이 채워진 고객을 수주가 참조 중인데 새 수주는 만들 수 없게 막아야 하나?"

**구분이 필요하다**:
- **이미 존재하는 수주가 참조 중인 마스터 행** → Soft Delete 되어 있어도 FK 입장에서는 "행이 살아 있다" 이므로 무결성 깨지지 않음 (Soft Delete = 행은 그대로, 표식만).
- **새 수주를 만들려는데 참조하려는 고객이 Soft Delete 됨** → Service 단에서 "활성 고객(`deleted_at IS NULL` AND `status = ACTIVE`)" 만 선택 가능하도록 막는다.

이게 Phase 1에서 "물리 삭제가 아니라 Soft Delete + status" 를 둔 이유다. 둘이 함께 작동해야 트랜잭션이 깨지지 않으면서 새 거래는 막을 수 있다.

#### Service 단 사전 검증 vs DB FK 위반
사용자에게 보여줄 메시지의 품질 차이:

| 보호 위치 | 사용자가 보는 에러 |
| --- | --- |
| Service에서 사전 검증 | `404: Customer not found: id=999` (이해 가능) |
| DB FK 위반만 | `500: SQLIntegrityConstraintViolationException: a foreign key constraint fails` (이해 불가) |

→ Service에서 먼저 검증, DB는 마지막 방어선. (Phase 1에서 이미 적용한 패턴.)

### 3.5 금액 계산 — 라인 합산과 부동소수점

#### 한 라인의 금액
```
line_total = quantity * unit_price - line_discount + line_tax
```

#### 헤더 총액
```
order_total = sum(line_total) - order_discount + freight + order_tax
```

#### 함정 1 — `BigDecimal` 강제
`double`로 0.1 + 0.2 를 더하면 0.30000000000000004 가 나오는 그 문제. 회계는 1원도 틀리면 안 되므로 **모든 금액은 `BigDecimal`**, scale은 보통 0 또는 2 자리로 고정.

#### 함정 2 — 합계를 **DB에 저장할 것인가, 매번 계산할 것인가**
두 가지 옵션:

- **저장 (Denormalization)** — `sales_order.total_amount` 컬럼을 둔다. 라인 변경 시마다 헤더의 합계를 다시 계산해 저장. 조회는 빠르지만 두 값이 어긋날 위험.
- **계산 (Normalization)** — `sales_order` 에는 total을 저장하지 않고 매번 `SUM(line_total)`. 정합성 보장. 조회는 느릴 수 있음.

**실무 ERP는 거의 다 "저장" 한다** (조회/리포트가 압도적으로 많고, 라인 합계가 매번 SQL `SUM`을 타면 성능이 안 나오므로). 단, 라인 변경 메서드 안에서 헤더의 `recalculateTotal()` 을 반드시 호출하도록 강제한다.

> 설계 단계에서 결정. Phase 2 학습 목적상 두 가지를 모두 짚는다.

#### 함정 3 — 세금 처리
한국 부가가치세는 공급가의 10%. 라인 단위로 부가세를 가질지, 헤더 단위로 한 번에 계산할지 회계 처리에 따라 달라진다.

Phase 2 학습 단계에서는 **라인 단위 부가세** 로 단순화 (`line_total = (qty × unit_price) × 1.1`). Phase 5(FI)에서 매출/매입 부가세 처리를 본격적으로 다룬다.

## 4. 업무 규칙(Business Rules) — 코드로 강제할 것들

위 5개 특성 위에 얹히는 "이건 절대 안 된다" 규칙들. Service/Domain 코드의 핵심 분량을 차지한다.

### 4.1 신용한도 체크 (Credit Limit Check)

#### 시나리오
신원전자의 `creditLimit = ₩50,000,000`. 이미 미수금 ₩40,000,000 이 있는 상태. ₩15,000,000 짜리 수주가 들어오면?

```
신용한도            : ₩50,000,000
사용 중 (확정된 SO) : ₩40,000,000
남은 한도          : ₩10,000,000
신규 수주 요청     : ₩15,000,000   ← 거부 (한도 초과)
```

#### 체크 타이밍
- **`DRAFT` 등록 시점이 아니라 `confirm()` 시점**에서 검증.
- 드래프트는 "고민 중" 상태라 한도 차감 X. 확정되어야 비로소 한도 차감.

#### 체크 위치
Service 단:
```java
@Transactional
public void confirm(Long orderId) {
    SalesOrder order = repo.findById(orderId).orElseThrow();
    BigDecimal used = repo.sumConfirmedAmountByCustomer(order.getCustomerId());
    BigDecimal remaining = order.getCustomer().getCreditLimit().subtract(used);
    if (order.getTotalAmount().compareTo(remaining) > 0) {
        throw new BusinessException("신용한도 초과: 남은 한도 " + remaining);
    }
    order.confirm();   // 도메인 메서드는 상태 전이만
}
```

도메인 메서드(`order.confirm()`)는 자기 자신의 상태만 보고 결정 가능한 규칙(이 검증 통과 후 DRAFT → CONFIRMED), Service는 **외부 의존이 필요한 규칙**(다른 SO들의 합계 계산) — 이 분리가 Phase 2의 학습 포인트 중 하나.

#### 한도 복원
수주가 `CANCELLED` 되면 한도가 복원된다. 인보이스가 발행되면 그건 별개 — 매출채권으로 넘어가는 단계라 회계 로직이 따라 붙음(Phase 5).

### 4.2 부분 출하 / 부분 인보이스 (Partial Delivery / Partial Invoice)

#### 무엇인가
수주 10대를 한 번에 못 보낼 때 (재고 부족, 분할 납기, 운송 제약 등) 6대 먼저 출하 → 나중에 4대 추가 출하. 인보이스도 마찬가지.

#### 데이터 구조에 미치는 영향
SO 라인이 "주문량(`order_qty`) / 출하량 누적(`shipped_qty`) / 청구량 누적(`invoiced_qty`)" 3개 수치를 갖는다.

```
SalesOrderLine
  order_qty     : 10  (주문량 — 변하지 않음)
  shipped_qty   :  6  (현재까지 출하된 누적)
  invoiced_qty  :  6  (현재까지 청구된 누적)
```

- 출하 라인이 추가될 때 `shipped_qty += delivery_qty`.
- 인보이스 라인이 추가될 때 `invoiced_qty += invoice_qty`.

#### 불변 조건 (Invariants)
- `shipped_qty ≤ order_qty` — 주문량을 초과해 출하 불가.
- `invoiced_qty ≤ shipped_qty` — 보내지도 않은 걸 청구하면 안 됨.

> 이 두 부등식이 깨지면 회계가 망가진다. 도메인 메서드 (`SalesOrderLine.recordShipment(qty)`) 안에서 검증.

#### 헤더 상태와 라인 상태
- 라인의 `shipped_qty == order_qty` 가 되면 그 라인은 "전량 출하".
- 모든 라인이 전량 출하면 헤더의 상태가 `SHIPPING → SHIPPED` 로.
- 부분 출하 중에는 헤더가 `SHIPPING` (중간 상태).

### 4.3 상태 전이 규칙 (재강조)

라인이든 헤더든, 상태 변경은 **메서드를 통해서만**. 도메인 객체 외부에서 `setStatus()` 호출 금지.

이를 강제하기 위해 자주 쓰는 패턴:
- `setStatus()` 를 `package-private` 으로 막거나 아예 없앤다.
- 변경자 메서드(`confirm()`, `ship()`, `cancel()`) 만 public.
- 모든 변경자 메서드 첫 줄에 "현재 상태가 허용된 상태인가" 검증.

## 5. 다른 모듈과의 연계 — 이번 Phase가 어디까지인가

| 영향 | Phase 2에서 다루나? | 어디서? |
| --- | --- | --- |
| 고객 마스터 참조 | **다룬다** | 수주 헤더의 customer_id |
| 상품 마스터 참조 | **다룬다** | 수주 라인의 item_id |
| 직원(담당자) 참조 | **다룬다** (optional) | 수주 헤더의 salesperson_id |
| **재고 차감** | **안 한다** | Phase 3(MM) 에서 재고 도입 → Phase 4(SD↔MM 연계) 에서 출하 시 자동 차감 |
| **자동 회계 전표** | **안 한다** | Phase 5(FI) 에서 인보이스 발행 시 매출 전표 자동 생성 |
| **수금 처리** | **안 한다** | Phase 5(FI) 에서 매출채권 소멸 처리 |
| **권한(영업팀만)** | **안 한다** | Phase 6(인증/인가) 에서 모듈별 권한 |

**Phase 2 단독으로는 "출하 처리 = 그냥 상태가 SHIPPED 가 된다" 까지만 한다.** 재고는 아직 마이너스가 되는지 모른다(MM 모듈이 없으니까). 이 한계가 다음 Phase에서 어떻게 채워지는지를 미리 그려두는 것이 학습의 큰 그림.

```
Phase 2:   수주 → 출하 → 인보이스          (상태만 흐름)
Phase 3:                                      재고 마스터 도입
Phase 4:   수주 → 출하 → 인보이스
                  └─ 재고 자동 차감          (한 트랜잭션으로 묶임)
Phase 5:                            인보이스
                                    └─ 매출 전표 자동 생성
                                          └─ 매출채권 소멸 (수금)
```

## 6. JPA에서의 구현 — 마스터와 달라지는 포인트

Phase 1과 비교했을 때 새로 등장하는 JPA 기법.

### 6.1 `@OneToMany` 양방향 + `cascade` + `orphanRemoval`
헤더(SO) ↔ 라인(SOL) 의 양방향 관계. `cascade = ALL + orphanRemoval = true` 면 헤더가 라인을 자식처럼 거느린다 — 헤더 저장 시 라인도 같이, 라인 컬렉션에서 빠진 라인은 삭제.

함정: 양방향 관계는 무한 직렬화/순환 참조를 일으키기 쉽다. DTO 변환을 일찍 적용해 엔티티 그래프를 API로 노출하지 않는다.

### 6.2 `FetchType.LAZY` 가 거의 모든 ManyToOne의 디폴트
SO 라인이 ITEM을 참조한다고 해서 SO 조회마다 ITEM을 같이 끌고 오면 N+1이 폭발한다. LAZY 로 두고, 화면이 필요할 때만 `fetch join` 으로 끌어온다.

### 6.3 도메인 메서드의 위치
"라인 추가" 같은 행위는 **헤더의 메서드**가 적절하다:

```java
class SalesOrder {
    public void addLine(Item item, int qty, BigDecimal unitPrice) {
        SalesOrderLine line = new SalesOrderLine(this, item, qty, unitPrice);
        this.lines.add(line);
        this.recalculateTotal();
    }
}
```

이렇게 두면 "총액 재계산을 잊는 실수" 가 원천 차단된다.

### 6.4 트랜잭션 경계 (`@Transactional`) 의 재정의
Phase 1에서는 메서드 하나가 한 트랜잭션이었다. Phase 2부터는 한 트랜잭션이 더 길어진다 — "수주 라인 5개 등록 + 헤더 총액 계산 + 신용한도 체크 + 시퀀스 발급" 이 모두 한 트랜잭션 안.

`@Transactional(readOnly = true)` (조회) vs `@Transactional` (변경) 의 구분도 본격적으로.

### 6.5 동시성 — 시퀀스 외에도
- 같은 고객의 신용한도 차감을 동시에 두 트랜잭션이 시도하면? → 신용한도 검증 시점부터 차감 완료 시점까지 락이 필요할 수 있음.
- Phase 1에서는 시퀀스에만 비관적 락을 걸었지만, Phase 2는 **고객의 한도 행 자체에 락**을 거는 패턴이 등장.

(설계 단계에서 트레이드오프 — 비관적 락의 범위 vs 낙관적 락 + 재시도.)

## 7. Phase 2에서 미리 알아둘 함정

| 함정 | 설명 | Phase 2에서의 결정 |
| --- | --- | --- |
| **헤더-라인 양방향 무한 직렬화** | JSON 직렬화 시 SO ↔ SOL ↔ SO 무한 루프 | DTO 사용 강제, `@JsonIgnore` 보조 |

| **트랜잭션 번호 중복** | 동시에 두 수주가 같은 번호 발급 | Phase 1의 비관적 락 패턴 재사용 |
| **취소된 수주의 신용한도 복원 누락** | `CANCELLED` 처리는 했지만 한도는 그대로 차감 중 | 취소 메서드 안에서 명시적으로 복원 |

| **인보이스 후 수주 수정 허용** | 라인 수량을 바꾸면 회계가 무너짐 | `INVOICED` 이후엔 수정 차단 |

## 9. 다음 단계로 넘어가기 전 — 자기 점검

아래 질문에 자기 말로 답할 수 있는지 점검한다. 막히는 항목이 있으면 그 번호를 알려 주면 더 자세히 풀어 설명한다.

1. "마스터 데이터와 트랜잭션 데이터의 차이를 OTC 시나리오로 5문장 안에 설명해 보세요."
2. "수주를 헤더-라인이 아니라 단일 테이블로 만들면 깨지는 케이스 두 가지를 들어 보세요."
3. "수주의 상태 전이 다이어그램을 5개 상태 이상으로 그릴 수 있나요? 왜 `DRAFT` 와 `CONFIRMED` 가 분리되어 있나요?"
4. "부분 출하 / 부분 인보이스 시 라인이 가져야 할 수치 세 가지(`order_qty / shipped_qty / invoiced_qty`)와, 이들 사이에 성립해야 할 부등식 두 개를 말해 보세요."
