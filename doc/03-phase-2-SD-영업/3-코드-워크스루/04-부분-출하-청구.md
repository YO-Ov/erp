# 4/7. 부분 출하·청구 — `Delivery` + `Invoice` 와 SO 라인의 동행

> 한 트랜잭션 안에서 출하/인보이스 라인 추가 + SO 라인 누적 + 헤더 상태 전이가 묶여 일어난다.
> 설계서 §3.3 `Delivery` + §3.4 `Invoice` + §5.2 부분 출하 흐름.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/sd/
├─ delivery/
│  ├─ Delivery.java
│  ├─ DeliveryLine.java
│  └─ DeliveryService.java
└─ invoice/
   ├─ Invoice.java
   ├─ InvoiceLine.java
   └─ InvoiceService.java
```

---

## 🔥 출하 라인이 SO 라인을 직접 가리킨다

```java
@Entity
@Table(name = "delivery_line")
public class DeliveryLine extends BaseEntity {

    @ManyToOne(fetch = LAZY, optional = false)
    @JoinColumn(name = "delivery_id", nullable = false)
    private Delivery delivery;

    @ManyToOne(fetch = LAZY, optional = false)
    @JoinColumn(name = "sales_order_line_id", nullable = false)
    private SalesOrderLine salesOrderLine;   // ← 핵심

    private int lineNo;
    private BigDecimal quantity;
}
```

설계서 §0 #8: **"Delivery/Invoice 라인은 `sales_order_line_id` 를 직접 FK 참조"**.

### 왜 `item_id` 가 없나

출하 라인은 `item_id` 컬럼을 따로 안 들고 SO 라인만 가리킨다. 이유:
- SO 라인이 이미 `item_id` 를 들고 있음.
- 출하는 "이 SO 라인의 일부" 가 본질 — item 을 따로 들면 SO 라인 의 item 과 어긋날 위험.

화면에서 item 정보가 필요하면:
```java
deliveryLine.getSalesOrderLine().getItem();
```

한 단계 거치면 된다.

### 왜 `quantity` 만 들고 `unit_price` 가 없나

출하는 "물건이 나갔다" 만 기록 — 가격 정보는 SO 라인에 있는 것을 인용한다. 회계 금액은 인보이스 라인에서 다시 동결(복사)되므로 출하 라인은 가격을 안 들어도 충분.

→ **출하 라인의 책임은 수량 + 시점**.

## 🔥 `DeliveryLine` 생성자 — 잔여 출하 가능량 검증

```java
DeliveryLine(Delivery delivery, int lineNo, SalesOrderLine salesOrderLine, BigDecimal quantity) {
    if (salesOrderLine == null) throw new IllegalArgumentException("salesOrderLine 은 null 일 수 없다.");
    if (quantity == null || quantity.signum() <= 0)
        throw new IllegalArgumentException("quantity 는 0보다 커야 한다.");
    BigDecimal remaining = salesOrderLine.remainingShippable();
    if (quantity.compareTo(remaining) > 0) {
        throw new IllegalStateException(
                "잔여 출하 가능량을 초과합니다. remainingShippable=" + remaining + ", quantity=" + quantity);
    }
    this.delivery = delivery;
    this.lineNo = lineNo;
    this.salesOrderLine = salesOrderLine;
    this.quantity = quantity;
}
```

### 두 단계 검증
1. **생성자에서 잔여 확인** — `remainingShippable()` 보다 더 많이 보내려 하면 거부.
2. **SalesOrderLine.addShippedQty 에서 다시 확인** — 같은 검증을 라인 누적 시 한 번 더.

중복처럼 보이지만 **방어선이 두 겹**인 게 의도된 설계.
- 생성자 단계: DeliveryLine 객체 자체가 만들어지지 않게 차단. 도메인 무결성 1차 방어.
- SO 누적 단계: 동시 출하 두 건이 들어와도 마지막 합산에서 한 번 더 잡힘.

> 💡 같은 트랜잭션 안이면 동시성은 큰 이슈가 아니지만, 동시 트랜잭션 시 SO 라인에 락이 없으니(Phase 2 의도적 미적용) 두 출하가 잔여를 동시에 확인하면 둘 다 통과할 수 있다. 그래서 SO 라인의 검증도 살아 있어야 함.

### `package-private` 생성자
```java
DeliveryLine(...)   // public 이 아님
```

`Delivery.addLine(sol, qty)` 만 부를 수 있게 막아 둔다. 외부에서 `new DeliveryLine(...)` 으로 만들면 `delivery.lines` 컬렉션과 동기화 안 됨.

## 🔥 `Delivery` 도메인 메서드 — 헤더 책임 한 줄

```java
public DeliveryLine addLine(SalesOrderLine salesOrderLine, BigDecimal quantity) {
    if (status != DeliveryStatus.DRAFT)
        throw new IllegalStateException("DRAFT 출하만 라인 추가 가능합니다. 현재: " + status);
    DeliveryLine line = new DeliveryLine(this, lines.size() + 1, salesOrderLine, quantity);
    lines.add(line);
    return line;
}

public void ship() {
    if (status != DeliveryStatus.DRAFT)
        throw new IllegalStateException("DRAFT 출하만 확정 가능합니다. 현재: " + status);
    if (lines.isEmpty())
        throw new IllegalStateException("라인이 비어 있는 출하는 확정할 수 없습니다.");
    this.status = DeliveryStatus.SHIPPED;
}

public void cancel() {
    if (status != DeliveryStatus.SHIPPED)
        throw new IllegalStateException("SHIPPED 출하만 취소 가능합니다. 현재: " + status);
    this.status = DeliveryStatus.CANCELLED;
}
```

상태 3개 (`DRAFT/SHIPPED/CANCELLED`) — 수주에 비하면 단순. 출하는 한 번 보내면 끝, 취소가 끝.

### `Delivery.ship()` 은 SO 누적을 안 부른다

`ship()` 이 끝나도 SO 라인의 `shipped_qty` 는 아직 0. 자기 헤더 상태만 SHIPPED 로 바꾼다. **SO 누적은 서비스가 책임**:

```java
// DeliveryService.create(...)
delivery.ship();   // 헤더만 SHIPPED 로
for (DeliveryLine dline : delivery.getLines()) {
    order.recordShipment(dline.getSalesOrderLine(), dline.getQuantity());  // SO 누적
}
```

왜 도메인이 직접 안 하는가?
- `Delivery.ship()` 이 `SalesOrder.recordShipment(...)` 를 부르려면 `delivery.getSalesOrder()` 를 깊이 들여다봐야 함.
- 트랜잭션 경계(`@Transactional`) 와 호출 순서를 서비스가 통제하는 게 더 명확.
- 두 엔티티가 양방향으로 부르는 결합이 줄어든다.

→ 도메인 메서드는 **자기 상태만**, 도메인 간 협업은 **서비스 단**에서.

## 🔥 `DeliveryService.create()` — 한 트랜잭션의 전체 흐름

```java
@Transactional
public DeliveryResponse create(DeliveryCreateRequest req) {
    SalesOrder order = salesOrderRepository.findById(req.salesOrderId())
            .orElseThrow(() -> new EntityNotFoundException("..."));

    String number = numberGenerator.nextDeliveryNumber(req.shippedDate());
    Delivery delivery = Delivery.draft(number, order, req.shippedDate());

    for (DeliveryLineRequest lineReq : req.lines()) {
        SalesOrderLine sol = order.findLineById(lineReq.salesOrderLineId());
        delivery.addLine(sol, lineReq.quantity());
    }

    delivery.ship();
    // SO 라인의 shipped_qty 누적 + SO 헤더 상태 전이 (같은 트랜잭션)
    for (DeliveryLine dline : delivery.getLines()) {
        order.recordShipment(dline.getSalesOrderLine(), dline.getQuantity());
    }

    return mapper.toResponse(repository.save(delivery));
}
```

### 한 `@Transactional` 안에서 일어나는 일

1. SO 헤더 로드 (한 SELECT).
2. 번호 발급 (`REQUIRES_NEW` 별도 트랜잭션 — 01편).
3. `Delivery.draft(...)` — 메모리에 헤더 생성.
4. SO 라인 검색 + 출하 라인 추가 — 각 라인마다 잔여 확인.
5. `delivery.ship()` — 출하 헤더 SHIPPED.
6. SO 라인 누적 — 각 라인마다 `recordShipment` → 헤더 `recomputeStatus`.
7. `delivery` 저장 — 헤더 + 라인 cascade INSERT.

이 모든 게 **한 트랜잭션**. 중간에 어디서 실패해도(예: 라인 5개 중 4번째에서 잔여 초과) **모두 롤백** — SO 누적도 안 변하고 Delivery 도 안 만들어짐.

→ 출하 등록이 **all-or-nothing**. 부분 상태 (3 라인은 누적됐는데 1 라인 실패) 가 운영 DB 에 남지 않는다.

### `findLineById(...)` 가 "이 수주" 보장

```java
SalesOrderLine sol = order.findLineById(lineReq.salesOrderLineId());
```

다른 수주의 라인 ID 를 넘기면 검색 실패 (`IllegalArgumentException`). API 요청에 잘못된 SO 라인 ID 가 끼어 있어도 데이터가 오염되지 않는다.

## 🔥 출하 취소 — 누적 차감 + 헤더 상태 복원

```java
@Transactional
public DeliveryResponse cancel(Long id) {
    Delivery delivery = getOrThrow(id);
    SalesOrder order = delivery.getSalesOrder();
    delivery.cancel();
    for (DeliveryLine dline : delivery.getLines()) {
        order.cancelShipment(dline.getSalesOrderLine(), dline.getQuantity());
    }
    return mapper.toResponse(delivery);
}
```

`cancelShipment` 가 `subtractShippedQty` 를 호출:
```java
void subtractShippedQty(BigDecimal qty) {
    BigDecimal next = this.shippedQty.subtract(qty);
    if (next.compareTo(invoicedQty) < 0)
        throw new IllegalStateException(
                "이미 청구된 수량보다 적게 출하 누적을 되돌릴 수 없습니다. ...");
    this.shippedQty = next;
}
```

핵심 방어:
- 6대 출하 후 6대 청구한 상태에서 6대 출하를 취소하려 하면 → "invoiced_qty=6 > shipped_qty(after)=0" → 거부.
- 청구가 먼저 취소되어야만 출하 취소 가능.

→ 회계 일관성: **청구된 적 없는 출하만 되돌릴 수 있다**.

`recomputeStatus()` 가 차감된 누적에서 헤더 상태를 다시 계산 — `anyShipped=false` 면 CONFIRMED 로 복원.

## 🔥 `Invoice` — 가격 동결의 자리

```java
@Entity
public class InvoiceLine extends BaseEntity {
    @ManyToOne private Invoice invoice;
    @ManyToOne private SalesOrderLine salesOrderLine;
    private int lineNo;
    private BigDecimal quantity;
    private BigDecimal unitPrice;   // ← SO 라인 단가 복사
    private BigDecimal lineTotal;

    InvoiceLine(Invoice invoice, int lineNo, SalesOrderLine salesOrderLine, BigDecimal quantity) {
        ...
        this.unitPrice = salesOrderLine.getUnitPrice();    // ← 동결
        this.lineTotal = quantity.multiply(this.unitPrice);
    }
}
```

설계서 §3.4 의 핵심 결정: **인보이스 라인이 단가를 복사**한다.

### 왜 단가를 또 복사하는가

SO 라인이 이미 단가를 동결하고 있다 (Item 마스터의 `standardPrice` 가 미래에 바뀌어도 SO 라인은 그대로). 그런데 인보이스 라인이 한 번 더 복사하는 건 **회계 보존** 때문:

```
Item.standardPrice 1_200_000 → 1_500_000 으로 변경 (정책)
   ↓ (SO 라인은 영향 없음 — 이미 1_200_000 동결)
SO.lines[i].unitPrice = 1_200_000  ← 안전
   ↓ 그런데 SO 라인 단가도 누군가 어드민에서 수정한다면?
   ↓ (Phase 2 에서는 SO 확정 후 수정 불가지만, 미래 확장 시...)
   ↓ 회계상 발행된 인보이스 금액은 절대 변하면 안 됨
Invoice.lines[i].unitPrice = 1_200_000  ← 두 번째 안전망
```

회계 원장은 **발행 시점의 금액**이 보존되어야 한다. 미래에 어떤 데이터가 어떻게 바뀌어도 발행된 인보이스 금액은 그대로.

→ **금액 무결성을 위한 의도된 데이터 중복**.

## 🔥 부가세 — 라인 합계 × 0.10

```java
public static final BigDecimal VAT_RATE = new BigDecimal("0.10");

private void recalculateTotals() {
    this.subtotal = lines.stream()
            .map(InvoiceLine::getLineTotal)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    this.taxAmount = subtotal.multiply(VAT_RATE).setScale(2, RoundingMode.HALF_UP);
    this.totalAmount = subtotal.add(taxAmount);
}
```

### `subtotal / tax_amount / total_amount` 세 컬럼 분리
한국 회계 처리상 공급가와 부가세를 **분리 표시**가 의무. 한 컬럼(`total_amount`)만 두고 라인에서 매번 계산하지 않은 이유:
- 리포트 (월/분기 부가세 신고) 에서 헤더 한 컬럼만 SUM 하면 됨.
- 발행 시점의 부가세율이 동결 — 향후 부가세율 변경 시에도 발행된 인보이스의 세액 보존.

### `setScale(2, RoundingMode.HALF_UP)`
부가세 계산이 소수점 셋째 자리 이하를 만들 수 있어 (예: `2370.555`). 한국 원화는 원 단위(소수 둘째 자리까지 의미 없음)지만 컬럼은 `DECIMAL(15,2)` 이라 두 자리 보존 + 반올림.

`HALF_UP` (0.5 올림) — 한국 회계의 일반 반올림 정책. 더 보수적으로 `HALF_EVEN` (은행원 반올림) 도 가능하지만 학습 단순화.

### Phase 2 의도적 단순화
- **부가세 10% 단일** — 면세 / 영세 / 다국가는 학습 범위 밖.
- **라인 단위 부가세 없음** — 라인은 공급가만, 세액은 헤더 단 합산. 다중 부가세율을 다루는 시점에 라인 단위 분리로 진화 가능.

## 🔥 `InvoiceLine` 생성자 — 잔여 청구 가능량 검증

```java
InvoiceLine(Invoice invoice, int lineNo, SalesOrderLine sol, BigDecimal quantity) {
    ...
    BigDecimal remaining = sol.remainingInvoiceable();   // shipped - invoiced
    if (quantity.compareTo(remaining) > 0) {
        throw new IllegalStateException(
                "잔여 청구 가능량을 초과합니다. remainingInvoiceable=" + remaining + ", quantity=" + quantity);
    }
    this.invoice = invoice;
    this.lineNo = lineNo;
    this.salesOrderLine = sol;
    this.quantity = quantity;
    this.unitPrice = sol.getUnitPrice();
    this.lineTotal = quantity.multiply(this.unitPrice);
}
```

`remainingInvoiceable() = shipped_qty - invoiced_qty` — **이미 보낸 것 중 미청구분만 청구 가능**. 보내지 않은 수량을 청구하려 하면 거부.

이게 **"보낸 만큼만 청구"** 라는 OTC 흐름의 도메인 규칙 (설계서 §3.2 의 불변 조건). 출하 라인에서는 `order_qty - shipped_qty`, 인보이스 라인에서는 `shipped_qty - invoiced_qty` — 각자의 잔여 계산식.

## 🔥 `InvoiceService.create()` + `cancel()` — Delivery 와 닮은 패턴

```java
@Transactional
public InvoiceResponse create(InvoiceCreateRequest req) {
    SalesOrder order = salesOrderRepository.findById(req.salesOrderId())...;
    String number = numberGenerator.nextInvoiceNumber(req.invoiceDate());
    Invoice invoice = Invoice.draft(number, order, req.invoiceDate());

    for (InvoiceLineRequest lineReq : req.lines()) {
        SalesOrderLine sol = order.findLineById(lineReq.salesOrderLineId());
        invoice.addLine(sol, lineReq.quantity());
    }

    invoice.issue();   // 자기 헤더만 ISSUED
    for (InvoiceLine iline : invoice.getLines()) {
        order.recordInvoicing(iline.getSalesOrderLine(), iline.getQuantity());  // SO 누적
    }
    return mapper.toResponse(repository.save(invoice));
}

@Transactional
public InvoiceResponse cancel(Long id) {
    Invoice invoice = getOrThrow(id);
    SalesOrder order = invoice.getSalesOrder();
    invoice.cancel();
    for (InvoiceLine iline : invoice.getLines()) {
        order.cancelInvoicing(iline.getSalesOrderLine(), iline.getQuantity());
    }
    return mapper.toResponse(invoice);
}
```

`DeliveryService` 와 거의 동일한 구조:
1. SO 헤더 조회.
2. 트랜잭션 번호 발급.
3. 헤더 `draft()`.
4. 라인 추가 (생성자 안에서 잔여 검증).
5. 자기 헤더 확정 (`issue()` / `ship()`).
6. SO 라인 누적 (`recordInvoicing` / `recordShipment`).
7. 저장.

→ **출하와 청구가 같은 패턴을 공유**. Phase 8 쯤 공통 베이스 (`TransactionEntity#postEntries(salesOrder)`) 추출도 고려할 수 있지만, 두 도메인의 차이 (수량만 vs 가격 복사 + 부가세) 가 작지 않아 아직은 두 벌이 명확.

## 🔥 시나리오 — 10대 수주 → 6+4 부분 출하·청구

```
[등록] SalesOrder.draft + addLine ×1 (orderQty=10)  → DRAFT
[확정] confirm()                                    → CONFIRMED
[출하1] DeliveryService.create({lines: [{sol, 6}]})
        ├ delivery.addLine(sol, 6)
        ├ delivery.ship()                          (DRAFT → SHIPPED for delivery)
        └ order.recordShipment(sol, 6)
           ├ sol.shippedQty = 6
           └ recomputeStatus → SHIPPING (6 < 10, anyShipped)
[청구1] InvoiceService.create({lines: [{sol, 6}]})
        ├ invoice.addLine(sol, 6)
        │   ├ remainingInvoiceable = 6 - 0 = 6 ✓
        │   └ unitPrice = 1_200_000 동결
        ├ invoice.issue()
        │   ├ subtotal = 7_200_000
        │   ├ taxAmount = 720_000
        │   └ totalAmount = 7_920_000
        └ order.recordInvoicing(sol, 6)
           ├ sol.invoicedQty = 6
           └ recomputeStatus → INVOICING (anyInvoiced, !allInvoiced)
[출하2] DeliveryService.create({lines: [{sol, 4}]})
        ├ delivery.addLine(sol, 4)  ✓ (잔여 출하 = 10 - 6 = 4)
        ├ delivery.ship()
        └ order.recordShipment(sol, 4)
           ├ sol.shippedQty = 10
           └ recomputeStatus → INVOICING (still anyInvoiced)
[청구2] InvoiceService.create({lines: [{sol, 4}]})
        ├ invoice.addLine(sol, 4)  ✓ (잔여 청구 = 10 - 6 = 4)
        ├ invoice.issue()  → subtotal 4_800_000, tax 480_000, total 5_280_000
        └ order.recordInvoicing(sol, 4)
           ├ sol.invoicedQty = 10
           └ recomputeStatus → INVOICED (allInvoiced)
```

상태 전이가 `CONFIRMED → SHIPPING → INVOICING → INVOICING → INVOICED` 로 매끄럽게 흘렀다. 어디서도 "현재 상태가 X 가 아니므로 거부" 분기에 막히지 않음 — `recomputeStatus()` 의 파생 상태 디자인 덕분.
