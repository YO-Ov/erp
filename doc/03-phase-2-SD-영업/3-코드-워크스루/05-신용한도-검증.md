# 5/7. 신용한도 검증 — `CreditLimitChecker` 와 외부 의존의 자리

> 도메인 메서드에 들어가지 못하는 규칙 — 다른 행을 조회해야 하는 검증을 어디에 둘 것인가.
> 설계서 §5.1 + §0의 핵심 결정 #9.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/sd/order/
├─ SalesOrderRepository.java          (sumActiveOrderAmountByCustomer JPQL)
├─ SalesOrderService.java             (confirm 흐름)
└─ creditcheck/
   └─ CreditLimitChecker.java         (외부 의존 묶음)
```

---

## 🔥 도메인 메서드에 못 두는 이유

`SalesOrder.confirm(...)` 안에서 신용한도 검사를 하지 않은 이유 — 자기 자신만으로 판단 불가하기 때문.

```
신용한도 검증 = (customer.creditLimit) - (이 고객의 다른 활성 수주 합계) >= 신규 수주 totalAmount
                          │                          │
                          │                          └─ 다른 SO 행들 SUM
                          └─ Customer 자기 컬럼만 보면 됨
```

다른 수주 합계는 **DB 조회가 필요**. 도메인 엔티티(`SalesOrder`)가 이 조회를 직접 한다면:
- `SalesOrder` 가 `SalesOrderRepository` 를 의존하게 됨 (DI 컨테이너 의존).
- 유닛 테스트가 mock 으로 무거워진다 (`mock(SalesOrderRepository.class)` ...).
- 엔티티 본분 — "자기 상태를 책임" — 이 흐려진다.

→ 외부 의존이 필요한 검증은 **서비스 단**에서 도메인 메서드 호출 직전에 처리.

## 🔥 `CreditLimitChecker` — 외부 의존 묶음

```java
@Component
@RequiredArgsConstructor
public class CreditLimitChecker {

    private final SalesOrderRepository repository;

    public void check(Customer customer, BigDecimal orderTotal, Long excludeOrderId) {
        if (customer.getStatus() != MasterStatus.ACTIVE) {
            throw new IllegalStateException("거래 불가 고객입니다 (status=" + customer.getStatus() + ").");
        }
        BigDecimal used = repository.sumActiveOrderAmountByCustomer(customer.getId(), excludeOrderId);
        if (used == null) used = BigDecimal.ZERO;
        BigDecimal remaining = customer.getCreditLimit().subtract(used);
        if (orderTotal.compareTo(remaining) > 0) {
            throw new IllegalStateException(
                    "신용한도 초과: 남은 한도 " + remaining + ", 요청 " + orderTotal);
        }
    }
}
```

### 왜 별도 클래스로 빼냈는가

`SalesOrderService.confirm(...)` 안에 인라인으로 한 검증을 두는 것도 가능하다. 그런데 별도 클래스로 빼면:

1. **재사용 가능성** — 추후 "수주 라인 추가 시점에 미리 한도 확인" 같은 기능이 생기면 같은 검증기를 호출.
2. **테스트 분리** — `CreditLimitChecker` 만 유닛 테스트 가능 (서비스 전체 컨텍스트 없이).
3. **이름이 의도를 전달** — `creditLimitChecker.check(...)` 가 무엇을 하는지 한 줄로 읽힌다.
4. **2개 규칙(고객 상태 + 한도)을 한 자리에 묶음** — 하나의 "거래 가능 여부" 검증.

→ 서비스가 비대해지지 않고, 규칙이 의미 단위로 응집.

### 패키지가 `sd/order/creditcheck/`

```
sd/order/
├─ SalesOrder.java
├─ SalesOrderService.java
├─ ...
└─ creditcheck/
   └─ CreditLimitChecker.java
```

`order` 의 하위 패키지로 둔다 — `creditcheck` 가 **수주 도메인의 보조** 임을 패키지 구조가 표현. 다른 도메인 (예: 결제) 의 신용 검사가 추가되면 그쪽 패키지로.

## 🔥 `sumActiveOrderAmountByCustomer` — JPQL 의 묘미

```java
@Query("""
    select coalesce(sum(o.totalAmount), 0)
      from SalesOrder o
     where o.customer.id = :customerId
       and o.status in (
            com.hwlee.erp.sd.order.SalesOrderStatus.CONFIRMED,
            com.hwlee.erp.sd.order.SalesOrderStatus.SHIPPING,
            com.hwlee.erp.sd.order.SalesOrderStatus.SHIPPED,
            com.hwlee.erp.sd.order.SalesOrderStatus.INVOICING,
            com.hwlee.erp.sd.order.SalesOrderStatus.INVOICED
       )
       and (:excludeId is null or o.id <> :excludeId)
    """)
BigDecimal sumActiveOrderAmountByCustomer(@Param("customerId") Long customerId,
                                          @Param("excludeId") Long excludeId);
```

### `coalesce(sum(o.totalAmount), 0)` — null 방어

`SUM` 은 매칭되는 행이 없으면 `null` 을 반환한다. 자바에서 받으면 `null BigDecimal`. 호출 측에서 `used.subtract(...)` 가 NPE.

`coalesce(SUM(...), 0)` 으로 DB 단에서 0 변환 → 자바에서 항상 `BigDecimal` 보장.

> ⚠️ `coalesce` 가 있어도 코드에 `if (used == null) used = BigDecimal.ZERO;` 한 줄을 더 둔 건 방어선 두 겹.

### `status in (...)` — "활성" 의 정의

설계서 §5.1: **"`CONFIRMED / SHIPPING / SHIPPED / INVOICING / INVOICED` 가 한도 차감 대상"**.

- `DRAFT`: 확정 전 — 한도에 영향 없음. **임시 입력은 한도 부담을 안 준다**.
- `CANCELLED`: 취소된 거래 — 한도 복원됨.
- `CLOSED`: 수금 완료 — 한도 복원됨.

→ "지금 회수해야 할 미수금이 있는 상태" 의 집합을 한 줄로 표현.

### enum 을 전체 경로로 명시

```jpql
o.status in (
    com.hwlee.erp.sd.order.SalesOrderStatus.CONFIRMED,
    ...
)
```

JPQL 에서 enum 을 쓰려면 **전체 경로** (FQN) 가 필요. import 없는 SQL 문맥이라.

대안으로 `@Param` 으로 `Set<SalesOrderStatus>` 를 넘기는 방법도 있지만, 호출 측 코드가 더 길어진다. **고정 집합** 이라서 쿼리에 박는 게 더 명확.

### `excludeId` — 자기 자신 제외

```jpql
and (:excludeId is null or o.id <> :excludeId)
```

`SalesOrderService.confirm(orderId)` 가 호출하는 시점에 현재 수주는 DRAFT (아직 CONFIRMED 아님) → SUM 결과에 포함 안 됨. **그런데** 다른 시나리오를 위해 안전망:
- 추후 "DRAFT 도 한도에 영향" 같은 정책이 도입되면 자기 자신을 빼야 함.
- 한 줄짜리 보험으로 미리 시그니처에 포함.

→ `Long excludeId` 가 null 이면 모든 행 포함, 값이 있으면 그 ID 제외.

## 🔥 `SalesOrderService.confirm()` — 검증 + 도메인 호출

```java
@Transactional
public SalesOrderResponse confirm(Long id) {
    SalesOrder order = getOrThrow(id);
    creditLimitChecker.check(order.getCustomer(), order.getTotalAmount(), order.getId());
    order.confirm(LocalDateTime.now(clock));
    return mapper.toResponse(order);
}
```

3줄짜리 흐름이지만 **순서가 의미** 있다:

1. 수주 조회 (없으면 404).
2. `creditLimitChecker.check(...)` — 통과 못 하면 `IllegalStateException` → 409.
3. `order.confirm(now)` — 자기 상태 검증 통과 후 DRAFT → CONFIRMED.

만약 2-3 순서가 뒤바뀌어 `order.confirm()` 이 먼저 호출되면:
- 신용한도 초과로 거부되어야 할 수주가 일단 CONFIRMED 로 전이됨.
- 그 후 한도 검증 실패로 트랜잭션 롤백 — DB 결과는 같지만, 도메인 메서드의 상태 전이가 의미 없이 일어남.

→ **검증이 통과한 후에만 상태를 전이**. 도메인 메서드의 책임이 "이미 검증된 상태 변경" 으로 단순해짐.

## 🔥 Phase 2 의 의도적 race condition

설계서 §0의 핵심 결정 #10: **"신용한도 동시 차감 race condition 은 의도적으로 노출 (락은 Phase 3에서)"**.

```
T1: customer.creditLimit = 1억, used = 8천 → remaining = 2천
T1: order.totalAmount = 1500 만 → 1500 ≤ 2000 OK
[T1 아직 commit 전]
T2: customer.creditLimit = 1억, used = 8천 (T1 미커밋이라 안 보임)
T2: → remaining = 2천
T2: order.totalAmount = 1500 만 → 1500 ≤ 2000 OK
T1 commit → used = 8 + 1500 = 9500
T2 commit → used = 8 + 1500 = 9500   (T1 의 +1500 이 반영 안 됨!)
실제 used = 11,000   ← 한도 1억을 초과한 상태로 둘 다 통과
```

이 시나리오는 **운영에서 실제로 일어나는 동시성 버그**. Phase 2 에서는 의도적으로 막지 않는다 — Phase 3 (MM 재고 차감) 에서 동시성 학습을 본격적으로 할 때 한 번에 다룬다.

대안 정도만 짚어둠:
- **비관적 락**: `Customer` 행을 `SELECT ... FOR UPDATE` 로 점유 후 검증/확정. 락 점유 시간이 길어 처리량 ↓.
- **낙관적 락** (`@Version`): Customer 엔티티에 버전 컬럼. 충돌 시 `OptimisticLockingFailureException` → 재시도. 트랜잭션 처리량 ↑ but 클라이언트가 재시도 로직 부담.
- **Application-level lock**: Redis 등 외부 락. 분산 환경에서 유효.

→ Phase 3 에서 재고 차감과 함께 본격 도입. 그때 이 신용한도 코드도 같이 손볼 예정.

## 🔥 활성/비활성 고객 처리

```java
if (customer.getStatus() != MasterStatus.ACTIVE) {
    throw new IllegalStateException("거래 불가 고객입니다 (status=" + customer.getStatus() + ").");
}
```

Phase 1 `Customer` 의 `MasterStatus`:
- `ACTIVE` — 정상 거래 가능
- `INACTIVE` — 휴면. 새 거래 등록 막음
- `BLOCKED` — 거래 정지 (신용 불량, 계약 종료 등)

→ INACTIVE/BLOCKED 고객의 수주 확정 시도는 거부. 단, **DRAFT 단계까지는 만들 수 있다** (확정만 막힘). 영업이 "이 고객이 거래 가능해지면 바로 확정하자" 같은 사전 작업을 허용.

### 왜 `confirm` 에서만 검증

`Customer.create` 단계에서는 ACTIVE 체크가 없다. 신용한도도 안 본다. 이유:
- **DRAFT 는 가벼운 임시 작업**. 영업이 견적 시뮬레이션처럼 자유롭게 등록.
- 실제로 부담 (다른 SO 합계에 들어감) 이 발생하는 시점은 **`CONFIRMED` 이후** — 그때만 검증.

→ "강한 검증은 강한 상태 전이 직전에" 라는 일관된 패턴.

## 🔥 시나리오 — 한도 초과 거부

```
[등록] DRAFT 수주 만들기 — 한도 검증 없음
  POST /api/sales-orders { customerId: 1, lines: [...total 8천만] }
  → 201 Created, status=DRAFT, totalAmount=80,000,000
[확정 시도] POST /api/sales-orders/{id}/confirm
   │
   ▼
[SalesOrderService.confirm(id)]
   │
   ├─ getOrThrow(id) → SalesOrder { totalAmount: 80,000,000 }
   │
   ├─ creditLimitChecker.check(customer, 80,000,000, id)
   │    │
   │    ├─ customer.status = ACTIVE → 통과
   │    │
   │    ├─ used = sumActiveOrderAmountByCustomer(customer.id, exclude=this)
   │    │    SQL: SELECT COALESCE(SUM(total_amount), 0)
   │    │           FROM sales_order
   │    │          WHERE customer_id = 1
   │    │            AND status IN ('CONFIRMED','SHIPPING',...,'INVOICED')
   │    │            AND id <> 42
   │    │    → 50,000,000
   │    │
   │    ├─ remaining = 100,000,000 - 50,000,000 = 50,000,000
   │    │
   │    └─ 80,000,000 > 50,000,000 → throw IllegalStateException
   │         "신용한도 초과: 남은 한도 50000000, 요청 80000000"
   │
   ▼
[GlobalExceptionHandler]
   → 409 Conflict
   { title: "Conflict", detail: "신용한도 초과: ..." }
```

상태는 그대로 DRAFT. 영업이 라인 수량을 줄이거나 다른 수주를 결제 받아 한도를 회복하면 다시 confirm 가능.
