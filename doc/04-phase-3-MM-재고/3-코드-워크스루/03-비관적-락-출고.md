# 3/7. 비관적 락 출고 — `findForUpdate` 와 race 의 원천 차단

> 같은 (item, warehouse) 행을 동시에 차감하려는 두 트랜잭션을 **DB 차원에서 줄 세운다**.
> 설계서 §3.1 + §0의 핵심 결정 #5 "출고 경로는 비관적 락 PESSIMISTIC_WRITE" 의 실제 구현.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/mm/
├─ stock/
│  ├─ Stock.java                    (issue 도메인 메서드)
│  ├─ StockRepository.java          (findForUpdate)
│  └─ InsufficientStockException.java
├─ goodsissue/
│  ├─ GoodsIssue.java               (DRAFT → POSTED 도메인 메서드)
│  └─ GoodsIssueService.java        (post — 락 + 검증 + 차감 + 원장)
└─ common/error/
   └─ GlobalExceptionHandler.java   (INSUFFICIENT_STOCK 매핑)
```

## 🔥 문제 — 락 없으면 어떻게 깨지는가

```
초기 상태: stock.qty_on_hand = 10

스레드 A (출고 6대 처리)            스레드 B (출고 7대 처리)
────────────────────────────────────────────────────────────
SELECT qty FROM stock WHERE ...     (qty=10)
                                    SELECT qty FROM stock WHERE ...  (qty=10)
6 ≤ 10 → 검증 통과                  7 ≤ 10 → 검증 통과
qty - 6 = 4
UPDATE qty = 4
                                    qty - 7 = 3        ← 10에서 뺐다고 생각!
                                    UPDATE qty = 3      ← A의 4를 덮어쓰기

결과: 6 + 7 = 13대를 보냈는데 재고는 3.
이론상 4대가 더 빠진 셈. 음수 재고가 안 돼야 하는데 음수와 같은 효과.
```

이게 **운영에서 매일 일어나는 동시성 버그의 전형**. Stock 잔액 같은 핵심 불변식이 걸린 자원은 보호 안 하면 반드시 깨진다.

→ Phase 3 의 첫 학습 주제. 두 가지 방향:
1. **비관적 락** — "내가 쓸 거니까 다른 사람은 기다려." 충돌 자체를 막음.
2. **낙관적 락** — "충돌은 드물 것. 일단 진행하고 충돌나면 한쪽 실패 + 재시도."

출고 경로는 **비관적 락**, 입고는 **낙관적 락**. 다음 편 (04) 에서 입고 쪽 다루고, 이 편은 출고 = 비관적 락에 집중.

## 🔥 `findForUpdate` — Spring Data JPA 의 `@Lock(PESSIMISTIC_WRITE)`

```java
public interface StockRepository
        extends JpaRepository<Stock, Long>, JpaSpecificationExecutor<Stock> {

    Optional<Stock> findByItemIdAndWarehouseId(Long itemId, Long warehouseId);

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("select s from Stock s where s.item.id = :itemId and s.warehouse.id = :warehouseId")
    Optional<Stock> findForUpdate(@Param("itemId") Long itemId,
                                  @Param("warehouseId") Long warehouseId);
}
```

`@Lock(LockModeType.PESSIMISTIC_WRITE)` → Hibernate 가 `SELECT ... FOR UPDATE` 로 SQL 을 발행. DB 가 그 행에 **쓰기 락** 을 건다.

같은 행을 다른 트랜잭션이 `SELECT ... FOR UPDATE` 하려고 하면 **대기** (락 풀릴 때까지). 그러므로 race condition 자체가 발생할 수 없다.

### 두 가지 메서드를 같이 두는 이유

- `findByItemIdAndWarehouseId(...)` — **락 없는 단순 조회**. 화면이 "재고 얼마 남았어요" 묻거나, 입고 경로처럼 낙관 락만으로 충분한 곳에서 사용.
- `findForUpdate(...)` — **비관 락**. 출고 경로의 핵심 보호.

같은 행을 다른 방식으로 가져갈 수 있게 한다. **호출 측이 의도를 명확히 표시** — 메서드 이름이 곧 락 모드.

> 💡 Spring Data JPA 의 메서드명 컨벤션상 `findForUpdate` 같은 이름은 자동 파생이 안 된다. 그래서 `@Query` JPQL 을 직접 적었다. JPQL 명시 + `@Lock` 을 같이 쓰는 게 표준 패턴.

## 🔥 `GoodsIssueService.post` — 락 → 검증 → 차감 → 원장의 한 흐름

```java
@Transactional
public GoodsIssueResponse post(Long id) {
    GoodsIssue gi = getOrThrow(id);
    gi.post(LocalDateTime.now(clock));               // DRAFT → POSTED (자기 상태 검증)
    Warehouse warehouse = gi.getWarehouse();
    LocalDateTime now = LocalDateTime.now(clock);

    for (GoodsIssueLine line : gi.getLines()) {
        Item item = line.getItem();

        // (1) 비관 락 점유 — 같은 행을 다른 트랜잭션이 SELECT FOR UPDATE 하려고 하면 대기
        Stock stock = stockRepository
                .findForUpdate(item.getId(), warehouse.getId())
                .orElseThrow(() -> new EntityNotFoundException(
                        "Stock not found: item=" + item.getId() + ", warehouse=" + warehouse.getId()
                                + " — 출고 전에 입고가 있어야 합니다."));

        // (2) 가용 검증 + 차감 (Stock.issue 내부)
        BigDecimal appliedCost = stock.issue(line.getQuantity());

        // (3) 원장에 음수 적재
        stockMovementRepository.save(StockMovement.of(
                item, warehouse, line.getQuantity().negate(), appliedCost,
                MovementReason.GOODS_ISSUE, "GI", gi.getId(), now));
    }
    return mapper.toResponse(gi);
}
```

각 단계의 의미:

1. **`findForUpdate(itemId, warehouseId)`** — 행 락 점유. 락은 트랜잭션 종료 (COMMIT 또는 ROLLBACK) 시까지 보유.
2. **`stock.issue(qty)`** — 도메인 메서드 안에서 가용 검증 후 차감 + 적용 단가 반환. 부족하면 `InsufficientStockException` → 트랜잭션 롤백 + 락 해제.
3. **`StockMovement.of(...)`** — 음수 행 (`qty.negate()`) 으로 원장 적재. `reason=GOODS_ISSUE`, 약한 참조 `(GI, gi.id)`.

같은 트랜잭션 안에서 3개 작업이 묶임. 락은 1단계에서 잡고, 트랜잭션 끝나는 4단계 (메서드 종료 후 자동 commit) 에서 풀린다.

## 🔥 `Stock.issue()` — 가용 검증의 마지막 보루

```java
public BigDecimal issue(BigDecimal qty) {
    if (qty == null || qty.signum() <= 0)
        throw new IllegalArgumentException("출고 수량은 0보다 커야 한다.");
    if (this.qtyOnHand.compareTo(qty) < 0) {
        throw new InsufficientStockException(
                item != null ? item.getId() : null,
                warehouse != null ? warehouse.getId() : null,
                this.qtyOnHand, qty);
    }
    BigDecimal applied = this.averageCost;
    this.qtyOnHand = this.qtyOnHand.subtract(qty);
    return applied;
}
```

도메인 메서드가 검증을 책임진다. 서비스가 "락 잡고 객체 가져왔으니 안전" 이라고 가정하지 않고, **객체 자기 자신** 이 자기 불변식을 강제. 락이 깨졌거나 (운영자가 직접 DB 조작 등) 사람이 잘못 호출해도 도메인이 막아준다.

`InsufficientStockException` 은 비즈니스 의미를 가진 도메인 예외. 일반 `IllegalStateException` 보다 강한 의미:

```java
public class InsufficientStockException extends RuntimeException {
    private final Long itemId;
    private final Long warehouseId;
    private final BigDecimal available;
    private final BigDecimal requested;
    ...
}
```

→ 어디서 얼마가 부족한지 4개 정보를 함께 들고 다님. `GlobalExceptionHandler` 가 이걸 모두 응답 본문에 박는다.

## 🔥 `InsufficientStockException` → 409 + 상세 속성

```java
@ExceptionHandler(InsufficientStockException.class)
public ProblemDetail handleInsufficientStock(InsufficientStockException ex) {
    ProblemDetail problem = ProblemDetail.forStatusAndDetail(HttpStatus.CONFLICT, ex.getMessage());
    problem.setTitle("Insufficient Stock");
    problem.setProperty("code", "INSUFFICIENT_STOCK");
    problem.setProperty("itemId", ex.getItemId());
    problem.setProperty("warehouseId", ex.getWarehouseId());
    problem.setProperty("available", ex.getAvailable());
    problem.setProperty("requested", ex.getRequested());
    return problem;
}
```

응답 예:

```json
{
  "type": "about:blank",
  "title": "Insufficient Stock",
  "status": 409,
  "detail": "재고 부족: itemId=1, warehouseId=1, available=5, requested=7",
  "code": "INSUFFICIENT_STOCK",
  "itemId": 1,
  "warehouseId": 1,
  "available": 5,
  "requested": 7
}
```

- **상태 코드 409 Conflict** — "자원은 있는데 현재 상태에서 그 동작 거부" (Phase 2 신용한도와 같은 의미).
- **`code` 속성** — 클라이언트가 메시지 문자열 파싱이 아니라 안정적 키로 분기 가능.
- **`available / requested`** — 화면이 "5개 있는데 7개 요청하셨어요. 5개로 줄이시겠어요?" 같은 UX 를 만들 수 있도록.

→ Phase 2 의 신용한도 메시지가 "남은 한도 X, 요청 Y" 처럼 변수 두 개를 한 응답에 박은 패턴과 동일.

## 🔥 락이 잡힌 동안 다른 트랜잭션은?

```
시점         T1 (출고 6대)                     T2 (출고 7대)
──────────────────────────────────────────────────────────────
t1   BEGIN
t2   findForUpdate(itemId, warehouseId)
     → 락 점유, qty=10 조회
t3                                       BEGIN
t4                                       findForUpdate(...)
                                         ⏸ 대기 (T1 의 락 풀릴 때까지)
t5   stock.issue(6) → qty=4
t6   StockMovement.save(-6)
t7   COMMIT → 락 해제
                                       (대기 해제)
t8                                       findForUpdate(...) → qty=4 조회 (T1 결과 반영)
t9                                       stock.issue(7) → qtyOnHand(4) < 7
                                         → InsufficientStockException
                                         → ROLLBACK → 락 해제
                                         → 409 INSUFFICIENT_STOCK
```

T2 는 **T1 의 결과를 보고** 검증한다. "10에서 뺐다고 착각" 같은 일이 안 일어남. 둘 다 통과해서 음수 재고가 되는 일도 없음.

### 대가

- T2 가 `t4 ~ t7` 동안 **대기**. 트랜잭션이 길수록 대기가 길어진다. 처리량 ↓.
- **데드락 가능성** — T1 이 행 A 잡고 B 기다리는데, T2 가 B 잡고 A 기다리면 둘이 영원히 대기. DB 가 감지해서 한쪽을 죽인다 (`LockAcquisitionException`).

Phase 3 의 출고 트랜잭션은 짧다 (락 → 검증 → 차감 → 원장 → commit, 보통 10ms 이내). 데드락도 한 행만 잡으므로 발생 가능성 낮음. → 비관 락의 대가가 작아 적합.

## 🔥 `GoodsIssue.post()` — 자기 상태 검증의 분리

```java
public void post(LocalDateTime now) {
    if (status != GoodsIssueStatus.DRAFT)
        throw new IllegalStateException("DRAFT 출고만 확정 가능합니다. 현재: " + status);
    if (lines.isEmpty())
        throw new IllegalStateException("라인이 비어 있는 출고는 확정할 수 없습니다.");
    this.status = GoodsIssueStatus.POSTED;
    this.postedAt = now;
}
```

GoodsIssue 자기 상태 검증 (`DRAFT → POSTED`) 은 도메인 메서드. **Stock 차감은 안 함** — 외부 의존(Stock 락) 이라 서비스 단으로.

Phase 2 의 `SalesOrder.confirm(now)` 와 같은 패턴:
- 자기 상태만 검증하는 책임은 엔티티 안에.
- 외부 자원 (DB 다른 행) 이 필요한 책임은 서비스 안에.

이 분리가 깨끗하면 단위 테스트가 가능해진다:

```java
// GoodsIssueTest — 외부 의존 0개
GoodsIssue gi = GoodsIssue.draft(...);
gi.addLine(item, qty);
gi.post(now);
assertThat(gi.getStatus()).isEqualTo(POSTED);
```

vs Stock 검증은 통합 테스트 (Testcontainers 로 실제 DB) 가 필요.

## 🔥 trans 흐름의 시각화

```
[클라이언트]
POST /api/goods-issues/{id}/post
   │
   ▼
[GoodsIssueController.post]
   └─ service.post(id)
        │
        ▼
[GoodsIssueService.post] ← @Transactional 시작 (BEGIN)
   │
   ├─ gi = repository.findById(id) ← GoodsIssue 헤더 + 라인 LAZY
   │
   ├─ gi.post(now) ← DRAFT → POSTED, 라인 검증
   │
   ├─ for each line:
   │   ├─ stock = stockRepo.findForUpdate(itemId, warehouseId)
   │   │     SQL: SELECT s.* FROM stock s
   │   │           WHERE s.item_id = ? AND s.warehouse_id = ?
   │   │           FOR UPDATE        ← ⭐ 락 점유
   │   │
   │   ├─ appliedCost = stock.issue(qty)
   │   │     ├─ qtyOnHand < qty ? throw InsufficientStockException
   │   │     └─ qtyOnHand -= qty; return averageCost
   │   │
   │   └─ stockMovementRepo.save(StockMovement.of(
   │           item, warehouse, qty.negate(), appliedCost,
   │           GOODS_ISSUE, "GI", gi.id, now))
   │
   └─ return mapper.toResponse(gi)
        ↓
   @Transactional 종료 (COMMIT) ← ⭐ 락 해제, 영속성 컨텍스트 flush
        ↓
   └─ 200 OK + GoodsIssueResponse

(가용 부족 시)
   InsufficientStockException 던짐
        ↓
   @Transactional ROLLBACK ← Stock 변경, StockMovement save 모두 취소
        ↓
   GlobalExceptionHandler 가 잡음
        ↓
   409 Conflict + { code: INSUFFICIENT_STOCK, available, requested, ... }
```

→ **락 시간 = `findForUpdate` 호출 ~ COMMIT/ROLLBACK** 의 짧은 구간. 그 안에서 모든 결정이 이뤄짐.

## 🔥 출고 취소도 같은 락

```java
@Transactional
public GoodsIssueResponse cancel(Long id) {
    GoodsIssue gi = getOrThrow(id);
    gi.cancel();   // POSTED → CANCELLED 자기 검증
    Warehouse warehouse = gi.getWarehouse();
    LocalDateTime now = LocalDateTime.now(clock);

    for (GoodsIssueLine line : gi.getLines()) {
        Item item = line.getItem();
        Stock stock = stockRepository
                .findForUpdate(item.getId(), warehouse.getId())   // ← 역시 비관 락
                .orElseThrow(...);
        BigDecimal appliedCost = stock.getAverageCost();
        stock.cancelIssue(line.getQuantity());                    // qty 만 복원

        stockMovementRepository.save(StockMovement.of(
                item, warehouse, line.getQuantity(), appliedCost,
                MovementReason.ADJUSTMENT_PLUS, "GI", gi.getId(), now));
    }
    return mapper.toResponse(gi);
}
```

취소는 수량을 다시 더하는 것 — 새 출고와의 race 가 가능하므로 마찬가지로 비관 락. `MovementReason.ADJUSTMENT_PLUS` 로 원장에 양수 행 추가 (원래 GOODS_ISSUE 행을 수정하지 않고 새 행으로 정정 — append-only).

## 🔥 음수 재고는 두 겹으로 막힘

1. **도메인 메서드** (`Stock.issue`) — `qtyOnHand < qty` 이면 `InsufficientStockException`.
2. **DB CHECK 제약** — `CHECK (qty_on_hand >= 0)`. 도메인 검증을 우회해 직접 SQL UPDATE 해도 거부.

두 겹 방어. 도메인이 일하면 (정상 흐름) CHECK 까지 갈 일이 없고, 코드 버그/직접 DB 조작 등 비정상 경로에서 CHECK 가 마지막 보루.

> 💡 DB CHECK 위반은 `DataIntegrityViolationException` → 기존 핸들러가 409 로 변환. 사용자 경험은 같음.
