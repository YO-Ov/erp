# 4/7. 낙관적 락 입고 — `@Version` 과 두 락의 공존

> 충돌이 드문 경로엔 가볍게, 잦은 경로엔 강하게 — **한 도메인에 두 락 패턴을 한 번에**.
> 설계서 §3.2 + §0의 핵심 결정 #4, #6, #7.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/mm/
├─ stock/
│  └─ Stock.java                       (@Version)
├─ goodsreceipt/
│  ├─ GoodsReceipt.java
│  └─ GoodsReceiptService.java         (post — 락 없는 조회 + receive + 원장)
└─ common/error/
   └─ GlobalExceptionHandler.java      (OPTIMISTIC_LOCK_CONFLICT 매핑)
```

---

## 🔥 입고는 왜 비관 락이 아닌가

출고는 "재고 부족" 이라는 강한 불변식이 걸려 race 가 곧 사고. 입고는 다른 성격:

- "10대 보낸 거 받았어요" 는 **충돌이 드물다** — 같은 (item, warehouse) 에 두 입고가 정확히 같은 순간에 들어오는 경우가 자주 없음.
- 충돌이 나도 **결과의 의미가 단순** — 두 입고가 합쳐져서 보유량이 늘어나는 것. 한 번에 한 트랜잭션씩 처리되면 그만.
- **재시도가 자연스럽다** — "입고 처리 실패, 다시 시도하세요" 가 사용자 입장에서 받아들일 수 있음 (vs 출고 거부는 비즈니스 영향이 큼).

→ 비관 락의 대가 (대기 시간, 데드락 위험) 를 치를 가치가 없다. **낙관 락** 으로 가볍게 — 충돌 시 한쪽이 실패하고 재시도하는 패턴.

---

## 🔥 `@Version` — JPA 가 자동으로 거는 보호

```java
@Entity
public class Stock extends BaseEntity {
    ...
    private BigDecimal qtyOnHand = BigDecimal.ZERO;
    private BigDecimal averageCost = BigDecimal.ZERO;

    @Version
    @Column(name = "version", nullable = false)
    private Long version;
    ...
}
```

`@Version` 어노테이션 한 줄로 끝. JPA/Hibernate 가 자동으로:

1. UPDATE 시점에 `WHERE id = ? AND version = ?` 조건을 SQL 에 추가.
2. UPDATE 가 0 row affected 면 (다른 트랜잭션이 먼저 갱신) `OptimisticLockException` 발생.
3. 성공 시 `version` 을 +1.

### 실제 SQL

```sql
-- 영속성 컨텍스트의 flush 시점
UPDATE stock
   SET qty_on_hand = ?,
       average_cost = ?,
       version = version + 1,
       updated_at = ?
 WHERE id = ?
   AND version = ?     ← ⭐ 낙관 락의 핵심
```

- 처음 조회 시 `version = 5` 였으면, UPDATE 의 WHERE 절도 `version = 5`.
- 그 사이 다른 트랜잭션이 같은 행을 갱신 (version=6 으로) 했으면 → UPDATE 0 row → `OptimisticLockException`.
- 충돌 없으면 → version = 6 으로 갱신 + UPDATE 1 row.

> 💡 `@Version` 컬럼 타입은 `Long`, `Integer`, `int`, `long`, `short`, `Timestamp` 중 하나여야 한다. 우리는 `Long` (nullable 가능, 일관성).

## 🔥 `GoodsReceiptService.post` — 락 없는 조회 + receive + 원장

```java
@Transactional
public GoodsReceiptResponse post(Long id) {
    GoodsReceipt gr = getOrThrow(id);
    gr.post(LocalDateTime.now(clock));                  // DRAFT → POSTED
    Warehouse warehouse = gr.getWarehouse();
    LocalDateTime now = LocalDateTime.now(clock);

    for (GoodsReceiptLine line : gr.getLines()) {
        Item item = line.getItem();

        // (1) 락 없는 조회 — 없으면 새 행 INSERT
        Stock stock = stockRepository
                .findByItemIdAndWarehouseId(item.getId(), warehouse.getId())
                .orElseGet(() -> stockRepository.save(Stock.empty(item, warehouse)));

        // (2) 가중평균 갱신 + 수량 누적 (Stock.receive 내부)
        stock.receive(line.getQuantity(), line.getUnitCost());

        // (3) 원장에 양수 적재
        stockMovementRepository.save(StockMovement.of(
                item, warehouse, line.getQuantity(), line.getUnitCost(),
                MovementReason.GOODS_RECEIPT, "GR", gr.getId(), now));
    }
    return mapper.toResponse(gr);
}
```

출고(`GoodsIssueService.post`) 와의 차이는 **(1) 단계** 한 줄:
- 출고: `findForUpdate(...)` (비관 락)
- 입고: `findByItemIdAndWarehouseId(...)` (락 없음)

그 외 흐름은 거의 동일. 가중평균 공식은 다음 편(05) 에서 자세히.

### `orElseGet` — 첫 입고 시 행 생성

```java
.orElseGet(() -> stockRepository.save(Stock.empty(item, warehouse)));
```

(item, warehouse) 행이 없으면 (`Optional.empty`) `Stock.empty(...)` 로 새로 만들고 저장. **이 INSERT 가 동시에 두 번 일어나면?** — UNIQUE 제약 (`uk_stock_item_warehouse`) 이 한쪽을 거부 → `DataIntegrityViolationException` → 트랜잭션 롤백 → 재시도 시 이미 행이 있으므로 정상 흐름.

→ DB 의 UNIQUE 가 첫 행 만들기 race 의 안전망. CodeGenerator 의 `createOrLockExisting` 패턴 (Phase 1) 과 동일한 사고.

## 🔥 충돌 시나리오 — 동시 입고

```
초기: stock.qty_on_hand=10, average_cost=1000, version=5

스레드 A (입고 5대 @ 1200)            스레드 B (입고 5대 @ 1100)
─────────────────────────────────────────────────────────────────
SELECT * FROM stock                  (qty=10, avg=1000, version=5)
                                     SELECT * FROM stock  (qty=10, avg=1000, version=5)
stock.receive(5, 1200)
  → qty=15, avg=(10×1000+5×1200)/15
       =1066.67
                                     stock.receive(5, 1100)
                                       → qty=15, avg=(10×1000+5×1100)/15
                                            =1033.33
(영속성 컨텍스트 flush)
UPDATE stock SET qty=15, avg=1066.67,
    version=6 WHERE id=1 AND version=5
→ 1 row → 성공, COMMIT
                                     (영속성 컨텍스트 flush)
                                     UPDATE stock SET qty=15, avg=1033.33,
                                         version=6 WHERE id=1 AND version=5
                                     → 0 row (version 은 이제 6) → ⚠️
                                     OptimisticLockException
                                     → ROLLBACK
                                     → 409 OPTIMISTIC_LOCK_CONFLICT
```

T2 는 실패. 클라이언트가 재시도하면:

```
SELECT * FROM stock                  (qty=15, avg=1066.67, version=6)
stock.receive(5, 1100)
  → qty=20, avg=(15×1066.67+5×1100)/20
       =1075.00
UPDATE ... WHERE version=6 → 1 row → 성공
```

→ 두 입고가 **순차 처리** 된 효과. 결과는 비관 락과 똑같다.

**차이는 처리 방식**:
- 비관 락: T2 가 처음부터 대기 (T1 commit 후 차감).
- 낙관 락: T2 가 일단 시도해보고, 충돌나면 재시도.

비관 락은 T2 가 한 번에 끝나지만 대기 시간이 있음. 낙관 락은 T2 가 두 번 일하지만 (실패+재시도) 대기는 없음. **둘 다 같은 정합성을 보장** — 선택은 충돌 빈도와 트랜잭션 길이에 달림.

## 🔥 `OptimisticLockException` → 409 + 재시도 안내

```java
@ExceptionHandler({OptimisticLockException.class, OptimisticLockingFailureException.class})
public ProblemDetail handleOptimisticLock(Exception ex) {
    log.warn("Optimistic lock conflict", ex);
    ProblemDetail problem = ProblemDetail.forStatusAndDetail(HttpStatus.CONFLICT,
            "동시 수정 충돌 — 다른 트랜잭션이 먼저 갱신했습니다. 재시도해 주세요.");
    problem.setTitle("Optimistic Lock Conflict");
    problem.setProperty("code", "OPTIMISTIC_LOCK_CONFLICT");
    return problem;
}
```

### 두 예외 클래스를 같이 잡는 이유

- `jakarta.persistence.OptimisticLockException` — JPA 표준 예외.
- `org.springframework.dao.OptimisticLockingFailureException` — Spring 의 DAO 추상화.

Spring 의 `@Transactional` 이 두 예외 모두 던질 수 있음. 환경에 따라 한쪽으로 통일되지 않아서 양쪽 다 매핑.

### 응답 메시지에 "재시도해 주세요"

낙관 락 충돌은 **재시도가 정답**. 의미적으로 잘못된 게 아니라 "타이밍이 안 좋았어" 인 상황. 클라이언트가 같은 요청을 다시 보내면 거의 성공한다.

→ Phase 9 (배치) 에서는 `@Retryable` 같은 어노테이션으로 서버 측 자동 재시도도 도입 가능. Phase 3 에서는 클라이언트 재시도로 단순화.

## 🔥 두 락이 한 행을 두고 만났을 때

```
시점         T1 (출고 6대, 비관 락)              T2 (입고 5대 @ 1200, 낙관 락)
──────────────────────────────────────────────────────────────────────
t1   findForUpdate(...) → 락 점유
     qty=10, avg=1100, version=5
t2                                       findByItemIdAndWarehouseId(...)
                                         (락 없는 조회)
                                         qty=10, avg=1100, version=5
t3   stock.issue(6) → qty=4
                                         stock.receive(5, 1200)
                                           qty=15, avg=(10×1100+5×1200)/15
                                                =1133.33
                                         (영속성 컨텍스트 flush 시점)
                                         UPDATE ... WHERE id=1 AND version=5
                                         ⏸ 대기 (T1 의 X-lock 이 막음)
t4   UPDATE stock SET qty=4, version=6
     WHERE id=1 AND version=5 → 1 row
t5   COMMIT → 락 해제
                                       (대기 해제)
                                         UPDATE ... → 0 row (version 은 이제 6)
                                         → OptimisticLockException
                                         → ROLLBACK
                                         → 409 OPTIMISTIC_LOCK_CONFLICT
```

비관 락이 잡은 동안 낙관 락 측의 UPDATE 가 **대기**. 락 해제되면 UPDATE 시도 — but `version` 이 이미 바뀌었으니 0 row → 충돌 검출 → 재시도.

→ **DB 의 행 락이 두 패턴 사이의 직렬화도 보장**. 한쪽이 비관, 한쪽이 낙관이어도 정합성 안 깨진다. 두 락의 공존 가능.

## 🔥 GoodsReceipt 의 도메인 모양

```java
@Entity
public class GoodsReceipt extends BaseEntity {
    private String number;            // GR-YYYYMMDD-NNN
    private Vendor vendor;            // 누구에게 받았는가
    private Warehouse warehouse;      // 어느 창고로
    private GoodsReceiptStatus status; // DRAFT/POSTED/CANCELLED
    private LocalDate receiptDate;
    private LocalDateTime postedAt;
    private List<GoodsReceiptLine> lines;
    ...

    public void post(LocalDateTime now) {
        if (status != GoodsReceiptStatus.DRAFT)
            throw new IllegalStateException("DRAFT 입고만 확정 가능합니다. 현재: " + status);
        if (lines.isEmpty())
            throw new IllegalStateException("라인이 비어 있는 입고는 확정할 수 없습니다.");
        this.status = GoodsReceiptStatus.POSTED;
        this.postedAt = now;
    }

    public void cancel() {
        if (status != GoodsReceiptStatus.POSTED)
            throw new IllegalStateException("POSTED 입고만 취소 가능합니다. 현재: " + status);
        this.status = GoodsReceiptStatus.CANCELLED;
    }
    ...
}
```

GoodsIssue 와 거의 동형 — 다른 점:
- `vendor` 필드 (입고는 매입처 정보가 의미 있음, 출고는 없음 — Phase 3 단순화).
- `reason` 필드 없음 (입고는 항상 "매입 입고" 한 종류, 출고는 SHIPMENT/ADJUSTMENT/SCRAP 다양).

상태 전이는 정확히 같은 패턴: `DRAFT → POSTED → CANCELLED`.

### 라인 컬럼 차이

```java
@Entity
public class GoodsReceiptLine extends BaseEntity {
    private GoodsReceipt goodsReceipt;
    private int lineNo;
    private Item item;
    private BigDecimal quantity;
    private BigDecimal unitCost;         // ← 입고만 있음
    private BigDecimal lineTotal;        // qty × unitCost
}

@Entity
public class GoodsIssueLine extends BaseEntity {
    private GoodsIssue goodsIssue;
    private int lineNo;
    private Item item;
    private BigDecimal quantity;
    // unit_cost 없음 — 출고는 Stock 의 평균 단가를 적용
    // line_total 없음 — 매출 인식은 Phase 4 의 Invoice 가 책임
}
```

입고 라인은 단가를 가지고 있음 (매입가가 라인마다 다를 수 있고, 가중평균 계산의 입력). 출고 라인은 단가를 안 가짐 — Stock 의 평균 단가를 그대로 적용해 StockMovement.unit_cost 에 박는다.

## 🔥 입고 취소도 비관 락 (예외)

```java
@Transactional
public GoodsReceiptResponse cancel(Long id) {
    GoodsReceipt gr = getOrThrow(id);
    gr.cancel();
    Warehouse warehouse = gr.getWarehouse();
    LocalDateTime now = LocalDateTime.now(clock);

    for (GoodsReceiptLine line : gr.getLines()) {
        Item item = line.getItem();
        // 취소는 비관 락 — 그 사이 출고로 부족해질 수 있어 가용 검증이 필요
        Stock stock = stockRepository
                .findForUpdate(item.getId(), warehouse.getId())
                .orElseThrow(...);
        BigDecimal appliedCost = stock.cancelReceipt(line.getQuantity());

        stockMovementRepository.save(StockMovement.of(
                item, warehouse, line.getQuantity().negate(), appliedCost,
                MovementReason.ADJUSTMENT_MINUS, "GR", gr.getId(), now));
    }
    return mapper.toResponse(gr);
}
```

`cancelReceipt(qty)` 내부에서 가용 검증 (수량을 빼는 것이므로 `Stock.issue` 와 동일 로직). 그 사이 다른 출고로 이미 빠져나가 부족하면 `InsufficientStockException` → 409.

→ **수량을 차감하는 경로는 모두 비관 락**. 입고 정상 흐름 (양수 누적) 만 낙관 락.

## 🔥 락 모드 선택의 의사결정 트리

```
                  같은 행을 동시에 바꾸는가?
                          │
              ┌───────────┴────────────┐
              아니오                    예
              │                        │
        락 필요 없음             충돌이 잦은가?
        (드물게 일어남)                │
                       ┌───────────────┴────────────┐
                       자주 (매분)                    드물게 (매시간 이하)
                       │                              │
                  비관적 락                       낙관적 락
                  - 대기 짧음                    - 대기 0
                  - 데드락 가능                  - 충돌 시 재시도
                  - 처리량 ↓                    - 처리량 ↑
                                                - 클라이언트 재시도 로직 필요

                       강한 불변식 (qty>=0)?
                       │
              ┌────────┴─────────┐
              예                  아니오
              │                  │
         비관 락 필수          낙관 락 충분
         (Stock.issue)         (단순 누적)
```

Phase 3 의 선택:
- **출고 (Stock.issue)**: 강한 불변식 + 충돌 가능성 ↑ → **비관 락**.
- **입고 (Stock.receive)**: 단순 누적, 충돌 드묾 → **낙관 락**.
- **취소 (수량 차감 경로)**: 가용 검증 필요 → **비관 락**.

→ 한 도메인에서 두 패턴이 자연스럽게 공존. "어디서 무엇을 쓰는가" 의 판단 기준을 직접 손에 잡는 것이 Phase 3 학습의 핵심 (설계서 §3.4).
