# 5/7. 이동평균 원가 — `Stock.receive` 의 공식과 그 한계

> 입고할 때마다 평균을 다시 계산, 출고는 그 평균을 그대로 사용.
> 설계서 §4 + §0의 핵심 결정 #9, #10 — Phase 5 (FI 회계) 매출원가의 기초.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/mm/stock/
└─ Stock.java
       ├─ receive(qty, unitCost)        ← 가중평균 갱신
       ├─ issue(qty)                    ← 평균 그대로
       ├─ cancelReceipt(qty)            ← 평균 그대로 (의도된 단순화)
       └─ cancelIssue(qty)              ← 평균 그대로
```

## 🔥 왜 평균을 추적하나 — 4가지 이유

공식 보기 전에 한 발 물러서서. **재고 수량만 들면 안 되나?** 안 된다. 회계는 항상 **수량 + 금액** 한 쌍으로 굴러간다.

### 1. 재고자산 금액 — 재무제표가 요구

재무제표의 "재고자산" 칸 = `수량 × 단가`. 단가 없이는 빈칸. Stock 의 `average_cost × qty_on_hand` 가 곧 그 행의 장부 가치.

### 2. 매출원가 (COGS) — 매 매출의 짝

회계 원칙(수익-비용 대응): 매출이 일어나는 순간 **그 매출에 대응되는 원가** 가 같이 잡혀야 한다.

```
노트북 100만원에 사서 150만원에 팔면
  → 매출 150  -  매출원가 100  =  이익 50
                           ↑
               "원가 100" 이 어디서? → Stock.average_cost
```

Phase 5 매출 인식 시점에 자동으로 떨어질 분개:
```
[매출 인식]
차변 매출채권   165 (부가세 포함)
대변 매출       150
대변 부가세대급금 15

[같은 시점, 자동 추가]
차변 매출원가   100   ← ⭐ Stock.average_cost × qty
대변 재고자산   100   ← Stock 에서 빠짐
```

이 "100" 의 출처가 곧 `StockMovement.unit_cost` (= 출고 시점의 `average_cost`). 평균법이 그 값을 만든다.

### 3. 매번 다른 매입가 — "이번에 빠지는 단가" 의 결정 규칙

같은 노트북을 100만원, 120만원, 110만원에 매입했다고 치자. 손님이 3대 사 가면 매출원가는?

- **개별법**: 시리얼 추적 — 운영 비용 ↑↑.
- **FIFO**: 먼저 받은 것부터 — "100짜리 3대".
- **평균법**: 다 섞어서 — "현재 평균 110짜리 3대".

→ 평균법은 "어차피 창고에서 섞여 있으니 구분이 무의미" 라는 관점. 관리 단순 + 한 줄 캐시로 끝.

### 4. **이동** 평균인 이유 — 실시간성

평균법 안에서도 갱신 시점에 따라:

| 방법 | 갱신 시점 | 특징 |
| --- | --- | --- |
| 총평균법 (Periodic) | 기간(월/연) 끝에 한 번 | 수기 장부 시대 |
| **이동평균법** (Moving / Perpetual) | **매 입고마다 즉시** | **컴퓨터 시스템의 방법** |

총평균법이면 매 출고 시점의 정확한 단가가 월말까지 미정 — Phase 5 의 자동 매출원가 분개가 불가능. 이동평균은 **매 출고가 그 시점의 정확한 단가를 즉시 박아준다** → 실시간 회계 가능.

### 정리 — 결정 트리

```
재고를 관리해야 함
   ├─ 수량만? → ❌ 재고자산/매출원가 계산 불가
   └─ 수량 + 단가
        ├─ 매번 매입가가 다름 → 어느 단가를 적용할지 규칙 필요
        │    ├─ 개별법 (시리얼 추적)         → 운영 ↑↑, 학습 범위 밖
        │    ├─ FIFO (선입선출)              → lot 행 누적, Phase 3 범위 밖
        │    └─ 평균법                       → ✅ Phase 3 채택
        │         ├─ 총평균 (월말 한 번)     → 실시간 매출원가 불가
        │         └─ 이동평균 (매 입고 즉시)  → ✅ 채택
        │
        └─ 가격 변동 평탄화 효과 보너스
             FIFO 는 오래된 입고가 먼저 빠져 매출원가가 한쪽으로 치우침,
             이동평균은 매입가 변동이 부드럽게 매출원가에 반영됨.
```

→ **"이동평균을 추적하는 이유"** = 매 출고 시점의 매출원가를 실시간으로 정확히 알려고. Phase 5 의 자동 분개가 이 한 값에 의존.

## 🔥 이동평균 (Weighted Average) 공식

```
            기존 수량 × 기존 평균  +  입고 수량 × 입고 단가
새 평균  =  ──────────────────────────────────────────────
                    기존 수량 + 입고 수량
```

코드로:

```java
public void receive(BigDecimal qty, BigDecimal unitCost) {
    if (qty == null || qty.signum() <= 0)
        throw new IllegalArgumentException("입고 수량은 0보다 커야 한다.");
    if (unitCost == null || unitCost.signum() < 0)
        throw new IllegalArgumentException("입고 단가는 0 이상이어야 한다.");

    BigDecimal newQty = this.qtyOnHand.add(qty);
    if (this.qtyOnHand.signum() == 0) {
        // 첫 입고 (또는 0 → 양수) — 입고 단가가 곧 평균.
        this.averageCost = unitCost;
    } else {
        BigDecimal totalValue = this.qtyOnHand.multiply(this.averageCost)
                .add(qty.multiply(unitCost));
        this.averageCost = totalValue.divide(newQty, 2, RoundingMode.HALF_UP);
    }
    this.qtyOnHand = newQty;
}
```

3가지 결정이 박혀 있다:
1. **`qtyOnHand == 0` 분기**: 첫 입고면 입고 단가가 곧 평균 (분모 = 입고 수량).
2. **`RoundingMode.HALF_UP` + `scale=2`**: 회계 표준 반올림.
3. **수량 갱신은 평균 계산 후**: `qtyOnHand` 를 먼저 바꾸면 공식의 "기존 수량" 이 손상됨.

## 🔥 숫자로 따라가기

```
초기:     qty=0,  avg=0

[T1] 입고 10대 @ 1000
   qtyOnHand=0 → 첫 입고 분기
   avg = 1000
   qty = 10
→ 결과: qty=10, avg=1000

[T2] 입고 10대 @ 1200
   qtyOnHand=10 → 가중평균 분기
   totalValue = 10 × 1000 + 10 × 1200 = 10000 + 12000 = 22000
   avg = 22000 / 20 = 1100
   qty = 20
→ 결과: qty=20, avg=1100

[T3] 출고 7대
   qty=20 ≥ 7 → 통과
   applied = 1100  (직전 평균)
   qty = 13
→ 결과: qty=13, avg=1100   (평균은 그대로)
→ StockMovement: qty_delta=-7, unit_cost=1100, reason=GOODS_ISSUE

[T4] 입고 5대 @ 1400
   qtyOnHand=13 → 가중평균 분기
   totalValue = 13 × 1100 + 5 × 1400 = 14300 + 7000 = 21300
   avg = 21300 / 18 = 1183.33 (HALF_UP)
   qty = 18
→ 결과: qty=18, avg=1183.33
```

매 입고가 평균을 다시 끌어올린다. 출고는 그 평균을 적용해 차감하지만 **평균 자체는 안 바꾼다**.

→ 통합 테스트 (`MmScenarioTest.입고_두번이면_가중평균으로_갱신된다`) 가 같은 시나리오를 자동 검증.

## 🔥 왜 출고는 평균을 안 바꾸나

직관: "10대 평균 1100 인 상태에서 7대 빼면, 남은 3대도 평균 1100 이다" — 자명. 같은 풀에서 일부만 빠지는 것이라 평균이 안 변함.

수식으로:
```
빠지기 전: 10대 × 1100 = 11000 원어치
빠진 후:    3대 × ?    =  3300 원어치 (3대 평균이 1100 이면)
빠진 7대:   7대 × ?    =  7700 원어치 (1100 으로 적용했다 치면)
합: 3300 + 7700 = 11000 ✓
```

각 단가에 같은 평균을 적용하면 합이 보존됨. 평균법은 "어느 입고분이 빠지는지" 구분하지 않으므로 평균을 유지하는 게 일관.

```java
public BigDecimal issue(BigDecimal qty) {
    ...
    BigDecimal applied = this.averageCost;     // ← 현재 평균을 그대로 적용
    this.qtyOnHand = this.qtyOnHand.subtract(qty);   // 수량만 차감
    return applied;
}
```

→ 출고의 적용 단가 = 차감 직전 평균. **이 값이 StockMovement.unit_cost 에 박힌다** → Phase 5 매출원가 계산의 원자료.

## 🔥 매출원가 계산의 미리보기 (Phase 5)

Phase 5 (FI) 에서 매출 인식 시 매출원가 (COGS, Cost of Goods Sold) 가 같이 잡힌다.

```
[수주 7대 @ 1500 → 출하 → 인보이스 발행]

매출 = 7 × 1500 = 10500
매출원가 = ?
```

매출원가는 **그 출고 시점의 평균 단가** 가 기반:

```sql
SELECT SUM(qty_delta * unit_cost) * -1 AS cogs
  FROM stock_movement
 WHERE ref_type = 'GI' AND ref_id = (출하의 GI ID)
   AND reason = 'GOODS_ISSUE'
```

위 예시 [T3] 의 행 (`qty_delta=-7, unit_cost=1100`) 이 그대로 사용:
```
cogs = -7 × 1100 × -1 = 7700
```

매출 10500 - 매출원가 7700 = **매출총이익 2800**.

→ StockMovement.unit_cost 가 Phase 5 의 자동 분개 (매출원가/재고자산) 의 입력. 지금 한 줄로 보이는 값이 회계 모듈 전체의 근간.

## 🔥 출고/입고 취소 — 평균은 건드리지 않는다

설계서 §0 #10: "입고 취소 시 평균 단가는 건드리지 않음 (수량만 차감, 평균 역산은 학습 범위 밖)".

```java
public BigDecimal cancelReceipt(BigDecimal qty) {
    // 의미상 issue 와 동일 — "수량만 빠지고 평균은 그대로"
    return issue(qty);
}

public void cancelIssue(BigDecimal qty) {
    if (qty == null || qty.signum() <= 0)
        throw new IllegalArgumentException("취소 수량은 0보다 커야 한다.");
    this.qtyOnHand = this.qtyOnHand.add(qty);
    // average 는 안 건드림
}
```

`cancelReceipt` 가 `issue` 와 같다는 게 의미 깊다 — **취소는 수량만 보면 출고와 동일한 효과**. 평균 단가는 둘 다 유지.

### "평균 역산" 이 왜 일반적으로 불가능한가

```
[T1] 입고 +10 @ 1000   → qty=10, avg=1000
[T2] 입고 +10 @ 1200   → qty=20, avg=1100
[T3] 출고  -5          → qty=15, avg=1100  (5대를 1100 으로 평가)
[T4] T2 입고를 취소     → ?
```

T2 의 영향을 정확히 되돌리려면:
- T3 에서 빠진 5대 중 "원래 100원짜리 5대" 였나 "새로 들어온 120원짜리 5대" 였나? 평균법은 그 구분이 **원리적으로 불가능**. (구분되는 게 FIFO — 입고 행마다 별도 추적 필요)

→ Phase 3 은 **수량만 빼고 평균은 유지** 라는 단순화. 결과:
- T4 후: `qty=5, avg=1100` (역산이라면 1000 이어야 할 것)
- 약간 부정확한 평균 — 다음 입고가 다시 가중평균으로 희석.

실무 ERP 도 같은 방식. 평가 차이는 **회계 모듈의 보정 분개** (재고평가손실 / 매입환출) 로 해결. 우리는 Phase 5 가 없으니 학습 노트로만 짚는다.

## 🔥 FIFO 와의 차이 (학습 노트)

| 차원 | 평균법 (우리 선택) | FIFO (선입선출) |
| --- | --- | --- |
| Stock 행 | (item, warehouse) 당 1행 | 입고 lot 단위 N 행 (`stock_lot`) |
| 출고 시 단가 | 현재 평균 | 가장 오래된 lot 의 단가부터 차감 |
| 매출원가 | 매번 평균에 의존 | 오래된 입고의 원가 |
| 가격 상승기 매출원가 | 평균 (낮은 입고 + 높은 입고 섞임) | 낮은 입고가 먼저 빠짐 → 매출원가 ↓ → 이익 ↑ |
| 행 수 | 일정 (관리 쉬움) | 입고할 때마다 행 추가 |
| 취소의 평균 역산 | 불가능 (방향성 없음) | 가능 (lot 추적) |

→ K-IFRS 한국 회계 기준은 둘 다 허용. 같은 회사도 상품 그룹별로 다르게 선택 가능. 학습 단계는 단순한 평균법.

### 만약 FIFO 였다면 (가정)

```java
// FIFO 라면 Stock 대신 StockLot 다중 행
public class StockLot {
    private Item item;
    private Warehouse warehouse;
    private LocalDateTime receivedAt;
    private BigDecimal originalQty;
    private BigDecimal remainingQty;     // 남아있는 수량
    private BigDecimal unitCost;          // 이 lot 의 단가 (고정)
}

// 출고 시 — 가장 오래된 lot 부터 차감
public List<StockMovement> issueFifo(BigDecimal qty) {
    List<StockLot> lots = repo.findByItemAndWarehouseOrderByReceivedAtAsc(...);
    BigDecimal remaining = qty;
    List<StockMovement> movements = new ArrayList<>();
    for (StockLot lot : lots) {
        if (remaining.signum() == 0) break;
        BigDecimal take = lot.getRemainingQty().min(remaining);
        lot.consume(take);
        movements.add(StockMovement.of(..., take.negate(), lot.getUnitCost(), ...));
        remaining = remaining.subtract(take);
    }
    if (remaining.signum() > 0) throw new InsufficientStockException(...);
    return movements;
}
```

복잡도가 폭증. lot 행이 누적되어 관리도 무거워짐. 학습 가치 대비 비용이 크지 않아 평균법으로.

## 🔥 `RoundingMode.HALF_UP` 의 선택

```java
this.averageCost = totalValue.divide(newQty, 2, RoundingMode.HALF_UP);
```

- `scale=2` — 소수 둘째 자리까지 (원화 기준).
- `RoundingMode.HALF_UP` — "반올림" (5는 올림).

대안:
- `HALF_EVEN` (banker's rounding) — 5일 때 짝수로. 통계적으로 편향이 적음. 회계용으로 표준이긴 한데 사용자 직관과 어긋남.
- `DOWN` (truncate) — 항상 내림. 의도적으로 매출원가를 줄여 매출총이익을 부풀리는 효과.
- `UP` — 항상 올림.

→ 학습 단계는 가장 직관적인 `HALF_UP`. Phase 5 회계 기준 도입 시 정책 재검토.

### scale 결정의 의미

`scale=2` 는 평균 단가가 1.234 가 되는 일이 없음. 1.23 으로 저장. 다음 가중평균 계산에서 0.004 가 누적되는 오차는 있지만:
- 한 시즌(3개월) 운영을 시뮬레이션해도 누적 오차가 평균 단가의 0.1% 미만.
- 실무 ERP 들도 보통 scale 2~4.

→ "회계 정밀도와 운영 효율의 균형점" 의 한 선택.

## 🔥 `BigDecimal` 을 쓰는 이유 (재확인)

평균 계산은 나눗셈을 동반. `double` / `float` 이라면:

```java
double avg = (10 * 1000 + 5 * 1100) / 15.0;
// 결과: 1033.3333333333333 ...
```

비교를 하면:
```java
double a = 0.1 + 0.2;
System.out.println(a == 0.3);   // false! (0.30000000000000004)
```

회계 시스템에서 `1100.00 != 1099.99` 같은 사고가 매출원가 계산 결과로 1원 차이를 만들면 분개가 안 맞음. **모든 금액/단가는 `BigDecimal`** 이 원칙 (Phase 1 메모).

```java
BigDecimal a = new BigDecimal("0.1");
BigDecimal b = new BigDecimal("0.2");
BigDecimal sum = a.add(b);   // 정확히 0.3
sum.compareTo(new BigDecimal("0.3"));   // 0 (같음)
```

> 💡 `BigDecimal` 비교는 `equals()` 가 아니라 **`compareTo() == 0`** 또는 AssertJ 의 `isEqualByComparingTo`. `new BigDecimal("1.10")` 과 `new BigDecimal("1.1")` 은 `equals` 로 다르지만 `compareTo` 로 같음 (scale 만 다름). 통합 테스트가 `isEqualByComparingTo` 를 일관 사용하는 이유.

## 🔥 가용 검증과 평균 적용의 순서

```java
public BigDecimal issue(BigDecimal qty) {
    if (qty == null || qty.signum() <= 0) ...        // (1) 입력 검증
    if (this.qtyOnHand.compareTo(qty) < 0) {         // (2) 가용 검증
        throw new InsufficientStockException(...);
    }
    BigDecimal applied = this.averageCost;            // (3) 평균 캡처
    this.qtyOnHand = this.qtyOnHand.subtract(qty);    // (4) 차감
    return applied;
}
```

(3) 의 평균 캡처가 (4) 차감보다 앞이어야 하는 이유 — 출고의 단가는 **차감 직전 평균**. 만약 (4) 가 먼저면:
- 차감 후 평균이 같으니 결과는 같지만,
- "어느 평균을 적용했나" 의 의미가 모호. **"내가 빠지기 전의 평균"** 이 더 자연스러움 (마치 출고가 가장 마지막 사건인 것처럼).

순서 한 줄의 의미가 코드 가독성을 다르게 만든다.

## 🔥 입고/출고 이후의 정합성

매 입고 후:
```
qty × avg = 누적된 매입 금액
```

매 출고 후:
```
(qty × avg) = (출고 전 qty × avg) - (출고된 qty × avg)
            = 출고 전 누적 금액 - 출고된 금액
```

→ "보유 금액 = 보유 수량 × 평균" 이 항상 성립. 단순한 곱으로 재고 평가액이 산출됨.

DB 차원의 정합성:
```sql
SELECT item_id, warehouse_id,
       qty_on_hand,
       average_cost,
       qty_on_hand * average_cost AS book_value
  FROM stock
 WHERE deleted_at IS NULL   -- Stock 은 soft delete 안 하지만 일관성 위해
```

`book_value` 가 곧 재무제표의 "재고자산" 금액. Phase 5 가 이 값을 그대로 받아 차변 분개.
