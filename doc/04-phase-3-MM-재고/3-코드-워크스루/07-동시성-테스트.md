## 🔥 단위 테스트 — 도메인만 검증 (외부 의존 0)

```java
class StockTest {

    @Test
    @DisplayName("두 번째 입고는 가중평균으로 평균을 갱신한다")
    void 두번째_입고는_가중평균으로_갱신된다() {
        Stock stock = Stock.empty(item(), warehouse());
        stock.receive(bd(10), bd(1000));     // qty=10, avg=1000
        stock.receive(bd(10), bd(1200));     // qty=20, avg=1100

        assertThat(stock.getQtyOnHand()).isEqualByComparingTo(bd(20));
        assertThat(stock.getAverageCost()).isEqualByComparingTo(bd(1100));
    }
    ...
}
```

`@SpringBootTest` 없음. JPA 없음. Testcontainers 없음. **순수 자바 객체** 만으로 동작.

→ 도메인 메서드의 책임이 자기 자신에만 있다는 증거. 외부 의존이 끼었다면 mock 이 필요했을 것.

### `BaseEntityWithCode` 와 테스트의 마찰

```java
private static Warehouse warehouse() {
    Warehouse w = Warehouse.create("WH-HQ", "본사창고", "서울시");
    if (w.getStatus() != MasterStatus.ACTIVE) {
        w.changeStatus(MasterStatus.ACTIVE);
    }
    return w;
}
```

마지막 if 가 어색해 보이지만 — `BaseEntityWithCode` 의 `status` 가 `= MasterStatus.ACTIVE` 로 필드 초기화돼 있으므로 보통은 if 가 안 탐. 방어용. 만약 베이스 정책이 바뀌어 ACTIVE 가 아닌 다른 상태로 시작하게 되면 이 한 줄이 안전망.

### `isEqualByComparingTo` 의 이유

```java
assertThat(stock.getAverageCost()).isEqualByComparingTo(bd(1100));
```

`isEqualTo(new BigDecimal("1100"))` 는 위험 — `equals()` 가 scale 까지 비교 (`1100.00 != 1100`). `compareTo() == 0` 만으로 동등 판정하는 게 회계 가치 비교의 표준.

→ AssertJ 가 `isEqualByComparingTo` 라는 명시적 메서드를 제공. Phase 2 의 모든 BigDecimal 단언이 이걸 쓰는 이유.

## 🔥 통합 시나리오 — 5개 핵심 흐름

`MmScenarioTest` 는 5개 테스트로 Phase 3 의 핵심 약속을 종합 검증.

### 1. `입고_두번이면_가중평균으로_갱신된다`

순수 흐름:
```
입고 10대 @ 1000 → post → stock(qty=10, avg=1000)
입고 10대 @ 1200 → post → stock(qty=20, avg=1100)
```

가중평균 공식이 실제 JPA/DB 를 통과해도 작동하는지.

### 2. `출고는_직전_평균을_원장에_기록한다`

```java
Stock stock = stockRepository.findByItemIdAndWarehouseId(...).orElseThrow();
assertThat(stock.getQtyOnHand()).isEqualByComparingTo(bd(13));
assertThat(stock.getAverageCost()).isEqualByComparingTo(bd(1100));   // 출고 후도 1100 유지

StockMovement giMovement = stockMovementRepository.findAll().stream()
        .filter(m -> m.getReason() == MovementReason.GOODS_ISSUE && m.getRefId().equals(gi.id()))
        .findFirst().orElseThrow();
assertThat(giMovement.getQtyDelta()).isEqualByComparingTo(bd(-7));
assertThat(giMovement.getUnitCost()).isEqualByComparingTo(bd(1100));
```

핵심 단언: `unit_cost=1100` — 출고 시 적용된 평균이 원장에 정확히 박혔는지. **Phase 5 매출원가의 근간이 이 한 값**.

### 3. `가용_재고_부족이면_거부된다`

```java
assertThatThrownBy(() -> goodsIssueService.post(draft.id()))
        .isInstanceOf(InsufficientStockException.class)
        .hasMessageContaining("재고 부족");

Stock stock = stockRepository.findByItemIdAndWarehouseId(...).orElseThrow();
assertThat(stock.getQtyOnHand())
        .as("실패한 출고는 트랜잭션 롤백 — 재고는 그대로 5대")
        .isEqualByComparingTo(bd(5));
```

두 가지 검증:
- 예외 종류 + 메시지.
- **트랜잭션 롤백 효과** — 재고가 정말 그대로 5대인지. 만약 도메인 검증이 누락되고 DB CHECK 만 막아도 같은 외적 결과지만, `InsufficientStockException` 종류 검증이 도메인 검증 경로를 확정.

### 4. `원장_합계는_Stock_과_일치한다` — 정합성 약속의 검증

```java
// 입고 10 → 출고 3 → 입고 5 → 출고 7 의 4단계 시나리오 후
Stock stock = stockRepository.findByItemIdAndWarehouseId(...).orElseThrow();
BigDecimal ledgerSum = stockMovementRepository.findAll().stream()
        .filter(m -> m.getItem().getId().equals(ctx.itemId)
                && m.getWarehouse().getId().equals(ctx.warehouseId))
        .map(StockMovement::getQtyDelta)
        .reduce(BigDecimal.ZERO, BigDecimal::add);

assertThat(stock.getQtyOnHand())
        .as("Stock.qty_on_hand == SUM(stock_movement.qty_delta)")
        .isEqualByComparingTo(ledgerSum);
assertThat(stock.getQtyOnHand()).isEqualByComparingTo(bd(5));   // 10 - 3 + 5 - 7 = 5
```

워크스루 02편의 정합성 식이 테스트로 강제됨. **시스템의 자기 검증 약속**. Phase 9 (배치) 에서 야간 검증 배치로 격상되면 같은 패턴을 SQL 로 풀 것.

### 5. `출고_취소시_수량이_복원된다`

```java
goodsIssueService.post(gi.id());   // qty: 10 → 3
goodsIssueService.cancel(gi.id()); // qty: 3 → 10 (복원)

Stock stock = ...;
assertThat(stock.getQtyOnHand()).isEqualByComparingTo(bd(10));
assertThat(stock.getAverageCost()).isEqualByComparingTo(bd(1000));   // 평균 그대로
```

수량 복원 + 평균 유지. 도메인의 `cancelIssue` 가 약속한 행위.

## 🔥 동시성 테스트 ⭐ — `ConcurrentGoodsIssueTest`

Phase 3 의 하이라이트. 코드를 한 줄씩 짚어본다.

```java
@SpringBootTest
@Import(TestcontainersConfiguration.class)
class ConcurrentGoodsIssueTest {
    ...

    @Test
    @DisplayName("동시_출고_요청은_비관적_락으로_직렬화되어_음수_재고가_발생하지_않는다")
    void 동시_출고_요청은_비관적_락으로_직렬화되어_음수_재고가_발생하지_않는다() throws Exception {
        // (1) 마스터 셋업 + 시작 재고 10대
        long nano = System.nanoTime();
        var item = itemService.create(...);
        var vendor = vendorService.create(...);
        var warehouse = warehouseService.create(...);

        var gr = goodsReceiptService.create(new GoodsReceiptCreateRequest(
                vendor.id(), warehouse.id(), LocalDate.now(),
                List.of(new GoodsReceiptLineRequest(item.id(), new BigDecimal("10"), new BigDecimal("1000")))));
        goodsReceiptService.post(gr.id());

        // (2) 8개 DRAFT 출고를 먼저 만들어 둔다 (각 2대씩)
        int threads = 8;
        BigDecimal perRequest = new BigDecimal("2");
        List<Long> issueDraftIds = IntStream.range(0, threads).mapToObj(i ->
                goodsIssueService.create(new GoodsIssueCreateRequest(
                        warehouse.id(), LocalDate.now(), GoodsIssueReason.SHIPMENT,
                        List.of(new GoodsIssueLineRequest(item.id(), perRequest)))).id()
        ).toList();

        // (3) 8개 스레드가 startGate 에서 일제히 출발 → post() 동시 호출
        ExecutorService pool = Executors.newFixedThreadPool(threads);
        CountDownLatch startGate = new CountDownLatch(1);
        AtomicInteger successes = new AtomicInteger();
        AtomicInteger failures = new AtomicInteger();

        List<Future<?>> futures = issueDraftIds.stream().<Future<?>>map(giId -> pool.submit(() -> {
            try {
                startGate.await();
                goodsIssueService.post(giId);
                successes.incrementAndGet();
            } catch (InsufficientStockException isx) {
                failures.incrementAndGet();
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
            return null;
        })).toList();

        startGate.countDown();   // ← 동시 출발
        pool.shutdown();
        assertThat(pool.awaitTermination(60, TimeUnit.SECONDS)).isTrue();
        for (Future<?> f : futures) f.get();

        // (4) 검증: 정확히 5 성공, 3 실패. 종료 재고 = 0.
        assertThat(successes.get()).as("성공한 출고 수").isEqualTo(5);
        assertThat(failures.get()).as("실패한 출고 수 (가용 부족)").isEqualTo(3);

        Stock stock = stockRepository
                .findByItemIdAndWarehouseId(item.id(), warehouse.id()).orElseThrow();
        assertThat(stock.getQtyOnHand()).as("종료 재고는 정확히 0").isEqualByComparingTo(BigDecimal.ZERO);
    }
    ...
}
```

### (1) 마스터 셋업

테스트마다 unique 한 item/vendor/warehouse 를 만든다. Phase 2 의 `PartialDeliveryAndInvoiceScenarioTest` 와 같은 패턴 — 테스트 간 격리 + V20 시드와의 충돌 회피.

### (2) DRAFT 출고를 미리 만들어두는 이유

만약 `post` 단계에서 DRAFT 만들기까지 같이 하면:
- `goodsIssueService.create(...)` 가 트랜잭션 번호를 발급 (CodeGenerator → 비관 락).
- 그 락 자체가 직렬화 → "재고 차감 동시성" 의 측정 노이즈.

→ 번호 발급 동시성은 `TransactionNumberConcurrencyTest` 에서 별도 검증된 사항. 이 테스트는 **순수하게 재고 차감의 직렬화** 만 측정하고 싶음.

### (3) `CountDownLatch startGate` — 동시 출발 트릭

```java
CountDownLatch startGate = new CountDownLatch(1);

// 각 스레드:
startGate.await();   // ← 대기
goodsIssueService.post(giId);

// 메인 스레드:
startGate.countDown();   // ← 0 으로 만들면 모든 스레드 일제히 풀림
```

스레드 풀에 submit 하면 작업이 즉시 시작될 수 있는데, 이러면 일찍 시작된 스레드가 다른 스레드보다 먼저 끝나버려 동시성 시나리오가 안 나옴. `startGate.await()` 로 모든 스레드가 한 지점에서 대기 → `countDown()` 로 일제히 출발 → 진짜 동시.

→ Phase 1 `CodeGeneratorConcurrencyTest` 의 동일 패턴.

### (3.5) 타입 단언 `.<Future<?>>map(...)`

```java
List<Future<?>> futures = issueDraftIds.stream().<Future<?>>map(giId -> pool.submit(() -> { ... })).toList();
```

자바 타입 추론이 `.toList()` 결과를 `List<Future<Object>>` 로 추론. `List<Future<?>>` 로 받으려면 명시적 type witness 가 필요 — `.<Future<?>>map(...)` 으로 강제.

→ 컴파일 에러 한 줄 잡는 패턴. 처음엔 컴파일이 실패해 알게 된 부분.

### (4) 검증의 두 단언

```java
assertThat(successes.get()).isEqualTo(5);
assertThat(failures.get()).isEqualTo(3);
assertThat(stock.getQtyOnHand()).isEqualByComparingTo(BigDecimal.ZERO);
```

- 성공 5 + 실패 3 = 총 8 (스레드 수와 일치 — 누락 없음).
- 성공 5 × 2대 = 10대 = 시작 재고. **정확히 다 빠짐** — 음수도, 잉여도 없음.
- 종료 재고 0.

이 세 단언이 동시에 통과한다는 것 = **비관 락이 race condition 을 완벽히 막고 있음의 증거**.

## 🔥 만약 락이 없었다면 — 테스트 시뮬레이션

이 테스트를 그대로 두고 `findForUpdate` 를 `findByItemIdAndWarehouseId` 로 바꾸면 어떻게 될까:

```java
// 가상의 시나리오 — 코드 변경 X, 머릿속 시뮬레이션
Stock stock = stockRepository
        .findByItemIdAndWarehouseId(item.getId(), warehouse.getId())  // 락 없음
        .orElseThrow(...);
BigDecimal appliedCost = stock.issue(line.getQuantity());
```

@Version 이 있으니 `OptimisticLockException` 으로 일부 충돌이 검출되긴 한다. 하지만:

```
스레드 A: SELECT (qty=10, version=5) → issue(2) → qty=8
스레드 B: SELECT (qty=10, version=5) → issue(2) → qty=8
A: UPDATE WHERE version=5 → 성공, version=6
B: UPDATE WHERE version=5 → 0 row → OptimisticLockException → ROLLBACK
```

낙관 락만 있어도 일관성은 보장. 하지만:
- B 가 실패만 함. 클라이언트는 재시도 책임.
- 재시도하지 않으면 **이론상** 가능한 출고가 실패로 끝남 (재고 있었는데).

**비관 락은 다르다**:
```
A: findForUpdate → 락 점유
B: findForUpdate → 대기
A: issue(2), commit → 락 해제
B: 대기 해제 → 락 점유 → issue(2) (실제 검증, 재고 8 보고 → 성공)
```

B 도 자기 차례에 가용 검증을 다시 함 → 실제로 부족할 때만 거부. **있는 재고는 모두 활용**.

→ 둘 다 정합성은 보장. 비관 락은 "재시도 없이 한 번에 정확한 결과", 낙관 락은 "재시도하면 같은 결과". Phase 3 은 출고가 너무 핵심이라 재시도 부담 없는 비관 락.

## 🔥 한 가지 한계 — 동시성 테스트는 운영 환경 보장이 아님

```java
@SpringBootTest
@Import(TestcontainersConfiguration.class)
class ConcurrentGoodsIssueTest {
```

Testcontainers 가 띄운 MySQL 8 컨테이너에서 동시 출고가 직렬화됨을 확인. 운영 환경:
- MySQL 의 트랜잭션 격리 수준이 같은 `REPEATABLE_READ` 인지.
- Connection Pool 의 동작이 동일한지.
- 트랜잭션 타임아웃 설정이 합리적인지.

운영 환경에서도 같은 결과가 나오려면 **같은 DB + 같은 설정** 이어야. Testcontainers 가 `mysql:8.0` 으로 운영과 같은 버전을 띄우는 게 신뢰의 기반.

> 💡 다른 DB (Postgres) 로 바꾸면 같은 코드가 다르게 동작할 수도. 비관 락의 의미는 SQL 표준이지만 구현 디테일이 다름. 그래서 통합 테스트는 운영 DB 와 같은 엔진으로 돌리는 게 안전.

| `OptimisticLockReceiptTest` — 동시 입고 충돌 시 한쪽이 재시도 대상 | 입고 동시 충돌이 거의 안 일어남. 작성 비용 > 학습 가치 (낙관 락의 의미는 워크스루 04편으로 충분). |
| MM 트랜잭션 번호 동시성 | Phase 2 `TransactionNumberConcurrencyTest` 가 SO/Q 로 같은 메커니즘을 이미 검증. GR/GI 도 같은 `CodeGenerator` 위임이라 자명. |
| 음수 재고 DB CHECK 단독 테스트 | 도메인 검증이 먼저 통과되므로 운영 경로에서 CHECK 가 보이지 않음. 코드 우회로 직접 SQL UPDATE 해야 검증 가능 — 학습 가치 ↓. |
| `StockTransfer` 시나리오 | 설계서 §11 "Phase 3 범위 밖". |
| FIFO vs 평균법 비교 | 평균법만 구현. |

→ 미루는 결정도 학습의 일부. 모든 케이스를 테스트하면 시그널 대비 노이즈가 커진다.

## 🔥 Phase 3 의 약속 — 한 화면에

```
✓ 첫 입고는 입고 단가가 평균이 된다
✓ 두 번째 입고는 가중평균으로 갱신된다
✓ 출고는 평균 단가를 유지한다 (수량만 빠짐)
✓ 가용 재고 미만이면 InsufficientStockException (409)
✓ 출고 취소는 수량을 다시 더하고 평균은 건드리지 않는다
✓ 입고 취소는 수량만 차감하고 평균은 건드리지 않는다 (역산 불가 한계 인정)
✓ 입고/출고 수량 0 이하는 거부

✓ StockMovement.qty_delta 0 은 거부
✓ reason 의 부호와 qty_delta 의 부호가 일치해야 함

✓ 입고 시 stock 행 생성, 두 번째 입고는 가중평균으로 갱신
✓ 출고는 직전 평균을 StockMovement.unit_cost 에 기록
✓ 가용 부족 출고는 트랜잭션 롤백 (재고 변동 없음)
✓ Stock.qty_on_hand == SUM(StockMovement.qty_delta) 정합성
✓ 출고 취소시 Stock 수량 복원 + 평균 유지

✓ 동시 출고 요청은 비관적 락으로 직렬화 → 음수 재고 절대 없음 ⭐
```

16개 항목. Phase 3 가 도입한 모든 비즈니스 약속이 자동 검증됨. 이 약속이 깨지면 = 빌드 빨강.
