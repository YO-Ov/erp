## 시연의 목표 (Phase 3 가 무엇을 보장하는가)

Phase 2 에서는 OTC 흐름의 절반 (견적~인보이스) 이 작동했다. 단, **출하(Delivery) 가 SO 라인의 `shippedQty` 만 갱신**했지 실제 창고 재고를 건드리지는 않았다 — Phase 3 가 그 빈칸을 메운다.

이번에 확인할 **5대 규칙**:

1. **재고는 (상품, 창고) 당 1행 캐시 + 원장** — `Stock` 한 행이 "지금" 을 들고 있고, `StockMovement` 가 모든 과거 변화를 append-only 로 보존. 항상 `SUM(qty_delta) == qty_on_hand`.
2. **입고는 가중평균으로 평균 단가 갱신** — `새 평균 = (기존 수량 × 기존 평균 + 입고 수량 × 입고 단가) / (기존 수량 + 입고 수량)`. 출고는 그 평균을 그대로 적용해 `StockMovement.unit_cost` 에 박는다 (Phase 5 매출원가의 근간).
3. **가용 재고가 부족하면 출고 거부 (409 INSUFFICIENT_STOCK)** — 응답에 `available, requested` 가 함께 박혀 운영자가 즉시 진단 가능. Phase 2 신용한도의 패턴 그대로.
4. **두 락 패턴의 공존** — 출고는 비관적 락 (`SELECT ... FOR UPDATE`) 으로 race 자체를 차단, 입고는 낙관적 락 (`@Version`) 으로 가볍게. 한 도메인에서 둘이 자연스럽게 만남.
5. **취소의 정합성** — 출고 취소 시 `Stock.qty_on_hand` 복원 + `StockMovement` 음수의 ADJUSTMENT_PLUS 행 추가 (append-only). 평균은 건드리지 않음.

거기에 V20 으로 시드된 **본사창고(`WH-HQ`) + 노트북 50대 @ 100만원 초기 재고** 가 보이는지도 확인한다.

### 4-2. 두 번째 입고 — 10대 @ 120만원

### 4-4. Stock 가중평균 확인

```bash
curl -s "http://localhost:8080/api/stocks/1" | jq '{itemCode, warehouseCode, qtyOnHand, averageCost, version}'
```

기대:
```json
{
  "itemCode": "ITEM-2026-0001",
  "warehouseCode": "WH-HQ",
  "qtyOnHand": 60.0000,
  "averageCost": 1033.33,
  "version": 1

```

확인 포인트:
- **공식 확인**: `(50 × 1000000 + 10 × 1200000) / 60 = 60000000/60 = 1033333.33` — 회계 표준 `HALF_UP` 으로 `1033333.33` (스케일 2).
  > 응답에서 `averageCost` 가 `1033333.33` 으로 보일 것 (위 예시는 jq 가 소수 축약했을 수 있음).
- **`version: 1`** — `@Version` 이 1 증가 (시드 직후 0, 첫 입고 확정 후 1). 낙관적 락의 흔적 (워크스루 04편).
- **`qty_on_hand: 60`** — 50 + 10 누적.

### 4-5. 원장 추가 확인


기대:
```json

```

확인 포인트:
- **원장은 append-only** — 첫 입고 행이 그대로 보존되고, 두 번째 행이 추가만 됨.
- **`unit_cost` 가 입고 단가 그대로** — 각 입고 시점의 단가가 영구 기록. 가중평균은 Stock 의 캐시일 뿐.
- 정합성: `50 + 10 = 60 = stock.qty_on_hand` ✓.

```bash
GI_OVER_ID=2   # 위에서 받은 id
curl -i -X POST "http://localhost:8080/api/goods-issues/$GI_OVER_ID/post"
```

기대:
```
HTTP/1.1 409 Conflict
Content-Type: application/problem+json

{
  "type": "about:blank",
  "title": "Insufficient Stock",
  "status": 409,
  "detail": "재고 부족: itemId=1, warehouseId=1, available=53, requested=100",
  "code": "INSUFFICIENT_STOCK",
  "itemId": 1,
  "warehouseId": 1,
```

확인 포인트:
- **상태 코드 409** — "자원은 있는데 현재 상태에서 그 동작 거부" (Phase 2 신용한도와 같은 의미).


기대: `qtyOnHand: 53` (그대로). 실패한 출고는 모두 롤백 — 재고는 단 1도 안 변함.

### 7-2. Stock 수량 복원 확인


기대:
```json
{

  "version": 3                  ← 또 +1

```

확인 포인트:

- **평균은 안 건드림** — Stock.cancelIssue 는 qty 만 더한다 (워크스루 05편 §취소 도메인 메서드).

### 7-3. 원장에 양수 정정 행 추가

```bash
curl -s "http://localhost:8080/api/stock-movements?itemId=1&warehouseId=1" | jq '.content[] | {qtyDelta, unitCost, reason, refType, refId}'
```

기대 (시간 역순):
```json
{ "qtyDelta": 7,   "unitCost": 1033333.33, "reason": "ADJUSTMENT_PLUS", "refType": "GI", "refId": 1 }   ← 취소 행
{ "qtyDelta": -7,  "unitCost": 1033333.33, "reason": "GOODS_ISSUE",     "refType": "GI", "refId": 1 }
{ "qtyDelta": 10,  "unitCost": 1200000,    "reason": "GOODS_RECEIPT",   "refType": "GR", "refId": 2 }
{ "qtyDelta": 50,  "unitCost": 1000000,    "reason": "GOODS_RECEIPT",   "refType": "GR", "refId": 1 }
```

확인 포인트:
- **원장은 수정/삭제 안 함** — 원래 GOODS_ISSUE 행은 그대로 두고, ADJUSTMENT_PLUS 행을 새로 추가해 정정 (append-only, 워크스루 02편).
- 정합성: `50 + 10 - 7 + 7 = 60 = stock.qty_on_hand` ✓.
- 양쪽 행이 같은 `refType=GI, refId=1` — 한 GI 에 묶인 한 쌍의 이력. `(ref_type, ref_id)` 인덱스로 빠르게 역추적.

## STEP 8. 원장 정합성 직접 SQL 검증

워크스루 02편 §정합성 검증 쿼리의 SQL 을 직접 돌려본다 — 시스템 자기 검증.

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT m.item_id, m.warehouse_id,
         SUM(m.qty_delta) AS ledger_qty,
         s.qty_on_hand
    FROM stock_movement m
    JOIN stock s ON s.item_id = m.item_id AND s.warehouse_id = m.warehouse_id
   GROUP BY m.item_id, m.warehouse_id, s.qty_on_hand
  HAVING SUM(m.qty_delta) <> s.qty_on_hand;"
```

기대:


0행이어야 한다. **시스템 약속이 깨지지 않았다는 증거** — 모든 (item, warehouse) 에 대해 원장 합 = 캐시.

## 트러블슈팅

### 1. STEP 4-2 `POST /api/goods-receipts` 에서 `400 Bad Request — vendorId 가 null`

- 본문 누락. `vendorId, warehouseId, receiptDate, lines` 4 필수.
- vendorId/warehouseId 는 마스터의 **PK** (id), 코드 문자열이 아님. `GET /api/vendors/by-code/VEND-2026-0001` 로 id 확인.

### 2. STEP 5-3 `averageCost` 가 예상보다 1원 차이

- `HALF_UP` 반올림의 결과. (50×1000000 + 10×1200000) / 60 = 1033333.333... → 1033333.33. 의도된 동작.
- 만약 응답에서 `1033333` 처럼 보이면 jq 또는 클라이언트의 표시 축약 — DB 의 실제 값은 1033333.33.

### 3. STEP 6 의 가용 부족 시도가 400 Bad Request 로 떨어진다

- DRAFT 생성 단계 (`POST /api/goods-issues`) 가 아니라 **post 단계 (`POST /api/goods-issues/{id}/post`)** 에서 검증된다. DRAFT 는 "장바구니" 처럼 자유롭게 만들 수 있고, post 가 "결제" 처럼 검증의 관문.
- 만약 quantity=0 같은 Bean Validation 위반이면 DRAFT 생성 자체가 400 거부. `@Positive` 검증 (DTO).

### 4. 통합 테스트 `ConcurrentGoodsIssueTest` 가 간헐 실패 (성공 6, 실패 2 등)

- 비관 락이 깨졌다는 의미가 아니라, **테스트 셋업의 race** 가능성. 예: 시드된 (item, warehouse) 와 같은 행을 다른 테스트가 동시 변경.
- 보통 isolated. 재실행: `./gradlew test --tests "*ConcurrentGoodsIssueTest"`.

### 5. Stock 행이 없는데 출고 시도하면?

```
HTTP/1.1 404 Not Found
detail: "Stock not found: item=X, warehouse=Y — 출고 전에 입고가 있어야 합니다."
```

- 의도된 거부. (item, warehouse) 조합으로 Stock 행이 생긴 적이 없음 = 한 번도 입고가 없었음.
- 부산창고를 새로 만들고 입고 없이 출고하려고 하면 이 메시지를 보게 됨.

### 6. STEP 10 의 새 창고가 보이지 않음

- `GET /api/warehouses` 로 전체 목록 확인. `deleted_at IS NULL` 자동 필터가 살아있는 행만 보여줌 (워크스루 01편).
- 만약 실수로 DELETE 했다면 — Soft Delete 라서 DB 에는 있지만 일반 조회에 안 나옴. 복구 API 는 의도적으로 없음 (Phase 1 정책).

## 시연 후 — 무엇이 학습됐는가

Phase 3 시연으로 **체감**으로 알게 된 것:

1. **"현재 재고는 캐시, 원장이 진실"** — Stock 한 행이 "지금" 을 빠르게 답하지만, 정답은 StockMovement 의 누적 SUM. 정합성 식 `SUM(qty_delta) == qty_on_hand` 가 시스템 약속의 본체. Phase 2 의 "SO 라인 `shippedQty` 캐시 / Delivery 행 진실" 과 같은 구조.
2. **"같은 상품이 다른 창고면 다른 재고"** — STEP 10 에서 직접 본 사실. `UNIQUE (item_id, warehouse_id)` 가 모델의 뼈대. 한 회사가 N개 창고를 가지면 N개 행이 자연스럽게 늘어남.
3. **"가중평균은 입고 시점에 갱신, 출고는 그 평균을 적용만"** — 출고가 평균을 바꾸지 않으니, 어느 시점의 출고든 그 직전 평균이 그대로 박힘. 이 단가가 Phase 5 매출원가의 입력이 된다 — **지금 한 줄의 값이 회계 모듈 전체의 근간**.
4. **"비관적 락이 race condition 을 원천 차단"** — 8개 스레드가 한 행을 두고 동시 출고해도 음수 재고가 발생하지 않음. `findForUpdate` 한 줄이 SELECT ... FOR UPDATE 로 직렬화. 반대로 입고는 가벼운 `@Version` 으로 충돌 시 재시도. **한 도메인에서 두 락 패턴이 자연스럽게 공존**.
5. **"취소도 append-only"** — 출고 취소 시 원래 GOODS_ISSUE 행을 수정하지 않고 ADJUSTMENT_PLUS 새 행을 추가. 원장의 신뢰성 유지. 회계 추적 가능.
