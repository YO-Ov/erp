# 04. 모듈 연계 — ① 도메인 편

> **이 글의 목적**: ERP 가 "단순 CRUD 가 아닌" 진짜 이유 — 한 거래가 여러 모듈을 자동으로 엮는 것 —
> 을 업무 관점에서 이해한다. 코드는 다음 글(② 구현 편)에서.

---

## 1. 먼저, 문제 상황

ERP 가 없다고 치자. 노트북 10대를 고객에게 **출하**하면 세 부서가 각자 장부를 적어야 한다.

- **영업팀**: "노트북 10대 ○○회사에 보냄"
- **창고팀(자재)**: "창고 재고 10대 줄여야지" ← 깜빡하면 재고가 안 맞음
- **회계팀**: "나간 물건 원가만큼 비용 잡고, 나중에 매출도 잡아야지" ← 빠뜨리면 장부가 틀어짐

**문제**: 사람이 따로따로 적으니 **누락·불일치**가 생긴다. 영업은 보냈다는데 창고 재고는 그대로,
회계 장부엔 기록이 없고… 기간계(회사 운영 시스템)에서 이건 치명적이다.

---

## 2. ERP 의 답 — "한 사건이 관련 부서를 자동으로 갱신한다"

ERP 에서는 **영업이 '출하 확정' 한 번 하면** 나머지가 자동으로 따라온다.

```
[영업] 출하 확정  ──자동──▶ [자재] 창고 재고 10대 차감
                  ──자동──▶ [회계] "매출원가" 전표 기록
```

담당자는 한 번만 처리했는데 **세 부서의 데이터가 동시에, 정확히** 맞춰진다.
이게 ERP 의 본질이자 "단순 CRUD 가 아닌" 이유다.

---

## 3. 우리 시스템의 실제 연계 3가지

> 배경 한 줄씩:
> - **출하(SD)** = 주문한 물건을 실제로 내보내는 것
> - **재고(MM)** = 창고에 몇 개 있는지
> - **전표(FI)** = 회계 장부에 적는 한 건의 기록 (복식부기 — 05 에서 깊게)

| 무슨 일이 생기면 (트리거) | 자동으로 따라오는 것 |
| --- | --- |
| **출하 확정** (SD) | ① 재고 차감 (MM) ② 매출원가 전표 (FI) |
| **세금계산서 발행** (SD) | 매출 / 부가세 / 매출채권 전표 (FI) |
| **입고 확정** (MM, 물건 사올 때) | ① 재고 증가 (MM) ② 매입 전표 (FI) |

핵심은 전부 **"한 모듈의 행동 → 다른 모듈이 자동 반응"** 구조라는 점.

---

## 4. 이 연계가 지켜야 할 두 가지 성질 ⭐ (도메인의 알맹이)

### (1) 원자성 — "다 되거나, 다 안 되거나"
출하했는데 재고 차감이 실패하면? **전부 취소(롤백)** 되어야 한다.
"물건은 나갔는데 재고는 그대로"인 어정쩡한 상태가 절대 남으면 안 된다.
→ 이 셋은 **하나의 트랜잭션**으로 묶인다.

### (2) 느슨한 결합 — "영업은 회계를 몰라도 된다"
영업 로직이 "출하했으면 재고 깎고, 전표도 써!" 라고 직접 다 시키면,
영업 코드가 재고·회계를 전부 알아야 해서 엉망이 된다(강결합).

그래서 ERP 는 이렇게 한다:
> **영업은 그냥 "출하했다!" 고 알리기만(이벤트 발행) 한다.**
> **재고팀·회계팀이 그 소식을 듣고(이벤트 구독) 알아서 자기 일을 한다.**

영업은 누가 듣는지 신경 안 쓴다. 나중에 "출하하면 통계도 갱신해줘" 가 추가돼도
**영업 코드는 안 바뀐다** — 새 구독자만 붙이면 되니까. (구현은 ② 에서 Spring Events.)

---

## 5. ERP 안은 "즉시", MES 는 "나중에"

- **ERP 내부 모듈 연계**(출하→재고→회계)는 **같은 순간, 같은 트랜잭션**에서 처리된다(동기).
- **MES 와의 연계**(공장 실적)는 **나중에 안전하게** 전달된다(비동기 Kafka).
- 같은 회사 안(ERP)이냐, 다른 시스템(MES)이냐에 따라 방식이 다르다. (11~14 에서)

---

## 요약

1. 부서가 따로 일하면 **누락·불일치** → ERP 는 한 사건으로 관련 모듈을 **자동 갱신**
2. 우리 시스템: 출하→재고·매출원가 / 계산서→매출·부가세·채권 / 입고→재고·매입
3. 필수 성질 둘: **원자성(롤백)** + **느슨한 결합(이벤트)**
4. ERP 내부는 동기, MES 는 비동기
