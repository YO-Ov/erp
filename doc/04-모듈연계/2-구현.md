# 04. 모듈 연계 — ② 구현 편

> ① 도메인 편에서 배운 **"원자성"** 과 **"느슨한 결합"** 이 코드에서 정확히 어떻게 구현됐는지 본다.
> 코드 조각을 인용하므로 파일을 안 열어도 따라올 수 있다.

---

## 0. 전체 그림 한 줄

> **SD 가 "사건"을 발행** → **MM·FI 리스너가 같은 트랜잭션에서 구독** → 하나라도 실패하면 출하 자체가 통째로 롤백.

---

## 1. "사건"을 정의한다 — 이벤트(record)

출하가 일어났다는 사실을 객체로 만든다.

```java
// sd/delivery/event/DeliveryShippedEvent.java
public record DeliveryShippedEvent(
        Long deliveryId,
        Long warehouseId,
        LocalDate shippedDate,
        List<Line> lines        // 어떤 품목 몇 개 나갔는지
) {
    public record Line(Long itemId, BigDecimal quantity) {}
}
```

💡 왜 `lines` 까지 담나? 듣는 쪽(재고)이 **같은 트랜잭션 안에서 즉시** 쓰기 때문에, DB 를 다시 조회하지 말고
사건에 실어 보낸다. 이벤트 타입은 **SD 패키지**에 있고 MM 이 가져다 쓴다 → 의존 방향 `MM → SD` 단방향.

---

## 2. 영업은 "알리기만" 한다 — 발행

```java
// sd/delivery/DeliveryService.java
@Transactional
public DeliveryResponse create(DeliveryCreateRequest req) {
    ...
    delivery.ship();
    repository.save(delivery);

    // ⭐ "출하했다!" 고 알리기만. 누가 듣는지는 모름.
    events.publishEvent(new DeliveryShippedEvent(
            delivery.getId(), delivery.getWarehouseId(),
            delivery.getShippedDate(), toEventLines(delivery)));

    return mapper.toResponse(delivery);
}
```

💡 ① 에서 "영업은 회계를 몰라도 된다" 고 했다. 코드로 증명된다 — `DeliveryService` 는
`GoodsIssueService`(재고)나 `AutoJournalService`(회계)를 **import 조차 안 한다.**
의존하는 건 `ApplicationEventPublisher events` 하나뿐.

---

## 3. 재고·회계가 "알아서 듣는다" — 구독

### 재고 리스너 (MM)
```java
// mm/integration/sd/DeliveryEventListener.java
@Order(10)   // 회계(20)보다 먼저!
@TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
void onShipped(DeliveryShippedEvent event) {
    goodsIssueService.createAndPostFromDelivery(event);  // 재고 차감
}
```

### 회계 리스너 (FI)
```java
// fi/integration/sd/CogsAccountingListener.java
@Order(20)   // 재고가 단가 박은 뒤에야 매출원가 계산 가능
@TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
void onShipped(DeliveryShippedEvent event) {
    autoJournal.createCogsEntry(event);   // 매출원가 전표
}
```

같은 `DeliveryShippedEvent` 하나를 **둘이 동시에 구독**한다. 영업 코드는 이들의 존재를 모른다.
나중에 "출하하면 통계도" 가 필요하면 **리스너만 하나 더** 붙이면 된다(영업 코드 무수정).

> 패키지명 `mm/integration/sd/`, `fi/integration/sd/` 자체가 "SD 로부터 들어오는 통합" 이라는
> 의존 방향(MM/FI → SD)을 드러낸다.

---

## 4. "원자성" 은 어떻게 보장되나 — `BEFORE_COMMIT` ⭐

① 의 "다 되거나 다 안 되거나" 가 바로 이 어노테이션이다.

```
@TransactionalEventListener(phase = BEFORE_COMMIT)
```

| 만약 이거였다면 | 결과 |
| --- | --- |
| `@EventListener` (그냥) | 발행 즉시 실행 — 트랜잭션 경계 애매 |
| 기본값 `AFTER_COMMIT` | 출하가 **이미 커밋된 뒤** 실행 → 재고 차감 실패해도 출하는 저장됨 = **부분 상태** ❌ |
| **`BEFORE_COMMIT`** | 출하 트랜잭션 **commit 직전, 같은 트랜잭션**에서 실행 → 실패 시 **전부 롤백** ✅ |

그리고 리스너가 부르는 서비스(`GoodsIssueService`, `AutoJournalService`)는 `@Transactional`(REQUIRED) 이라
**새 트랜잭션을 열지 않고 기존 것에 합류**한다. 그래서 셋이 한 트랜잭션으로 묶인다.
(리스너 자체엔 `@Transactional` 을 안 붙이는 이유 = 합류시키려고.)

> 통합 테스트 `SdMmIntegrationTest` 가 "재고 부족 시 출하까지 롤백" 을 실제로 검증한다.

---

## 5. `@Order` 가 왜 필요한가 — 순서 정합성

재고(@Order 10) → 회계(@Order 20) 순서가 **강제**돼야 한다.
1. 재고 리스너가 출고하면서 `StockMovement` 에 **출고 단가(unit_cost)** 를 박는다.
2. 회계 리스너는 그 단가를 **읽어서** 매출원가를 계산한다.

순서가 바뀌면 회계가 "아직 안 박힌 단가" 를 읽어 깨진다. 그래서 `@Order` 로 못박았다.
→ **한 사건을 여러 모듈이 들을 때는 실행 순서를 명시해야 한다** 는 교훈.

---

## 6. 매출 전표 (세금계산서 흐름)

세금계산서 발행도 같은 패턴 — `InvoiceService.create` 가 `InvoiceIssuedEvent` 발행 →
`SalesAccountingListener` 가 받아 분개:

```java
// fi/journal/AutoJournalService.java — createSalesEntry
entry.addDebit(account(AR),           event.totalAmount());  // 차) 매출채권 (받을 돈)
entry.addCredit(account(SALES),       event.subtotal());     // 대) 매출
entry.addCredit(account(VAT_PAYABLE), event.taxAmount());    // 대) 부가세예수금
entry.post(...);
```

💡 차변 합(totalAmount) = 대변 합(subtotal + tax). **복식부기는 차변=대변이 항상 맞아야** 한다
(05 FI 의 핵심). 안 맞으면 예외 → 인보이스 발행까지 롤백.

---

## 7. 출하 1건이 코드에서 거치는 순서

```
POST 출하 확정 → DeliveryService.create  [트랜잭션 T 시작]
 1. delivery.ship() + 저장
 2. publishEvent(DeliveryShippedEvent)   ← 아직 commit 안 됨
 3. (commit 직전) BEFORE_COMMIT 리스너들이 @Order 순으로 실행:
     @Order(10) MM  → 재고 차감 + StockMovement(단가 기록)
     @Order(20) FI  → 그 단가 읽어 매출원가 전표
 4. 모두 성공 → commit  /  하나라도 실패 → 전부 롤백
```

---

## 요약 — 도메인 ↔ 구현 대응

| ① 도메인 개념 | ② 구현 |
| --- | --- |
| 한 사건이 여러 모듈 갱신 | `DeliveryShippedEvent` 발행 → MM·FI 리스너 |
| 느슨한 결합 | `DeliveryService` 는 재고·회계를 import 안 함 (이벤트만) |
| 원자성(롤백) | `@TransactionalEventListener(BEFORE_COMMIT)` + REQUIRED 합류 |
| 순서 정합성 | `@Order(10/20)` (재고가 단가 박은 뒤 회계가 읽음) |

> **다음 주제 후보**: 05 FI(복식부기·자동분개) — 방금 본 전표가 무엇이고 왜 차변=대변인지. 또는 08 PP(생산·원가).
