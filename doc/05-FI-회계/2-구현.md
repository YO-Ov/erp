# 05. FI(회계) — ② 구현 편 : 차대 검증과 자동분개를 코드로

> **이 글의 목적**: ① 도메인 편의 두 가지 — **"차변 합 = 대변 합" 불변식**과 **자동분개** — 가
> 실제 코드에서 어떻게 강제되고 생성되는지 본다. 그리고 04 의 이벤트 리스너가 이 자동분개를
> 어떻게 호출하는지(SD → FI 연결)까지 잇는다. 파일은 전부 `com.hwlee.erp.fi.journal` 패키지.

---

## 1. 모델 구조 — 헤더(JournalEntry) + 라인(JournalLine)

전표 하나는 **헤더 1 + 라인 N** 구조다. (Invoice/SalesOrder 와 똑같은 패턴)

```
JournalEntry (헤더)          JournalLine (라인) — 여러 개
 ├ number   JE-20260601-001   ├ account  매출채권(1200)   debit 1,100,000  credit 0
 ├ entryDate 2026-06-01        ├ account  매출(4100)       debit 0          credit 1,000,000
 ├ status   DRAFT→POSTED       └ account  부가세예수금(2200) debit 0         credit 100,000
 ├ sourceType INV  sourceId 42
 └ lines  [ ... ]
```

핵심 설계 두 가지가 여기 들어 있다.

### (a) 한 라인은 차변·대변 중 **한 쪽만** 양수

① 도메인 편에서 "모든 기록은 왼쪽(차변) 또는 오른쪽(대변)" 이라 했다. 코드에선 `JournalLine` 이
`debit`·`credit` **두 컬럼**을 갖되, **한 쪽만 > 0** 이도록 생성자가 막는다:

```java
// JournalLine 생성자 (발췌)
boolean debitPositive  = debit.signum() > 0;
boolean creditPositive = credit.signum() > 0;
if (debitPositive == creditPositive) {   // 둘 다 0 이거나 둘 다 양수면 위반
    throw new IllegalArgumentException(
            "라인은 차변 또는 대변 중 한 쪽만 양수여야 한다. debit=" + debit + ", credit=" + credit);
}
```

> 이 규칙은 **DB CHECK 제약(V24)**으로도 한 번 더 강제된다 — 애플리케이션을 우회해 DB 에 직접
> 꽂아도 깨지지 않게. "한 라인 = 한 방향" 은 그만큼 절대 어기면 안 되는 불변식이다.

또 `account.isPostable()` 도 검사한다 — 계정과목 트리에서 **상위(헤더) 계정**(예: "유동자산" 같은
분류 노드)엔 직접 분개를 달 수 없고, **말단(전기 가능) 계정**에만 달 수 있다.

### (b) 차변/대변 라인은 의도를 드러내는 메서드로 추가

`AutoJournalService` 가 분개를 만들 때 `debit/credit` 컬럼을 직접 다루지 않고, 읽기 좋은 메서드를 쓴다:

```java
public JournalLine addDebit(Account account, BigDecimal amount)  { return addLine(account, amount, ZERO); }
public JournalLine addCredit(Account account, BigDecimal amount) { return addLine(account, ZERO, amount); }
```

→ 분개 코드가 **"차) … 대) …"** 한국식 분개장과 거의 1:1로 읽힌다 (3절에서 확인).

## 2. 핵심 불변식 — `post()` 가 차변 합 = 대변 합을 검증 ⭐

① 도메인 편의 **황금률**이 코드로 사는 곳은 단 한 메서드, `JournalEntry.post()` 다.

```java
public void post(LocalDateTime now) {
    if (status != JournalEntryStatus.DRAFT)  throw new IllegalStateException("DRAFT 전표만 확정 가능");
    if (lines.isEmpty())                      throw new IllegalStateException("라인 없는 전표는 확정 불가");

    BigDecimal totalDebit  = lines.stream().map(JournalLine::getDebit).reduce(ZERO, BigDecimal::add);
    BigDecimal totalCredit = lines.stream().map(JournalLine::getCredit).reduce(ZERO, BigDecimal::add);

    if (totalDebit.compareTo(totalCredit) != 0) {
        throw new UnbalancedJournalException(totalDebit, totalCredit);  // ← 균형 깨지면 전기 거부
    }
    this.status   = JournalEntryStatus.POSTED;
    this.postedAt = now;
}
```

읽을 점:

1. **전기(post) 직전에만 검증한다.** DRAFT 단계에선 차/대가 안 맞아도 된다(작성 중). "확정" 버튼을
   누르는 순간 검증 → 통과해야 `POSTED`. 못 맞으면 `UnbalancedJournalException` 으로 막힌다.
2. `compareTo(...) != 0` — `BigDecimal` 은 `equals` 가 scale(소수 자릿수)까지 비교하므로, 금액 비교는
   반드시 `compareTo`. (`1000` vs `1000.00` 을 같다고 봐야 함)
3. `POSTED` 이후엔 `addLine` 도 막힌다(`status != DRAFT` 체크) — **확정된 전표는 불변**.
4. 취소는 `cancel()` 로 헤더 상태만 `CANCELLED` 로 — 라인을 지우지 않는다. 원장은 append-only,
   바로잡을 땐 역분개를 새 전표로 추가하는 게 회계 정석(① 도메인 편 7절).

> 정리: **복식부기의 무결성 = `post()` 한 군데**. 자동분개든 수동 전표든 전기는 모두 이 문을 통과한다.

## 3. 자동분개 — 사건을 분개로 옮기는 `AutoJournalService`

① 도메인 편의 핵심: **담당자는 업무 행동만, 분개는 시스템이 자동 생성.** 그 변환기가
`AutoJournalService` 다. "범용 규칙 엔진" 대신 **사건마다 전용 메서드**를 둔 게 설계 결정(#7)이다 —
분개 규칙이 자바 코드로 한눈에 읽히기 때문.

### 매출 분개 — `createSalesEntry`

① 도메인 편 사례 ①(세금계산서 발행)이 그대로 코드가 된다:

```java
@Transactional
public JournalEntry createSalesEntry(InvoiceIssuedEvent event) {
    String number = numberGenerator.nextJournalEntryNumber(event.invoiceDate());
    JournalEntry entry = JournalEntry.draft(number, event.invoiceDate(),
            "매출 " + event.number(), JournalSource.INV, event.invoiceId());

    entry.addDebit (account(SystemAccounts.AR),          event.totalAmount());  // 차) 매출채권 110만
    entry.addCredit(account(SystemAccounts.SALES),       event.subtotal());     // 대) 매출      100만
    entry.addCredit(account(SystemAccounts.VAT_PAYABLE), event.taxAmount());    // 대) 부가세    10만

    entry.post(LocalDateTime.now(clock));   // ← 차변 110만 = 대변(100만+10만) 검증 통과 후 POSTED
    return journalRepository.save(entry);
}
```

→ 코드가 분개장과 1:1. `addDebit`/`addCredit` 세 줄이 곧 "차) 매출채권 / 대) 매출 + 부가세" 다.
사건(`InvoiceIssuedEvent`)이 금액(total/subtotal/tax)을 실어 오므로, 메서드는 **어느 계정 어느 쪽에**
꽂을지만 안다. 이게 "사건 → 분개 규칙" 의 코드화다.

### 다른 사건들 — 같은 골격, 계정만 다르다

| 메서드 | 사건 | 분개 (차 / 대) |
| --- | --- | --- |
| `createSalesEntry` | 인보이스 발행 | 차)매출채권 / 대)매출 + 부가세예수금 |
| `createCogsEntry` | 출하 확정 | 차)매출원가 / 대)재고자산 |
| `createPurchaseEntry` | 입고 확정 | 차)재고(제품)+원재료 / 대)매입채무 |
| `createProductionEntry` | 생산 완료 | 차)제품 / 대)원재료 |
| `createReceiptEntry` | 입금 | 차)현금 / 대)매출채권 |
| `createDisbursementEntry` | 출금 | 차)매입채무 / 대)현금 |
| `createPayrollEntry` | 급여 확정 | 차)급여비용+법정복리비 / 대)예수금×2 + 미지급급여 |
| `createSalaryPaymentEntry` | 급여 지급 | 차)미지급급여 / 대)현금 |

전부 **draft → addDebit/addCredit → post → save** 의 같은 골격. 계정과 금액 출처만 다르다.

### 계정코드 → 계정 엔티티 조회

`SystemAccounts.AR` 같은 코드 상수를 실제 `Account` 로 바꾸는 건 작은 헬퍼:

```java
private Account account(String code) {
    try { return accountService.getEntityByCode(code); }
    catch (EntityNotFoundException e) {
        throw new IllegalStateException("시스템 계정 " + code + " 가 마스터에 없다. 시드 점검 필요.", e);
    }
}
```

→ 분개에 쓰는 계정은 마스터에 **반드시 시드되어 있어야** 한다(마이그레이션). 없으면 분개 자체가 실패.

## 4. 04 와 연결 — 이벤트 리스너가 자동분개를 호출한다

여기서 04(모듈 연계)와 05(회계)가 만난다. 영업(SD)은 회계를 직접 호출하지 않는다 —
**"인보이스 발행했다" 고 이벤트만 발행**하고, FI 쪽 리스너가 그걸 듣고 자동분개를 부른다.

```java
// fi/integration/sd/SalesAccountingListener  — "SD 로부터 들어오는 통합" (패키지 위치가 의존 방향)
@Component
@RequiredArgsConstructor
class SalesAccountingListener {
    private final AutoJournalService autoJournal;

    @TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
    void onIssued(InvoiceIssuedEvent event) {
        autoJournal.createSalesEntry(event);   // ← 3절의 매출 자동분개 호출
    }
}
```

04 구현 편에서 본 규칙이 그대로 적용된다:

- **`BEFORE_COMMIT`**: 인보이스 발행 트랜잭션과 **같은 트랜잭션**에서 실행. 그래서 분개가
  `UnbalancedJournalException` 으로 실패하면 **인보이스 발행 자체가 롤백** — "매출은 잡혔는데
  전표가 없는" 불일치가 구조적으로 불가능(① 도메인 편의 원자성).
- **이벤트 소유권**: 이벤트(`InvoiceIssuedEvent`)는 발행자 SD 가, 리스너는 수신자 FI 가 소유.
  패키지 `fi/integration/sd/` 이름이 "SD→FI 방향" 을 드러낸다.

### 순서가 정합성을 좌우하는 경우 — 매출원가(`createCogsEntry`)

매출 분개는 리스너가 하나뿐이라 순서 문제가 없지만, **출하 확정**은 두 리스너가 같은 사건을 듣는다:

```
DeliveryShippedEvent 발행
   ├─ @Order(10) 재고 리스너  : GoodsIssue.post → StockMovement 에 unit_cost(출고 단가) 기록
   └─ @Order(20) 회계 리스너  : 그 StockMovement 를 읽어 매출원가 합산 → createCogsEntry
```

`createCogsEntry` 는 출고 **단가**가 필요한데, 그 단가는 `DeliveryShippedEvent` 본문에 없다.
재고 리스너(@Order 10)가 **먼저** 돌아 `StockMovement.unit_cost` 를 박아두면, 회계 리스너(@Order 20)가
**나중에** 그걸 읽어 합산한다. `@Order` 가 보장하는 실행 순서가 곧 정합성 — 이게 04 대비 05 의 새 포인트다.

```java
// createCogsEntry 발췌 — 재고 리스너가 만들어 둔 StockMovement 를 역추적해 원가 합산
GoodsIssue gi = goodsIssueRepository.findByDeliveryId(event.deliveryId())
        .orElseThrow(() -> new IllegalStateException("재고 리스너(@Order=10)가 먼저 돌지 않았다"));
BigDecimal totalCost = stockMovementRepository
        .findByRefTypeAndRefIdAndReason("GI", gi.getId(), MovementReason.GOODS_ISSUE)
        .stream().map(m -> m.getUnitCost().multiply(m.getQtyDelta().abs()))
        .reduce(BigDecimal.ZERO, BigDecimal::add);
```

## 5. 왜 차대가 "항상" 맞을까 — 라운딩까지 무사한 이유

자동분개가 **절대 `UnbalancedJournalException` 을 던지지 않는** 이유는, 각 메서드의 분개가
**수학적으로 균형이 보장**되도록 짜였기 때문이다. 급여 분개가 좋은 예다:

```
차) 급여비용   Σgross
차) 법정복리비 Σins_company
        대) 예수금-소득세   Σtax
        대) 예수금-사회보험 Σ(ins_emp + ins_company)
        대) 미지급급여      Σnet
```

- 차변 합 = `Σgross + Σins_company`
- 대변 합 = `Σtax + Σ(ins_emp + ins_company) + Σnet`
- 그런데 `net = gross − tax − ins_emp` (실수령액 정의) 이므로 대입하면 **양변이 항상 일치**.

→ 라운딩(원 단위 반올림)을 해도 `net` 을 같은 반올림 값으로 정의하는 한 균형이 안 깨진다.
매출·매입 분개도 마찬가지로 "한 금액을 양쪽에 같이 적는" 구조라 본질적으로 균형이다.
`post()` 의 검증은 그래서 **자동분개엔 안전망**, **수동 전표(사람이 입력)엔 실질적 방어선**으로 작동한다.

## 요약

1. 전표 = **헤더(JournalEntry) + 라인(JournalLine)**. 라인은 **차변·대변 한 쪽만 양수**(생성자 + DB CHECK)
2. **황금률은 `post()` 한 곳**: 차변 합 ≠ 대변 합이면 `UnbalancedJournalException` → 전기 거부.
   `BigDecimal` 비교는 `compareTo`. POSTED 후 라인 불변, 취소는 역분개
3. **자동분개 = `AutoJournalService`**: 사건별 전용 메서드, `addDebit/addCredit` 로 분개장처럼 읽힘.
   8개 사건(매출/매출원가/매입/생산/입금/출금/급여확정/급여지급) 같은 골격
4. **04 연결**: SD 가 이벤트 발행 → `fi/integration/sd` 리스너가 `BEFORE_COMMIT` 으로 자동분개 호출 →
   실패 시 발행까지 롤백(원자성). 매출원가는 `@Order` 로 재고→회계 순서를 보장해 단가를 넘겨받음
5. 자동분개는 **수학적으로 균형이 보장**돼 예외를 안 던진다 — `post()` 검증은 수동 전표의 방어선

> 또는 조감도 표의 다른 주제. 회계는 이로써 "전표의 정체 → 자동분개 → 모듈 연계와의 결합" 까지 한 바퀴.
