# Phase 4 도메인 브리핑 — SD ↔ MM 연계 (모듈을 한 트랜잭션으로 엮기) ⭐

> 7단계 사이클의 **단계 1: 도메인 브리핑**.
> 이 단계의 목적은 "코드를 짜기 전에, 모듈 간 연계가 ERP 의 본질인 이유와 그 구현 패턴을 글로 이해" 하는 것.
> 학습 계획서가 **⭐ ERP 의 핵심** 으로 지정한 Phase. 충분히 읽고 모호한 부분이 있으면 **단계 2 Q&A** 에서 질문.

---

## 0. 한 줄 요약

> **한 비즈니스 사건이 여러 모듈의 상태를 한 트랜잭션으로 동시에 바꾸는 것 — 그게 ERP 의 본질이다.**
> Phase 2 의 출하(Delivery) 와 Phase 3 의 출고(GoodsIssue) 는 지금까지 **서로 모르는 사이**. Phase 4 가 두 모듈을 한 흐름으로 엮는다 — 출하 확정 한 번이 재고를 실제로 차감한다.

---

## 1. ERP 가 단순 CRUD 와 다른 결정적 이유

**CRUD 시스템** (게시판, 쇼핑몰의 상품 관리 등): 한 화면의 동작이 한 테이블만 건드림. "글 등록 → 글 테이블에 INSERT 하나". 끝.

**ERP**: 한 비즈니스 사건이 **여러 모듈의 상태를 동시에** 바꿈. 한 사건 = 여러 테이블의 일관성 있는 변화. 그것도 **한 트랜잭션 안에서**.

예시 — 노트북 6대 출하 확정 한 번이 일으키는 일:

| 모듈 | 어떤 변화 |
| --- | --- |
| SD (영업) | `Delivery.status` = SHIPPED, `SalesOrderLine.shipped_qty += 6` |
| **MM (재고)** | **`Stock.qty_on_hand -= 6`, `StockMovement` 음수 행 추가** |
| FI (회계, Phase 5) | (장차) 매출원가 분개 + 재고자산 감소 분개 |
| HR (실적, Phase 8) | (장차) 출하 담당자의 처리 건수 +1 |

**모두 한 트랜잭션** — 어느 하나가 실패하면 다 롤백. 절반만 일어나면 데이터가 어긋남:
- 출하는 됐는데 재고는 안 줄었음 → 다음 출하가 잘못된 가용 재고로 검증.
- 재고는 줄었는데 매출원가는 안 잡힘 → 손익 계산서 거짓말.

→ **"왜 ERP 가 단순 CRUD 가 아닌가"** 의 답이 이 한 줄. Phase 4 가 그 차이를 코드로 처음 만나는 자리.

## 2. 지금까지의 결손 — Phase 2 의 출하가 안 한 일

Phase 2 의 `Delivery.ship()` 은 SO 라인의 `shipped_qty` 만 갱신했다. 시연 STEP 6 에서 노트북 6대 출하했을 때:

```
SO 라인:    order_qty=10, shipped_qty=6 ← 갱신됨
Stock:      qty_on_hand=??? ← 건드리지 않음
```

"노트북 6대 보냈는데 창고 재고는 그대로." 이게 실무에선 사고:
- 영업이 "10대 출하" 보고 → 실제로 10대 못 보냄 (재고 부족).
- 다른 영업이 같은 노트북 7대 수주 → 가용 검증에서 통과 (재고 안 줄었으니) → 또 사고.
- 회계가 "이번 달 매출원가" 계산 → Stock 의 평균 단가가 의미 없는 값 (실 출고가 반영 안 됨).

Phase 3 이 MM 모듈을 만들었지만, Phase 2 의 출하는 여전히 MM 을 모른다. **Phase 4 가 그 다리를 놓는다**.

## 3. Phase 4 가 메우는 흐름

```
[Before — Phase 2 까지]
POST /api/deliveries/{id}/ship
   │
   ├─ Delivery.status = SHIPPED
   └─ SalesOrderLine.shipped_qty += qty
   ←━━━━━━━━━━━━ Stock 은 안 건드림

[After — Phase 4]
POST /api/deliveries/{id}/ship
   │
   ├─ Delivery.status = SHIPPED
   ├─ SalesOrderLine.shipped_qty += qty
   ├─ ⭐ GoodsIssue 자동 생성 + post (DRAFT → POSTED)
   │      ├─ Stock.qty_on_hand -= qty  (비관 락)
   │      └─ StockMovement(-) 적재
   │
   └─ 한 @Transactional 안 — 어느 하나 실패하면 다 롤백
```

**핵심**: 영업 사용자는 `POST /api/deliveries/{id}/ship` 한 번만 부른다. 시스템 내부에서 SD/MM 두 모듈이 협력해 한 트랜잭션을 완성. 사용자에겐 단일 동작.

## 4. 두 가지 통합 패턴

모듈 간 연계의 두 가지 방향. 각각 트레이드오프가 있다.

### 4-1. 직접 호출 (강결합) — 가장 단순

```java
class DeliveryService {
    private final GoodsIssueService goodsIssueService;   // ← 직접 의존

    public void ship(Long id) {
        Delivery dlv = ...;
        dlv.ship();
        order.recordShipment(...);
        // ⭐ 직접 호출
        goodsIssueService.createAndPostFromDelivery(dlv);
    }
}
```

장점:
- 코드 흐름이 한눈에 보임. 직선적.
- 디버깅 쉬움 — 스택 트레이스가 곧 흐름.
- 트랜잭션 경계가 자명 (`@Transactional` 하나).

단점:
- SD 모듈이 MM 모듈을 알아야 함 — **모듈 간 의존 방향이 박힘**.
- MM 이 사라지면 SD 도 깨짐 — 격리 안 됨.
- 추후 같은 사건에 새 모듈 (회계, HR, MES) 이 반응하려면 SD 코드를 매번 수정.

### 4-2. 도메인 이벤트 (느슨한 결합) — Spring Application Events

```java
class DeliveryService {
    private final ApplicationEventPublisher events;   // ← Spring 표준

    public void ship(Long id) {
        Delivery dlv = ...;
        dlv.ship();
        order.recordShipment(...);
        // ⭐ 사건 발행 — "누가 듣는지 모름"
        events.publishEvent(new DeliveryShippedEvent(dlv.getId(), ...));
    }
}

// MM 측 (다른 패키지)
@Component
class StockDeductionListener {
    @TransactionalEventListener(phase = BEFORE_COMMIT)
    void on(DeliveryShippedEvent event) {
        goodsIssueService.createAndPostFromDelivery(event);
    }
}
```

장점:
- **SD 가 MM 을 모름** — 이벤트만 발행. 누가 듣든 상관 X.
- 새 모듈이 같은 사건에 반응하려면 listener 하나 추가하면 끝 (SD 코드 무수정).
- 모듈 격리 — MM 이 사라져도 SD 는 그대로.

단점:
- 흐름이 흩어짐 — "출하 확정하면 무슨 일이 일어나?" 답이 코드 한 군데에 없음.
- 디버깅 어려움 — 이벤트 발행 후 어디로 가는지 추적이 무거움.
- `@TransactionalEventListener` 의 phase 옵션 이해 필요 (다음 절).

### 4-3. `@TransactionalEventListener` — 트랜잭션 경계와 이벤트의 만남

```java
@TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
void on(DeliveryShippedEvent event) { ... }
```

`phase` 옵션의 4가지:

| phase | 호출 시점 | 같은 트랜잭션? |
| --- | --- | --- |
| `BEFORE_COMMIT` | commit 직전 | **예** — 실패 시 전체 롤백 |
| `AFTER_COMMIT` (기본) | commit 후 | 아니오 — 별도 트랜잭션 |
| `AFTER_ROLLBACK` | rollback 후 | 아니오 |
| `AFTER_COMPLETION` | commit/rollback 후 | 아니오 |

Phase 4 의 핵심 선택:
- **재고 차감이 출하와 같은 트랜잭션에 묶여야** → `BEFORE_COMMIT`.
- 만약 `AFTER_COMMIT` 으로 했다가 재고 차감 실패하면 → 출하는 이미 commit, 재고는 안 빠짐 → **정합성 깨짐**.

→ "사건 발행" 의 가벼움과 "한 트랜잭션 원자성" 을 동시에 잡는 패턴. 이게 Spring Events 의 진가.

### 4-4. Phase 4 의 선택 — 어떤 패턴을 쓸까

학습 계획서 § Phase 4: "Spring Application Events 로 모듈 간 결합도 낮추기".

→ **이벤트 패턴** 으로 진행. 직접 호출의 단순함을 포기하는 대신:
- 모듈 격리 학습.
- Phase 5 (회계) 의 자동 분개 listener 가 같은 사건에 자연스럽게 붙음.
- `@TransactionalEventListener` 의 phase 의미를 손에 잡음.

> 💡 처음엔 직접 호출로 만들었다가 이벤트로 리팩토링하는 흐름도 학습 가치가 있음. 설계 제안 단계에서 결정.

## 5. 트랜잭션 경계 설계 — `@Transactional` 전파 옵션

한 트랜잭션 vs 두 트랜잭션의 선택은 **전파 옵션 (Propagation)** 으로 표현.

| 옵션 | 의미 |
| --- | --- |
| `REQUIRED` (기본) | 기존 트랜잭션 있으면 참여, 없으면 새로 시작 |
| `REQUIRES_NEW` | **항상 새 트랜잭션** — 기존 트랜잭션 일시 정지 |
| `SUPPORTS` | 있으면 참여, 없으면 트랜잭션 없이 진행 |
| `MANDATORY` | 반드시 기존 트랜잭션 안 — 없으면 예외 |
| `NEVER` | 트랜잭션 안에서 호출되면 예외 |

Phase 4 의 핵심 결정:
- `DeliveryService.ship` 의 `@Transactional` 안에서 listener 가 `GoodsIssueService` 호출.
- listener 도 `@Transactional` (`REQUIRED` 기본) → **같은 트랜잭션 참여**.
- 어느 하나 실패하면 전체 롤백.

> 만약 listener 가 `REQUIRES_NEW` 였다면 — 출하 트랜잭션과 재고 트랜잭션이 독립. 출하 실패해도 재고는 commit 된 상태로 남음. **부분 상태 = 사고**.

→ 한 비즈니스 사건의 정합성은 거의 항상 `REQUIRED`. `REQUIRES_NEW` 는 "로그/감사 기록처럼 본 흐름과 무관한 곁가지" 에만 적절.

## 6. Phase 4 의 새 비즈니스 규칙

### 6-1. 출하 확정 시 가용 재고 부족이면 거부

Phase 2 에서는 OK 였던 출하가 이제는 막힐 수 있다:

```
POST /api/deliveries/{id}/ship  (Delivery 의 10대 출하 시도)
   │
   ├─ Delivery.ship() — 자기 검증 OK
   ├─ event: DeliveryShippedEvent 발행
   │
   └─ listener: goodsIssueService.createAndPostFromDelivery(...)
        │
        └─ GoodsIssue.post() → stock.issue(10) → qty_on_hand=5 < 10
             → InsufficientStockException
             → 트랜잭션 전체 롤백
             → 클라이언트에 409 INSUFFICIENT_STOCK
             → Delivery 도 SHIPPED 되지 않음 (롤백)
```

**사용자 경험**: 출하 시도 → "재고 5대만 있음, 10대 못 보냅니다" 응답. SO 라인은 그대로, 재고도 그대로, 출하도 안 됨. **모든 게 시도 전 상태**.

### 6-2. 출하 취소 시 재고 복원

`DELETE /api/deliveries/{id}/cancel` (Phase 2 에서 만든 취소 흐름) 가 이제 추가로:
- 연결된 `GoodsIssue` 도 취소.
- `Stock` 수량 복원.
- `StockMovement` 에 ADJUSTMENT_PLUS 정정 행 추가 (append-only).

이것도 한 트랜잭션 안에서.

### 6-3. 의도적으로 안 하는 것

- **수주 확정 시 가용 재고 사전 체크** — 학습 계획서 §Phase 4 에는 없음. "수주 확정 = 신용한도 검증" 만 (Phase 2). 가용 재고는 출하 시점에. 실무에서도 양쪽 패턴 다 있음 — 학습 범위 단순화.
- **입고 측 통합** — 입고(GoodsReceipt) 는 Phase 3 의 `post` 에서 이미 자동으로 Stock 갱신함. Phase 4 에서 추가할 게 없음.

## 9. Phase 2/3 와의 누적성

```
Phase 1: 마스터 데이터 (코드 자동 발급, Soft Delete, Auditing)
   ↓
Phase 2: 트랜잭션 (헤더-라인, 상태 머신, 부분 누적, 신용한도 race 노출)
   ↓
Phase 3: 재고 (동시성, 가중평균, 원장, INSUFFICIENT_STOCK)
   ↓
Phase 4: ⭐ 위 모든 것을 한 트랜잭션으로 엮기
            ├─ 헤더-라인 (Delivery + GoodsIssue 둘 다)
            ├─ 상태 머신 (양쪽 모두 DRAFT/POSTED/CANCELLED 전이)
            ├─ 부분 누적 (Delivery 라인별 출하)
            ├─ 동시성 보호 (출하 시 비관 락이 한 번 더 일함)
            └─ 한 트랜잭션 원자성 + 이벤트 패턴 (새)
```

→ **누적된 패턴이 한 흐름에 다 등장**. Phase 4 는 새 개념이 적지만, 기존 패턴들을 한 그림으로 합치는 자리.

## 10. 사용자가 이번 브리핑에서 잡아야 할 핵심 5가지

1. **ERP = 한 비즈니스 사건이 여러 모듈의 상태를 한 트랜잭션으로 동시에 바꾸는 시스템**. 단순 CRUD 가 아닌 결정적 이유.
2. **Phase 2 의 출하가 재고를 안 건드린 게 사고의 씨앗** — Phase 4 가 그 다리를 놓는다.
3. **직접 호출 vs 이벤트 — 결합도와 추적성의 트레이드오프**. Phase 4 는 이벤트 선택, Phase 5 가 같은 패턴으로 회계 분개 listener 를 얹을 예정.
4. **`@TransactionalEventListener(BEFORE_COMMIT)` 의 의미** — 이벤트 발행과 한 트랜잭션 원자성을 동시에. AFTER_COMMIT 이면 정합성 깨짐.
5. **Phase 4 후의 출하 시도 = SO + Delivery + Stock + StockMovement 4 테이블이 한 트랜잭션에서 변함**. 어느 하나라도 실패하면 모두 시도 전 상태로 복귀.

## 11. 다음 단계 (단계 2: 이해 확인 Q&A)

이 브리핑을 읽고 **모호하거나 의문이 있는 부분을 질문** 해 주세요. 예시 질문 ("정답을 모르는데 어떤 걸 물어야 할지 모를 때" 출발점):

- "굳이 이벤트 패턴을 쓰는 이유 — Phase 3 시연에서 GoodsIssue 를 직접 호출해도 잘 됐는데, 뭐가 그렇게 다른가?"
- "`@TransactionalEventListener` 와 그냥 `@EventListener` 의 차이를 코드로 보면 어떻게 다른가?"
- "`BEFORE_COMMIT` 에서 listener 가 예외 던지면 그 예외가 호출자에 그대로 전파되나? 어떻게 받나?"
- "한 출하에 여러 라인(여러 상품) 이 있으면 한 GoodsIssue 가 다 묶이나, 라인마다 GoodsIssue 하나씩 생기나?"
