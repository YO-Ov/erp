# 1/6. 도메인 이벤트 — Spring Application Events 와 record

> Phase 4 의 출발점. "한 비즈니스 사건이 여러 모듈을 건드린다" 를 **직접 호출이 아니라 사건 발행으로** 풀어내는 첫 코드.
> 설계서 §2 "도메인 이벤트 설계" + §0의 핵심 결정 #1·#2 "Spring Application Events + record 2종" 의 실제 구현.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/sd/delivery/
├─ DeliveryService.java               (이벤트 발행 — create / cancel)
└─ event/                              ← 새 서브패키지 ⭐
   ├─ DeliveryShippedEvent.java       (출하 확정 사건 — record)
   └─ DeliveryCancelledEvent.java     (출하 취소 사건 — record)
```

---

## 🔥 왜 이벤트인가 — "출하했다" 한 사건, 두 모듈의 변화

출하가 확정되면 두 가지 일이 동시에 일어나야 한다.

```
[Delivery 가 SHIPPED 로 확정]
   ├─ SD 측: SalesOrder 라인의 shipped_qty 누적 + 헤더 상태 전이
   └─ MM 측: GoodsIssue 자동 생성 + 재고 차감 + 원장 적재
```

순진하게 짜면 `DeliveryService` 가 `GoodsIssueService` 를 **직접 호출** 한다.

```java
// ❌ 직접 호출 — SD 가 MM 을 안다
deliveryService {
    goodsIssueService.createAndPostFromDelivery(...);  // SD → MM 의존 발생
}
```

이러면 SD 모듈이 MM 모듈에 묶인다. 나중에 "출하하면 회계 분개도 만들어라" 가 추가되면 SD 가 회계 모듈까지 알아야 한다. SD 의 핵심 로직이 점점 다른 모듈 호출로 더러워진다.

→ 대신 SD 는 **"출하가 확정됐다" 는 사건만 알린다**. 누가 그 사건에 반응하는지는 SD 의 관심사가 아니다.

```java
// ✅ 이벤트 발행 — SD 는 "사건이 일어났다" 고 알릴 뿐
events.publishEvent(new DeliveryShippedEvent(...));
```

MM 이 "그 사건에 관심 있어요" 하고 구독한다. SD 는 구독자가 0명인지 3명인지 모른다. **구독자에 대한 결합도 0**.

> 💡 의존 방향은 `MM → SD` 단방향이 된다 (MM 이 SD 의 이벤트 타입을 import). SD 는 MM 을 전혀 모른다. 이 방향이 왜 중요한지는 **3편 (패키지 방향 의존성)** 에서 본격적으로 다룬다. 이 편에서는 "발행자는 구독자를 모른다" 만 가볍게 잡고 간다.

## 🔥 두 이벤트가 Java `record` 인 이유

```java
public record DeliveryShippedEvent(
        Long deliveryId,
        Long warehouseId,
        LocalDate shippedDate,
        List<Line> lines
) {
    public record Line(Long itemId, BigDecimal quantity) {}
}
```

```java
public record DeliveryCancelledEvent(
        Long deliveryId
) {}
```

이벤트는 **"이미 일어난 사건의 스냅샷"** 이다. 사건은 과거형이고, 과거는 바뀌지 않는다. 그래서 이벤트 객체의 본질은 **불변 값 객체 (immutable value object)**.

`record` 가 이 본질에 정확히 맞는다.

| record 의 성질 | 이벤트에 왜 맞는가 |
| --- | --- |
| 모든 필드 `final` (불변) | 발행된 사건은 수정 불가. listener 가 값을 바꿔 다른 listener 를 오염시킬 수 없음 |
| 생성자/접근자/`equals`/`hashCode`/`toString` 자동 | 단순 데이터 전달 객체 — 보일러플레이트 0 |
| 값 동등성 (필드가 같으면 같은 객체) | 사건의 정체성 = 그 안의 데이터. "같은 내용 = 같은 사건" |

> 💡 엔티티 (`Delivery`, `GoodsIssue`) 는 `class` + 식별자 동등성 (같은 id = 같은 객체) 이고, 이벤트는 `record` + 값 동등성이다. **"살아 있는 상태를 가진 객체" 와 "찍어 둔 사건의 사진"** 의 차이. 이벤트를 엔티티로 짜면 영속성 컨텍스트와 얽혀 오히려 위험하다.

### 중첩 `record Line` 의 자리

`DeliveryShippedEvent.Line(itemId, quantity)` 가 이벤트 안에 중첩돼 있다. 이 `Line` 은 **오직 출하 이벤트의 일부로서만 의미** 가 있다 — 따로 떠다닐 일이 없는 값. 그래서 바깥에 별도 파일로 빼지 않고 이벤트 안에 둔다. 이름도 `DeliveryShippedEvent.Line` 으로 소속이 분명해진다.

## 🔥 본문 설계 — 두 이벤트가 담는 것이 다른 이유

여기가 이 편의 가장 도메인적인 결정이다. 두 이벤트의 본문이 비대칭이다.

```
DeliveryShippedEvent  : deliveryId + warehouseId + shippedDate + lines[(itemId, qty)]   ← 두껍다
DeliveryCancelledEvent: deliveryId                                                       ← 얇다
```

왜 한쪽은 라인까지 다 담고, 한쪽은 ID 하나뿐인가? 설계서 §2.2 의 결정이다.

### 출하 확정 — "ID + 핵심 데이터" (옵션 B)

`DeliveryShippedEvent` 의 javadoc 이 직접 말한다.

```java
/**
 * <p>본문은 <b>ID + 핵심 라인 정보</b>를 함께 담는다(설계 §2.2 옵션 B). listener 가
 * 같은 트랜잭션 안에서 즉시 사용하므로 다시 DB 를 조회할 필요가 없고, 일관성 우려도 없다.
 */
```

출하 listener (MM) 는 **새 GoodsIssue 를 만들어야** 한다. 만들려면 "어느 창고 (warehouseId)", "어떤 품목을 몇 개 (lines)", "언제 (shippedDate)" 가 전부 필요하다.

이걸 이벤트에 담아 두면 listener 가 **Delivery 를 다시 조회하지 않아도 된다**. 만약 `deliveryId` 만 담았다면 listener 가 `deliveryRepository.findById(...)` 로 라인을 다시 끌어와야 한다 — 쿼리 추가 + 트랜잭션이 길어짐.

> 💡 "ID 만 담으면 항상 최신 상태를 다시 조회하니 일관성이 더 좋지 않나?" — 일반적으로는 맞다. 하지만 이 listener 는 **같은 트랜잭션 안에서 commit 직전에** 도는 (`BEFORE_COMMIT`, 2편 주제). 발행 시점과 처리 시점 사이에 다른 트랜잭션이 끼어들 틈이 없으므로 스냅샷이 곧 최신이다. **일관성 우려가 없으니 옵션 B 의 단점이 사라진다.**

### 출하 취소 — "deliveryId 만" (역추적)

`DeliveryCancelledEvent` 의 javadoc.

```java
/**
 * <p>{@link DeliveryShippedEvent} 와 달리 <b>라인 정보를 담지 않는다</b>(설계 §2.2):
 * 취소 listener 는 {@code deliveryId} 로 연결된 GoodsIssue 를 직접 찾아 그 GI 의 라인을
 * 복원하는 게 더 깔끔하다(라인 매칭 로직 불필요). 차감했던 창고도 GI 가 알고 있으므로
 * 이벤트에 warehouseId 조차 필요 없다.
 */
```

취소 listener 가 할 일은 "출하 때 차감했던 재고를 **그대로 되돌리기**" 다. 그런데 그때 차감한 정보 (창고, 품목, 수량) 는 이미 **GoodsIssue 에 통째로 박혀 있다**.

Phase 4 의 GoodsIssue 는 `delivery_id` FK 로 자기 출처 Delivery 를 가리킨다 (설계 §4, 5편 주제). 그러니 취소 listener 는 `deliveryId` 하나만 받으면:

```
deliveryId  →  goodsIssueRepository.findByDeliveryId(...)  →  그 GI 의 라인을 그대로 복원
```

이벤트에 라인을 또 담으면 **두 군데에 같은 정보가 생긴다** (이벤트의 라인 vs GI 의 라인). 둘이 어긋날 위험만 생기고 이득은 없다. **차감의 원본 = GI** 이므로, 취소는 GI 를 진실의 원천으로 삼는 게 맞다.

> ⚠️ 설계 초안 (§6.3) 에서는 `DeliveryCancelledEvent(deliveryId, warehouseId)` 로 warehouseId 도 담을지 고민했다. 실제 구현은 **deliveryId 만** 으로 확정됐다 — 창고조차 GI 가 알고 있으므로 이벤트에 넣을 이유가 없다는 결론. javadoc 마지막 줄이 그 결정을 못 박는다.

### 대칭이 아닌 게 정상

| | 출하 확정 | 출하 취소 |
| --- | --- | --- |
| listener 가 할 일 | GI 를 **새로 만든다** | 기존 GI 를 **되돌린다** |
| 필요한 데이터의 출처 | 아직 GI 가 없음 → 이벤트가 줘야 함 | 이미 GI 에 있음 → 이벤트는 ID 만 |
| 이벤트 본문 | 두껍다 (lines 포함) | 얇다 (deliveryId) |

→ "두 이벤트니까 모양을 맞춰야 한다" 는 강박은 틀렸다. **각 사건이 일으킬 반응에 꼭 필요한 만큼만** 담는다.

## 🔥 발행 코드 — 트랜잭션 안에서 `events.publishEvent(...)`

발행은 `DeliveryService` 가 한다. 먼저 발행기 주입부터.

```java
import org.springframework.context.ApplicationEventPublisher;

@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class DeliveryService {

    private final DeliveryRepository repository;
    private final DeliveryMapper mapper;
    private final SalesOrderRepository salesOrderRepository;
    private final TransactionNumberGenerator numberGenerator;
    private final ApplicationEventPublisher events;   // ← Spring 이 자동 주입
```

`ApplicationEventPublisher` 는 Spring 이 컨테이너에 항상 등록해 두는 기반 빈. `@RequiredArgsConstructor` 가 `final` 필드로 생성자 주입을 만들어 준다. 별도 설정 없이 주입만 받으면 끝이다.

### `create` — 출하 확정 발행

```java
@Transactional
public DeliveryResponse create(DeliveryCreateRequest req) {
    SalesOrder order = salesOrderRepository.findById(req.salesOrderId())
            .orElseThrow(() -> new EntityNotFoundException("SalesOrder not found: id=" + req.salesOrderId()));

    String number = numberGenerator.nextDeliveryNumber(req.shippedDate());
    Delivery delivery = Delivery.draft(number, order, req.warehouseId(), req.shippedDate());

    for (DeliveryLineRequest lineReq : req.lines()) {
        SalesOrderLine sol = order.findLineById(lineReq.salesOrderLineId());
        delivery.addLine(sol, lineReq.quantity());
    }

    delivery.ship();
    // SO 라인의 shipped_qty 누적 + SO 헤더 상태 전이 (같은 트랜잭션)
    for (DeliveryLine dline : delivery.getLines()) {
        order.recordShipment(dline.getSalesOrderLine(), dline.getQuantity());
    }

    repository.save(delivery);

    // ⭐ Phase 4 — 출하 확정 사건 발행. MM 의 @TransactionalEventListener(BEFORE_COMMIT) 가
    // 같은 트랜잭션 안에서 GoodsIssue 자동 생성 + 재고 차감. 가용 부족이면 여기까지 전부 롤백.
    events.publishEvent(new DeliveryShippedEvent(
            delivery.getId(), delivery.getWarehouseId(), delivery.getShippedDate(),
            toEventLines(delivery)));

    return mapper.toResponse(delivery);
}
```

발행 위치가 중요하다.

1. **메서드 전체가 `@Transactional`** — 발행은 이 트랜잭션 **안에서** 일어난다.
2. **`repository.save(delivery)` 다음에 발행** — `delivery.getId()` 가 채번된 뒤라야 이벤트에 유효한 ID 가 담긴다. (실제로는 BEFORE_COMMIT 시점에 flush 되지만, ID 가 필요하므로 save 후에 발행하는 게 흐름상 자연스럽다.)
3. **발행은 "출하 확정 처리가 다 끝난 다음"** — SD 자기 일 (`delivery.ship()`, `order.recordShipment(...)`) 을 먼저 마치고, 마지막에 "이제 사건이 일어났다" 고 알린다.

> 💡 `publishEvent` 는 동기다. 이 줄에서 등록된 listener 들이 호출된다 (`@Async` 안 씀 — 설계 #8). 다만 `@TransactionalEventListener(BEFORE_COMMIT)` 이면 즉시 실행이 아니라 **commit 직전으로 미뤄진다**. 이 미묘한 타이밍이 2편의 핵심 주제다. 여기서는 "발행 = 같은 트랜잭션 안에서의 사건 알림" 까지만.

### `cancel` — 출하 취소 발행

```java
@Transactional
public DeliveryResponse cancel(Long id) {
    Delivery delivery = getOrThrow(id);
    SalesOrder order = delivery.getSalesOrder();
    delivery.cancel();
    for (DeliveryLine dline : delivery.getLines()) {
        order.cancelShipment(dline.getSalesOrderLine(), dline.getQuantity());
    }

    // ⭐ Phase 4 — 출하 취소 사건 발행. MM 리스너가 연결된 GoodsIssue 를 취소하고 재고를 복원.
    events.publishEvent(new DeliveryCancelledEvent(delivery.getId()));

    return mapper.toResponse(delivery);
}
```

여기서도 SD 자기 일 (`delivery.cancel()`, `order.cancelShipment(...)`) 을 먼저 끝내고, `delivery.getId()` 하나만 담아 발행한다. 라인을 모으는 코드가 없다 — 위에서 본 대로 취소 listener 가 GI 에서 알아서 가져가기 때문.

## 🔥 `toEventLines` — DeliveryLine 을 이벤트 Line 으로 변환

확정 이벤트가 담는 `lines` 는 이 헬퍼가 만든다.

```java
private static List<DeliveryShippedEvent.Line> toEventLines(Delivery delivery) {
    return delivery.getLines().stream()
            .map(dline -> new DeliveryShippedEvent.Line(
                    dline.getSalesOrderLine().getItem().getId(), dline.getQuantity()))
            .toList();
}
```

`DeliveryLine` (엔티티) → `DeliveryShippedEvent.Line` (이벤트 값) 으로의 **번역**.

```
DeliveryLine                              DeliveryShippedEvent.Line
─────────────────────────                 ──────────────────────────
salesOrderLine ──getItem()──getId()  ──▶  itemId
quantity                             ──▶  quantity
```

눈여겨볼 점.

- **품목 ID 가 한 단계 깊다** — `DeliveryLine` 은 품목을 직접 들지 않고 `SalesOrderLine` 을 통해 가리킨다 (`dline.getSalesOrderLine().getItem().getId()`). 출하 라인이 "어느 수주 라인에 대한 출하인지" 로 정의되기 때문. listener 에게는 그 체인이 필요 없으니 여기서 `itemId` 로 납작하게 펴서 넘긴다.
- **엔티티를 이벤트에 직접 넣지 않는다** — `Item` 객체나 `DeliveryLine` 을 그대로 담지 않고, ID 와 수량이라는 **원시 값** 만 뽑는다. 이벤트는 영속성 컨텍스트에서 분리된 순수 스냅샷이어야 LAZY 로딩·detached 문제에 안 걸린다.
- **`static` 헬퍼** — 인스턴스 상태에 의존하지 않는 순수 변환이라 `static`.

> 💡 이게 "이벤트 본문을 가볍게 유지한다" (설계 #3) 의 실제 모습. 엔티티 그래프를 통째로 안 넘기고, listener 가 필요한 최소 값 `(itemId, quantity)` 만 추려서 넘긴다.

## 🔥 전체 그림 — 발행자는 구독자를 모른다

```
                        sd/delivery/  (발행자, MM 을 모름)
   ┌─────────────────────────────────────────────────────────┐
   │  DeliveryService.create()           @Transactional       │
   │     ├─ delivery.ship()                                   │
   │     ├─ order.recordShipment(...)                         │
   │     ├─ repository.save(delivery)                         │
   │     └─ events.publishEvent(                              │
   │            new DeliveryShippedEvent(id, wh, date, lines))│
   │                          │                               │
   └──────────────────────────┼───────────────────────────────┘
                              │  "출하가 확정됐다"  (사건만 던짐)
                              ▼
                  ┌───────────────────────┐
                  │ ApplicationContext     │  ← Spring 이 사건을
                  │ (이벤트 디스패처)        │     구독자에게 전달
                  └───────────┬───────────┘
                              │  누가 듣는지는 SD 의 관심사 아님
                              ▼
              mm/integration/sd/  (구독자, SD 의 이벤트를 import)
   ┌─────────────────────────────────────────────────────────┐
   │  DeliveryEventListener.onShipped(event)   ← 2편·3편 주제  │
   │     └─ GoodsIssue 생성 + 재고 차감                        │
   └─────────────────────────────────────────────────────────┘
```

- SD 의 코드 어디에도 `GoodsIssueService` 나 `mm.*` import 가 없다 — `DeliveryService` 의 import 목록을 보면 전부 `sd.*` + Spring 기반 클래스뿐.
- SD 는 `ApplicationEventPublisher` 에게 사건을 넘길 뿐, 그 사건이 어디로 가는지 모른다.
- 새 구독자 (예: Phase 5 회계 분개 listener) 를 추가해도 **SD 코드는 한 줄도 안 바뀐다**. 그냥 새 `@EventListener` 클래스를 MM/회계 쪽에 하나 더 두면 된다.

→ 이게 "모듈을 느슨하게 연결한다 (loose coupling)" 의 핵심. 사건은 발행자가 정의하고, 반응은 구독자가 각자 책임진다.
