# 3/6. 패키지 방향 의존성 — SD→MM 가 아니라 MM←SD

> Phase 4 의 진짜 주제는 "이벤트 기술" 이 아니라 **모듈 경계와 의존 방향** 이다. 두 모듈이 한 트랜잭션 안에서 협력하면서도 **자바 코드 레벨의 의존은 단 한 방향 (`MM → SD`)** 으로만 흐르게 만든 결정을 푼다.
> 설계서 §2.3 (이벤트는 발행자 소유) + §3.1 (`mm/integration/sd/` 의 자리) + §7 (패키지 구조가 의존 방향을 표현) 의 실제 구현.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/
├─ sd/delivery/
│  ├─ Delivery.java                          (warehouseId — Long, ManyToOne 아님 ⭐)
│  └─ event/
│     └─ DeliveryShippedEvent.java           (이벤트 = SD 가 소유)
└─ mm/
   ├─ goodsissue/
   │  └─ GoodsIssue.java                      (deliveryId — Long, ManyToOne 아님 ⭐)
   └─ integration/sd/
      └─ DeliveryEventListener.java           (리스너 = MM 이 소유, SD 를 import)
```

---

## 🔥 의존 방향이란 무엇인가 — "누가 누구를 import 하는가"

자바에서 모듈 A 가 모듈 B 에 **의존한다** 는 건, A 의 어떤 파일이 B 의 타입을 `import` 한다는 뜻이다. 컴파일 시점에 A 는 B 가 있어야만 빌드된다. B 가 바뀌면 A 가 흔들린다.

Phase 4 는 SD (영업·출하) 와 MM (자재·재고) 두 모듈을 엮는다. 출하가 확정되면 재고가 빠져야 하니까. 그런데 **누가 누구를 알아야 하는가** 가 설계의 갈림길이다.

- 순진한 방법: `DeliveryService` 가 `GoodsIssueService` 를 직접 주입받아 호출 → **SD 가 MM 을 import** → `SD → MM`.
- Phase 4 의 방법: SD 가 **이벤트만 발행** 하고, MM 이 그 이벤트를 **구독** → **MM 이 SD 를 import** → `MM → SD`.

두 방향 다 "한 방향" 이긴 하다. 그런데 왜 굳이 `MM → SD` 를 골랐나? 그리고 그 한 방향을 지키기 위해 **엔티티 필드까지 `Long` 으로 깎아낸** 결정이 이 편의 핵심이다.

> 💡 의존이 양방향이 되면 (`SD → MM` 이면서 `MM → SD`) 두 모듈은 사실상 한 덩어리다. 따로 떼어 테스트할 수도, 한쪽만 이해할 수도 없다. **순환 의존** 은 모듈화의 가장 흔한 실패. Phase 4 의 모든 결정이 "이 순환을 절대 안 만든다" 한 줄로 수렴한다.

## 🔥 이벤트는 발행자(SD)가 소유한다 — `sd/delivery/event/`

```java
package com.hwlee.erp.sd.delivery.event;   // ← SD 모듈 안

public record DeliveryShippedEvent(
        Long deliveryId,
        Long warehouseId,
        LocalDate shippedDate,
        List<Line> lines
) {
    public record Line(Long itemId, BigDecimal quantity) {}
}
```

이벤트 타입이 `sd/delivery/event/` 에 산다. **출하라는 사건의 주인은 SD** 이기 때문이다. SD 는 "내가 출하를 확정했다" 를 세상에 알릴 뿐, **누가 그 소식을 듣는지 모른다.** Javadoc 에 그대로 박혀 있다:

> *"이벤트는 발행자(SD)가 소유한다. MM 모듈이 이 타입을 import 해서 구독할 뿐, SD 는 누가 듣는지 모른다 — 의존 방향은 `MM → SD` 단방향."*

여기서 본문이 전부 `Long` (deliveryId, warehouseId, itemId) 과 원시 타입 (`LocalDate`, `BigDecimal`) 으로만 채워진 점에 주목하라. 이벤트가 `Warehouse` 나 `GoodsIssue` 같은 MM 엔티티를 들고 다녔다면, SD 패키지가 MM 을 import 하게 되어 그 자리에서 `SD → MM` 의존이 생긴다. **이벤트 본문조차 ID 로만 말한다.**

## 🔥 리스너는 수신자(MM)가 소유한다 — `mm/integration/sd/`

```java
package com.hwlee.erp.mm.integration.sd;          // ← MM 모듈 안, "SD 로부터" 자리

import com.hwlee.erp.mm.goodsissue.GoodsIssueService;
import com.hwlee.erp.sd.delivery.event.DeliveryCancelledEvent;   // ← MM 이 SD 를 import
import com.hwlee.erp.sd.delivery.event.DeliveryShippedEvent;     // ← MM 이 SD 를 import

@Slf4j
@Component
@RequiredArgsConstructor
class DeliveryEventListener {

    private final GoodsIssueService goodsIssueService;

    @TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
    void onShipped(DeliveryShippedEvent event) {
        log.info("출하 확정 사건 수신: deliveryId={}, warehouseId={}, lines={}",
                event.deliveryId(), event.warehouseId(), event.lines().size());
        goodsIssueService.createAndPostFromDelivery(event);
    }
    // onCancelled(...) 동일 패턴
}
```

리스너는 MM 안에 산다. **소식을 듣고 자기 일을 하는 주체가 MM** 이기 때문이다. 이 클래스의 import 문이 곧 의존 방향의 증거다:

- `import com.hwlee.erp.sd.delivery.event.DeliveryShippedEvent` → MM 파일이 SD 타입을 안다.
- 반대로 SD 의 어떤 파일도 `com.hwlee.erp.mm.*` 를 import 하지 않는다.

→ 의존은 오직 `MM → SD` 한 방향. 그리고 그 의존이 **딱 한 곳** (`DeliveryEventListener` 의 import 두 줄) 에 모여 있다. SD 와 MM 의 결합 지점이 이 파일 하나라는 것 — 이게 통합 코드를 한 자리에 모은 이유다.

### `integration/sd/` 네이밍의 미래 확장성 (설계서 §3.1)

```
mm/
├─ warehouse/
├─ stock/
├─ goodsreceipt/
├─ goodsissue/
└─ integration/          ← "다른 모듈/시스템으로부터 들어오는 통합" 의 한 자리
   └─ sd/                ← SD 로부터 들어오는 통합 (이번 Phase)
      └─ DeliveryEventListener.java
```

`integration/` 아래에 **출처 모듈명** 으로 디렉토리를 나눈다. 미래에:

- Phase 11~ MES 연계 → `integration/mes/`
- 외부 ERP 시스템 → `integration/external/`

"이 모듈이 바깥 세계로부터 반응하는 코드" 가 전부 `integration/` 한 곳에 모인다. 통합 흐름을 디버깅할 때 "어디서 SD 소식을 받지?" 의 답이 폴더 이름에 박혀 있다.

## 🔥 ⭐ 가장 중요한 결정 — `Long` 식별자 컬럼 (설계서 §6.5 를 뒤집다)

설계서 §6.5 / §4 는 원래 **엔티티 객체 참조** 를 제안했다. `Delivery` 에 `Warehouse warehouse` (ManyToOne) 필드를, `GoodsIssue` 에 `Delivery` 참조를 두자고. 그런데 **실제 구현은 둘 다 `Long` 컬럼으로만 매핑** 했다. 이 차이가 이 편 전체에서 가장 중요한 학습 포인트다.

### `Delivery.warehouseId` — SD 가 MM 의 Warehouse 를 모르게

```java
// sd/delivery/Delivery.java
@ManyToOne(fetch = FetchType.LAZY, optional = false)
@JoinColumn(name = "sales_order_id", nullable = false)
private SalesOrder salesOrder;          // ← 같은 SD 모듈이라 엔티티 참조 OK

/**
 * 출하지 창고 ID (Phase 4). DB 에는 fk_delivery_warehouse FK 가 걸려 있지만,
 * SD 가 MM 의 Warehouse 엔티티를 import 하지 않도록 Long 으로만 매핑한다
 * (의존 방향 MM → SD 단방향 유지). 존재 검증은 출하 시점 MM 리스너가 수행.
 */
@Column(name = "warehouse_id", nullable = false)
private Long warehouseId;               // ← MM 의 Warehouse 이지만 Long 으로만!
```

같은 파일 안에서 두 참조가 **다른 방식** 으로 매핑된 걸 보라:

- `salesOrder` 는 `@ManyToOne SalesOrder` — **같은 SD 모듈** 이라 엔티티 참조해도 모듈 경계를 넘지 않는다.
- `warehouseId` 는 `Long` — `Warehouse` 는 **MM 모듈** 이라, 엔티티로 잡으면 `SD → MM` 의존이 생긴다. 그래서 ID 로만.

만약 `private Warehouse warehouse` 였다면? `Delivery.java` 가 `import com.hwlee.erp.mm.warehouse.Warehouse` 를 하게 되고, 그 순간 SD 가 MM 을 import → `SD → MM` 발생.

### `GoodsIssue.deliveryId` — MM 이 SD 의 Delivery 를 모르게 (엔티티로는)

```java
// mm/goodsissue/GoodsIssue.java
@ManyToOne(fetch = FetchType.LAZY, optional = false)
@JoinColumn(name = "warehouse_id", nullable = false)
private Warehouse warehouse;            // ← 같은 MM 모듈이라 엔티티 참조 OK

/**
 * 이 출고가 비롯된 출하 ID (Phase 4). ... DB 에는 fk_goods_issue_delivery FK 가
 * 걸려 있지만, MM 이 SD 의 Delivery 엔티티를 import 하지 않도록 Long 으로만 매핑한다
 * (의존 방향 MM → SD 단방향).
 */
@Column(name = "delivery_id")
private Long deliveryId;                // ← SD 의 Delivery 이지만 Long 으로만!
```

여기도 대칭이다. `warehouse` 는 같은 MM 이라 엔티티 참조, `deliveryId` 는 SD 라서 `Long`.

### 왜 둘 다 ID 인가 — 양방향 의존을 막는 한 수

만약 설계서 §6.5 원안대로 둘 다 엔티티 참조였다면:

| 매핑 | 발생하는 import | 의존 방향 |
| --- | --- | --- |
| `Delivery.warehouse : Warehouse` | SD → mm.warehouse | `SD → MM` |
| `GoodsIssue.delivery : Delivery` | MM → sd.delivery | `MM → SD` |

**두 줄이 동시에 있으면 `SD ↔ MM` 양방향(순환) 의존.** 설계서 §2.3 / §3 의 "단방향 의존" 원칙이 그 자리에서 깨진다. 객체 그래프상 `delivery.getWarehouse()` 도 되고 `goodsIssue.getDelivery()` 도 되니 편해 보이지만, 두 모듈은 영영 분리 불가능해진다.

**해결:** 엔티티 객체 참조 대신 **`Long` 식별자 컬럼 + DB FK 제약**. 자바 코드 레벨의 모듈 의존은 0, 정합성은 DB FK 가 강제.

```sql
-- V21__add_delivery_id_to_goods_issue.sql
ALTER TABLE goods_issue ADD COLUMN delivery_id BIGINT NULL;
ALTER TABLE goods_issue ADD CONSTRAINT fk_goods_issue_delivery
    FOREIGN KEY (delivery_id) REFERENCES delivery(id);
CREATE INDEX idx_goods_issue_delivery ON goods_issue(delivery_id);

-- V22__add_warehouse_to_delivery.sql
ALTER TABLE delivery ADD CONSTRAINT fk_delivery_warehouse
    FOREIGN KEY (warehouse_id) REFERENCES warehouse(id);
CREATE INDEX idx_delivery_warehouse ON delivery(warehouse_id);
```

> 💡 핵심 통찰 — **"강한 FK" 가 노리는 두 가지 (정합성·인덱스) 는 그대로 살아 있다.** `delivery_id` 가 진짜 `delivery(id)` 를 가리키도록 DB 가 강제하고 (dangling 불가), 인덱스로 역추적도 빠르다. 자바 측에서 포기한 건 오직 **객체 그래프 탐색** 뿐. 즉 "강한 FK" 의 본질은 DB 레벨에 두고, 자바 레벨에서는 ID 만 들고 모듈 격리를 챙긴 것.

### 트레이드오프 — 객체 탐색을 잃고 모듈 격리를 산다

| 잃는 것 | 대신 치르는 비용 |
| --- | --- |
| `gi.getDelivery().getShippedDate()` 같은 객체 그래프 탐색 | 필요할 때 `deliveryRepository.findById(gi.getDeliveryId())` 로 **명시적 조회** |
| JPA 가 알아서 LAZY 로딩해 주는 편의 | 호출 측이 "지금 SD 데이터가 필요하다" 를 의식적으로 표현 |

이 비용은 **버그가 아니라 의도된 설계 가격** 이다. 모듈을 격리한 대가로 약간의 불편을 치르는 것. 그리고 실제로 Phase 4 흐름에서는 이 탐색이 거의 필요 없다 — 이벤트 본문이 이미 `warehouseId`, `lines` 를 다 담아 오므로 (설계서 §2.2 옵션 B), 리스너는 추가 SD 조회 없이 일을 끝낸다.

> ⚠️ 약한 참조 (StockMovement 의 `ref_type/ref_id`) 와 혼동 금지. 저건 **FK 자체가 없는** 다형성 참조. 여기 `delivery_id` 는 **진짜 FK 가 걸린** 단일 종속 (설계서 §4.2). "자바 의존을 끊으려고 Long 으로 매핑" 한 것이지, "FK 를 약하게 한" 게 아니다. DB 정합성은 오히려 강하다.

## 🔥 의존 방향 다이어그램

```
        ┌─────────────────────────────────────┐
        │              SD 모듈                  │
        │  com.hwlee.erp.sd.delivery           │
        │                                       │
        │   Delivery                            │
        │     ├─ salesOrder : SalesOrder (SD)   │  ← 같은 모듈, 엔티티 참조 OK
        │     └─ warehouseId : Long  ···········│··· (Warehouse 를 Long 으로만)
        │                                       │      컴파일 의존 없음
        │   event/                              │
        │     DeliveryShippedEvent (record)     │
        │       deliveryId, warehouseId,        │
        │       shippedDate, lines  (전부 ID/원시)│
        └───────────────▲───────────────────────┘
                        │
       import (이벤트 타입)│  ← 유일한 컴파일 의존: MM → SD 한 방향
                        │
        ┌───────────────┴───────────────────────┐
        │              MM 모듈                    │
        │  com.hwlee.erp.mm                      │
        │                                        │
        │   integration/sd/                      │
        │     DeliveryEventListener              │
        │       import sd...DeliveryShippedEvent ◀── 여기 한 곳에만 결합
        │       → goodsIssueService.create...()  │
        │                                        │
        │   goodsissue/                          │
        │     GoodsIssue                         │
        │       ├─ warehouse : Warehouse (MM)    │  ← 같은 모듈, 엔티티 참조 OK
        │       └─ deliveryId : Long  ···········│··· (Delivery 를 Long 으로만)
        │                                        │      컴파일 의존 없음
        └────────────────────────────────────────┘

  ──▶  컴파일(import) 의존 : 오직 MM → SD, 그것도 리스너 import 두 줄뿐
  ···  Long id 참조        : DB FK 는 있으나 자바 컴파일 의존은 0
```

읽는 법:

1. **실선 화살표(`▲`) 는 단 하나** — `DeliveryEventListener` 가 SD 의 이벤트 타입을 import. 방향은 `MM → SD`.
2. **점선(`···`) 두 줄** — `Delivery.warehouseId` 와 `GoodsIssue.deliveryId`. DB FK 로는 연결돼 있지만 **자바 import 가 없으므로 컴파일 의존이 아니다.** 만약 이게 엔티티 참조였다면 점선이 실선이 되어 양방향 화살표가 그려졌을 것.
3. SD 박스에서 MM 박스로 나가는 화살표는 **하나도 없다.** SD 는 자기가 발행한 이벤트를 누가 듣는지 모른다.

## 🔥 그래서 무엇이 좋아지는가

- **단독 이해 가능** — SD 코드만 읽어도 출하 로직이 완결된다. "재고는 누가 빼지?" 의 답을 SD 안에서 찾을 필요 없다 (SD 는 모른다). MM 만 읽으면 "출하 소식을 받아 재고를 뺀다" 가 `integration/sd/` 한 폴더에 완결.
- **테스트 격리** — SD 의 `DeliveryService` 테스트에 MM 을 띄울 필요 없다 (이벤트 발행만 검증). MM 리스너 테스트는 이벤트 객체를 만들어 던지면 끝.
- **변경 파급 최소** — MM 의 `GoodsIssue` 구조가 바뀌어도 SD 는 빌드조차 영향 없다. 반대로 SD 가 `DeliveryShippedEvent` 본문을 바꾸면 MM 리스너 한 곳만 본다.
- **확장 자리 예약** — 새 통합 (회계, MES) 은 `integration/<출처>/` 에 리스너만 추가. 발행자 SD 는 손대지 않는다 (Open/Closed).
