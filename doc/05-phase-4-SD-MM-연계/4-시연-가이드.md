# Phase 4 시연 가이드

> 7단계 사이클의 마지막 — **단계 7: 시연 + 회고**.
> 도메인 브리핑 → 모델링 → 코드 워크스루(6편)로 SD↔MM 연계의 설계를 머리로 이해했다.
> 이번에는 직접 Swagger UI / cURL / 통합 테스트로 **"출하 한 번이 두 모듈을 한 트랜잭션으로 움직인다"** 를 손으로 확인한다.

---

## 시연의 목표 (Phase 4 가 무엇을 보장하는가)

Phase 2 에서는 `Delivery.ship()` 이 SO 라인의 `shippedQty` 만 갱신했지 실제 창고 재고는 건드리지 않았다. Phase 3 에서는 재고/입출고가 독립적으로 작동했지만 SD 와 연결되지 않았다. **Phase 4 가 그 둘을 잇는다.**

이번에 확인할 **4대 사실**:

1. **출하 확정 한 번 = 자동 GoodsIssue 발행 + 재고 차감** — `POST /api/deliveries` 한 번에 ① Delivery SHIPPED ② SO 라인 `shippedQty` 누적 ③ GoodsIssue 자동 생성·POSTED ④ Stock 차감 ⑤ StockMovement(-) 적재가 **한 트랜잭션**으로 일어난다. SD 는 MM 을 직접 호출하지 않고 **이벤트를 발행**할 뿐.
2. **가용 부족이면 출하 자체가 거부 (409 INSUFFICIENT_STOCK)** — Phase 2 에서 OK 였던 출하가 이제 재고가 없으면 막힌다. 그리고 거부 시 **출하·SO 누적·재고가 모두 시도 전 상태로 롤백**. 리스너의 예외가 `BEFORE_COMMIT` 이라 발행자 트랜잭션까지 통째로 되돌아간다.
3. **출하 취소 = 연결된 GoodsIssue 자동 취소 + 재고 복원** — `POST /api/deliveries/{id}/cancel` 한 번에 SO 누적 복원 + GI CANCELLED + Stock 복원 + ADJUSTMENT_PLUS 정정 행까지.
4. **Delivery ↔ GoodsIssue 가 강한 FK 로 연결** — 자동 생성된 GI 의 `delivery_id` 가 원천 출하를 가리킨다. 사용자가 직접 만든 GI(실사조정 등)는 `delivery_id = NULL` 로 같은 테이블에 공존.

> 💡 핵심 한 문장: **"한 비즈니스 사건(출하) = 한 트랜잭션 = 여러 모듈의 변화"**. 이게 ERP 가 단순 CRUD 가 아닌 이유다.

## STEP 2. 스키마 변화 확인 — 두 컬럼이 생겼다

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SHOW COLUMNS FROM delivery LIKE 'warehouse_id';
  SHOW COLUMNS FROM goods_issue LIKE 'delivery_id';"
```

기대:
```
+--------------+--------+------+-----+---------+-------+
| Field        | Type   | Null | Key | Default | Extra |
+--------------+--------+------+-----+---------+-------+
| warehouse_id | bigint | NO   | MUL | NULL    |       |   ← Delivery: NOT NULL FK
+--------------+--------+------+-----+---------+-------+
+-------------+--------+------+-----+---------+-------+
| Field       | Type   | Null | Key | Default | Extra |
+-------------+--------+------+-----+---------+-------+
| delivery_id | bigint | YES  | MUL | NULL    |       |   ← GoodsIssue: nullable FK
+-------------+--------+------+-----+---------+-------+
```

확인 포인트:
- `delivery.warehouse_id` 는 **`NO`(NOT NULL)** — 모든 출하는 출하지 창고를 반드시 가진다.
- `goods_issue.delivery_id` 는 **`YES`(nullable)** — 자동 GI 는 채워지고 수동 GI 는 NULL (워크스루 04편).
- `Key = MUL` — 둘 다 FK 인덱스(`idx_delivery_warehouse`, `idx_goods_issue_delivery`)가 걸려 있다.

FK 제약도 확인:
```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT constraint_name, table_name, referenced_table_name
    FROM information_schema.key_column_usage
   WHERE constraint_name IN ('fk_delivery_warehouse','fk_goods_issue_delivery');"
```

기대: 두 FK 가 각각 `delivery→warehouse`, `goods_issue→delivery` 를 가리킨다.

> 💡 자바 코드에서는 이 둘을 **`Long` 컬럼**으로만 매핑했다(엔티티 객체 참조 아님). 그래도 DB FK 가 정합성을 강제 — "코드 레벨 모듈 격리 + DB 레벨 강한 FK" 의 조합(워크스루 03편).

## STEP 6. delivery_id 로 GI 역추적 (SQL)

워크스루 04편의 "강한 FK 로 역추적" 을 직접 SQL 로 확인한다.

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT gi.id, gi.number, gi.status, gi.delivery_id, d.number AS delivery_number
    FROM goods_issue gi
    JOIN delivery d ON d.id = gi.delivery_id
   WHERE gi.delivery_id = $DLV_ID;"
```

기대: GI 한 행이 출하 `DLV-20260529-001` 과 연결돼 나온다.

수동 GI 와의 공존도 확인 — 한 테이블에 두 출처:
```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT id, number, reason, delivery_id,
         CASE WHEN delivery_id IS NULL THEN '수동등록' ELSE '출하자동' END AS origin
    FROM goods_issue ORDER BY id;"
```

확인 포인트: `delivery_id` 가 채워진 행(출하자동) 과 `NULL` 인 행(Phase 3 에서 직접 만든 GI 가 있다면)이 **같은 테이블에 공존**, NULL 여부로 출처가 갈린다 (워크스루 04편 §nullable FK).

## STEP 7. ⭐ 가용 부족 출하 → 409 + 전체 롤백

재고는 현재 14대. **20대 출하**를 시도해 부족 상황을 만든다.

```bash
# 부족 시연용 두 번째 수주 — 노트북 20대
SO2=$(curl -s -X POST "http://localhost:8080/api/sales-orders" \
  -H "Content-Type: application/json" \
  -d "{
    \"customerId\": $CUST_ID,
    \"orderDate\": \"2026-05-29\",
    \"lines\": [{ \"itemId\": $ITEM_ID, \"quantity\": 20, \"unitPrice\": 1200000 }]
  }")
SO2_ID=$(echo "$SO2" | jq '.id')
curl -s -X POST "http://localhost:8080/api/sales-orders/$SO2_ID/confirm" > /dev/null
SOL2_ID=$(curl -s "http://localhost:8080/api/sales-orders/$SO2_ID" | jq '.lines[0].id')

# 20대 출하 시도 — 재고 14 < 20
curl -i -X POST "http://localhost:8080/api/deliveries" \
  -H "Content-Type: application/json" \
  -d "{
    \"salesOrderId\": $SO2_ID,
    \"warehouseId\": $WH_ID,
    \"shippedDate\": \"2026-05-29\",
    \"lines\": [{ \"salesOrderLineId\": $SOL2_ID, \"quantity\": 20 }]
  }"
```

기대:
```
HTTP/1.1 409 Conflict
Content-Type: application/problem+json

{
  "title": "Insufficient Stock",
  "status": 409,
  "detail": "재고 부족: itemId=1, warehouseId=3, available=14, requested=20",
  "code": "INSUFFICIENT_STOCK",
  "available": 14.0000,
  "requested": 20
}
```

**핵심 — 전체 롤백 검증.** 출하가 실패했으니 그 트랜잭션에서 시도한 모든 변화가 되돌아가야 한다:

```bash
# (1) 재고는 14 그대로 (1도 안 빠짐)
curl -s "http://localhost:8080/api/stocks?itemId=$ITEM_ID&warehouseId=$WH_ID" \
  | jq '.content[] | {qtyOnHand}'

# (2) SO2 라인 shipped_qty 도 0 — 출하 누적이 안 됨
curl -s "http://localhost:8080/api/sales-orders/$SO2_ID" \
  | jq '{status, shippedQty: .lines[0].shippedQty}'

# (3) SO2 에 연결된 Delivery 가 아예 안 생김
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT COUNT(*) AS deliveries_for_so2 FROM delivery WHERE sales_order_id = $SO2_ID;"
```

기대: `qtyOnHand: 14`, `status: "CONFIRMED"` + `shippedQty: 0`, `deliveries_for_so2: 0`.

확인 포인트:
- **MM 의 예외(InsufficientStockException)가 SD 의 트랜잭션까지 롤백**시켰다. 리스너가 `@TransactionalEventListener(BEFORE_COMMIT)` 이라 commit '전에' 예외가 터졌고, 그래서 Delivery·SO 누적·재고가 전부 시도 전 상태로 복귀 (워크스루 02편 §예외 전파).
- 만약 `AFTER_COMMIT` 이었다면 → 출하는 이미 커밋돼 SHIPPED 로 남고, 재고만 따로 실패하는 **부분 상태 사고**가 났을 것.

## STEP 9. 원장 정합성 직접 SQL 검증

Phase 3 와 같은 자기 검증 — 모든 (item, warehouse) 에서 원장 합 = 캐시여야 한다.

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT m.item_id, m.warehouse_id,
         SUM(m.qty_delta) AS ledger_qty, s.qty_on_hand
    FROM stock_movement m
    JOIN stock s ON s.item_id = m.item_id AND s.warehouse_id = m.warehouse_id
   GROUP BY m.item_id, m.warehouse_id, s.qty_on_hand
  HAVING SUM(m.qty_delta) <> s.qty_on_hand;"
```

## 트러블슈팅

### 1. `POST /api/deliveries` 에서 `400 Bad Request — warehouseId 가 null`

- Phase 4 부터 `DeliveryCreateRequest` 에 `warehouseId` 가 **필수**로 추가됐다 (`@NotNull`). Phase 2/3 시연 때 쓰던 본문(warehouseId 없음)은 이제 거부된다 (워크스루 04편).

### 2. 출하했는데 `404 Not Found — Stock not found ... 출고 전에 입고가 있어야 합니다`

- 출하지 창고(`warehouseId`)에 그 품목 재고 행이 한 번도 생긴 적이 없음 = 입고가 없었음.
- STEP 3 처럼 그 창고에 먼저 입고해야 한다. **재고 없는 창고로의 출하는 전체 롤백**(STEP 7 의 5번 케이스와 같은 EntityNotFoundException 경로).

### 3. 출하가 `409 INSUFFICIENT_STOCK` 인데 어느 트랜잭션이 롤백된 건지 헷갈림

- 출하 트랜잭션 **전체**가 롤백된다. Delivery 생성도, SO `shippedQty` 누적도, 재고 차감도 전부. STEP 7 의 (1)(2)(3) 으로 직접 확인.
- 원인: 리스너가 `BEFORE_COMMIT` 이라 예외가 commit '전'에 발생 → 발행자(DeliveryService) 트랜잭션까지 rollback.

### 4. GI 목록에서 방금 만든 출하의 GI 가 안 보임

- `reason=SHIPMENT` + 올바른 `warehouseId` 로 필터했는지 확인. GoodsIssue 조회엔 `deliveryId` 필터가 없으므로, 목록 응답의 `deliveryId` 필드로 매칭하거나 STEP 6 의 SQL 로 직접 역추적.

### 5. 출하 취소했는데 `GI 가 안 보임 / 이미 CANCELLED` 로그

- `cancelByDeliveryId` 는 방어적이다 — 연결된 GI 가 없거나(Phase 4 적용 전 출하) 이미 POSTED 가 아니면 조용히 건너뛴다 (워크스루 02편). 정상 동작.

## 시연 후 — 무엇이 학습됐는가

Phase 4 시연으로 **체감**으로 알게 된 것:

1. **"한 비즈니스 사건 = 한 트랜잭션 = 여러 모듈 변화"** — `POST /api/deliveries` 한 번이 SD(출하·SO)와 MM(GI·재고·원장)을 동시에 움직였다. 사용자는 "출하했다" 한 동작을 했을 뿐인데 6개 테이블이 정합성을 유지하며 변했다. 이게 ERP 가 단순 CRUD 와 다른 지점.
2. **"모듈은 이벤트로 느슨하게, 트랜잭션은 한 덩어리로"** — SD 는 MM 을 직접 호출하지 않고 `DeliveryShippedEvent` 를 발행만 했다(결합도 ↓). 그런데도 `BEFORE_COMMIT` 덕분에 같은 트랜잭션(정합성 ↑). **느슨한 결합과 강한 정합성을 동시에** 잡는 게 이 패턴의 핵심.
3. **"실패는 전부 아니면 전무(all-or-nothing)"** — 재고가 1대라도 모자라면 출하 자체가 없던 일이 된다. 부분 출하·부분 차감 같은 어중간한 상태가 DB 에 남지 않는다. STEP 7 에서 직접 확인.
4. **"역추적은 강한 FK 로"** — 자동 생성된 GI 가 `delivery_id` 로 원천 출하를 가리킨다. 취소 흐름이 이 FK 로 GI 를 즉시 찾아낸다. 수동 GI(NULL)와 한 테이블에 공존하면서도 출처가 명확.
