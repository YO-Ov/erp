# Phase 5 도메인 브리핑 — FI(회계) / 거래가 자동으로 전표가 되는 흐름 ⭐

> 7단계 사이클의 **단계 1: 도메인 브리핑**.
> 이 단계의 목적은 "코드를 짜기 전에, **왜 ERP 에서 회계 모듈이 가장 중요한가** 와 복식부기의 기본 골격을 글로 이해" 하는 것.
> 회계 도메인 지식이 없어도 따라올 수 있게 §2 복식부기를 충분히 친절하게 풀었다. 모호한 부분은 **단계 2 Q&A** 에서 질문.

---

## 0. 한 줄 요약

> **모든 비즈니스 거래는 결국 "돈의 이동" 이고, 그 이동을 차변/대변 두 줄로 적은 게 회계 전표다.**
> Phase 2~4 에서 만든 매출·입고·출하·입출금 사건이 **자동으로 회계 전표(분개)** 가 되는 흐름을 만든다. Phase 4 의 이벤트 패턴이 여기서 **두 번째 구독자(회계 listener)** 로 자연스럽게 확장된다.

---

## 1. 왜 ERP 에서 회계가 "심장" 인가

다른 모듈(SD/MM/HR/PP)은 **각자의 현장 업무**를 본다 — 영업은 주문, 자재는 재고, 생산은 작업지시. 하지만 이 모든 활동은 결국 **돈의 언어로 번역**되어야 한다. 그 번역기가 회계(FI)다.

- 사장이 "이번 달 얼마 벌었어?" → **손익계산서**(매출 − 비용).
- 은행이 "이 회사 재무 상태는?" → **재무상태표**(자산 = 부채 + 자본).
- 세무서가 "부가세 얼마 낼래?" → 매출세액 − 매입세액.

이 세 질문의 답이 **모두 회계 전표의 누적**에서 나온다. 그래서:

> **ERP 도입의 진짜 목적은 "현장 거래를 사람 손 안 거치고 회계로 자동 연결" 하는 것.**
> 영업이 송장 한 장 발행하면, 경리가 따로 분개를 안 쳐도 매출·매출채권·부가세가 자동으로 장부에 잡힌다. 이 "자동 전표" 가 ERP 의 핵심 가치.

Phase 5 가 바로 그 자동 연결을 만든다. 지금까지(Phase 2~4)는 "현장" 만 있었고 "장부" 가 없었다.

## 2. 복식부기 기초 — 이것만 잡으면 된다

회계가 어렵게 느껴지는 건 용어 때문이다. 골격은 단순하다.

### 2-1. 모든 거래는 두 줄로 — 차변(Debit)과 대변(Credit)

복식부기(複式簿記, double-entry)의 핵심 규칙 단 하나:

> **하나의 거래를 항상 "차변" 과 "대변" 양쪽에 같은 금액으로 적는다. 그리고 차변 합 = 대변 합.**

"복식" = 한 거래를 두 번(양쪽) 적는다는 뜻. 왜? 돈은 **어디선가 나와서 어디론가 간다**. 그 출처와 사용처를 동시에 기록하면 절대 안 틀린다.

예 — 노트북을 외상으로 1,200만원 팔았다:

```
차변(왼쪽)              대변(오른쪽)
─────────────────────────────────────
매출채권  12,000,000    매출  12,000,000
(받을 돈이 생김)        (매출이 발생함)
```

"받을 권리(매출채권)가 1,200만원 늘었고, 그 원인은 매출 1,200만원" — 한 사건의 양면. **차변 합 12,000,000 = 대변 합 12,000,000.** 이 균형이 안 맞으면 전표 자체가 성립 안 한다(시스템이 거부).

### 2-2. 차변/대변에 뭐가 오는가 — 5가지 계정 유형의 증감 규칙

계정(과목)은 5가지 유형으로 나뉘고, 각 유형이 **늘 때 차변에 오는지 대변에 오는지**가 정해져 있다:

| 계정 유형 | 예시 | 증가(↑) 시 | 감소(↓) 시 |
| --- | --- | --- | --- |
| **자산** (Asset) | 현금, 매출채권, 재고자산 | **차변** | 대변 |
| **부채** (Liability) | 매입채무, 부가세예수금 | 대변 | **차변** |
| **자본** (Equity) | 자본금, 이익잉여금 | 대변 | 차변 |
| **수익** (Revenue) | 매출 | 대변 | 차변 |
| **비용** (Expense) | 매출원가, 급여 | **차변** | 대변 |

> 💡 외울 필요 없다. **"자산·비용은 늘면 차변, 수익·부채는 늘면 대변"** 한 줄만 기억하면 이 프로젝트의 모든 분개를 읽을 수 있다.

위 매출 예시를 다시 보면:
- 매출채권(자산)이 ↑ → 차변. ✅
- 매출(수익)이 ↑ → 대변. ✅

### 2-3. 분개(Journal Entry) = 한 거래의 차/대 기록

이렇게 한 거래를 차변·대변으로 적은 한 묶음을 **분개** 또는 **회계 전표(Journal Entry)** 라고 부른다. 이게 Phase 5 의 주인공 테이블이다.

## 3. 이 프로젝트가 다룰 5가지 핵심 분개

우리가 Phase 2~4 에서 만든 사건들이 각각 어떤 분개가 되는지 — 이게 Phase 5 의 설계 지도다.

### 3-1. 매출 (인보이스 발행) — Phase 2 의 Invoice

노트북 6대를 1,200만원에 외상 판매, 부가세 10%:

```
차변                          대변
──────────────────────────────────────────────
매출채권  13,200,000          매출           12,000,000
                              부가세예수금    1,200,000
```

- 받을 돈(매출채권, 자산↑) = 공급가 + 부가세 = 13,200,000.
- 매출(수익↑) 12,000,000, 부가세예수금(부채↑ — 나중에 국가에 낼 돈) 1,200,000.
- 차변 13,200,000 = 대변 13,200,000. ✅

### 3-2. 매출원가 (출하) — Phase 4 의 GoodsIssue ⭐

물건을 내보냈으니 **그 원가만큼 재고자산이 줄고 비용(매출원가)이 잡힌다**. 노트북 6대 원가 @100만원:

```
차변                          대변
──────────────────────────────────────────────
매출원가  6,000,000           재고자산  6,000,000
(비용↑)                       (자산↓)
```

> ⭐ **여기서 Phase 3 의 복선이 회수된다.** 출고 시 `StockMovement.unit_cost` 에 박았던 "직전 평균 단가" 가 바로 이 매출원가의 금액이다. Phase 3 시연에서 "지금 박는 이 단가가 회계의 근간" 이라 했던 그 값이 Phase 5 에서 실제로 비용 전표가 된다.

매출(§3-1)과 매출원가(§3-2)는 **다른 사건**(인보이스 발행 vs 출하)에서 나온다. 그래서 이익 = 매출 − 매출원가가 자동 계산된다.

### 3-3. 매입 (입고) — Phase 3 의 GoodsReceipt

외상으로 자재 입고, 1,000만원:

```
차변                          대변
──────────────────────────────────────────────
재고자산  10,000,000          매입채무  10,000,000
(자산↑)                       (부채↑ — 갚을 돈)
```

### 3-4. 입금 (채권 소멸)

고객이 외상값 13,200,000 을 현금으로 갚음:

```
차변                          대변
──────────────────────────────────────────────
현금  13,200,000              매출채권  13,200,000
(자산↑)                       (자산↓ — 받을 돈이 사라짐)
```

### 3-5. 출금 (채무 소멸)

거래처에 매입대금 10,000,000 지급:

```
차변                          대변
──────────────────────────────────────────────
매입채무  10,000,000          현금  10,000,000
(부채↓)                       (자산↓)
```

> 💡 §3-1~3-5 다섯 패턴이 OTC(수주-수금) + PTP(구매-지급) 의 회계 전부다. 학습 계획서의 권고대로 **가장 단순한 매출 분개(§3-1)부터** 만들고 하나씩 늘린다.

## 4. 계정과목(Chart of Accounts) — 회계의 마스터 데이터

위 분개에 등장한 "매출채권", "매출", "현금"... 이것들이 **계정과목(CoA)**. 회사가 쓸 수 있는 계정의 목록이자 코드 체계다.

```
1000 자산
├─ 1100 현금
├─ 1200 매출채권
└─ 1400 재고자산
2000 부채
├─ 2100 매입채무
└─ 2200 부가세예수금
4000 수익
└─ 4100 매출
5000 비용
└─ 5100 매출원가
```

특징:
- **트리 구조** — Phase 1 의 `Department` 자기참조 트리와 같은 패턴(부모 계정 ↔ 자식 계정). 누적성 ⭐.
- **계정 유형(자산/부채/자본/수익/비용)** 을 각 계정이 가진다 — §2-2 의 증감 규칙을 코드로 강제할 때 쓰임.
- **마스터 데이터** — 코드 체계 + Soft Delete + 활성 상태. Phase 1 에서 만든 마스터 패턴을 또 재사용.

→ 계정과목은 "또 하나의 마스터" 다. Phase 1 자산이 여기서도 일한다.

---

## 5. 회계 전표(Journal Entry) — 헤더-라인의 또 다른 등장

전표도 **헤더-라인 구조**(Phase 2 의 수주/인보이스와 동일):

```
JournalEntry (헤더)              전표번호, 전표일자, 적요, 상태(임시/확정)
├─ JournalLine (라인 1)          계정, 차변금액 or 대변금액
├─ JournalLine (라인 2)          계정, 차변금액 or 대변금액
└─ ...
```

핵심 불변식 — **차변 합 = 대변 합**:

```
SUM(line.debit) == SUM(line.credit)
```

이게 안 맞으면 전표 확정(post)을 거부한다. Phase 3 의 "음수 재고 거부", Phase 2 의 "신용한도 초과 거부" 와 같은 자리 — **도메인 불변식을 코드가 지킨다**.

> 💡 Phase 2 의 "헤더-라인 + 상태 머신(DRAFT→POSTED)", Phase 3 의 "원장 append-only" 가 회계 전표에서 거의 그대로 반복된다. 회계 전표야말로 원장(ledger)의 원조다.

## 6. 자동 전표 — Phase 4 이벤트 패턴의 확장 ⭐

여기가 Phase 5 의 진짜 핵심이자 Phase 4 와의 연결고리다.

### 6-1. 사람이 안 친다 — 사건이 전표를 만든다

전통 회계: 경리가 매출 송장 보고 손으로 분개 입력. ERP: **사건이 발생하면 규칙에 따라 전표가 자동 생성**.

Phase 4 에서 출하 확정 시 `DeliveryShippedEvent` 를 발행했고, MM 의 리스너가 재고를 깠다. **Phase 5 는 같은 사건에 회계 리스너를 하나 더 붙인다**:

```
인보이스 발행 (InvoiceIssuedEvent)
   │
   ├─ [기존] (Phase 2 에서 SO invoiced_qty 갱신)
   └─ ⭐ [신규] 회계 리스너 → 매출 분개 자동 생성 (§3-1)

출하 확정 (DeliveryShippedEvent)
   │
   ├─ [Phase 4] 재고 차감 리스너 (StockDeductionListener)
   └─ ⭐ [신규] 회계 리스너 → 매출원가 분개 자동 생성 (§3-2)

입고 확정 (GoodsReceiptPostedEvent)
   │
   └─ ⭐ [신규] 회계 리스너 → 매입 분개 자동 생성 (§3-3)
```

→ Phase 4 브리핑 §10-3 에서 예고한 "Phase 5 의 자동 분개 listener 가 같은 사건에 자연스럽게 붙는다" 가 실현되는 자리. **이벤트 패턴을 쓴 보람**이 여기서 나온다 — SD/MM 코드를 한 줄도 안 고치고 회계 기능이 얹힌다.

### 6-2. 두 리스너가 한 사건을 들을 때 — 순서 문제

이제 `DeliveryShippedEvent` 를 **두 리스너**가 듣는다(재고 차감 + 회계 분개). 순서가 중요할까?

- 매출원가 분개는 **출고 단가**가 필요한데, 그 단가는 재고 차감(`StockMovement`) 시점에 확정된다.
- → **재고 리스너가 먼저, 회계 리스너가 나중**에 실행돼야 한다.
- Spring 의 **`@Order`** 로 리스너 실행 순서를 제어한다(Phase 4 설계서 §10 에서 "두 listener 의 순서 — Phase 5 에서 회계 listener 추가 시" 라고 미뤄둔 항목).

→ 이게 Phase 5 의 새 학습 포인트: **한 사건에 여러 구독자가 반응할 때의 순서 보장**.

### 6-3. 자동 전표 생성 "규칙 엔진"

각 사건 타입 → 어떤 분개를 만들지가 **규칙**이다:

| 사건 | 차변 | 대변 |
| --- | --- | --- |
| 인보이스 발행 | 매출채권 | 매출 + 부가세예수금 |
| 출하 확정 | 매출원가 | 재고자산 |
| 입고 확정 | 재고자산 | 매입채무 |
| 입금 | 현금 | 매출채권 |
| 출금 | 매입채무 | 현금 |

이 표를 코드로 옮긴 게 **자동 전표 생성 규칙**. 설계 단계(단계 3)에서 "규칙을 어디에 둘까 — 사건별 전용 메서드 vs 매핑 테이블 vs 전략 패턴" 을 결정한다.

## 7. 채권(AR)/채무(AP) — 미수금·미지급금 관리

- **매출채권(AR, Accounts Receivable)**: 외상 매출로 "받을 돈". 인보이스 발행 시 ↑, 입금 시 ↓.
- **매입채무(AP, Accounts Payable)**: 외상 매입으로 "갚을 돈". 입고 시 ↑, 출금 시 ↓.

이 둘은 §3 의 분개에서 이미 등장했다. Phase 5 에서는 추가로 **"고객별/거래처별 잔액 조회"**(누가 얼마 안 갚았나) 를 다룰 수 있다 — 다만 학습 범위는 설계 단계에서 확정(전표 자체로 충분한지, 별도 잔액 캐시가 필요한지).

> 💡 채권/채무 잔액도 Phase 3 의 "Stock(캐시) vs StockMovement(원장)" 와 같은 구조 — 전표(원장)의 누적이 진실, 잔액은 그 SUM. 또 한 번의 누적성.

## 8. 만들 것 — 한눈에 (개략, 상세는 단계 3)

| 만들 것 | 종류 | 비고 |
| --- | --- | --- |
| 계정과목(`Account`) | 마스터 (트리) | Phase 1 마스터 + Department 트리 패턴 재사용 |
| 회계 전표(`JournalEntry` + `JournalLine`) | 트랜잭션 (헤더-라인) | 차변합=대변합 불변식 |
| 자동 전표 규칙 | 서비스/규칙 | 사건 → 분개 매핑 |
| 회계 이벤트 리스너 | 리스너 (FI 측) | `DeliveryShippedEvent` 등 구독, `@Order` 로 순서 |
| 새 이벤트(`InvoiceIssuedEvent`, `GoodsReceiptPostedEvent` 등) | 도메인 이벤트 | Phase 4 의 이벤트 가족 확장 |
| 입금/출금(`Payment`) | 트랜잭션 | 채권/채무 소멸 |
| 마이그레이션 V23~ | Flyway | 계정/전표/결제 테이블 + 계정 시드 |

**새 테이블은 늘지만, 패턴은 전부 기존 것** — 마스터(Phase1) + 헤더-라인/상태머신(Phase2) + 원장(Phase3) + 이벤트(Phase4). Phase 5 는 **누적된 모든 패턴의 종합**이다.

## 9. Phase 1~4 와의 누적성

```
Phase 1: 마스터 (코드 체계, Soft Delete, 자기참조 트리)
   ↓                                    └─→ 계정과목이 그대로 재사용
Phase 2: 트랜잭션 (헤더-라인, 상태 머신 DRAFT→POSTED)
   ↓                                    └─→ 회계 전표가 그대로 재사용
Phase 3: 재고 (원장 append-only, 캐시 vs 원장, unit_cost)
   ↓                                    └─→ 전표=원장, 채권잔액=캐시, 매출원가=unit_cost
Phase 4: 이벤트 (@TransactionalEventListener, 한 트랜잭션 원자성)
   ↓                                    └─→ 회계 리스너가 같은 사건에 추가 구독
Phase 5: ⭐ 위 모든 패턴을 모아 "거래 → 자동 회계" 를 완성
```

→ Phase 5 는 **새로 배울 개념이 복식부기 하나** 뿐. 나머지는 전부 Phase 1~4 의 패턴 재사용 + 한 번 더 깊게(`@Order`, 규칙 엔진).

## 10. 사용자가 이번 브리핑에서 잡아야 할 핵심 5가지

1. **복식부기 = 한 거래를 차변·대변 두 줄로, 차변합=대변합**. 자산·비용↑은 차변, 수익·부채↑은 대변. 이 한 줄이면 이 프로젝트 모든 분개를 읽는다.
2. **ERP 의 진짜 가치 = 현장 거래가 사람 손 안 거치고 자동으로 회계 전표가 되는 것**. Phase 5 가 그 자동 연결을 만든다.
3. **5가지 분개**(매출/매출원가/매입/입금/출금)가 OTC+PTP 회계의 전부. 매출 분개부터 시작.
4. **매출원가 분개의 금액 = Phase 3 의 `StockMovement.unit_cost`** — Phase 3 의 복선이 여기서 회수된다.
5. **Phase 4 이벤트 패턴의 확장** — 같은 사건(출하/인보이스/입고)에 회계 리스너를 추가로 붙인다. 두 리스너의 순서는 `@Order` 로. SD/MM 코드는 무수정.

## 11. 다음 단계 (단계 2: 이해 확인 Q&A)

이 브리핑을 읽고 **모호하거나 의문이 있는 부분을 질문** 해 주세요. 출발점이 될 예시 질문:

- "차변·대변이 헷갈린다 — 왜 매출채권이 늘면 '차변' 이고 매출이 늘면 '대변' 인가? 직관으로 잡는 법은?"
- "매출(인보이스)과 매출원가(출하)가 다른 사건에서 나온다고 했는데, 한 번에 안 잡고 따로 잡는 이유는?"
- "자동 전표가 틀리면(차변≠대변) 어떻게 되나? 누가 막나?"
- "부가세는 왜 매출과 따로 '부가세예수금' 으로 빼서 기록하나?"
- "Phase 4 의 출하 이벤트를 회계 리스너가 또 듣는다는데, 재고 리스너랑 순서가 꼬이면 어떻게 되나?"
- "채권 잔액을 별도 테이블로 캐시할까, 전표를 매번 SUM 할까 — Phase 3 의 Stock 처럼?"
- "회계가 어렵다고 했는데, 이 프로젝트에서 어디까지 단순화하고 어디부터 진짜 회계인가?"

질문이 없거나 충분히 이해됐다면 **"단계 3 (모델링 제안) 으로 진행"** 이라고 알려 주세요. — 그러면 계정과목 트리 / 전표 ERD / 자동 분개 규칙 / 이벤트·리스너 순서 / 마이그레이션 계획을 설계해 승인받겠습니다.
