> 7단계 사이클의 **단계 3: 모델링 제안**.
> 도메인 브리핑(`1-도메인-브리핑.md`)의 복식부기·5가지 분개·이벤트 확장 개념을 **ERD / 엔티티 / 자동 분개 규칙 / 이벤트·리스너 순서 / 마이그레이션** 으로 변환한다.

## 0. 한눈에 — 이번 설계의 핵심 결정 12개

| # | 결정 | 한 줄 이유 |
| --- | --- | --- |
| 1 | 계정과목 `Account` = **`BaseEntityWithCode` + 자기참조 트리 + `AccountType`** | Phase 1 `Department` 트리 패턴 그대로 재사용 |
| 2 | 정상 잔액 방향(차변/대변)은 **`AccountType` 에서 파생** (자산·비용=차변, 부채·자본·수익=대변) | 별도 컬럼 불필요, 유형이 곧 방향 |
| 3 | 전표 = **`JournalEntry`(헤더) + `JournalLine`(라인)** 헤더-라인 | Phase 2 수주/인보이스 패턴 재사용 |
| 4 | 라인은 **`debit`, `credit` 두 컬럼** — 한 라인은 한쪽만 > 0 | 회계 장부의 관행적 표현, 사람이 읽기 쉬움 |
| 5 | **차변 합 = 대변 합** 불변식 — `post()` 에서 검증, 불일치 시 거부 | 복식부기의 본질. Phase 3 음수재고 거부와 같은 자리 |
| 6 | 전표 출처 = **`sourceType`(INV/GI/GR/PAY) + `sourceId` 약한 참조** | StockMovement 다형성 참조 패턴 재사용 (다양한 출처) |
| 7 | 자동 전표 = **사건별 FI 리스너 + 시스템 계정 코드 상수 매핑** | 규칙 엔진 과설계 회피, 학습 단순화 |
| 8 | 이벤트 확장: **`InvoiceIssuedEvent`(매출), `GoodsReceiptPostedEvent`(매입)** 신규 + 기존 **`DeliveryShippedEvent` 재구독(매출원가)** | Phase 4 이벤트 가족 확장. SD/MM 코드 거의 무수정 |
| 9 | 매출원가 단가 = **회계 리스너가 `deliveryId` → GoodsIssue → StockMovement 조회로 합산** | Phase 3 `unit_cost` 복선 회수. FI 가 MM 원장을 읽음 |
| 10 | **`@Order`** 로 리스너 순서 — 재고 차감(Phase 4) **먼저**, 회계 분개 **나중** | 매출원가는 출고 단가 확정 후에야 계산 가능 |
| 11 | 채권/채무 **별도 잔액 테이블 안 만듦** — 계정별 전표 라인 SUM 으로 조회 | 원장이 진실. Phase 3 "캐시 vs 원장" 의 원장-only 선택 |
| 12 | 입금/출금 = **`Payment` 엔티티** — 채권/채무 소멸 분개 | OTC 의 마지막 단계(수금) 완성 |

원칙(Phase 4 §3.1 재확인):
- **이벤트는 발행자가 소유** — `InvoiceIssuedEvent` 는 `sd/invoice/event/`, `GoodsReceiptPostedEvent` 는 `mm/goodsreceipt/event/`.
- **리스너는 수신자(FI)가 소유** — `fi/integration/sd/`, `fi/integration/mm/`.
- 의존 방향: **`FI → SD`, `FI → MM`** 단방향. SD/MM 은 FI 를 모른다. "회계는 모든 모듈을 알지만, 현장 모듈은 회계를 모른다" — 회계의 본질이 패키지로 표현됨.

> 💡 FI 가 SD·MM 을 아는 건 Phase 4 의 "MM←SD 단방향" 과 일관된 방향성. 회계가 맨 아래에서 모두를 구독하는 구조.

## 2. 계정과목(`Account`) — 또 하나의 트리 마스터

### 2.1 엔티티

```java
@Entity @Table(name = "account")
@SQLDelete(...) @SQLRestriction("deleted_at IS NULL")
public class Account extends BaseEntityWithCode {   // id, code, status, audit, soft delete

    @Column(nullable = false) private String name;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false) private AccountType type;   // ASSET/LIABILITY/EQUITY/REVENUE/EXPENSE

    @ManyToOne(fetch = LAZY)
    @JoinColumn(name = "parent_id")
    private Account parent;                                // 자기참조 트리 (Department 패턴)

    @Column(nullable = false) private boolean postable;    // 말단 계정만 전표 가능 (헤더 계정 X)
}
```

- **`BaseEntityWithCode`** — Phase 1 의 Customer/Item/Warehouse 와 같은 베이스. 코드 수동 입력(`1200`, `4100`).
- **자기참조 `parent`** — Phase 1 `Department.parent` 와 동일. 계정 트리(자산 > 매출채권).
- **`postable`** — 헤더 계정(예: "자산 1000")엔 직접 전표 못 달고, 말단 계정(예: "매출채권 1200")만 가능. 실무 회계의 기본 규칙.

### 2.2 `AccountType` 과 정상 잔액 방향 (결정 #2)

```java
public enum AccountType {
    ASSET(NormalSide.DEBIT),       // 자산  — 늘면 차변
    LIABILITY(NormalSide.CREDIT),  // 부채  — 늘면 대변
    EQUITY(NormalSide.CREDIT),     // 자본  — 늘면 대변
    REVENUE(NormalSide.CREDIT),    // 수익  — 늘면 대변
    EXPENSE(NormalSide.DEBIT);     // 비용  — 늘면 차변

    private final NormalSide normalSide;
}
```

→ 브리핑 §2-2 의 증감표를 enum 으로 박음. "정상 잔액 방향" 컬럼을 따로 두지 않고 **유형에서 파생** — 자산은 언제나 차변이 정상.

### 2.3 시드 계정 (V23)

학습에 필요한 최소 계정 트리:

| 코드 | 이름 | 유형 | postable | 부모 |
| --- | --- | --- | --- | --- |
| 1000 | 자산 | ASSET | N | — |
| 1100 | 현금 | ASSET | Y | 1000 |
| 1200 | 매출채권 | ASSET | Y | 1000 |
| 1400 | 재고자산 | ASSET | Y | 1000 |
| 2000 | 부채 | LIABILITY | N | — |
| 2100 | 매입채무 | LIABILITY | Y | 2000 |

## 3. 회계 전표(`JournalEntry` + `JournalLine`)

### 3.1 헤더-라인 ERD

```
JournalEntry (전표 헤더)
├─ number        VARCHAR  전표번호 JE-YYYYMMDD-NNN (TransactionNumberGenerator 확장)
├─ entry_date    DATE     전표일자
├─ description   VARCHAR  적요 ("INV-... 매출")
├─ status        ENUM     DRAFT / POSTED / CANCELLED
├─ source_type   VARCHAR  INV / GI / GR / PAY / MANUAL  (약한 참조)
├─ source_id     BIGINT   출처 트랜잭션 id
└─ posted_at     DATETIME

JournalLine (전표 라인)
├─ journal_entry_id  BIGINT  FK
├─ line_no           INT
├─ account_id        BIGINT  FK → account
├─ debit             DECIMAL(15,2)  차변 (대변 라인이면 0)
└─ credit            DECIMAL(15,2)  대변 (차변 라인이면 0)
```

### 3.2 라인의 차/대 표현 — `debit`/`credit` 두 컬럼 (결정 #4)

세 가지 옵션:

| 옵션 | 표현 | 채택 |
| --- | --- | --- |
| A. `amount` + `side`(DEBIT/CREDIT) enum | 한 컬럼 + 방향 | — |
| **B. `debit`, `credit` 두 컬럼** (한쪽만 > 0) | 회계 장부 그대로 | ✅ |
| C. 부호 있는 `amount` (차변 +, 대변 −) | 한 컬럼, 부호로 구분 | — |

→ **옵션 B**. 실제 회계 장부(분개장)가 차변·대변 두 칸으로 나뉜 모습 그대로라 **사람이 읽기 가장 쉽고**, 차변 합/대변 합 계산이 `SUM(debit)`, `SUM(credit)` 으로 직관적. 학습 가치 ↑.

> 💡 옵션 C(부호)는 코드는 짧지만 "−1,200,000 이 대변" 이라는 게 회계 직관과 어긋남. 학습 단계엔 B 가 적합.

### 3.3 차대 균형 불변식 (결정 #5) ⭐

```java
public void post(LocalDateTime now) {
    if (status != DRAFT) throw new IllegalStateException(...);
    if (lines.isEmpty()) throw new IllegalStateException(...);
    BigDecimal totalDebit  = lines.stream().map(JournalLine::getDebit).reduce(ZERO, BigDecimal::add);
    BigDecimal totalCredit = lines.stream().map(JournalLine::getCredit).reduce(ZERO, BigDecimal::add);
    if (totalDebit.compareTo(totalCredit) != 0)
        throw new UnbalancedJournalException(totalDebit, totalCredit);   // ← 복식부기의 핵심
    this.status = POSTED;
    this.postedAt = now;
}
```

- 차변 합 ≠ 대변 합이면 `UnbalancedJournalException` → `GlobalExceptionHandler` 가 **422 (또는 409)** 로 변환.
- Phase 3 `InsufficientStockException`, Phase 2 신용한도와 같은 "도메인 불변식을 코드가 지킨다" 패턴.

### 3.4 상태 머신

```
DRAFT ──post()──▶ POSTED ──cancel()──▶ CANCELLED
```

- **자동 생성 전표는 생성 즉시 POSTED** (Phase 4 의 `createAndPostFromDelivery` 처럼 draft+post 한 번에).
- 취소는 **역분개**(차/대 반대 전표 추가, append-only)로 할지 status 만 바꿀지 — 회계 정석은 역분개. 학습 단순화로 **status=CANCELLED + 역분개 전표 생성** 채택(원장 신뢰성, Phase 3 ADJUSTMENT_PLUS 와 같은 append-only 철학).

## 4. 자동 분개 규칙 (결정 #7) — 사건 → 분개

### 4.1 규칙 표 (브리핑 §6-3 재확인)

| 사건 | 차변 | 대변 | 금액 출처 |
| --- | --- | --- | --- |
| 인보이스 발행 | 매출채권(1200) | 매출(4100) + 부가세예수금(2200) | `Invoice.subtotal/taxAmount/totalAmount` |
| 출하 확정 | 매출원가(5100) | 재고자산(1400) | GoodsIssue 의 StockMovement `unit_cost × qty` 합 |
| 입고 확정 | 재고자산(1400) | 매입채무(2100) | `GoodsReceipt` 라인 `unitCost × qty` 합 |
| 입금 | 현금(1100) | 매출채권(1200) | `Payment.amount` |
| 출금 | 매입채무(2100) | 현금(1100) | `Payment.amount` |

> 부가세대급금(2300)은 매입에 부가세를 분리할 경우. 학습 1차 범위에선 매입 부가세 생략 가능 — 설계 단순화 여지(§10).

### 4.2 규칙을 어디 두나 — 과설계 회피

- ❌ 범용 "규칙 엔진"(DB 에 분개 템플릿 저장 + 인터프리터) — 학습 단계엔 과함.
- ✅ **사건별 FI 서비스 메서드 + 시스템 계정 코드 상수**:

```java
class SystemAccounts {   // 시스템이 자동 분개에 쓰는 계정 코드
    static final String AR = "1200";   // 매출채권
    static final String SALES = "4100";
    static final String VAT_PAYABLE = "2200";
    static final String COGS = "5100";
    static final String INVENTORY = "1400";
    static final String AP = "2100";
    static final String CASH = "1100";
}

@Service
class AutoJournalService {
    @Transactional
    JournalEntry createSalesEntry(InvoiceIssuedEvent e) {
        JournalEntry je = JournalEntry.draft(number, e.invoiceDate(), "매출 " + e.invoiceNumber(),
                                             "INV", e.invoiceId());
        je.addLine(account(AR),          e.totalAmount(), ZERO);   // 차) 매출채권
        je.addLine(account(SALES),       ZERO, e.subtotal());      // 대) 매출
        je.addLine(account(VAT_PAYABLE), ZERO, e.taxAmount());     // 대) 부가세예수금
        je.post(now());                                            // 차대 검증 + POSTED
        return repo.save(je);
    }
    // createCogsEntry, createPurchaseEntry, createPaymentEntry ...
}
```

→ 분개 규칙이 **읽기 쉬운 자바 코드**로 한 곳에. 계정은 코드 상수로 조회(`accountRepository.findByCode`).

## 5. 이벤트 확장 + 리스너 순서 (결정 #8, #9, #10) ⭐

### 5.1 이벤트 가족 — Phase 4 에서 확장

| 이벤트 | 발행 위치 | 본문 | 신규/기존 |
| --- | --- | --- | --- |
| `InvoiceIssuedEvent` | `InvoiceService.create` | invoiceId, number, date, subtotal, taxAmount, totalAmount | **신규** |
| `GoodsReceiptPostedEvent` | `GoodsReceiptService.post` | grId, number, date, lines(itemId, qty, unitCost) | **신규** |
| `DeliveryShippedEvent` | (Phase 4 기존) | deliveryId, warehouseId, date, lines | **재구독** |

→ SD/MM 에 **이벤트 발행 한 줄씩만 추가**. 나머지 로직 무수정. Phase 4 이벤트 패턴의 배당금.

### 5.2 회계 리스너 3종 (FI 측)

```java
@Component @RequiredArgsConstructor
class SalesAccountingListener {
    private final AutoJournalService autoJournal;
    @TransactionalEventListener(phase = BEFORE_COMMIT)
    void on(InvoiceIssuedEvent e) { autoJournal.createSalesEntry(e); }
}
```

`CogsAccountingListener`(매출원가), `PurchaseAccountingListener`(매입)도 같은 모양.

### 5.3 매출원가 단가를 어떻게 얻나 (결정 #9) ⭐ — 설계의 까다로운 지점

`DeliveryShippedEvent` 본문엔 단가가 없다(itemId, quantity 만). 출고 평균 단가는 GoodsIssue 가 post 되며 `StockMovement.unit_cost` 에 박힌다. 세 옵션:

| 옵션 | 방법 | 트레이드오프 |
| --- | --- | --- |
| A | 회계 리스너가 `deliveryId` → GoodsIssue → StockMovement(GOODS_ISSUE, refId=giId) 조회해 `Σ unit_cost×|qty|` | FI 가 MM 원장 조회(FI→MM 의존, 자연스러움). **@Order 순서 학습** 직결 |
| B | GoodsIssue.post 시 매출원가 담은 `GoodsIssuePostedEvent` 별도 발행 → 회계가 구독 | 이벤트 중첩(리스너 안에서 또 발행). 깔끔하나 흐름 추적 ↑ |
| C | `DeliveryShippedEvent` 에 단가를 미리 실어 보냄 | SD 가 원가를 알아야 함 — 책임 위반 |

→ **옵션 A 채택**. 이유:
- 학습 계획서가 콕 집은 **"`@Order` 로 리스너 순서"** 학습 포인트를 정면으로 다룸 — 재고 리스너가 StockMovement 를 먼저 만들고(낮은 Order), 회계 리스너가 나중에(높은 Order) 그걸 읽는다.
- 같은 트랜잭션이라 정합성 OK. 회계가 MM 원장을 읽는 건 회계 본질에 부합.

### 5.4 `@Order` — 한 사건, 두 리스너의 순서 (결정 #10)

`DeliveryShippedEvent` 를 **두 리스너**가 듣는다:

```java
// MM 측 (Phase 4) — 재고 차감 + StockMovement 적재
@Order(10)
@TransactionalEventListener(BEFORE_COMMIT)
void onShipped(DeliveryShippedEvent e) { goodsIssueService.createAndPostFromDelivery(e); }

// FI 측 (Phase 5) — 매출원가 분개 (StockMovement 의 unit_cost 필요)
@Order(20)   // ← 더 큰 값 = 나중 실행
@TransactionalEventListener(BEFORE_COMMIT)
void onShippedForCogs(DeliveryShippedEvent e) { autoJournal.createCogsEntry(e); }
```

```
DeliveryShippedEvent 발행
   │
   ├─ @Order(10) 재고 리스너 → GoodsIssue post → StockMovement(unit_cost 확정)
   │
   └─ @Order(20) 회계 리스너 → deliveryId 로 StockMovement 조회 → 매출원가 분개
        (10 이 먼저 끝났으므로 unit_cost 가 이미 있다)
```

> ⚠️ `@Order` 를 안 주면 순서 미보장 → 회계 리스너가 먼저 돌면 StockMovement 가 아직 없어 매출원가 0. **순서가 정합성을 좌우하는 첫 사례** — Phase 5 의 새 학습 포인트.

## 6. 입금/출금(`Payment`) — OTC 의 마지막 (결정 #12)

```
Payment
├─ number      PAY-YYYYMMDD-NNN
├─ type        RECEIPT(입금) / DISBURSEMENT(출금)
├─ party       customer_id 또는 vendor_id (type 에 따라)
├─ amount      DECIMAL
├─ payment_date DATE
└─ status      DRAFT / POSTED / CANCELLED
```

- **입금(RECEIPT)**: 고객이 외상값 갚음 → `차) 현금 / 대) 매출채권` → 채권 감소.
- **출금(DISBURSEMENT)**: 거래처에 지급 → `차) 매입채무 / 대) 현금` → 채무 감소.
- Payment 확정 시 자동 분개(직접 호출 or 이벤트 — §9 단순화 여지).

### 채권/채무 잔액 (결정 #11)

별도 잔액 테이블을 **만들지 않는다**. "고객 A 가 얼마 안 갚았나" = `매출채권 계정의 그 고객 관련 전표 라인 SUM`. 원장이 진실(Phase 3 의 "캐시 vs 원장" 에서 **원장-only** 선택 — 회계는 정확성이 성능보다 우선).

> 💡 단, "고객별" 채권을 전표 라인 SUM 으로 내려면 라인에 party 정보가 필요. 학습 1차 범위에선 **계정 전체 잔액**(전 고객 합)까지만, 고객별 보조원장(sub-ledger)은 §10 으로 미룸.

## 7. 전체 흐름 — 매출 한 건의 회계 여정

```
① 인보이스 발행  POST /api/invoices
   InvoiceService.create
     ├─ Invoice.issue() (DRAFT→ISSUED)
     └─ publish InvoiceIssuedEvent ───────▶ SalesAccountingListener
                                              └─ 차)매출채권 / 대)매출+부가세  [전표1]

② 출하 확정  POST /api/deliveries  (Phase 4 + Phase 5)
   DeliveryService.create → publish DeliveryShippedEvent
     ├─ @Order(10) 재고 리스너 → 재고 차감 + StockMovement(unit_cost)
     └─ @Order(20) 회계 리스너 → 차)매출원가 / 대)재고자산           [전표2]

③ 입금  POST /api/payments (RECEIPT)
   PaymentService → 차)현금 / 대)매출채권                            [전표3]

결과: 손익 = 매출(전표1) − 매출원가(전표2),  채권 0 (전표1 +13.2M, 전표3 −13.2M)
```

→ 영업이 누른 건 인보이스·출하·입금 세 번. 회계 전표 셋이 사람 손 없이 생성. **이게 ERP 자동 회계.**

## 11. 승인 체크리스트 ⭐

수정/이의 있는 항목은 **번호로** 알려 주세요. 다 OK면 **"승인"** 이라고 답하시면 단계 5(코드 구현)로 진입합니다.

- [ ] **1.** 새 `fi/` 모듈 — `account` / `journal` / `payment` / `integration`(sd, mm). 의존 방향 `FI → SD/MM` 단방향
- [ ] **2.** `Account` = `BaseEntityWithCode` + 자기참조 트리 + `postable` (Department 패턴 재사용)
- [ ] **3.** `AccountType`(5종) + 정상 잔액 방향을 유형에서 파생
- [ ] **4.** 시드 계정 12개 (V23) — 현금/매출채권/재고자산/매입채무/부가세예수금/매출/매출원가 등
- [ ] **5.** `JournalEntry`(헤더) + `JournalLine`(라인), 라인은 `debit`/`credit` 두 컬럼
- [ ] **6.** **차변 합 = 대변 합** 불변식 — `post()` 검증, 불일치 시 `UnbalancedJournalException`
- [ ] **7.** 전표 출처 = `sourceType`+`sourceId` 약한 참조 (INV/GI/GR/PAY/MANUAL)
- [ ] **8.** 자동 분개 = 사건별 FI 서비스 메서드 + 시스템 계정 코드 상수 (범용 규칙 엔진 X)
- [ ] **9.** 이벤트: `InvoiceIssuedEvent`(매출), `GoodsReceiptPostedEvent`(매입) 신규 + `DeliveryShippedEvent` 재구독(매출원가)

- [ ] **11.** **`@Order`** 로 재고 리스너(10) → 회계 리스너(20) 순서 보장
- [ ] **12.** 채권/채무 별도 잔액 테이블 없음 — 계정 전표 라인 SUM (원장-only)
