# 3/7. 자동 분개 규칙 — 사건이 어떻게 전표가 되는가

> "영업이 인보이스를 발행했다 → 회계 전표 한 줄 자동 생성." 이 흐름이 ERP 의 핵심 매력. 이 글은 그 매핑 규칙이 **자바 코드 어디** 에 어떤 모양으로 박혔는지를 본다 — `AutoJournalService` 와 `SystemAccounts` 상수.
>
> 설계 결정 #7: 범용 "규칙 엔진"(DB 에 분개 템플릿 + 인터프리터) 은 학습 단계에 과함. 사건별 자바 메서드로 한눈에 보이게.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/fi/journal/
├─ AutoJournalService.java     사건별 분개 메서드 5종
└─ SystemAccounts.java         시스템 계정 코드 상수 (1100/1200/4100/...)
```

---

## 🔥 5가지 사건 → 5가지 분개

ERP 의 핵심 거래는 다섯 가지. 각각이 한 가지 분개 패턴을 만든다:

| # | 사건 | 차변 | 대변 | 금액 출처 |
| --- | --- | --- | --- | --- |
| 1 | 인보이스 발행 | 매출채권(1200) | 매출(4100) + 부가세예수금(2200) | `Invoice.subtotal/taxAmount/totalAmount` |
| 2 | 출하 확정 | 매출원가(5100) | 재고자산(1400) | GoodsIssue 의 StockMovement `unit_cost × qty` 합 |
| 3 | 입고 확정 | 재고자산(1400) | 매입채무(2100) | `GoodsReceipt` 라인 `unitCost × qty` 합 |
| 4 | 입금 | 현금(1100) | 매출채권(1200) | `Payment.amount` |
| 5 | 출금 | 매입채무(2100) | 현금(1100) | `Payment.amount` |

`AutoJournalService` 에 메서드 다섯 개로 1:1 매핑:

```
createSalesEntry       (InvoiceIssuedEvent)         → #1
createCogsEntry        (DeliveryShippedEvent)        → #2
createPurchaseEntry    (GoodsReceiptPostedEvent)     → #3
createReceiptEntry     (Payment RECEIPT)             → #4
createDisbursementEntry(Payment DISBURSEMENT)        → #5
```

## 🔥 왜 "규칙 엔진" 이 아니고 자바 메서드인가 (설계 결정 #7)

세 가지 옵션:

| 옵션 | 형태 | 트레이드오프 |
| --- | --- | --- |
| A | 범용 규칙 엔진 — DB 에 분개 템플릿 저장, 런타임 인터프리트 | 새 거래 유형 추가에 SQL 만 손대면 됨. 학습 단계엔 과함. 디버깅 지옥 가능성. |
| **B** | **사건별 FI 서비스 메서드 + 코드 상수** | ✅ 분개가 한눈에 자바 코드로 읽힘. 새 거래 유형 = 메서드 추가. |
| C | 사건별 JSON 스크립트 + 사용자 정의 표현식 | 운영자가 분개 규칙을 바꿀 수 있음. 학습 가치 낮음. |

→ **옵션 B 채택**. 이유:

1. **읽기 쉬움** — "매출 분개는 어디 짜여 있어?" → IDE 에서 `createSalesEntry` 찾기.
2. **컴파일 타임 안전** — 계정 코드를 상수로 두면 오타가 컴파일 에러로 잡힌다.
3. **테스트 쉬움** — 단순한 자바 메서드라 unit test 가 자연스럽다.
4. **실무 ERP 도 보통 여기서 시작** — 회계 거래는 종류가 한정적(영업/구매/수금/지급/조정). 범용 엔진은 거래 유형이 폭증할 때만 가치 있음.

> 💡 실무에서 옵션 A 로 가려면 (1) 회계 키 테이블 (2) 키 → 차/대 라인 템플릿 (3) 금액 식 평가기 (4) 회사별/사업장별 매핑 다 만들어야 한다. SAP 의 자동 회계 모듈이 정확히 그것. 학습용 ERP 가 따라갈 길은 아니다.

## 🔥 `SystemAccounts` — 자동 분개가 쓰는 계정 코드 상수

```java
public final class SystemAccounts {
    public static final String CASH = "1100";
    public static final String AR = "1200";
    public static final String INVENTORY = "1400";
    public static final String AP = "2100";
    public static final String VAT_PAYABLE = "2200";
    public static final String SALES = "4100";
    public static final String COGS = "5100";
    private SystemAccounts() {}
}
```

설계 의도 3가지:

1. **상수만 노출** — 분개 메서드는 이 7개 코드만 안다. 분개 규칙 한 곳 + 계정 코드 한 곳 = 변경 영향 추적이 쉽다.
2. **AR / AP / COGS / VAT_PAYABLE** — 영문 약어가 회계 관용어. 한국어 이름(`MAECHUL_CHAEGWON`)보다 가독성 ↑.
3. **`final` + private 생성자** — 인스턴스화 금지. 정적 상수 모음용 클래스의 정석.

> 💡 실무 ERP 는 이 매핑을 DB(예: `accounting_key` 테이블)로 빼서 회사별/사업장별로 다르게 둔다 — 한 회사가 여러 사업부를 가질 때 매출 계정을 사업부별로 분리하는 식. Phase 5 학습 범위는 단일 회사라 코드 상수가 적절.

## 🔥 매출 분개 — `createSalesEntry`

```java
@Transactional
public JournalEntry createSalesEntry(InvoiceIssuedEvent event) {
    String number = numberGenerator.nextJournalEntryNumber(event.invoiceDate());
    JournalEntry entry = JournalEntry.draft(
            number, event.invoiceDate(),
            "매출 " + event.number(),
            JournalSource.INV, event.invoiceId());

    entry.addDebit(account(SystemAccounts.AR),           event.totalAmount());
    entry.addCredit(account(SystemAccounts.SALES),       event.subtotal());
    entry.addCredit(account(SystemAccounts.VAT_PAYABLE), event.taxAmount());

    entry.post(LocalDateTime.now(clock));
    return journalRepository.save(entry);
}
```

분개 패턴은 라인 3개:

```
차) 매출채권     totalAmount  (= subtotal + taxAmount)
대) 매출         subtotal
대) 부가세예수금 taxAmount
```

차변 합 = 대변 합 = totalAmount → `post()` 통과.

> 💡 **부가세예수금** 의 의미 — 우리가 받은 부가세는 "고객 돈을 임시로 보관 → 다음 신고 때 국세청 납부" 라는 부채. 그래서 LIABILITY. 매출 100만원 + VAT 10만원을 고객한테서 받으면, **우리 수익은 100만** 이고 **국세청에 갚을 빚이 10만** 이라는 게 회계의 진실. 110만 전체를 매출로 잡으면 안 된다.

## 🔥 매입 분개 — `createPurchaseEntry`

```java
@Transactional
public JournalEntry createPurchaseEntry(GoodsReceiptPostedEvent event) {
    BigDecimal totalCost = event.lines().stream()
            .map(l -> l.unitCost().multiply(l.quantity()))
            .reduce(BigDecimal.ZERO, BigDecimal::add);

    if (totalCost.signum() == 0) {
        log.warn("매입 0 — 전표 생략. grId={}", event.goodsReceiptId());
        return null;
    }

    String number = numberGenerator.nextJournalEntryNumber(event.receiptDate());
    JournalEntry entry = JournalEntry.draft(
            number, event.receiptDate(),
            "매입 " + event.number(),
            JournalSource.GR, event.goodsReceiptId());

    entry.addDebit(account(SystemAccounts.INVENTORY), totalCost);
    entry.addCredit(account(SystemAccounts.AP),       totalCost);

    entry.post(LocalDateTime.now(clock));
    return journalRepository.save(entry);
}
```

분개 패턴:

```
차) 재고자산  totalCost
대) 매입채무  totalCost
```

매출과 다른 두 가지:

1. **금액 계산이 라인 합산** — `Σ unitCost × quantity`. 매출은 인보이스 헤더에 이미 합계가 있어서 그대로 가져옴.
2. **부가세를 분리 안 함** — 매출은 부가세예수금을 따로 빼지만, 매입은 부가세대급금(2300)을 분리 안 한다. 학습 단순화 (설계 §4.1 ⚠️) — 매입 단가에 부가세 포함으로 가정. 추후 도입 시 라인 한 줄 추가만 하면 됨.

`totalCost == 0` 가드는 라인이 0원이면 전표 생략 — `JournalLine` 생성자가 `debit/credit > 0` 을 요구하므로 안 만들면 안전.

## 🔥 매출원가 분개 — `createCogsEntry` ⭐ (설계 결정 #9, #10)

이 글에선 큰 그림만, 자세한 건 다음 두 글(`04-리스너-순서-Order.md`, `05-매출원가-단가연결.md`).

```java
@Transactional
public JournalEntry createCogsEntry(DeliveryShippedEvent event) {
    // 1) deliveryId → GoodsIssue
    GoodsIssue gi = goodsIssueRepository.findByDeliveryId(event.deliveryId())
            .orElseThrow(() -> new IllegalStateException(
                    "GoodsIssue 가 없다 — 재고 리스너(@Order=10) 가 먼저 돌지 않았다. "
                            + "deliveryId=" + event.deliveryId()));

    // 2) StockMovement 합산 → 총 원가
    BigDecimal totalCost = stockMovementRepository
            .findByRefTypeAndRefIdAndReason("GI", gi.getId(), MovementReason.GOODS_ISSUE)
            .stream()
            .map(m -> m.getUnitCost().multiply(m.getQtyDelta().abs()))
            .reduce(BigDecimal.ZERO, BigDecimal::add);

    if (totalCost.signum() == 0) return null;

    String number = numberGenerator.nextJournalEntryNumber(event.shippedDate());
    JournalEntry entry = JournalEntry.draft(
            number, event.shippedDate(),
            "매출원가 " + gi.getNumber(),
            JournalSource.GI, gi.getId());

    entry.addDebit(account(SystemAccounts.COGS),       totalCost);
    entry.addCredit(account(SystemAccounts.INVENTORY), totalCost);

    entry.post(LocalDateTime.now(clock));
    return journalRepository.save(entry);
}
```

분개 패턴 자체는 단순:

```
차) 매출원가  totalCost
대) 재고자산  totalCost
```

핵심은 **`totalCost` 를 어떻게 구하는가** — `DeliveryShippedEvent` 본문엔 단가가 없다(itemId, quantity 만). 그래서 **이미 박힌 `StockMovement.unit_cost` 를 읽어와서 합산**. 그러려면 그 StockMovement 가 이 회계 리스너보다 먼저 만들어져 있어야 한다 → `@Order(10)` 재고 리스너 → `@Order(20)` 회계 리스너 순서 보장. 다음 두 글이 이 한 줄을 깊게 다룬다.

## 🔥 입금/출금 분개 — `createReceiptEntry` / `createDisbursementEntry`

```java
@Transactional
public JournalEntry createReceiptEntry(Long paymentId, LocalDate paymentDate,
                                       String paymentNumber, BigDecimal amount) {
    String number = numberGenerator.nextJournalEntryNumber(paymentDate);
    JournalEntry entry = JournalEntry.draft(
            number, paymentDate, "입금 " + paymentNumber,
            JournalSource.PAY, paymentId);

    entry.addDebit(account(SystemAccounts.CASH), amount);
    entry.addCredit(account(SystemAccounts.AR),  amount);

    entry.post(LocalDateTime.now(clock));
    return journalRepository.save(entry);
}
```

입금(RECEIPT):

```
차) 현금     amount   ← 우리 자산이 늘었고
대) 매출채권 amount   ← 받을 돈이 줄었다
```

출금(DISBURSEMENT) 은 거울:

```
차) 매입채무 amount   ← 갚을 돈이 줄었고
대) 현금     amount   ← 우리 자산이 줄었다
```

입금/출금만 다른 메서드 시그니처를 쓴 이유: 매출/매입/매출원가는 이벤트를 받지만, 입금/출금은 `Payment` 도메인이 같은 모듈(`fi`)에 있어서 이벤트로 분리할 이유가 없다 — 그래서 `PaymentService.createAndPost` 가 `AutoJournalService.create...Entry(...)` 를 **직접 호출**. 인자가 풀어진 형태(`paymentId`, `paymentDate`, `paymentNumber`, `amount`)인 것은 그 직접 호출에 맞춘 것.

## 🔥 `account(code)` 헬퍼 — 시드 누락 디텍션

```java
private Account account(String code) {
    try {
        return accountService.getEntityByCode(code);
    } catch (EntityNotFoundException e) {
        throw new IllegalStateException(
                "시스템 계정 " + code + " 가 마스터에 없다. V23 시드 점검 필요.", e);
    }
}
```

`accountService.getEntityByCode("1200")` 가 NotFound 면 `EntityNotFoundException` → 404 가 응답으로 가는 게 기본. 하지만 **시스템 계정이 없는 것은 운영자 입력 오류가 아니라 마이그레이션 결함** — `IllegalStateException` (500) 으로 변환하고 메시지에 V23 시드 점검 안내. 디버깅 메시지가 곧 문서.

## 🔥 호출 흐름 — 누가 부르는가

`AutoJournalService` 는 자체적으론 아무도 안 부른다. 부르는 쪽:

| 사건 | 호출자 |
| --- | --- |
| 매출(`createSalesEntry`) | `SalesAccountingListener` (`fi/integration/sd/`) ← `InvoiceIssuedEvent` |
| 매출원가(`createCogsEntry`) | `CogsAccountingListener` (`fi/integration/sd/`) ← `DeliveryShippedEvent` |
| 매입(`createPurchaseEntry`) | `PurchaseAccountingListener` (`fi/integration/mm/`) ← `GoodsReceiptPostedEvent` |
| 입금(`createReceiptEntry`) | `PaymentService.createAndPost` (같은 FI 모듈, 직접 호출) |
| 출금(`createDisbursementEntry`) | `PaymentService.createAndPost` (같은 FI 모듈, 직접 호출) |

세 가지 리스너는 다음 두 글(`04-리스너-순서-Order.md`, `05-매출원가-단가연결.md`) 에서 자세히. 패키지 위치(`fi/integration/...`) 가 의존 방향을 표현한다는 점은 Phase 4 의 `mm/integration/sd/` 와 동일.

## 🔥 클래스 레벨 `@Transactional` 이 없는 이유

`AutoJournalService` 클래스에는 `@Transactional` 이 없다. 메서드 각각만 `@Transactional`:

```java
@Slf4j
@Service
@RequiredArgsConstructor
public class AutoJournalService {   // ← 클래스 레벨 @Transactional 없음

    @Transactional
    public JournalEntry createSalesEntry(InvoiceIssuedEvent event) { ... }
    ...
}
```

- 다른 서비스(`JournalEntryService`, `AccountService`) 는 `@Transactional(readOnly = true)` 를 클래스에 붙이고 쓰기 메서드에 `@Transactional` 을 덮어쓰는 패턴.
- `AutoJournalService` 는 **읽기 메서드가 없다** — 항상 새 전표를 만들고 저장. 그래서 클래스 레벨 readOnly 가 의미 없음.

이건 자동 분개가 호출하는 쪽 트랜잭션에 **참여** (REQUIRED 기본값) 한다는 것과도 맞물린다 — 새 트랜잭션을 열지 않고 호출자(=리스너 → 호출자 = 발행자 = `InvoiceService.create`) 의 트랜잭션에 합류. 발행자가 롤백되면 분개도 롤백 (Phase 4 와 같은 패턴).

## 정리

| 학습 포인트 | 코드 위치 |
| --- | --- |
| 5가지 사건 → 5가지 분개 매핑 | 본문 표 |
| 범용 규칙 엔진 회피 (옵션 B) | `AutoJournalService` 메서드별 분리 |
| 시스템 계정 코드 상수 | `SystemAccounts` |
| 매출 = 차)AR / 대)매출+VAT | `createSalesEntry` |
| 매입 = 차)재고 / 대)AP (부가세 분리 X) | `createPurchaseEntry` |
| 입금/출금 = 직접 호출 (이벤트 X) | `PaymentService` |
| 시드 누락 → IllegalStateException 변환 | `account(code)` 헬퍼 |
| 자동 분개는 호출자 트랜잭션에 참여(REQUIRED) | 메서드별 `@Transactional` |

다음 글(`04-리스너-순서-Order.md`) 에서는 매출원가의 까다로운 지점 — **두 리스너가 한 사건을 들을 때 순서를 어떻게 보장하는가** — 를 본다.
