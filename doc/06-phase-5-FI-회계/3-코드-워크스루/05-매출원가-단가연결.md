# 5/7. 매출원가 단가 연결 — FI 가 MM 원장을 읽는다

> 매출원가 분개의 까다로움은 "**단가가 출하 시점엔 사건에 없다**" — `DeliveryShippedEvent` 는 itemId 와 quantity 만 가진다. 출고 단가는 Phase 3 의 `Stock.averageCost` 를 따라 결정되고, `GoodsIssue.post` 가 그 값을 `StockMovement.unit_cost` 에 박는다. 회계는 그 원장을 읽어 매출원가를 계산.
>
> 설계 결정 #9 옵션 A: 회계 리스너가 `deliveryId → GoodsIssue → StockMovement` 두 단계 조회로 단가 합산.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/
├─ fi/journal/AutoJournalService.java   createCogsEntry(DeliveryShippedEvent)
├─ mm/stock/StockMovementRepository.java findByRefTypeAndRefIdAndReason 추가
├─ mm/stock/StockMovement.java          refType/refId/unitCost/qtyDelta 필드
├─ mm/goodsissue/GoodsIssue.java        deliveryId 필드 (Phase 4)
└─ mm/goodsissue/GoodsIssueRepository.java findByDeliveryId (Phase 4)
```

---

## 🔥 큰 그림 — Phase 3·4·5 의 단가 릴레이

```
[Phase 3]   GoodsReceipt.post     → Stock.averageCost 갱신 (가중평균)
            GoodsIssue.post       → StockMovement(unit_cost = 직전 averageCost) 적재

[Phase 4]   Delivery.create + ship → DeliveryShippedEvent
            DeliveryEventListener  → GoodsIssueService.createAndPostFromDelivery
                                     ↓
                                     postStock() — Stock 차감 + StockMovement 적재
                                                   (unit_cost = 출고 시점 averageCost)

[Phase 5]   같은 DeliveryShippedEvent
            CogsAccountingListener → AutoJournalService.createCogsEntry
                                     ↓
                                     ① deliveryId → GoodsIssue (Phase 4 가 박아둔 매핑)
                                     ② StockMovement(GI 출처) 조회 → Σ unit_cost × |qty|
                                     ③ 차)매출원가 / 대)재고자산 분개
```

**Phase 3 의 단가 추적성** + **Phase 4 의 GoodsIssue ↔ Delivery 연결** + **Phase 5 의 @Order 순서** 가 합쳐져 매출원가가 자동으로 계산된다. 세 Phase 의 결정이 누적된 결과.

---

## 🔥 1단계 — `deliveryId → GoodsIssue` (Phase 4 가 깔아 둔 길)

```java
GoodsIssue gi = goodsIssueRepository.findByDeliveryId(event.deliveryId())
        .orElseThrow(() -> new IllegalStateException(
                "GoodsIssue 가 없다 — 재고 리스너(@Order=10) 가 먼저 돌지 않았다. "
                        + "deliveryId=" + event.deliveryId()));
```

Phase 4 가 `goods_issue.delivery_id` 컬럼을 추가하고(V21), `GoodsIssueRepository.findByDeliveryId` 를 만들어 뒀다. Phase 5 는 그것을 **재사용**.

> 💡 `deliveryId → GoodsIssue` 라는 약한 FK 매핑은 Phase 4 의 "출하 취소 → 재고 복원" 을 위해 만들었던 길. Phase 5 가 "출하 → 매출원가" 라는 새 용도로 또 쓴다. 잘 깔린 도메인 매핑은 여러 모듈이 후속 사용. **재사용성** 은 미리 설계가 아니라 결과로 나온다.

`orElseThrow` 메시지가 의도적으로 길다 — "재고 리스너가 먼저 돌지 않았다" 라는 원인까지 적시. 운영 중 이 예외가 뜨면 즉시 `@Order` 점검 신호. 디버깅 비용을 미리 갚는다.

## 🔥 2단계 — `GoodsIssue → StockMovement` (출고 원장)

```java
BigDecimal totalCost = stockMovementRepository
        .findByRefTypeAndRefIdAndReason("GI", gi.getId(), MovementReason.GOODS_ISSUE)
        .stream()
        .map(m -> m.getUnitCost().multiply(m.getQtyDelta().abs()))
        .reduce(BigDecimal.ZERO, BigDecimal::add);
```

`StockMovementRepository.findByRefTypeAndRefIdAndReason` 는 Phase 5 에서 새로 추가한 메서드:

```java
public interface StockMovementRepository
        extends JpaRepository<StockMovement, Long>, JpaSpecificationExecutor<StockMovement> {

    /**
     * 특정 출처(예: GoodsIssue) + 사유(예: GOODS_ISSUE)로 이동 원장을 조회한다.
     *
     * <p>Phase 5 의 매출원가 분개가 출하 단가를 구할 때 사용:
     * {@code findByRefTypeAndRefIdAndReason("GI", giId, GOODS_ISSUE)} → 라인별 {@code unitCost × |qtyDelta|} 합산.
     */
    List<StockMovement> findByRefTypeAndRefIdAndReason(String refType, Long refId, MovementReason reason);
}
```

Spring Data JPA 의 method name query — 메서드 이름만으로 쿼리 자동 생성. 본문 0줄.

### 왜 `reason = GOODS_ISSUE` 도 같이 거는가

`StockMovement` 의 `(refType, refId)` 는 출처 GoodsIssue 한 건이지만, 그 GI 의 **취소**(`POSTED → CANCELLED`) 시 `MovementReason.ADJUSTMENT_PLUS` 행이 같은 (`refType="GI"`, `refId`) 로 추가된다 (Phase 4 `GoodsIssueService.cancel`). 매출원가 합산엔 GOODS_ISSUE 만 — 그래서 `reason` 필터.

> 💡 Phase 4 의 "취소 시 복원 행 추가" 가 Phase 5 의 조회 조건을 바꾸는 식으로 이어진다. 도메인 모듈은 외따로 살지 않는다 — 한 모듈의 결정이 다른 모듈의 코드를 바꾼다.

## 🔥 단가 합산 — `unitCost × |qtyDelta|`

```java
.map(m -> m.getUnitCost().multiply(m.getQtyDelta().abs()))
.reduce(BigDecimal.ZERO, BigDecimal::add);
```

- **`qtyDelta.abs()`** — 출고는 `qty_delta < 0` 이라 절댓값으로 양수화. 단가는 항상 양수라 곱셈 결과가 양수.
- **`reduce(ZERO, BigDecimal::add)`** — 합산.

라인이 2개(노트북 5 × 800,000 + 모니터 3 × 200,000) 인 출하면 결과 = 4,000,000 + 600,000 = 4,600,000. 분개:

```
차) 매출원가 4,600,000
대) 재고자산 4,600,000
```

차변 합 = 대변 합 = 4,600,000 → `post()` 통과.

## 🔥 0원 가드 — 라인이 0이면 전표 생략

```java
if (totalCost.signum() == 0) {
    log.warn("매출원가 0 — 전표 생략. deliveryId={}, giId={}", event.deliveryId(), gi.getId());
    return null;
}
```

매출원가 0 인 출하 (= 모든 라인이 무상 출고 또는 시드 단가 없음) 는 회계적으로 의미 없는 전표가 된다. 게다가 `JournalLine` 생성자가 `debit > 0 OR credit > 0` 을 요구하므로 만들면 예외. 그래서 **만들지 않고 경고만 남김**.

> ⚠️ 학습용 데이터에서 이 분기가 발동했다면 거의 항상 시드 데이터의 문제다 (단가 0으로 입고 → 단가 0으로 출고). 실무에선 무상 출고/샘플 출고가 별도 분개를 가지므로 사유(`reason`) 분기가 필요.

## 🔥 왜 FI 가 MM 원장을 직접 읽는가 — 설계 철학

옵션 A 채택의 깊은 이유 (설계 §5.3):

| 옵션 | 책임 분리 관점 |
| --- | --- |
| A (현 채택) | **회계가 다른 모듈의 사실을 읽어와 정리** — 회계 본질에 부합 |
| B (별도 이벤트) | 깔끔하지만 이벤트가 이벤트를 부르는 중첩 구조 |
| C (이벤트에 단가 포함) | SD 가 원가를 알아야 — SD의 책임 위반 |

**회계 모듈의 본질** 은 "다른 모듈에서 일어난 사실들을 화폐 가치로 정리해 손익·재무상태를 도출" — 즉 **읽는 모듈**. 영업 모듈은 회계가 어떻게 정리하는지 몰라도 되지만, 회계는 영업/구매/재고가 어떻게 굴러가는지 알아야 한다.

이게 패키지 의존 방향과도 일치 — `FI → SD`, `FI → MM` 단방향 (Phase 4 의 `MM → SD` 와 같은 방향성 원칙):

```
[FI]
 ├─ AutoJournalService     ───reads──→ [MM] StockMovementRepository
 │                                          GoodsIssueRepository
 └─ integration/sd/        ───listens─→ [SD] DeliveryShippedEvent
                                            InvoiceIssuedEvent
                                       [MM] GoodsReceiptPostedEvent

SD/MM 은 FI 의 존재를 모름 — fi.* 를 import 하는 SD/MM 코드 없음.
```

→ **회계는 모든 모듈을 알지만, 현장 모듈은 회계를 모른다** — 이 회계의 본질이 패키지 구조로 그대로 표현된다.

## 🔥 같은 트랜잭션이라 정합성 OK

`stockMovementRepository.findByRefTypeAndRefIdAndReason` 호출은 **같은 트랜잭션** 안에서 일어난다 (BEFORE_COMMIT 단계). 그 트랜잭션 안에서 직전에 (@Order=10 의 재고 리스너가) save 한 StockMovement 가 보일까?

답: **보인다**. JPA EntityManager 는 same-transaction commit 전이라도 flush 후 SELECT 시 영속성 컨텍스트의 변경을 반영한다. JPA의 auto-flush (FlushModeType.AUTO) 가 SELECT 직전에 미커밋 INSERT/UPDATE 를 flush. 그래서 같은 트랜잭션 안에서 "방금 저장한 StockMovement" 도 조회 결과에 포함.

> 💡 만약 자동 flush 가 꺼져 있거나(`FlushModeType.COMMIT`), 다른 트랜잭션이었다면 — 본 InSERT 가 commit 전이라 보이지 않아 매출원가 0. **같은 트랜잭션** 이라는 것이 옵션 A 의 전제.

테스트(`FiAccountingIntegrationTest`)가 이 보장을 정확히 검증 — 매출원가 분개의 totalDebit 이 `unitCost × qty` 와 정확히 일치해야 통과.

## 🔥 정합성 검증 — 만약 단가가 어긋났다면?

`StockMovement.unit_cost` 가 잘못 박혔거나, `qtyDelta` 가 잘못 음수가 됐다면 어떻게 될까:

| 시나리오 | 결과 |
| --- | --- |
| `unit_cost = 0` 으로 박힘 | totalCost = 0 → 전표 생략 (위 가드) |
| `qtyDelta = 0` 으로 박힘 | `StockMovement.of` 가 거부 (Phase 3 의 도메인 검증) |
| `unit_cost = 음수` | `StockMovement.of` 가 거부 (`unitCost.signum() < 0` 검사) |

→ 잘못된 입력이 매출원가까지 흘러갈 가능성이 거의 없도록 **상류 모듈이 자기 데이터를 먼저 검증**. Phase 5 회계는 그 위에 올라가기만 한다. 책임 분리의 정석.

## 🔥 차이 — 매입 단가는 사건에 직접 실음

비교 — `createPurchaseEntry` 는 단가를 사건 본문에서 그대로 가져온다:

```java
BigDecimal totalCost = event.lines().stream()
        .map(l -> l.unitCost().multiply(l.quantity()))   // 사건 본문에 unitCost
        .reduce(BigDecimal.ZERO, BigDecimal::add);
```

`GoodsReceiptPostedEvent.Line(itemId, quantity, unitCost)` — `unitCost` 가 사건에 포함. 왜?

- **매입 단가** 는 사용자가 입력한 값 (`GoodsReceipt` 라인의 `unitCost`) — 사건 발생 시점에 이미 확정.
- **매출원가** 는 시스템이 계산한 값 (출고 시점의 `Stock.averageCost`) — 사건 발생 시점엔 미정.

→ 매입은 옵션 C 처럼 사건 본문에 단가 실음, 매출원가는 옵션 A 처럼 원장 조회. 단가의 **발견 시점** 에 따라 옵션이 갈린다.

## 🔥 정리 — 세 Phase 의 결정이 누적

| Phase | 결정 | Phase 5 에서의 쓰임 |
| --- | --- | --- |
| Phase 3 | `StockMovement(unit_cost, refType, refId)` — 원장에 출고 단가 박음 | 매출원가 조회 대상 |
| Phase 4 | `GoodsIssue.deliveryId` + `findByDeliveryId` — 출하↔GI 매핑 | deliveryId → giId 변환 |
| Phase 4 | `DeliveryShippedEvent` 발행 | CogsAccountingListener 가 구독 |
| **Phase 5** | `@Order(10) → @Order(20)` | StockMovement 가 박힌 뒤 회계가 읽는 순서 보장 |
| **Phase 5** | `findByRefTypeAndRefIdAndReason` | GOODS_ISSUE 사유만 합산 (취소 ADJUSTMENT_PLUS 제외) |
| **Phase 5** | `AutoJournalService.createCogsEntry` | 두 단계 조회 → 분개 |

이 다섯이 모여야 출하 한 번에 매출원가 자동 분개가 박힌다. 학습 가치가 가장 높은 부분 중 하나 — **모듈 간 데이터가 흐르는 모양** 의 실전 케이스.

| 학습 포인트 | 코드 위치 |
| --- | --- |
| 매출원가의 단가는 사건 본문에 없다 | `DeliveryShippedEvent` |
| 두 단계 조회: deliveryId → GI → StockMovement | `createCogsEntry` |
| Phase 4 매핑(`findByDeliveryId`) 의 재사용 | — |
| `qtyDelta.abs()` 로 출고 음수를 양수화 | `createCogsEntry` |
| `reason = GOODS_ISSUE` 필터로 취소 행 제외 | `findByRefTypeAndRefIdAndReason` |
| 0원 가드로 빈 전표 회피 | `if (totalCost.signum() == 0) return null` |
| 같은 트랜잭션의 auto-flush 가 정합성 보장 | JPA `FlushModeType.AUTO` |
| 매입은 사건 본문에 단가 실음 (옵션 C) | `createPurchaseEntry` |
| FI → MM 단방향 의존 (회계는 다른 모듈을 읽음) | 패키지 구조 |

다음 글(`06-Flyway-V23-V25.md`) 에서는 이 모든 도메인을 떠받치는 **스키마와 시드** 를 본다.
