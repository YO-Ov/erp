# 7/7. 회계 시나리오 테스트 — 거래 한 묶음이 닫히는 모양

> 단위 테스트는 도메인 객체의 불변식을 검증한다 — `JournalEntry` 가 차/대 일치를 거부하는지, `Account` 가 헤더 계정을 만들어 내는지. 통합 테스트는 그 위에 한 단계 — **"사건 → 자동 분개 → 잔액"** 의 흐름이 끝까지 닫히는지를 검증.
>
> Phase 5 의 모든 결정(@Order, 이벤트, 자동 분개, 차/대 균형)이 **한 테스트 안에서 한꺼번에 통과** 해야 의미가 있다.

대상 파일:

```
hwlee-erp/src/test/java/com/hwlee/erp/
├─ fi/
│  ├─ account/AccountTest.java           단위 — 트리, 유형→방향, 입력 검증
│  ├─ journal/JournalEntryTest.java      단위 — 차/대 균형, 상태 머신, 헤더 계정 거부
│  └─ FiAccountingIntegrationTest.java   통합 — 5가지 자동 분개 + OTC 전체 시나리오
└─ ...
```

## 🔥 단위 vs 통합 — 무엇을 어느 쪽에서 검증하는가

| 검증 대상 | 단위 | 통합 |
| --- | --- | --- |
| `Account` 생성 시 빈 코드 거부 | ✅ | — |
| `AccountType.ASSET.getNormalSide() == DEBIT` | ✅ | — |
| 헤더 계정(postable=false)에 라인 거부 | ✅ | — |
| 차변 합 ≠ 대변 합이면 `UnbalancedJournalException` | ✅ | — |
| 인보이스 발행 → 매출 분개 자동 생성 | — | ✅ |
| 출하 → 매출원가 분개 (단가가 정확히 합산) | — | ✅ |
| 입금 → 매출채권 잔액 감소 | — | ✅ |
| OTC 전체 시나리오에서 채권 잔액 = 0 | — | ✅ |

원칙:
- **도메인 객체 한 개의 규칙** → 단위. Spring Boot 컨텍스트 부팅 없이 빠르게 (`AccountTest`, `JournalEntryTest`).
- **여러 모듈을 잇는 흐름** → 통합. `@SpringBootTest` + Testcontainers + 이벤트·트랜잭션 실제 동작.

## 🔥 단위 — `JournalEntryTest`

`차변_대변_불일치_시_예외`:

```java
@Test
@DisplayName("차변 합 ≠ 대변 합 → UnbalancedJournalException, 상태는 DRAFT 유지")
void 차변_대변_불일치_시_예외() {
    JournalEntry je = JournalEntry.draft(
            "JE-20260524-002", today, "잘못된 분개", JournalSource.MANUAL, null);
    je.addDebit(ar,    new BigDecimal("1000"));
    je.addCredit(sales, new BigDecimal("999"));   // 1원 불일치

    assertThatThrownBy(() -> je.post(now))
            .isInstanceOf(UnbalancedJournalException.class)
            .hasMessageContaining("1000")
            .hasMessageContaining("999");

    assertThat(je.getStatus())
            .as("post 실패 시 상태는 DRAFT 그대로 — 트랜잭션 롤백 없이도 도메인이 안전")
            .isEqualTo(JournalEntryStatus.DRAFT);
}
```

세 가지 검증:
1. `UnbalancedJournalException` 발생.
2. 메시지에 양쪽 금액이 들어있어 디버깅 가능.
3. 예외 후에도 상태가 DRAFT — 도메인 객체가 "반쯤 망가진 상태" 로 남지 않음.

세 번째가 중요한 학습 포인트 — 트랜잭션이 롤백되지 않아도 도메인이 안전하다. "in-memory 안전성" 이라는 또 다른 차원의 정합성.

`헤더_계정에는_라인_불가`:

```java
@Test
void 헤더_계정에는_라인_불가() {
    JournalEntry je = JournalEntry.draft(...);
    assertThatThrownBy(() -> je.addDebit(asset, new BigDecimal("1000")))
            .isInstanceOf(IllegalArgumentException.class)
            .hasMessageContaining("postable=false");
}
```

→ `Account.postable = false` 인 헤더 계정(자산 1000 등)에 라인을 달려고 하면 거부. 도메인 객체가 자기 규칙을 지킨다.

| 테스트 | 검증 |
| --- | --- |
| `입고_확정_시_매입_분개_자동_생성` | 입고 → `차)재고자산 / 대)매입채무` |

### 가장 중요한 테스트 한 개 — 전체 OTC 시나리오

```java
@Test
@DisplayName("매입→출하→인보이스→입금 전체 시나리오 — 채권 잔액 0 / 손익 = 매출 - 매출원가")
void 전체_OTC_시나리오로_채권이_닫히고_손익이_맞는다() {
    var ctx = setupMasters();

    // 입고 10대 × 800,000 → 매입 전표 8,000,000
    stockUp(ctx, 10, bd(800000));
    var so = createConfirmedOrder(ctx, 6, bd(1200000));

    // 출하 6대 → 매출원가 전표 4,800,000 (= 6 × 800,000)
    deliveryService.create(new DeliveryCreateRequest(
            so.salesOrderId, ctx.warehouseId, LocalDate.now(),
            List.of(new DeliveryLineRequest(so.salesOrderLineId, bd(6)))));

    // 인보이스 6대 → 매출 7,200,000 + 부가세 720,000, 매출채권 +7,920,000
    invoiceService.create(new InvoiceCreateRequest(
            so.salesOrderId, LocalDate.now(),
            List.of(new InvoiceLineRequest(so.salesOrderLineId, bd(6)))));

    // 입금 7,920,000 → 매출채권 -7,920,000, 현금 +7,920,000
    paymentService.createAndPost(new PaymentCreateRequest(
            PaymentType.RECEIPT, ctx.customerId, null,
            new BigDecimal("7920000"), LocalDate.now(), "수금"));

    // 매출채권 잔액 = +7,920,000 (인보이스) - 7,920,000 (입금) = 0
    assertThat(accountBalance(SystemAccounts.AR))
            .as("외상값을 모두 회수했으므로 매출채권 잔액 0")
            .isEqualByComparingTo(BigDecimal.ZERO);

    // 손익 = 매출 7,200,000 - 매출원가 4,800,000 = 2,400,000
    BigDecimal sales = accountBalance(SystemAccounts.SALES).abs();
    BigDecimal cogs  = accountBalance(SystemAccounts.COGS);
    assertThat(sales.subtract(cogs))
            .as("총이익 = 매출(7.2M) - 매출원가(4.8M) = 2.4M")
            .isEqualByComparingTo("2400000");
}
```

이 한 테스트가 검증하는 것:

1. **매입 자동 분개** 가 입고 확정 시점에 박힘 (`stockUp` 안에서).
2. **매출원가 자동 분개** 가 출하 확정 시점에 박힘. `@Order(10) → @Order(20)` 가 올바르게 동작.
3. **매출 자동 분개** 가 인보이스 발행 시점에 박힘. 부가세 분리.
4. **입금 자동 분개** 가 채권을 정확히 상쇄.
5. **계정 잔액**(원장 SUM) 이 비즈니스 직관과 일치 — 채권 0, 손익 2.4M, 부가세예수금 0.72M (다음 신고 시 납부 대상).

→ "거래 한 묶음이 닫힌다" 의 회계적 의미가 그대로 어셔션이 됨. 이게 ERP 회계 모듈의 완성 모양.

## 🔥 `accountBalance` 헬퍼 — 원장 SUM (결정 #11)

```java
/**
 * 계정 잔액 — POSTED 전표의 그 계정 라인 (debit - credit) 합.
 * 차변 정상 계정(자산/비용)은 양수, 대변 정상 계정(부채/자본/수익)은 음수가 정상 방향.
 */
private BigDecimal accountBalance(String accountCode) {
    return journalRepository.findPostedLinesByAccountCode(accountCode).stream()
            .map(line -> line.getDebit().subtract(line.getCredit()))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
}
```

**별도 잔액 테이블 없음** (설계 결정 #11). 원장(`journal_line`) 이 진실:
- 채권/채무/현금/매출 잔액 모두 `SUM(debit) - SUM(credit)` 로 즉시 계산.
- Phase 3 의 "Stock = 캐시, StockMovement = 원장" 의 자매 결정 — 회계는 더 엄격해 **캐시조차 두지 않음**.

테스트에선 단순한 stream 합산이지만, 실무에선 이게 곧 **계정원장 (account ledger)** 조회 — 한 계정의 모든 거래 + 누계 잔액을 보여주는 표.

`findPostedLinesByAccountCode` — POSTED 만 합산 (CANCELLED 제외):

```java
@Query("SELECT l FROM JournalLine l "
        + "JOIN FETCH l.account a "
        + "WHERE a.code = :accountCode "
        + "AND l.journalEntry.status = com.hwlee.erp.fi.journal.JournalEntryStatus.POSTED")
List<JournalLine> findPostedLinesByAccountCode(@Param("accountCode") String accountCode);
```

`JOIN FETCH` 가 두 가지 일을 한다:
1. **`account` 까지 함께 페치** — 테스트가 트랜잭션 밖에서 account 를 참조해도 LazyInit 안 남.
2. **`status = POSTED` 필터** — CANCELLED 가 잔액에 끼지 않음.

## 🔥 JOIN FETCH 패턴 — LazyInit 회피

`findBySourceTypeAndSourceIdWithLines`:

```java
@Query("SELECT DISTINCT je FROM JournalEntry je "
        + "LEFT JOIN FETCH je.lines l "
        + "LEFT JOIN FETCH l.account "
        + "WHERE je.sourceType = :sourceType AND je.sourceId = :sourceId "
        + "ORDER BY je.id ASC")
List<JournalEntry> findBySourceTypeAndSourceIdWithLines(...);
```

테스트가 트랜잭션 밖에서 `je.getLines()` 를 부르려면 라인이 미리 fetch 돼 있어야 한다. `LEFT JOIN FETCH je.lines` 한 줄로 N+1 회피 + LazyInit 회피.

> 💡 일반 메서드 `findBySourceTypeAndSourceId` 는 그대로 둠 — 같은 트랜잭션 안에서만 쓰는 경우 lazy 가 더 효율적. WithLines 버전은 트랜잭션 밖에서 라인 순회가 필요한 케이스용. 같은 의도의 두 메서드를 가질 때의 명명 관습.

## 🔥 테스트 데이터 격리 — `uniqueBusinessNo()` 패턴

```java
private static final AtomicLong SEQ = new AtomicLong(System.nanoTime());

private static String uniqueBusinessNo() {
    long n = SEQ.incrementAndGet();
    return String.format("%03d-%02d-%05d", ...);
}
```

Phase 4 의 `SdMmIntegrationTest` 와 동일 — 같은 컨텍스트 안에서 여러 테스트가 돌면 customer/vendor 의 UNIQUE 제약과 충돌. nano 시간 + 시퀀스로 매번 유일한 사업자번호를 생성.

> 💡 대안: 각 `@Test` 메서드에 `@Transactional` + 롤백. 하지만 `@TransactionalEventListener(BEFORE_COMMIT)` 가 테스트 트랜잭션이 commit 안 되면 안 발동 → 이벤트 검증 불가. **테스트 격리는 데이터 유일화 / 롤백 두 가지 중 하나** 를 선택해야 하고, BEFORE_COMMIT 리스너를 검증하는 우리는 데이터 유일화 채택.

## 🔥 어느 어셔션이 깨지면 무엇이 잘못인가

학습용 가이드 — 통합 테스트가 실패할 때 어디부터 보나:

| 깨지는 어셔션 | 의심할 곳 |
| --- | --- |
| `entries.hasSize(1)` 가 0 | 이벤트 발행이 안 됐거나 (`InvoiceService.create` 의 `publishEvent` 호출 누락), 리스너가 안 등록 (`@Component` 누락) |
| `totalDebit ≠ totalCredit` | 자동 분개 메서드의 라인 계산 오류 |
| 매출원가 totalDebit 이 0 | `@Order` 미설정 — 회계 리스너가 재고 리스너보다 먼저 실행 |
| 매출원가 totalDebit 이 잘못된 값 | `findByRefTypeAndRefIdAndReason` 의 reason 필터 누락 (취소 행이 합산됨) |
| 채권 잔액 ≠ 0 | 입금 분개의 라인 방향 잘못 (차/대 거꾸로) |
| `chk_journal_line_side` CHECK 위반 | 라인 한 쪽만 양수가 깨졌거나 양쪽 0 |
| `chk_payment_party` CHECK 위반 | type/party 조합 잘못 |
| `IllegalStateException("시스템 계정 ... 없음")` | V23 시드 누락 |
| `IllegalStateException("GoodsIssue 가 없다")` | `@Order(10)` 누락 — Phase 4 리스너가 회계 리스너보다 늦게 실행 |

→ 한 테스트가 깨질 때 어디를 봐야 할지 디버깅 지도. 실패 메시지 디자인의 가치.

## 🔥 부가세예수금 — 닫히지 않는 잔액의 의미

전체 OTC 시나리오 끝에서:

```java
assertThat(accountBalance(SystemAccounts.VAT_PAYABLE).abs())
        .as("부가세예수금 잔액 — 다음 부가세 신고 시 납부 대상")
        .isEqualByComparingTo("720000");
```

채권은 0으로 닫혔지만 **부가세예수금은 720,000 이 남는다**. 이게 왜 정상인가:

| 사건 | 부가세예수금 변화 |
| --- | --- |
| 인보이스 발행 | +720,000 (대변, 부채 증가) |
| 입금 | 0 (현금/채권 사이의 이동) |
| **잔액** | **-720,000 in (debit-credit), 즉 720,000 대변 잔액** |

이 720,000 은 "고객한테서 받은 부가세 = 다음 분기 신고 시 국세청에 납부해야 할 빚". 부가세 신고가 따로 일어나야 비로소 ZERO 로 닫힌다. → 학습용 ERP 에선 그 단계 생략(설계 §10 미루는 것).

→ 회계의 모든 계정이 OTC 한 묶음으로 닫히지는 않는다. **자기 사이클** 이 있다 — 부가세는 분기 단위, 매출채권은 거래 단위, 재고자산은 입출고 단위. 각 사이클이 잘 작동하는지가 ERP 회계의 핵심.

## 🔥 Phase 5 회계 학습의 완결

Phase 5 7편의 워크스루를 통과한 시점에서 다음을 말할 수 있어야 한다:

- **복식부기** — 한 거래의 차변 합 = 대변 합. `JournalEntry.post()` 가 지킨다.
- **계정과목** — 5대 유형이 정상 잔액 방향을 파생. 트리 + postable.
- **자동 분개** — 5가지 사건이 5가지 분개 패턴. `AutoJournalService` 한 클래스에.
- **@Order** — 한 사건을 두 리스너가 들을 때 순서가 정합성을 좌우. 매출원가가 그 대표 케이스.
- **FI → MM 단방향** — 회계는 다른 모듈을 읽지만, 다른 모듈은 회계를 모름. 회계의 본질이 패키지로 표현됨.
- **원장이 진실** — 별도 잔액 테이블 없음. `SUM(debit-credit)` 가 정확성이 성능보다 우선되는 결정의 표현.
- **방어선 다중화** — 도메인 객체 + DB CHECK + 명시적 가드. 회계 정합성을 두세 겹으로 잠금.

→ "왜 ERP 에서 회계 모듈이 가장 중요한가" 의 답이 어느 정도 보일 것. 회계는 모든 거래의 **최종 정착지** — 영업/구매/재고/생산 어디서 일어난 일이든 마지막엔 회계 전표 한 줄로 정리. 이 한 줄이 틀리면 회사의 손익과 재무상태가 거짓이 된다.

다음은 Phase 6(인증/인가) — 그 회계를 누가 보고/누가 만들 수 있는지의 문제로 옮겨간다.
