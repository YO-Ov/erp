# Phase 5 시연 가이드 — FI 자동 회계

> 7단계 사이클의 마지막 — **단계 7: 시연 + 회고**.
> 도메인 브리핑 → 모델링 → 코드 워크스루 7편으로 FI 의 설계와 구현을 머리로 이해했다.
> 이번에는 Swagger UI / cURL / MySQL CLI 로 **"거래 한 번이 자동으로 회계 전표가 되는 흐름"** 을 손으로 확인한다.

---

## 시연의 목표 (Phase 5 가 무엇을 보장하는가)

Phase 1~4 가 영업/구매/재고를 만들었지만, 그 거래들이 **회계 장부로 들어가는 길** 은 없었다. Phase 5 가 그 길을 만든다.

이번에 확인할 **5대 사실**:

1. **거래 = 자동 분개** — 입고/인보이스/출하/입금/출금 어느 하나만 하면 회계 전표가 한 트랜잭션 안에서 자동으로 박힌다. 사용자가 분개를 입력하지 않는다.
2. **차변 합 = 대변 합 — 복식부기 강제** — 잘못된 분개(차/대 불일치) 는 422 UNPROCESSABLE_ENTITY 로 거부. 도메인이 회계 원리를 직접 지킨다.
3. **@Order 가 정합성을 좌우** — 한 사건(`DeliveryShippedEvent`) 을 두 모듈(MM 재고 + FI 회계) 이 듣는다. 재고가 먼저 빠진 뒤 회계가 매출원가를 계산. 순서가 깨지면 매출원가 0.
4. **원장이 진실** — 별도 잔액 테이블 없음. 채권/채무/현금 잔액 모두 전표 라인 `SUM(debit - credit)` 으로 조회.
5. **OTC 한 묶음이 닫힌다** — 매입→출하→인보이스→입금 끝에 매출채권 잔액 0, 손익 = 매출 - 매출원가 (단, 부가세예수금은 신고 전이라 남는다).

> 💡 한 문장: **"거래가 일어나는 곳마다 회계 전표가 자동으로 따라간다 — 사용자는 회계를 의식하지 않는다"**. 이게 ERP 가 단순히 데이터 관리가 아니라 **자기 회계가 닫히는 시스템** 인 이유.

## STEP 4. 차변 합 ≠ 대변 합 → 422 거부 ⭐

복식부기 핵심 — 1원이라도 어긋나면 거부.

```bash
curl -i -X POST http://localhost:8080/api/journal-entries \
  -H 'Content-Type: application/json' \
  -d '{
    "entryDate": "2026-05-29",
    "description": "잘못된 분개 — 1원 어긋남",
    "lines": [
      { "accountCode": "1100", "debit": 100000, "credit": 0 },
      { "accountCode": "4100", "debit": 0, "credit": 99999 }
    ]
  }'
```

기대:

```
HTTP/1.1 422
Content-Type: application/problem+json

{
  "type": "about:blank",
  "title": "Unbalanced Journal Entry",
  "status": 422,
  "detail": "전표 차/대 불일치 — 차변 합 100000, 대변 합 99999",
  "code": "UNBALANCED_JOURNAL",
  "totalDebit": 100000,
  "totalCredit": 99999
}
```

확인 포인트:
- `422 UNPROCESSABLE_ENTITY` — 입력 형식은 맞지만 도메인 규칙 위반.
- `totalDebit`, `totalCredit` 두 값이 응답에 포함 — 클라이언트가 어디서 어긋났는지 즉시 안다.

다른 사례 — **헤더 계정(postable=0) 에 라인 시도**:

```bash
curl -i -X POST http://localhost:8080/api/journal-entries \
  -H 'Content-Type: application/json' \
  -d '{
    "entryDate": "2026-05-29",
    "description": "헤더 계정 시도",
    "lines": [
      { "accountCode": "1000", "debit": 100, "credit": 0 },
      { "accountCode": "4100", "debit": 0, "credit": 100 }
    ]
  }'
```

기대: `400 Bad Request` (IllegalArgumentException) — `"헤더 계정에는 라인을 달 수 없다 — postable=false: code=1000"`.

## STEP 5. 매입 자동 분개 — 입고 한 번에 회계 전표 ⭐

# 상품

  -H 'Content-Type: application/json' \

## STEP 7. 매출원가 자동 분개 — @Order 의 성과 ⭐

STEP 6 의 출하 시점에 이미 매출원가 분개도 같이 박혔다. 그것을 확인.

먼저 DLV_ID 에 자동 생성된 GoodsIssue id 를 찾는다:

```bash
GI_ID=$(curl -s "http://localhost:8080/api/goods-issues?deliveryId=$DLV_ID" 2>/dev/null | jq -r '.content[0].id')
echo "GI_ID=$GI_ID"
```

> 만약 `?deliveryId` 검색이 없으면 (Phase 4 의 GoodsIssueController 가 그 필터를 노출 안 했을 수 있음), MySQL 로 직접:
> ```bash
> docker exec hwlee-erp-mysql mysql -uerp -perp -D erp_db -sNe "SELECT id FROM goods_issue WHERE delivery_id=$DLV_ID;"
> ```

이제 매출원가 전표 조회 — `sourceType=GI, sourceId=GI_ID`:

```bash
curl -s "http://localhost:8080/api/journal-entries?sourceType=GI&sourceId=$GI_ID" | jq '.content[0]'
```

기대:

```json
{
  "description": "매출원가 GI-20260529-001",
  "status": "POSTED",
  "sourceType": "GI",
  "totalDebit": 4800000.00,
  "totalCredit": 4800000.00,
  "lines": [
    { "accountCode": "5100", "accountName": "매출원가", "debit": 4800000.00, "credit": 0.00 },
    { "accountCode": "1400", "accountName": "재고자산", "debit": 0.00,       "credit": 4800000.00 }
  ]
}
```

확인 포인트:
- **매출원가 = 6대 × 800,000원 = 4,800,000** — 출고된 단가가 정확히 합산됨.
- 차)매출원가 / 대)재고자산.
- **이 전표가 정확하려면 @Order(10) MM 리스너가 먼저 돌아 `StockMovement.unit_cost` 를 박은 뒤, @Order(20) FI 리스너가 그것을 읽어와야 한다**. 워크스루 04, 05편의 핵심 결정이 실제로 동작했다.

서버 로그에서 두 리스너의 순서를 본다:

```
o.s.t.e.s.DeliveryEventListener           : 출하 확정 사건 수신: deliveryId=..., warehouseId=..., lines=1
c.h.e.f.i.s.CogsAccountingListener        : 출하 확정 사건 수신(매출원가용): deliveryId=...
c.h.e.f.journal.AutoJournalService        : 매출원가 자동 전표 생성: JE-... (deliveryId=..., giId=..., totalCost=4800000)
```

→ MM 리스너 로그가 **먼저** 찍히고 FI 리스너 로그가 **나중에** 찍히는지 확인. `@Order` 가 보장.

## STEP 9. 원장 SUM 으로 잔액 검증 — 한 묶음이 닫히는 모양 ⭐

별도 잔액 테이블이 없으므로 MySQL 로 원장을 직접 합산해 본다 (설계 결정 #11):

```bash
docker exec hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
SELECT
    a.code, a.name, a.type,
    ROUND(SUM(jl.debit), 0)  AS total_debit,
    ROUND(SUM(jl.credit), 0) AS total_credit,
    ROUND(SUM(jl.debit) - SUM(jl.credit), 0) AS balance
FROM journal_line jl
JOIN journal_entry je ON je.id = jl.journal_entry_id
JOIN account a        ON a.id  = jl.account_id
WHERE je.status = 'POSTED'
  AND a.code IN ('1100','1200','1400','2100','2200','4100','5100')
GROUP BY a.code, a.name, a.type
ORDER BY a.code;"
```

기대 (위 STEP 5~8 만 수행한 상태):

```
+------+----------------+-----------+-------------+--------------+----------+
| code | name           | type      | total_debit | total_credit | balance  |
+------+----------------+-----------+-------------+--------------+----------+
| 1100 | 현금           | ASSET     |     7920000 |            0 |  7920000 |   ← +7.92M 현금 보유
| 1200 | 매출채권       | ASSET     |     7920000 |      7920000 |        0 |   ← ⭐ 채권 닫힘
| 1400 | 재고자산       | ASSET     |     8000000 |      4800000 |  3200000 |   ← 4대 남음 × 800K
| 2100 | 매입채무       | LIABILITY |           0 |      8000000 | -8000000 |   ← 거래처 미지급 8M
| 2200 | 부가세예수금   | LIABILITY |           0 |       720000 |  -720000 |   ← 다음 신고 시 납부
| 4100 | 매출           | REVENUE   |           0 |      7200000 | -7200000 |   ← 매출 7.2M
| 5100 | 매출원가       | EXPENSE   |     4800000 |            0 |  4800000 |   ← 원가 4.8M
+------+----------------+-----------+-------------+--------------+----------+
```

읽는 법:
- **차변 정상 계정**(자산/비용) 은 balance 가 양수면 정상 (자산이 그만큼 있음, 비용이 그만큼 발생).
- **대변 정상 계정**(부채/수익) 은 balance 가 음수면 정상 — `debit - credit` 이라 대변 누계가 음수로 표시될 뿐, 실제 잔액은 `|balance|` (예: 부가세예수금 잔액 = 720,000).

검증:

| 검증 | 값 | 의미 |
| --- | --- | --- |
| 매출채권 잔액 = 0 | ✅ | 7.92M 청구 + 7.92M 회수 — **OTC 한 묶음이 닫혔다** |
| 매출 - 매출원가 = 손익 | 7,200,000 − 4,800,000 = **2,400,000** | 이번 거래의 총이익 |
| 매입채무 잔액 = -8M | 거래처에 8M 미지급 | 다음 출금(STEP 10)으로 해소 |
| 부가세예수금 = -720K | 다음 부가세 신고 시 납부 | Phase 5 범위 밖 |

## STEP 10. 출금 — 매입채무 절반 갚기

## 회고 (Phase 5 학습 일지 작성)

`LEARNING-LOG.md` 에 다음 질문에 직접 답해 보기:

1. **복식부기를 한 문장으로 설명한다면?** — 모든 거래는 차변과 대변 두 측면을 가지며, 차변 합과 대변 합은 항상 같다.
2. **부가세대급금(2300) 의 유형이 ASSET 인 이유는?** — 매입 시 미리 낸 부가세는 다음 신고 시 차감받을 자산.
3. **자동 분개는 누가 호출하는가?** — 발행자(InvoiceService/GoodsReceiptService/DeliveryService/PaymentService) 가 호출하지 않는다. 이벤트 리스너(`SalesAccountingListener`/`PurchaseAccountingListener`/`CogsAccountingListener`) 가 `AutoJournalService` 의 메서드를 호출한다. (Payment 만 같은 모듈이라 PaymentService 가 직접 호출.)
4. **차/대 불일치 시 어떻게 되는가?** — `UnbalancedJournalException` → 422 응답. 도메인 객체의 상태는 DRAFT 그대로 (예외 전 변경 없음).
5. **매출원가의 단가는 어디서 오는가?** — `StockMovement.unit_cost` (Phase 3 가 박아둠). FI 리스너가 `findByRefTypeAndRefIdAndReason("GI", giId, GOODS_ISSUE)` 로 조회.
6. **`@Order(10)` 과 `@Order(20)` 의 의미는?** — 같은 이벤트(`DeliveryShippedEvent`) 를 듣는 두 리스너의 실행 순서. 작은 값이 먼저. 매출원가 계산이 재고 차감 뒤에 일어나도록 보장.
7. **별도 잔액 테이블이 없는 이유는?** — 원장(`journal_line`) 이 진실. 캐시(잔액 테이블) 를 두면 동기화 비용/정합성 위험 증가. 회계는 정확성이 성능보다 우선.
8. **OTC 한 묶음이 닫혔다는 것은 어떻게 확인하는가?** — 매출채권 잔액(`SUM(debit-credit)`) = 0. 매출과 매출원가의 차이 = 손익. 부가세예수금은 별도 신고 사이클이라 남는 게 정상.

## 정리 — Phase 5 가 보장하는 것

| 사용자 작업 | 자동으로 박히는 것 |
| --- | --- |
| `POST /api/goods-receipts/{id}/post` | 차)재고자산 / 대)매입채무 |
| `POST /api/invoices` | 차)매출채권 / 대)매출+부가세예수금 |
| `POST /api/deliveries` (출하 확정) | 재고 차감 + StockMovement(-) + **차)매출원가 / 대)재고자산** |
| `POST /api/payments` (RECEIPT) | 차)현금 / 대)매출채권 |
| `POST /api/payments` (DISBURSEMENT) | 차)매입채무 / 대)현금 |

이 다섯 자동 분개가 **한 트랜잭션 안에서, 차/대 균형 검증을 통과한 후에** 박힌다. Phase 4 의 "출하 = 한 트랜잭션" 원자성이 회계로 확장된 모양.
