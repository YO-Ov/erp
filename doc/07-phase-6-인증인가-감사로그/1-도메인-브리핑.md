# Phase 6 도메인 브리핑 — 인증/인가 + 감사 로그 / "누가 무엇을 했는가" ⭐

> 7단계 사이클의 **단계 1: 도메인 브리핑**.
> 이 단계의 목적은 코드를 짜기 전에 **"왜 기간계에서 인증·인가·감사가 선택이 아니라 필수인가"** 와 그 골격(JWT / 역할 기반 권한 / 감사 로그 두 층위)을 글로 이해하는 것.
> 보안 지식이 없어도 따라올 수 있게 §2~§3 을 친절하게 풀었다. 모호한 부분은 **단계 2 Q&A** 에서 질문.

## 0. 한 줄 요약

> 지금까지(Phase 1~5)는 **"무엇을 할 수 있는가"** 를 만들었다. Phase 6 은 **"누가 그것을 해도 되는가(인증·인가)"** 와 **"누가 그것을 실제로 했는가(감사)"** 를 채운다.
> Phase 1 에서 `AuditorAware` 가 항상 `"system"` 을 반환하도록 심어 둔 복선, `Employee.email` 에 "Phase 6 부터 로그인 사용자가 된다" 고 적어 둔 복선이 **여기서 회수**된다.

## 1. 왜 기간계에 인증/인가/감사가 "필수" 인가

학습 계획서가 ERP/기간계의 5대 특성으로 꼽은 것 중 하나가 **감사(audit)** 였고, 다른 하나가 **한정된 사용자**였다. 이 둘이 Phase 6 의 주제다.

기간계 시스템은 **회사의 돈과 책임이 걸린 데이터**를 다룬다. 그래서 세 가지 질문에 항상 답할 수 있어야 한다:

| 질문 | 답하는 기능 | 영어 |
| --- | --- | --- |
| "너 **누구**야?" | 로그인(신원 증명) | **인증** (Authentication) |
| "너 이거 **해도 돼**?" | 권한 검사 | **인가** (Authorization) |
| "이 데이터 **누가 언제** 바꿨어?" | 변경 이력 | **감사** (Audit) |

왜 ERP 에서 특히 중요한가:

- **분식·횡령 방지** — 회계 전표(Phase 5)를 아무나 확정/수정하면 회사가 무너진다. "재무팀만 전표 확정 가능" 같은 권한이 곧 **내부통제(internal control)**.
- **법적 요구** — 세무·감사 대응 시 "이 매출 전표를 2026-03-15 14:22 에 누가 만들었는가" 를 제출해야 한다. 추적 불가능한 시스템은 감사를 통과 못 한다.
- **책임 추적(accountability)** — 잘못된 데이터가 발견됐을 때 "누가 왜 그랬는지" 를 알아야 고치고 재발을 막는다.

> 💡 지금까지 우리 API 는 **인증 없이 누구나 호출 가능**했다(Swagger UI 에서 바로). 이건 학습 편의를 위한 임시 상태였고, 실무 기간계에서는 **있을 수 없는 일**. Phase 6 이 그 구멍을 막는다.

## 2. 인증 vs 인가 — 가장 먼저 분리해야 할 두 개념

초보자가 가장 많이 헷갈리는 지점. **둘은 완전히 다른 단계**다.

```
요청이 들어옴
   │
   ├─ ① 인증(Authentication): "이 요청을 보낸 게 누구인가?"
   │     → 토큰/세션을 까서 "아, 영업팀 김철수다" 확정. 실패하면 401 Unauthorized.
   │
   └─ ② 인가(Authorization): "그 김철수가 이 동작을 해도 되는가?"
         → "김철수는 영업팀(SD)이라 회계 전표 확정은 안 됨" → 거부. 실패하면 403 Forbidden.
```

- **인증 실패 = 401 Unauthorized** — "너가 누군지 모르겠다"(로그인 안 했거나 토큰 만료).
- **인가 실패 = 403 Forbidden** — "너가 누군진 알겠는데, 이건 하면 안 돼"(권한 없음).

> 💡 이 401 vs 403 구분은 Phase 2 에서 배운 **HTTP 상태 코드의 의미 차이**(400/404/409)의 연장이다. 클라이언트가 "다시 로그인하라(401)" vs "권한 신청하라(403)" 를 다르게 안내할 수 있게 하는 약속.

## 3. 인증 — JWT 무상태 토큰 방식

### 3-1. 세션 방식 vs 토큰(JWT) 방식

로그인 상태를 서버가 어떻게 기억하느냐의 두 갈래:

| 방식 | 서버가 기억하는 것 | 비유 |
| --- | --- | --- |
| **세션(Session)** | 서버 메모리/DB 에 "세션 ID → 로그인 정보" 보관, 클라이언트는 세션 ID 쿠키만 | 클럽 입구에서 손목밴드 받고, 명단은 클럽이 보관 |
| **JWT(토큰)** | **아무것도 기억 안 함**(무상태). 토큰 자체에 정보가 들어 있고 **서명**으로 위조 방지 | 위조 불가 도장이 찍힌 신분증을 본인이 들고 다님 |

우리는 **JWT** 를 쓴다(학습 계획서 지정). 이유: 무상태 → 서버 확장이 쉽고, 나중에 **Phase 11~ 의 ERP↔MES 두 시스템**이 같은 토큰을 공유하기 좋다.

### 3-2. JWT 의 생김새 — 3토막

JWT 는 점(`.`)으로 나뉜 세 부분이다:

```
eyJhbGc...   .   eyJzdWI...   .   SflKxwRJ...
  헤더               페이로드(내용)        서명
  (알고리즘)      (누구/언제만료/역할)   (위조 검증용)
```

- **페이로드(Payload)**: "이 토큰의 주인은 `kim@hwlee.com`, 역할은 `SALES`, 만료는 2026-05-29 15:00" 같은 **클레임(claim)** 이 담긴다. ⚠️ 페이로드는 암호화가 아니라 **인코딩**일 뿐 — 누구나 까볼 수 있다. **비밀번호 같은 민감정보는 절대 안 넣는다.**
- **서명(Signature)**: 서버만 아는 **비밀키(secret)** 로 헤더+페이로드를 서명. 누군가 페이로드를 "역할: ADMIN" 으로 위조하면 서명이 깨져서 **서버가 즉시 거부**. 이게 JWT 의 핵심 안전장치.

### 3-3. 로그인 ~ 보호된 API 호출 흐름

```
① POST /api/auth/login {email, password}
      서버: 이메일·비밀번호 확인 → 맞으면 JWT 발급
      ← { "accessToken": "eyJ..." }

② GET /api/sd/sales-orders
      Authorization: Bearer eyJ...        ← 매 요청에 토큰 첨부
      서버: 토큰 서명 검증 → 페이로드에서 "kim, SALES" 추출
           → SecurityContext 에 "현재 사용자 = kim" 세팅 (인증 완료)
           → kim 이 SD 권한 있나 검사 (인가)
           → OK 면 처리
```

### 3-4. 비밀번호는 어떻게 저장하나 — 절대 평문 금지

`password` 컬럼에 비밀번호를 **그대로 저장하면 안 된다**. DB 가 유출되면 전원 계정이 털린다. 대신 **단방향 해시**(BCrypt)로 저장한다:

```
입력 "1234"  ──BCrypt──▶  "$2a$10$N9qo8uLOick...(60자)"   ← 이걸 DB 에 저장
```

- 해시는 **역산 불가** — 저장된 해시로는 원래 비밀번호를 못 구한다.
- 로그인 시엔 입력값을 같은 방식으로 해시해서 **저장된 해시와 비교**.
- BCrypt 는 **솔트(salt)** 가 내장돼 같은 "1234" 라도 매번 다른 해시가 나온다 → 무지개표(rainbow table) 공격 방어.

> 💡 Spring Security 가 `PasswordEncoder` 로 이걸 표준 제공한다. 우리가 해시 알고리즘을 직접 구현하지 않는다.

## 4. 인가 — 역할 기반 접근 제어(RBAC)

### 4-1. 사용자 → 역할 → 권한

권한을 사용자에게 직접 붙이면 관리가 지옥이 된다(직원 100명 × 권한 50개). 그래서 **역할(Role)** 이라는 중간 묶음을 둔다 — **RBAC(Role-Based Access Control)**:

```
사용자(User)  ──N:M──▶  역할(Role)  ──N:M──▶  권한(Permission)
  김철수                  SALES                  SD_READ, SD_WRITE
  박영희                  FINANCE                FI_READ, FI_WRITE, FI_POST
  관리자                  ADMIN                  (전부)
```

학습 계획서의 예시 — **"영업팀은 SD만, 재무팀은 FI만"** — 이 그대로 역할이 된다:

| 역할(Role) | 접근 가능 모듈 | 현우전자 부서 매핑 |
| --- | --- | --- |
| `SALES` | SD(영업), 마스터 조회 | 영업팀 |
| `PURCHASING` | MM(자재/구매), 마스터 조회 | 구매팀 |
| `FINANCE` | FI(회계) — 전표 확정 권한 포함 | 재무팀 |
| `ADMIN` | 전 모듈 + 사용자 관리 | 시스템 관리자 |

### 4-2. 우리 프로젝트의 단순화 — 역할 단위로 시작

진짜 ERP 는 권한을 메뉴/버튼 단위까지 쪼갠다. 우리는 학습이 목적이므로 **역할(Role) 단위 인가**부터 시작한다:

- `@PreAuthorize("hasRole('FINANCE')")` 한 줄로 **회계 컨트롤러 전체를 재무팀에게만** 연다.
- 더 세밀하게 가면 `hasAuthority('FI_POST')` 처럼 권한(Permission) 단위로 — 설계 단계(단계 3)에서 어디까지 쪼갤지 결정.

> 💡 이건 Phase 5 의 "규칙을 어디에 둘까(메서드 vs 매핑 테이블 vs 전략)" 와 같은 종류의 설계 결정 — **단순하게 시작하고, 필요하면 쪼갠다.**

## 5. 로그인 주체는 누구인가 — Employee 마스터와의 연결 ⭐

여기서 Phase 1 의 복선이 회수된다. `Employee` 엔티티에 이렇게 적혀 있었다:

```java
/** 직원 마스터 — 부서에 속하며, Phase 6 부터는 로그인 사용자(email)이 된다. */
@Column(name = "email", nullable = false, unique = true, length = 200)
private String email;
```

즉 **로그인 주체 = 직원(Employee)**. `email` 이 unique 인 것도 이걸 위한 복선이었다. 두 가지 설계 갈림길이 있고, 단계 3 에서 정한다:

- **(A) Employee 에 `passwordHash` + 역할을 직접 붙인다** — 단순. 직원이 곧 사용자.
- **(B) 별도 `app_user` 테이블을 두고 Employee 와 1:1** — 관심사 분리(인사 정보 vs 로그인 정보). 실무에 가까움.

→ 어느 쪽이든 **부서(Department)가 역할을 결정**하는 자연스러운 연결이 생긴다(영업팀 직원 → SALES 역할). Phase 1 의 부서 트리가 여기서 또 일한다.

## 6. 감사(Audit) — 두 개의 층위를 구분하라

"감사" 라는 말이 두 가지를 동시에 가리켜서 헷갈린다. **명확히 두 층위로 나눈다:**

### 6-1. 층위 ① — 변경자 추적 (이미 있다, BaseEntity)

모든 엔티티가 상속하는 `BaseEntity` 에 이미 4개 컬럼이 있다:

```java
@CreatedBy   private String createdBy;    // 누가 만들었나
@CreatedDate private LocalDateTime createdAt;
@LastModifiedBy   private String updatedBy;   // 누가 마지막에 고쳤나
@LastModifiedDate private LocalDateTime updatedAt;
```

문제: 지금 `createdBy`/`updatedBy` 에는 항상 `"system"` 이 박힌다. 왜냐하면 `JpaAuditingConfig` 가:

```java
// Phase 1 에서는 인증 시스템이 없으므로 항상 "system" 을 반환한다.
// Phase 6 (Spring Security 도입) 에서 SecurityContext 기반으로 교체된다.   ← 복선!
@Bean
public AuditorAware<String> auditorProvider() {
    return () -> Optional.of(SYSTEM_USER);   // ← 항상 "system"
}
```

→ **Phase 6 의 첫 회수**: 이 `auditorProvider` 를 **SecurityContext 에서 현재 로그인 사용자를 꺼내도록** 바꾼다. 그러면 그날부터 모든 테이블의 `created_by`/`updated_by` 에 `"system"` 대신 **`kim@hwlee.com`** 이 자동으로 박힌다. **기존 SD/MM/FI 코드는 한 줄도 안 고친다** — `BaseEntity` 가 이미 깔아둔 인프라라서.

### 6-2. 층위 ② — 변경 이력 로그 (새로 만든다)

층위 ①은 "**현재** 누가 마지막에 고쳤나" 만 안다 — **이전 값은 덮어써져서 사라진다**. 하지만 감사는 "**이력 전체**" 를 요구한다: "이 고객 신용한도가 100만 → 500만 → 300만 으로 바뀐 과정 전부".

이걸 위해 **감사 로그(Audit Log)** 를 만든다 — 변경 한 건마다 한 행(append-only):

| 컬럼 | 예시 |
| --- | --- |
| 누가 | `kim@hwlee.com` |
| 언제 | `2026-05-29 14:22:01` |
| 무엇을 (엔티티) | `Customer #42` |
| 어떤 동작 | `UPDATE` |
| 무엇이 바뀌었나 | `creditLimit: 1000000 → 5000000` |

> 💡 이 "append-only 이력 테이블" 구조는 새 개념이 아니다 — **Phase 3 의 `StockMovement`(재고 원장), Phase 5 의 `JournalEntry`(회계 원장)와 똑같은 패턴**. ERP 에서 "진실은 이력의 누적, 현재값은 그 결과" 라는 사상이 또 반복된다. 누적성 ⭐.

### 6-3. 감사 로그 구현 방식 — 설계 단계의 핵심 결정

학습 계획서가 두 선택지를 제시했다(**"Spring Data Envers 또는 직접 구현"**):

| 방식 | 장점 | 단점 |
| --- | --- | --- |
| **Spring Data Envers** | `@Audited` 한 줄로 자동 이력 테이블 생성·기록 | 마법이 많아 "어떻게 도는지" 학습엔 불투명 |
| **직접 구현** (JPA 이벤트 리스너 / AOP) | 흐름을 눈으로 봄 — 학습 가치 ↑ | 코드 더 씀 |

→ 학습 목적상 어느 쪽이 나은지는 단계 2 Q&A 와 단계 3 설계에서 확정. (도메인 이해 중심 학습이라면 **직접 구현 쪽이 "어떻게 도는가" 를 보여줘서** 교육적일 수 있다.)

## 7. 화면 — 이 시점부터 Thymeleaf 도입 시작 (점진적)

학습 계획서: *"Phase 6 이후 Thymeleaf + HTMX + Bootstrap"*, *"이 시점부터 Thymeleaf로 간단한 메뉴 화면 시작"*.

지금까지는 **Swagger UI 가 곧 테스트 화면**이었다. Phase 6 부터는 **로그인 화면**이 필요해진다(토큰을 어디서 발급받나). 그래서:

- **최소한**: 로그인 폼 + 로그인 후 "내 역할로 접근 가능한 메뉴" 한 페이지.
- 단, **화면 만드느라 도메인 학습이 늦어지면 안 됨**(계획서 원칙). 백엔드(인증/인가/감사)가 본체, 화면은 맛보기.

→ 화면을 어디까지 만들지는 단계 3 에서 범위 확정. JWT 를 쓰는데 브라우저 화면에서 토큰을 어떻게 들고 다닐지(쿠키 vs 헤더)도 그때 결정.

## 8. 만들 것 — 한눈에 (개략, 상세는 단계 3)

| 만들 것 | 종류 | 비고 |
| --- | --- | --- |
| 사용자/비밀번호 (`Employee` 확장 or `AppUser`) | 마스터 | 로그인 주체. BCrypt 해시 |
| 역할(`Role`) / 권한 매핑 | 마스터 (N:M) | RBAC. 부서 → 역할 연결 |
| 로그인 API (`POST /api/auth/login`) | 인증 | 이메일·비밀번호 → JWT 발급 |
| JWT 발급/검증 필터 | 보안 인프라 | `SecurityFilterChain`, JWT 필터 |
| 인가 규칙 | 보안 설정 | `@PreAuthorize("hasRole('FINANCE')")` 등 모듈별 |
| `AuditorAware` 교체 ⭐ | 인프라 수정 | `"system"` → SecurityContext 사용자 (복선 회수) |
| 감사 로그(`AuditLog`) + 기록 메커니즘 | 트랜잭션 (append-only) | 변경 이력. Envers vs 직접 구현 |
| 로그인/메뉴 화면 | Thymeleaf | 최소 범위 |
| 마이그레이션 V26~ | Flyway | 사용자/역할/권한/감사로그 테이블 + 시드 |

**새 의존성도 늘어난다**: `spring-boot-starter-security`, JWT 라이브러리(jjwt 등), `spring-boot-starter-thymeleaf`. (Phase 5 까지는 보안 의존성이 아예 없었다 — build.gradle 에 처음 추가된다.)

## 9. Phase 1~5 와의 누적성

```
Phase 1: 마스터 + Auditing 4컬럼(BaseEntity) + AuditorAware "system" 자리만 잡음
   │                                  └─→ §6-1: auditorProvider 교체로 회수
   │      Employee.email unique ────────→ §5: 로그인 주체로 회수
Phase 2: HTTP 상태 코드 의미(400/404/409)
   │                                  └─→ §2: 401/403 으로 확장
Phase 3: StockMovement = append-only 원장
   │                                  └─→ §6-2: AuditLog 가 같은 패턴
Phase 4: 이벤트 리스너(한 사건에 여러 구독자)
   │                                  └─→ 감사 로그도 "변경 사건"에 반응하는 리스너로 구현 가능
Phase 5: JournalEntry 원장 + "내부통제(누가 전표를 확정하나)"
   │                                  └─→ §4: FINANCE 역할만 전표 확정 — 인가로 강제
Phase 6: ⭐ "누가 해도 되는가(인증·인가)" + "누가 했는가(감사)" 를 전 모듈에 한 번에 입힘
```

→ Phase 6 의 묘미: **기존 5개 Phase 의 코드를 거의 안 고치고** 보안·감사가 횡단(cross-cutting)으로 얹힌다. `BaseEntity`·`AuditorAware`·이벤트·HTTP 상태코드가 미리 깔아둔 레일 위에 올라타는 것 — **앞에서 심은 복선이 가장 많이 회수되는 Phase**.

## 10. 사용자가 이번 브리핑에서 잡아야 할 핵심 5가지

1. **인증(너 누구야? 401) vs 인가(해도 돼? 403) 는 완전히 다른 단계.** 인증이 먼저, 인가가 나중.
2. **JWT = 서버가 상태를 안 갖는 "서명된 신분증".** 페이로드는 누구나 까보므로 민감정보 금지, 위조는 서명으로 막는다. 비밀번호는 BCrypt 해시로만 저장.
3. **RBAC: 사용자→역할→권한.** "영업팀은 SD만, 재무팀은 FI만" 이 곧 역할. 부서가 역할을 결정.
4. **감사는 두 층위** — ① 변경자 추적(이미 BaseEntity 에 있음, `"system"` 만 진짜 사용자로 교체) + ② 변경 이력 로그(새로 만드는 append-only 테이블, StockMovement/JournalEntry 와 같은 패턴).
5. **Phase 6 은 복선 회수의 Phase** — `auditorProvider` 와 `Employee.email` 에 미리 심은 자리가 채워지고, 기존 모듈 코드는 거의 안 고친 채 보안·감사가 횡단으로 얹힌다.
