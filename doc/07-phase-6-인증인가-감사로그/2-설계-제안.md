# Phase 6 설계 제안 — 인증/인가 + 감사 로그 ⭐

> 7단계 사이클의 **단계 3: 모델링 제안**.
> 단계 2(이해 확인 Q&A)에서 확정한 네 가지 설계 결정을 토대로, ERD / 인증·JWT 흐름 / 인가 규칙 배치 / `AuditorAware` 교체 / 감사 로그 구현 / Thymeleaf 범위 / 마이그레이션(V26~) 을 설계한다.
> 이 문서를 검토하고 **단계 4(승인)** 을 주시면 단계 5(코드 구현)로 넘어간다.

## 2. ERD — 새 테이블 5개 + 시드

확정 ①(별도 AppUser) + ③(N:M 역할)에 따른 모델.

```
                 ┌───────────────┐
                 │   employee    │ (기존, Phase 1)
                 │  id, email,   │
                 │  department_id│
                 └───────┬───────┘
                         │ 1:1 (employee_id UNIQUE)
                         ▼
   ┌──────────────────────────────────┐
   │            app_user              │  ← 로그인 주체
   │  id, employee_id(UQ),            │
   │  username(UQ=email),             │
   │  password_hash(BCrypt),          │
   │  enabled, account_locked         │
   └───────────────┬──────────────────┘
                   │ N:M
                   ▼
        ┌──────────────────────┐        ┌──────────────────────┐
        │   user_role (조인)    │        │   role_permission(조인)│
        │  user_id, role_id     │        │  role_id, permission_id│
        └──────────┬───────────┘        └───────────┬──────────┘
                   │ N:M                              │ N:M
                   ▼                                  ▼
            ┌─────────────┐                    ┌──────────────┐
            │    role     │────────────────────│  permission  │
            │ id, code,   │   role_permission   │ id, code,    │
            │ name        │                     │ name         │
            └─────────────┘                     └──────────────┘

   ┌────────────────────────────────────────────────────────┐
   │                       audit_log                         │  ← 변경 이력 (append-only)
   │  id, entity_type, entity_id, action(INSERT/UPDATE/      │
   │  DELETE), changed_by, changed_at, changes(JSON)         │
   └────────────────────────────────────────────────────────┘
```

### 2-1. 테이블 정의

#### `app_user` — 로그인 계정 (BaseEntity 상속)
| 컬럼 | 타입 | 제약 | 설명 |
| --- | --- | --- | --- |
| id | BIGINT | PK, AI | |
| employee_id | BIGINT | FK→employee, **UNIQUE**, NOT NULL | 1:1. 직원이 곧 사용자 |
| username | VARCHAR(200) | **UNIQUE**, NOT NULL | 로그인 ID. = employee.email 로 시드 |
| password_hash | VARCHAR(100) | NOT NULL | BCrypt 60자 (+여유) |
| enabled | BOOLEAN | NOT NULL, default TRUE | 계정 활성/비활성 |
| account_locked | BOOLEAN | NOT NULL, default FALSE | 잠금(로그인 실패 누적 대비, 이번엔 컬럼만) |
| created_at/by, updated_at/by | | | BaseEntity 4컬럼 |

> 코드 채번(`code`)이 필요 없으므로 `BaseEntityWithCode` 가 아닌 **`BaseEntity`** 만 상속. Soft Delete 도 계정엔 부적합(enabled 로 대체).

#### `role` — 역할 (BaseEntity 상속)
| 컬럼 | 타입 | 제약 | 설명 |
| --- | --- | --- | --- |
| id | BIGINT | PK | |
| code | VARCHAR(30) | UNIQUE, NOT NULL | `SALES`, `PURCHASING`, `FINANCE`, `ADMIN` |
| name | VARCHAR(100) | NOT NULL | "영업", "재무" 등 표시명 |

> Spring Security 의 `hasRole('FINANCE')` 는 내부적으로 `ROLE_FINANCE` 권한을 찾는다. → 코드는 `FINANCE` 로 저장하고, UserDetails 구성 시 `ROLE_` 접두어를 붙인다(§4-3).

#### `permission` — 세부 권한 (BaseEntity 상속)
| 컬럼 | 타입 | 제약 | 설명 |
| --- | --- | --- | --- |
| id | BIGINT | PK | |
| code | VARCHAR(50) | UNIQUE, NOT NULL | `SD_READ`, `SD_WRITE`, `FI_POST` 등 |
| name | VARCHAR(100) | NOT NULL | 표시명 |

> 이번 Phase 의 인가는 **역할 단위**로 시작(§4). permission 테이블은 **구조만 만들고 시드**해 둔다. `@PreAuthorize` 에서 `hasAuthority('FI_POST')` 로 쪼갤 여지를 남기는 복선(Phase 7+ 또는 본 Phase 후반 선택).

#### `user_role` — 사용자↔역할 N:M (조인)
| user_id (FK→app_user) | role_id (FK→role) | PK(user_id, role_id) |

#### `role_permission` — 역할↔권한 N:M (조인)
| role_id (FK→role) | permission_id (FK→permission) | PK(role_id, permission_id) |

#### `audit_log` — 변경 이력 append-only (BaseEntity 상속 안 함, 독립)
| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| id | BIGINT PK | |
| entity_type | VARCHAR(100) | `Customer`, `SalesOrder` 등 클래스 단순명 |
| entity_id | BIGINT | 대상 PK |
| action | VARCHAR(10) | `INSERT` / `UPDATE` / `DELETE` |
| changed_by | VARCHAR(64) | 변경자 (AuditorAware 와 같은 소스) |
| changed_at | DATETIME | 변경 시각 |
| changes | JSON | `{"creditLimit": ["1000000","5000000"]}` 형태 |

> `audit_log` 는 자기 자신이 또 감사되면 안 되므로 BaseEntity 를 상속하지 않고 직접 컬럼을 갖는다(무한 루프 차단).

### 2-2. 역할/권한 시드 (확정 ③: 부서 기반)

기존 `department` 시드(V8)에 부서가 있다. 부서코드 → 역할 매핑으로 `user_role` 을 시드:

| 부서 (Phase 1 시드) | → 역할 | 권한(role_permission) |
| --- | --- | --- |
| 영업팀 | `SALES` | SD_READ, SD_WRITE, MASTER_READ |
| 구매팀 | `PURCHASING` | MM_READ, MM_WRITE, MASTER_READ |
| 재무팀 | `FINANCE` | FI_READ, FI_WRITE, FI_POST, MASTER_READ |
| (관리자 계정) | `ADMIN` | ALL |

→ V28 시드에서 **각 직원의 부서를 보고 자동으로 user_role 행을 INSERT**. 이후 관리자가 화면/API 로 수동 조정(예: 팀장에게 ADMIN 추가).

## 3. 인증(Authentication) 설계

### 3-1. 컴포넌트 구성 (`com.hwlee.erp.security` 신설)

```
security/
├─ config/
│   └─ SecurityConfig.java          ← SecurityFilterChain, PasswordEncoder, AuthenticationManager
├─ jwt/
│   ├─ JwtTokenProvider.java        ← 토큰 생성/파싱/검증 (jjwt)
│   ├─ JwtAuthenticationFilter.java ← OncePerRequestFilter: 헤더 토큰 → SecurityContext
│   └─ JwtProperties.java           ← secret, 만료시간 (application.yml 바인딩)
├─ auth/
│   ├─ AuthController.java          ← POST /api/auth/login
│   ├─ LoginRequest / LoginResponse ← DTO
│   └─ AuthService.java             ← 인증 + 토큰 발급
├─ user/
│   ├─ AppUser.java / AppUserRepository
│   ├─ Role.java / Permission.java / *Repository
│   └─ ErpUserDetailsService.java   ← UserDetailsService 구현 (username→권한 로딩)
└─ support/
    └─ SecurityAuditorAware.java    ← §5 층위① 교체본 (AuditorAware<String>)
```

### 3-2. 로그인 흐름

```
POST /api/auth/login  { "username": "kim@hwlee.com", "password": "1234" }
   │
   ├─ AuthService: AppUserRepository.findByUsername
   ├─ PasswordEncoder.matches(raw, password_hash)  ── 불일치 → 401
   ├─ enabled=false / account_locked=true          ── → 401(or 403)
   └─ JwtTokenProvider.createToken(username, roles)
         payload: { sub=username, roles=[SALES,...], iat, exp }
   ← 200  { "accessToken": "eyJ...", "tokenType": "Bearer", "expiresIn": 3600 }
```

### 3-3. 보호된 요청 흐름

```
GET /api/journal-entries
Authorization: Bearer eyJ...
   │
   ├─ JwtAuthenticationFilter
   │     ├─ 헤더에서 Bearer 토큰 추출 (없으면 그냥 통과 → 뒤에서 401)
   │     ├─ JwtTokenProvider.validate(token)   ── 서명/만료 실패 → SecurityContext 비움
   │     └─ UsernamePasswordAuthenticationToken(username, null, authorities) 를
   │          SecurityContextHolder 에 set
   ├─ @PreAuthorize("hasRole('FINANCE')") 평가
   │     ├─ 미인증 → 401 (AuthenticationEntryPoint)
   │     └─ 인증됐으나 권한 없음 → 403 (AccessDeniedHandler)
   └─ 통과 → 컨트롤러 실행
```

### 3-4. 비밀번호 저장

- `PasswordEncoder` = `BCryptPasswordEncoder` (Spring Security 표준).
- 시드 비밀번호: 학습 편의상 전 계정 동일(예: `pass1234`)을 **BCrypt 해시로 미리 인코딩한 값**을 V28 에 박는다. (평문 저장 절대 금지 원칙 준수)

### 3-5. 토큰 만료 / 리프레시 — 이번 범위

- **Access Token 1종만**(만료 1시간). **Refresh Token 은 이번 Phase 범위 밖**(만료되면 재로그인).
- 이유: 리프레시 토큰은 저장/회전/폐기 등 별도 설계가 커서, 인증 기본기 학습을 흐린다. Phase 후반 또는 추후 확장으로 메모만 남긴다.

## 4. 인가(Authorization) 설계 — RBAC

### 4-1. ⚠️ 중요 정정 — 실제 URL 은 모듈 prefix 가 없다

브리핑 §3-3 은 `/api/sd/sales-orders` 를 가정했지만, **실제 컨트롤러 경로는 모듈 prefix 가 없다**:

| 모듈 | 실제 경로 |
| --- | --- |
| SD | `/api/sales-orders`, `/api/quotations`, `/api/deliveries`, `/api/invoices` |
| MM | `/api/goods-issues`, `/api/goods-receipts`, `/api/warehouses`, `/api/stocks`, `/api/stock-movements` |
| FI | `/api/journal-entries`, `/api/payments`, `/api/accounts` |
| master | `/api/customers`, `/api/items`, `/api/vendors`, `/api/departments`, `/api/employees` |

→ URL 패턴(`/api/sd/**`)으로 묶는 방식은 **부적합**(경로에 모듈명이 없음).
→ **메서드 보안(`@EnableMethodSecurity` + `@PreAuthorize`)** 을 컨트롤러 **클래스 단위**로 부착하는 방식을 채택. 모듈 소속이 코드 위치로 자명하므로 클래스에 한 줄 거는 게 가장 명확하다.

### 4-2. 인가 규칙 배치 (클래스 단위 `@PreAuthorize`)

```java
// SD 컨트롤러 4개 (클래스 레벨)
@PreAuthorize("hasAnyRole('SALES','ADMIN')")

// MM 컨트롤러 5개
@PreAuthorize("hasAnyRole('PURCHASING','ADMIN')")

// FI 컨트롤러 3개
@PreAuthorize("hasAnyRole('FINANCE','ADMIN')")

// master 컨트롤러 5개 — 조회는 전원, 변경은 ADMIN
@PreAuthorize("hasAnyRole('SALES','PURCHASING','FINANCE','ADMIN')")  // GET
@PreAuthorize("hasRole('ADMIN')")                                    // POST/PUT/DELETE
```

- 메서드별로 읽기/쓰기를 다르게 걸 곳(master)은 **메서드 레벨**로 내려 건다.
- 전표 확정처럼 더 센 동작은 `hasRole('FINANCE')` 단독(ADMIN 도 허용할지는 구현 시 확정).
- `/api/auth/login`, `/swagger-ui/**`, `/api-docs/**`, 로그인 화면, 정적 리소스는 **permitAll**.

### 4-3. `hasRole` 과 `ROLE_` 접두어

- DB `role.code` 는 `FINANCE` 로 저장.
- `ErpUserDetailsService` 가 권한 부여 시 `new SimpleGrantedAuthority("ROLE_" + code)` 로 변환.
- `hasRole('FINANCE')` 는 자동으로 `ROLE_FINANCE` 를 찾으므로 일치. (permission 은 접두어 없이 `hasAuthority('FI_POST')`)

## 5. 감사(Audit) 설계 — 두 층위

### 5-1. 층위① — `AuditorAware` 교체 (복선 회수)

기존 `JpaAuditingConfig.auditorProvider()` 의 `"system"` 고정을 **SecurityContext 기반**으로 교체:

```java
// security/support/SecurityAuditorAware.java
public class SecurityAuditorAware implements AuditorAware<String> {
    @Override
    public Optional<String> getCurrentAuditor() {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth == null || !auth.isAuthenticated() || auth instanceof AnonymousAuthenticationToken) {
            return Optional.of("system");   // 배치/시드/미인증 컨텍스트 → system 유지
        }
        return Optional.of(auth.getName()); // username = email
    }
}
```

- `JpaAuditingConfig` 는 이 빈을 가리키도록 `auditorAwareRef` 만 바꾼다(또는 빈 교체).
- **효과**: 그날부터 모든 테이블 `created_by`/`updated_by` 에 `kim@hwlee.com` 자동 기록. **기존 SD/MM/FI/master 엔티티·서비스 코드는 0줄 수정**.
- 배치·시드·로그인 전 컨텍스트는 `system` 으로 안전하게 fallback.

### 5-2. 층위② — `audit_log` 직접 구현 (확정 ②)

**방식: JPA `@EntityListeners` + 공통 리스너**.

```java
// audit/AuditLogListener.java
public class AuditLogListener {
    @PostPersist void onInsert(Object e) { record(e, "INSERT"); }
    @PostUpdate  void onUpdate(Object e) { record(e, "UPDATE"); }
    @PostRemove  void onDelete(Object e) { record(e, "DELETE"); }
}
```

- 감사 대상 엔티티에 `@EntityListeners({AuditingEntityListener.class, AuditLogListener.class})` 를 부착.
  - **선택 A (권장)**: `BaseEntity` 에 리스너를 추가 → 전 엔티티 자동 감사. 단 audit 양이 많아짐.
  - **선택 B**: 감사가 꼭 필요한 핵심 마스터/전표(Customer, SalesOrder, JournalEntry 등)에만 부착 → 학습 시연에 적당.
  - → **구현 시 B 로 시작**(시연에서 "어떤 엔티티가 왜 감사 대상인가"를 의식적으로 고른다). 데모: `Customer.creditLimit` 변경 이력.
- **변경 필드 추출(`changes` JSON)**: Hibernate 의 변경 추적을 직접 파싱하는 건 복잡하므로, 학습용으로는 **`@PreUpdate` 시점에 더티 필드를 비교**하거나, 1차로 **엔티티 스냅샷(JSON 직렬화) 전체를 저장**하는 단순안 중 택1.
  - → **단순안**: action 별로 핵심 필드 위주 직렬화. (완벽한 diff 는 Envers 의 강점이고, 우리는 "흐름을 눈으로 본다"가 목적이므로 적정선)
- **트랜잭션 경계**: 리스너가 같은 트랜잭션에서 `audit_log` 에 INSERT(원자성). 단 `@PostPersist` 안에서 EntityManager 재진입 이슈가 있어, **이벤트 발행(Phase 4 패턴) → `@TransactionalEventListener(BEFORE_COMMIT)` 에서 기록**하는 방식도 후보. 구현 단계에서 둘 중 안정적인 쪽 확정.

> 💡 이 audit_log 는 Phase 3 `StockMovement`, Phase 5 `JournalEntry` 와 같은 **append-only 원장 패턴**. "진실은 이력의 누적" 사상의 3번째 등장.

### 5-3. 감사 로그 조회 API

- 성공 기준(계획서): "누가 언제 어떤 데이터를 바꿨는지 조회할 수 있는 화면/API 가 있다" 충족.

## 6. 예외 처리 확장 (401/403)

`GlobalExceptionHandler` 에 ProblemDetail 핸들러 추가:

| 예외 | 상태 | 의미 |
| --- | --- | --- |
| `AuthenticationException` (EntryPoint) | **401** | 미인증/토큰 무효 |
| `AccessDeniedException` (Handler) | **403** | 인증됐으나 권한 없음 |

- Security 필터 단계 예외는 `@RestControllerAdvice` 가 못 잡으므로, `AuthenticationEntryPoint` / `AccessDeniedHandler` 를 구현해 **동일한 ProblemDetail JSON 포맷**으로 응답(기존 에러 포맷 일관성 유지).

| 화면 | 경로 | 내용 |

| 로그인 | `GET /login` | username/password 폼. 제출 → 토큰 발급 후 쿠키 보관 |
| 메뉴(대시보드) | `GET /` | 내 역할로 접근 가능한 메뉴만 노출 |
| 사용자 관리 | `GET /admin/users` | 목록 + 역할 부여/회수 (ADMIN) |
| 역할 관리 | `GET /admin/roles` | 역할-권한 매핑 조회 (ADMIN) |

### 7-1. 브라우저에서 JWT 를 어떻게 들고 다니나

- **결정: HttpOnly 쿠키에 Access Token 저장**.
  - 이유: Thymeleaf 서버 렌더링 화면에서 매 요청 `Authorization` 헤더를 수동으로 붙이기 번거롭다. HttpOnly 쿠키면 브라우저가 자동 전송 + JS 접근 차단(XSS 완화).
  - `JwtAuthenticationFilter` 가 **헤더(Bearer) 와 쿠키 둘 다**에서 토큰을 찾도록 구현 → API 클라이언트(Swagger/테스트)는 헤더, 브라우저는 쿠키.
  - CSRF: 쿠키 방식이라 본래 CSRF 토큰이 필요하나, 학습 범위에서는 SameSite=Strict 쿠키로 1차 방어(메모로 한계 명시).
- Swagger UI 는 계속 병행(헤더 `Authorize` 버튼으로 Bearer 입력).

## 10. 테스트 전략 — 기존 통합 테스트가 깨지는 문제

브리핑 §11 의 질문("인증 걸리면 Testcontainers 통합 테스트가 다 깨지지 않나?")에 대한 답:

1. **`@WithMockUser` / `@WithUserDetails`** — Spring Security Test 로 테스트에 가짜 인증 주입. 기존 통합 테스트에 역할 부여한 mock 사용자를 얹는다.
2. **테스트 프로파일에서 일부 완화** 옵션도 있으나, **인가까지 테스트로 검증**하는 게 학습 가치가 크므로 `@WithMockUser(roles=...)` 채택.
3. 새 테스트:
   - `AuthControllerTest` — 로그인 성공/실패(401), 토큰 발급 검증.
   - `AuthorizationTest` — SALES 가 `/api/journal-entries` 호출 시 403, FINANCE 는 200.
   - `AuditLogTest` — Customer 수정 후 audit_log 1행 + changed_by = mock 사용자.
   - `AuditorAwareTest` — 인증 컨텍스트에서 created_by 가 username 으로 박히는지.
