# Phase 6 코드 워크스루 ② — 사용자·역할·권한 모델과 UserDetailsService

> 대상 파일
> - `security/user/AppUser.java`
> - `security/user/Role.java`
> - `security/user/Permission.java`
> - `security/user/ErpUserDetailsService.java`
>
> 이 글의 목표: **RBAC(역할 기반 접근 제어)를 엔티티로 표현**하고, Spring Security가 로그인 시 그 모델을 어떻게 읽어 들이는지 이해한다. 브리핑 §4(RBAC)·§5(로그인 주체)가 코드가 되는 지점.

---

## 0. 한눈에

```
Employee (Phase 1 인사 마스터)
   │ 1:1 (employee_id UNIQUE)
   ▼
AppUser ──N:M──▶ Role ──N:M──▶ Permission
 (로그인 계정)   (SALES 등)    (FI_POST 등)
   username
   passwordHash(BCrypt)

ErpUserDetailsService: username → AppUser 로딩 → 권한 목록(ROLE_*, *) 구성
```

핵심 한 줄: **사용자에 권한을 직접 붙이지 않고 역할을 거친다.** "직원(인사)"과 "계정(로그인)"을 분리해(확정 ①·B), 관심사를 나눈다.

---

## 1. `AppUser` — 로그인 계정 (Employee와 1:1)

```java
@Entity
@Table(name = "app_user")
public class AppUser extends BaseEntity {       // ← BaseEntity(코드 없음), not BaseEntityWithCode

    @OneToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "employee_id", nullable = false)
    private Employee employee;                   // 1:1 — 직원이 곧 사용자

    @Column(nullable = false, unique = true, length = 200)
    private String username;                     // = employee.email

    @Column(name = "password_hash", nullable = false, length = 100)
    private String passwordHash;                 // BCrypt 해시만 (평문 금지)

    @Column(nullable = false) private boolean enabled = true;
    @Column(name = "account_locked", nullable = false) private boolean accountLocked = false;

    @ManyToMany(fetch = FetchType.LAZY)
    @JoinTable(name = "user_role",
            joinColumns = @JoinColumn(name = "user_id"),
            inverseJoinColumns = @JoinColumn(name = "role_id"))
    private Set<Role> roles = new HashSet<>();
```

짚을 점:

- **`extends BaseEntity` (≠ `BaseEntityWithCode`)**: 고객·상품처럼 `CUST-2026-0001` 같은 채번 코드가 필요 없다. 또 Soft Delete(`deletedAt`)도 안 쓴다 — 계정 비활성화는 `enabled = false` 로 표현하는 게 더 자연스럽다(삭제가 아니라 "잠금"). 그래서 BaseEntity의 Auditing 4컬럼(created_by 등)만 물려받는다.
- **`employee` 1:1**: 브리핑 §5의 복선 회수. `Employee.email` 이 unique였던 게 여기서 `username` 의 출처가 된다.
- **`passwordHash`**: 평문이 아니라 BCrypt 해시 60자(+여유)만 저장. 해싱은 `PasswordEncoder`가 한다(워크스루 ③).
- **`enabled` / `accountLocked`**: Spring Security의 `UserDetails`가 요구하는 표준 계정 상태. `accountLocked` 는 이번 Phase에선 컬럼만 두고 로그인 실패 누적 잠금 로직은 넣지 않았다(범위 절충).
- **`roles` N:M**: `user_role` 조인 테이블로 매핑. 한 사람이 여러 역할(예: 팀장 = FINANCE + ADMIN)을 가질 수 있다(확정 ③).

행위 메서드도 엔티티 안에:

```java
public void grantRole(Role role)  { this.roles.add(role); }
public void revokeRole(Role role) { this.roles.removeIf(r -> ...equals(role.getId())); }
public void changeEnabled(boolean enabled) { this.enabled = enabled; }
```

→ "역할 부여/회수"를 서비스가 컬렉션을 직접 만지는 대신 엔티티 메서드로 표현(Phase 1~5에서 지킨 도메인 응집 원칙 동일).

## 2. `Role` / `Permission` — 역할과 세부 권한

```java
@Entity @Table(name = "role")
public class Role extends BaseEntity {
    @Column(nullable = false, unique = true, length = 30)
    private String code;          // "SALES" — ROLE_ 접두어 없이 저장
    @Column(nullable = false, length = 100)
    private String name;          // "영업"

    @ManyToMany(fetch = FetchType.LAZY)
    @JoinTable(name = "role_permission", ...)
    private Set<Permission> permissions = new HashSet<>();
}
```

```java
@Entity @Table(name = "permission")
public class Permission extends BaseEntity {
    @Column(nullable = false, unique = true, length = 50) private String code;  // "FI_POST"
    @Column(nullable = false, length = 100) private String name;
}
```

- **`code` 에 `ROLE_` 접두어를 안 붙인다.** Spring Security의 `hasRole('FINANCE')` 는 내부에서 자동으로 `ROLE_FINANCE` 권한을 찾는다. 그래서 DB엔 깔끔하게 `FINANCE` 로 저장하고, 권한 객체로 변환할 때만 접두어를 붙인다(§3). 접두어를 DB에 같이 저장하면 `hasRole('ROLE_FINANCE')` 같은 이중 접두어 버그가 난다.
- **`Permission` 은 이번 Phase에선 "구조 + 시드만"**: 인가는 역할 단위로 시작한다(브리핑 §4-2). permission은 `role_permission` 으로 역할에 매달아 두지만, `@PreAuthorize("hasAuthority('FI_POST')")` 처럼 권한 단위로 쪼개는 건 추후 확장의 복선. 지금은 "쪼갤 자리"를 만들어 둔 것.

## 3. `ErpUserDetailsService` — 로그인 시 모델을 권한으로 변환

Spring Security가 로그인할 때 호출하는 **유일한 연결 고리**다.

```java
@Service
public class ErpUserDetailsService implements UserDetailsService {

    private final AppUserRepository appUserRepository;

    @Override
    @Transactional(readOnly = true)
    public UserDetails loadUserByUsername(String username) {
        AppUser user = appUserRepository.findByUsername(username)
                .orElseThrow(() -> new UsernameNotFoundException("사용자를 찾을 수 없습니다: " + username));

        Set<GrantedAuthority> authorities = new HashSet<>();
        for (Role role : user.getRoles()) {
            authorities.add(new SimpleGrantedAuthority("ROLE_" + role.getCode()));   // 역할 → ROLE_SALES
            for (Permission permission : role.getPermissions()) {
                authorities.add(new SimpleGrantedAuthority(permission.getCode()));   // 권한 → FI_POST (접두어 X)
            }
        }

        return User.withUsername(user.getUsername())
                .password(user.getPasswordHash())     // 저장된 해시 — 비교는 Security가 함
                .disabled(!user.isEnabled())
                .accountLocked(user.isAccountLocked())
                .authorities(authorities)
                .build();
    }
}
```

한 줄씩:

- **`UserDetailsService` 구현 = Spring Security와 우리 도메인의 경계**. Security는 "username을 주면 UserDetails를 다오"만 안다. 우리 `AppUser`/`Role` 구조는 전혀 모른다 — 이 클래스가 통역한다.
- **`@Transactional(readOnly = true)`**: `roles`·`permissions` 가 LAZY라서, 트랜잭션 안에서 컬렉션을 순회해야 초기화된다. 안 그러면 `LazyInitializationException`.
- **역할은 `ROLE_` 접두어 + 권한은 접두어 없이** → `hasRole('SALES')` 와 `hasAuthority('FI_POST')` 가 각각 맞아떨어진다.
- **`.password(passwordHash)`**: 여기서 비밀번호를 "비교"하지 않는다. 저장된 해시를 UserDetails에 실어주기만 하면, `AuthenticationManager` 가 `PasswordEncoder.matches(입력, 해시)` 로 대조한다(워크스루 ③).
- **`disabled`/`accountLocked`**: `enabled=false` 계정은 Security가 로그인 단계에서 자동으로 막아준다(우리가 if문 안 써도 됨).

### 3-1. 로그인 필터와 무엇이 다른가?

| | `ErpUserDetailsService` | `JwtAuthenticationFilter` (워크스루 ①) |
| --- | --- | --- |
| 언제 | **로그인 시 1회** (`/api/auth/login`) | **매 요청** |
| DB 조회 | 함 (username → AppUser + 역할) | **안 함** (토큰 payload의 역할 사용) |
| 목적 | 비밀번호 검증 + 토큰에 넣을 역할 수집 | 토큰의 역할로 컨텍스트 복원 |

→ DB는 로그인 때 딱 한 번만 본다. 그 결과를 토큰에 구워두면 이후 요청은 무상태로 흐른다.

---

## 4. 한 줄 요약

> 로그인 주체는 `Employee`와 1:1인 `AppUser`이고, 권한은 `AppUser → Role → Permission` N:M으로 표현한다(RBAC). `ErpUserDetailsService`가 Spring Security와 이 도메인 모델 사이의 통역사로서, 로그인 시 username을 받아 역할을 `ROLE_*` 권한으로 변환해 넘긴다.
