# Phase 7 도메인 브리핑 — HR 간이 모듈: 사람을 회계에 연결한다

> 7단계 사이클의 **단계 1: 도메인 브리핑**.
> 이 단계의 목적은 코드를 짜기 전에 **"인사(HR)가 ERP에서 무슨 일을 하고, 왜 결국 회계(FI)로 흘러가는가"** 를 글로 이해하는 것.
> 모호한 부분은 **단계 2 Q&A** 에서 질문.

---

## 0. 한 줄 요약

> 지금까지의 모듈(SD/MM/FI)은 **물건과 돈**의 흐름이었다. Phase 7은 **사람**을 다룬다 — 직원 정보를 넓히고(직급·급여등급), 근태를 기록하고, 그것으로 **급여를 계산**한다. 그리고 그 급여는 회사 입장에서 **인건비(비용)** 이므로, Phase 5에서 만든 자동 분개를 타고 **회계 전표가 된다**.
> 핵심 한 줄: **"급여 확정 = 인건비 전표"** — Phase 1의 `Employee` 마스터가 Phase 5의 `JournalEntry` 까지 이어진다.

---

## 1. HR이 ERP에서 하는 일 — 왜 회계와 한 몸인가

회사 입장에서 **인건비는 보통 가장 큰 비용 항목** 중 하나다. 그래서 HR은 "사람을 관리하는 부서 기능"이면서 동시에 **매달 거대한 회계 거래(급여 지급)를 일으키는 모듈**이다.

ERP에서 HR이 중요한 이유:

- **인건비는 손익에 직결** — 급여를 회계에 안 넣으면 손익계산서가 거짓말을 한다. Phase 5에서 "매출 − 매출원가 = 이익"을 봤지만, 진짜 이익은 거기서 인건비·기타비용을 더 빼야 나온다.
- **근태가 급여의 근거** — "이 사람이 며칠 일했나, 몇 시간 초과근무했나"가 급여 금액을 만든다. 출퇴근 기록이 곧 돈이 된다.
- **공제(deduction)와 예수금** — 급여에서 소득세·4대보험을 회사가 떼어(원천징수) 나중에 국가/기관에 대신 낸다. 직원이 받는 실수령액과 회사가 부담하는 총액이 다르다. 이게 회계에서 **예수금(부채)** 으로 표현된다.

> 💡 핵심 통찰: HR은 독립된 섬이 아니다. **"근태 → 급여 → 인건비 전표 → 손익"** 으로 흐르며, Phase 5의 자동 분개 패턴이 여기서 그대로 재사용된다. 출하가 매출 전표가 됐듯, 급여 확정이 인건비 전표가 된다.

## 2. 직원 정보 확장 — 이미 있는 것 위에 쌓기

Phase 1에서 만든 `Employee` 마스터는 이미 이만큼 가지고 있다:

```
Employee (Phase 1)
├─ code          EMP-2026-0001  (채번)
├─ name          김영업
├─ email         kim@hwlee-erp.example  (Phase 6에서 로그인 ID로 회수됨)
├─ department    부서 FK
├─ hireDate      입사일  ← 이미 있다!
└─ status        ACTIVE/INACTIVE/BLOCKED
```

Phase 7에서 급여 계산에 필요한 것을 **더한다**:

| 추가 후보 | 설명 | 왜 필요한가 |
| --- | --- | --- |
| 직급(grade/position) | 사원/대리/과장... | 급여 등급·수당의 기준 |
| 급여등급 또는 기본급(baseSalary) | 월 기본급 | 급여 계산의 출발점 |
| (선택) 시급/연봉 구분 | 월급제 vs 시급제 | 계산 방식 분기 |

> 학습 단순화 방향: "직급 + 월 기본급"만 우선. 호봉 테이블·수당 체계 같은 복잡한 급여 규정은 범위 밖. 어디까지 둘지는 단계 2~3에서 확정.

설계 갈림길(단계 3에서 결정): **Employee에 급여 필드를 직접 붙일까(A)**, 아니면 **별도 급여 마스터/계약 테이블로 분리할까(B)**. Phase 6의 `AppUser` 분리와 같은 종류의 결정.

## 3. 근태(Attendance) — 급여의 근거

"이 사람이 이번 달에 얼마나 일했나"를 기록한다.

가장 단순한 모델:

| 컬럼 | 예시 |
| --- | --- |
| 직원 | 김영업 |
| 근무일 | 2026-05-30 |
| 출근/퇴근 | 09:00 / 18:30 |
| (파생) 근무시간 / 초과근무 | 8h / 0.5h |

근태가 급여로 연결되는 방식의 두 갈래(단계 3 결정):

- **(간이) 근무일수 기반** — "이번 달 22일 출근 → 기본급 만근"처럼 단순 집계. 결근하면 일할 계산.
- **(상세) 시간 기반** — 초과근무 × 1.5배 같은 수당까지. 더 현실적이지만 복잡.

> 💡 Phase 3의 `StockMovement`(재고 원장), Phase 5의 `JournalEntry`(회계 원장)처럼, 근태도 **"하루 한 건씩 쌓이는 기록"** 이다. 누적된 기록을 월 단위로 집계해 급여를 만든다 — 익숙한 "이력 누적 → 집계" 패턴.

## 4. 급여 계산(Payroll) — 헤더-라인으로 표현되는 한 달치 급여

급여는 Phase 2의 수주, Phase 5의 전표와 같은 **헤더-라인 구조**가 자연스럽다:

```
PayrollRun (급여 대장 헤더 — "2026년 5월 급여")
└─ Payslip (직원별 급여 명세서, N건)
     ├─ 기본급        3,000,000
     ├─ (+) 수당      200,000        ← 지급 항목
     ├─ (−) 소득세    -150,000       ← 공제 항목
     ├─ (−) 4대보험   -270,000       ← 공제 항목
     └─ 실수령액      2,780,000      (gross 3,200,000 − 공제 420,000)
```

핵심 개념 세 가지:

- **총지급액(gross)** = 기본급 + 수당. 회사가 부담하는 "비용"의 근간.
- **공제(deduction)** = 소득세 + 4대보험. 회사가 떼어서 **대신 보관**(예수금).
- **실수령액(net)** = gross − 공제. 직원 통장에 실제 들어가는 돈.

> ⚠️ 가장 헷갈리는 지점: **회사가 쓰는 비용(gross)** ≠ **직원이 받는 돈(net)**. 그 차액(공제)은 사라진 게 아니라 회사가 잠시 들고 있다가 국가에 낸다 — 그래서 **부채(예수금)**. 이게 §5의 분개에서 그대로 드러난다.

학습 단순화: 소득세·4대보험을 정밀 계산하지 않고 **고정 비율(예: 합산 13%)** 로 간이 처리. "공제가 예수금 부채가 된다"는 구조만 체득하면 목적 달성.

## 5. ⭐ HR → FI 연계: 급여 확정 = 인건비 전표

이 Phase의 하이라이트. Phase 5의 자동 분개 + Phase 4의 이벤트 패턴이 그대로 재사용된다.

급여를 확정하면(`PayrollRun.confirm`) 자동으로 이런 전표가 박힌다:

```
차) 급여비용(인건비)   3,200,000     ← 회사가 쓴 총비용 (gross)
        대) 예수금(소득세·4대보험)   420,000     ← 회사가 떼어 보관 (부채)
        대) 미지급급여 or 현금        2,780,000   ← 직원에게 줄/준 돈
```

읽는 법:
- **차변 = 비용 발생** (급여비용 gross 전액). 손익에 −3,200,000.
- **대변 = 두 갈래**: 떼어둔 공제(예수금 부채 증가) + 실제 줄 돈(미지급급여 부채 또는 현금 감소).
- **차변 합 = 대변 합** — Phase 5의 복식부기 불변식 그대로. 3,200,000 = 420,000 + 2,780,000. ✅

이후 실제 지급(이체)하면:
```
차) 미지급급여   2,780,000
        대) 현금   2,780,000
```
→ Phase 5의 Payment(출금)와 같은 모양. "비용 인식"과 "현금 지급"을 분리하는 게 발생주의 회계.

> 💡 구현 관점: Phase 5에서 만든 `AutoJournalService`에 "급여 분개" 규칙을 한 줄기 더 추가하면 된다. 사건(급여 확정)이 일어나는 곳에서 이벤트를 발행하거나 서비스를 직접 호출 — Phase 5의 매출/매입/매출원가 분개와 같은 자리. **새 패턴이 아니라 익숙한 패턴의 다섯 번째 적용.**

## 6. 새 계정과목 — 인건비를 위한 계정

Phase 5에서 만든 계정 트리(1000 자산, 2000 부채, 4000 수익, 5000 비용)에 **말단 계정 몇 개를 더한다**:

| 코드(예) | 계정명 | 유형 | 정상잔액 | 역할 |
| --- | --- | --- | --- | --- |
| 5200 | 급여비용 | EXPENSE | 차변 | 인건비 (gross) |
| 2400 | 예수금 | LIABILITY | 대변 | 떼어둔 소득세·4대보험 |
| 2500 | 미지급급여 | LIABILITY | 대변 | 아직 안 준 급여 |

> 코드 번호·계정 구성은 단계 3에서 확정. Phase 5의 시드(V23) 패턴을 그대로 따라 새 마이그레이션으로 추가.

## 7. 인가 — Phase 6 회수: HR 역할

Phase 6에서 만든 RBAC가 여기서 또 일한다. 급여 정보는 **극도로 민감**(개인 연봉)하므로 접근 통제가 필수:

- 새 역할 후보: `HR`(인사팀) — HR/Payroll 모듈 접근.
- `@PreAuthorize("hasAnyRole('HR','ADMIN')")` — Phase 6의 인가 패턴 그대로 컨트롤러에 부착.
- 부서 매핑: 인사팀(DEPT-HR) → HR 역할 (Phase 6 V28 시드 방식 확장).

> 💡 "재무팀은 FI만, 영업팀은 SD만"에 "인사팀은 HR만"이 자연스럽게 추가된다. 급여 명세는 본인+인사+관리자만 봐야 한다는 게 내부통제의 상식.

## 9. Phase 1~6 와의 누적성

```
Phase 1: Employee 마스터 (code/email/department/hireDate)
   │                              └─→ §2: 직급·기본급으로 확장
Phase 2: 헤더-라인 구조 (SalesOrder↔Line)
   │                              └─→ §4: PayrollRun↔Payslip 같은 구조
Phase 3: StockMovement = 이력 누적 → 집계
   │                              └─→ §3: Attendance 가 같은 패턴
Phase 4: 이벤트 리스너 (사건 → 여러 모듈)
   │                              └─→ §5: 급여 확정 사건 → 회계
Phase 5: AutoJournalService 자동 분개 + 복식부기
   │                              └─→ §5: 급여 분개가 다섯 번째 규칙. 차/대 균형 그대로
Phase 6: RBAC 인가 + 부서→역할 시드
   │                              └─→ §7: HR 역할 추가, 급여 정보 접근 통제
Phase 7: ⭐ 사람(HR)을 회계(FI)에 연결 — "급여 확정 = 인건비 전표"
```

→ Phase 7은 **새 개념이 거의 없다.** 헤더-라인(P2), 이력 누적(P3), 이벤트(P4), 자동 분개(P5), 인가(P6)가 "급여"라는 새 도메인에 다시 적용될 뿐. 그래서 이번 Phase는 **앞에서 배운 패턴들의 종합 연습** 성격이 강하다.

## 10. 사용자가 이번 브리핑에서 잡아야 할 핵심 5가지

1. **HR은 결국 회계로 흐른다** — 인건비는 큰 비용 항목. "급여 확정 = 인건비 전표"가 이 Phase의 본질.
2. **gross ≠ net** — 회사가 쓰는 비용(총지급액)과 직원이 받는 돈(실수령액)은 다르다. 그 차액(공제)은 **예수금 부채**로, 회사가 잠시 들고 있다 국가에 낸다.
3. **근태 → 급여** — 출퇴근 기록(이력 누적)이 월 단위로 집계되어 급여 금액을 만든다. Phase 3 원장 패턴의 재현.
4. **급여 분개도 차변 합 = 대변 합** — `급여비용 = 예수금 + 미지급급여`. Phase 5의 복식부기 불변식이 그대로 강제된다.
5. **거의 모든 게 복습** — 헤더-라인(P2)·이력누적(P3)·이벤트(P4)·자동분개(P5)·인가(P6)의 종합. 새 패턴은 적고, 익숙한 패턴을 사람 도메인에 적용한다.

- "급여 정보를 Employee에 직접 붙일지(A) 별도 급여계약 테이블로 뺄지(B) — 학습엔 어느 쪽이 나은가?"
- "근태를 시간 기반(초과근무 수당)까지 갈지, 근무일수 기반 간이로 둘지?"

- "급여 확정 시 분개를 이벤트로 발행할지(Phase 4 방식), PayrollService가 AutoJournalService를 직접 부를지(Payment 방식)?"
- "'미지급급여'를 거치나, 아니면 확정과 동시에 현금 지급으로 단순화하나? (발생주의 vs 현금주의)"

질문이 없거나 충분히 이해됐다면 **"단계 3 (모델링 제안) 으로 진행"** 이라고 알려 주세요. — 그러면 직원 확장/근태/급여(PayrollRun·Payslip) ERD, 급여 계산 규칙, HR→FI 자동 분개 흐름, 새 계정과목, HR 역할·인가, 마이그레이션 계획을 설계해 승인받겠습니다.
