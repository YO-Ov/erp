## 1. 설계 전체 그림

```
Employee (Phase 1, 그대로)
   │ 1:N (한 직원의 급여계약 이력)
   ▼
EmploymentContract  ── 발효일/기본급(월)/직급/소정근로시간 ──┐
   │                                                        │ 급여 계산의 기준
Attendance (근태, 하루 N건 누적) ──월 집계: 근무시간/연장시간─┤
                                                            ▼
                                              PayrollRun (월 급여대장 헤더)
                                                 └─ Payslip (직원별 명세, N건)
                                                       ├─ 기본급 / 연장수당  → gross
                                                       ├─ 공제(소득세·4대보험)
                                                       └─ net (실수령)
                                                            │ 확정(confirm)
                                                            ▼
                                              PayrollConfirmedEvent  ──(Phase 4 이벤트)
                                                            ▼
                                       PayrollAccountingListener (fi/integration/hr/)
                                                            ▼
                                       AutoJournalService.createPayrollEntry()
                                          차) 급여비용 + 법정복리비
                                          대) 예수금(소득세/4대직원/4대회사) + 미지급급여
                                                            │ 나중에 지급
                                                            ▼
                                       Payment(DISBURSEMENT) → 차)미지급급여 / 대)현금
```

핵심 한 줄: **근태(이력) → 급여 계산(헤더-라인) → 확정 이벤트 → 인건비 전표(복식부기)**. Phase 1~6 패턴의 종합.

## 2. ERD — 새 테이블 4개

```
employment_contract                attendance
├─ id                              ├─ id
├─ employee_id   FK→employee       ├─ employee_id  FK→employee
├─ position      (직급 enum)       ├─ work_date    (근무일)
├─ base_salary   (월 기본급)       ├─ clock_in     (출근 시각)
├─ contracted_hours (월 소정근로)  ├─ clock_out    (퇴근 시각)
├─ effective_from (발효일)         ├─ worked_minutes  (파생: 근무 분)
├─ effective_to   (만료일, null=현재)└─ overtime_minutes(파생: 연장 분)
└─ status

payroll_run (헤더)                 payslip (라인)
├─ id                              ├─ id
├─ period        (YYYY-MM)         ├─ payroll_run_id FK→payroll_run
├─ run_date                        ├─ employee_id    FK→employee
├─ status (DRAFT/CONFIRMED/PAID)   ├─ base_pay       (기본급)
├─ total_gross                     ├─ overtime_pay   (연장수당)
├─ total_deduction                 ├─ gross_pay      (= base + overtime)
├─ total_net                       ├─ income_tax     (소득세 공제)
└─ confirmed_at                    ├─ insurance_employee (4대보험 직원분 공제)
                                   ├─ insurance_company  (4대보험 회사분 — 공제아님, 회사비용)
                                   ├─ total_deduction (= income_tax + insurance_employee)
                                   └─ net_pay        (= gross - total_deduction)
```

### 2-1. `employment_contract` — 급여계약 (BaseEntity 상속)

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| id | BIGINT PK | |
| employee_id | FK→employee, NOT NULL | 한 직원이 여러 계약(시간순) 가질 수 있음 |
| position | VARCHAR(16) | 직급 enum: STAFF/SENIOR/MANAGER/DIRECTOR |
| base_salary | DECIMAL(15,2) | 월 기본급 |
| contracted_hours | INT | 월 소정근로시간 (예: 209h) — 시급 환산 분모 |
| effective_from | DATE NOT NULL | 발효일 |
| effective_to | DATE NULL | 만료일(null = 현재 유효) |
| status | VARCHAR(16) | ACTIVE/INACTIVE |

> **왜 이력 테이블인가**(확정 ①): 연봉은 매년 바뀐다. "2025년엔 3,000만, 2026년부터 3,300만"을 `effective_from/to`로 표현. 급여 계산 시 "그 달에 유효한 계약"을 찾아 쓴다. → 시급 = `base_salary / contracted_hours`.

### 2-2. `attendance` — 근태 (BaseEntity 상속, 이력 누적)

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| employee_id | FK→employee | |
| work_date | DATE | 근무일 (직원+일자 UNIQUE) |
| clock_in / clock_out | TIME | 출퇴근 시각 |
| worked_minutes | INT | 파생: 근무 분 (휴게 제외 단순화) |
| overtime_minutes | INT | 파생: 소정 초과분 (예: 8h=480분 초과) |

> Phase 3 `StockMovement`처럼 "하루 한 건씩 쌓이는" 이력. 월 급여 계산 때 `SUM`으로 집계. `(employee_id, work_date)` UNIQUE로 하루 중복 기록 방지.

### 2-3. `payroll_run` ↔ `payslip` — 헤더-라인 (Phase 2 패턴)

`payroll_run`은 "2026년 5월 급여 대장"(헤더), `payslip`은 직원별 명세서(라인). 상태 머신:

```
DRAFT ──confirm()──▶ CONFIRMED ──markPaid()──▶ PAID
         (급여 계산 확정       (실제 이체 완료)
          + 회계 전표 생성)
```

- `period` (YYYY-MM)에 UNIQUE — 한 달에 급여대장 하나.
- 합계 컬럼(total_gross/deduction/net)은 payslip 집계 캐시(조회 편의).

## 3. 급여 계산 규칙 (시간 기반, 확정 ②)

한 직원의 한 달 명세서(`Payslip`) 계산:

```
유효계약 = 그 period 에 유효한 EmploymentContract (effective_from ≤ 말일, effective_to null or ≥ 1일)
시급      = base_salary / contracted_hours
월근태집계 = Σ attendance.overtime_minutes (그 달, 그 직원)

base_pay     = 유효계약.base_salary                       (만근 가정 — 결근 일할은 범위 밖)
overtime_pay = (overtime_minutes / 60) × 시급 × 1.5       (연장수당 1.5배)
gross_pay    = base_pay + overtime_pay

income_tax          = round(gross_pay × 6%)              (소득세 간이)
insurance_employee  = round(gross_pay × 4.5%)            (4대보험 직원분)
insurance_company   = round(gross_pay × 4.5%)            (4대보험 회사분 — 회사 추가비용)

total_deduction = income_tax + insurance_employee        (직원에게서 떼는 것만)
net_pay         = gross_pay - total_deduction            (실수령)
```

> 💡 **회사분 4대보험(insurance_company)은 공제가 아니다.** 직원 월급에서 떼는 게 아니라 회사가 별도로 더 부담하는 비용 → `total_deduction`에 안 들어가고, 회계에선 `법정복리비`(비용)로 잡힌다(§5). "회사가 직원 한 명에 쓰는 진짜 총비용 = gross + insurance_company".
>
> 세율은 학습용 고정값. 실제 4대보험은 항목별(국민연금·건강·고용·산재) 요율이 다르고 상한이 있지만, "구조"가 목적이라 단순화. 상수로 빼서 한눈에 보이게 한다.

### 3-1. 일할/결근은?

이번 범위에선 **만근 가정**(base_pay = 계약 기본급 전액). 결근 일할 차감은 넣지 않는다(복잡도 대비 학습가치 낮음). 단 연장수당은 근태 기반으로 실제 계산 → "근태가 급여에 반영된다"는 핵심은 연장수당으로 체득.

## 4. ⭐ HR → FI 자동 분개 (확정 ③④⑤)

### 4-1. 새 계정과목 (V29 시드)

Phase 5 계정 트리(1000/2000/4000/5000)에 말단 계정 추가:

| 코드 | 계정명 | 유형 | 정상잔액 | 역할 |
| --- | --- | --- | --- | --- |
| 5200 | 급여비용 | EXPENSE | 차변 | 직원 총지급액(gross) |
| 5300 | 법정복리비 | EXPENSE | 차변 | 4대보험 회사부담분 |
| 2400 | 예수금-소득세 | LIABILITY | 대변 | 떼어둔 소득세 |
| 2500 | 예수금-사회보험 | LIABILITY | 대변 | 4대보험 직원분+회사분 |
| 2600 | 미지급급여 | LIABILITY | 대변 | 직원에게 줄 실수령액 |

`SystemAccounts`에 상수 추가:
```java
public static final String SALARY_EXPENSE   = "5200";
public static final String LEGAL_WELFARE     = "5300";
public static final String WITHHOLDING_TAX   = "2400";
public static final String WITHHOLDING_INSURANCE = "2500";
public static final String SALARY_PAYABLE    = "2600";
```

### 4-2. 급여 확정 분개 (`createPayrollEntry`)

`PayrollRun.confirm()` → `PayrollConfirmedEvent` 발행 → `PayrollAccountingListener`가 수신 → `AutoJournalService.createPayrollEntry()`. 대장 전체 합산 분개:

```
차) 급여비용(5200)           Σ gross_pay
차) 법정복리비(5300)         Σ insurance_company
        대) 예수금-소득세(2400)        Σ income_tax
        대) 예수금-사회보험(2500)      Σ (insurance_employee + insurance_company)
        대) 미지급급여(2600)           Σ net_pay
```

검증: 차변 합 = `Σgross + Σinsurance_company`,
대변 합 = `Σincome_tax + Σ(insurance_employee+insurance_company) + Σnet_pay`.
`net = gross - income_tax - insurance_employee` 이므로 양변이 항상 일치 → Phase 5 복식부기 불변식 통과. ✅

예시(직원 1명, gross 3,500,000):
```
차) 급여비용      3,500,000
차) 법정복리비      157,500   (3.5M × 4.5%)
        대) 예수금-소득세      210,000   (× 6%)
        대) 예수금-사회보험    315,000   (직원 157,500 + 회사 157,500)
        대) 미지급급여      3,132,500   (3.5M - 210,000 - 157,500)
   차변 3,657,500 = 대변 3,657,500 ✅
```

### 4-3. 지급 분개 (2단계, 확정 ④)

`PayrollRun.markPaid()` 시 (또는 기존 Payment 재사용):
```
차) 미지급급여(2600)   Σ net_pay
        대) 현금(1100)        Σ net_pay
```
→ Phase 5 출금(`Payment.DISBURSEMENT`)과 같은 모양. "비용 인식(확정)"과 "현금 유출(지급)"의 분리 = 발생주의.

전표 출처 추적: `findBySourceTypeAndSourceId(PAYROLL, payrollRunId)`.
