# 1/6. 급여계약 — "발효일로 끊어 읽는" 이력 테이블

> Phase 7 의 모든 계산은 **"그 달에 유효한 급여 조건이 무엇인가"** 에서 출발한다.
> 그 답을 주는 게 `EmploymentContract` — 직원과 분리된 별도 이력 테이블이다.
> 이 글은 **왜 Employee 에 연봉 컬럼을 박지 않고 이력 테이블로 뺐는가**, 그리고 그 이력을 어떻게 "한 시점의 진실"로 좁혀 읽는가를 본다.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/hr/contract/
├─ EmploymentContract.java     발효일 이력 + 시급 환산
├─ Position.java               직급 enum (STAFF/SENIOR/MANAGER/DIRECTOR)
├─ ContractStatus.java         ACTIVE / INACTIVE
├─ EmploymentContractRepository.java   findEffectiveOn (그 날 유효한 계약)
├─ EmploymentContractService.java      새 계약 발효 시 직전 계약 자동 종료
└─ EmploymentContractController.java   /api/employment-contracts (HR/ADMIN)
```

---

## 🔥 왜 Employee 에 연봉을 안 넣었나

가장 단순한 설계는 `employee` 테이블에 `base_salary` 컬럼 하나를 두는 것이다. 그런데 연봉은 **매년 바뀐다**. 컬럼 하나로는 이런 질문에 답할 수 없다:

- "이 직원의 **2025년 5월** 급여를 다시 계산해줘" (그땐 연봉이 얼마였지?)
- "연봉 인상 **이력**을 보여줘"

급여는 **시점(point-in-time)** 의 사실이다. 그래서 "현재 값" 하나가 아니라 **"발효일 구간을 가진 행들"** 로 저장한다.

```
employment_contract
├─ employee_id
├─ base_salary       3,000,000  (2025년 계약)
├─ effective_from    2025-01-01
├─ effective_to      2025-12-31   ← 닫힘
└─ status            INACTIVE

employment_contract
├─ employee_id       (같은 직원)
├─ base_salary       3,300,000  (2026년 계약)
├─ effective_from    2026-01-01
├─ effective_to      NULL         ← 현재 유효
└─ status            ACTIVE
```

> 💡 같은 패턴을 이미 봤다 — Phase 3 의 `StockMovement`(재고 변동 이력), Phase 5 의 `JournalLine`(전표 라인). ERP 는 **"덮어쓰지 않고 쌓는다"**. 과거를 지우면 재계산·감사가 불가능하기 때문.

## 🔥 "그 날 유효한 계약" 한 줄로 좁히기

이력으로 쌓아두면, 읽을 때 **한 시점으로 좁히는 규칙**이 필요하다. 그게 `isEffectiveOn`:

```java
/** 주어진 일자에 이 계약이 유효한가 (발효일 ≤ date ≤ 만료일, 만료일 null 이면 현재 유효). */
public boolean isEffectiveOn(LocalDate date) {
    if (date.isBefore(effectiveFrom)) return false;
    return effectiveTo == null || !date.isAfter(effectiveTo);
}
```

그리고 DB 레벨에서 같은 조건을 거는 Repository 쿼리 — 급여 계산이 직접 쓴다:

```java
@Query("SELECT c FROM EmploymentContract c JOIN FETCH c.employee "
        + "WHERE c.status = com.hwlee.erp.hr.contract.ContractStatus.ACTIVE "
        + "AND c.effectiveFrom <= :date "
        + "AND (c.effectiveTo IS NULL OR c.effectiveTo >= :date) "
        + "ORDER BY c.employee.id ASC")
List<EmploymentContract> findEffectiveOn(@Param("date") LocalDate date);
```

급여 계산은 **"5월 말일(2026-05-31)에 유효한 계약 전부"** 를 이 쿼리로 가져온다. 한 직원당 한 줄이 떨어지는 게 정상 — 그게 다음 절의 불변식이다.

## 🔥 "한 시점에 유효 계약은 하나" — 발효 시 직전 계약 자동 종료

이력 테이블의 함정: 발효 구간이 **겹치면** `findEffectiveOn` 이 한 직원에 대해 두 줄을 반환하고, 급여가 이중 계산된다. 이걸 막는 건 "새 계약을 넣을 때 직전 계약을 닫는" 규칙이다 — 서비스가 강제한다:

```java
@Transactional
public EmploymentContractResponse create(EmploymentContractCreateRequest req) {
    Employee employee = employeeRepository.findById(req.employeeId())
            .orElseThrow(() -> new EntityNotFoundException("Employee not found: id=" + req.employeeId()));

    repository.findByEmployeeIdAndStatusAndEffectiveToIsNull(employee.getId(), ContractStatus.ACTIVE)
            .ifPresent(prev -> {
                if (!req.effectiveFrom().isAfter(prev.getEffectiveFrom())) {
                    throw new IllegalArgumentException(
                            "새 계약 발효일은 직전 계약 발효일(" + prev.getEffectiveFrom() + ") 이후여야 합니다.");
                }
                prev.terminate(req.effectiveFrom().minusDays(1));   // 직전 계약을 새 발효일 '전날' 로 닫는다
            });

    EmploymentContract contract = EmploymentContract.create(
            employee, req.position(), req.baseSalary(), req.contractedHours(), req.effectiveFrom());
    return mapper.toResponse(repository.save(contract));
}
```

엔티티 쪽 `terminate` 는 상태 전이를 자기 책임으로 지킨다:

```java
public void terminate(LocalDate effectiveTo) {
    if (status != ContractStatus.ACTIVE)
        throw new IllegalStateException("ACTIVE 계약만 종료 가능합니다. 현재: " + status);
    if (effectiveTo.isBefore(effectiveFrom))
        throw new IllegalArgumentException("effectiveTo 는 effectiveFrom 이후여야 한다.");
    this.effectiveTo = effectiveTo;
    this.status = ContractStatus.INACTIVE;
}
```

> 2026-03-01 발효 새 계약을 넣으면 → 직전 ACTIVE 계약은 `effective_to = 2026-02-28` 로 닫히고 INACTIVE. 두 계약의 구간이 **하루도 겹치지 않는다**. 이로써 `findEffectiveOn` 은 항상 직원당 0~1줄.

## 🔥 시급 = 기본급 ÷ 소정근로시간

연장수당(다음 글들)의 분모가 되는 시급은 계약이 직접 환산한다 — "월급쟁이의 시간당 단가":

```java
/** 시급 = 월 기본급 / 월 소정근로시간 (원 단위, 반올림). */
public BigDecimal hourlyWage() {
    return baseSalary.divide(BigDecimal.valueOf(contractedHours), 2, RoundingMode.HALF_UP);
}
```

`contractedHours`(월 소정근로시간, 예 209h 또는 200h)가 **시급 환산의 분모**다. 기본급 3,000,000 / 200h = 시급 15,000. 연장 1시간이면 15,000 × 1.5 = 22,500 이 붙는다.

> 왜 계약이 시급을 계산하나? "시급은 기본급과 소정근로시간에서 **파생**되는 값" 이라, 그 두 값을 가진 계약 객체가 계산 책임을 갖는 게 자연스럽다(도메인 응집). 급여 서비스는 `contract.hourlyWage()` 만 부른다 — 나눗셈 규칙(반올림 자리수)을 서비스가 알 필요 없다.

## 정리

| 설계 결정 | 코드에 박힌 방법 |
| --- | --- |
| 연봉은 시점의 사실 | Employee 컬럼 ❌ → 별도 이력 테이블 `employment_contract` |
| 한 시점에 유효 계약 하나 | 새 계약 발효 시 `terminate(new.from - 1)` 로 직전 계약 자동 종료 |
| 그 달 급여 조건 조회 | `findEffectiveOn(말일)` — from/to 구간 필터 |
| 시급은 파생값 | `contract.hourlyWage()` 가 계산 책임 보유 |

다음 글: **근태(Attendance)** — 하루 한 건씩 쌓이는 또 다른 이력, 그리고 출퇴근 시각에서 근무·연장 분이 어떻게 파생되는가.
