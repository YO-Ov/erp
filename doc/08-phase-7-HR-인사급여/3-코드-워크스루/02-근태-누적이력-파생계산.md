# 2/6. 근태 — 쌓이는 이력 + 파생 계산, 그리고 하루 한 건 보장

> 급여계약이 "급여의 기준"이라면, 근태는 "그 달 실제로 얼마나 일했나"의 사실이다.
> `Attendance` 는 Phase 3 의 `StockMovement` 와 똑같은 모양 — **하루 한 건씩 쌓이고, 나중에 SUM 으로 집계**된다.
> 이 글은 ① 출퇴근 시각에서 근무·연장 분을 **파생**하는 법, ② `(직원, 일자)` 중복을 **DB와 코드 양쪽에서** 막는 법, ③ 월 연장근로를 한 번에 집계하는 쿼리를 본다.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/hr/attendance/
├─ Attendance.java             출퇴근 → 근무/연장 분 파생
├─ AttendanceRepository.java   중복 체크 + 월 연장 분 SUM
├─ AttendanceService.java      중복 거부
└─ AttendanceController.java   /api/attendances (HR/ADMIN)
```

## 🔥 파생 컬럼 — 입력은 시각, 저장은 분

근태 등록 시 사용자가 주는 건 **출근 시각 / 퇴근 시각** 두 개뿐이다. `worked_minutes`(근무 분) 와 `overtime_minutes`(연장 분) 는 **거기서 계산되는 파생값** — 입력받지 않고 엔티티가 만든다:

```java
public static final int STANDARD_DAILY_MINUTES = 480;   // 소정근로 8시간

public static Attendance create(Employee employee, LocalDate workDate,
                                LocalTime clockIn, LocalTime clockOut) {
    if (!clockOut.isAfter(clockIn))
        throw new IllegalArgumentException("clockOut 은 clockIn 이후여야 한다.");

    Attendance a = new Attendance();
    a.employee = employee;
    a.workDate = workDate;
    a.clockIn = clockIn;
    a.clockOut = clockOut;
    a.workedMinutes = (int) Duration.between(clockIn, clockOut).toMinutes();        // 근무 = 퇴근 - 출근
    a.overtimeMinutes = Math.max(0, a.workedMinutes - STANDARD_DAILY_MINUTES);      // 연장 = 소정 초과분
    return a;
}
```

예: 08:00 출근 → 18:00 퇴근.
- `workedMinutes` = 600분 (10시간)
- `overtimeMinutes` = 600 − 480 = **120분 (2시간)**

> **왜 파생값을 컬럼으로 저장하나** (계산이 아니라 저장?): 월 급여 계산 때 수천 건 근태를 `SUM(overtime_minutes)` 으로 빠르게 집계하기 위해서다. 매번 출퇴근 시각을 빼서 더하면 DB가 인덱스/집계를 못 쓴다. **"쓸 때 한 번 계산, 읽을 때 집계"** — 전형적인 비정규화 트레이드오프.
>
> 단순화 명시: 휴게시간 차감은 없다(출근~퇴근 전체를 근무로 본다). 근로기준법 풀스펙(야간·휴일 가산율 분리 등)도 제외 — "근태가 급여에 반영된다"는 구조를 보는 게 목적이라 연장수당 한 갈래로 좁혔다.

## 🔥 "하루 한 건" — 두 겹의 방어

같은 직원이 같은 날짜에 근태를 두 번 등록하면 연장수당이 이중 계산된다. 이걸 **DB 제약 + 서비스 체크** 두 겹으로 막는다.

**1겹 — DB UNIQUE 제약** (V29):

```sql
UNIQUE KEY uk_attendance_emp_date (employee_id, work_date)
```

**2겹 — 서비스 사전 체크** (사용자에게 친절한 메시지):

```java
@Transactional
public AttendanceResponse create(AttendanceCreateRequest req) {
    Employee employee = employeeRepository.findById(req.employeeId())
            .orElseThrow(() -> new EntityNotFoundException("Employee not found: id=" + req.employeeId()));
    if (repository.existsByEmployeeIdAndWorkDate(employee.getId(), req.workDate())) {
        throw new IllegalArgumentException(
                "이미 등록된 근태입니다. employeeId=" + employee.getId() + ", workDate=" + req.workDate());
    }
    Attendance attendance = Attendance.create(employee, req.workDate(), req.clockIn(), req.clockOut());
    return mapper.toResponse(repository.save(attendance));
}
```

> **왜 두 겹인가**: 서비스 체크만 두면 동시 요청 두 개가 둘 다 "없음"을 보고 둘 다 INSERT 하는 경합(race)이 가능하다 → 그땐 DB UNIQUE 가 최후의 보루로 막는다(두 번째는 제약 위반). 반대로 DB 제약만 두면 사용자는 의미 불명의 SQL 에러를 본다. **DB 제약 = 정합성의 진실, 서비스 체크 = 친절한 UX** — 역할이 다르다. Phase 3 의 재고 음수 방어와 같은 이중 구조.

## 🔥 월 연장근로 — 한 번의 SUM 으로

급여 계산은 직원별로 "그 달 연장근로 총 분"이 필요하다. 근태 행을 자바로 순회해 더하지 않고, DB 가 집계하게 한다:

```java
/** 한 직원의 기간 내 연장근로 분 합계 — 급여 계산용. 데이터 없으면 0. */
@Query("SELECT COALESCE(SUM(a.overtimeMinutes), 0) FROM Attendance a "
        + "WHERE a.employee.id = :employeeId "
        + "AND a.workDate BETWEEN :from AND :to")
int sumOvertimeMinutes(@Param("employeeId") Long employeeId,
                       @Param("from") LocalDate from,
                       @Param("to") LocalDate to);
```

`COALESCE(..., 0)` 가 핵심 — 근태가 한 건도 없는 직원은 SUM 이 `NULL` 이 된다. 그걸 0 으로 떨어뜨려 **"근태 없음 = 연장 0 = 기본급만"** 으로 자연스럽게 이어진다(NPE 없음).

급여 서비스에서의 호출:

```java
int overtimeMinutes = attendanceRepository.sumOvertimeMinutes(employee.getId(), firstDay, lastDay);
```

`firstDay`/`lastDay` 는 급여 대상 월의 1일과 말일 — 다음 글에서 `YearMonth` 로 어떻게 만드는지 본다.

## 정리

| 개념 | 코드 |
| --- | --- |
| 입력은 시각, 저장은 분 | `Attendance.create` 가 `Duration.between` 으로 파생 |
| 연장 = 소정(480) 초과분 | `max(0, worked - 480)` |
| 하루 한 건 | DB `UNIQUE(employee_id, work_date)` + 서비스 `existsByEmployeeIdAndWorkDate` |
| 월 연장 집계 | `SUM(overtime_minutes) ... BETWEEN` + `COALESCE 0` |
