# Phase 8 도메인 브리핑 — PP(생산): "사서 파는 회사"에서 "만들어 파는 회사"로

> 7단계 사이클의 **단계 1: 도메인 브리핑**.
> 이 단계의 목적은 코드를 짜기 전에 **"제조(PP)가 ERP에서 무슨 일을 하고, 왜 결국 재고(MM)·회계(FI)와 한 몸인가"** 를 글로 이해하는 것.
> 모호한 부분은 **단계 2 Q&A** 에서 질문.

---

## 0. 한 줄 요약

> 지금까지 우리 회사는 **완제품을 사다가(MM 입고) 그대로 팔았다(SD 출하)**. Phase 8은 회사를 **제조업**으로 바꾼다 — 부품을 사서, **레시피(BOM)** 대로 조립해 완제품을 **만든다**. 그러면 창고에서 **부품이 빠져나가고(자재 소모) 완제품이 들어온다(완제품 입고)**.
> 핵심 한 줄: **"생산 완료 = 부품 출고(재고 −) + 완제품 입고(재고 +)"** — Phase 3의 재고 원장과 Phase 4의 이벤트 연계가 그대로 재사용된다. 새 거래(생산)가 MM 재고를 양방향으로 움직인다.

---

## 1. PP가 ERP에서 하는 일 — 제조업의 심장, 왜 MM과 한 몸인가

지금까지의 흐름을 돌아보자:

```
지금까지(상품 매매업):   부품/완제품 구매(MM 입고) ──▶ 재고 ──▶ 판매(SD 출하)
Phase 8 이후(제조업):    부품 구매(MM 입고) ──▶ [생산: 부품 소모 → 완제품 생성] ──▶ 완제품 재고 ──▶ 판매
```

제조업에서 **생산(Production Planning)** 은 "재고를 소비해 더 가치 있는 재고로 바꾸는" 활동이다. 그래서 PP는 독립된 섬이 아니라 **MM(자재) 위에 올라탄다**:

- **투입(input)**: 생산하려면 부품이 창고에 있어야 한다 → MM 재고 **차감**(Phase 3 출고와 동일).
- **산출(output)**: 생산이 끝나면 완제품이 창고에 생긴다 → MM 재고 **증가**(Phase 3 입고와 동일).
- **원가(cost)**: 빠져나간 부품들의 원가 합이 곧 완제품의 원가가 된다 → Phase 3의 이동평균 단가, Phase 5의 원가 회계로 이어진다.

> 💡 핵심 통찰: 생산은 **"재고의 변신(transformation)"** 이다. Phase 3에서 입고/출고는 재고가 **들어오고 나가기만** 했다. Phase 8의 생산은 **한 거래 안에서 어떤 재고는 나가고(부품) 어떤 재고는 들어온다(완제품)**. 이 양방향성이 생산을 특별하게 만든다.

## 2. BOM (Bill of Materials) — 완제품을 부품으로 분해하는 레시피

생산의 출발점은 **"이 완제품 하나를 만들려면 무슨 부품이 몇 개 필요한가"** 다. 이걸 적어둔 표가 **BOM(자재 명세서)** — 요리의 레시피와 같다.

예시 — `hyunwoo 노트북 15"` 1대의 BOM:

```
hyunwoo 노트북 15" (완제품)  ← 만들 대상
├─ 15" LCD 패널        × 1
├─ 메인보드            × 1
├─ 16GB 메모리         × 2
├─ 512GB SSD          × 1
└─ 배터리팩            × 1
```

데이터로 표현하면 **"완제품 - 부품" 의 부모-자식 관계**:

| 컬럼 | 예시 | 설명 |
| --- | --- | --- |
| 완제품(부모 Item) | hyunwoo 노트북 15" | 만들 대상 |
| 부품(자식 Item) | 16GB 메모리 | 투입될 자재 |
| 소요량(qty) | 2 | 완제품 1개당 필요 수량 |

> 💡 익숙한 패턴의 재현: 이건 Phase 1 `Department` 의 **자기참조 트리**(부서가 부서를 부모로 가짐)와 같은 모양이다. 다만 여기선 Item이 Item을 참조한다 — 완제품 Item이 여러 부품 Item을 자식으로 거느린다(헤더-라인 구조, Phase 2 수주와 동형).

**설계 갈림길(단계 3에서 결정):**
- **(간이) 단일 레벨 BOM** — "완제품 = 부품들의 1단계 조합". 학습엔 이걸로 충분.
- **(상세) 다단계 BOM** — 부품이 또 하위 부품으로 쪼개짐(반제품). 재귀 전개 필요. 현실적이지만 복잡 → 범위 밖 후보.

## 3. 생산지시(Production Order) — 무엇을·얼마나 만들지 (헤더-라인)

BOM이 "레시피"라면, **생산지시(Production Order)** 는 "이번에 실제로 만들 주문서"다. "5월 3일, hyunwoo 노트북 100대 생산하라."

Phase 2 수주, Phase 5 전표와 같은 **헤더-라인 구조**가 자연스럽다:

```
ProductionOrder (생산지시 헤더 — "노트북 100대 만들기")
├─ 대상 완제품   hyunwoo 노트북 15"
├─ 생산 수량     100
├─ 지시일/완료예정일
├─ 상태          PLANNED / RELEASED / COMPLETED
└─ 소요 자재 라인 (BOM × 수량으로 자동 산출, N건)
     ├─ 15" LCD 패널   소요 100 (= 1 × 100)
     ├─ 16GB 메모리    소요 200 (= 2 × 100)
     └─ ...
```

> 💡 BOM(품목당 레시피)과 생산지시(이번 회차 주문)는 다른 개념이다. BOM은 **마스터성**(잘 안 바뀜), 생산지시는 **트랜잭션**(매번 생김). SD에서 `Item`(마스터)과 `SalesOrderLine`(트랜잭션)이 다른 것과 같다.

## 4. ⭐ 자재 소요량 계산 (MRP 미니버전) — BOM × 생산수량

이 Phase의 **학습 하이라이트**. "노트북 100대를 만들려면 부품이 각각 몇 개 필요한가?"

```
필요 자재량 = Σ (BOM 소요량 × 생산 수량)

메모리:  2 (BOM) × 100 (생산) = 200개 필요
SSD:     1 (BOM) × 100 (생산) = 100개 필요
LCD:     1 (BOM) × 100 (생산) = 100개 필요
```

이게 바로 **MRP(Material Requirements Planning, 자재 소요량 계획)** 의 가장 단순한 형태다. 현실의 MRP는 여기에 "현재 재고", "리드타임", "안전재고", "발주 제안"까지 얹지만, 우리는 **"BOM × 수량 = 필요량"** 이라는 골격만 체득한다.

생산 직전 자연스럽게 따라오는 질문 — **"부품이 충분한가?"**:

```
필요량 200  vs  현 재고 150  →  50개 부족 → 생산 불가(또는 경고)
```

> 💡 여기서 Phase 3의 **가용재고 검증**(출고 시 "재고보다 많이 못 뺀다")이 그대로 재현된다. 생산도 결국 "부품을 출고하는 것"이라, 재고가 모자라면 막아야 한다. 새 규칙이 아니라 Phase 3 규칙의 재적용.

**설계 갈림길(단계 3):** 재고 부족을 **언제** 막을까 — 생산지시 확정(RELEASED) 시점? 아니면 생산 완료(COMPLETED) 시점? (현실은 보통 자재 출고가 일어나는 완료/착수 시점.)

---

## 5. 생산 완료 시 처리 — 자재 출고(소모) + 완제품 입고

생산지시를 **완료(complete)** 하면, 한 번에 두 가지 재고 이동이 일어난다:

```
① 부품 소모 (출고):  필요한 부품들이 창고에서 빠진다     → MM GoodsIssue (재고 −)
② 완제품 입고:        만들어진 완제품이 창고에 들어온다   → MM GoodsReceipt (재고 +)
```

핵심은 **이 두 이동을 우리가 Phase 3에서 이미 만들어 뒀다는 것**이다. 생산은 새 재고 메커니즘을 만들지 않고, MM의 출고/입고를 **호출**한다.

```
ProductionOrder.complete()
   ├─▶ (각 부품마다) GoodsIssue 생성·전기 → 부품 재고 차감 (Phase 3 출고 로직 그대로)
   └─▶ GoodsReceipt 생성·전기            → 완제품 재고 증가 (Phase 3 입고 로직 그대로)
```

> 💡 이게 **정확히 Phase 4 패턴**이다. Phase 4에서 "출하 확정(SD) → 재고 차감·GoodsIssue 생성(MM)"을 **이벤트 한 트랜잭션**으로 묶었다. Phase 8은 "생산 완료(PP) → 부품 GoodsIssue + 완제품 GoodsReceipt(MM)"를 같은 방식으로 묶는다. **사건 하나 = 트랜잭션 하나 = 여러 모듈 변화**, 세 번째 적용(SD→MM, HR→FI 에 이어).

**설계 갈림길(단계 3):** PP→MM 연결을 **이벤트 발행**(Phase 4 방식)으로 할지, `ProductionService` 가 MM 서비스를 **직접 호출**할지. (출고가 여러 건이라 트랜잭션 경계·롤백을 어떻게 잡을지가 포인트.)

## 6. PP → MM/FI 흐름: 원가의 변신

생산은 회계적으로도 의미가 있다 — **원가가 형태를 바꾼다**:

```
부품 재고(자산)  ──생산──▶  완제품 재고(자산)
   원가 800,000        →        원가 800,000 (+ 가공원가)
```

빠져나간 부품들의 **원가 합**이 완제품의 **원가**가 된다(투입 = 산출의 원가 보존). Phase 3의 이동평균 단가로 "부품이 얼마짜리였는지"를 알 수 있으니, 그 합을 완제품 입고 단가로 쓸 수 있다.

```
완제품 1대 원가 = Σ (부품 소요량 × 부품 이동평균 단가)
              = LCD 200,000 + 메모리 2×50,000 + SSD 150,000 + ...
```

> ⚠️ 학습 범위 결정 포인트: 현실 제조 회계는 **재공품(WIP) 계정**·노무비·제조간접비 배부까지 가지만, 우리는 단순화한다. 후보:
> - **(간이) 재고 이동만** — 부품 재고 ↓, 완제품 재고 ↑. 회계 전표는 생략하거나 "재고 대체" 한 줄.
> - **(상세) 생산 전표** — 차)완제품 / 대)부품(원재료). Phase 5 `AutoJournalService` 에 생산 분개 규칙 추가.
>
> Phase 7에서 "급여 확정 = 인건비 전표"를 했듯, 원하면 "생산 완료 = 재고 대체 전표"를 여섯 번째 자동분개로 넣을 수 있다. 넣을지 말지는 단계 2~3에서 결정.

## 7. 상태머신 — 생산지시의 일생

Phase 2 수주(DRAFT→CONFIRMED→...), Phase 3 입출고(DRAFT→POSTED→CANCELLED)처럼 생산지시도 상태를 갖는다:

```
PLANNED ──release()──▶ RELEASED ──complete()──▶ COMPLETED
(계획만 잡힘        (생산 착수·확정       (생산 완료:
 자재 소요 산출)     자재 확보 확인)        부품 출고 + 완제품 입고)
                                          │
                                    (취소: CANCELLED)
```

각 전이에서 무슨 일이 일어나는지가 곧 비즈니스 규칙이다 — 완료 시점에 재고가 실제로 움직인다(§5). 어느 시점에 자재 가용성을 검증할지(§4)도 여기서 정해진다.

> 💡 익숙한 패턴: "상태가 바뀌는 순간 부수효과(재고 이동·전표)가 일어난다"는 건 Phase 2~7 내내 봐온 것. 생산지시는 그 일곱 번째 상태머신일 뿐.

## 8. 마스터 보강 — Item에 "완제품 / 부품" 구분이 필요하다

현재 `Item` 마스터는 이렇다(Phase 1):

```
Item
├─ code / name
├─ category    NOTEBOOK | MONITOR   ← 지금은 '완제품' 종류만 있다
├─ unit
├─ standardCost  (구매·생산 원가 근간)
└─ standardPrice (판매가)
```

BOM을 만들려면 **부품(LCD·메모리·SSD…) Item** 이 있어야 하고, "이게 완제품인지 부품인지" 구분이 필요하다. 갈림길(단계 3):

- **(A)** `ItemCategory` 에 부품 카테고리 추가(PART, RAW_MATERIAL 등).
- **(B)** Item에 `itemType`(FINISHED/COMPONENT) 같은 **새 분류 축**을 추가. (category=종류, itemType=완제품/부품 역할로 분리.)

> 💡 한 Item이 누군가의 완제품이면서 다른 누군가의 부품일 수도 있다(다단계 BOM의 반제품). 단일 레벨로 간다면 "완제품 / 부품" 2분류로 충분.

부품 Item과 그 BOM은 Phase 1/3 시드 패턴대로 새 마이그레이션으로 추가한다.

## 9. 인가 — Phase 6 회수: PRODUCTION 역할 (생산팀은 이미 있다)

Phase 6 RBAC가 또 일한다. 그리고 **복선이 이미 깔려 있다**:

- Phase 1 시드(V8)에 **생산팀(DEPT-PRODUCTION)** 과 직원 **박생산(park@hyunwoo.com)** 이 있다.
- Phase 6에서 박생산은 **일부러 역할을 안 줬다**("park = 역할 없음, 403 시연용"). → Phase 8에서 **PRODUCTION 역할**을 만들어 부여하면 그 복선이 회수된다.
- `@PreAuthorize("hasAnyRole('PRODUCTION','ADMIN')")` — Phase 6 인가 패턴 그대로 PP 컨트롤러에 부착.
- 부서 매핑: 생산팀(DEPT-PRODUCTION) → PRODUCTION 역할.

> 💡 "영업팀=SD, 구매팀=MM, 재무팀=FI, 인사팀=HR"에 "생산팀=PP"가 마지막 조각으로 채워진다. 부서-역할 대응이 완성된다.

## 11. Phase 1~7 과의 누적성

```
Phase 1: Item 마스터 + Department 자기참조 트리
   │                    └─→ §2,§8: Item↔Item BOM 트리, 완제품/부품 구분
Phase 2: 헤더-라인 + 상태머신
   │                    └─→ §3,§7: ProductionOrder↔소요자재 라인, PLANNED→COMPLETED
Phase 3: 재고 원장 + 가용 검증 + 이동평균 단가
   │                    └─→ §4,§5,§6: 자재 출고/완제품 입고, 부족 검증, 원가 보존
Phase 4: 이벤트 연계 (한 사건 → 여러 모듈 한 트랜잭션)
   │                    └─→ §5: 생산 완료 → 부품 GoodsIssue + 완제품 GoodsReceipt
Phase 5: 자동 분개 + 복식부기
   │                    └─→ §6: (선택) 생산 = 재고 대체 전표, 여섯 번째 규칙
Phase 6: RBAC 인가 + 부서→역할 시드
   │                    └─→ §9: PRODUCTION 역할, park 복선 회수
Phase 7: 사건→회계 연계의 종합 연습
   │                    └─→ Phase 8: 사건→재고 연계의 종합 연습 (양방향 재고 이동)
Phase 8: ⭐ "사서 파는 회사 → 만들어 파는 회사" — 생산이 재고를 변신시킨다
```

→ Phase 8도 **새 개념이 거의 없다.** 자기참조 트리(P1), 헤더-라인·상태머신(P2), 재고 원장·가용검증·원가(P3), 이벤트 연계(P4), 자동분개(P5), 인가(P6)가 "생산"이라는 도메인에 다시 적용된다. 유일한 새 맛은 **BOM(레시피)** 과 **양방향 재고 이동(부품 ↓ + 완제품 ↑)**.

## 12. 사용자가 이번 브리핑에서 잡아야 할 핵심 5가지

1. **생산 = 재고의 변신** — 부품 재고가 빠지고(출고) 완제품 재고가 생긴다(입고). 한 거래가 재고를 **양방향**으로 움직이는 게 핵심.
2. **BOM = 레시피** — "완제품 1개당 부품 몇 개"의 마스터 데이터. Item↔Item 자기참조 트리(Phase 1 Department 패턴).
3. **자재 소요량 = BOM × 생산수량** — MRP의 미니버전. 그리고 "부품이 충분한가"는 Phase 3 가용재고 검증의 재현.
4. **생산 완료는 MM을 호출한다** — 출고/입고 메커니즘을 새로 안 만들고 Phase 3·4의 GoodsIssue/GoodsReceipt + 이벤트를 재사용.
5. **거의 모든 게 복습** — 트리(P1)·헤더라인/상태머신(P2)·재고원장(P3)·이벤트(P4)·자동분개(P5)·인가(P6)의 종합. 새 패턴은 BOM 하나뿐.

이 브리핑을 읽고 **모호하거나 의문이 있는 부분을 질문** 해 주세요. 출발점이 될 예시 질문:

- "BOM을 단일 레벨(완제품=부품 1단계)로 둘지, 다단계(반제품 포함, 재귀 전개)까지 갈지?"
- "Item의 완제품/부품 구분을 `ItemCategory` 확장(A)으로 할지, 별도 `itemType` 축(B)으로 분리할지?"
- "자재 부족을 언제 막을지 — 생산지시 확정(RELEASED) 시점인가, 완료(COMPLETED) 시점인가?"
- "생산 완료 시 부품 출고를 부품별 GoodsIssue 여러 건으로 할지, 한 건에 여러 라인으로 묶을지?"
- "PP→MM 연결을 이벤트 발행(Phase 4 방식)으로 할지, ProductionService가 MM 서비스를 직접 부를지?"
- "완제품 입고 단가를 투입 부품의 실제(이동평균) 원가 합으로 산정할지, 그냥 표준원가로 둘지?"
- "생산 회계 전표(재고 대체)를 넣을지, 재고 이동만 하고 회계는 생략할지? (재공품 WIP까지는 범위 밖)"
- "생산지시 화면(목록/상세/생성)도 이번에 같이 만들지, 도메인+API+테스트까지만 하고 화면은 뒤로 미룰지?"
