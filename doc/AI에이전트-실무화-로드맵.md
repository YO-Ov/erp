# AI 에이전트 실무화 로드맵 — 실무는 어떻게 하는가 & 다음 과제

> 목적: ERP 챗봇/에이전트를 실무(상용 ERP·AI 코파일럿) 수준으로 끌어올리기 위한 정리.
> 지금까지 구현한 것과 실무의 차이를 짚고, 남은 3대 과제를 할 일로 등록한다.
> 관련: [AI에이전트-로컬LLM-업무자동화-설계.md](AI에이전트-로컬LLM-업무자동화-설계.md), `PROGRESS.md`

## 1. 현재 구현 요약 (실무 대비 "잘 맞는" 것)

ERP 웹 챗봇 → 프록시 → 로컬 LLM 에이전트 → ERP 구조로, 다음은 이미 실무 정석에 가깝다.

| 실무 원칙 | 우리 구현 |
|---|---|
| **툴/함수 호출**(정의된 안전한 액션만 LLM이 선택) | 도메인 API 매핑(결재·수주·고객·손익). 범용 Text-to-SQL은 관리자 폴백으로 축소 |
| **사용자 권한 상속**(on-behalf-of) | 프록시가 사용자 JWT 전달 → 에이전트가 그 사용자로 ERP API 호출 → ERP RBAC 적용 |
| **읽기 우선 + 쓰기는 승인 게이트**(human-in-the-loop) | plan → 미리보기 → y/n → apply |
| **인가는 단일 통제점**(앱이 판단, LLM 신뢰 X) | 인가는 전부 ERP `@PreAuthorize`. 권한 규칙 한 곳 → 화면·챗봇 동시 적용 |
| **감사 로그** | `chat_log.jsonl`(조회/답변 기록) |

상용 제품(SAP Joule, Microsoft Copilot for Dynamics 365, Salesforce Einstein Copilot,
Oracle Fusion AI, ServiceNow Now Assist)도 "정의된 스킬 + 사용자 권한 + 감사" 구조이며,
**범용 Text-to-SQL이 아니라 큐레이션된 액션**을 쓴다 — 우리가 마지막에 전환한 방향과 같다.

## 2. 실무가 "더 나가는" 지점 (= 다음 과제)

지금은 **역할(Role) 단위**까지만 통제한다. 실무는 아래로 더 내려간다.

### 과제 ② 직무분리(SoD, Segregation of Duties) — 규모: 중
- **무엇/왜**: 한 사람이 **등록 + 승인**을 동시에 못 하게 강제(부정·오류 방지). 감사의 핵심 통제.
- **실무 방식**: "기안자 ≠ 결재자", "등록자 ≠ 승인자" 강제. 위반 시 차단. 민감 조합(발주+입고, 전표작성+승인) 분리.
- **우리 현황**: 전자결재(상신≠승인)로 절반은 이미 있음. **명시적 self-approval 차단 규칙은 미확인/부분**.
- **할 일**: 결재 엔진에 "본인이 상신한 문서는 본인이 결재 불가" 규칙 + 결재선 구성 시 본인 제외 검증 + 위반 케이스 테스트.
- **전제/리스크**: 기존 결재 시나리오 테스트 회귀 주의. 결재선 자동계산 로직과 충돌 없는지 확인.

### 과제 ① 데이터 레벨(행 단위) 보안 — 규모: 대, 가장 값짐
- **무엇/왜**: "손익 조회 가능/불가"가 아니라 **"어느 본부/사업부의 손익까지"**. 영업본부장은 자기 본부 실적만, CFO는 전사.
- **실무 방식**: 거래에 조직 차원(회사코드·사업부·코스트센터) 태그 → 조회를 사용자 조직 스코프로 필터(row-level security).
- **우리 현황**: 손익은 전사 합산(조직 차원 없음). 수주는 담당자/고객은 있으나 "본부" 스코프는 없음.
- **할 일(선 조사 필요)**: ⓐ 거래 데이터(전표·수주)에 조직 차원이 있는지 확인 → ⓑ 없으면 매핑 규칙(담당자→본부)으로 **매출/수주부터** 본부 스코프 데모 → ⓒ 리포트/쿼리에 사용자 본부 필터 주입.
- **전제/리스크**: 데이터 모델에 조직 차원이 없으면 손익(전표) 스코프는 제한적. 매핑 기반 근사부터.

### 과제 ③ 상용 AI 코파일럿 아키텍처 정리 — 규모: 소(대부분 문서/정합)
- **무엇/왜**: 우리가 만든 것을 상용 코파일럿 구조에 맞춰 **정리·문서화**하고 빈 곳을 메움.
- **실무 요소**: 스킬 레지스트리(툴 카탈로그), on-behalf-of 인증 표준, 불변 감사 로그, PII 마스킹, 프롬프트 인젝션 방어, 데이터 거버넌스.
- **우리 현황**: 대부분의 뼈대는 있음. **거버넌스(불변 로그·PII·인젝션 방어)와 스킬 레지스트리 형식화가 빈 곳**.
- **할 일**: 현재 아키텍처를 상용 대비표로 정리 + 최소 보강(예: 로그 불변화, 민감필드 마스킹, 툴 목록 형식화).

## 3. 추천 진행 순서

**한 번에 몰아서 하지 않고 하나씩 만들고 검증**한다.

1. **② 직무분리(SoD)** — 기존 결재 엔진 위라 빠르게 성과. (먼저)
2. **① 데이터 레벨 보안** — 데이터 모델 확인이 필요, 실무 체감 최대. (중간)
3. **③ 코파일럿 아키텍처 정리** — 앞의 결과를 반영해 마지막 정리.
