## 1. ERP를 5분 안에 이해하는 법

**ERP = Enterprise Resource Planning = 전사적 자원 관리**

한 줄 정의: **"회사를 운영하는 데 필요한 모든 업무 데이터를 한 시스템에서 관리하는 소프트웨어"**

### 회사가 돈을 버는 일반적 흐름 (이게 ERP의 척추)

```
1. 고객이 주문         (영업 = SD)
        ↓
2. 창고에서 물건 꺼냄  (자재 = MM)
        ↓
3. 부족하면 만든다     (생산 = PP)
   또는 사온다         (구매 = MM)
        ↓
4. 고객에게 배송       (영업 = SD)
        ↓
5. 인보이스 → 입금     (회계 = FI)
        ↓
6. 직원에게 월급       (인사 = HR)
```

이 6가지 활동이 한 시스템 안에서 데이터로 연결되어 흐르는 것 — 이게 ERP의 본질.
모듈을 따로따로 만들면 그건 ERP가 아니라 그냥 관리 시스템 6개의 묶음.

## 2. 학습용 가상 회사 — 현우전자

| 항목 | 설정 |
|------|------|
| 업종 | 노트북 + 모니터 제조업 |
| 판매 형태 | B2B (기업 고객 대상) |
| 부서 | 영업 / 구매 / 생산 / 재무 / 인사 / 공장 |
| 시스템 | ERP (본사 사무직) + MES (공장 현장) — 별도 두 시스템 |

**왜 이런 시나리오인가?**
- 노트북·모니터는 BOM(부품 구성)이 적절히 복잡해서 PP(생산) 학습에 좋음
- B2B라 신용한도/계약/대량 거래 학습 가능 (B2C 결제 복잡도는 회피)
- 한국 IT 종사자에게 직관적

## 3. 마스터 데이터 vs 트랜잭션 데이터 ⭐ 첫 번째 큰 산

ERP 학습의 가장 첫 번째 개념. 이걸 이해 못하면 이후 모든 게 흐릿해진다.

| 구분 | 마스터 데이터 | 트랜잭션 데이터 |
|------|---------------|------------------|
| **의미** | 변하지 않는 기준 정보 | 매일 발생하는 거래 |
| **예시** | 고객, 상품, 직원, 계정과목, 창고 | 주문, 출하, 입고, 회계전표, 급여 |
| **변경 빈도** | 거의 안 바뀜 | 끊임없이 생성됨 |
| **식별자** | 코드 (예: `CUST-2026-0001`) | 일련번호 (예: `SO-20260506-001`) |
| **삭제** | 거의 못 함 → Soft Delete | 일반적으로 안 지움 (수정만) |
| **만드는 순서** | **먼저** | 나중 |

### 왜 이 구분이 중요한가?

1. **모든 트랜잭션은 마스터를 참조한다**
   주문(트랜잭션)에는 반드시 고객(마스터), 상품(마스터)이 들어간다. 마스터가 없으면 트랜잭션을 만들 수 없다.
   → 그래서 학습 순서도 **Phase 1(마스터) → Phase 2(트랜잭션)** 다.

2. **마스터를 함부로 바꾸거나 지우면 과거 거래가 깨진다**
   고객을 지웠는데 그 고객의 작년 매출 기록이 살아있으면? "고객 없는 매출"이 되어버림.
   → 그래서 ERP는 Soft Delete + 변경 이력 관리가 필수.

3. **마스터가 곧 회사 자산이다**
   고객 1만 명을 잘못 다뤄서 데이터를 잃으면 회사가 무너질 수 있다.

## 4. 기간계 시스템의 5대 특성 ⭐ 두 번째 큰 산

ERP는 **기간계** = 회사의 핵심 업무를 처리하는 시스템.
일반 IT 서비스(블로그, SNS, 쇼핑몰 프론트)와 본질적으로 다르다.

### 4-1. 데이터 무결성이 절대적
1원이라도 틀리면 안 된다. 회계는 정확히 0원으로 맞아떨어져야 한다.
→ 트랜잭션, 외래키 제약, 동시성 제어가 디폴트.

### 4-2. 트랜잭션이 길고 복잡
한 거래(예: 출하)가 여러 모듈에 걸친다 (재고 차감 + 매출 인식 + 채권 생성).
→ `@Transactional` 경계 설계가 매우 중요.

### 4-3. 감사 가능(Auditable) 해야 함
"누가 언제 뭘 바꿨는가"가 모두 추적 가능해야 한다 (법적 요구사항).
→ 모든 테이블에 `created_by, created_at, updated_by, updated_at` + 변경 이력 테이블.

### 4-4. 배치 처리가 핵심
실시간 처리 + **야간 일괄 처리(Batch)** 가 함께 돌아간다.
예: 매일 자정 매출 마감, 월말 결산.
→ Spring Batch 학습이 후반부에 등장하는 이유.

### 4-5. 사용자 = 회사 직원
일반 인터넷 서비스와 달리, 사용자가 한정되어 있고 권한 분리가 핵심.
영업팀은 SD만, 재무팀은 FI만 보여야 한다.

## 5. Phase 0에서 셋업할 도구들 — 각각이 왜 필요한가

각 도구를 "있어서 쓰는" 게 아니라 "왜 필요해서 쓰는지" 이해하는 게 학습의 시작.

### 🐳 Docker + docker-compose
- ERP는 인프라가 많다: MySQL + (후반부) Kafka, Redis, Zookeeper, Zipkin
- OS에 직접 설치하면 학습 끝나고 정리가 어렵고 버전 충돌 가능
- Docker로 띄우면 `docker compose up` 한 번으로 모든 인프라 시작/정지

### 🐬 MySQL 8
- 사용자 지정. 한국 SI 표준
- InnoDB 엔진: 트랜잭션 + 외래키 + Row-level Lock 지원 → 기간계 필수
- 후반부에 동시성 학습(Phase 3) 시 진짜 락이 걸리는 걸 봐야 함

### ☕ Java 21 + Spring Boot 3.3
- Java 21: LTS 최신. record(불변 DTO), pattern matching, 가상 스레드 활용 가능
- Spring Boot 3.3: Jakarta EE 9+ 기반, 최신 Spring 6, JdbcClient/RestClient 같은 신규 API 활용

### 📦 Spring Data JPA + Hibernate
- ERP의 엔티티 관계는 복잡 (고객 1:N 주문 1:N 라인 N:1 상품 ...)
- 객체 그래프로 표현하면 SQL 직접 쓰는 것보다 직관적
- "왜 ORM인가?" → 마스터 + 트랜잭션 데이터의 연관관계 탐색이 매우 빈번하기 때문

### 🚀 Flyway
- DB 스키마를 SQL 파일로 버전 관리 (Git에 함께 들어감)
- "팀원 PC에서 스키마 어긋나는 사고" 방지
- ERP는 스키마 변경이 매우 잦음 (요구사항 추가가 끝없음)

### 📖 Swagger UI (springdoc-openapi)
- 화면 없이 API 테스트 가능
- 우리는 Phase 6까지 일부러 화면을 안 만든다 (도메인 학습에 집중)
- → Swagger UI가 그 동안 "임시 화면" 역할

### 🧪 Testcontainers (통합 테스트용)
- 통합 테스트 시 진짜 MySQL 컨테이너를 띄워서 검증
- H2 같은 인메모리 DB로 테스트하면 SQL 방언 차이로 운영에서 깨짐
- 기간계는 "운영과 동일 환경에서 테스트" 가 디폴트

### 🔍 QueryDSL (Phase 2부터 본격 사용)
- ERP는 다중 필터 조회/리포트가 압도적으로 많다
- JPQL 문자열로는 한계 (오타 컴파일 타임에 못 잡음)
- → 컴파일 타임에 안전한 동적 쿼리 도구

### 🏗️ Lombok + MapStruct
- Lombok: getter/setter/생성자 보일러플레이트 제거
- MapStruct: 엔티티 ↔ DTO 변환을 컴파일 타임에 코드 생성 (런타임 리플렉션 X)
