# 서버 구성 — Oracle Cloud Always Free 배포 검토

> **이 글의 목적**: ERP/MES 포트폴리오를 **외부에 공개**하기 위한 호스팅 방식을 검토한 기록.
> 결론은 **집 서버 대신 Oracle Cloud Always Free(무료)에 배포 + Cloudflare 연결**.

---

## 0. 왜 집 서버가 아니라 클라우드인가

- 포트폴리오는 **불특정 다수에게 공개**되는 서비스 → 봇·악성 이용자가 반드시 들어온다.
- 집 서버의 진짜 위험은 **IP 노출이 아니라** "앱이 뚫리면 집 내부망(다른 PC·NAS·공유기)까지 뚫린다"는 점.
  - 참고: 공인 IP로 알 수 있는 건 통신사·대략적 도시 수준뿐. **집 주소·물리적 위치는 특정 불가.**
- 클라우드에 올리면 **집 네트워크와 완전히 분리** → 서버가 털려도 집은 무관.
- 채용담당자 입장에서도 "클라우드 배포"가 더 프로다워 보이고, **배포 경험 자체가 포트폴리오 가점.**

---

## 1. Oracle Cloud Always Free 사양

핵심은 잘 알려지지 않은 **ARM(Ampere A1) 무료 인스턴스**.

| 항목 | 무료 제공량 |
|---|---|
| ARM CPU/RAM | **4 OCPU + 24GB RAM** |
| 블록 스토리지 | **200GB** |
| VM 구성 | 최대 4대로 분할 또는 **한 대에 몰아서** 사용 |
| 기간 | **평생 무료 (12개월 제한 아님), 24/7 상시 가동** |

→ **ERP + MES + MySQL + Docker 를 한 대에서 돌리고도 여유가 크다.**

예상 리소스 사용:

| 컨테이너 | RAM |
|---|---|
| ERP (Spring Boot 가정) | ~1GB |
| MES | ~1GB |
| MySQL | ~1GB |
| nginx/기타 | ~0.5GB |
| **합계** | **~4GB / 24GB** |

## 2. "평생 무료 24시간" 정확히 이해하기

Oracle 무료는 **두 종류**이며 혼동 주의.

- **① Always Free** — ARM 4코어/24GB, 스토리지 200GB 등.
  - **기간 제한 없음(평생), 24/7 상시 가동.** (Fly.io처럼 안 쓰면 잠드는 방식 아님)
  - **← 우리가 목표하는 것.**
- **② $300 / 30일 무료 크레딧 (Free Trial)** — 가입 시 딸려오는 체험 크레딧.
  - 30일 지나거나 소진되면 끝. **이게 끝나도 Always Free는 계속 유지됨.**

### 유휴 회수 정책 주의
- Oracle은 **완전히 놀고 있는(idle, CPU 장기간 ~0%) Always Free 인스턴스**를 회수할 수 있음.
- 포트폴리오 서버는 상시 앱이 떠 있어 **해당 안 됨.**
- 100% 안전하게 하려면 → 계정을 **Pay As You Go로 업그레이드**.
  - 카드만 등록, **Always Free 범위 내 사용 시 요금 0원.**
  - 이렇게 하면 **유휴 회수 대상에서 완전 제외** → 정석 세팅.

---

## 3. 아키텍처 주의사항 (ARM)

- 인스턴스가 **ARM(aarch64)** 아키텍처.
- MySQL, OpenJDK, Node, nginx, Postgres 등 **유명 이미지는 ARM 멀티아치 지원** → 그대로 동작.
- 직접 빌드하는 앱 이미지는 **ARM 타깃으로 빌드**하거나 **그 서버 위에서 빌드**하면 됨.
- Java/Kotlin은 아키텍처 무관이라 신경 쓸 게 거의 없음.
- **인스턴스 생성이 "out of capacity"로 막히는 경우가 잦음** (인기 리전은 무료 ARM 물량 부족).
  - 해결: 덜 붐비는 리전 선택, 또는 생성 재시도(스크립트). 일단 잡히면 계속 사용 가능.

## 4. 도메인 연결

Always Free 인스턴스도 공인 IP가 있어 **도메인 연결 가능.** 방식은 일반 서버와 동일.

### 기본 흐름
1. ARM 인스턴스 생성 → **공인 IP** 부여
2. 도메인 등록처에서 **A 레코드 → 공인 IP** 설정
3. `내도메인.com` → 서버 연결 완료

### Oracle에서 꼭 챙길 3가지 (여기서 다들 막힘)
1. **공인 IP를 Reserved(고정)로 전환**
   - 기본은 Ephemeral(임시)라 재부팅·재생성 시 바뀔 수 있음.
   - 콘솔에서 **Reserved Public IP**로 전환 (Always Free 범위 내 무료).
2. **Security List(클라우드 방화벽) 포트 열기**
   - 기본적으로 80/443 막힘 → VCN → Security List에 **Ingress 80, 443 허용** 추가.
3. **인스턴스 내부 iptables — 대표적 함정**
   - Oracle 리눅스 이미지는 OS 안에도 iptables로 80/443을 막음.
   - Security List만 열고 "왜 안 되지?" 하는 대표 원인. **서버 안에서도 80/443 허용** 필요.

## 5. 최종 추천 구성

```
도메인 → Cloudflare(무료) → Oracle ARM VM (Reserved IP)
                              └ 80/443 열기 (Security List + iptables)
                              └ Docker: [nginx] [ERP] [MES] [MySQL]
```

- **Cloudflare를 앞에 두면**: A 레코드를 Cloudflare로 넣고 Cloudflare가 Oracle 실서버 IP를 가림.
  - **실서버 IP 숨김 + 무료 HTTPS + DDoS 방어 + WAF** 무료.
  - 도메인 등록처의 네임서버만 Cloudflare로 변경 → Cloudflare에서 A 레코드 관리.
- 집 IP·집 네트워크 **완전 무관** → 앞서 우려한 위험 원천 제거.
- 회사 RDS/EC2와도 분리 → 눈치 볼 일 없음.
- **비용 0원.**

### 포트폴리오라서 추가로 챙길 것 (호스팅 무관)
- **실제 개인정보 넣지 말 것** — 데모용 가짜 데이터만.
- 관리자 계정·기본 비밀번호 제거, DB 외부 포트 노출 금지.
- 가능하면 **읽기 위주 데모** (회원가입/글쓰기 무제한 열면 봇 스팸밭).
- HTTPS 필수, 의존성·OS 상시 업데이트.

## 6. AI 기능(Phase 17) 배포 검토

Phase 17 = **자연어 데이터 검색(Text-to-SQL)**. 사용자가 한국어로 물으면 LLM이 SQL을 만들어 실행·응답. 원래 계획은 **로컬 Ollama + `qwen2.5-coder:7b`**(M2 Pro 16GB 기준). 이걸 클라우드에 올릴 때의 검토.

### 6.1 결정적 제약 — Always Free엔 GPU가 없다

- Always Free의 쓸 만한 리소스는 **ARM 4 OCPU + 24GB RAM**뿐, **GPU 없음**(GPU 인스턴스는 유료).
- **메모리는 여유** — 7B 양자화 모델 ~5GB. ERP+MES+MySQL(~4GB)에 얹어도 24GB 안에 들어옴.
- **문제는 속도** — CPU-only ARM에서 7B는 대략 **2~5 tok/s** → Text-to-SQL 한 건에 **30초~2분**. "느리지만 돌아는 간다" 수준.
  - M2 Pro가 견딜 만했던 건 GPU + 높은 메모리 대역폭 덕분. Ampere A1엔 둘 다 없음.

### 6.2 선택지 3가지

| 방식 | 구동 | 특징 |
|---|---|---|
| **① 전부 클라우드 + 작은 모델(3B/1.5B)** ⭐추천 | 됨 | 로컬 LLM 학습 취지 유지 + CPU에서도 몇 초. 정확도 하락은 가드레일(17-C)로 보완 |
| **② 전부 클라우드 + 7B 유지** | 됨 | 학습·데모용이면 느린 속도 감수 가능. 가드레일 학습엔 속도 무관 |
| **③ 클라우드 API 전환(Claude 등)** | 됨(쾌적) | 리소스 부담 0·빠름. 단 "로컬·무비용·무유출"이라는 Phase 17 취지가 바뀜 |

→ **AI까지 전부 Oracle Cloud 한 대에 올리는 게 가장 단순·안전.** 속도는 모델 크기로 조절.

### 6.3 요청 흐름 — 안전한 설계

```
사용자 → 웹서버(ERP) : "지난달 매출 top10" 호출
웹서버 → AI(Ollama)  : 질문 텍스트만 전달        ← DB 인증키 안 보냄
AI    → 웹서버        : "SELECT ... LIMIT 10" (SQL 문자열만 반환)  ← DB 접근 0
웹서버               : SQL 검증(가드레일) → 읽기전용 계정으로 DB 실행
웹서버 → 사용자        : 결과 표시
```

- **핵심 원칙: AI는 "번역기"일 뿐, DB는 안 만진다.** AI는 한국어→SQL 텍스트만 반환하고, **실제 DB 실행은 웹서버가** 한다.
  - DB 인증키가 AI 쪽으로 흐르지 않음 → 인증키가 신뢰 영역(웹서버) 밖으로 안 나감.
  - 가드레일 4종(JSqlParser 검증·읽기전용 계정 `erp_ro`/`mes_ro`·강제 LIMIT·타임아웃)이 **전부 웹서버 한 곳**에서 돎 = Phase 17 학습 핵심(17-B).
- 전부 한 인스턴스 안이면 웹서버→AI 호출이 **localhost 내부 통신** → 인터넷을 안 탐, 노출 원천 없음.

### 6.4 (참고) AI만 집 PC에 두는 하이브리드

성능을 위해 AI만 집 M2 Pro에 두는 안도 가능하나 **필수 아님**.

- 걱정거리인 "집 노출"의 핵심은 **집이 문을 열고 밖의 접속을 기다리는 것(웹서버화)**. 그게 아니면 안 위험.
- 집이 **밖으로 나가는(outbound) 연결만** 하면 포트를 안 열어 노출 없음. (지금 개발 중 집→AWS DB 접속이 바로 이 안전한 방향. 정상.)
- 그래도 "웹서버→AI" 방향(클라우드가 집을 부름)이 필요하면 → **Tailscale/WireGuard 사설 터널**로 묶으면 공용 포트 없이 내부처럼 통신. 집 노출 0.
- 요약: **집을 서비스 서버로 공개하지만 않으면**(6.3처럼 전부 클라우드거나, 하이브리드라도 outbound/터널) **집 IP 노출 위험은 생기지 않는다.**
