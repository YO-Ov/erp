# 전자결재(Approval) 설계 — 전사 횡단 워크플로

> "실무 리얼리즘 확장" 프로젝트의 **결재 기능** 설계 문서. (마스터 데이터=STEP 4 와 별개의 신규 기능)
> hwlee님 지적: "결재는 견적만 필요한 게 아니라 다른 메뉴들도 필요하다" → 맞음. 실무 전자결재는 **여러 문서 공통**.

## 1. 배경 — 결재는 횡단 관심사

실무에서 전자결재(전자문서 결재)는 특정 화면 하나가 아니라, **여러 업무 문서가 공통으로 타는 승인 절차**다.
- 보통 **조건부**로 결재를 탄다: 금액 구간, 표준 할인율 초과, 비표준 결제조건, 신용도 낮은 고객, 신규 거래처 등.
- **결재선(라인)**: 담당 작성(상신) → 팀장 → (큰 건) 본부장 → 대표. **금액 구간별 결재권자**가 보통.
- 표준·소액 건은 1단계(팀장)만, 또는 결재 없이 담당자 처리.

## 2. 우리 ERP의 결재 대상 문서 (모듈별)

| 모듈 | 문서 | 결재 트리거(예) |
|---|---|---|
| SD 영업 | **견적(Quotation)** 발송 | 금액·특가·비표준 결제조건 |
| SD 영업 | 수주(SalesOrder) 확정 | 대형 수주·신용 초과 |
| MM 구매 | **구매발주/입고** | 금액 구간 |
| MM 구매 | 거래처(Vendor) 신규 등록 | 항상/대형 |
| FI 재무 | **지급결의(Payment)** | 금액 구간 |
| FI 재무 | 전표(Journal) 전기 | 수동 전표·고액 |
| FI 재무 | 여신 상향 *(이미 구현됨)* | 항상 |
| HR 인사 | 근태/연장근로·휴가 | 항상 |
| HR 인사 | 급여대장 확정 | 항상 |
| PP 생산 | 생산지시 발행 | 대형 |
| PP 생산 | 계획오더 승인 *(이미 구현됨)* | 항상 |
| 마스터 | 고객/품목 신규 등록 | 정책에 따라 |

> **기존에 이미 "승인" 흐름이 있는 것**: 여신 상향 요청(`fi.credit.CreditLimitRequest` submit→approve, 알림 연동), 계획오더(`pp.planning.PlannedOrder` PROPOSED→승인→전환). → 범용 엔진으로 **통합/일원화** 가능(나중에 마이그레이션).

## 3. 설계 방향 — 범용 전자결재 엔진 (권장)

문서마다 개별 승인 필드를 다는 대신, **어떤 문서든 붙일 수 있는 재사용 결재 모듈**을 만든다(포트폴리오에서 돋보임).

### 핵심 모델
- **`approval_request`** (결재 문서 한 건)
  - `id`, `doc_type`(QUOTATION/SALES_ORDER/PURCHASE_ORDER/PAYMENT/JOURNAL/…), `ref_id`/`ref_no`(원본 문서 참조), `title`, `amount`,
  - `requester`(상신자 username), `status`(DRAFT/PENDING/APPROVED/REJECTED/WITHDRAWN), `current_step`, `requested_at`/`decided_at`.
- **`approval_step`** (결재선의 각 단계)
  - `request_id`, `step_no`(순차), `approver`(username 또는 role/position), `status`(PENDING/APPROVED/REJECTED/SKIPPED), `decided_at`, `comment`.
  - **순차 처리**: step 1 승인돼야 step 2 차례. 한 단계라도 반려면 전체 REJECTED.
- **`approval_rule`** (결재선 결정 규칙, 선택)
  - `doc_type`, `min_amount`/`max_amount`, `step_no`, `approver_role`/`position` → 금액 구간별 결재선 자동 구성.

### 상태머신
```
DRAFT → (상신) PENDING → (각 단계 승인) … → APPROVED → [원본 문서 진행]
                       ↘ (반려) REJECTED
        (상신 취소) WITHDRAWN
```

### 연동
- **알림(Notification, 이미 있음)**: 상신 시 1차 결재자에게, 각 단계 승인 시 다음 결재자에게, 최종 승인/반려 시 상신자에게. 딥링크.
- **문서 콜백/이벤트**: 최종 APPROVED 시 원본 문서 상태 전이(예: 견적 SENT, 구매발주 ISSUED). Spring Event 또는 서비스 콜백으로 느슨하게.
- **직급/결재권자**: 결재 단계는 팀장(MANAGER)/본부장(DIRECTOR)에 매핑. 직원 직급은 `employment_contract.position`. → **STEP 4에서 깔아둔 부서별 "담당+팀장" 로그인**이 상신자/결재자가 됨.

### 대안 (비권장)
- 문서별 개별 승인 필드(여신처럼 각각): 빠르지만 중복·비일관. 범용 엔진이 학습/포트폴리오 가치 큼.

## 4. 단계적 도입 계획 (제안)

큰 기능이라 한 번에 다 하지 않고 쪼갠다:
1. **결재 엔진 코어** + **견적(Quotation)** 1종 적용 (상신→승인→발송, 반려, 알림).
2. **구매발주/입고** 적용.
3. **지급결의(Payment)·전표 전기** 적용.
4. **HR(근태/휴가/급여 확정)·생산지시** 적용.
5. 기존 **여신·계획오더**를 범용 엔진으로 통합(선택).
6. 결재 화면: 상신함/결재함/문서별 결재 진행 타임라인(여신 타임라인 패턴 재사용).

## 5. 미결정 사항 (다음 세션에 정할 것)

1. **범위**: 위 대상 문서 중 어디까지 결재 적용? (최소=견적+구매발주+지급, 최대=전 모듈)
2. **엔진 방식**: 범용 엔진(권장) vs 문서별 개별.
3. **결재선 규칙**: 금액 구간 자동 결재선 vs 고정 결재선(담당→팀장→본부장).
4. **타이밍**: 데이터(STEP 4~5) 먼저 채우고 결재를 그 위에 올리기(권장) vs 먼저.
5. 기존 여신/계획오더 통합 여부.
