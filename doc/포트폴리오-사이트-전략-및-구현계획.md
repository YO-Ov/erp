## 0. 이 문서를 쓴 이유 — 목표가 바뀌었다

최초 포트폴리오는 "개발자 개인 사이트"로 만들었다. 하지만 실제 목적이 명확해졌다:

> **제조업(공장) IT 직군 경력직 취업.**

목적이 정해지면 콘텐츠 우선순위가 달라진다. 지금 사이트는 "개인 프로젝트 3개"가 주인공이고 실무 경력은 타임라인 한 줄씩인데, **이 비중이 목표와 어긋난다**(→ §3).

## 1. 상황 진단 (사실 기반)

### 1-1. 지원자 조건

| 항목 | 내용 |
| --- | --- |
| 경력 | 10년 9개월 (2015.12~), 백엔드/풀스택 |
| 도메인 | 공공기관 단속 시스템 · 항공 ERP · 물류배송 · 시설예약 플랫폼 |
| 목표 직군 | 제조업 사내 IT / ERP·MES 운영·개발 |
| 본인이 느끼는 약점 | ⓐ ERP·MES 경력 없음 ⓑ 나이 있는 경력직 ⓒ 경력이 "웹개발·유지보수"라 경쟁력 부족 ⓓ 유지보수 사이트 대부분 비공개·품질 낮음 |

### 1-2. 약점 진단 — 3개 중 2개는 사실이 아니다

**ⓐ "ERP 경력이 없다" → 틀렸다.**
`우정항공 시스템개발팀(2022.05–2023.07)` = **ERP 커스터마이징 · 쿼리 최적화 · 서버 안정화 + 삼성SDS 첼로스퀘어 연동 인터페이스**.
이건 명백한 ERP 실무 경력이고, 심지어 **외부 물류 플랫폼 연동**까지 했다. 현재 사이트에서 한 줄로 흘려보내는 중.
→ **가장 크게 키워야 할 항목.** (MES 경력이 없는 건 사실. 그건 개인 프로젝트로 커버)

**ⓒ "웹개발·유지보수라 경쟁력 없다" → 재프레이밍이 필요할 뿐, 오히려 강점.**
제조업 사내 IT의 실제 업무 = **레거시 유지보수 + 현업 요구 반영 + 벤더/SI 관리 + 안정적 운영**. 신규 개발 비중은 작다.
- 공공기관 단속 시스템 유지보수(엑스아이티) = 레거시·장기운영·보안지침(KISA) 준수 → 공장 시스템과 성격 동일
- 전 경력이 **"현업이 매일 쓰는 업무 시스템"** → B2C 서비스 개발자보다 제조업 적합도가 높다
→ "유지보수를 했다"가 아니라 **"멈추면 안 되는 시스템을 오래 책임졌다"** 로 서술한다.

**ⓑ 나이/경력직 → 이건 실재하는 허들.**
채용 측의 진짜 의심은 "새 도메인을 배울 의지가 있나 / 와서 오래 있을까"다.
→ **ERP·MES 개인 프로젝트의 진짜 역할은 기술 자랑이 아니라 이 의심에 대한 답이다.**
   "경력 없음을 메우는 증빙"이 아니라 **"모르는 도메인을 스스로 학습해 끝까지 완성한 증거"** 로 포지셔닝한다.

### 1-3. 결론 — 이 사이트가 해야 할 일

> **"업무 시스템을 10년 책임져 온 사람이, 제조 도메인을 스스로 학습해 실제로 만들어냈다."**

개인 프로젝트와 실무 경력이 **각자 다른 의심에 답한다**:

| 채용 측 의심 | 답하는 콘텐츠 |
| --- | --- |
| "ERP/MES 도메인을 아나?" | 개인 프로젝트 (ERP·MES·AI) |
| "우리 레거시 운영·개선을 맡길 수 있나?" | 실무 경력 10년 (특히 우정항공 ERP · 공공 유지보수) |
| "새 도메인 배울 의지가 있나? 오래 있을까?" | 개인 프로젝트를 **왜·어떻게** 만들었는지의 서술 |

## 2. 핵심 결정 — 유지보수하던 사이트를 넣을 것인가

### 결정: **넣는다. 단 "프로젝트 링크 카드"가 아니라 "경력 서술"로 넣는다.**

**링크 카드로 넣지 않는 이유**
1. 품질이 낮은 사이트를 링크로 걸면 평가 축이 **"결과물의 디자인 품질"** 로 이동한다. 이길 수 없는 싸움이고, 애초에 본인이 디자인한 것도 아니다.
2. 접속 안 되는 링크(비공개)는 **신뢰만 깎는다.** "링크 걸어놨는데 안 열림" = 검증 안 된 사이트라는 인상.
3. 이미 **품질로 승부할 카드(ERP·MES)가 따로 있다.** 저품질 카드를 옆에 두면 좋은 카드까지 평균으로 끌어내린다.

**대신 이렇게 넣는다**
- **"비공개 (사내 시스템)" 라벨을 당당히 붙인다.** 업무 시스템 개발자에게 비공개는 정상이고, 오히려 **엔터프라이즈 경험의 신호**다. 변명하듯 쓰지 않는다.
- 서술 축을 **결과물 → 책임·문제해결**로 바꾼다:
  `무엇을 책임졌는가` / `규모(사용자·데이터·기간)` / `어떤 문제를 어떻게 해결했는가` / `사용 기술`
- **예외**: 접속 가능하고 보여줄 만한 것 **1~2개만** 선별해 링크. (나머지는 링크 없이 서술)

**선별 기준** (hwlee님이 판단할 것)
- ✅ 넣기: 지금도 접속되고 · 본인 기여가 명확하고 · 제조/업무 시스템 성격
- ❌ 빼기: 접속 불가 · 디자인이 본인 손이 아님 · 단순 퍼블리싱 · 기여 설명이 어려움

> ⚠️ **NDA·계약 확인**: 고객사명·화면 캡처는 계약상 공개 불가일 수 있다. **회사명은 이미 이력서에 있으니 그대로 쓰되, 화면 캡처와 내부 수치는 넣지 않는다.** 애매하면 "물류 대기업 A사" 식으로.

## 3. 페이지 구조 개편안

### 현재 구조 (문제)

```
Hero → 01 이런 걸 합니다 → 02 개인 프로젝트(주인공) → 03 기술 스택 → 04 실무 경력(타임라인 한 줄씩) → 연락
```

**문제점**
1. 실무 경력 10년이 **맨 아래 + 한 줄 요약** = 가장 큰 자산을 가장 약하게 배치
2. **ERP 경력(우정항공)이 안 보인다** — 제조업 지원에서 가장 중요한 한 줄인데 5개 중 3번째에 묻힘
3. "왜 ERP·MES를 만들었는가"가 없다 → 취미 프로젝트로 읽힐 위험
4. 제조업 타깃 신호가 전혀 없음 — 누가 봐도 범용 개발자 사이트
5. 🐞 **푸터에 시안 문구가 그대로 남아 있음** (`index.html:485` "시안(mockup) — 콘텐츠는 자유롭게 교체하세요") → 미완성으로 읽힘. **최우선 수정**

### 개편 구조

```
Hero            ← 제조업 타깃 한 줄 + 핵심 숫자
01 왜 ERP·MES를 만들었나   ← 신설. 도메인 학습 서사 (ⓑ 의심에 답)
02 개인 프로젝트          ← 유지 + 실제 스크린샷 + 기술 하이라이트
03 실무 경력              ← 대폭 강화. ERP 경력 최상단 강조 + 비공개 시스템 서술
04 기술 스택              ← 유지 + 레거시 스택 추가(제조업은 레거시가 많다)
05 일하는 방식            ← 신설. 검증 습관 (짧게)
연락                     ← 이력서 CTA 포함
```

**변경 요약**
| 변경 | 내용 | 근거 |
| --- | --- | --- |
| 신설 | `01 왜 만들었나` | 개인 프로젝트를 "학습 서사"로 프레이밍 (§1-2 ⓑ) |
| 승격 | 실무 경력 04 → 03 | 10년이 가장 큰 자산 (§1-3) |
| 강화 | 우정항공 ERP를 **별도 강조 블록**으로 | "ERP 경력 없음"은 사실이 아님 (§1-2 ⓐ) |
| 추가 | 경력 항목에 비공개 시스템 서술 | §2 결정 |
| 추가 | 스택에 레거시(JSP·Oracle·MSSQL·전자정부 등) | 제조업 현장은 레거시 비중이 큼 — 오히려 가점 |
| 신설 | `05 일하는 방식` | 자소서 "직접 확인 안 하면 안 믿는다" 검증 습관 |
| 수정 | 푸터 시안 문구 제거 | 🐞 버그 |

## 4. 섹션별 작성 지침

### Hero — 타깃을 명시한다

- 현재: "업무 시스템을 설계하고 만드는 개발자" (범용)
- 방향: **제조·물류 업무 시스템** 을 명시. 읽는 사람이 "우리 쪽 사람이네"라고 느끼게.
- 핵심 숫자 3개를 칩으로: `경력 10년+` · `업무 시스템 5개 도메인` · `ERP·MES 직접 설계·구현`
- ⚠️ 비용 40% 절감은 Hero가 아니라 경력 항목 안에 둔다(맥락 없이 쓰면 과장으로 읽힘)

### 01 왜 ERP·MES를 만들었나 (신설, 3~4문장)

**톤**: 겸손하되 변명하지 않는다. "경력이 없어서 만들었다"고 쓰지 말 것 — 약점을 자백하는 문장이 된다.

담을 것:
- 업무 시스템을 10년 만들다 보니 **그 위쪽(전사 자원관리·생산 실행)이 궁금해졌다**
- 책으로 읽는 대신 **직접 설계하고 만들어서 배포**했다 — 복식부기, BOM 전개, 상태 전이, 결재선을 코드로
- 그 과정에서 **도메인 규칙을 불변식으로 지키는 법**을 익혔다

### 02 개인 프로젝트 (유지 + 보강)

- ⓐ **CSS 목업 프리뷰 → 실제 스크린샷 교체** (임팩트 가장 큼)
- ⓑ 카드마다 **"기술 하이라이트" 1~2줄** — 소재는 `PROGRESS.md` 에 풍부:
  - **ERP**: 복식부기 차대 균형을 불변식으로 → 불일치 시 `422 UNBALANCED_JOURNAL`
  - **ERP**: 전자결재 승인 → 원본 문서 자동 전기/전이 (업무 흐름이 화면에서 닫힘)
  - **MES**: TypeScript 전환이 **런타임 버그를 컴파일 단계에서 검출** (실적 등록 API 반환 타입 불일치)
  - **연동**: ERP 작업지시 → **Kafka** → MES 수신 (시스템 간 비동기 연동)
  - **AI**: 로컬 LLM(Ollama) 자연어 조회 + **역할 권한(RBAC)을 챗봇에도 그대로 적용**
- ⓒ **MES 카드에 "제조 실행" 성격을 더 강조** — 작업지시·설비 가동률·품질검사·BOM. 제조업 담당자가 아는 단어를 쓴다.

### 03 실무 경력 (대폭 강화 — 이번 작업의 핵심)

각 항목을 **한 줄 요약 → 3~4줄 구조**로 확장:

```
[회사 · 직위 · 기간]
역할:    무엇을 책임졌는가
규모:    사용자 / 데이터량 / 운영 기간
문제해결: 어떤 문제를 어떻게 (정량 가능하면 숫자)
기술:    스택 태그
공개여부: 라이브 링크 or "비공개 (사내 시스템)"
```

**우선순위**
1. **우정항공 (ERP)** — 최상단 강조. `ERP 커스터마이징 / 쿼리 최적화 / 서버 안정화 / 삼성SDS 첼로스퀘어 연동`. 제조업 지원에서 **가장 중요한 항목**. 별도 배지(`ERP 실무`)를 달아 눈에 띄게.
2. **엑스아이티 (공공 유지보수)** — "레거시 장기 운영 + KISA 보안 지침 준수". 제조업 사내 IT 업무와 성격이 같음을 드러낸다.
3. **딱플 (현재)** — Spring Boot·배치·CI/CD·React Native + **AWS 비용 40% 절감**(자동화/최적화 역량).
4. 위두쉽·웰텍 — 업무 시스템 운영 + 외부 연동 API.

> ✍️ **작성 시 금칙어**: "단순", "보조", "참여" → 기여를 축소하는 단어. "담당", "설계", "개선", "책임" 으로.

### 04 기술 스택 (레거시 추가)

현재는 최신 스택만 있다. **제조업 현장은 레거시가 많아 이게 오히려 감점**이 될 수 있다("우리 시스템 못 다루겠네").
→ `legacy/` 카테고리 신설: JSP · Oracle · MSSQL · jQuery · 전자정부 프레임워크 등 **실제 다뤄본 것만**.

### 05 일하는 방식 (신설, 3항목 정도)

- **직접 확인하기 전엔 믿지 않는다** — 만든 기능은 실제 API·실제 브라우저로 끝까지 확인한다 (자소서 검증 습관)
- **규칙은 코드로 강제한다** — 차대 균형, 상태 전이, 권한을 문서가 아니라 불변식/타입으로
- **만들고 끝내지 않는다** — 배포·모니터링·비용까지
> ⚠️ **야근·시간 투입 표현은 넣지 않는다.** "문제해결/효율/자동화"로 대체(hwlee님 방침).
