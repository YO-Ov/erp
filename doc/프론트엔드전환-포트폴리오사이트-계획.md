## 1. 포트폴리오 사이트 아키텍처 결정

### 결론: 포트폴리오는 **ERP/MES/AI 중 어디에도 종속시키지 않고 독립**으로 둔다

특정 프로젝트(예: ERP) 안에 넣으면 생기는 문제:

| 문제 | 설명 |
|---|---|
| 배포 커플링 | 프로필 한 줄 고치려고 ERP 전체를 재배포 |
| 관심사 불일치 | 포트폴리오=정적 소개 / ERP=상태 있는 업무 앱 |
| 비대칭 | 3개를 동등하게 소개해야 하는데 한 곳만 "특별"해짐 |
| 도메인 오염 | `hyunwoo.pro` 루트가 ERP 앱이 되어버림 |

→ 포트폴리오는 **3개 프로젝트로 뻗는 "현관/인덱스"**. 방(ERP) 안에 현관을 둘 수 없다.

### 채택 구조 (서브도메인 분리)

```
hyunwoo.pro          → 포트폴리오 (정적, 나 소개 + 3개로 링크)
erp.hyunwoo.pro      → ERP
mes.hyunwoo.pro      → MES
agent.hyunwoo.pro    → AI 에이전트
```

- repo 개수가 부담되면 **모노레포**(`portfolio/`, `erp/`, `mes/`, `agent/` 폴더)로 물리적 위치만 한곳에 모으되 배포는 독립. (현 `my-app/` 하위 구조가 이미 이 방향)

### 프레임워크는 필요 없다 — "정적 + 외부 서비스"로 시작

**"접속자 정보 = DB = 프레임워크"는 오해.** 방문자 분석은 외부 서비스 `<script>` 한 줄로 끝:

- Cloudflare Web Analytics / Plausible / Google Analytics / Vercel Analytics → 내 쪽 백엔드·DB 0개

프레임워크(백엔드)가 진짜 필요한 순간 = **내가 직접 데이터를 소유·저장**할 때만:

| 단계 | 필요한 것 | 예시 |
|---|---|---|
| ① 순수 정적 | 없음 | HTML/CSS/JS + 외부 analytics |
| ② 정적 + 서버리스 함수 | 함수 몇 개(프레임워크 아님) | 연락 폼 처리 |
| ③ BaaS | 백엔드 코드 거의 없음 | Supabase/Firebase가 DB+API 제공 |
| ④ 풀 프레임워크 | Next.js 등 | SSR·통합 백엔드가 정말 필요할 때 |

→ **YAGNI: ①로 시작 + analytics 스크립트.** 방명록/조회수 소유 욕구가 실제로 생기면 그때 ②·③만 추가.

## 2. 포트폴리오 페이지 구성 (2026 표준 "4섹션 공식")

방문자의 3가지 질문에 답하는 구조: **누구인가 / 뭘 만들 수 있나 / 어떻게 연락하나.**

```
① Hero     — 이름 + 한 줄 정체성("ERP/MES 도메인을 파는 백엔드 개발자") + CTA 2개
② About    — 짧은 스토리, 관심 도메인(제조/ERP), 기술 스택
③ Projects — 핵심. 카드 → 클릭 시 케이스 스터디 상세
④ Contact  — 이메일 / GitHub / LinkedIn
```

### 핵심은 ③ Projects — "카드"가 아니라 "케이스 스터디"

프로젝트마다 4단 서사:
1. **문제** — 어떤 도메인 문제 (역할별 권한, 재고/생산 흐름 등)
2. **접근/설계** — 어떻게 모델링했고 왜 그 선택 ← **백엔드 실력 증명 지점**
3. **기술 스택 & 아키텍처** — 다이어그램 하나가 텍스트 열 줄보다 강력
4. **결과** — 라이브 데모 / GitHub / 배운 점

> ERP는 이미 `PROGRESS.md`, `ERP-STUDY-PLAN.md`, `doc/` 산출물이 쌓여 있어 그대로 케이스 스터디 본문으로 재활용 가능. "역할별 대시보드", "AI 어시스턴트" 등이 좋은 소제목 소재.

## 3. Vue vs React — ERP SPA 전환 관점

현재 ERP = **Thymeleaf 서버 렌더링 MVC**, 단 `@RestController` 39개로 **REST 기반이 이미 상당함** → SPA 전환에 유리.

| 기준 | Vue | React |
|---|---|---|
| Thymeleaf에서 넘어가기 | ★ 쉬움 (`v-if`/`v-for` ≈ `th:if`/`th:each`) | JSX 낯섦 |
| 학습 곡선 | 완만, 한국어 입문 자료 좋음 | 개념 많음(hooks), 대신 자료 최다 |
| 취업/생태계 | 괜찮으나 좁음 | 국내외 채용 표준, 레퍼런스 최다 |
| 폼/테이블 위주 앱 | 초반 생산성 살짝 높음 | 라이브러리 생태계 두꺼움 |

**한 줄:** 부드러운 전환·빠른 화면 → **Vue** / 이력서 가치·장기 생태계 → **React**

### ✅ 확정 (2026-07-15): 프로젝트마다 다른 프레임워크로 **둘 다 학습** (A안)

하나 고르느라 고민 X. 독립 프로젝트가 3개 + 포트폴리오가 있으니 나눠서 배운다.
목적 = **두 프레임워크 경험** (hwlee님은 React·Vue 둘 다 초심).

**배분 (확정):**
- **ERP → React** — 화면 수가 많고 복잡한 폼·대용량 그리드·권한별 분기가 핵심.
  이 영역은 React 생태계(AG Grid·TanStack Table·MUI DataGrid, React Hook Form + Zod, MUI/Ant Design,
  레퍼런스·채용 시장)가 확실히 두껍다. **양이 많은 쪽에 생태계가 두꺼운 프레임워크**를 배치.
- **MES → Vue** — 현장 모니터링·실시간 상태·설비 대시보드 성격. 화면 단위가 응집적이라 SFC 구조와 잘 맞고,
  반응성 모델이 직관적이라 러닝커브가 낮다. 규모가 작아 부담이 적다.
- **포트폴리오 → 아직 안 써본 쪽으로 복습** (선택).

### ✅ 확정: **Next.js / Nuxt 안 씀 → 순수 SPA (Vite + React, Vite + Vue)**

이유 = **백엔드(Spring REST)가 이미 있으므로.** 이 저장소는 Spring(hwlee-erp) + REST API 39개 + JWT 인증이
이미 완성돼 있어, 프론트의 역할은 **REST API를 소비하는 순수 SPA 하나뿐**이다. 이 구조에서 Next/Nuxt의
서버사이드 기능은 전부 중복·불필요하거나 오히려 인증 흐름을 꼬이게 만든다.

| Next.js/Nuxt가 주는 것 | 이 프로젝트에서 |
|---|---|
| SSR/SSG (SEO) | **불필요** — 로그인 뒤 내부 업무 화면, 검색 노출 무의미 |
| API Routes (백엔드 역할) | **중복** — 이미 Spring이 백엔드. 계층만 하나 더 생김 |
| Server Components / 서버 실행 | **오히려 혼란** — JWT 쿠키 인증과 얽히면 초심자에게 부담 |
| 풀스택 통합 | 백엔드가 이미 분리돼 있어 이점 소멸 |

Next/Nuxt를 얹으면 **"Spring 백엔드 + Next 서버 + 브라우저"** 3계층이 되어 토큰이 어디서 흐르는지부터 꼬인다.
또 초심자 학습 목적상 Next.js는 "React 그 자체"가 아니라 "Next.js라는 프레임워크"를 배우는 것에 가까워,
순수 React 감각을 익히기엔 방해가 된다. → **순수 React/Vue에 집중.**

**확정 스택:**
- ERP: **Vite + React + React Router + TanStack Query(서버 상태) + Axios**
- MES: **Vite + Vue + Vue Router + Pinia**
- 배포: 정적 빌드 산출물을 **Caddy로 서빙** (이미 사용 중), 기존 `erp.hyunwoo.pro`·`mes.hyunwoo.pro` REST API 호출.
  **백엔드는 손대지 않는다.**

### ✅ 확정: 진행 순서 = **MES(Vue) 먼저 → ERP(React) 나중**

둘 다 초심이라 동시 병행은 인지 부하가 크다. "쉬운 것 → 어려운 것" 순서로:
1. 먼저 **MES를 Vue로** — 작고 러닝커브 낮은 조합으로 프론트-백엔드 연동·라우팅·상태관리 기본기 체득.
2. 그다음 **ERP를 React로** — Vue에서 익힌 개념(컴포넌트/상태/props)을 React 문법에 매핑하며 확장.
   → 두 프레임워크의 공통 개념 vs 차이가 선명해져 학습 효과가 크다.

## 4. 전환 전략 — 통째로 갈아엎지 않는다 (점진적)

Thymeleaf ERP를 한 번에 재작성하면 규모 과다로 학습 붕괴. REST API가 이미 있으니 **모듈 단위 점진 전환**:

1. **Spring을 순수 API 서버로** — 기존 `@RestController` 활용, 부족한 API만 보강
2. **모듈 하나만 먼저 SPA로** — 예: SD 견적(quotation)만 교체, 나머지는 Thymeleaf 유지 (공존)
3. 익숙해지면 모듈 단위로 확장
4. 빌드 도구 = **Vite** (Vue·React 공통 표준), 프레임워크 서버(Next/Nuxt)는 안 씀 → 3절 참조
5. 인증 = **이미 JWT** (`JwtAuthenticationFilter`, `Authorization: Bearer` 또는 `ACCESS_TOKEN` 쿠키, CSRF disabled).
   SPA는 이 JWT를 그대로 쓴다 → 별도 인증 재설계 불필요. (※ 이 REST+JWT 기반이 갖춰져 있다는 점이
   Next/Nuxt를 배제하고 순수 SPA를 택한 결정적 근거)
