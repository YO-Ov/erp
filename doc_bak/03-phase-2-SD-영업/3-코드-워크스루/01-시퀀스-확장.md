# 1/7. 시퀀스 확장 — `code_sequence` 일반화 + `TransactionNumberGenerator`

> Phase 1의 `CUST-2026-0042` 와 Phase 2의 `SO-20260524-001` 을 **한 테이블** 로 통합한다.
> 설계서 §4 "트랜잭션 번호 발급 (CodeSequence 확장)" 의 실제 구현.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/common/code/
├─ CodeSequence.java            (year → periodKey 일반화)
├─ CodeSequenceRepository.java  (findForUpdate 시그니처 변경)
└─ CodeGenerator.java           (nextCode / nextTransactionCode 2 메서드)

hwlee-erp/src/main/java/com/hwlee/erp/sd/
└─ TransactionNumberGenerator.java  (SD 모듈의 발급 래퍼)

hwlee-erp/src/main/resources/db/migration/
└─ V9__alter_code_sequence_period_key.sql
```

## 🔥 왜 컬럼을 일반화했는가

Phase 1의 카운터 테이블 모양:

```
code_sequence
  prefix       VARCHAR(16)    -- 'CUST', 'ITEM', 'EMP', ...
  year         INT            -- 2026
  next_number  INT
  UNIQUE (prefix, year)
```

마스터는 **연 단위**로 번호가 리셋된다 (`CUST-2026-0042` → 2027년이 되면 `CUST-2027-0001`).

Phase 2의 요구는 다르다 — 트랜잭션은 **일 단위**로 리셋된다 (`SO-20260524-001` → 자정 지나면 `SO-20260525-001`).

설계서에서 세 옵션을 비교했다:

| 옵션 | 평가 |
| --- | --- |
| A. `year` 두고 `period_key` 컬럼 추가 | 이중 컬럼 — 마스터/트랜잭션 분기가 모든 코드에 퍼짐 |
| B. **`year` 를 `period_key VARCHAR(8)` 로 변경** | ✅ 한 테이블, 분기 없음 |
| C. `transaction_sequence` 테이블 분리 | 락 패턴 / Repository / CodeGenerator 두 벌 |

→ **옵션 B 채택**. 이유는 단순하다 — 락 메커니즘과 catch 분기 등 동시성 처리 코드가 그대로 재사용된다. 컬럼 하나 바꾸는 게 가장 작은 변경.

기존 데이터 보존도 자연스럽다:
- `year=2026` (INT) 행이 → `period_key='2026'` (VARCHAR) 로 변환됨.
- MySQL의 `CHANGE COLUMN year period_key VARCHAR(8)` 이 알아서 INT → 문자열 캐스팅.

## 🔥 V9 마이그레이션 — 컬럼 변경의 안전망

```sql
-- V9__alter_code_sequence_period_key.sql
ALTER TABLE code_sequence
    CHANGE COLUMN year period_key VARCHAR(8) NOT NULL
        COMMENT '발급 단위 — 마스터: YYYY, 트랜잭션: YYYYMMDD';

ALTER TABLE code_sequence
    DROP INDEX uk_code_sequence_prefix_year,
    ADD UNIQUE KEY uk_code_sequence_prefix_period (prefix, period_key);
```

두 가지 신경 쓸 점:

### 1. `CHANGE COLUMN` 이 `MODIFY COLUMN` 과 다른 점
- `MODIFY COLUMN year ...` → 타입만 변경, 이름 그대로.
- `CHANGE COLUMN year period_key ...` → **이름과 타입을 동시에 변경**.

`year INT` → `period_key VARCHAR(8)` 은 이름/타입 둘 다 바뀌니 `CHANGE` 가 맞다.

### 2. UNIQUE 인덱스 이름 갱신
원래의 `uk_code_sequence_prefix_year` 는 의미상 부적절해진다. DROP 후 새 이름으로 ADD. 인덱스가 가리키는 컬럼은 `(prefix, period_key)` 로 동일.

MySQL은 한 `ALTER TABLE` 안에서 DROP + ADD 를 동시에 처리할 수 있어 한 statement 로 끝낸다.

### V8 시드와의 호환성

V8 은 Phase 1 시드인데, 거기서 이렇게 쓴다:

```sql
INSERT INTO code_sequence (prefix, year, next_number, updated_at) VALUES
    ('CUST', YEAR(NOW()), 3, NOW()),
    ...
```

V8 → V9 순서로 실행되므로:
1. V8 실행 시점엔 `year INT` 컬럼이 살아있음 → INSERT 성공 (year=2026)
2. V9 실행 시 `CHANGE COLUMN` → year=2026 이 period_key='2026' 으로 변환

→ V8 을 손대지 않고도 마이그레이션이 매끄럽게 흐른다. **Flyway 원칙: 적용된 마이그레이션은 절대 수정하지 말 것** (체크섬이 깨지면 운영에서 Flyway가 거부함).

## 🔥 `CodeSequence` — `int year` 에서 `String periodKey` 로

```java
@Column(name = "period_key", nullable = false, length = 8)
private String periodKey;

public static CodeSequence initial(String prefix, String periodKey) {
    CodeSequence seq = new CodeSequence();
    seq.prefix = prefix;
    seq.periodKey = periodKey;
    seq.nextNumber = 1;
    return seq;
}
```

- 정적 팩토리 시그니처: `(String prefix, int year)` → `(String prefix, String periodKey)`.
- `length = 8` — `"20260524"` 가 최대 길이.
- 다른 부분(`issueAndIncrement()`, `@LastModifiedDate`, `@NoArgsConstructor(PROTECTED)`)은 그대로.

→ **Phase 1에서 이미 잘 잡은 약속들(정적 팩토리, 캡슐화, package-private `issueAndIncrement`)은 손대지 않았다**. 한 컬럼만 일반화.

## 🔥 `CodeGenerator` — 메서드 2개로 분리

```java
private static final String MASTER_FORMAT = "%s-%s-%04d";
private static final String TRANSACTION_FORMAT = "%s-%s-%03d";

@Transactional(propagation = Propagation.REQUIRES_NEW)
public String nextCode(String prefix) {
    String periodKey = String.valueOf(LocalDate.now(clock).getYear());
    int issued = issue(prefix, periodKey);
    return String.format(MASTER_FORMAT, prefix, periodKey, issued);
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public String nextTransactionCode(String prefix, String periodKey) {
    int issued = issue(prefix, periodKey);
    return String.format(TRANSACTION_FORMAT, prefix, periodKey, issued);
}

private int issue(String prefix, String periodKey) {
    CodeSequence sequence = repository.findForUpdate(prefix, periodKey)
            .orElseGet(() -> createOrLockExisting(prefix, periodKey));
    return sequence.issueAndIncrement();
}
```

### 왜 메서드 2개인가 — "마스터/트랜잭션을 호출 측이 의식하지 않게"

설계 때 한 메서드(`nextCode(prefix, periodKey)`)로 통합할까 고민했지만, 호출 코드가 더 지저분해진다:

```java
// 한 메서드안으로 통합한 가상의 코드
String code = codeGenerator.nextCode("CUST", String.valueOf(LocalDate.now().getYear()));
String number = codeGenerator.nextCode("SO", LocalDate.now().format(YYYYMMDD));
```

→ 매번 호출 측이 periodKey 포맷을 의식해야 한다. **정책(연 단위인가 일 단위인가)이 호출 측으로 새어 나간다**.

두 메서드로 나누면:
- `nextCode("CUST")` → 마스터 — periodKey 는 자동으로 올해 연도.
- `nextTransactionCode("SO", "20260524")` → 트랜잭션 — periodKey 는 명시적.

추가로 포맷도 다르다 (`%04d` vs `%03d`) — 마스터는 1만 건, 트랜잭션은 하루 1000건 한도. 이 정책 차이도 메서드별 상수로 분리되어 명확.

### 공통 로직은 `private int issue(...)` 한 메서드로

UNIQUE 위반 catch / FOR UPDATE 락 / orElseGet 흐름은 Phase 1과 동일하므로 한 메서드로 모은다. 두 public 메서드가 형식만 다르게 감싸는 구조.

→ 동시성 처리(설계서 §4.1과 02편 워크스루)는 손대지 않았다. **검증된 인프라를 재사용한다**.

### `REQUIRES_NEW` 와 "번호 구멍" 은 그대로

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
```

이게 트랜잭션 영역에서 가장 중요한 한 줄. Phase 1 워크스루 02편에서 짚었듯이 — 호출 측 트랜잭션이 롤백되어도 번호는 이미 커밋되어 있으니 "구멍" 이 생긴다. 동시성 안전성과의 트레이드오프.

→ Phase 2의 트랜잭션 번호도 같은 정책. **SO-20260524-005 가 갑자기 안 보일 수 있다는 가능성을 운영팀이 알고 있어야 한다**.

## 🔥 `TransactionNumberGenerator` — 의미를 입히는 얇은 래퍼

```java
@Component
@RequiredArgsConstructor
public class TransactionNumberGenerator {

    static final String PREFIX_QUOTATION = "Q";
    static final String PREFIX_SALES_ORDER = "SO";
    static final String PREFIX_DELIVERY = "DLV";
    static final String PREFIX_INVOICE = "INV";

    private static final DateTimeFormatter PERIOD_FORMAT =
            DateTimeFormatter.ofPattern("yyyyMMdd");

    private final CodeGenerator codeGenerator;

    public String nextQuotationNumber(LocalDate issuedDate) {
        return codeGenerator.nextTransactionCode(PREFIX_QUOTATION, periodKey(issuedDate));
    }

    public String nextSalesOrderNumber(LocalDate orderDate) {
        return codeGenerator.nextTransactionCode(PREFIX_SALES_ORDER, periodKey(orderDate));
    }

    public String nextDeliveryNumber(LocalDate shippedDate) { /* 동일 */ }
    public String nextInvoiceNumber(LocalDate invoiceDate)  { /* 동일 */ }

    private static String periodKey(LocalDate date) {
        return date.format(PERIOD_FORMAT);
    }
}
```

### 왜 얇은 래퍼를 또 만들었나

`CodeGenerator.nextTransactionCode("SO", "20260524")` 를 서비스에서 직접 부르면 안 될까? — 동작은 한다. 그런데:

1. **Prefix 가 코드 곳곳에 string literal 로 흩어진다** — `"SO"`, `"DLV"`, `"INV"`, `"Q"` 가 4개 서비스 클래스에 등장. 오타가 컴파일러를 통과해 운영에서 발견.
2. **날짜 포맷 결정이 호출 측에 흩어진다** — `LocalDate.format(DateTimeFormatter.ofPattern("yyyyMMdd"))` 가 매번 반복. 형식 바꾸려면 grep.
3. **메서드 이름이 의미를 전달한다** — `nextSalesOrderNumber(orderDate)` 는 "수주 번호 발급" 이라는 의도가 명확. `nextTransactionCode("SO", ...)` 는 의미가 분산.

→ 얇아도 **prefix 결정 + 날짜 포맷팅을 한 곳에 모은다**. 12편의 `CodeGenerator` 가 인프라라면, 이 클래스는 **SD 모듈의 어휘**.

### `LocalDate` 받기 — `Clock` 까지 갈 필요는?

`Clock` 을 주입받지 않고 `LocalDate` 를 인자로 받는다. 호출 측이 이미 영업일/주문일 같은 "도메인 날짜" 를 가지고 있기 때문이다:

```java
salesOrderService.create(SalesOrderCreateRequest(..., orderDate, ...))
  ↓
String number = numberGenerator.nextSalesOrderNumber(req.orderDate());
```

`orderDate` 는 사용자가 명시한 "이 주문은 5/24일자" — 발급 시각의 시스템 시계가 아니다. 마감 직전(23:59)에 등록한 주문도 "내일자 주문" 으로 의도되었다면 그날 번호를 받아야 한다.

→ **번호의 날짜와 주문의 날짜를 일치시킨다**. 운영에서 "5/24일자 주문인데 번호는 5/25" 같은 어긋남이 안 생긴다.

## 🔥 마스터 호출부 변경 — 0 줄

여기서 살짝 놀라운 점이 있다 — **Phase 1의 호출부 `customerService.create(...)` 는 한 줄도 안 바뀌었다**.

```java
// CustomerService.java (Phase 1 그대로)
String code = codeGenerator.nextCode(CODE_PREFIX);  // "CUST"
```

이게 가능한 이유는 `CodeGenerator.nextCode(String prefix)` 의 **외부 시그니처를 유지**했기 때문이다. 내부적으로 periodKey 를 올해 연도 문자열로 자동 결정한다.

→ **인터페이스를 깨지 않고 내부를 일반화** — 이게 좋은 리팩토링의 모습.

## 🔥 시나리오 — `SO-20260524-001` 이 발급되기까지

```
[웹: POST /api/sales-orders { orderDate: "2026-05-24", ... }]
   │
   ▼
[SalesOrderService.create(req)]
   │ — T0 시작
   │
   ├─ numberGenerator.nextSalesOrderNumber(LocalDate.parse("2026-05-24"))
   │     │
   │     ├─ periodKey("2026-05-24") → "20260524"
   │     │
   │     └─ codeGenerator.nextTransactionCode("SO", "20260524")
   │           │ — REQUIRES_NEW → T1 시작
   │           │
   │           ├─ findForUpdate("SO", "20260524")
   │           │    → SELECT ... WHERE prefix='SO' AND period_key='20260524'
   │           │       FOR UPDATE
   │           │
   │           ├─ 없음 → createOrLockExisting → INSERT (next_number=1)
   │           │
   │           ├─ issueAndIncrement() → 1 반환, next_number=2
   │           │
   │           └─ T1 커밋 → "SO-20260524-001" 반환
   │
   ├─ SalesOrder.draft("SO-20260524-001", customer, ..., orderDate)
   ├─ ...
   └─ T0 커밋
```

다음 요청은 같은 날짜라면 002, 003, ... 으로 진행. 자정이 지나 다른 날짜로 들어온 요청은 `findForUpdate("SO", "20260525")` 가 없는 행을 만나 새로 INSERT, 1부터 시작.
