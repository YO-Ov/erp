# 2/7. 수주 헤더-라인 — `SalesOrder` + `SalesOrderLine` 구조

> Phase 2 의 핵심 엔티티. 헤더-라인 관계, OneToMany cascade, 도메인 메서드 패턴을 한 번에.
> 설계서 §3.2 "SalesOrder (수주) — Phase 2의 핵심" 의 코드 버전.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/sd/order/
├─ SalesOrder.java             (헤더 엔티티 + 도메인 메서드)
├─ SalesOrderLine.java         (라인 엔티티 + 누적 메서드)
├─ SalesOrderStatus.java       (8개 상태 enum)
├─ SalesOrderRepository.java   (활성 합계 JPQL 쿼리)
└─ ... (Service/Controller/Mapper 는 풀스택은 04편/이번 편 마지막에)
```

견적도 같은 패턴(`Quotation` + `QuotationLine`)이라 마지막에 짧게 비교.

## 🔥 트랜잭션은 `BaseEntity` 만 상속 — Soft Delete 안 한다

```java
@Entity
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@Table(name = "sales_order")
public class SalesOrder extends BaseEntity {   // ← BaseEntityWithCode 가 아니다
    ...
}
```

Phase 1 워크스루 01편에서 미리 짚어둔 결정 — **트랜잭션은 `BaseEntityWithCode` 의 `code/status/deletedAt` 셋을 안 쓴다**.

| 마스터 (`BaseEntityWithCode`) | 트랜잭션 (`BaseEntity`) |
| --- | --- |
| `code VARCHAR(30)` — 외부 식별자 한 번만 할당 | `number VARCHAR(30)` — 각 엔티티가 직접 선언 |
| `status MasterStatus` — ACTIVE/INACTIVE/BLOCKED | 각자 다른 상태 enum (DRAFT/CONFIRMED/...) |
| `deleted_at` — Soft Delete | **없음** — 잘못된 입력은 `CANCELLED` 상태로 |

### 왜 트랜잭션은 Soft Delete 안 하나

설계서 §2.1 의 한 줄: **"트랜잭션은 물리 삭제 자체가 금기"**. 회계 감사 추적상 한 번 발급된 수주는 행이 그대로 남아야 한다.

- 마스터의 `deleted_at IS NULL` 자동 필터는 트랜잭션엔 안 어울린다 — "취소된 거래 목록" 도 조회해야 함.
- "지운 표식" 인 `deleted_at` 도 의미가 없다 — 어차피 안 지우니까.

→ 잘못된 입력은 **`CANCELLED` 상태로 표현**. 데이터는 살아 있고, 일반 목록 쿼리에서는 필터로 빠진다.

### `code` 대신 `number` 를 직접 선언

```java
@Column(name = "number", nullable = false, unique = true, length = 30, updatable = false)
private String number;
```

- `unique = true` — 헤더 번호 중복 금지.
- `updatable = false` — 한 번 발급한 번호는 변경 불가 (감사 추적).
- 마스터의 `BaseEntityWithCode.code` 와 비슷한 약속이지만 베이스에 끌어올리지 않았다.

### 왜 `BaseTransactionEntity` 베이스 클래스를 만들지 않았나

설계 4단계에서 두 옵션을 비교했었다:

- **A. `BaseEntity` 만 상속, `number` 는 각자 선언** ← 채택
- B. `BaseTransactionEntity extends BaseEntity` 새 베이스 + `number`

→ 공통화로 묶을 게 결국 `number` 한 컬럼이고, 상태 enum 타입과 라인 컬렉션 타입은 모두 다르다. **한 컬럼을 위해 추상 클래스를 또 만드는 건 과한 추상화**.

설계서 §2.2: "추후 트랜잭션 모듈이 더 늘어 공통 행위(취소 가능 여부 등)가 명확해지면 그때 추출". → **YAGNI**.

## 🔥 `@OneToMany` cascade + orphanRemoval — 헤더-라인의 일관성

```java
@OneToMany(mappedBy = "salesOrder", cascade = CascadeType.ALL,
           orphanRemoval = true, fetch = FetchType.LAZY)
@OrderBy("lineNo ASC")
private List<SalesOrderLine> lines = new ArrayList<>();
```

이 어노테이션이 헤더-라인 관계의 모든 일관성을 보장한다.

### `cascade = ALL` — 한 트랜잭션으로 헤더 + 라인 저장
```java
SalesOrder order = SalesOrder.draft(...);
order.addLine(item1, qty1, price1);   // 라인 객체가 lines 에 추가됨
order.addLine(item2, qty2, price2);
repository.save(order);
   ↓
   - INSERT sales_order
   - INSERT sales_order_line (×2)  ← cascade 가 자동
```

라인을 따로 `lineRepository.save(...)` 할 필요가 없다. 헤더 저장이 곧 라인 저장.

### `orphanRemoval = true` — 라인을 빼면 DB에서도 빠진다
```java
order.clearLines();   // lines.clear() 내부에서
repository.save(order);   // implicit 으로 flush
   ↓
   - DELETE FROM sales_order_line WHERE id IN (...)
```

`orphanRemoval` 이 없으면 `lines.clear()` 가 자바 컬렉션만 비우고 DB 행은 남는다 — "버려진 행" 이 생긴다.

### `mappedBy = "salesOrder"` — 외래 키 주인은 라인
```java
// SalesOrderLine.java
@ManyToOne(fetch = FetchType.LAZY, optional = false)
@JoinColumn(name = "sales_order_id", nullable = false)
private SalesOrder salesOrder;
```

`sales_order_line` 테이블의 `sales_order_id` 컬럼이 외래 키. 양방향 매핑이지만 **주인은 자식(`SalesOrderLine.salesOrder`)** 이고, 헤더의 `lines` 는 그 거울(`mappedBy`).

→ 양방향을 굳이 두는 이유는 도메인 메서드(`order.addLine(...)`, `order.getLines()`)를 헤더에서 부르고 싶기 때문.

### `fetch = LAZY` 디폴트
헤더 로드 시 라인을 자동으로 안 가져온다. 라인이 필요한 메서드(`order.recordShipment(...)`) 에서 컬렉션을 건드는 순간 자동으로 SQL 한 번 발사.

N+1은 의도된 비용 — Phase 2에서는 한 수주의 라인이 보통 3~10개라 큰 부담이 아니다. 대량 조회(`Page<SalesOrder>`)에 라인까지 필요하면 fetch join 으로 해결 (학습 범위 밖).

### `@OrderBy("lineNo ASC")` — 결정적 라인 순서
DB가 반환하는 행 순서는 기본적으로 보장되지 않는다. `@OrderBy` 가 SELECT 에 `ORDER BY line_no ASC` 를 자동 추가 → 1, 2, 3, ... 순서 보장.

→ "라인 3을 추가했더니 갑자기 화면에 위로 올라간다" 같은 사고를 막는다.

## 🔥 `SalesOrderLine` — 부분 출하/청구의 단일 진실

```java
@Entity
public class SalesOrderLine extends BaseEntity {

    @ManyToOne(fetch = LAZY, optional = false)
    @JoinColumn(name = "sales_order_id", nullable = false)
    private SalesOrder salesOrder;

    @Column(name = "line_no", nullable = false)
    private int lineNo;

    @ManyToOne(fetch = LAZY, optional = false)
    @JoinColumn(name = "item_id", nullable = false)
    private Item item;

    @Column(name = "order_qty",    precision = 15, scale = 4) private BigDecimal orderQty;
    @Column(name = "shipped_qty",  precision = 15, scale = 4) private BigDecimal shippedQty = BigDecimal.ZERO;
    @Column(name = "invoiced_qty", precision = 15, scale = 4) private BigDecimal invoicedQty = BigDecimal.ZERO;
    @Column(name = "unit_price",   precision = 15, scale = 2) private BigDecimal unitPrice;
    @Column(name = "line_total",   precision = 15, scale = 2) private BigDecimal lineTotal;
```

### 세 수량 컬럼이 OTC 흐름의 척추

```
order_qty   = 10  (확정 후 불변)
shipped_qty =  6  (이 라인의 출하 누적)
invoiced_qty=  6  (이 라인의 청구 누적)
```

설계서 §0의 핵심 결정 #7: **"부분 출하/인보이스 추적은 수주 라인의 `shipped_qty`, `invoiced_qty` 누적값"**.

이 한 행을 보면 "10대 중 6대 보냈고 6대 청구했다" 가 즉시 드러난다. 출하 테이블이나 인보이스 테이블을 join 해서 합산할 필요가 없다.

### 불변 조건은 도메인 메서드가 강제

```java
void addShippedQty(BigDecimal qty) {
    BigDecimal next = this.shippedQty.add(qty);
    if (next.compareTo(orderQty) > 0)
        throw new IllegalStateException(
                "출하 누적이 주문량을 초과합니다. orderQty=" + orderQty + ", shippedQty(after)=" + next);
    this.shippedQty = next;
}

void addInvoicedQty(BigDecimal qty) {
    BigDecimal next = this.invoicedQty.add(qty);
    if (next.compareTo(shippedQty) > 0)
        throw new IllegalStateException(
                "청구 누적이 출하 누적을 초과합니다. shippedQty=" + shippedQty + ", invoicedQty(after)=" + next);
    this.invoicedQty = next;
}
```

규칙:
- `shipped_qty ≤ order_qty` — 주문량보다 더 보낼 수 없다.
- `invoiced_qty ≤ shipped_qty` — **보낸 것만 청구 가능**.

이 규칙은 DB 제약(CHECK)에도 못 박을 수 있겠지만, **메서드 안에서 검증**한 이유는:
1. 비즈니스 메시지를 한국어로 던질 수 있다 (사용자에게 친절한 에러).
2. CHECK 위반 대신 도메인 예외(`IllegalStateException`)로 통일 — `GlobalExceptionHandler` 가 409 Conflict 로 변환.
3. DB 제약은 MySQL 8.0+에서만 enforce — 학습용으로 한 곳에 모아 두는 게 단순.

### 가시성 — `package-private void addShippedQty(...)`

```java
void addShippedQty(BigDecimal qty) { ... }       // ← package-private
void subtractShippedQty(BigDecimal qty) { ... }  // ← package-private
```

`public` 이 아니다. 외부 서비스가 라인을 직접 잡고 `shippedQty` 를 올리면 헤더 상태가 안 따라간다 — 라인은 SHIPPING 인데 헤더는 CONFIRMED.

→ **누적은 반드시 `SalesOrder.recordShipment(line, qty)` 를 거치게** 한다. 그래야 헤더의 `recomputeStatus()` 가 같이 호출되어 일관성 유지 (03편).

### `remainingShippable()` — 도메인 의미를 메서드 이름에

```java
public BigDecimal remainingShippable() {
    return orderQty.subtract(shippedQty);
}

public BigDecimal remainingInvoiceable() {
    return shippedQty.subtract(invoicedQty);
}
```

호출 측 (`DeliveryLine` 생성자) 에서 사용:
```java
BigDecimal remaining = salesOrderLine.remainingShippable();
if (quantity.compareTo(remaining) > 0)
    throw new IllegalStateException("잔여 출하 가능량을 초과합니다. ...");
```

`orderQty.subtract(shippedQty)` 를 호출 측에서 직접 계산하면 의미가 옅어진다. **메서드 이름이 도메인 어휘** — "잔여 출하 가능량".

### `precision/scale` — 왜 4자리 소수?

```java
@Column(name = "order_qty", precision = 15, scale = 4)
private BigDecimal orderQty;
```

상품 단위가 EA (정수) 면 `scale=0` 이어도 충분하지만, 무게/길이 단위 (`KG` 등) 는 소수가 필요. Phase 1 `Item.unit` 이 다양한 단위를 허용하므로 **수량 컬럼은 일반화해 둔다**.

`scale=4` 는 통상 4자리 소수까지 — `0.0025 KG` 같은 표현 가능. 금액(`unitPrice`, `lineTotal`)은 `scale=2` (원 단위 정수 + 소수 둘째 자리까지).

## 🔥 `SalesOrder.draft(...)` — 정적 팩토리

```java
public static SalesOrder draft(String number, Customer customer, Employee salesperson,
                               Quotation quotation, LocalDate orderDate) {
    if (number == null || number.isBlank()) throw new IllegalArgumentException("number 는 비어 있을 수 없다.");
    if (customer == null) throw new IllegalArgumentException("customer 는 null 일 수 없다.");
    if (orderDate == null) throw new IllegalArgumentException("orderDate 는 null 일 수 없다.");
    SalesOrder so = new SalesOrder();
    so.number = number;
    so.customer = customer;
    so.salesperson = salesperson;   // nullable
    so.quotation = quotation;       // nullable
    so.orderDate = orderDate;
    return so;
}
```

- 메서드명이 **`create` 가 아니라 `draft`** — 만들어진 직후 상태가 `DRAFT` 라는 약속을 이름으로 표현.
- 필수 (`number`, `customer`, `orderDate`) 와 선택 (`salesperson`, `quotation`) 을 명확히.
- 시그니처를 보면 "수주는 무엇이 있어야 시작되나?" 가 한눈에.

### 왜 라인은 생성자에 안 들어가나

```java
SalesOrder order = SalesOrder.draft("SO-...", customer, null, null, LocalDate.now());
order.addLine(item1, qty1, price1);
order.addLine(item2, qty2, price2);
```

생성자에 `List<SalesOrderLine>` 을 받을 수도 있지만 두 가지 단점:

1. **순환 참조 생성 문제** — 라인이 헤더를 가리키려면 헤더가 먼저 만들어져야 함.
2. **검증 단계화** — 헤더 생성 후 한 라인씩 추가하면서 각 라인의 검증을 따로 처리.

→ `addLine` 메서드가 라인 검증 (수량/가격) 과 라인 번호 할당, 헤더 총액 재계산을 한 번에 처리.

## 🔥 `addLine()` — 캡슐화된 라인 추가

```java
public SalesOrderLine addLine(Item item, BigDecimal qty, BigDecimal unitPrice) {
    ensureEditable();
    SalesOrderLine line = new SalesOrderLine(this, lines.size() + 1, item, qty, unitPrice);
    lines.add(line);
    recalculateTotal();
    return line;
}

private void recalculateTotal() {
    this.totalAmount = lines.stream()
            .map(SalesOrderLine::getLineTotal)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
}

private void ensureEditable() {
    if (status != SalesOrderStatus.DRAFT)
        throw new IllegalStateException("DRAFT 상태에서만 수정 가능합니다. 현재: " + status);
}
```

이 한 메서드에 **세 가지 도메인 규칙**이 모여 있다:

1. **`ensureEditable()`** — DRAFT 가 아닌 수주에는 라인 추가 불가.
2. **`lines.size() + 1`** — 라인 번호 자동 부여 (1, 2, 3, ...). 호출 측이 결정하지 않는다.
3. **`recalculateTotal()`** — 라인 추가 직후 헤더 총액 즉시 갱신.

설계서 §0의 핵심 결정 #6: **"헤더 총액(`total_amount`)을 DB에 저장 (Denormalization)"**. 매번 라인 합산 SQL을 발사하지 않고, 변경 시점에 한 번 재계산 후 헤더에 저장.

> 💡 외부에서 `order.setTotalAmount(...)` 를 호출할 수 없도록 setter 비공개. 도메인 메서드 안에서만 갱신.

## 🔥 `SalesOrderStatus` — 8개 상태

```java
public enum SalesOrderStatus {
    DRAFT,
    CONFIRMED,
    SHIPPING,
    SHIPPED,
    INVOICING,
    INVOICED,
    CLOSED,       // Phase 5 (수금 완료)
    CANCELLED
}
```

8개가 많아 보이지만, OTC 흐름이 직선이 아닌 게 본질적 복잡도다 (브리핑 §1 참고).

- `DRAFT → CONFIRMED → SHIPPING ↔ SHIPPED → INVOICING ↔ INVOICED → CLOSED`
- 어디서든 `CANCELLED` 로 분기 (단, 출하 실적이 없을 때만)

상태 머신의 자세한 흐름은 다음 편 (03편).

`@Enumerated(EnumType.STRING)` — DB에는 문자열로 저장 (`'CONFIRMED'`). 순서 의존 (`ORDINAL`) 안 쓰는 이유: enum 값을 추가하면 순서가 바뀌어 운영 DB가 깨질 위험.

## 🔥 SalesOrder → SalesOrderLine 의 양방향 매핑 — 일관성 위한 `findLineById`

```java
public SalesOrderLine findLineById(Long lineId) {
    return lines.stream()
            .filter(l -> l.getId() != null && l.getId().equals(lineId))
            .findFirst()
            .orElseThrow(() -> new IllegalArgumentException(
                    "이 수주에 해당 라인이 없습니다. salesOrderLineId=" + lineId));
}
```

`DeliveryService.create()` 가 라인을 검색할 때 사용:
```java
SalesOrderLine sol = order.findLineById(lineReq.salesOrderLineId());
```

왜 `SalesOrderLineRepository.findById(...)` 를 안 쓰는가? — 출하 라인이 **이 수주의 라인이 맞는지** 같이 확인하기 위해서다. 다른 수주의 라인 ID 를 넘기면 검색 실패 (`IllegalArgumentException`). 잠재적 데이터 오염을 메서드 시그니처가 막아준다.

## 🔥 `Quotation` 도 같은 패턴 — 짧게 비교

```java
@Entity @Getter @NoArgsConstructor(access = PROTECTED) @Table(name = "quotation")
public class Quotation extends BaseEntity {
    private String number;                   // Q-YYYYMMDD-NNN
    @ManyToOne(...) private Customer customer;
    @Enumerated(STRING) private QuotationStatus status = DRAFT;
    private LocalDate issuedDate;
    private LocalDate validUntil;            // nullable
    private BigDecimal totalAmount = ZERO;
    @OneToMany(mappedBy = "quotation", cascade = ALL, orphanRemoval = true)
    private List<QuotationLine> lines = new ArrayList<>();
    ...
}
```

수주와 거의 동일. 차이점만 짚으면:
- `validUntil` 컬럼 (견적 유효 기한) — nullable.
- 상태 enum: `DRAFT/SENT/ACCEPTED/EXPIRED/CANCELLED`.
- 견적 라인은 `quantity`만 — 출하/청구 누적이 없다 (견적은 "제안" 까지만).

설계서 §0의 #14: "Quotation → SalesOrder 변환은 `quotation_id` 추적만, 라인 자동 복사는 안 함". 학습 단순화 — 견적이 수락되어도 수주 라인은 사용자가 다시 입력.

→ **두 도메인이 같은 헤더-라인 구조 + 도메인 메서드 패턴을 공유**한다. Phase 2 마지막에 공통 추출을 고민할 수 있지만, 아직은 두 벌이 명확하다 (YAGNI).
