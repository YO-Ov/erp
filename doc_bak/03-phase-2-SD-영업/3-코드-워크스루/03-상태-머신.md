# 3/7. 상태 머신 — `recomputeStatus()` 와 도메인 메서드의 약속

> 8개 상태가 라인 누적값에서 자연스럽게 파생되도록 설계.
> 설계서 §3.2 상태 전이도와 §0의 핵심 결정 #10 "상태 변경은 도메인 메서드만, `setStatus()` 비공개" 의 실제 구현.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/sd/order/
└─ SalesOrder.java   (confirm / recordShipment / recordInvoicing / cancel*  + recomputeStatus)
```

---

## 🔥 설계 진화 — 분기 폭발 → 파생 상태

설계 제안서에서는 도메인 메서드별로 상태를 직접 세팅하는 형태였다:

```java
// 초기 설계 — 안 채택
public void recordShipment(SalesOrderLine line, BigDecimal qty) {
    if (status != CONFIRMED && status != SHIPPING)
        throw new IllegalStateException("...");
    line.addShippedQty(qty);
    this.status = isAllShipped() ? SHIPPED : SHIPPING;
}
```

구현하면서 통합 시나리오 테스트(10대 → 6+4 부분 출하·청구)를 돌려보니 한 가지 케이스가 막혔다:

```
수주 확정 → 6대 출하 (SHIPPING) → 6대 인보이스 (INVOICING)
   → 4대 추가 출하 시도 → ❌ "출하 등록 가능 상태가 아닙니다. 현재: INVOICING"
```

`recordShipment` 가 CONFIRMED/SHIPPING 만 허용했기 때문. INVOICING 중에도 추가 출하가 자연스러운 흐름인데 막혀버렸다.

→ "상태별로 허용 메서드를 명시" 하는 방식은 분기 폭발이 일어난다. 대안은 **라인 누적값에서 헤더 상태를 파생** 하는 방식.

```
shipped_qty  → 누적
invoiced_qty → 누적
        │
        ▼ (recomputeStatus)
   header.status   ← "한 줄로 계산된다"
```

각 도메인 메서드는 **라인 누적값만 변경**하고 마지막에 `recomputeStatus()` 한 번을 부른다. 상태 전이 로직이 한 곳에 모인다.

---

## 🔥 `recomputeStatus()` — 헤더 상태를 한 함수로

```java
private void recomputeStatus() {
    if (status == SalesOrderStatus.DRAFT
            || status == SalesOrderStatus.CANCELLED
            || status == SalesOrderStatus.CLOSED) {
        return;   // 이 세 상태는 라인 누적과 무관
    }
    boolean anyShipped   = lines.stream().anyMatch(l -> l.getShippedQty().signum() > 0);
    boolean anyInvoiced  = lines.stream().anyMatch(l -> l.getInvoicedQty().signum() > 0);
    if (isAllInvoiced()) {
        this.status = SalesOrderStatus.INVOICED;
    } else if (anyInvoiced) {
        this.status = SalesOrderStatus.INVOICING;
    } else if (isAllShipped()) {
        this.status = SalesOrderStatus.SHIPPED;
    } else if (anyShipped) {
        this.status = SalesOrderStatus.SHIPPING;
    } else {
        this.status = SalesOrderStatus.CONFIRMED;
    }
}
```

### 진리표

| 라인 누적값 | 결과 상태 |
| --- | --- |
| `shipped=0`, `invoiced=0` (모든 라인) | CONFIRMED |
| 일부 라인 `shipped>0`, 모든 라인 `invoiced=0` | SHIPPING |
| 모든 라인 `shipped == orderQty`, 모든 라인 `invoiced=0` | SHIPPED |
| 일부 라인 `invoiced>0`, 누군가 `invoiced < shipped` | INVOICING |
| 모든 라인 `invoiced == orderQty` | INVOICED |

**중요**: "shipped 가 다 끝나지 않아도 invoiced 가 시작될 수 있다" 가 자연스럽게 표현된다 — `anyInvoiced && !isAllInvoiced()` 면 INVOICING.

### 왜 DRAFT/CANCELLED/CLOSED 는 제외

- **DRAFT**: 아직 confirm 안 됨. 라인 누적값이 0인 게 당연. 재계산하면 CONFIRMED 로 잘못 전이.
- **CANCELLED**: 종착 상태. 라인 누적과 관계없이 그대로 둬야 함.
- **CLOSED**: Phase 5 (수금 완료) 의 종착 상태. 마찬가지.

→ "전이 가능한 상태 집합" (CONFIRMED ~ INVOICED) 안에서만 동작.

## 🔥 도메인 메서드 4개 — 모두 한 패턴

```java
public void recordShipment(SalesOrderLine line, BigDecimal qty) {
    ensureLineBelongsToThis(line);
    ensureProgressable("출하");
    line.addShippedQty(qty);
    recomputeStatus();
}

public void cancelShipment(SalesOrderLine line, BigDecimal qty) {
    ensureLineBelongsToThis(line);
    ensureProgressable("출하 취소");
    line.subtractShippedQty(qty);
    recomputeStatus();
}

public void recordInvoicing(SalesOrderLine line, BigDecimal qty) {
    ensureLineBelongsToThis(line);
    ensureProgressable("청구");
    line.addInvoicedQty(qty);
    recomputeStatus();
}

public void cancelInvoicing(SalesOrderLine line, BigDecimal qty) {
    ensureLineBelongsToThis(line);
    ensureProgressable("청구 취소");
    line.subtractInvoicedQty(qty);
    recomputeStatus();
}
```

네 메서드가 **동일한 4단계**:
1. `ensureLineBelongsToThis(line)` — 라인이 이 수주에 속하는지.
2. `ensureProgressable("...")` — 진행 가능 상태인지.
3. 라인 누적값 변경 (`add/subtract ShippedQty / InvoicedQty`).
4. `recomputeStatus()`.

상태 전이 분기가 메서드별로 흩어지지 않고 한 군데에 모인다. 새 흐름(예: 반품)이 추가되어도 라인 누적 → recomputeStatus 패턴이 그대로 적용 가능.

## 🔥 `ensureProgressable()` — 진행 가능 상태 게이트

```java
private void ensureProgressable(String op) {
    if (status == SalesOrderStatus.DRAFT
            || status == SalesOrderStatus.CANCELLED
            || status == SalesOrderStatus.CLOSED) {
        throw new IllegalStateException(op + " 가능 상태가 아닙니다. 현재: " + status);
    }
}
```

진행 불가 상태:
- **DRAFT**: 확정 전이므로 출하/청구 불가.
- **CANCELLED**: 취소된 수주에 새 출하/청구 불가.
- **CLOSED**: 수금까지 마친 수주. 추가 작업 없음.

→ 허용 상태는 CONFIRMED / SHIPPING / SHIPPED / INVOICING / INVOICED. 이 안에서 라인 누적이 자유롭게 변하고, 결과 상태는 `recomputeStatus()` 가 결정.

### `op` 파라미터의 역할

```java
ensureProgressable("출하");       // → "출하 가능 상태가 아닙니다. 현재: CANCELLED"
ensureProgressable("청구 취소");   // → "청구 취소 가능 상태가 아닙니다. 현재: DRAFT"
```

같은 검증 로직을 4개 메서드가 공유하면서, **에러 메시지에는 호출 맥락**이 드러난다. 사용자에게 "왜 막혔는지" 가 더 명확.

## 🔥 `ensureLineBelongsToThis()` — Hibernate 프록시와의 동거

```java
private void ensureLineBelongsToThis(SalesOrderLine line) {
    if (line == null) throw new IllegalArgumentException("line 은 null 일 수 없다.");
    SalesOrder owner = line.getSalesOrder();
    if (owner == null) throw new IllegalArgumentException("이 수주에 속하지 않는 라인입니다.");
    if (this.getId() != null) {
        if (!this.getId().equals(owner.getId())) {
            throw new IllegalArgumentException("이 수주에 속하지 않는 라인입니다.");
        }
    } else if (owner != this) {
        throw new IllegalArgumentException("이 수주에 속하지 않는 라인입니다.");
    }
}
```

처음에는 한 줄이었다:
```java
if (line.getSalesOrder() != this) throw new IllegalArgumentException(...);
```

`==` 참조 비교만으로 충분해 보이지만, **Hibernate 프록시**가 끼면 깨진다. 통합 테스트에서 발견된 시나리오:

```
DeliveryService.cancel(deliveryId):
  Delivery delivery = repo.findById(...);
  SalesOrder order = delivery.getSalesOrder();   // ← Hibernate 프록시
  order.cancelShipment(dlvLine.getSalesOrderLine(), qty)
       ↓ 프록시가 실 객체에 delegate
  // 실 SalesOrder 내부에서 ensureLineBelongsToThis 호출
  // this = 실 객체
  // line.getSalesOrder() = 프록시 (또는 다른 인스턴스)
  // this != line.getSalesOrder()  → 잘못된 거부
```

해결책: **ID 가 있으면 ID 기반**, 없으면(persisted 전) 참조 기반.

- DB에 저장된 엔티티는 ID 가 있으니 `getId().equals(owner.getId())` 가 안전.
- 유닛 테스트의 비영속 엔티티는 ID 가 null → 참조 비교로 떨어진다.

`getId()` 호출은 Hibernate 프록시도 fetch 없이 안전하게 반환 (proxy 가 id 만은 미리 가짐).

> 💡 이 한 메서드가 "운영 환경의 트랜잭션 흐름" 과 "유닛 테스트의 새로 만든 객체" 양쪽을 모두 통과시킨다.

## 🔥 `confirm()` — DRAFT → CONFIRMED 만은 직접 세팅

```java
public void confirm(LocalDateTime now) {
    if (status != SalesOrderStatus.DRAFT)
        throw new IllegalStateException("DRAFT 수주만 확정 가능합니다. 현재: " + status);
    if (lines.isEmpty())
        throw new IllegalStateException("라인이 비어 있는 수주는 확정할 수 없습니다.");
    this.status = SalesOrderStatus.CONFIRMED;
    this.confirmedAt = now;
}
```

`recomputeStatus()` 가 DRAFT 를 무시하므로, **DRAFT → CONFIRMED 만은 직접 세팅**한다. 이게 "전이 가능 영역으로 진입하는 한 번뿐인 이벤트".

`confirmedAt` 타임스탬프도 같이 기록 — Phase 5 (회계) 에서 "주문이 확정된 시각" 이 회계 기간 분류 기준이 될 수 있어 미리 저장.

### `LocalDateTime now` 를 인자로 받는 이유

```java
public void confirm(LocalDateTime now)   // ← Clock 안 받음
```

`LocalDateTime.now()` 를 메서드 안에서 부르면 결정적 테스트가 어렵다. 호출 측 서비스가 `Clock.instant()` 로 결정적 시각을 만들어 넘김:

```java
// SalesOrderService.confirm(...)
order.confirm(LocalDateTime.now(clock));
```

→ 도메인 메서드는 외부 의존(시계) 없이 입력만 받아 처리. 시계는 서비스 계층의 책임.

## 🔥 `cancel()` — 종착 상태 진입의 검증

```java
public void cancel() {
    if (status != SalesOrderStatus.DRAFT && status != SalesOrderStatus.CONFIRMED)
        throw new IllegalStateException(
                "DRAFT 또는 CONFIRMED 수주만 취소 가능합니다. 현재: " + status);
    boolean anyShipped = lines.stream().anyMatch(l -> l.getShippedQty().signum() > 0);
    if (anyShipped)
        throw new IllegalStateException("출하 실적이 있는 수주는 취소할 수 없습니다.");
    this.status = SalesOrderStatus.CANCELLED;
}
```

두 게이트:
1. **상태 게이트**: DRAFT/CONFIRMED 만 허용 — SHIPPING 이후엔 직접 취소 불가 (반품 흐름은 Phase 2 범위 밖).
2. **실적 게이트**: CONFIRMED 라도 출하 실적이 있으면 거부 — 이론상 CONFIRMED 면 shipped=0 이지만, 다른 흐름으로 일부 출하 후 cancelShipment 한 경우 등 방어적 체크.

설계서 §0 #2 "잘못 입력했다 → `CANCELLED` 상태로 둔다" — 물리 삭제 대신 상태로 표현.

## 🔥 상태 전이의 전체 그림

```
        ┌──────┐
        │DRAFT │  ← addLine / updateHeader / clearLines 가능
        └──┬───┘
           │ confirm(now) [라인 비어있지 않으면 OK]
           ▼
       ┌────────┐
       │CONFIRMED│
       └──┬─────┘
          │
          │ recordShipment(line, qty)  ─→ shipped_qty 증가
          ▼ (recomputeStatus 계산)
       ┌────────┐               ┌────────┐
       │SHIPPING│ ←─ all? ─────→│SHIPPED │
       └──┬─────┘               └──┬─────┘
          │                        │
          │ recordInvoicing(line, qty) ─→ invoiced_qty 증가
          ▼ (recomputeStatus 계산)         ▼
       ┌─────────┐              ┌─────────┐
       │INVOICING│ ←─ all? ────→│INVOICED │
       └─────────┘              └────┬────┘
                                     │
                                     │ Phase 5: 수금 완료
                                     ▼
                                ┌────────┐
                                │ CLOSED │
                                └────────┘

DRAFT/CONFIRMED ─ cancel() (실적 없을 때만) ─→ CANCELLED
```

화살표는 **도메인 메서드 호출**. 상태 자체를 직접 바꾸는 방법은 없다 (`setStatus()` 비공개).

## 🔥 외부 의존이 있는 검증은 도메인 밖

`confirm()` 안에서 신용한도를 검사하지 않았다 — 왜?

```java
public void confirm(LocalDateTime now) {
    // 신용한도 검증 X
    if (status != DRAFT) throw ...;
    if (lines.isEmpty()) throw ...;
    this.status = CONFIRMED;
    this.confirmedAt = now;
}
```

신용한도 검증은 **"이 고객의 다른 활성 수주 합계"** 가 필요하다 — 자기 자신만으로 판단 불가. Repository 의존이 도메인 엔티티에 끼면:
- 엔티티 객체 그래프가 무거워짐 (DI 컨테이너 의존).
- 유닛 테스트할 때 mock 이 필요.
- 엔티티 본분 (자기 상태 책임) 이 흐려진다.

→ 외부 의존이 필요한 규칙은 **서비스 단**에서 `confirm()` 호출 직전 처리. 자세한 패턴은 05편.
