# 7/7. 테스트 — 도메인 단위 + 부분 출하·청구 시나리오

> Phase 2 의 마지막 편. 테스트가 시스템의 살아있는 명세라는 관점에서, 어떤 규칙이 어떤 테스트에 박혔는지 점검.

대상 파일:

```

├─ PartialDeliveryAndInvoiceScenarioTest.java       ⭐ Phase 2 의 하이라이트
├─ order/
│  ├─ SalesOrderTest.java                           (도메인 메서드 9개)
│  └─ SalesOrderCrudIntegrationTest.java            (서비스 통합 4개)
└─ invoice/
   └─ InvoiceTest.java                              (부가세 계산 2개)
```

총 21개의 새 테스트 + Phase 1 의 18개 = 누계 39개. 모두 통과.

## 🔥 테스트가 명세 — `@DisplayName` 만 모아 보기

Phase 1 워크스루 09편처럼, **테스트 이름만 읽어도 Phase 2 가 보장하는 규칙이 한 눈에**:

```
[번호 동시성]
- 같은 날짜에 동시 수주 등록시 번호가 중복되지 않는다

[SalesOrder 도메인 — 단위 테스트]
- DRAFT 가 아닌 수주를 confirm 하면 거부된다
- 라인이 비어 있는 수주는 confirm 불가
- 확정된 수주는 라인 추가 불가
- 출하 누적이 주문량을 초과하면 거부된다
- 전량 출하되면 SHIPPED 상태로 전이된다
- 청구 누적이 출하 누적을 초과하면 거부된다 — 보낸 만큼만 청구 가능
- 출하 실적이 있는 수주는 취소할 수 없다 — SHIPPING 이후 cancel 금지
- DRAFT 또는 CONFIRMED 수주만 취소 가능
- totalAmount 는 라인 합계로 재계산된다

[SalesOrder — 통합 테스트]
- 수주를 생성하면 SO-YYYYMMDD-NNN 형식의 번호가 자동 발급된다
- 신용한도 부족이면 confirm 이 거부된다
- DRAFT 수주는 한도 영향 없음 — 같은 고객으로 한도 한 번 더 가능
- DRAFT 수주만 수정 가능 — 확정 후 수정은 거부

[Invoice — 부가세]
- 부가세는 공급가의 10퍼센트
- subtotal + tax = total 합산 정합성

[부분 출하·청구 시나리오] ⭐
- 10대 수주 → 6대 출하 → 6대 청구 → 4대 출하 → 4대 청구 전체 흐름
- 주문량 10대를 초과해 11대를 출하하려고 하면 거부된다
- 출하 취소시 SO 라인 shipped_qty 가 복원되고 헤더 상태가 CONFIRMED 로 되돌아간다
```

→ **이 21줄이 Phase 2 의 명세서**. 변경하려는 규칙이 있으면 깨질 테스트가 있고, 변경한 후 통과하면 새 규칙이 명세가 된다.

## 🔥 단위 테스트 — Spring 부트 없는 도메인 검증

```java
class SalesOrderTest {

    @Test
    @DisplayName("전량 출하되면 SHIPPED 상태로 전이된다")
    void recordShipment_전량_출하되면_SHIPPED_상태로_전이() {
        SalesOrder order = orderWith2Lines();
        order.confirm(LocalDateTime.now());
        SalesOrderLine line1 = order.getLines().get(0);  // qty=10
        SalesOrderLine line2 = order.getLines().get(1);  // qty=5

        order.recordShipment(line1, bd(6));
        assertThat(order.getStatus()).isEqualTo(SalesOrderStatus.SHIPPING);

        order.recordShipment(line1, bd(4));   // line1 완료
        assertThat(order.getStatus()).isEqualTo(SalesOrderStatus.SHIPPING);

        order.recordShipment(line2, bd(5));   // line2 완료 — 전량 출하
        assertThat(order.getStatus()).isEqualTo(SalesOrderStatus.SHIPPED);
    }
    ...
}
```

### Spring 없이 도메인만 검증
- `@SpringBootTest` 없음 — Testcontainers MySQL 도 안 띄움.
- 그냥 자바 객체를 만들어 메서드를 호출.
- `Customer.create(...)` 같은 도메인 정적 팩토리만 사용 → DB 의존 없음.

→ **빠른 피드백**. 통합 테스트는 한 번 부팅에 2~3초, 단위 테스트는 ms 단위.

### `orderWith2Lines()` 헬퍼

```java
private static SalesOrder orderWith2Lines() {
    SalesOrder order = SalesOrder.draft("SO-20260524-001", customer(), null, null, LocalDate.now());
    order.addLine(itemNotebook(), bd(10), bd(1200000));
    order.addLine(itemMonitor(), bd(5), bd(350000));
    return order;
}
```

매 테스트의 fixture 를 짧게 만들어 본문이 핵심에 집중. **테스트 본문에 줄 수가 적을수록 메시지가 명확**.

### 한 테스트가 한 규칙

```java
@DisplayName("청구 누적이 출하 누적을 초과하면 거부된다 — 보낸 만큼만 청구 가능")
void recordInvoicing_출하량_초과면_거부된다() {
    SalesOrder order = orderWith2Lines();
    order.confirm(LocalDateTime.now());
    SalesOrderLine line = order.getLines().get(0);

    order.recordShipment(line, bd(6));
    order.recordInvoicing(line, bd(6));
    assertThatThrownBy(() -> order.recordInvoicing(line, bd(1)))
            .isInstanceOf(IllegalStateException.class)
            .hasMessageContaining("출하 누적을 초과");
}
```

- 시나리오: 6개 보내고 6개 청구한 다음, 1개 더 청구 시도.
- 검증: `IllegalStateException` + 메시지에 "출하 누적을 초과".

`hasMessageContaining` 이 정확한 메시지 매칭이 아니라 **키워드 일치** — 메시지 문구를 살짝 다듬어도 테스트가 깨지지 않게.

## 🔥 `InvoiceTest` — 금액 계산의 정확성

```java
@Test
@DisplayName("부가세는 공급가의 10퍼센트")
void 부가세는_공급가의_10퍼센트() {
    SalesOrder order = orderWithLine10();
    SalesOrderLine sol = order.getLines().get(0);
    order.recordShipment(sol, new BigDecimal("6"));

    Invoice invoice = Invoice.draft("INV-20260524-001", order, LocalDate.now());
    invoice.addLine(sol, new BigDecimal("6"));

    // 6 × 1_200_000 = 7_200_000, tax = 720_000, total = 7_920_000
    assertThat(invoice.getSubtotal()).isEqualByComparingTo(new BigDecimal("7200000"));
    assertThat(invoice.getTaxAmount()).isEqualByComparingTo(new BigDecimal("720000"));
    assertThat(invoice.getTotalAmount()).isEqualByComparingTo(new BigDecimal("7920000"));
}
```

### `isEqualByComparingTo` 가 BigDecimal 비교의 핵심

```java
assertThat(invoice.getSubtotal()).isEqualByComparingTo("7200000");
   ↕ (다름)
assertThat(invoice.getSubtotal()).isEqualTo("7200000");
```

`BigDecimal.equals` 는 **scale 도 비교**한다: `7200000` (scale 0) ≠ `7200000.00` (scale 2). DB 에서 읽으면 scale 이 2, 자바 리터럴은 0 — 매번 깨짐.

`compareTo` 는 값만 비교 → 안전. **BigDecimal 단언은 항상 `isEqualByComparingTo`**.

### 합산 정합성 한 줄

```java
BigDecimal sum = invoice.getSubtotal().add(invoice.getTaxAmount());
assertThat(invoice.getTotalAmount()).isEqualByComparingTo(sum);
```

`totalAmount = subtotal + taxAmount` 라는 도메인 약속을 한 줄로 검증. 미래에 부가세 정책이 바뀌어도 이 정합성은 유지되어야 한다.

## 🔥 `PartialDeliveryAndInvoiceScenarioTest` — Phase 2 의 하이라이트

```java
@SpringBootTest
@Import(TestcontainersConfiguration.class)
class PartialDeliveryAndInvoiceScenarioTest {

    @Test
    @DisplayName("10대 수주 → 6대 출하 → 6대 청구 → 4대 출하 → 4대 청구 전체 흐름")
    void 부분_출하와_청구가_각_단계마다_누적되며_상태가_전이된다() {
        // given — 한도 충분한 고객 + 노트북
        var customer = customerService.create(...);
        var item = itemService.create(...);

        // step 1: 10대 수주 → 확정
        SalesOrderResponse order = salesOrderService.create(new SalesOrderCreateRequest(
                customer.id(), null, null, LocalDate.now(),
                List.of(new SalesOrderLineRequest(item.id(), new BigDecimal("10"), new BigDecimal("1200000")))
        ));
        salesOrderService.confirm(order.id());

        Long soId = order.id();
        Long solId = salesOrderService.findById(soId).lines().get(0).id();

        // step 2: 6대 출하 → SHIPPING
        deliveryService.create(new DeliveryCreateRequest(
                soId, LocalDate.now(),
                List.of(new DeliveryLineRequest(solId, new BigDecimal("6")))));

        var afterShip1 = salesOrderService.findById(soId);
        assertThat(afterShip1.status()).isEqualTo(SalesOrderStatus.SHIPPING);
        assertThat(afterShip1.lines().get(0).shippedQty()).isEqualByComparingTo("6");

        // step 3: 6대 인보이스 → INVOICING + 부가세 검증
        InvoiceResponse inv1 = invoiceService.create(new InvoiceCreateRequest(
                soId, LocalDate.now(),
                List.of(new InvoiceLineRequest(solId, new BigDecimal("6")))));
        assertThat(inv1.subtotal()).isEqualByComparingTo("7200000");
        assertThat(inv1.taxAmount()).isEqualByComparingTo("720000");
        assertThat(inv1.totalAmount()).isEqualByComparingTo("7920000");

        // step 4: "보내지 않은 수량은 청구 불가" 검증
        assertThatThrownBy(() -> invoiceService.create(new InvoiceCreateRequest(
                soId, LocalDate.now(),
                List.of(new InvoiceLineRequest(solId, new BigDecimal("1"))))))
                .hasMessageContaining("잔여 청구 가능량");

        // step 5: 4대 추가 출하 → SO 라인 누적 10, 상태는 여전히 INVOICING
        deliveryService.create(new DeliveryCreateRequest(...));
        assertThat(salesOrderService.findById(soId).lines().get(0).shippedQty())
                .isEqualByComparingTo("10");

        // step 6: 잔여 4대 청구 → INVOICED
        invoiceService.create(new InvoiceCreateRequest(...));

        var finalState = salesOrderService.findById(soId);
        assertThat(finalState.status()).isEqualTo(SalesOrderStatus.INVOICED);
        assertThat(finalState.lines().get(0).shippedQty()).isEqualByComparingTo("10");
        assertThat(finalState.lines().get(0).invoicedQty()).isEqualByComparingTo("10");
    }
}
```

### 한 테스트에서 검증되는 7가지

1. **부분 출하 누적** — 6 → 10 으로 단계별 갱신.
2. **부분 청구 누적** — 6 → 10 으로 단계별 갱신.
3. **상태 머신** — CONFIRMED → SHIPPING → INVOICING → INVOICED.
4. **부가세 계산** — 6대 × 120만 = 720만 + 72만 부가세 = 792만.
5. **"보낸 만큼만 청구"** — 출하 안 한 수량 청구 시도 → 거부.
6. **INVOICING 중 추가 출하** — 상태 유지.
7. **전량 청구 완료 → INVOICED 자동 전이**.

→ **OTC 흐름의 모든 도메인 규칙이 한 시나리오에 살아있다**. 어느 규칙이 깨지면 이 테스트가 깨진다.

### `@SpringBootTest` + Testcontainers

- 진짜 MySQL 컨테이너 (`mysql:8.0`) 가 띄워짐.
- Flyway 가 V1~V14 모두 실행.
- 서비스 계층을 통해 호출 — `@Transactional`, JPA 영속성, 트랜잭션 경계가 모두 운영과 동일.

→ "단위 테스트가 통과해도 통합에서 깨지는" 케이스를 잡는다. 실제로 구현 중 발견된 **Hibernate 프록시 비교 버그** (03편 `ensureLineBelongsToThis`) 가 이 통합 테스트에서 처음 잡혔다.

### `var` 의 적절한 사용

```java
var afterShip1 = salesOrderService.findById(soId);
```

타입이 길고 (`SalesOrderResponse`) 메서드명에서 의도가 분명할 때 `var` 가 시그널 노이즈를 줄인다. 자바 17+ 의 권장 패턴.

## 🔥 `SalesOrderCrudIntegrationTest` — 신용한도 흐름

```java
@Test
@DisplayName("DRAFT 수주는 한도 영향 없음 — 같은 고객으로 한도 한도 한 번 더 가능")
void DRAFT_수주는_한도_영향_없음() {
    var customerId = createCustomer(new BigDecimal("200000")).id();
    var itemId = createItem(new BigDecimal("100000")).id();

    // 한 건은 확정 (10만 차감)
    var so1 = salesOrderService.create(new SalesOrderCreateRequest(
            customerId, null, null, LocalDate.now(),
            List.of(new SalesOrderLineRequest(itemId, new BigDecimal("1"), new BigDecimal("100000")))
    ));
    salesOrderService.confirm(so1.id());

    // 또 한 건은 DRAFT 로만 두면 한도에 영향 없음
    var so2 = salesOrderService.create(new SalesOrderCreateRequest(...));
    // 남은 한도 10만 = 신규 10만, OK
    salesOrderService.confirm(so2.id());
    assertThat(salesOrderService.findById(so2.id()).status()).isEqualTo(SalesOrderStatus.CONFIRMED);
}
```

설계서 §5.1 "DRAFT 는 한도 영향 없음" 의 검증. 위 시나리오:
- 한도 20만.
- 1번 수주 확정 (10만 차감) → 남은 한도 10만.
- 2번 수주 DRAFT → 한도 영향 X.
- 2번 수주 확정 시도 → "신규 10만 ≤ 남은 10만" → 통과.

만약 DRAFT 도 한도에 영향 주면 이 테스트가 깨진다 — **한도 정책이 명세에 박혀 있다**.

## 🔥 `TransactionNumberConcurrencyTest` — 동시성 보장

```java
@Test
@DisplayName("같은 날짜에 동시 수주 등록시 번호가 중복되지 않는다")
void 같은_날짜에_동시_수주_등록시_번호가_중복되지_않는다() throws ... {
    int threads = 50;
    LocalDate today = LocalDate.now();
    ExecutorService pool = Executors.newFixedThreadPool(threads);
    CountDownLatch startGate = new CountDownLatch(1);

    List<Future<String>> futures = IntStream.range(0, threads).mapToObj(i -> pool.submit(() -> {
        startGate.await();
        return generator.nextSalesOrderNumber(today);
    })).toList();

    startGate.countDown();
    pool.shutdown();
    boolean finished = pool.awaitTermination(60, TimeUnit.SECONDS);

    Set<String> issued = new HashSet<>();
    for (Future<String> f : futures) issued.add(f.get());
    assertThat(issued).hasSize(threads);   // 모두 unique
}
```

Phase 1 의 `CodeGeneratorConcurrencyTest` 와 동일한 패턴 — `CountDownLatch` 로 50 스레드를 같은 순간에 출발시켜 모든 발급 번호가 unique 함을 검증.

→ 01편의 일반화 (`year` → `periodKey`) 가 동시성 안전성을 깨뜨리지 않았음을 검증.

## 🔥 `BigDecimal` + nanoTime 으로 unique business no 생성

```java
private static final java.util.concurrent.atomic.AtomicLong SEQ =
        new java.util.concurrent.atomic.AtomicLong(System.nanoTime());

private static String uniqueBusinessNo() {
    long n = SEQ.incrementAndGet();
    return String.format("%03d-%02d-%05d",
            (int) ((n / 10_000_000L) % 900) + 100,
            (int) ((n / 100_000L) % 100),
            (int) (n % 100_000L));
}
```

처음에는 `static int counter` 패턴이었지만, 두 테스트 클래스가 같은 패턴을 쓰면서 **둘 다 `200-01-00001` 부터 시작** → 첫 테스트끼리 충돌.

수정안:
- 시드를 `System.nanoTime()` 으로 — JVM 마다, 클래스마다, 매번 다른 시작점.
- `AtomicLong` 으로 thread-safe.
- 형식은 사업자번호 3-2-5 패턴.

→ 학습 단계에서 자주 마주치는 "한 JVM 안에서 데이터 충돌" 함정. **테스트 격리는 픽스처 설계의 일부**.

## 🔥 단위 vs 통합 — 어떻게 나눴나

| 검증 대상 | 단위 (`@Test` only) | 통합 (`@SpringBootTest`) |
| --- | --- | --- |
| 도메인 상태 머신 | ✅ `SalesOrderTest` | |
| 부가세 계산 | ✅ `InvoiceTest` | |
| 잔여 출하/청구 검증 | ✅ `SalesOrderTest` | ✅ 시나리오 |
| 트랜잭션 번호 발급 | | ✅ |
| 신용한도 | | ✅ `SalesOrderCrudIntegrationTest` |
| 부분 흐름 전체 | | ✅ `PartialDeliveryAndInvoiceScenarioTest` |
| 동시성 | | ✅ `TransactionNumberConcurrencyTest` |

원칙:
- **단위**: 외부 의존 없는 도메인 규칙 (상태 머신, 누적 검증, 부가세).
- **통합**: DB/Repository/번호 발급기/Hibernate 프록시가 필요한 경우.

→ 단위로 가능하면 단위로. 통합 테스트 비용 (부팅 시간, 데이터 충돌) 을 아낀다.
