# Phase 2 시연 가이드

> 7단계 사이클의 마지막 — **단계 7: 시연 + 회고**.
> 도메인 브리핑 → 모델링 → 코드 워크스루(7편)로 OTC 흐름의 설계를 머리로 이해했다.
> 이번에는 직접 Swagger UI / cURL / 통합 테스트로 **OTC 흐름의 6대 규칙**이 실제로 동작하는지 손으로 확인한다.

---

## 시연의 목표 (Phase 2가 무엇을 보장하는가)

Phase 1에서는 마스터 데이터의 5대 특성(자동 코드/Soft Delete/Auditing/페이징/참조 무결성)을 확인했다. Phase 2에서는 **트랜잭션 데이터**가 처음 등장하며, 마스터를 참조해 OTC(Order-to-Cash) 흐름의 절반(견적~인보이스)이 동작한다.

이번에 확인할 **6대 규칙**:

1. **트랜잭션 번호 자동 발급** — `Q-YYYYMMDD-NNN`, `SO-YYYYMMDD-NNN`, `DLV-YYYYMMDD-NNN`, `INV-YYYYMMDD-NNN` 형식이 날짜별로 자동 발급된다 (마스터는 연도별, 트랜잭션은 일자별로 시퀀스 분리)
2. **헤더-라인 구조** — 한 수주에 여러 라인, `cascade=ALL` 로 헤더 저장 시 라인까지 함께 영속화
3. **상태 머신** — DRAFT → CONFIRMED → SHIPPING → SHIPPED → INVOICING → INVOICED. 단계 건너뛰기/역행 금지
4. **신용한도 검증** — DRAFT는 한도 영향 없음, **CONFIRMED 시점에 차감 누계 + 신규 금액이 한도를 초과하면 거부**
5. **부분 출하/청구 누적** — 한 수주 라인에 출하/청구가 여러 번 누적되며, 출하/청구 누적량이 주문량을 넘으면 거부. **"보낸 만큼만 청구"** 규칙
6. **취소의 정합성** — 출하 취소 시 SO 라인의 `shipped_qty`가 복원되고 헤더 상태가 되돌아간다 (CONFIRMED). 출하 실적이 있는 수주는 직접 cancel 불가

거기에 V14로 시드된 **DRAFT 견적 1건 + DRAFT 수주 1건(노트북 10대)** 이 보이는지도 확인한다.

## STEP 2. 시드 트랜잭션 + 시퀀스 상태 확인

```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT 'quotation' AS t, COUNT(*) FROM quotation
  UNION ALL SELECT 'quotation_line', COUNT(*) FROM quotation_line
  UNION ALL SELECT 'sales_order',    COUNT(*) FROM sales_order
  UNION ALL SELECT 'sales_order_line', COUNT(*) FROM sales_order_line
  UNION ALL SELECT 'delivery',       COUNT(*) FROM delivery
  UNION ALL SELECT 'invoice',        COUNT(*) FROM invoice;"
```

기대:
```
+------------------+----------+
| t                | COUNT(*) |
+------------------+----------+
| quotation        |        1 |   ← V14 시드
| quotation_line   |        2 |   ← 노트북 5대 + 모니터 3대
| sales_order      |        1 |   ← V14 시드
| sales_order_line |        1 |   ← 노트북 10대 (시연용)
| delivery         |        0 |
| invoice          |        0 |
+------------------+----------+
```

시퀀스 상태:
```bash
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db -e "
  SELECT prefix, period_key, next_number FROM code_sequence ORDER BY prefix;"
```

기대 (오늘이 2026-05-24 가정):
```
+--------+------------+-------------+
| prefix | period_key | next_number |
+--------+------------+-------------+
| CUST   | 2026       |           3 |
| EMP    | 2026       |           4 |
| ITEM   | 2026       |           3 |
| Q      | 20260524   |           2 |   ← V14가 001 사용 → next=2
| SO     | 20260524   |           2 |   ← V14가 001 사용 → next=2
| VEND   | 2026       |           2 |
+--------+------------+-------------+
```

확인 포인트:
- **`period_key` 형식이 prefix 별로 다르다** — CUST/ITEM/VEND/EMP 는 4자리 연도, Q/SO 는 8자리 일자. 한 컬럼이 두 정책을 모두 담는다 (01편 워크스루 §2 참조).
- DLV/INV 행은 아직 없다. 첫 출하/인보이스 등록 시점에 자동 생성된다.

## STEP 4. 견적(Quotation) — 발송과 수락

V14 시드된 DRAFT 견적의 ID 가 `1` 이라고 가정 (`GET /api/quotations` 로 확인).

### 4-1. 시드 견적 조회

```bash
curl -s "http://localhost:8080/api/quotations/1" | jq
```

기대 (요약):
```json
{
  "id": 1,
  "number": "Q-20260524-001",
  "status": "DRAFT",
  "customerId": 1,
  "totalAmount": 7050000.00,
  "lines": [
    { "lineNo": 1, "itemId": 1, "quantity": 5, "unitPrice": 1200000, "lineTotal": 6000000 },
    { "lineNo": 2, "itemId": 2, "quantity": 3, "unitPrice": 350000,  "lineTotal": 1050000 }
  ]
}
```

확인 포인트:
- **`totalAmount` 가 라인 합계와 일치** — 6,000,000 + 1,050,000 = 7,050,000. 도메인 메서드 `recomputeTotal()` 의 결과.

### 4-2. 발송 (DRAFT → SENT)

```bash
curl -i -X POST "http://localhost:8080/api/quotations/1/send"
```

기대: **200 OK** + `"status": "SENT"`.

### 4-3. 수락 (SENT → ACCEPTED)

```bash
curl -i -X POST "http://localhost:8080/api/quotations/1/accept"
```

기대: **200 OK** + `"status": "ACCEPTED"`.

### 4-4. 상태 머신 위반 — DRAFT 가 아닌데 수정 시도

```bash
curl -i -X POST "http://localhost:8080/api/quotations/1/send"   # 이미 ACCEPTED
```

기대:
```
HTTP/1.1 409 Conflict
{
  "title": "Conflict",
  "status": 409,
  "detail": "DRAFT 상태의 견적만 발송할 수 있습니다 — 현재 상태: ACCEPTED"
}
```

확인 포인트:
- **상태 머신은 도메인 메서드에서 검증** — Service 가 아니라 `Quotation.send()` / `accept()` 안에 if 가 박혀 있다 (03편 워크스루 참조).
- 한 번 전이한 상태는 되돌릴 수 없음. 취소(`/cancel`) 는 별도 경로.

## STEP 5. 수주(Sales Order) — 신용한도 검증

여기가 Phase 2 의 첫 번째 하이라이트.

### 5-1. 신원전자(`CUST-2026-0001`)의 현재 신용한도 확인

```bash
curl -s "http://localhost:8080/api/customers/1" | jq '{name, creditLimit}'
```

기대 — `신원전자`, `creditLimit: 50000000.00` 같은 큰 값 (시드 기준). 이미 V14 시드된 1,200만원 DRAFT 수주는 한도에 영향 없음 (DRAFT 이므로).

### 5-2. 시드된 DRAFT 수주 확정 (10대 × 120만 = 1,200만 차감)

```bash
curl -i -X POST "http://localhost:8080/api/sales-orders/1/confirm"
```

기대:
```
HTTP/1.1 200 OK
{
  "id": 1,
  "number": "SO-20260524-001",
  "status": "CONFIRMED",
  "totalAmount": 12000000.00,
  "lines": [
    { "id": 1, "itemId": 1, "orderQty": 10, "shippedQty": 0, "invoicedQty": 0, ... }
  ]
}
```

확인 포인트:
- 상태가 **DRAFT → CONFIRMED**.
- 이 시점부터 **신용한도에서 1,200만이 차감되어 잡힌다** (실제로 차감하는 게 아니라 "확정된 미회수 매출 합계" 로 계산).

### 5-3. 한도 초과 수주 — 새 고객 만들고 시도

신원전자는 한도가 크니, **작은 한도의 신규 고객**을 만들어 한도 위반을 본다.

```bash
# 한도 50만원짜리 고객 만들기
curl -s -X POST "http://localhost:8080/api/customers" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "소한도사",
    "businessNo": "777-66-55555",
    "creditLimit": 500000,
    "paymentTerms": "COD"
  }' | jq '{id, code, creditLimit}'
```

응답에서 `id` (예: 4) 와 `code` 를 기억.

```bash
# 100만원짜리 수주 DRAFT 생성
curl -s -X POST "http://localhost:8080/api/sales-orders" \
  -H "Content-Type: application/json" \
  -d '{
    "customerId": 4,
    "orderDate": "2026-05-24",
    "lines": [
      { "itemId": 1, "orderQty": 1, "unitPrice": 1000000 }
    ]
  }' | jq '{id, number, status, totalAmount}'
```

기대: DRAFT 로 생성 성공 (한도 검증은 confirm 시점). 여기서 응답된 `id` (예: 2) 기억.

```bash
# DRAFT → CONFIRMED 시도 — 한도 50만 < 신규 100만 → 거부
curl -i -X POST "http://localhost:8080/api/sales-orders/2/confirm"
```

기대:
```
HTTP/1.1 409 Conflict
{
  "title": "Conflict",
  "status": 409,
  "detail": "신용한도 초과: 한도=500000, 차감 누계=0, 신규=1000000"
}
```

확인 포인트:
- **DRAFT 등록은 통과, CONFIRMED 시점에 거부** — 영업이 자유롭게 견적/주문 입력은 하되, 확정은 한도 정책으로 통제. 실무 패턴.
- 메시지 형식: "한도 / 현재 차감 누계 / 신규" 3개가 들어가 운영자가 즉시 원인 파악 가능.

### 5-4. DRAFT 수주 수정 (라인 추가/단가 변경)

```bash
curl -i -X PUT "http://localhost:8080/api/sales-orders/2" \
  -H "Content-Type: application/json" \
  -d '{
    "orderDate": "2026-05-24",
    "lines": [
      { "itemId": 1, "orderQty": 1, "unitPrice": 400000 }
    ]
  }'
```

기대: 200 OK, `totalAmount: 400000`.

```bash
# 다시 확정 시도 — 이번엔 한도 50만 ≥ 신규 40만 → 통과
curl -i -X POST "http://localhost:8080/api/sales-orders/2/confirm"
```

기대: **200 OK** + `status: CONFIRMED`.

```bash
# 확정 후 수정 시도 — 거부
curl -i -X PUT "http://localhost:8080/api/sales-orders/2" \
  -H "Content-Type: application/json" \
  -d '{ "orderDate": "2026-05-24", "lines": [{ "itemId": 1, "orderQty": 2, "unitPrice": 400000 }] }'
```

기대:
```
HTTP/1.1 409 Conflict
detail: "DRAFT 상태의 수주만 수정할 수 있습니다 — 현재 상태: CONFIRMED"
```

확인 포인트:
- **DRAFT 만 수정 가능** — 확정 이후는 수정 금지. 회계/감사 추적이 무너지지 않게 막는 핵심 규칙.

## STEP 6. 부분 출하 (Phase 2 의 핵심)

`SO-20260524-001` (시드, 노트북 10대, 라인 ID=1) 를 사용한다. 라인 ID는 `GET /api/sales-orders/1` 로 확인 (응답 `lines[0].id`).

### 6-1. 6대 출하

```bash
# salesOrderLineId 는 위에서 확인한 값으로 (예시 1)
curl -i -X POST "http://localhost:8080/api/deliveries" \
  -H "Content-Type: application/json" \
  -d '{
    "salesOrderId": 1,
    "shippedDate": "2026-05-24",
    "lines": [
      { "salesOrderLineId": 1, "quantity": 6 }
    ]
  }'
```

기대:
```
HTTP/1.1 201 Created
{
  "id": 1,
  "number": "DLV-20260524-001",
  "salesOrderId": 1,
  "status": "SHIPPED",
  "lines": [{ "salesOrderLineId": 1, "quantity": 6 }]
}
```

수주 상태도 같이 바뀜:
```bash
curl -s "http://localhost:8080/api/sales-orders/1" | jq '{status, "line0": .lines[0] | {orderQty, shippedQty, invoicedQty}}'
```

기대:
```json
{
  "status": "SHIPPING",
  "line0": { "orderQty": 10, "shippedQty": 6, "invoicedQty": 0 }
}
```

확인 포인트:
- **수주 헤더 상태 = `SHIPPING`** (전량 출하 전). `recomputeStatus()` 가 라인 누적을 보고 파생 (03편 워크스루 참조).
- **`shippedQty: 6`** — SO 라인에 출하 누적이 직접 기록된다 (Delivery 라인을 매번 합산하지 않음).

### 6-2. 주문량 초과 출하 시도 — 거부

```bash
curl -i -X POST "http://localhost:8080/api/deliveries" \
  -H "Content-Type: application/json" \
  -d '{
    "salesOrderId": 1,
    "shippedDate": "2026-05-24",
    "lines": [
      { "salesOrderLineId": 1, "quantity": 5 }
    ]
  }'
```

기대:
```
HTTP/1.1 409 Conflict
detail: "출하 누적이 주문량을 초과합니다 — 주문=10, 기출하=6, 신규=5"
```

확인 포인트:
- 남은 출하 가능량 = 10 - 6 = 4. 5 시도 → 거부.
- 메시지에 주문량 / 기출하 / 신규 3개가 들어가 잔여량 계산이 즉시 가능.

## STEP 7. 부분 청구 — "보낸 만큼만"

### 7-1. 6대 청구 (출하한 만큼)

```bash
curl -i -X POST "http://localhost:8080/api/invoices" \
  -H "Content-Type: application/json" \
  -d '{
    "salesOrderId": 1,
    "invoiceDate": "2026-05-24",
    "lines": [
      { "salesOrderLineId": 1, "quantity": 6 }
    ]
  }'
```

기대:
```
HTTP/1.1 201 Created
{
  "id": 1,
  "number": "INV-20260524-001",
  "status": "ISSUED",
  "subtotal": 7200000.00,
  "taxAmount": 720000.00,
  "totalAmount": 7920000.00,
  "lines": [...]
}
```

확인 포인트 — **부가세 정합성**:
- `subtotal = 6 × 1,200,000 = 7,200,000`
- `taxAmount = subtotal × 10% = 720,000`
- `totalAmount = subtotal + taxAmount = 7,920,000`

수주 상태:
```bash
curl -s "http://localhost:8080/api/sales-orders/1" | jq '{status, "line0": .lines[0] | {shippedQty, invoicedQty}}'
```

기대: `status: "INVOICING"`, `invoicedQty: 6`.

### 7-2. 출하 안 한 수량 청구 시도 — 거부

```bash
curl -i -X POST "http://localhost:8080/api/invoices" \
  -H "Content-Type: application/json" \
  -d '{
    "salesOrderId": 1,
    "invoiceDate": "2026-05-24",
    "lines": [
      { "salesOrderLineId": 1, "quantity": 1 }
    ]
  }'
```

기대:
```
HTTP/1.1 409 Conflict
detail: "잔여 청구 가능량을 초과합니다 — 출하=6, 기청구=6, 신규=1"
```

확인 포인트:
- "**보낸 만큼만 청구**" — 현재 출하 6 = 청구 6, 추가 청구하려면 추가 출하부터 해야 한다.
- 회계상 "발송하지도 않은 매출"이 잡히지 않도록 막는 핵심 규칙.

### 8-1. 4대 추가 출하

```bash
curl -s -X POST "http://localhost:8080/api/deliveries" \
```

## STEP 9. 출하 취소 — 복원 정합성

> 위에서 이미 전량 마감했으므로, **이 단계는 새 SO 를 하나 만들어서** 시연한다.

### 9-1. 새 수주 + 출하 만들기

```bash

NEW_SO=$(curl -s -X POST "http://localhost:8080/api/sales-orders" \

```

기대: `status: "SHIPPING"`, `shipped: 2`.

### 9-2. 출하 취소

```bash

```

기대: **200 OK** + `"status": "CANCELLED"`.

수주 상태 복원 확인:
```bash

```

기대: **`status: "CONFIRMED"`** (출하 누적 0 → SHIPPING 이전 상태), `shipped: 0`.

확인 포인트:
- 출하 취소가 **SO 라인의 `shipped_qty` 복원 + 헤더 상태 재계산** 까지 한 트랜잭션으로 처리됨.

### 9-3. 출하 실적이 있는 수주는 cancel 불가 (단위 테스트로 검증된 규칙)

확인 포인트:
- 수주 취소는 **출하 이전 (DRAFT/CONFIRMED) 에서만** 가능. 이미 출하 흔적이 있으면 자식부터 정리해야 한다.

## 트러블슈팅

### 1. 견적/수주 ID 가 시연 예시와 다르다
- `GET /api/quotations` / `GET /api/sales-orders` 로 실제 ID 를 확인하고 명령어의 `1`, `2` 부분을 바꿔서 실행.
- Phase 1 시연에서 추가 등록을 했다면 customer/item ID 도 다를 수 있으니 함께 확인.

### 2. `409 Conflict — 신용한도 초과` 가 예상 안 한 곳에서 발생
- 신원전자(`CUST-2026-0001`) 의 한도가 어떤 값인지 시드(`V8__seed_master_data.sql`) 또는 현재 DB 에서 직접 확인.
- 한도가 작게 시드되어 있으면 STEP 5-2 의 1,200만 차감만으로도 한도 초과가 날 수 있다.

### 3. 부가세 계산이 1원 차이가 난다 (`isEqualByComparingTo` 가 아니라 `isEqualTo` 사용 시)
- `BigDecimal.equals` 는 scale 도 비교 — `7200000` (scale 0) ≠ `7200000.00` (scale 2). DB에서 읽으면 scale 2.
- 단정문은 항상 `isEqualByComparingTo`. (07편 워크스루 § "InvoiceTest" 참조)

### 4. `400 Bad Request` — `salesOrderLineId` 가 null 이라는 메시지
- 출하/인보이스 요청 본문은 **SO 라인 ID** (수주의 라인 PK) 를 보내야 한다. SO ID 가 아님.
- `GET /api/sales-orders/{soId}` 응답의 `lines[].id` 가 그것.

### 5. `409 Conflict — DRAFT 상태의 수주만 ...` 메시지
- 의도된 거부. 한 번 confirm 한 수주는 수정 불가. 한도 영향만 보고 싶었다면 새 수주를 DRAFT 로 다시 만들어 테스트.

## 시연 후 — 무엇이 학습됐는가

Phase 2 시연으로 **체감**으로 알게 된 것:

1. **"트랜잭션은 마스터 위에 쌓이는 시간 축의 사건"** — 견적/수주/출하/인보이스는 마스터(고객/상품)를 참조하면서 발생일이 함께 박힌 데이터다. 그래서 시퀀스도 날짜별로 끊는다 (`SO-20260524-NNN`). 같은 날 안에서만 번호가 의미를 가진다.
2. **"상태 머신은 정적인 컬럼이 아니라 메서드"** — `status` 컬럼이 있긴 하지만, 그것을 바꿀 수 있는 것은 도메인 메서드 (`confirm()`, `recordShipment()`, ...) 뿐이다. Service 가 직접 `setStatus` 를 호출하지 않는다. 이게 무너지면 "DRAFT 인데 출하됐다" 같은 상태가 가능해진다.
3. **"부분 출하·청구의 누적값은 SO 라인이 들고 있다"** — Delivery 라인을 매번 합산하는 대신, SO 라인이 `shippedQty` / `invoicedQty` 를 직접 갱신한다. 조회 성능 + 검증 단순성. 단, 취소 시 복원도 반드시 같은 트랜잭션에서 처리해야 함.
4. **"한도는 DRAFT 가 아니라 CONFIRMED 에서 검증한다"** — 영업이 자유롭게 견적/주문 입력은 가능, 회사가 책임지는 시점은 확정. 실무 ERP 의 표준 패턴.
5. **"보낸 만큼만 청구"** — 매출 인식 시점의 핵심 규칙. 출하 ≠ 청구 가 가능해야 부분 청구를 처리할 수 있고, 그래서 둘을 분리한 라인 누적값을 두 개 (`shippedQty`, `invoicedQty`) 만든다.
