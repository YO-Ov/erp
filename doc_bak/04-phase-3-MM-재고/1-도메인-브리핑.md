# Phase 3 도메인 브리핑 — MM (자재 / 재고와 입출고)

> 7단계 사이클의 **단계 1: 도메인 브리핑**.
> 이 단계의 목적은 "코드를 짜기 전에, 이 모듈이 현실에서 무엇을 하고, 왜 그렇게 설계되는가" 를 글로 이해하는 것.
> 충분히 읽고 모호한 부분이 있으면 **단계 2 Q&A** 에서 질문 — 이해 안 된 채로 코드 단계로 넘어가지 않는다.

## 0. 한 줄 요약

> **"실제 창고에 무엇이 몇 개 있는가" 를 단 하나의 진실로 관리하는 모듈.**
> Phase 2 의 출하(Delivery) 는 SO 라인의 `shippedQty` 만 갱신했다 — **그 출하가 실제 창고 재고를 차감하지는 않았다**. Phase 3 에서 그 빈칸을 메운다.

---

## 1. MM 모듈이 현실에서 하는 일

회사의 **창고** 에는 매일 일이 벌어진다:

| 사건 | 누가 | 결과 |
| --- | --- | --- |
| 매입 입고 | 구매팀이 거래처에서 부품/완제품을 받음 | 창고 재고 **증가** |
| 출하 출고 | 영업팀의 출하를 창고가 실제로 보냄 | 창고 재고 **감소** |
| 창고 간 이동 | 본사 창고 → 지방 창고로 이동 | A 창고 감소, B 창고 증가 |
| 재고 실사 조정 | 월말에 실물 카운트와 시스템 차이 보정 | 양/음 둘 다 가능 |
| 폐기 | 불량/유효기간 만료 | 감소 |

MM 모듈은 이 모든 사건을 **"재고 변동" 한 종류의 이벤트로 추상화** 한다. 즉, "왜 변했는가(reason)" 만 다르고, "**언제 / 어떤 창고에서 / 어떤 상품이 / 얼마나 / 어느 방향으로 변했는가**" 는 모두 똑같이 한 테이블 (`stock_movement`) 에 행으로 쌓인다.

---

## 2. 핵심 개념 4가지

### 2-1. 창고 (Warehouse)
물건이 실제로 있는 장소. 같은 상품이라도 **어느 창고에 있는지** 가 다르면 다른 재고. 현우전자 시연용으로는 본사창고 (`WH-HQ`) 하나만으로 시작하지만, 모델은 N 개를 가정.

### 2-2. 재고 (Stock) — *상품 × 창고* 의 현재 보유량
```
( item_id, warehouse_id ) → qty_on_hand
```
- 노트북 / 본사창고 → 23 대
- 노트북 / 지방창고 → 0 대  ← 행 자체가 없을 수도 있고 0 인 행이 있을 수도 있음 (설계 결정 포인트)
- 모니터 / 본사창고 → 7 대

→ 이게 **"현재 진실"**. 화면이나 API 가 "재고 얼마 남았어요?" 물을 때 즉시 답을 주는 캐시된 값.

### 2-3. 재고 이동 (Stock Movement) — *과거의 모든 변화 기록*
```
( id, item_id, warehouse_id, qty_delta, reason, ref_id, moved_at )
```
- `qty_delta` = +5 (입고) / -3 (출고) / 등
- `reason` = `GOODS_RECEIPT` / `GOODS_ISSUE` / `TRANSFER_OUT` / `TRANSFER_IN` / `ADJUSTMENT` / `SCRAP`
- `ref_id` = 어느 거래에서 발생했는가 (출하 ID, 입고 ID 등)

→ 이게 **"역사 / 감사 추적"**. `SUM(qty_delta) WHERE item_id=A AND warehouse_id=W` 가 이론적으로 `stock.qty_on_hand` 와 항상 같아야 한다 (정합성 검증의 기준).

### 2-4. 입고 (Goods Receipt) / 출고 (Goods Issue)
**Stock Movement 를 발생시키는 *행위*** 의 두 가지 주요 유형.

- 입고: 매입처에서 물건을 받았다는 사건. 헤더-라인 구조 (Phase 2 와 동일 패턴).
- 출고: Phase 2 의 `Delivery` 가 "출고 지시" 였다면, MM 의 `Goods Issue` 는 "**실제 창고에서 물건이 빠져나가는 행위**".

→ Phase 4 에서 둘이 엮인다: 출하 확정 = 출고 실행 = 재고 차감 = Stock Movement 행 추가.

## 3. Phase 2 와 결정적으로 다른 점 — **동시성**

Phase 2 의 신용한도는 한 고객 단위로 검증했다. 두 사람이 동시에 같은 고객한테 수주를 넣는 일은 흔치 않다.

**재고는 다르다.** 같은 노트북 재고 (item=노트북, warehouse=본사창고) 를 **두 출하가 동시에 차감** 하려는 일이 매일 벌어진다. 만약 보호 없이 그냥 처리하면:

```
초기 상태: 노트북 재고 = 10대

스레드 A (출하 X 처리)         스레드 B (출하 Y 처리)
─────────────────────────────────────────────────
SELECT qty FROM stock          (qty=10)
                               SELECT qty FROM stock  (qty=10)
qty - 6 = 4
UPDATE qty = 4
                               qty - 7 = 3   ← 10에서 뺐다고 생각!
                               UPDATE qty = 3   ← 4 가 덮어쓰임!

결과: 6 + 7 = 13 대를 보냈는데 재고는 3.
실제로는 음수가 되어야 하는데, 그게 보호되지 않음.
```

이걸 막는 두 가지 패턴 — **Phase 3 의 학습 주제 1순위**:

### 3-1. 비관적 락 (`PESSIMISTIC_WRITE`, SELECT ... FOR UPDATE)
- "내가 쓸 거니까, 다른 사람은 일단 기다려."
- A 가 SELECT 하면 DB 가 그 행에 락을 건다. B 의 SELECT 는 A 의 트랜잭션 끝날 때까지 **대기**.
- **장점**: 충돌 자체가 발생 안 함. 가장 안전.
- **단점**: 동시성 떨어짐. 대기 → 응답 시간 길어짐 → 데드락 위험.

### 3-2. 낙관적 락 (`@Version`, 버전 컬럼)
- "충돌은 드물 것. 일단 양쪽 다 진행하고, 충돌 났으면 한쪽이 실패하고 재시도."
- 행에 `version` 컬럼이 있어서 UPDATE 시 `WHERE version = N` 조건. 다른 사람이 먼저 갱신했으면 `version` 이 N+1 이라 0 row affected → 실패.
- **장점**: 동시성 좋음. 락 대기 없음.
- **단점**: 충돌 시 재시도 로직이 클라이언트/서비스 책임.

Phase 3 는 **두 패턴을 모두 구현해 직접 비교** 한다. "재고 차감" 같은 핵심 경로는 비관적 락이 정답인 경우가 많지만, "기준 정보 변경 충돌" 같은 경우는 낙관적 락이 더 자연스럽다. **언제 무엇을 쓰는가** 의 판단 기준을 직접 손에 잡는 게 학습의 핵심.

## 4. 학습 계획서가 정한 Phase 3 의 학습 포인트

1. **재고 부족 시 출고 거부** — 가용 재고 ≥ 요청 수량 검증, 부족 시 409 (Phase 2 의 신용한도와 같은 패턴).
2. **Stock Movement 이력으로 추적성** — 현재 재고는 캐시, 과거는 누적 이력.
3. **동시성 보호** — 비관적 락 vs 낙관적 락 비교 학습 (위 §3).
4. **재고 평가 방법 개념** — 선입선출(FIFO) vs 평균법(Weighted Average) 차이만 글로 이해. **구현은 평균법만**.

> 평가 방법이 왜 중요한가: 재고 수량은 같아도 "원가가 얼마짜리인가" 는 입고 시점마다 다르다 — 매입가가 변하기 때문. Phase 5 (FI 회계) 에서 매출원가 (COGS) 를 계산하려면 "이번에 나간 노트북의 원가" 가 얼마인지 결정 규칙이 필요. FIFO 는 "먼저 들어온 것부터", 평균법은 "가중평균". 학습용으로는 평균법만.

## 5. 만들 것 — 한눈에

| 만들 것 | 종류 | 핵심 컬럼/관계 |
| --- | --- | --- |
| `Warehouse` 마스터 | Phase 1 식 마스터 | code, name, address, status |

→ Phase 2 의 견적/수주/출하/인보이스 와 같은 **헤더-라인 패턴**이 반복된다. Phase 2 에서 쌓은 패턴이 그대로 재사용된다는 게 ERP 학습의 누적성.

| 시점 | 동작 |

## 7. 사용자가 이번 브리핑에서 잡아야 할 핵심 5가지

1. **"현재 재고" 는 캐시이고, "재고 이동 이력" 이 진실의 원천**. 라운드 5 의 "SO 라인 `shippedQty` = 캐시 / Delivery 행 = 진실" 패턴과 같은 구조.
2. **재고 차감은 반드시 동시성 보호가 필요하다** — 비관적 락(가장 흔한 정답) vs 낙관적 락의 차이.
3. **`stock_movement.qty_delta` 의 부호 하나가 입고/출고/조정을 다 표현** — 추상화의 힘. 새 이동 유형이 생겨도 테이블 구조 안 바꿈.
4. **Phase 2 와 같은 헤더-라인 패턴이 GoodsReceipt / GoodsIssue 에서도 반복** — ERP 의 모든 트랜잭션이 결국 이 모양.
5. **Phase 3 는 MM 만 완성. Phase 4 에서 SD ↔ MM 이 엮인다** — 출하 한 번이 두 모듈을 동시에 바꾸는 흐름이 다음 학습의 핵심.
