# 1/7. 창고 마스터 — Phase 1 패턴의 재사용

> Phase 3 의 시작점. 새 모듈 `mm/` 의 첫 엔티티가 **새로운 게 아니라 익숙한 것** 이라는 사실 자체가 학습 포인트.
> 설계서 §0의 핵심 결정 #2 "`Warehouse` 는 `BaseEntityWithCode` 상속 + 코드 수동 입력 (Department 와 같은 방침)" 의 실제 구현.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/mm/warehouse/
├─ Warehouse.java
├─ WarehouseRepository.java
├─ WarehouseService.java
├─ WarehouseController.java
├─ WarehouseMapper.java
├─ WarehouseSpecifications.java
└─ dto/
   ├─ WarehouseCreateRequest.java
   ├─ WarehouseUpdateRequest.java
   └─ WarehouseResponse.java
```

---

## 🔥 새 Phase 의 첫 코드가 새로운 코드가 아닌 이유

`mm/warehouse/` 의 10개 파일을 펼쳐 보면 **Phase 1 의 `master/department/` 와 거의 똑같다**. 의도된 결과:

- 베이스: `BaseEntityWithCode` 그대로 (id + code + status + audit + soft delete).
- Soft Delete: `@SQLDelete` + `@SQLRestriction` 패턴 그대로.
- 코드 발급: 수동 입력 (`WH-HQ`, `WH-BUS`) — Department 의 `DEPT-SALES` 와 같은 방침.
- CRUD 6종 + 페이징 + 필터 Specification.
- DTO 3종(Create/Update/Response) + MapStruct.

**왜 그대로 베끼는가** — 마스터 데이터의 본질이 같기 때문. "참조되는 기준 정보 + 함부로 못 삭제 + 활성/휴면 상태 분리" 라는 3박자가 Customer, Item, Vendor, Department, Warehouse 모두 동일. **베이스 클래스 + 컨벤션 한 번 잘 만들어 두면 새 마스터는 30분 작업** 이라는 게 Phase 1 의 누적 효과.

> 💡 ERP 학습의 누적성 — Phase 가 쌓일수록 새로 배워야 할 부분이 줄어든다. Phase 3 의 진짜 학습은 다음 편들(Stock/락/평균)에 모여 있고, Warehouse 는 그저 "MM 의 토대 마스터 한 줄" 이다.

## 🔥 `Warehouse.java` — 4가지 필드의 의미

```java
@Entity
@Table(name = "warehouse")
@SQLDelete(sql = "UPDATE warehouse SET deleted_at = NOW() WHERE id = ?")
@SQLRestriction("deleted_at IS NULL")
public class Warehouse extends BaseEntityWithCode {

    @Column(name = "name", nullable = false, length = 200)
    private String name;

    @Column(name = "address", length = 500)
    private String address;
    ...
}
```

자기 필드는 단 2개 (`name`, `address`). 나머지는 베이스에서:
- `id`, `created_at/by`, `updated_at/by` ← `BaseEntity`
- `code`, `status`, `deleted_at` ← `BaseEntityWithCode`

→ 새 마스터의 추가 컬럼만 자기 클래스에 선언. Phase 1 학습 메모 "공통 컬럼은 베이스로, 도메인 컬럼만 자기 자리" 의 결과.

### `address` 가 nullable 인 이유

`@Column(name = "address", length = 500)` — `nullable = false` 가 없다. Customer 와 같은 방침:
- 단일 필드 (구조화된 주소가 아닌 자유 입력 문자열).
- 학습 단계에서는 "주소가 없는 창고" 도 등록 가능 (예: 본사 안 구역).
- 실무에서는 주소가 NOT NULL 이 자연스럽지만, 학습 범위 단순화.

### Soft Delete 어노테이션의 테이블명 직접 박기

```java
@SQLDelete(sql = "UPDATE warehouse SET deleted_at = NOW() WHERE id = ?")
```

`#{table}` 같은 매크로는 JPA 표준이 아니라서 각 엔티티가 자기 테이블명을 직접 적는다. Phase 1 메모대로 — `BaseEntityWithCode` 에 한 줄로 묶을 수 없는 이유.

## 🔥 코드 수동 입력 — Department 와 같은 방침

```java
public static Warehouse create(String code, String name, String address) {
    validate(code, name);
    Warehouse w = new Warehouse();
    w.assignCode(code);   // ← 외부에서 받은 코드를 그대로 박음
    w.name = name;
    w.address = address;
    return w;
}
```

`CodeGenerator.nextCode(...)` 호출이 없다. Phase 1 의 Customer/Item/Vendor/Employee 는 자동 발급 (`CUST-2026-0001`), Department 와 Warehouse 만 수동.

**이유** (설계서 §5.1):
- 창고는 회사 조직과 1:1. 자주 안 만들고 (한 회사에 보통 1~10개).
- **의미 있는 이름** 의 코드(`WH-HQ`, `WH-BUS`, `WH-INC`) 가 자동 시퀀스(`WH-2026-0001`) 보다 훨씬 유용.
- API/시드에서 "본사창고" 를 가리키려 할 때 `WH-HQ` 로 직접 부르는 게 자연스러움.

### 코드 형식 검증은 DTO 에서

```java
public record WarehouseCreateRequest(
        @NotBlank @Size(max = 30) @Pattern(regexp = "^WH-[A-Z0-9_-]+$",
                message = "code 는 WH-XXX 형식이어야 한다") String code,
        ...
) {}
```

Bean Validation 의 `@Pattern` — 컨트롤러 진입 단계에서 형식 검증. 도메인이 받는 시점에는 형식이 보장된 상태. Department (`^DEPT-...$`) 와 같은 패턴.

→ 형식 검증은 입력 계층 (DTO), 비즈니스 검증 (중복, 자기참조 등) 은 서비스 계층, 자기 상태 검증 (null/blank) 은 도메인 메서드. **검증의 3층 분리** 가 Phase 1 부터 일관.

## 🔥 `WarehouseService.create()` — 중복 코드 게이트

```java
@Transactional
public WarehouseResponse create(WarehouseCreateRequest req) {
    if (repository.existsByCode(req.code())) {
        throw new IllegalStateException("이미 등록된 창고 코드입니다: " + req.code());
    }
    Warehouse warehouse = Warehouse.create(req.code(), req.name(), req.address());
    return mapper.toResponse(repository.save(warehouse));
}
```

수동 코드라서 **중복 가능성** 이 있다 (자동 발급은 시퀀스가 막아주지만). 서비스 단에서 `existsByCode` 로 선검사.

> ⚠️ 동시성: 두 트랜잭션이 같은 코드를 동시에 등록 시도하면 둘 다 `existsByCode = false` 를 보고 둘 다 INSERT 시도 → 한쪽은 UNIQUE 제약 위반 (`DataIntegrityViolationException`).
> → `GlobalExceptionHandler` 가 409 로 변환. 사용자 경험은 같음.

**왜 비관적 락 안 거나** — 창고 등록은 일상 운영이 아니라 초기 셋업 단계의 작업. 동시 등록 시도가 사실상 일어나지 않음. Phase 1 Department 와 같은 방침.

## 🔥 `WarehouseRepository` — 2개 메서드면 충분

```java
public interface WarehouseRepository
        extends JpaRepository<Warehouse, Long>, JpaSpecificationExecutor<Warehouse> {

    Optional<Warehouse> findByCode(String code);

    boolean existsByCode(String code);
}
```

- `findByCode` — 코드로 조회 (`GET /by-code/{code}`).
- `existsByCode` — 중복 검증 (위 서비스 단계).
- 나머지는 `JpaRepository` + `JpaSpecificationExecutor` 가 제공 — 페이징/필터 조회는 `findAll(spec, pageable)`.

`@SQLRestriction("deleted_at IS NULL")` 이 베이스에 박혀 있어서 모든 조회가 자동으로 살아있는 행만 본다 (Phase 1 패턴).

## 🔥 `WarehouseController` — 6종 API 의 일관 형태

```java
@RestController
@RequestMapping("/api/warehouses")
public class WarehouseController {

    @PostMapping                              public ResponseEntity<...> create(...)
    @GetMapping("/{id}")                      public ... findById(...)
    @GetMapping("/by-code/{code}")            public ... findByCode(...)
    @GetMapping                                public Page<...> search(...)
    @PutMapping("/{id}")                      public ... update(...)
    @DeleteMapping("/{id}")                   public ResponseEntity<Void> delete(...)
}
```

Customer / Item / Vendor / Department 모두 똑같은 6종. **마스터 모듈의 표준 컨벤션** — 새 마스터를 보면 API 모양을 외울 필요가 없다.

### 페이징 + 필터

```java
@GetMapping
public Page<WarehouseResponse> search(
        @RequestParam(required = false) String name,
        @RequestParam(required = false) MasterStatus status,
        Pageable pageable
) {
    return service.search(where(nameContains(name)).and(statusEquals(status)), pageable);
}
```

Phase 1 의 `ItemController.search` 와 패턴 동일. JPA `Specification` 로 조합 — Phase 2 부터 QueryDSL 로 전환할 만큼 조건이 복잡하지 않음 (Phase 1 메모).

## 🔥 V20 시드의 첫 행 — `WH-HQ`

```sql
INSERT INTO warehouse (code, name, address, status, ...)
VALUES ('WH-HQ', '본사창고', '서울시 강남구', 'ACTIVE', ...);
```

Phase 3 시연용 단 한 창고. Phase 1 의 `DEPT-HQ` (회사 루트 부서) 와 같은 시드 정책 — 시연 흐름이 시작하기 위한 최소한의 마스터만 한 줄 박아둠.

→ 시연에서 새 창고 등록을 직접 해보려면 `POST /api/warehouses { "code": "WH-BUS", "name": "부산창고", ... }`. 자동 코드가 아니므로 운영자가 직접 정한 코드를 그대로 박는 흐름.

## 🔥 무엇이 빠져 있는가 — 의도된 단순함

| 항목 | 빠진 이유 |
| --- | --- |
| 위치 (Bin Location) — 창고 내부 위치 | 한 창고당 위치 다중화는 학습 범위 밖. Phase 3 은 "재고 = (상품, 창고)" 한 단위만. |
| 창고 간 이동 (StockTransfer) | 설계서 §11 "범위 밖" — 두 Movement 행으로 흉내 가능. |
| 창고별 권한 (담당자) | Phase 6 보안 모듈로 미룸. |
| 면적/용량 같은 물리 속성 | 학습 초점이 아님. |

→ "재고 캐시의 단위가 되는 장소" 라는 최소 책임만. 다음 편부터 실제 학습이 시작.
