# Phase 4 모델링 제안 — SD ↔ MM 연계 설계 ⭐

> 7단계 사이클의 **3단계: 모델링 제안**.
> 도메인 브리핑(`1-도메인-브리핑.md`)에서 정리한 "한 비즈니스 사건 = 한 트랜잭션 = 여러 모듈 변화" 의 원칙을 **이벤트 / 리스너 / 트랜잭션 경계 / FK 결정 / 마이그레이션 계획**으로 변환한다.
> 코드 한 줄 쓰기 전에 "직접 호출 vs 이벤트 / 강 FK vs 약한 참조 / 동기 vs 비동기" 의 결정을 박는다.
> 이 문서 마지막의 **승인 체크리스트** 에 사용자가 OK 하면 단계 5(코드 구현)로 진입한다.

---

## 0. 한눈에 — 이번 설계의 핵심 결정 10개

| # | 결정 | 한 줄 이유 |
| --- | --- | --- |
| 1 | 통합 패턴 = **Spring Application Events + `@TransactionalEventListener(BEFORE_COMMIT)`** | 학습 계획서가 지정. 모듈 격리 + 한 트랜잭션 원자성을 동시에 잡는 패턴 |
| 2 | 이벤트 2종: **`DeliveryShippedEvent`, `DeliveryCancelledEvent`** (Java record) | 출하 확정/취소의 두 사건. 각각 listener 가 반응 |
| 3 | 이벤트 본문 = **ID + 핵심 라인 정보만** (warehouseId, [(itemId, qty)]) | listener 가 다시 DB 조회 안 해도 되게 + 이벤트는 가벼움 유지 |
| 4 | 리스너 위치 = **`mm/integration/sd/` 새 패키지** | "다른 모듈로부터 들어오는 통합" 자리. Phase 11~ MES 연계와 컨벤션 통일 |
| 5 | `GoodsIssue.delivery_id` **nullable FK 컬럼 추가 (V21)** | 출하 취소 시 "이 Delivery 의 GI 찾기" 가 자주 필요 → 강한 FK + 인덱스 |
| 6 | 사용자 직접 만든 GI (`delivery_id = NULL`) 와 자동 GI 가 **한 테이블에 공존** | nullable FK 의 효용. 두 출처를 단일 모델로 |
| 7 | 트랜잭션 전파 = **`REQUIRED` (기본)** — listener 가 출하 트랜잭션에 참여 | 한 비즈니스 사건의 정합성. `REQUIRES_NEW` 면 부분 상태 사고 |
| 8 | 동기 호출 — **`@Async` 안 쓴다** | 비동기면 다른 트랜잭션 → BEFORE_COMMIT 의 의미가 깨짐 |
| 9 | **출하 시 가용 부족이면 출하 자체가 실패** (전체 롤백, 409 INSUFFICIENT_STOCK) | Phase 3 의 예외가 SD 경로로 자연스럽게 노출 |
| 10 | **수주 확정 시 가용 재고 사전 체크는 안 한다** | 학습 계획서 범위 밖. 출하 시점에 첫 검증 (학습 노트로만 짚음) |

## 1. 전체 흐름 — 출하 확정 시퀀스

```
[Client]
POST /api/deliveries  (만약 Phase 2 의 create 가 자동 ship 한다면)
   │
   ▼
[DeliveryService.create / ship]  ───────────────────────  @Transactional 시작
   │
   ├─ delivery.ship()                                ← 자기 상태 검증
   ├─ order.recordShipment(...)                      ← SO 라인 shipped_qty
   │
   ├─ events.publishEvent(DeliveryShippedEvent(...))  ← ⭐ 이벤트 발행
   │       │
   │       ▼
   │  [StockDeductionListener]   @TransactionalEventListener(BEFORE_COMMIT)
   │       │  (같은 트랜잭션 안에서, commit 직전 호출)
   │       ▼
   │  [GoodsIssueService.createAndPostFromDelivery(event)]
   │       │
   │       ├─ GoodsIssue.draft(번호, warehouse, today, SHIPMENT, deliveryId)
   │       ├─ for each event.line: gi.addLine(item, qty)
   │       ├─ goodsIssueRepo.save(gi)
   │       │
   │       └─ gi.post() ─────────────────────────────────  내부:
   │              │
   │              for each line:
   │                ├─ stockRepo.findForUpdate(...)        ← 비관 락
   │                ├─ stock.issue(qty)                    ← 가용 검증
   │                │      ⚠ 부족 시 InsufficientStockException
   │                └─ stockMovementRepo.save(...)         ← 원장 (-)
   │
   └─ return mapper.toResponse(delivery)
        ↓
   @Transactional 종료 (COMMIT)  ────────────────────────  실제 commit
        ↓
   201 Created + DeliveryResponse

[실패 흐름]
가용 부족 → InsufficientStockException
        ↓
   @Transactional ROLLBACK ─────────────────────────────  Delivery, SO 라인,
                                                          Stock, StockMovement
                                                          모두 시도 전 상태로 복귀
        ↓
   GlobalExceptionHandler → 409 INSUFFICIENT_STOCK
```

핵심:
- **이벤트 발행은 SD 측 트랜잭션 안에서** 일어남.
- **`BEFORE_COMMIT` phase** → 리스너가 commit 직전에 실행 + 같은 트랜잭션.
- 리스너의 예외 → 전체 트랜잭션 롤백 → 호출자(Client) 에 그대로 전파.

## 2. 도메인 이벤트 설계

### 2.1 두 이벤트 — `DeliveryShippedEvent`, `DeliveryCancelledEvent`

```java
public record DeliveryShippedEvent(
        Long deliveryId,
        Long warehouseId,
        LocalDate shippedDate,
        List<Line> lines
) {
    public record Line(Long itemId, BigDecimal quantity) {}
}

public record DeliveryCancelledEvent(
        Long deliveryId,
        Long warehouseId
        // 라인 정보는 listener 가 deliveryId 로 GI 찾아서 그 GI 의 lines 사용
) {}
```

### 2.2 본문 — ID 만 vs 핵심 데이터 같이

| 옵션 | 장점 | 단점 |
| --- | --- | --- |
| A. ID 만 | 가장 가벼움, 일관성 ↑ (listener 가 최신 상태 다시 조회) | listener 가 다시 fetch → 쿼리 늘어남, 트랜잭션 길어짐 |
| B. **ID + 핵심 데이터** (채택) | listener 가 즉시 사용, 한 트랜잭션이라 일관성도 OK | 이벤트 객체가 살짝 두꺼움 |

→ **옵션 B**. `DeliveryShippedEvent` 가 (deliveryId, warehouseId, shippedDate, lines) 까지 들고 다님. 같은 트랜잭션이라 일관성 우려 없음.

`DeliveryCancelledEvent` 는 라인 정보 없이 deliveryId 만 — 취소 listener 가 `delivery_id` 로 GI 를 직접 찾는 게 더 깔끔 (라인 매칭 로직 불필요).

### 2.3 이벤트 위치 — 발행자 소유

```
sd/delivery/
├─ Delivery.java
├─ DeliveryService.java
├─ event/                           ← 새 서브패키지
│  ├─ DeliveryShippedEvent.java
│  └─ DeliveryCancelledEvent.java
└─ ...
```

**이벤트는 발행자가 정의** — SD 모듈이 자기 사건을 자기 패키지에 둠. MM 측이 SD 의 이벤트 타입을 import 해서 구독. **방향이 SD → MM** 이 아니라, MM 이 SD 의 사건에 "관심을 표시" 하는 형태.

→ SD 는 누가 듣는지 모름. MM 만이 SD 의 이벤트를 안다 — **단방향 의존**.

## 3. 리스너 배치

### 3.1 새 패키지 — `mm/integration/sd/`

```
mm/
├─ warehouse/
├─ stock/
├─ goodsreceipt/
├─ goodsissue/
└─ integration/             ← 새 패키지
   └─ sd/                   ← "SD 로부터 들어오는 통합" 자리
      └─ DeliveryEventListener.java
```

`integration/` 아래에 출처 모듈명(`sd/`) 으로 디렉토리를 둠. 미래 확장:
- Phase 11~ MES 연계: `integration/mes/`
- 외부 ERP 시스템: `integration/external/`

→ "다른 모듈/시스템으로부터 들어오는 통합 코드" 의 한 자리. 통합 디버깅 시 한 곳에서 찾을 수 있음.

### 3.2 리스너 클래스

```java
@Component
@RequiredArgsConstructor
@Slf4j
class DeliveryEventListener {

    private final GoodsIssueService goodsIssueService;

    @TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
    void onShipped(DeliveryShippedEvent event) {
        log.info("Delivery shipped event received: deliveryId={}", event.deliveryId());
        goodsIssueService.createAndPostFromDelivery(event);
    }

    @TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
    void onCancelled(DeliveryCancelledEvent event) {
        log.info("Delivery cancelled event received: deliveryId={}", event.deliveryId());
        goodsIssueService.cancelByDeliveryId(event.deliveryId());
    }
}
```

핵심:
- **`@TransactionalEventListener(BEFORE_COMMIT)`** — 같은 트랜잭션의 commit 직전 실행. 예외 시 전체 롤백.
- 리스너 자체에 `@Transactional` 안 씀 — 호출하는 `goodsIssueService` 메서드가 `@Transactional(REQUIRED)` 이므로 자동 참여.
- 로그 한 줄 — 통합 흐름 추적용. 시연/디버깅에 필수.

### 3.3 phase 의 4가지 옵션 — 왜 BEFORE_COMMIT 인가

| phase | 시점 | 트랜잭션 참여 | 이번 결정 |
| --- | --- | --- | --- |
| **`BEFORE_COMMIT`** | commit 직전 | **참여 (같은 트랜잭션)** | ✅ **채택** |
| `AFTER_COMMIT` (기본) | commit 후 | 별도 트랜잭션 | ❌ 부분 상태 사고 |
| `AFTER_ROLLBACK` | rollback 후 | 별도 | (사용 안 함) |
| `AFTER_COMPLETION` | commit/rollback 후 | 별도 | (사용 안 함) |

`AFTER_COMMIT` 으로 했다가 listener 가 실패하면:
- Delivery 는 이미 commit 됨 → DB 에 SHIPPED 로 박혀 있음.
- Stock 차감은 별도 트랜잭션이라 알 수 없는 상태 (롤백돼도 Delivery 는 살아있음).
- **사고**: 출하했는데 재고 안 빠짐. 다음 출하의 가용 검증이 거짓.

→ `BEFORE_COMMIT` 만이 "한 비즈니스 사건의 원자성" 을 보장.

## 4. `GoodsIssue.delivery_id` — Delivery 참조의 방법

### 4.1 세 가지 옵션 비교

**옵션 A — 약한 참조 (StockMovement 패턴 재사용)**

`GoodsIssue` 에 `ref_type=DLV, ref_id=delivery.id` 컬럼 두 개. FK 없음.

장점: 새 출처 추가 (예: Phase 8 생산 자재 소모) 시 컬럼 안 늘림.
단점: 출하 취소 시 "이 Delivery 의 GI 찾기" 쿼리가 `WHERE ref_type='DLV' AND ref_id=?` — 인덱스는 있지만 의도가 흐려짐.

**옵션 B — 강한 FK (`delivery_id BIGINT NULL FK`) ⭐ 채택**

`GoodsIssue` 에 `delivery_id` 컬럼. nullable FK. 사용자 직접 만든 GI 는 NULL.

장점:
- 의도가 명확 — "이 GI 는 Delivery 와 연결됐다" 가 컬럼명에 박힘.
- FK 인덱스로 빠른 역추적.
- DB 가 FK 정합성 강제 — dangling 안 됨.

단점: 미래에 출처가 더 다양해지면 NULL FK 컬럼이 늘어남. → 그때 가서 옵션 A 로 일반화 검토.

**옵션 C — 별도 매핑 테이블 `delivery_goods_issue_link`**

`(delivery_id, goods_issue_id)` 한 테이블.

장점: 두 도메인 둘 다 깨끗 (서로 참조 없음).
단점: 테이블 하나 더, JOIN 한 단계 더. 1:1 관계인데 매핑 테이블은 과함.

### 4.2 결정 — 옵션 B

이유:
- **Phase 4 단계의 SD↔MM 연계는 1:1 강한 관계** — 한 Delivery 는 한 GI 와 1:1.
- 출하 취소 흐름이 빈번 (시연 STEP 7 의 패턴). FK 인덱스로 빠르게 찾는 게 자연스러움.
- "이 GI 는 어디서 왔나" 가 한 컬럼으로 명확.

→ 약한 참조 (StockMovement.ref_type) 와 강한 FK 는 **다른 성격**. StockMovement 는 다형성 (GR, GI, ADJ, 향후 더), GoodsIssue.delivery_id 는 단일 종속.

### 4.3 마이그레이션 V21 (자세히 §8)

```sql
ALTER TABLE goods_issue
    ADD COLUMN delivery_id BIGINT NULL COMMENT '출하 자동 발생 시 원천 Delivery (수동 등록은 NULL)';

ALTER TABLE goods_issue
    ADD CONSTRAINT fk_goods_issue_delivery FOREIGN KEY (delivery_id) REFERENCES delivery(id);

CREATE INDEX idx_goods_issue_delivery ON goods_issue(delivery_id);
```

## 5. 트랜잭션 경계 + 전파 옵션

### 5.1 전체 흐름의 트랜잭션 구조

```
@Transactional   ← DeliveryService.ship
    │
    │   /* 도메인 메서드 */
    │   delivery.ship()
    │   order.recordShipment(...)
    │
    │   /* 이벤트 발행 */
    │   events.publishEvent(DeliveryShippedEvent(...))
    │
    │   /* @TransactionalEventListener(BEFORE_COMMIT) */
    │   onShipped(event) {
    │       /* @Transactional(REQUIRED) — 위 트랜잭션에 참여 */
    │       goodsIssueService.createAndPostFromDelivery(event) {
    │           gi.draft(...)
    │           gi.addLine(...)
    │           gi.post() {
    │               /* @Transactional(REQUIRED) — 위에 또 참여 */
    │               stockRepo.findForUpdate(...)
    │               stock.issue(...)
    │               stockMovementRepo.save(...)
    │           }
    │       }
    │   }
    │
    │   /* return — commit 직전 */
COMMIT  ← 한 번의 commit 으로 4 테이블 (delivery, sales_order_line, stock, stock_movement) + goods_issue + goods_issue_line + stock_movement 변화가 모두 박힘
```

### 5.2 `REQUIRED` 의 의미

| 전파 옵션 | DeliveryService.ship 의 트랜잭션 안에서 호출되면 |
| --- | --- |
| **`REQUIRED`** (기본) | **참여 — 같은 트랜잭션**. ✅ 채택 |
| `REQUIRES_NEW` | 별도 새 트랜잭션. 본 흐름 일시 정지. 어느 한쪽 실패해도 다른 쪽은 독립 commit. **사고** |
| `SUPPORTS` | 트랜잭션 있으면 참여, 없으면 트랜잭션 없이. 위험 |
| `MANDATORY` | 트랜잭션 반드시 있어야 — 없으면 예외. 안전하지만 호출 측 제약 |
| `NEVER` | 트랜잭션 안에서 호출되면 예외. 사용 안 함 |

→ **`REQUIRED`** 가 정답. `@Transactional` 어노테이션을 따로 안 적어도 기본값.

### 5.3 BEFORE_COMMIT 의 예외 전파

```java
@TransactionalEventListener(phase = BEFORE_COMMIT)
void onShipped(DeliveryShippedEvent event) {
    goodsIssueService.createAndPostFromDelivery(event);
    //                       ↑
    //  InsufficientStockException 발생 시 →
    //  Spring 이 잡아서 트랜잭션을 rollbackOnly 마킹 →
    //  최종 commit 단계에서 rollback 으로 전환 →
    //  예외는 DeliveryService.ship 호출자에 전파 →
    //  GlobalExceptionHandler 가 409 로 변환
}
```

`BEFORE_COMMIT` 의 핵심 특성:
- listener 가 예외 던지면 **commit 전에 발생** → 자연스럽게 전체 트랜잭션 rollback.
- `AFTER_COMMIT` 이었으면 commit 후 발생 → rollback 불가 (이미 commit) → 데이터 부분 상태.

## 6. 새 비즈니스 로직

### 6.1 `GoodsIssueService.createAndPostFromDelivery(event)`

```java
@Transactional
public GoodsIssueResponse createAndPostFromDelivery(DeliveryShippedEvent event) {
    Warehouse warehouse = warehouseRepository.findById(event.warehouseId())
            .orElseThrow(() -> new EntityNotFoundException(...));

    String number = numberGenerator.nextGoodsIssueNumber(event.shippedDate());
    GoodsIssue gi = GoodsIssue.draftForDelivery(
            number, warehouse, event.shippedDate(),
            GoodsIssueReason.SHIPMENT, event.deliveryId());

    for (DeliveryShippedEvent.Line line : event.lines()) {
        Item item = itemRepository.findById(line.itemId()).orElseThrow(...);
        gi.addLine(item, line.quantity());
    }

    goodsIssueRepository.save(gi);
    gi.post(LocalDateTime.now(clock));   // Stock 차감 + StockMovement (기존 post 로직 재사용)
    return mapper.toResponse(gi);
}
```

핵심:
- **새 팩토리 메서드** `GoodsIssue.draftForDelivery(...)` — `delivery_id` 를 받는 점만 기존 `draft(...)` 와 다름.
- `reason = SHIPMENT` 고정.
- `gi.post()` 는 Phase 3 의 기존 로직 그대로 — 비관 락 + Stock.issue + StockMovement.

### 6.2 `GoodsIssueService.cancelByDeliveryId(deliveryId)`

```java
@Transactional
public void cancelByDeliveryId(Long deliveryId) {
    GoodsIssue gi = goodsIssueRepository.findByDeliveryId(deliveryId).orElse(null);
    if (gi == null) {
        // Phase 4 적용 전 Delivery 는 GI 없을 수 있음 — 무시.
        log.info("No GI for delivery id={} — skipping cancel", deliveryId);
        return;
    }
    if (gi.getStatus() != GoodsIssueStatus.POSTED) {
        // 이미 취소된 경우 등 — 무시.
        log.info("GI id={} not in POSTED — skipping", gi.getId());
        return;
    }
    // 기존 cancel 로직 재사용 (Phase 3) — 비관 락 + Stock 복원 + ADJUSTMENT_PLUS
    cancel(gi.getId());
}
```

방어적 처리:
- GI 없으면 무시 — Phase 4 적용 전 데이터 호환.
- 이미 CANCELLED 면 무시 — 멱등성.

### 6.3 `DeliveryService.cancel(id)` 수정

기존 (Phase 2):
```java
@Transactional
public DeliveryResponse cancel(Long id) {
    Delivery delivery = getOrThrow(id);
    SalesOrder order = delivery.getSalesOrder();
    delivery.cancel();
    for (DeliveryLine dline : delivery.getLines()) {
        order.cancelShipment(dline.getSalesOrderLine(), dline.getQuantity());
    }
    return mapper.toResponse(delivery);
}
```

Phase 4 추가 — 마지막에 이벤트 발행:
```java
@Transactional
public DeliveryResponse cancel(Long id) {
    Delivery delivery = getOrThrow(id);
    SalesOrder order = delivery.getSalesOrder();
    delivery.cancel();
    for (DeliveryLine dline : delivery.getLines()) {
        order.cancelShipment(dline.getSalesOrderLine(), dline.getQuantity());
    }
    // ⭐ Phase 4 추가
    events.publishEvent(new DeliveryCancelledEvent(
            delivery.getId(), delivery.getSalesOrder().getWarehouse()... ?));
    return mapper.toResponse(delivery);
}
```

> ⚠️ `warehouseId` 의 출처가 모호 — `Delivery` 자체엔 `warehouse` 가 없음. Phase 2 모델에서 출하지 창고를 안 잡았기 때문. 두 가지 해결:
> (a) GoodsIssue 의 warehouse_id 가 충분 → `cancelByDeliveryId` 에서 GI 의 warehouseId 사용 → 이벤트엔 deliveryId 만.
> (b) Delivery 에 warehouseId 컬럼 추가 — 모델 변경.
>
> → 학습 단순화 + 정합성 측면에서 **(a)** 권장. `DeliveryCancelledEvent` 는 `deliveryId` 만 들고 다님.

### 6.4 `DeliveryService.create` (Phase 2 의 `ship` 흐름) 수정

`Delivery.ship()` 후 마지막에 이벤트 발행:
```java
@Transactional
public DeliveryResponse create(DeliveryCreateRequest req) {
    // ... 기존 Phase 2 로직 ...
    delivery.ship();
    for (DeliveryLine dline : delivery.getLines()) {
        order.recordShipment(dline.getSalesOrderLine(), dline.getQuantity());
    }
    repository.save(delivery);

    // ⭐ Phase 4 추가 — Delivery 가 어느 창고로 보낼지를 알아야 함
    // → DeliveryCreateRequest 에 warehouseId 추가 또는
    // → 어딘가에서 결정 (예: 항상 본사창고 default)
    events.publishEvent(new DeliveryShippedEvent(
            delivery.getId(), warehouseId, delivery.getShippedDate(),
            mapToEventLines(delivery.getLines())));

    return mapper.toResponse(delivery);
}
```

### 6.5 새 모델 변경 — `Delivery` 에 `warehouse_id` 추가

`Delivery` 자체에 출하지 창고 정보가 없는 게 §6.3 의 문제. Phase 4 의 자연스러운 보강:

**옵션 X**: `Delivery` 엔티티에 `Warehouse warehouse` 필드 추가.
- 마이그레이션 `ALTER TABLE delivery ADD COLUMN warehouse_id BIGINT NOT NULL`
- `DeliveryCreateRequest` 에 `warehouseId` 추가.
- 기존 V14 시드는 영향 없음 (Delivery 시드 없음).
- 이벤트가 `warehouseId` 를 정확히 담을 수 있음.

**옵션 Y**: 일단 `default = WH-HQ` 로 고정 — 학습 단순화.
- 마이그레이션 없음.
- 코드: `warehouseRepository.findByCode("WH-HQ")` 호출해 항상 같은 창고.
- 단점: 다중 창고 출하 시연 불가.

→ **옵션 X 권장**. Phase 3 시연 STEP 10 에서 부산창고 추가도 시연했고, 다중 창고는 ERP 의 정상 상태. Phase 4 가 그 자연스러운 보강을 같이 한다.

추가 마이그레이션: **V22 `add_warehouse_to_delivery`** (V21 다음).

## 8. 마이그레이션 계획 (Flyway V21~V22)

| 버전 | 파일명 | 내용 |
| --- | --- | --- |
| V21 | `V21__add_delivery_id_to_goods_issue.sql` | `goods_issue.delivery_id` nullable FK + 인덱스 |
| V22 | `V22__add_warehouse_to_delivery.sql` | `delivery.warehouse_id` NOT NULL FK + 인덱스 + 기존 행 backfill |

### 8.1 V21 — GoodsIssue.delivery_id

```sql
-- Phase 4: GoodsIssue 가 자기 출처(Delivery) 를 강한 FK 로 참조.
-- nullable — 사용자 직접 만든 GI 는 NULL.

ALTER TABLE goods_issue
    ADD COLUMN delivery_id BIGINT NULL COMMENT '출하 자동 발생 시 원천 Delivery (수동 등록은 NULL)';

ALTER TABLE goods_issue
    ADD CONSTRAINT fk_goods_issue_delivery FOREIGN KEY (delivery_id) REFERENCES delivery(id);

CREATE INDEX idx_goods_issue_delivery ON goods_issue(delivery_id);
```

### 8.2 V22 — Delivery.warehouse_id

```sql
-- Phase 4: Delivery 가 출하 창고를 직접 안다 (이벤트 본문 + 직접 조회용).
-- 기존 Delivery 행이 없거나 (Phase 2 시연 시 사용자가 만든 행) 적은 경우 단순:

ALTER TABLE delivery
    ADD COLUMN warehouse_id BIGINT NULL;

-- 기존 행이 있다면 본사창고로 backfill (시연 환경 가정)
UPDATE delivery d
   SET warehouse_id = (SELECT id FROM warehouse WHERE code = 'WH-HQ')
 WHERE warehouse_id IS NULL;

-- NOT NULL 강제
ALTER TABLE delivery
    MODIFY COLUMN warehouse_id BIGINT NOT NULL COMMENT '출하지 창고';

ALTER TABLE delivery
    ADD CONSTRAINT fk_delivery_warehouse FOREIGN KEY (warehouse_id) REFERENCES warehouse(id);

CREATE INDEX idx_delivery_warehouse ON delivery(warehouse_id);
```

> ⚠️ **운영 마이그레이션** 이면 backfill 정확도가 중요. 학습 환경은 단순화.

### 8.3 V20 까지의 시드 영향 없음

- V14 시드는 Delivery 행 안 만듦 (Quotation + SalesOrder 만).
- V20 시드는 Warehouse + Stock + StockMovement + GoodsReceipt 만. GoodsIssue 시드 없음.
- → V21/V22 는 시드와 충돌 없음. 깔끔한 ALTER.

## 10. 의도적으로 미루는 것 (Phase 4 범위 밖)

| 항목 | 미루는 이유 | 도입 시점 |
| --- | --- | --- |
| 수주 확정 시 가용 재고 사전 체크 | 학습 계획서 범위 밖. 출하 시점에 첫 검증으로 충분 | (실무 검토용, 학습 단계는 안 함) |

| 이벤트 발행 실패 시 재시도 (Outbox Pattern) | 같은 트랜잭션이라 발행 실패 자체가 없음 (메모리 내 dispatch) | Phase 14 (Kafka 발행) 에서 본격 |

| 두 listener 의 순서 (`@Order`) | 현재 listener 가 하나뿐 | Phase 5 이상에서 회계 listener 추가 시 |
| 입고 측 자동 흐름 (PurchaseOrder → GoodsReceipt) | PO 모듈이 없음 (학습 범위 밖) | 학습 범위 밖 |
| 다중 창고 출하 분배 (한 SO 가 두 창고로 나눠 출하) | 실무 패턴이지만 학습 가치 대비 복잡도 ↑ | 학습 범위 밖 |
| 부분 가용 출하 (10대 요청, 7대만 보내고 3대는 백오더) | 학습 단순화 | 학습 범위 밖 |
| Saga / 보상 트랜잭션 | 단일 시스템 내라 단순 트랜잭션으로 충분 | Phase 14~ 분산 시스템 |
