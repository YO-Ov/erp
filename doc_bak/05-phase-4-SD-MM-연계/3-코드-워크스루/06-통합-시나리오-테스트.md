# 6/6. 통합 시나리오 테스트 — 한 트랜잭션의 원자성을 검증한다

> Phase 4 의 학습이 코드로 굳어지는 마지막 자리. 앞 편들에서 설계한 "출하 = 이벤트 → 리스너 → 재고 차감" 의 한 흐름이 **정말 하나의 트랜잭션으로 묶이는가** 를 테스트가 증명한다.
> 설계서 §9 "테스트 전략" + Phase 3 `ConcurrentGoodsIssueTest` 의 "테스트가 곧 명세" 정신 계승.

대상 파일:

```
hwlee-erp/src/test/java/com/hwlee/erp/sd/
├─ SdMmIntegrationTest.java                    ⭐ Phase 4 의 하이라이트 (5개 테스트)
└─ PartialDeliveryAndInvoiceScenarioTest.java  (Phase 2 테스트가 Phase 4 로 어떻게 진화했나)
```

---

## 🔥 테스트가 곧 명세 — 5개 메서드명이 Phase 4 의 약속

```java
@Test void 출하_확정_시_재고가_자동_차감되고_GoodsIssue_가_POSTED_로_생성된다()
@Test void 가용_재고_부족_출하는_전체_롤백된다()
@Test void 출하_취소_시_GoodsIssue_도_취소되고_재고가_복원된다()
@Test void 다중_라인_출하는_한_GoodsIssue_가_여러_라인을_가진다()
@Test void 리스너_예외는_Delivery_트랜잭션을_롤백시킨다()
```

→ 5개 한국어 문장이 곧 Phase 4 의 약속. 누군가 "SD 와 MM 을 연결하면 무엇이 보장되나?" 물으면 이 5줄을 보여주면 답이 된다.

Phase 마다 자신만의 약속 한 묶음을 보탰다:
- Phase 1: 코드 생성 동시성 / Soft Delete / 마스터 CRUD
- Phase 2: 부분 출하/청구 누적 / 신용한도 / 상태 전이
- Phase 3: 가중평균 / 가용 검증 / 원장-캐시 정합성 / 동시 출고 직렬화
- **Phase 4**: 출하-재고 **연계의 원자성** — 한 사건은 전부 일어나거나 전부 안 일어난다

이 모든 검증이 `@SpringBootTest` + Testcontainers 로 **실제 MySQL(V1~V22 마이그레이션 적용)** 위에서, 컨트롤러 아래 전 계층을 통과해 돌아간다. 도메인 단위 테스트가 아니라 "시스템이 실제로 그렇게 동작하는가" 의 증명.

## 🔥 테스트 1 — 출하 한 번에 재고·GI·원장이 모두 따라온다

```java
@Test
@DisplayName("출하 확정 시 재고가 자동 차감되고 GoodsIssue 가 POSTED 로 생성된다")
void 출하_확정_시_재고가_자동_차감되고_GoodsIssue_가_POSTED_로_생성된다() {
    var ctx = setup(10);   // 10대 입고

    // 6대 출하 확정 — 이벤트 발행 → 리스너가 GI 생성 + 재고 차감
    DeliveryResponse dlv = deliveryService.create(new DeliveryCreateRequest(
            ctx.soId, ctx.warehouseId, LocalDate.now(),
            List.of(new DeliveryLineRequest(ctx.solId, bd(6)))));

    // 재고는 4대 남는다
    Stock stock = stockRepository
            .findByItemIdAndWarehouseId(ctx.itemId, ctx.warehouseId).orElseThrow();
    assertThat(stock.getQtyOnHand()).isEqualByComparingTo(bd(4));

    // 출하에 연결된 GoodsIssue 가 자동 생성되고 POSTED
    Long giId = goodsIssueRepository.findByDeliveryId(dlv.id()).orElseThrow().getId();
    GoodsIssueResponse gi = goodsIssueService.findById(giId);
    assertThat(gi.status()).isEqualTo(GoodsIssueStatus.POSTED);
    assertThat(gi.deliveryId()).isEqualTo(dlv.id());
    assertThat(gi.lines()).hasSize(1);
    assertThat(gi.lines().get(0).quantity()).isEqualByComparingTo(bd(6));

    // 원장에 출고(-6) 한 줄
    StockMovement giMv = stockMovementRepository.findAll().stream()
            .filter(m -> m.getReason() == MovementReason.GOODS_ISSUE && m.getRefId().equals(giId))
            .findFirst().orElseThrow();
    assertThat(giMv.getQtyDelta()).isEqualByComparingTo(bd(-6));
}
```

이 한 테스트가 검증하는 **4개의 연쇄**:

| # | 검증 | 의미 |
| --- | --- | --- |
| 1 | `stock.qtyOnHand == 4` | 10 입고, 6 출하 → 재고 4. 출하가 재고를 자동 차감 |
| 2 | `gi.status == POSTED`, `gi.deliveryId == dlv.id` | 리스너가 GI 를 만들고 **바로 확정**까지. 강한 FK 로 출하와 연결 |
| 3 | `gi.lines.get(0).quantity == 6` | 출하 라인의 수량이 GI 라인으로 그대로 흘러옴 |
| 4 | `StockMovement(GOODS_ISSUE, -6)` | 원장에 음수 한 줄. Phase 3 의 차감 메커니즘이 그대로 호출됨 |

호출자는 `deliveryService.create(...)` **단 한 줄** 만 했다. 그 한 줄이 이벤트를 발행하고, 리스너가 GI 를 만들고, GI 가 재고를 차감하고, 원장이 적재됐다. **사용자 코드는 출하만 알고, 재고·GI·원장은 시스템이 자동으로 따라온다** — Phase 4 가 산 가치.

> 💡 `findByDeliveryId` — Phase 4 에서 추가된 조회. GI 테이블에 자동 GI(`delivery_id != NULL`)와 수동 GI(`delivery_id IS NULL`)가 공존하고, 출하로부터 역추적할 때 쓴다 (설계서 §0 결정 #4~5, V21 마이그레이션).

## 🔥 테스트 2 — 재고 부족이면 **출하·SO 누적까지 전부 롤백**

```java
@Test
@DisplayName("가용 재고가 부족하면 InsufficientStockException 과 함께 출하 전체가 롤백된다")
void 가용_재고_부족_출하는_전체_롤백된다() {
    var ctx = setup(5);   // 5대만 입고

    // 10대 출하 시도 (SO 잔여는 10이라 라인 생성은 통과, 재고 차감에서 실패)
    assertThatThrownBy(() -> deliveryService.create(new DeliveryCreateRequest(
            ctx.soId, ctx.warehouseId, LocalDate.now(),
            List.of(new DeliveryLineRequest(ctx.solId, bd(10))))))
            .isInstanceOf(InsufficientStockException.class);

    // 전체 롤백 — 재고는 5대 그대로
    Stock stock = stockRepository
            .findByItemIdAndWarehouseId(ctx.itemId, ctx.warehouseId).orElseThrow();
    assertThat(stock.getQtyOnHand())
            .as("리스너 예외가 BEFORE_COMMIT 이라 재고 차감도 롤백")
            .isEqualByComparingTo(bd(5));

    // SO 라인의 shipped_qty 도 누적되지 않음 (출하 트랜잭션 통째로 롤백)
    var so = salesOrderService.findById(ctx.soId);
    assertThat(so.lines().get(0).shippedQty())
            .as("출하가 롤백됐으므로 SO 출하 누적도 0")
            .isEqualByComparingTo(BigDecimal.ZERO);
}
```

이 테스트가 Phase 4 의 **핵심**. 단계별로 무슨 일이 벌어지는가:

```
deliveryService.create(10대)  ← @Transactional BEGIN
  ├─ DeliveryLine 생성: SO 잔여 10 ≥ 요청 10 → 통과 (여기선 아직 OK)
  ├─ SalesOrderLine.shippedQty += 10  ← 영속성 컨텍스트에 쌓임
  ├─ DeliveryShippedEvent 발행
  │   └─ @TransactionalEventListener(BEFORE_COMMIT)
  │       └─ GI 생성 → post → Stock.issue(10)
  │           └─ qtyOnHand(5) < 10 → InsufficientStockException 💥
  │
  └─ 예외가 트랜잭션 경계까지 전파 → ROLLBACK
       ↑ Delivery, DeliveryLine, shippedQty 증가, GI 생성 모두 취소
```

검증의 두 단언이 **원자성의 증거**:

- `stock.qtyOnHand == 5` — 차감 시도가 롤백돼 입고분 그대로.
- **`so.shippedQty == 0`** — 더 중요한 단언. 출하 라인 생성과 SO 누적은 리스너가 **터지기 전에** 이미 일어났다. 그런데도 0 이라는 건, 리스너의 예외가 BEFORE_COMMIT 단계에서 **출하 트랜잭션 전체를 되감았다**는 뜻.

> ⚠️ 만약 리스너가 `@TransactionalEventListener(AFTER_COMMIT)` 였다면? 출하는 이미 커밋된 뒤라 SO 의 `shippedQty=10` 이 남고, 재고 차감만 실패하는 **불일치 상태**가 된다. BEFORE_COMMIT 선택(설계서 §0 결정 #6, 워크스루 02편)이 이 단언을 통과시키는 이유.

## 🔥 테스트 3 — 출하 취소도 거꾸로 한 트랜잭션

```java
@Test
@DisplayName("출하 취소 시 연결된 GoodsIssue 도 CANCELLED 되고 재고가 복원된다")
void 출하_취소_시_GoodsIssue_도_취소되고_재고가_복원된다() {
    var ctx = setup(10);

    DeliveryResponse dlv = deliveryService.create(new DeliveryCreateRequest(
            ctx.soId, ctx.warehouseId, LocalDate.now(),
            List.of(new DeliveryLineRequest(ctx.solId, bd(6)))));
    Long giId = goodsIssueRepository.findByDeliveryId(dlv.id()).orElseThrow().getId();

    // 출하 취소 → 이벤트 → 리스너가 GI 취소 + 재고 복원
    deliveryService.cancel(dlv.id());

    Stock stock = stockRepository
            .findByItemIdAndWarehouseId(ctx.itemId, ctx.warehouseId).orElseThrow();
    assertThat(stock.getQtyOnHand()).isEqualByComparingTo(bd(10));

    assertThat(goodsIssueService.findById(giId).status()).isEqualTo(GoodsIssueStatus.CANCELLED);

    // 복원 원장(ADJUSTMENT_PLUS +6) 추가
    StockMovement restore = stockMovementRepository.findAll().stream()
            .filter(m -> m.getReason() == MovementReason.ADJUSTMENT_PLUS && m.getRefId().equals(giId))
            .findFirst().orElseThrow();
    assertThat(restore.getQtyDelta()).isEqualByComparingTo(bd(6));
}
```

출하(6대)와 정반대 흐름:
- `qtyOnHand` 가 4 → **10 으로 복원**.
- GI 가 POSTED → **CANCELLED**.
- 원장에 `ADJUSTMENT_PLUS(+6)` **새 줄 추가**. 원래 `GOODS_ISSUE(-6)` 행을 지우지 않고 양수 정정 행을 더한다 — append-only 원장 (Phase 3 03편의 출고 취소와 동일 정책).

핵심: 취소도 `DeliveryCancelledEvent` 를 발행하고 리스너가 `cancelByDeliveryId(...)` 로 GI 취소 + 재고 복원을 같은 트랜잭션에 묶는다. **정방향이든 역방향이든 "출하 사건 = 한 트랜잭션"** 이라는 약속이 일관.

## 🔥 테스트 4 — 한 출하, 두 품목, 하나의 GoodsIssue

```java
@Test
@DisplayName("여러 라인 출하는 한 GoodsIssue 가 여러 라인을 가지고 생성된다")
void 다중_라인_출하는_한_GoodsIssue_가_여러_라인을_가진다() {
    // 노트북 5대 + 모니터 3대 한 출하
    ...
    stockUp(notebook.id(), warehouseId, 5);
    stockUp(monitor.id(), warehouseId, 3);
    ...
    DeliveryResponse dlv = deliveryService.create(new DeliveryCreateRequest(
            order.id(), warehouseId, LocalDate.now(),
            List.of(new DeliveryLineRequest(notebookSolId, bd(5)),
                    new DeliveryLineRequest(monitorSolId, bd(3)))));

    // 한 출하 → 한 GoodsIssue, 라인 2개
    Long giId = goodsIssueRepository.findByDeliveryId(dlv.id()).orElseThrow().getId();
    GoodsIssueResponse gi = goodsIssueService.findById(giId);
    assertThat(gi.lines()).hasSize(2);

    // 두 품목 재고 모두 0 차감
    assertThat(stockRepository.findByItemIdAndWarehouseId(notebook.id(), warehouseId)
            .orElseThrow().getQtyOnHand()).isEqualByComparingTo(BigDecimal.ZERO);
    assertThat(stockRepository.findByItemIdAndWarehouseId(monitor.id(), warehouseId)
            .orElseThrow().getQtyOnHand()).isEqualByComparingTo(BigDecimal.ZERO);
}
```

두 라인(노트북·모니터) 출하가 **GI 1개 + 라인 2개** 로 매핑됨을 검증. "출하 1건 = GI 1건" 의 1:1 관계가 라인 수와 무관하게 성립.

`gi.lines().hasSize(2)` 와 두 품목 재고가 각각 0 으로 떨어진 것 — 리스너가 출하 라인을 순회하며 품목별로 재고를 차감했고(Phase 3 의 라인 루프 + `findForUpdate` 가 그대로 재사용됨), 모두 같은 한 GI 안에 묶였다는 증거.

> 💡 헬퍼 `stockUp(itemId, warehouseId, qty)` — 입고(GoodsReceipt 생성 + 확정)를 한 줄로 압축한 테스트 픽스처. 품목별로 따로 입고를 세팅해야 다중 라인 출하가 성립한다. Phase 4 에서 모든 출하 테스트의 공통 전제가 됐다.

## 🔥 테스트 5 — 다른 예외 경로, 같은 결과(전체 롤백)

```java
@Test
@DisplayName("BEFORE_COMMIT 리스너에서 어떤 예외가 나도 Delivery 트랜잭션이 통째로 롤백된다")
void 리스너_예외는_Delivery_트랜잭션을_롤백시킨다() {
    // 입고를 전혀 안 함 → 출하 시 리스너에서 Stock not found(EntityNotFoundException)
    ...
    Long warehouseId = warehouseId();   // 재고 행 없음
    ...
    assertThatThrownBy(() -> deliveryService.create(new DeliveryCreateRequest(
            order.id(), warehouseId, LocalDate.now(),
            List.of(new DeliveryLineRequest(solId, bd(3))))))
            .isInstanceOf(EntityNotFoundException.class);

    // Delivery·SO 누적 모두 시도 전 상태 — shipped_qty 0
    var so = salesOrderService.findById(order.id());
    assertThat(so.lines().get(0).shippedQty()).isEqualByComparingTo(BigDecimal.ZERO);
}
```

테스트 2 와 짝이다. 둘을 나란히 놓으면 **"예외 종류와 무관하게 전체 롤백"** 이라는 더 강한 약속이 드러난다:

| | 테스트 2 | 테스트 5 |
| --- | --- | --- |
| 셋업 | 5대만 입고 | **입고 전혀 안 함** (재고 행 없음) |
| 리스너에서 던지는 예외 | `InsufficientStockException` (가용 부족) | `EntityNotFoundException` (Stock not found) |
| 발생 지점 | `Stock.issue()` 의 가용 검증 | `findForUpdate(...).orElseThrow()` |
| **결과** | **shipped_qty == 0** (전체 롤백) | **shipped_qty == 0** (전체 롤백) |

비즈니스 예외(`InsufficientStock`)든 인프라성 예외(`EntityNotFound`)든, **리스너 안에서 던져진 어떤 RuntimeException 도** 같은 트랜잭션에 참여 중이므로 출하 전체를 되감는다. 특정 예외 하나만 잡아 처리하는 게 아니라 "트랜잭션 경계가 모든 예외를 동일하게 다룬다" 는 Spring 트랜잭션의 본질을 두 경로로 못 박은 것.

> ⚠️ "재고 행이 아예 없는" 경우(`EntityNotFoundException`)는 출하 전 입고가 한 번도 없었다는 뜻. 정상 운영에선 시드/입고가 선행되지만, 코드가 그 가정을 깨도 출하가 안전하게 거부된다.

## 🔥 Phase 2 테스트의 진화 — 출하는 이제 재고 없이 성립 안 한다

`PartialDeliveryAndInvoiceScenarioTest` 는 **Phase 2 에 작성된 부분 출하/청구 시나리오** 테스트다. Phase 4 가 출하-재고 연계를 도입하면서 이 테스트가 **고쳐졌다**. 무엇이 바뀌었나:

```java
/**
 * Phase 2 의 하이라이트 — 부분 출하/청구 시나리오.
 * ...
 * <p><b>Phase 4 변경</b>: 출하 확정 시 MM 재고가 자동 차감되므로, 출하 전에 충분한 재고를
 * 입고해 둬야 한다(재고 없으면 출하 자체가 롤백). 출하/청구 누적 검증의 본질은 그대로.
 */
```

```java
// Phase 4: 출하될 창고에 10대 입고 (출하 시 재고 차감)
Long warehouseId = warehouseId();
stockUp(item.id(), warehouseId, 10);

// step 2: 6대 출하 → SHIPPING
DeliveryResponse dlv1 = deliveryService.create(new DeliveryCreateRequest(
        soId, warehouseId, LocalDate.now(),                       // ← warehouseId 인자 추가
        List.of(new DeliveryLineRequest(solId, new BigDecimal("6")))));
```

| Phase 2 (원래) | Phase 4 (변경 후) |
| --- | --- |
| 고객 + 품목만 셋업 | **+ 창고 생성 + `stockUp` 으로 입고** |
| `DeliveryCreateRequest(soId, date, lines)` | `DeliveryCreateRequest(soId, **warehouseId**, date, lines)` |
| 출하는 SO 누적만 다룸 | 출하가 **재고를 차감** — 충분한 입고가 전제 |
| 검증: shipped_qty / invoiced_qty 누적, 상태 전이 | **동일** (본질은 안 변함) |

이게 **Phase 4 도입의 자연스러운 파급**. 출하가 더 이상 "SO 숫자만 더하는 행위" 가 아니라 "실제 재고를 움직이는 행위" 가 됐으므로, **재고 없이는 출하가 성립하지 않는다.** Phase 2 의 시나리오를 그대로 돌리려면 입고 셋업이 반드시 선행돼야 한다.

> 💡 주목할 점은 **검증 자체는 한 줄도 안 바뀌었다**는 것. shipped_qty/invoiced_qty 누적과 상태 전이(SHIPPING → INVOICING → INVOICED)는 그대로. Phase 4 는 출하의 *부수 효과*(재고 차감)를 추가했을 뿐, SO 누적이라는 *본질*은 건드리지 않았다. 잘 분리된 설계의 증거 — 새 책임을 더해도 기존 약속이 깨지지 않는다.

`DeliveryCreateRequest` 에 `warehouseId` 인자가 늘어난 것도 같은 맥락이다. "어느 창고에서 빼는가" 를 출하 시점에 알아야 재고를 차감할 수 있으므로 (V22 마이그레이션이 `delivery.warehouse_id` 컬럼 추가, 05편 참조).

## 🔥 테스트 격리 — unique 셋업과 V20~V22 시드의 공존

```java
private TestContext setup(int stockQty) {
    long nano = System.nanoTime();
    var customer = customerService.create(new CustomerCreateRequest(
            "현우테크-" + nano, uniqueBusinessNo(), ...));
    var item = itemService.create(...);
    Long warehouseId = warehouseId();
    stockUp(item.id(), warehouseId, stockQty);
    ...
}

private Long warehouseId() {
    return warehouseService.create(new WarehouseCreateRequest(
            "WH-" + whSuffix(), "테스트창고", "서울시")).id();
}
```

테스트마다 `System.nanoTime()` + `AtomicLong SEQ` 로 **고유한** 고객·품목·창고·사업자번호를 만든다. 이유는 Phase 3 동시성 테스트와 동일:
- 테스트 간 격리 — 한 테스트가 만든 데이터가 다른 테스트에 안 새도록.
- 시드(V20 의 `WH-HQ` 등)와의 충돌 회피 — 고정 코드를 쓰면 UNIQUE 제약에 걸린다.

`setup(int stockQty)` 의 인자 하나로 "재고를 얼마나 채워둘지" 만 바꿔가며 테스트 1(10대), 2(5대), 3(10대)이 같은 픽스처를 재사용한다. 테스트 5 만 "재고 행 없음" 을 만들려고 `setup` 을 안 쓰고 직접 셋업한다.

> 💡 `whSuffix()` 가 `WH-T...` 형식을 만드는 이유 — 창고 코드는 `^WH-[A-Z0-9_-]+$` 패턴(워크스루 01편)을 통과해야 하므로 접두사 `WH-` 를 붙인다. DTO 검증이 통합 테스트에서도 살아있다는 부수적 확인.
