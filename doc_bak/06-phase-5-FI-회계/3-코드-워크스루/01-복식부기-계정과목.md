# 1/7. 복식부기와 계정과목 — 회계의 두 토대를 코드로

> Phase 5 의 모든 코드는 두 개의 토대 위에 서 있다.
> **(1) 복식부기 — 한 거래는 차변 합 = 대변 합** (다음 글들에서 깊게 다룸)
> **(2) 계정과목(Account) — 모든 분개가 참조하는 마스터, 자기참조 트리**
> 이 글은 (2)를 짚는다. Department 트리 패턴을 그대로 가져오면서, 회계 고유의 두 개념 — **유형 → 정상 잔액 방향 파생** 과 **postable 헤더/말단 구분** — 을 어떻게 코드로 박았는지 본다.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/fi/account/
├─ Account.java                자기참조 트리 + postable + type
├─ AccountType.java            5종 enum + NormalSide 파생
├─ NormalSide.java             DEBIT / CREDIT
├─ AccountRepository.java      findByCode (계정 코드 lookup)
├─ AccountService.java         CRUD + getEntityByCode (FI 내부용)
├─ AccountController.java      /api/accounts
└─ dto/                        Request/Response

hwlee-erp/src/main/resources/db/migration/
└─ V23__create_account.sql     테이블 + 12개 계정 시드
```

## 🔥 복식부기, 30초 요약

회계의 "모든 거래는 두 측면을 가진다" 는 직관:

| 거래 | 어떤 측면이 늘었나? | 어떤 측면이 줄었나? (또는 부채/수익이 늘었나?) |
| --- | --- | --- |
| 외상으로 1만원어치 팜 | 매출채권(받을 돈, 자산) +10,000 | 매출(수익) +10,000 |
| 그 매출의 원가가 6천원이었다 | 매출원가(비용) +6,000 | 재고자산 -6,000 |
| 외상값 1만원 회수 | 현금 +10,000 | 매출채권 -10,000 |

이 두 측면을 회계는 **차변(Debit)** 과 **대변(Credit)** 으로 부른다. 한 거래의 차변 합과 대변 합은 **항상 같다**. 안 같으면 어딘가 빠뜨린 것이다 — 이게 "복식부기" 의 본질.

→ Phase 5 의 모든 자동 분개는 이 규칙을 코드로 박은 것. 그 규칙을 적용하려면 먼저 "**어느 계정이 차변/대변 어느 쪽이 정상인가**" 를 알아야 한다 → **계정과목**.

## 🔥 다섯 가지 유형 — 정상 잔액 방향이 결정된다

`AccountType.java`:

```java
@Getter
@RequiredArgsConstructor
public enum AccountType {
    ASSET(NormalSide.DEBIT),
    LIABILITY(NormalSide.CREDIT),
    EQUITY(NormalSide.CREDIT),
    REVENUE(NormalSide.CREDIT),
    EXPENSE(NormalSide.DEBIT);

    private final NormalSide normalSide;
}
```

회계의 5대 분류 — 모든 계정은 이 다섯 중 하나에 속한다. 그리고 **유형이 정해지면 정상 잔액 방향이 자동으로 결정**된다:

| 유형 | 정상 방향 | 직관 |
| --- | --- | --- |
| 자산(ASSET) | 차변 | "있는 것" — 늘면 차변에 적힌다 |
| 비용(EXPENSE) | 차변 | "쓴 것" — 늘면 차변 (자산이 줄고 비용이 늘었다) |
| 부채(LIABILITY) | 대변 | "갚을 것" — 늘면 대변 |
| 자본(EQUITY) | 대변 | "주주 몫" — 늘면 대변 |
| 수익(REVENUE) | 대변 | "번 것" — 늘면 대변 (자산이 늘고 수익이 늘었다) |

> 💡 외우는 요령 — "**자비차, 부자수대**" (자산·비용은 차변, 부채·자본·수익은 대변). 차변에 적힌 게 늘면 그 계정이 증가, 대변에 적힌 게 늘면 그 계정이 증가.

### 왜 `normalSide` 컬럼을 따로 안 두나 (설계 결정 #2)

**유형이 곧 방향**이기 때문이다. ASSET 인데 NormalSide=CREDIT 인 계정은 회계 원칙상 존재할 수 없다. 컬럼을 따로 두면 두 값이 어긋날 위험만 늘어난다 — **단일 진실(single source of truth)** 의 원칙.

`Account.normalSide()`:

```java
public NormalSide normalSide() {
    return type.getNormalSide();
}
```

DB 컬럼 한 줄 늘리는 대신 enum 한 줄로 끝낸다.

## 🔥 자기참조 트리 — Department 패턴 그대로

`Account.java`:

```java
@Entity
@Table(name = "account")
@SQLDelete(sql = "UPDATE account SET deleted_at = NOW() WHERE id = ?")
@SQLRestriction("deleted_at IS NULL")
public class Account extends BaseEntityWithCode {

    @Column(name = "name", nullable = false, length = 100)
    private String name;

    @Enumerated(EnumType.STRING)
    @Column(name = "type", nullable = false, length = 16)
    private AccountType type;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "parent_id")
    private Account parent;

    @Column(name = "postable", nullable = false)
    private boolean postable;
    ...
}
```

`Department` 와 똑같은 도구:

| 도구 | 같이 쓰는 이유 |
| --- | --- |
| `BaseEntityWithCode` | code/status/deletedAt + 감사 컬럼 한 묶음. 마스터 데이터의 공통 베이스 |
| `@ManyToOne self` | 트리 — 부모 계정을 self join 으로 가리킴 |
| `@SQLDelete` + `@SQLRestriction` | 소프트 삭제 — 회계 계정은 트랜잭션이 참조 중이라 함부로 못 지운다 |

다른 점은 두 컬럼:

- `type` — Department 엔 없는, 회계 고유의 분류
- `postable` — 헤더 계정인가 말단 계정인가

## 🔥 `postable` — 헤더와 말단의 구분

실무 회계는 계정을 **트리** 로 본다:

```
1000 자산          (postable=false, 헤더)
├─ 1100 현금        (postable=true,  말단)
├─ 1200 매출채권    (postable=true,  말단)
├─ 1400 재고자산    (postable=true,  말단)
└─ 2300 부가세대급금 (postable=true,  말단; 자산이라 1000 아래에 둠)
2000 부채          (postable=false, 헤더)
├─ 2100 매입채무    (postable=true,  말단)
└─ 2200 부가세예수금 (postable=true,  말단)
...
```

"자산이 1억" 이라고 말할 때 1000(자산) 계정에 직접 1억을 적지 않는다. 1100(현금), 1200(매출채권), 1400(재고) ... 말단 계정에 각각 적고, 헤더는 그 **합계** 만 본다. 그래서 **헤더 계정에는 전표 라인을 직접 달 수 없다** — 이게 실무 회계의 기본 규칙.

이 규칙을 코드로 박은 곳은 `JournalLine.java` (다음 글에서 자세히):

```java
JournalLine(JournalEntry entry, int lineNo, Account account, BigDecimal debit, BigDecimal credit) {
    ...
    if (!account.isPostable()) {
        throw new IllegalArgumentException(
                "헤더 계정에는 라인을 달 수 없다 — postable=false: code=" + account.getCode());
    }
    ...
}
```

생성자에서 거부. 도메인 객체가 회계 규칙을 직접 지킨다.

## 🔥 시드 계정 12개 (V23) — 학습에 필요한 최소

`V23__create_account.sql`:

| 코드 | 이름 | 유형 | postable | 부모 |
| --- | --- | --- | --- | --- |
| 1000 | 자산 | ASSET | N | — |
| 1100 | 현금 | ASSET | Y | 1000 |
| 1200 | 매출채권 | ASSET | Y | 1000 |
| 1400 | 재고자산 | ASSET | Y | 1000 |
| 2300 | 부가세대급금 | **ASSET** | Y | 1000 |
| 2000 | 부채 | LIABILITY | N | — |
| 2100 | 매입채무 | LIABILITY | Y | 2000 |
| 2200 | 부가세예수금 | LIABILITY | Y | 2000 |
| 4000 | 수익 | REVENUE | N | — |
| 4100 | 매출 | REVENUE | Y | 4000 |
| 5000 | 비용 | EXPENSE | N | — |
| 5100 | 매출원가 | EXPENSE | Y | 5000 |

> ⚠️ **부가세대급금(2300)** — 코드는 2xxx 이라 부채처럼 보이지만 유형은 ASSET. "매입 시 미리 낸 부가세는 자산(받을 돈)" 이라는 회계 원리 때문. **코드 번호와 유형이 항상 일치하지는 않는다 — 유형 컬럼이 진실**. 이게 §2의 핵심 결정 #2 "별도 normalSide 컬럼 없음" 의 또 다른 이유 — 유형만 보면 안전, 코드 보고 추측하면 위험.

시드 INSERT 는 두 단계로 나뉜다:

```sql
-- 1) 헤더 계정 4개 — parent_id = NULL
INSERT INTO account (code, name, type, parent_id, postable, status, ...)
VALUES
    ('1000', '자산',          'ASSET',     NULL, 0, 'ACTIVE', ...),
    ('2000', '부채',          'LIABILITY', NULL, 0, 'ACTIVE', ...),
    ('4000', '수익',          'REVENUE',   NULL, 0, 'ACTIVE', ...),
    ('5000', '비용',          'EXPENSE',   NULL, 0, 'ACTIVE', ...);

-- 2) 말단 계정 — 부모를 code 로 lookup
INSERT INTO account (code, name, type, parent_id, postable, status, ...)
SELECT '1100', '현금', 'ASSET', id, 1, 'ACTIVE', ...
FROM account WHERE code = '1000';
...
```

FK(`fk_account_parent`) 제약 때문에 **부모를 먼저 넣고**, 자식은 부모의 id 를 `SELECT id FROM account WHERE code = '...'` 로 가져와서 INSERT 한다. 시드 SQL 의 흔한 패턴.

## 🔥 `AccountService.getEntityByCode` — FI 내부용 공개 메서드

```java
public Account getEntityByCode(String code) {
    return repository.findByCode(code)
            .orElseThrow(() -> new EntityNotFoundException("Account not found: code=" + code));
}
```

다른 모든 서비스 메서드는 `AccountResponse` (DTO) 를 반환하는데, 이 하나만 **엔티티**를 그대로 반환한다. 이유:

- `AutoJournalService` 가 분개 라인을 만들 때 `Account` 엔티티가 필요 (FK 매핑).
- 외부 컨트롤러가 부르는 게 아니라 같은 FI 모듈의 내부 호출.
- 매번 DTO → 엔티티 변환을 거치는 게 무의미.

> 💡 "엔티티를 서비스 밖으로 내보내지 마라" 는 일반 원칙은 컨트롤러/외부 모듈을 향한 것. 같은 모듈 내부 협력에서는 엔티티 직접 전달이 깔끔.

`AutoJournalService` 에서 쓰는 모양:

```java
private Account account(String code) {
    try {
        return accountService.getEntityByCode(code);
    } catch (EntityNotFoundException e) {
        throw new IllegalStateException(
                "시스템 계정 " + code + " 가 마스터에 없다. V23 시드 점검 필요.", e);
    }
}
```

시드가 누락되면 `IllegalStateException` 으로 변환 — "어디서 무엇이 잘못됐는지" 가 stack trace 에서 바로 보이게.

## 🔥 다른 모듈과의 비교

| Phase | 트리 마스터 | 자기참조 트리 | 추가 컬럼 |
| --- | --- | --- | --- |
| Phase 1 | `Department` | parent_id | name |
| Phase 3 | (없음) | — | — |
| **Phase 5** | **`Account`** | parent_id | name, type, postable |

같은 패턴(BaseEntityWithCode + 자기참조)을 두 번 적용한 것. 이 트리 패턴은 ERP 마스터에서 자주 나오는 모양 — 다음에 또 트리가 필요하면(예: 품목 카테고리, 거래처 그룹) 같은 방법으로 작성한다.

## 정리

| 학습 포인트 | 코드 위치 |
| --- | --- |
| 회계의 5대 유형 → 정상 잔액 방향 파생 | `AccountType.java` |

| FI 내부용 엔티티 노출 (`getEntityByCode`) | `AccountService` |
