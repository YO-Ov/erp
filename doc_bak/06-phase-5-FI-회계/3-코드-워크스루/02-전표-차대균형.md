# 2/7. 회계 전표와 차대 균형 — 복식부기의 본질을 코드가 지킨다 ⭐

> 복식부기의 핵심 한 줄: **"한 거래의 차변 합 = 대변 합"**. 이게 깨지면 회계는 무너진다. 이 글은 그 한 줄을 도메인 객체가 어떻게 직접 지키는지를 본다 — `JournalEntry.post()` 의 검증과 `UnbalancedJournalException`.
>
> Phase 2 의 "수주 → 인보이스 상태 전이", Phase 3 의 "음수 재고 거부(`InsufficientStockException`)" 와 같은 자리 — **"도메인 불변식을 코드가 지킨다"** 패턴의 회계판.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/fi/journal/
├─ JournalEntry.java               헤더 + post() 차대 검증
├─ JournalLine.java                debit/credit 두 컬럼, postable 검증
├─ JournalEntryStatus.java         DRAFT / POSTED / CANCELLED
├─ JournalSource.java              INV/GI/GR/PAY/MANUAL — 다형성 참조
├─ UnbalancedJournalException.java 차/대 불일치 시 던지는 예외
├─ JournalEntryService.java        수동 전표 등록 + 공통 조회
└─ ...

hwlee-erp/src/main/java/com/hwlee/erp/common/error/
└─ GlobalExceptionHandler.java     UnbalancedJournalException → 422

hwlee-erp/src/main/resources/db/migration/
└─ V24__create_journal.sql         journal_entry + journal_line + CHECK 제약
```

## 🔥 헤더-라인 구조 — 익숙한 패턴

`JournalEntry` 는 헤더, `JournalLine` 은 라인. Phase 2 의 `SalesOrder` ↔ `SalesOrderLine`, `Invoice` ↔ `InvoiceLine` 과 같은 모양:

```
JournalEntry (헤더)
├─ number          JE-YYYYMMDD-NNN
├─ entryDate       전표일자
├─ description     적요 (사람이 읽는 한 줄 설명)
├─ status          DRAFT / POSTED / CANCELLED
├─ sourceType      INV / GI / GR / PAY / MANUAL  (출처 분류)
├─ sourceId        출처 트랜잭션 id (다형성 약한 참조)
└─ postedAt        확정 시각

JournalLine (라인, 헤더당 N개)
├─ accountId       FK → account (분개 라인이 가리키는 계정)
├─ lineNo          순서
├─ debit           차변 금액 (대변 라인이면 0)
└─ credit          대변 금액 (차변 라인이면 0)
```

차/대 표현 방식은 세 옵션 중 **B(두 컬럼)** 채택 (설계 §3.2):

| 옵션 | 표현 | 비고 |
| --- | --- | --- |
| A | `amount` + `side`(DEBIT/CREDIT) enum | 한 컬럼이지만 SUM 시 조건 분기 |
| **B** | **`debit`, `credit` 두 컬럼** (한쪽만 > 0) | ✅ 분개장 그대로. SUM 직관적 |
| C | 부호 있는 `amount` (차변 +, 대변 -) | 코드는 짧지만 "-1,200,000 이 대변" 이 회계 직관과 어긋남 |

**B를 고른 이유** — 회계 장부(분개장) 가 실제로 차변·대변 두 칸으로 나뉜 모습 그대로다. 사람이 보기 가장 쉽고, "차변 합 / 대변 합" 계산이 `SUM(debit)`, `SUM(credit)` 으로 직관적. 학습 가치 ↑.

## 🔥 한 라인은 한쪽만 양수 — 세 겹의 방어

같은 라인에 `debit > 0, credit > 0` 이면 분개가 모호해진다. 그래서 **한 라인은 한쪽만 양수** — 이 규칙을 세 겹으로 막는다.

### 1겹 — `JournalLine` 생성자

`JournalLine.java`:

```java
JournalLine(JournalEntry entry, int lineNo, Account account, BigDecimal debit, BigDecimal credit) {
    ...
    boolean debitPositive  = debit.signum() > 0;
    boolean creditPositive = credit.signum() > 0;
    if (debitPositive == creditPositive) {
        throw new IllegalArgumentException(
                "라인은 차변 또는 대변 중 한 쪽만 양수여야 한다. debit=" + debit + ", credit=" + credit);
    }
    if (debit.signum() < 0 || credit.signum() < 0) {
        throw new IllegalArgumentException("debit/credit 은 음수일 수 없다.");
    }
    ...
}
```

`debitPositive == creditPositive` 한 줄로 **둘 다 양수** + **둘 다 0** 두 경우를 동시에 막는다.

### 2겹 — `JournalEntry.addDebit` / `addCredit` 만 공개

```java
public JournalLine addDebit(Account account, BigDecimal amount) {
    return addLine(account, amount, BigDecimal.ZERO);
}

public JournalLine addCredit(Account account, BigDecimal amount) {
    return addLine(account, BigDecimal.ZERO, amount);
}

private JournalLine addLine(Account account, BigDecimal debit, BigDecimal credit) { ... }
```

호출자가 차/대를 직접 못 섞게 — `addDebit` 은 credit 을 ZERO 로, `addCredit` 은 debit 을 ZERO 로 강제. private `addLine` 만 두 값을 받지만 외부에서 부를 수 없다.

### 3겹 — DB CHECK 제약

`V24__create_journal.sql`:

```sql
CONSTRAINT chk_journal_line_side CHECK (
    (debit > 0 AND credit = 0) OR (debit = 0 AND credit > 0)
)
```

코드 우회(JPA 외 직접 INSERT, 마이그레이션 실수 등)도 DB 가 막는다 — **방어선의 마지막 줄**.

## 🔥 차변 합 = 대변 합 — 복식부기의 본질 ⭐

`JournalEntry.post()`:

```java
public void post(LocalDateTime now) {
    if (status != JournalEntryStatus.DRAFT) {
        throw new IllegalStateException("DRAFT 전표만 확정 가능합니다. 현재: " + status);
    }
    if (lines.isEmpty()) {
        throw new IllegalStateException("라인이 없는 전표는 확정할 수 없다.");
    }
    BigDecimal totalDebit = lines.stream()
            .map(JournalLine::getDebit)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    BigDecimal totalCredit = lines.stream()
            .map(JournalLine::getCredit)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    if (totalDebit.compareTo(totalCredit) != 0) {
        throw new UnbalancedJournalException(totalDebit, totalCredit);
    }
    this.status = JournalEntryStatus.POSTED;
    this.postedAt = now;
}
```

세 가지를 짚는다:

1. **상태 가드** — DRAFT 만 post 가능. 두 번 post 막음.
2. **빈 전표 거부** — 라인 0개는 회계적으로 무의미.
3. **차변 합 = 대변 합** — `BigDecimal.compareTo(other) != 0` 로 비교. `equals` 는 scale 까지 같아야 같다고 보지만(`1.0 != 1.00`), `compareTo` 는 수학적 동치만 본다. 금액 비교의 표준 패턴.

전부 통과해야 비로소 `status = POSTED` 로 전이.

> 💡 한 가지 안 보일 수도 있는 디테일 — `post()` 가 실패해도 `status` 는 **그대로 DRAFT**. 예외가 나기 전에 상태를 바꾸지 않았기 때문. 트랜잭션이 롤백되지 않아도 도메인 객체가 안전한 상태를 유지한다 (in-memory 안전성).

## 🔥 `UnbalancedJournalException` — Phase 3 의 자매

`UnbalancedJournalException.java`:

```java
@Getter
public class UnbalancedJournalException extends RuntimeException {
    private final BigDecimal totalDebit;
    private final BigDecimal totalCredit;

    public UnbalancedJournalException(BigDecimal totalDebit, BigDecimal totalCredit) {
        super(String.format("전표 차/대 불일치 — 차변 합 %s, 대변 합 %s", totalDebit, totalCredit));
        ...
    }
}
```

Phase 3 의 `InsufficientStockException`, Phase 2 의 "신용한도 초과" 와 같은 디자인:

- **RuntimeException** 상속 — 도메인 위반은 보통 unchecked.
- **불일치 양쪽 값을 필드로** — 응답에 노출해 디버깅 도움.

`GlobalExceptionHandler` 가 422 (UNPROCESSABLE_ENTITY) 로 변환:

```java
@ExceptionHandler(UnbalancedJournalException.class)
public ProblemDetail handleUnbalancedJournal(UnbalancedJournalException ex) {
    ProblemDetail problem = ProblemDetail.forStatusAndDetail(HttpStatus.UNPROCESSABLE_ENTITY, ex.getMessage());
    problem.setTitle("Unbalanced Journal Entry");
    problem.setProperty("code", "UNBALANCED_JOURNAL");
    problem.setProperty("totalDebit", ex.getTotalDebit());
    problem.setProperty("totalCredit", ex.getTotalCredit());
    return problem;
}
```

`422` 를 고른 이유: 입력 형식(JSON 파싱)은 맞지만 **도메인 규칙 위반**. `400` 은 형식 오류, `422` 는 의미 오류 — 회계 차/대 불일치는 후자에 정확히 들어맞는다 (Phase 3 의 `InsufficientStockException` 은 동시성 충돌 의미로 `409`).

응답:

```json
{
  "type": "about:blank",
  "title": "Unbalanced Journal Entry",
  "status": 422,
  "detail": "전표 차/대 불일치 — 차변 합 13200000.00, 대변 합 12000000.00",
  "code": "UNBALANCED_JOURNAL",
  "totalDebit": 13200000.00,
  "totalCredit": 12000000.00
}
```

클라이언트가 "어디서 얼마나 어긋났는지" 바로 본다.

## 🔥 상태 머신 — DRAFT → POSTED → CANCELLED

```
DRAFT ──post()──▶ POSTED ──cancel()──▶ CANCELLED
                            ▲
                            └─ 역분개 별도 전표 추가 (실무 회계 정석)
```

자동 분개는 **draft + post 한 번에**:

```java
// AutoJournalService.createSalesEntry
JournalEntry entry = JournalEntry.draft(...);
entry.addDebit(...);
entry.addCredit(...);
entry.post(now());      // ← 차/대 검증 + POSTED
return repo.save(entry);
```

사람 손이 거치지 않으니 DRAFT 로 둘 이유가 없다. Phase 4 의 `GoodsIssueService.createAndPostFromDelivery` 와 같은 패턴 — "한 번에 만들고 확정".

수동 전표(`MANUAL`) 도 같은 패턴 (학습 단순화). 실무에선 결재 워크플로가 들어가지만, 그건 Phase 6(인증/인가) 이후에 도입.

`cancel()` 은 status 만 CANCELLED 로 바꾼다:

```java
public void cancel() {
    if (status != JournalEntryStatus.POSTED) {
        throw new IllegalStateException("POSTED 전표만 취소 가능합니다. 현재: " + status);
    }
    this.status = JournalEntryStatus.CANCELLED;
}
```

> ⚠️ 회계 정석은 "취소" 가 아니라 **역분개(reverse entry)** — 차/대 반대인 별도 전표를 추가하는 것. 원장은 절대 수정/삭제하지 않고 새 행만 쌓는 append-only 가 회계의 철학(Phase 3 stock_movement 와 동일). 학습 단순화로 status 변경만 채택했지만, 잔액 SUM 시 `status = POSTED` 만 더하므로 결과적으로 같다 — `JournalEntryRepository.findPostedLinesByAccountCode` 가 이를 강제 (다음 글들에서).

## 🔥 출처 — `sourceType` + `sourceId` 약한 참조

`JournalEntry` 헤더의 두 컬럼:

```java
@Enumerated(EnumType.STRING)
@Column(name = "source_type", nullable = false, length = 16)
private JournalSource sourceType;   // INV / GI / GR / PAY / MANUAL

@Column(name = "source_id")
private Long sourceId;              // 출처 트랜잭션 id (MANUAL 이면 null)
```

Phase 3 의 `StockMovement.refType` / `refId` 와 같은 **다형성 약한 참조** 패턴:

- 출처가 4가지(invoice/goods_issue/goods_receipt/payment) — 단일 FK 로 묶을 수 없다.
- 그래서 (분류, id) 쌍으로 표현. **FK 제약은 안 건다** — 출처 테이블별로 따로 걸어야 하는데 그럼 다형성이 깨진다.

`draft()` 팩토리에서 정합성 검증:

```java
public static JournalEntry draft(String number, LocalDate entryDate, String description,
                                 JournalSource sourceType, Long sourceId) {
    ...
    if (sourceType != JournalSource.MANUAL && sourceId == null) {
        throw new IllegalArgumentException("MANUAL 이 아닌 경우 sourceId 는 필수다.");
    }
    ...
}
```

자동 전표(INV/GI/GR/PAY)는 sourceId 필수, 수동 전표(MANUAL)는 null 허용.

조회 시 활용:

```java
// "INV-001 인보이스로 만든 전표를 보여줘"
List<JournalEntry> entries = journalRepository
        .findBySourceTypeAndSourceId(JournalSource.INV, invoiceId);
```

→ 영업이 인보이스 한 건 잘못 발행했을 때 "회계에 뭐가 박혔지?" 를 한 번에 추적. 시연·디버깅의 단골 쿼리. 인덱스는 `(source_type, source_id)` 복합 (V24).

## 🔥 `JournalEntryService.createManual` — 수동 전표 등록

자동 분개가 9할이지만, 수동 입력 API도 있어야 한다 (월말 조정/오류 정정/이체).

```java
@Transactional
public JournalEntryResponse createManual(JournalEntryCreateRequest req) {
    String number = numberGenerator.nextJournalEntryNumber(req.entryDate());
    JournalEntry entry = JournalEntry.draft(
            number, req.entryDate(), req.description(), JournalSource.MANUAL, null);
    for (JournalLineRequest lineReq : req.lines()) {
        Account account = accountService.getEntityByCode(lineReq.accountCode());
        BigDecimal debit = lineReq.debit();
        BigDecimal credit = lineReq.credit();
        if (debit.signum() > 0) {
            entry.addDebit(account, debit);
        } else if (credit.signum() > 0) {
            entry.addCredit(account, credit);
        } else {
            throw new IllegalArgumentException(
                    "라인은 차변 또는 대변 중 한 쪽이 > 0 이어야 한다. accountCode=" + lineReq.accountCode());
        }
    }
    entry.post(LocalDateTime.now(clock));
    return mapper.toResponse(repository.save(entry));
}
```

세 가지 결정:

1. **즉시 POSTED** — DRAFT 단계 생략. 학습 단순화.
2. **DTO 의 debit/credit 을 그대로 도메인에 넘기지 않음** — DTO 는 양쪽 0 도 받지만, 서비스가 명시적으로 한 쪽이 양수인지 확인 후 `addDebit` / `addCredit` 분기.
3. **계정 lookup 은 `accountCode`** — DTO 에 `accountId` 를 받으면 호출자가 ID 를 외워야 한다. 회계 코드 (`1200`, `4100`) 가 자연스러운 식별자.

## 🔥 정리 — 이 한 줄이 회계 모듈의 본체

```java
if (totalDebit.compareTo(totalCredit) != 0) {
    throw new UnbalancedJournalException(totalDebit, totalCredit);
}
```

Phase 5 전체에서 가장 중요한 코드 한 줄. 자동 분개(매출/매입/매출원가/입금/출금) 모두 이 검증을 거친다. 자동 분개 규칙이 잘못 짜여 차/대가 안 맞으면 → 발행자(인보이스 발행/출하 확정/입고 확정/입금) 트랜잭션 통째가 롤백 (다음 글 §리스너 BEFORE_COMMIT).

| 학습 포인트 | 코드 위치 |
| --- | --- |
| 헤더-라인 (Invoice 패턴 재사용) | `JournalEntry` ↔ `JournalLine` |
| debit/credit 두 컬럼 (옵션 B) | `JournalLine` `debit`, `credit` |
| 한 라인은 한쪽만 양수 — 3겹 방어 | 생성자 + addDebit/addCredit + DB CHECK |
| 차변 합 = 대변 합 — 복식부기의 본질 ⭐ | `JournalEntry.post()` |
| BigDecimal `compareTo` for 금액 비교 | `post()` 내 비교 |
| UnbalancedJournalException → 422 | `GlobalExceptionHandler` |
| 출처 = sourceType + sourceId (다형성 약 참조) | `JournalEntry`, `StockMovement` 와 같은 패턴 |
| 자동 전표는 draft + post 한 번에 | `AutoJournalService` (다음 글) |

다음 글(`03-자동분개-규칙.md`) 에서는 이 전표를 **자동으로 만드는 규칙** — 사건별 분개 매핑과 `AutoJournalService` 를 본다.
