# Phase 6 시연 가이드 — 인증/인가 + 감사 로그 직접 돌려보기

> 7단계 사이클의 **단계 7: 시연 + 회고**.
> 지금까지는 Swagger UI에서 **누구나** API를 호출했다. 이번엔 **로그인해야 호출되고, 역할에 따라 막히고, 누가 바꿨는지 기록되는** 흐름을 직접 확인한다.
> 네 가지를 눈으로 본다: ① 로그인 → JWT 발급, ② 역할별 인가(401/403), ③ 변경자 자동 기록(층위①), ④ 감사 이력 조회(층위②).

## 0. 사전 준비

### 0-1. 인프라 기동

```bash
cd hwlee-erp
docker compose up -d        # MySQL 8
./gradlew bootRun
```

- 로그인 화면: http://localhost:8080/login
- Swagger UI: http://localhost:8080/swagger-ui.html
- 로그는 `bootRun` 콘솔에서 실시간 확인.

### 0-2. 시드 계정 (V28) — 모두 비밀번호 `pass1234`

| 계정(username) | 역할 | 접근 가능 | 비고 |
| --- | --- | --- | --- |
| `kim@hwlee-erp.example` | SALES | SD(영업), 마스터 조회 | 영업팀 |
| `lee@hwlee-erp.example` | FINANCE | FI(회계), 마스터 조회 | 재무팀 |
| `park@hwlee-erp.example` | (없음) | 없음 | 인증은 되나 전부 403 — 시연용 |
| `admin@hwlee-erp.example` | ADMIN | 전 모듈 + 사용자/역할 관리 + 감사로그 | 관리자 |

> 💡 비밀번호는 DB에 BCrypt 해시로만 저장돼 있다(`$2y$10$...`). 평문 `pass1234`는 어디에도 저장되지 않는다.

## 1. 시연 시나리오 전체 그림

```
① 로그인 없이 호출        → 401 (인증 필요)
② 로그인 → JWT 발급       → accessToken + 역할
③ kim(SALES) 으로 호출
      ├─ GET /api/sales-orders   → 200 (SD 허용)
      └─ GET /api/journal-entries → 403 (FI 거부) ⭐ 인가
④ admin 으로 고객 수정     → created_by/updated_by = admin (층위①) ⭐
⑤ 감사 로그 조회           → 변경 이력 누적 (층위②) ⭐
⑥ 브라우저 화면            → 로그인 → 역할별 메뉴 → 관리자 역할 조정
```

## 2. API 시연 (Swagger UI 또는 curl)

### ① 로그인 없이 호출 → 401

먼저 인증이 진짜 걸리는지 확인. 토큰 없이:

```bash
curl -i http://localhost:8080/api/customers
```

→ **401 Unauthorized** + ProblemDetail JSON:
```json
{ "status": 401, "detail": "인증이 필요합니다. 로그인 후 다시 시도해 주세요." }
```

> Phase 5까지는 이게 200이었다. 보안의 "구멍"이 막혔다(브리핑 §1).

---

### ② 로그인 → JWT 발급

```bash
curl -i -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"kim@hwlee-erp.example","password":"pass1234"}'
```

→ **200** + 토큰:
```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJraW0...",
  "tokenType": "Bearer",
  "expiresIn": 3600,
  "roles": ["SALES"]
}
```

- 응답 헤더에 `Set-Cookie: ACCESS_TOKEN=eyJ...; HttpOnly; SameSite=Strict` 도 함께 온다(브라우저용).
- `accessToken` 값을 복사해 둔다(아래 `$TOKEN`).

> 💡 **틀린 비밀번호로 시도하면 401.** `"password":"wrong"` 으로 바꿔 확인해 보라.
> 💡 토큰 페이로드(가운데 토막)를 https://jwt.io 에 붙여 보면 `sub`, `roles`, `exp`가 보인다 — "누구나 까볼 수 있다"(브리핑 §3-2)를 직접 확인. 단 서명은 못 위조한다.

### Swagger에서 토큰 넣는 법

Swagger UI 우측 상단 **Authorize** 버튼 → `Bearer eyJ...` 입력 → 이후 모든 호출에 자동 첨부.

curl이면 매 요청에:
```bash
TOKEN="여기에-accessToken-붙여넣기"
```

---

### ③ 인가 — kim(SALES)은 SD만, FI는 403 ⭐

SD 모듈 — **허용(200)**:
```bash
curl -i http://localhost:8080/api/sales-orders \
  -H "Authorization: Bearer $TOKEN"
```
→ 200, 수주 목록.

FI 모듈 — **거부(403)**:
```bash
curl -i http://localhost:8080/api/journal-entries \
  -H "Authorization: Bearer $TOKEN"
```
→ **403 Forbidden**:
```json
{ "status": 403, "detail": "이 작업을 수행할 권한이 없습니다." }
```

> ⭐ 핵심 차이: ②의 401("너 누구야?")과 여기 403("누군진 알겠는데 권한 없어")은 **완전히 다른 실패**다(브리핑 §2). kim은 인증은 됐지만 FINANCE 역할이 없다.

마스터는 조회 OK, 변경은 403:
```bash
curl -i http://localhost:8080/api/customers -H "Authorization: Bearer $TOKEN"          # 200 (조회)
curl -i -X DELETE http://localhost:8080/api/customers/1 -H "Authorization: Bearer $TOKEN"  # 403 (변경은 ADMIN만)
```

> 대조: `park@hwlee-erp.example`로 로그인해 같은 호출을 하면 **SD조차 403**(역할이 아예 없음). `lee@...`(FINANCE)로 하면 `/api/journal-entries`가 **200**.

---

### ④ 변경자 자동 기록 (층위 ①) ⭐ — 복선 회수

`admin`으로 로그인해 토큰을 받은 뒤(`$ADMIN_TOKEN`), 고객을 하나 수정:

```bash
# admin 로그인
curl -s -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin@hwlee-erp.example","password":"pass1234"}'
# → accessToken 복사 → $ADMIN_TOKEN

# 고객 1번 수정
curl -i -X PUT http://localhost:8080/api/customers/1 \
  -H "Authorization: Bearer $ADMIN_TOKEN" -H "Content-Type: application/json" \
  -d '{"name":"신원전자","address":"서울 변경","creditLimit":150000000,"paymentTerms":"NET30"}'

# 수정 결과 조회 — updatedBy 확인
curl -s http://localhost:8080/api/customers/1 -H "Authorization: Bearer $ADMIN_TOKEN"
```

→ 응답의 `updatedBy` 가 **`"system"`이 아니라 `"admin@hwlee-erp.example"`**.

> ⭐ Phase 1에서 심은 `auditorProvider` 복선이 회수된 결과(워크스루 ④). **Customer 코드는 한 줄도 안 고쳤는데** 변경자가 진짜 사용자로 박힌다. Phase 5까지 생성된 행들의 `created_by`는 여전히 `system`(그땐 인증이 없었으니까) — 대조해 보면 명확하다.

---

### ⑤ 감사 이력 조회 (층위 ②) ⭐

④에서 고객을 수정했으니 `audit_log`에 이력이 쌓였다. 감사 로그는 ADMIN/FINANCE만 조회 가능:

```bash
curl -s "http://localhost:8080/api/audit-logs?entityType=Customer&entityId=1" \
  -H "Authorization: Bearer $ADMIN_TOKEN"
```

→ 최신순 이력:
```json
{
  "content": [
    {
      "entityType": "Customer", "entityId": 1, "action": "UPDATE",
      "changedBy": "admin@hwlee-erp.example",
      "changedAt": "2026-05-30T...",
      "changes": "{\"code\":\"CUST-2026-0001\",\"name\":\"신원전자\",\"creditLimit\":150000000,...}"
    }
  ]
}
```

> ⭐ 층위 ①(현재 변경자)과 층위 ②(이력 전체)의 차이를 직접 본다. ④의 `updatedBy`는 "마지막에 누가"만 알지만, 여기 `audit_log`는 **변경할 때마다 한 행씩 쌓여** 신용한도가 어떻게 바뀌어 왔는지 전 과정이 남는다(브리핑 §6). `changes` JSON에 그 시점 스냅샷이 담긴다.
>
> kim(SALES) 토큰으로 같은 감사 로그를 조회하면 **403** — 감사 정보 자체가 통제 대상.

## 3. 브라우저 화면 시연 (Thymeleaf)

### 3-1. 로그인 → 메뉴

1. http://localhost:8080/login 접속 (인증 안 했으니 `/`로 가도 자동 리다이렉트됨).
2. `kim@hwlee-erp.example` / `pass1234` 로 로그인.
3. → `/` 메뉴 화면. **영업(SD) 카드만 보인다.** 회계·관리자 카드는 숨겨짐(`sec:authorize`).
4. 로그아웃 후 `admin@hwlee-erp.example`로 로그인 → **모든 카드 + 관리자 카드**가 보인다.

> 💡 로그인 시 받은 토큰은 **HttpOnly 쿠키**로 저장돼, 이후 페이지 이동에 자동 첨부된다. 개발자도구 → Application → Cookies에서 `ACCESS_TOKEN`을 확인할 수 있지만, JS로는 읽히지 않는다(XSS 방어).

### 3-2. 관리자 — 역할 수동 조정

1. admin으로 로그인 → 관리자 카드의 **사용자 관리** → `/admin/users`.
2. `park@hwlee-erp.example` 행에서 역할 체크박스에 **SALES** 체크 → [저장].
3. 로그아웃 → `park`로 로그인 → 이제 영업(SD) 메뉴가 보이고 `/api/sales-orders`도 200.

> 💡 시드는 부서로 기본 역할을 깔고(park은 생산팀이라 역할 없음), 이후 변화는 관리자가 화면에서 손본다 — "부서 기반 시드 + 수동 조정"(확정 ③)의 실제 동작.
