## 🔥 새 계정 5개 — 인건비의 양면

급여 한 번은 회계적으로 여러 측면을 동시에 만든다. 그걸 담을 말단 계정 5개를 Phase 5 트리(1000/2000/4000/5000)에 추가했다:

| 코드 | 계정명 | 유형 | 정상 | 역할 |
| --- | --- | --- | --- | --- |
| 5200 | 급여비용 | EXPENSE | 차변 | 직원 총지급액(gross) |
| 5300 | 법정복리비 | EXPENSE | 차변 | **4대보험 회사부담분** |
| 2400 | 예수금-소득세 | LIABILITY | 대변 | 떼어둔 소득세(나라에 낼 돈) |
| 2500 | 예수금-사회보험 | LIABILITY | 대변 | 4대보험 직원분+회사분(공단에 낼 돈) |
| 2600 | 미지급급여 | LIABILITY | 대변 | 직원에게 줄 실수령액 |

```java
// SystemAccounts.java
public static final String SALARY_EXPENSE = "5200";        // 급여비용
public static final String LEGAL_WELFARE = "5300";         // 법정복리비
public static final String WITHHOLDING_TAX = "2400";       // 예수금-소득세
public static final String WITHHOLDING_INSURANCE = "2500"; // 예수금-사회보험
public static final String SALARY_PAYABLE = "2600";        // 미지급급여
```

> **"예수금"이 뭔가**: 회사가 직원 월급에서 소득세·보험료를 **미리 떼어 보관**했다가 나중에 나라/공단에 대신 내준다. 떼어둔 그 돈은 회사 것이 아니라 "남의 돈을 잠시 맡은 것" → **부채(예수금)**. 직원에게 줄 실수령액(미지급급여)도 "아직 안 준 빚" → 부채.

## 🔥 급여 확정 분개 — 차) 비용 / 대) 떼어둔 것 + 줄 것

`createPayrollEntry` 가 대장 전체 합산으로 한 장의 전표를 만든다:

```java
@Transactional
public JournalEntry createPayrollEntry(PayrollConfirmedEvent event) {
    String number = numberGenerator.nextJournalEntryNumber(event.entryDate());
    JournalEntry entry = JournalEntry.draft(
            number, event.entryDate(), "급여 " + event.number(),
            JournalSource.PAYROLL, event.payrollRunId());

    entry.addDebit(account(SystemAccounts.SALARY_EXPENSE), event.totalGross());           // 차) 급여비용
    entry.addDebit(account(SystemAccounts.LEGAL_WELFARE),  event.totalInsuranceCompany()); // 차) 법정복리비
    entry.addCredit(account(SystemAccounts.WITHHOLDING_TAX),       event.totalIncomeTax());          // 대) 예수금-소득세
    entry.addCredit(account(SystemAccounts.WITHHOLDING_INSURANCE),
            event.totalInsuranceEmployee().add(event.totalInsuranceCompany()));                       // 대) 예수금-사회보험
    entry.addCredit(account(SystemAccounts.SALARY_PAYABLE),        event.totalNet());                 // 대) 미지급급여

    entry.post(LocalDateTime.now(clock));   // ← 여기서 차/대 균형 검증 (Phase 5)
    return journalRepository.save(entry);
}
```

분개를 한글로 읽으면:

```
차) 급여비용(5200)        Σgross           "직원에게 줄 총액은 비용이다"
차) 법정복리비(5300)      Σ회사분 보험      "회사가 더 내는 보험료도 비용이다"
        대) 예수금-소득세(2400)      Σ소득세              "월급에서 뗀 소득세는 나중에 나라에"
        대) 예수금-사회보험(2500)    Σ(직원분+회사분)     "뗀 보험료 + 회사분도 공단에"
        대) 미지급급여(2600)         Σnet                 "직원에게 실제 줄 돈은 아직 빚"
```

## 🔥 왜 항상 균형인가 — net 의 정의가 증명이다 ⭐

Phase 5 의 `JournalEntry.post()` 는 차변 합 ≠ 대변 합이면 `UnbalancedJournalException` 을 던진다. 급여 전표는 **왜 절대 안 던지는가**?

차변 합과 대변 합을 식으로 펼쳐보자 (Σ 는 대장 전체 직원 합):

```
차변 합 = Σgross + Σ회사분
대변 합 = Σ소득세 + Σ(직원분 + 회사분) + Σnet
```

그런데 명세서 정의(3글)에서 **각 직원의** `net = gross − 소득세 − 직원분`. 양변에 Σ 를 씌우면:

```
Σnet = Σgross − Σ소득세 − Σ직원분
```

이걸 대변 합에 대입:

```
대변 합 = Σ소득세 + Σ직원분 + Σ회사분 + (Σgross − Σ소득세 − Σ직원분)
        = Σgross + Σ회사분
        = 차변 합   ✅
```

> 💡 **라운딩과 무관하다.** 각 직원의 소득세·보험료를 원 단위로 아무리 반올림해도, `net` 을 **그 반올림된 값들의 뺄셈으로 정의**했기 때문에 양변이 항상 같다. 만약 net 을 따로 `gross × (1 − 세율합)` 식으로 반올림했다면 1원씩 안 맞을 수 있었다. **"net 은 계산이 아니라 차액으로 정의한다"** — 이 한 줄이 복식부기 불변식을 공짜로 보증한다.

이건 코드가 아니라 **설계**가 만든 안전성이다. `Payslip` 생성자가 `netPay = grossPay.subtract(totalDeduction)` 으로 둔 그 선택(3글)이 여기서 보상받는다.

## 🔥 합계는 어디서 오나 — `PayrollRun` 의 집계 메서드

전표가 쓰는 다섯 합계(Σgross / Σ소득세 / Σ직원분 / Σ회사분 / Σnet)는 대장이 payslip 에서 집계해 이벤트에 실어 보낸다. gross/net 은 캐시 컬럼이지만, 소득세·보험 합은 그때그때 계산:

```java
public BigDecimal totalIncomeTax()        { return sum(Payslip::getIncomeTax); }
public BigDecimal totalInsuranceEmployee(){ return sum(Payslip::getInsuranceEmployee); }
public BigDecimal totalInsuranceCompany() { return sum(Payslip::getInsuranceCompany); }
```

> 왜 이 셋은 컬럼이 아닌가: 평소 대장 조회엔 안 쓰이고, **급여 확정 이벤트를 채울 때만** 필요해서다. 컬럼을 늘리는 대신 필요한 순간 집계 — "캐시는 정말 자주 읽는 것만". gross/deduction/net 은 대장 목록에서 자주 보여줘서 캐시했다.
