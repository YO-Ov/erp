# 5/6. 이벤트로 분리, 2단계로 지급 — HR 은 FI 를 모른다

> 앞 글에서 급여 확정이 인건비 전표를 만드는 걸 봤다. 그런데 `PayrollService` 코드 어디에도 `AutoJournalService` 가 없다.
> HR 은 FI 를 **import 하지 않는다**. 둘은 오직 **이벤트**로만 연결된다 — Phase 4(SD↔MM), Phase 5(SD→FI)에서 쌓은 그 원칙의 세 번째 적용.
> 그리고 급여는 **확정(비용 인식)** 과 **지급(현금 유출)** 두 단계로 나뉜다 — 발생주의.

대상 파일:

```
hwlee-erp/src/main/java/com/hwlee/erp/hr/payroll/event/
├─ PayrollConfirmedEvent.java   확정 사건 (HR 소유)
└─ PayrollPaidEvent.java        지급 사건 (HR 소유)

hwlee-erp/src/main/java/com/hwlee/erp/fi/integration/hr/
└─ PayrollAccountingListener.java   수신 → 분개 (FI 소유)
```

---

## 🔥 의존 방향 — HR → 이벤트 ← FI

만약 `PayrollService` 가 `AutoJournalService` 를 직접 호출하면, HR 모듈이 FI 모듈을 알아야 한다(컴파일 의존). 그러면 "급여를 회계 없이 테스트"할 수 없고, HR 이 FI 변경에 끌려다닌다.

대신 **이벤트**로 끊는다:

```
PayrollService.confirm()
      │  publishEvent(PayrollConfirmedEvent)      ← HR 은 "사건이 일어났다"만 알린다
      ▼
 (Spring ApplicationEventPublisher)
      ▼
PayrollAccountingListener.onConfirmed()           ← FI 가 그 사건을 구독해 분개
      │  autoJournal.createPayrollEntry(event)
      ▼
 인건비 전표
```

이벤트는 **발행자(HR)가 소유**한다 — `hr/payroll/event/`. 리스너는 **수신자(FI)가 소유** — `fi/integration/hr/`. 패키지 경로 `fi/integration/hr/` 자체가 "FI 로 들어오는, HR 발 통합"을 표현한다(Phase 5 의 `fi/integration/sd/` 와 같은 규칙).

서비스는 이벤트만 던진다:

```java
@Transactional
public PayrollRunResponse confirm(Long id) {
    PayrollRun run = getOrThrow(id);
    run.confirm(LocalDateTime.now(clock));
    events.publishEvent(new PayrollConfirmedEvent(
            run.getId(), run.getNumber(), run.getRunDate(),
            run.getTotalGross(), run.totalIncomeTax(),
            run.totalInsuranceEmployee(), run.totalInsuranceCompany(), run.getTotalNet()));
    return mapper.toResponse(run);
}
```

리스너가 받아 분개한다:

```java
@TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
void onConfirmed(PayrollConfirmedEvent event) {
    log.info("급여 확정 사건 수신: payrollRunId={}, gross={}", event.payrollRunId(), event.totalGross());
    autoJournal.createPayrollEntry(event);
}
```

---

## 🔥 `BEFORE_COMMIT` — 한 트랜잭션에서 같이 살고 같이 죽는다

`@TransactionalEventListener(phase = BEFORE_COMMIT)` 의 의미: 리스너가 **발행자 트랜잭션 안에서, 커밋 직전에** 실행된다. 결과:

- 분개가 성공해야 급여 확정도 커밋된다.
- 분개가 실패(차/대 불일치 등)하면 **급여 확정 자체가 롤백**된다.

즉 "급여는 확정됐는데 전표는 없는" 어긋난 상태가 **원천적으로 불가능**하다. Phase 5 에서 인보이스 발행과 매출 분개가 한 몸이었던 것과 똑같다.

> 일반 `@EventListener` 와의 차이: 일반 리스너는 발행 즉시(트랜잭션 커밋과 무관하게) 실행돼, 발행자가 나중에 롤백되면 "전표만 남는" 유령이 생긴다. `BEFORE_COMMIT` 은 그걸 막는다 — 정합성이 성능보다 우선인 회계에 맞는 선택.

## 🔥 왜 2단계인가 — 발생주의 (확정 ≠ 지급)

급여는 "확정"과 "지급"이 **다른 시점**이다:

- **5월 31일** — 5월 급여가 확정된다. 회사는 "직원에게 줄 빚"이 생겼다. 아직 안 줬다.
- **6월 10일** — 실제로 통장에 이체한다. 그제서야 현금이 나간다.

회계는 이 둘을 따로 기록한다(**발생주의** — 현금이 아니라 사건이 일어난 시점에 인식):

```
[확정] 5/31   차) 급여비용 등  /  대) 미지급급여(빚 생김)
[지급] 6/10   차) 미지급급여   /  대) 현금(빚 갚고 현금 나감)
```

`createSalaryPaymentEntry` 가 지급 단계를 담당 — 확정 때 인식한 미지급급여(부채)를 현금으로 상계:

```java
@Transactional
public JournalEntry createSalaryPaymentEntry(PayrollPaidEvent event) {
    String number = numberGenerator.nextJournalEntryNumber(event.paymentDate());
    JournalEntry entry = JournalEntry.draft(
            number, event.paymentDate(), "급여지급 " + event.number(),
            JournalSource.PAYROLL, event.payrollRunId());

    entry.addDebit(account(SystemAccounts.SALARY_PAYABLE), event.totalNet());   // 차) 미지급급여 (빚 소멸)
    entry.addCredit(account(SystemAccounts.CASH),          event.totalNet());   // 대) 현금 (유출)

    entry.post(LocalDateTime.now(clock));
    return journalRepository.save(entry);
}
```

상태머신도 이 두 단계를 강제한다:

```java
// PayrollStatus:  DRAFT ──confirm()──▶ CONFIRMED ──markPaid()──▶ PAID

public void markPaid(LocalDateTime now) {
    if (status != PayrollStatus.CONFIRMED)
        throw new IllegalStateException("CONFIRMED 급여대장만 지급 처리 가능합니다. 현재: " + status);
    this.status = PayrollStatus.PAID;
    this.paidAt = now;
}
```

> 확정 안 한 대장(DRAFT)은 지급할 수 없다 — "비용도 인식 안 했는데 현금부터 나가는" 순서 역전을 막는다. Phase 5 의 출금(`Payment.DISBURSEMENT` — 차)매입채무/대)현금)과 **똑같은 모양**의 전표라는 점도 음미할 만하다. 외상 갚기든 급여 지급이든 "부채를 현금으로 상계"는 같은 회계 행위.

지급도 똑같이 이벤트로 — `PayrollPaidEvent` → 같은 리스너의 `onPaid`:

```java
@TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
void onPaid(PayrollPaidEvent event) {
    autoJournal.createSalaryPaymentEntry(event);
}
```

> 두 전표 모두 `sourceType=PAYROLL, sourceId=대장ID` 로 박힌다. 그래서 `findBySourceTypeAndSourceId(PAYROLL, runId)` 하면 **확정 전표 + 지급 전표 2건**이 함께 조회된다 — "이 급여대장이 만든 모든 회계 흔적".

## 정리

| 개념 | 코드 |
| --- | --- |
| HR ↛ FI 직접 의존 | `publishEvent` → `fi/integration/hr/` 리스너 |
| 같이 커밋/롤백 | `@TransactionalEventListener(BEFORE_COMMIT)` |
| 발생주의 2단계 | 확정(대)미지급급여) → 지급(차)미지급급여/대)현금) |
| 순서 강제 | 상태머신 `DRAFT→CONFIRMED→PAID` |
| 흔적 추적 | `sourceType=PAYROLL` 로 확정+지급 전표 묶음 조회 |
