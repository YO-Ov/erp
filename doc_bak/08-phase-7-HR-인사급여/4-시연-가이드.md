# Phase 7 시연 가이드 — HR 급여와 자동 인건비 회계

> 7단계 사이클의 마지막 — **단계 7: 시연 + 회고**.
> 도메인 브리핑 → 모델링 → 코드 워크스루 6편으로 HR 의 설계와 구현을 머리로 이해했다.
> 이번에는 cURL / Swagger UI / MySQL CLI 로 **"근태를 넣고 급여를 돌리면, 인건비 전표가 복식부기로 자동으로 박히는 흐름"** 을 손으로 확인한다.

---

## 시연의 목표 (Phase 7 이 무엇을 보장하는가)

이번에 확인할 **6대 사실**:

1. **근태가 급여에 반영된다** — 연장근로(소정 8h 초과)를 넣으면 그만큼 연장수당(시급×1.5)이 붙는다.
2. **급여 = 자동 인건비 전표** — 급여대장을 확정하면 5개 계정에 걸친 전표가 한 트랜잭션에 자동으로 박힌다.
3. **차변 합 = 대변 합** — 인건비 전표도 복식부기 불변식을 지킨다(`net` 정의가 보증, 워크스루 4글).
4. **회사부담 보험 = 법정복리비** — 직원 공제가 아니라 회사 추가비용(차변 비용)으로 잡힌다.
5. **확정 ≠ 지급 (발생주의 2단계)** — 확정은 미지급급여(부채), 지급은 미지급급여→현금.
6. **급여는 아무나 못 본다** — HR/ADMIN 만 200, 그 외 403.

> 💡 한 문장: **"근태 한 줄이 들어가면, 그 끝에 복식부기 인건비 전표가 자동으로 닫힌다 — 인사 담당자는 회계를 의식하지 않는다."**

## STEP 2. HR 계정으로 로그인 → JWT

급여는 HR/ADMIN 전용이라, 먼저 인사팀 `jung` 으로 로그인한다(V31 시드).

```bash
curl -i -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"jung@hwlee-erp.example","password":"pass1234"}'
```

→ **200** + 토큰:

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJqdW5n...",
  "tokenType": "Bearer",
  "expiresIn": 3600,
  "roles": ["HR"]
}
```

`accessToken` 을 환경변수에 담아 둔다:

```bash
TOKEN="여기에-accessToken-붙여넣기"
```

> 직원 id 를 미리 확보해 두면 편하다. V31 이 4명에게 급여계약을 시드했다. 시연에선 **김영업(kim)** 의 급여를 돌려보자. kim 의 employee id 가 필요하다 — DB 로 한 번 확인:
> ```bash
> docker compose exec mysql mysql -uroot -proot erp \
>   -e "SELECT id, code, name, email FROM employee WHERE email LIKE 'kim@%' OR email LIKE 'jung@%';"
> ```
> 여기선 kim 의 id 를 **1** 이라고 가정한다(환경에 맞게 바꾼다).

## STEP 3. 근태 등록 — 연장근로를 넣는다 ⭐

kim 의 5월 근태 중 하루를 **08:00~19:00 (11시간)** 으로 등록한다 → 연장 180분(3시간).

```bash
curl -s -X POST http://localhost:8080/api/attendances \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"employeeId":1,"workDate":"2026-05-04","clockIn":"08:00","clockOut":"19:00"}' | jq
```

→ 응답에서 **파생 계산** 확인:

```json
{
  "id": 1, "employeeId": 1, "workDate": "2026-05-04",
  "clockIn": "08:00:00", "clockOut": "19:00:00",
  "workedMinutes": 660,        // 11시간
  "overtimeMinutes": 180       // 660 - 480 = 3시간 ⭐ 연장
}
```

> 💡 **같은 직원·같은 날짜로 한 번 더 POST 하면 400** — "이미 등록된 근태입니다"(워크스루 2글, 하루 한 건). 확인해 보라.

원하면 다른 날짜도 몇 건 더 넣어 연장을 누적할 수 있다(예: 2026-05-11 도 연장 2시간). 월 연장은 `SUM` 으로 집계된다.

## STEP 4. 급여대장 생성 — DRAFT (자동 계산)

5월 급여대장을 만든다. **대상 월만** 주면, 그 달 유효계약 전원의 명세서가 자동 계산된다.

```bash
curl -s -X POST http://localhost:8080/api/payroll-runs \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"period":"2026-05"}' | jq
```

→ **201** + 대장(DRAFT). V31 이 4명에게 계약을 시드했으니 payslip 4건이 나온다. kim 의 명세를 보면(기본급 3,500,000 / 소정 209h / 연장 180분):

```json
{
  "id": 1, "number": "PR-202605-001", "period": "2026-05", "status": "DRAFT",
  "totalGross": ...,
  "payslips": [
    {
      "employeeName": "김영업",
      "basePay": 3500000,
      "overtimePay": 75359,      // 시급(3.5M/209≈16746.41) × 1.5 × 3h
      "grossPay": 3575359,
      "incomeTax": 214522,       // 6%
      "insuranceEmployee": 160891,  // 4.5%
      "insuranceCompany": 160891,   // 4.5% (공제 아님)
      "totalDeduction": 375413,  // 소득세 + 직원분
      "netPay": 3199946          // gross - 공제
    },
    ... (이재무/박생산/정인사)
  ]
}
```

> 숫자는 시급 반올림에 따라 1원 단위가 달라질 수 있다 — 중요한 건 **관계**다: `grossPay = basePay + overtimePay`, `netPay = grossPay − incomeTax − insuranceEmployee`, 그리고 `insuranceCompany` 는 net 에 안 들어간다.
>
> 같은 달로 다시 POST 하면 **400** — "이미 해당 월의 급여대장이 있습니다"(period UNIQUE, 월 1건).

`id` 를 기억해 둔다(여기선 `RUN=1`).

## STEP 5. 급여 확정 → 인건비 전표 자동 생성 ⭐

```bash
curl -s -X POST http://localhost:8080/api/payroll-runs/1/confirm \
  -H "Authorization: Bearer $TOKEN" | jq '.status'
```

→ `"CONFIRMED"`. 이 한 번의 호출이 **같은 트랜잭션에서 인건비 전표를 박았다**. 확인:

```bash
curl -s "http://localhost:8080/api/journal-entries?sourceType=PAYROLL" \
  -H "Authorization: Bearer $TOKEN" | jq
```

> `journal-entries` 는 FI 조회 — HR 역할에 `MASTER_READ` 만 있어 막힐 수 있다. 그땐 **admin 으로 로그인**해 조회하거나, DB 로 직접 본다(아래). admin 토큰: `{"username":"admin@hwlee-erp.example","password":"pass1234"}`.

DB 로 전표 라인을 직접 펼쳐 보는 게 가장 명확하다:

```bash
docker compose exec mysql mysql -uroot -proot erp -e "
SELECT je.number, a.code, a.name, jl.debit, jl.credit
FROM journal_entry je
JOIN journal_line jl ON jl.journal_entry_id = je.id
JOIN account a ON a.id = jl.account_id
WHERE je.source_type='PAYROLL' AND je.description LIKE '급여 %'
ORDER BY je.id, jl.line_no;"
```

→ 대장 전체 합산 분개(차변 2 / 대변 3):

```
차) 5200 급여비용        Σgross
차) 5300 법정복리비      Σ회사분 보험
       대) 2400 예수금-소득세      Σ소득세
       대) 2500 예수금-사회보험    Σ(직원분+회사분)
       대) 2600 미지급급여         Σnet
```

**차변 합 = 대변 합** 인지 확인:

```bash
docker compose exec mysql mysql -uroot -proot erp -e "
SELECT SUM(jl.debit) AS 차변합, SUM(jl.credit) AS 대변합
FROM journal_entry je JOIN journal_line jl ON jl.journal_entry_id = je.id
WHERE je.source_type='PAYROLL' AND je.description LIKE '급여 %';"
```

→ 두 값이 **정확히 같다**(워크스루 4글의 수학적 보증). ⭐

## STEP 6. 급여 지급 → 미지급급여 → 현금 (발생주의 2단계)

확정만으론 아직 현금이 안 나갔다(미지급급여 = 빚). 실제 이체:

```bash
curl -s -X POST http://localhost:8080/api/payroll-runs/1/pay \
  -H "Authorization: Bearer $TOKEN" | jq '.status'
```

→ `"PAID"`. 지급 전표 확인:

```bash
docker compose exec mysql mysql -uroot -proot erp -e "
SELECT je.number, a.code, a.name, jl.debit, jl.credit
FROM journal_entry je
JOIN journal_line jl ON jl.journal_entry_id = je.id
JOIN account a ON a.id = jl.account_id
WHERE je.source_type='PAYROLL' AND je.description LIKE '급여지급 %'
ORDER BY jl.line_no;"
```

→

```
차) 2600 미지급급여   Σnet     (빚 소멸)
       대) 1100 현금       Σnet     (현금 유출)
```

> 이제 `source_type=PAYROLL` 전표가 **2건**(확정 + 지급)이 됐다. "이 대장이 만든 모든 회계 흔적". 확정 안 한 대장을 `/pay` 하면 **400**(상태머신 — CONFIRMED 만 지급 가능).

미지급급여(2600) 잔액이 **0** 으로 닫혔는지 — 확정에서 +Σnet, 지급에서 −Σnet:

```bash
docker compose exec mysql mysql -uroot -proot erp -e "
SELECT SUM(jl.debit - jl.credit) AS 미지급급여_잔액
FROM journal_line jl JOIN account a ON a.id = jl.account_id
JOIN journal_entry je ON je.id = jl.journal_entry_id
WHERE a.code='2600' AND je.status='POSTED';"
```

→ **0**. 인식한 빚을 모두 갚았다. ⭐

## STEP 7. 인가 — 급여는 아무나 못 본다 ⭐

영업(kim)으로 로그인해 급여 API 를 때려본다:

```bash
# kim 로그인
curl -s -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"kim@hwlee-erp.example","password":"pass1234"}' | jq -r '.accessToken' > /tmp/kim.txt
KIM=$(cat /tmp/kim.txt)

# 급여대장 조회 시도
curl -i http://localhost:8080/api/payroll-runs/1 \
  -H "Authorization: Bearer $KIM"
```

→ **403 Forbidden**. 급여는 `hasAnyRole('HR','ADMIN')` 전용 — 영업도, 심지어 재무(lee)도 막힌다.

| 계정 | 역할 | `/api/payroll-runs` |
| --- | --- | --- |
| jung | HR | ✅ 200 |
| admin | ADMIN | ✅ 200 |
| kim | SALES | ⛔ 403 |
| lee | FINANCE | ⛔ 403 |
| park | (없음) | ⛔ 403 |

> "전표의 총액은 회계가 보지만, 개인별 급여 명세는 인사만" — 실무의 정보 경계를 인가로 그었다(워크스루 6글).

## 회고 — Phase 7 이 모은 것

Phase 7 은 새 개념을 거의 안 만들었다. **Phase 1~6 의 패턴을 한데 모았다**:

| Phase | 패턴 | Phase 7 에서 |
| --- | --- | --- |
| 1 | 마스터 + 이력 | `employment_contract`(발효일 이력) |
| 2 | 헤더-라인 + 상태머신 | `payroll_run` ↔ `payslip`, DRAFT→CONFIRMED→PAID |
| 3 | 누적 이력 + 파생 | `attendance`(StockMovement 형) |
| 4 | 이벤트로 모듈 분리 | `PayrollConfirmed/PaidEvent`, HR↛FI |
| 5 | 자동 분개 + 복식부기 | `createPayrollEntry`, 차/대 균형 |
| 6 | RBAC 인가 | 급여 HR/ADMIN 전용 |

**핵심 한 줄**: 근태(이력) → 급여 계산(헤더-라인) → 확정 이벤트 → 인건비 전표(복식부기) → 2단계 지급. 그리고 `net ≡ gross − 소득세 − 직원분` 이라는 **정의 하나**가 라운딩과 무관하게 복식부기를 닫는다.
