# Phase 8 설계 제안 — PP(생산): BOM · 생산지시 · 재고 변신

> 7단계 사이클의 **단계 3: 모델링 제안**. 브리핑(§단계 1)의 갈림길들을 **코드 현실에 맞춰** 확정하고, ERD·계산 규칙·연계 흐름·마이그레이션·인가·테스트까지 설계한다.
> 이 문서를 검토하고 **"승인"** 하면 단계 5(코드 구현)로 들어간다. 바꾸고 싶은 결정이 있으면 알려 달라.

## 0. 한 줄 설계 요약

> **생산은 재고 메커니즘(Stock·StockMovement·락·이동평균)을 재사용하되, GoodsReceipt/GoodsIssue 헤더는 재사용하지 않는다.** 대신 `ProductionService` 가 완료 시점에 **부품 출고(stock.issue) + 완제품 입고(stock.receive)** 를 한 트랜잭션에서 직접 오케스트레이션한다. 완제품 단가 = 투입 부품 실제원가 합 ÷ 수량(원가 보존).

## 1. ⭐ 설계 결정 요약 (브리핑 갈림길 확정)

| # | 갈림길 | **결정** | 근거 |
| --- | --- | --- | --- |
| 1 | BOM 단일 vs 다단계 | **단일 레벨** | 완제품=부품 1단계. 재귀 전개 복잡성 회피. 반제품(다단계)은 범위 밖. |
| 2 | Item 완제품/부품 구분 | **`itemType` 축 추가**(FINISHED/COMPONENT) + `ItemCategory.PART` 추가 | category(종류)와 직교하는 '역할' 축. 기존 노트북/모니터=FINISHED 백필. |

| 6 | 완제품 입고 단가 | **투입 부품 실제(이동평균) 원가 합 ÷ 생산수량** | "투입=산출 원가 보존"이 이 Phase 회계 핵심. Phase 3·5와 일관. |
| 7 | 생산 자동분개(FI) | **현실형 간이 — 원재료/제품 계정 분리, 포함** (단계 4 확정) | 실제 ERP 제조회계(원재료→재공품→제품)를 따르되 **재공품(WIP)·노무비·간접비는 생략**(착수=완성 1단계, 직접재료비만). 상세 §7. |
| 8 | 생산 화면(UI) | **이번에 화면까지 포함** (단계 4 확정) | BOM·생산지시 목록/상세/생성 + 사이드바 PP 그룹. SD/MM/FI/HR 패턴 재사용. |

## 2. ERD — 새 테이블 3개 + Item 보강 1개

```
Item (마스터, Phase 1 — 보강)
├─ ... 기존 (code/name/category/unit/standard_cost/standard_price)
└─ item_type   FINISHED | COMPONENT   ← 신규 컬럼 (기존 행은 FINISHED 백필)

bom (마스터 — 완제품의 레시피, 자기참조 Item↔Item)
├─ id
├─ product_item_id    FK → item   (완제품, itemType=FINISHED)
├─ component_item_id  FK → item   (부품,  itemType=COMPONENT)
├─ quantity           완제품 1개당 소요량 (DECIMAL)
└─ (UNIQUE: product_item_id + component_item_id — 같은 부품 중복 라인 방지)

production_order (트랜잭션 — 헤더)
├─ id
├─ number              PO-YYYYMMDD-NNN (채번, 일자 리셋 — Phase 2 패턴)
├─ product_item_id     FK → item (만들 완제품)
├─ warehouse_id        FK → warehouse (부품 출고·완제품 입고 창고)
├─ quantity            생산 수량
├─ status              PLANNED | RELEASED | COMPLETED | CANCELLED
├─ order_date          지시일
├─ due_date            완료 예정일 (nullable)
├─ completed_at        실제 완료 일시 (nullable)
└─ lines               소요 자재 라인 (1:N)

production_order_line (트랜잭션 — 소요 자재, BOM×수량 스냅샷)
├─ id
├─ production_order_id FK
├─ component_item_id   FK → item (부품)
├─ required_qty        소요량 = BOM.quantity × order.quantity (생성 시 고정)
└─ issued_unit_cost    완료 시 실제 출고된 이동평균 단가 (nullable, 완료 전 NULL)
```

> 💡 **스냅샷 원칙**(Phase 2 재현): 생산지시 라인은 생성 시점의 BOM을 **복제**해 굳힌다. 나중에 BOM이 바뀌어도 이미 낸 지시는 안 흔들린다(SalesOrderLine이 단가를 굳히는 것과 동일).

## 3. 자재 소요량 계산 (MRP 미니버전)

생산지시 **생성 시** BOM을 전개해 소요 자재 라인을 만든다:

```
1) product_item_id 의 BOM 조회 → [(component, qty_per), ...]
2) 각 부품마다: required_qty = qty_per × order.quantity
3) production_order_line 으로 저장 (스냅샷)
```

BOM이 없는 완제품(=레시피 미등록)으로 생산지시를 내면 **거부**(IllegalState). 가용재고 검증은 완료 시점에 자동(아래 §4).

추가로 **가용성 조회**(읽기 전용, 막지 않음): 각 부품의 `required_qty` vs 현재 가용재고를 비교해 "부족 여부"를 보여주는 API 1개(`GET /api/production-orders/{id}/material-availability`). RELEASED 전에 "이거 지금 만들 수 있나" 확인용.

## 4. ⭐ 생산 완료 처리 — 직접 오케스트레이션 + 원가 보존

`ProductionService.complete(id)` 한 트랜잭션:

```
complete(orderId):
  order = load(orderId); requireStatus(RELEASED)

  totalCost = 0
  for line in order.lines:                          # ① 부품 출고
      stock = stockRepo.findForUpdate(component, warehouse)   # 비관 락 (Phase 3)
      appliedCost = stock.issue(line.requiredQty)            # 재고 차감 + 이동평균 단가 반환
                                                             #   ↳ 재고 부족이면 InsufficientStockException → 전체 롤백
      line.issuedUnitCost = appliedCost
      totalCost += appliedCost × line.requiredQty
      save StockMovement.of(component, wh, -requiredQty, "PRODUCTION_ORDER", orderId, now)  # reason=PRODUCTION_OUT

  unitCost = totalCost / order.quantity              # ② 완제품 단위원가 (원가 보존)
  stock = stockRepo.findForUpdate(product, warehouse)
  stock.receive(order.quantity, unitCost)            # 완제품 재고 증가 + 이동평균 갱신
  save StockMovement.of(product, wh, +order.quantity, "PRODUCTION_ORDER", orderId, now)     # reason=PRODUCTION_IN

  order.complete(now)  # 상태 COMPLETED
```

핵심:
- **재고 메커니즘 재사용**: `findForUpdate`(비관 락)·`stock.issue`/`receive`(이동평균)·`StockMovement` 원장 — 전부 Phase 3 자산 그대로.
- **원가 보존**: 빠져나간 부품 원가 합(totalCost) = 완제품으로 들어간 원가(unitCost × qty). 재고자산 총액 불변 → §7에서 "회계 전표 불필요"의 근거.
- **원자성**: 한 부품이라도 재고 부족이면 `InsufficientStockException` 으로 **전체 롤백** — 부분 출고 없음.

새 무브먼트 사유 2개 추가: `MovementReason.PRODUCTION_IN`(+), `PRODUCTION_OUT`(−). `refType="PRODUCTION_ORDER"`, `refId=생산지시 id` 로 추적.

## 5. 왜 GoodsReceipt/GoodsIssue 헤더를 재사용하지 않는가 (핵심 근거)

브리핑은 "MM 출고/입고를 호출"이라 했지만, **코드를 보니 헤더 재사용은 부적합**하다:

| 재사용 시도 | 문제 |
| --- | --- |
| **GoodsReceipt**(완제품 입고) | ① `vendor` **필수**(NOT NULL) — 자가생산 완제품은 거래처가 없다. ② 전기 시 `GoodsReceiptPostedEvent` 발행 → `fi/integration/mm/PurchaseAccountingListener` 가 **매입 전표**(차)재고/대)매입채무)를 잘못 생성. |
| **GoodsIssue**(부품 출고) | 회계는 안전(매출원가는 `DeliveryShippedEvent`로만 트리거 → 생산 출고는 COGS 안 만듦). 단 `reason` enum 확장 필요하고, 헤더를 굳이 만들 이유가 약함. |

> ✅ 결론: **저수준 재고 메커니즘(Stock + StockMovement + 락)만 재사용**하고, GoodsReceipt/GoodsIssue **헤더(트랜잭션 문서)는 PP가 만들지 않는다.** 생산의 재고 이동은 `production_order` 자체가 원천 문서이고, StockMovement에 `refType=PRODUCTION_ORDER`로 직접 적재한다. → vendor 강제·매입/COGS 오트리거를 깔끔히 회피.

이것이 결정 5(직접 호출)의 진짜 이유다: 이벤트로 GoodsReceipt를 만들게 하면 위 문제를 그대로 떠안는다.

## 6. 상태머신 — ProductionOrder

```
PLANNED ──release()──▶ RELEASED ──complete()──▶ COMPLETED
   │  (생성: BOM 전개,    (착수 확정)    (부품 출고 + 완제품 입고,
   │   소요자재 산출)                    재고 실제 이동)
   └──cancel()──┐    └──cancel()──┐
                ▼                 ▼
            CANCELLED         CANCELLED
(COMPLETED·CANCELLED 는 종착 — 재고 이동은 COMPLETED 전이에서만)
```

- `PLANNED`: 생성 직후. 소요 자재만 산출됨(재고 미반영). 수정/취소 가능.
- `RELEASED`: 생산 착수 확정. (재고는 아직 안 움직임 — 완료 시 일괄.)
- `COMPLETED`: §4 실행. 종착.
- `CANCELLED`: PLANNED/RELEASED 에서만. 재고 미반영 상태라 원복 불필요.

> 💡 단순화: 완료를 한 번에(부품 소모+완제품 입고 동시). 현실의 "착수 시 자재 투입 → 완료 시 완제품 산출" 2단계 분리는 범위 밖(재공품 WIP 개념과 함께).

## 7. ⭐ 회계(FI) — 현실형 간이: 원재료/제품 계정 분리

### 실제 ERP 제조회계 (참고)
제조기업은 재고를 3단계로 분리한다: **원재료 → 재공품(WIP) → 제품**. 분개 흐름:
```
① 원재료 구매   차)원재료   / 대)매입채무
② 자재 투입     차)재공품   / 대)원재료
③ 노무비 투입   차)재공품   / 대)미지급노무비
④ 간접비 배부   차)재공품   / 대)제조간접비
⑤ 생산 완료     차)제품     / 대)재공품
⑥ 제품 판매     차)매출원가 / 대)제품
```
제조원가 = 직접재료비 + 직접노무비 + 제조간접비.

### 우리 구현 (간이 — 재공품·노무비·간접비 생략)
현재 CoA는 전 재고가 단일 `재고자산(1400)`. 기존 1400 잔액은 **전부 완제품(노트북/모니터)** 이므로, **1400을 "제품"으로 의미 부여**하고 **원재료(1410)만 신설**하면 재분류 없이 코herent하다.

| 사건 | 분개 | 변경점 |
| --- | --- | --- |
| 부품(COMPONENT) 매입 | 차)**원재료(1410)** / 대)매입채무 | MM 매입 리스너가 itemType 보고 계정 분기 |
| 완제품(FINISHED) 매입 | 차)제품(1400) / 대)매입채무 | 기존과 동일(1400) |
| **생산 완료** ⭐ | 차)**제품(1400)** / 대)**원재료(1410)** | 신규 — 직접재료비만(투입 부품 원가 합) |
| 제품 판매(출하) | 차)매출원가 / 대)제품(1400) | 기존과 동일(변경 없음) |

> ⚠️ 재공품(WIP)을 생략하므로 "착수→완성"을 한 전표로 본다(차)제품/대)원재료). 노무비·제조간접비 배부는 범위 밖 → 완제품 원가 = **직접재료비뿐**(가공원가 0). 이건 명시적 단순화.

### 계정·기초재고 처리
- **신설 계정**: `원재료(1410, ASSET)`, 그리고 부품 **기초재고의 GL 상계**를 위해 `자본(3000 header)` + `이익잉여금(3100, EQUITY)`. (1400은 이름만 "제품(재고자산)"으로 갱신, 코드·용도 유지 → `SystemAccounts.INVENTORY="1400"` 불변.)
- **부품 기초재고 시드**(V34): 재고 원장(Stock+StockMovement)과 함께 **개시 분개** `차)원재료(1410) / 대)이익잉여금(3100)` 1건으로 GL=재고원장 일치 보장. (실무의 "기초재고 이월" 분개.)
- **변경 코드**: `SystemAccounts`(+RAW_MATERIAL, +RETAINED_EARNINGS), `AutoJournalService.createPurchaseEntry`(라인 itemType별 계정 분기) + 생산완료 분개 메서드 신규. COGS·SALES 리스너는 **무변경**(완제품만 팔리므로 1400 그대로).
- ⚠️ Phase 5 매입 통합테스트가 단일 1400을 가정하면 itemType 분기 후에도 통과하는지 **재검증 필수**(기존 테스트는 완제품 매입이라 1400 유지 → 영향 없을 것으로 예상하나 확인).

## 8. 인가 — PRODUCTION 역할 (Phase 6 복선 회수)

- 새 역할 **`PRODUCTION`** 추가(role 테이블).
- 부서 매핑 시드: 생산팀(`DEPT-PRODUCTION`) → `PRODUCTION` 역할 → 직원 **박생산(park\@hyunwoo.com)** 이 드디어 역할을 갖는다(Phase 6 "역할 없음 403 시연" 복선 회수).
- 컨트롤러: `@PreAuthorize("hasAnyRole('PRODUCTION','ADMIN')")` (BOM 조회는 마스터라 좀 더 열 수도 있으나, 기본은 PRODUCTION/ADMIN).
- (선택) permission 시드 `PP_READ`/`PP_WRITE` — 기존 권한 시드 패턴과 일관되게 구조만.
