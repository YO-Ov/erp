# Phase 11 — MES + ERP-MES 연계: 시스템 구성도와 기술 개요

> **이 문서의 목적**: Part 2(MES)에 들어가기 전에 **"무엇을, 왜 만드는가"**를 먼저 잡는다.
> 인프라 셋업을 시작하면 새로운 기술(Kafka·Outbox·Circuit Breaker·분산추적)이 한꺼번에 등장한다.
> 코드를 보기 전에 각 기술이 **현실에서 어떤 문제를 풀려고 존재하는지**를 이해해 두면, 이후 구현이 "왜 이렇게 짜는지"로 읽힌다.

## 1. 한눈에 보기 — 두 시스템과 그 사이

지금까지 만든 것은 **하나의 시스템(ERP)** 이었다. Part 2부터는 **두 번째 시스템(MES)** 을 따로 만들고, 둘을 **네트워크 너머로 연결**한다.

```
        ┌──────────────────────────────┐         ┌──────────────────────────────┐
        │        hwlee-erp (8080)       │         │        hwlee-mes (8082)       │
        │     본사 사무직 · 기간계      │         │   공장 현장 · 현장 실행계     │
        │                              │         │                              │
        │  SD 영업   MM 재고   FI 회계  │         │  설비   작업자   작업지시     │
        │  HR 인사   PP 생산   배치/리포트│        │  생산실적  자재투입  품질     │
        │                              │         │                              │
        │      ┌──────────────┐        │         │      ┌──────────────┐        │
        │      │   erp_db     │        │         │      │   mes_db     │        │
        │      │  (MySQL)     │        │         │      │  (MySQL)     │        │
        │      └──────────────┘        │         │      └──────────────┘        │
        └───────────┬──────────────────┘         └───────────┬──────────────────┘
                    │                                         │
                    │   ① 작업지시 발행 (동기 REST 호출)       │
                    │   ───────────────────────────────────▶  │
                    │   "이 제품 100개 만들어라"                │
                    │                                         │
                    │   ② 생산 실적 전송 (비동기 Kafka 메시지)  │
                    │   ◀───────────────────────────────────  │
                    │   "80개 완성, 자재 이만큼 썼다"           │
                    │                                         │
              ┌─────┴─────────────────────────────────────────┴─────┐
              │                  공유 인프라                          │
              │   Kafka + Zookeeper   (비동기 메시지 버스)            │
              │   Zipkin              (분산 추적 — 두 시스템 걸친 흐름) │
              └──────────────────────────────────────────────────────┘
```

핵심 한 줄: **ERP는 "계획·재무"를, MES는 "현장 실행"을 담당하고, 둘은 두 가지 방식(동기 REST + 비동기 Kafka)으로 데이터를 주고받는다.**

## 2. 왜 ERP와 MES를 굳이 나누는가?

같은 DB에 테이블 몇 개 더 만들면 되는데 왜 시스템을 쪼갤까? — 이게 Part 2의 가장 중요한 학습 포인트다.

| 이유 | 설명 |
| --- | --- |
| **책임이 다르다** | ERP = "무엇을 얼마나 만들/팔지"(계획·돈). MES = "지금 라인에서 실제로 어떻게 돌아가는지"(실행·현장). 관심사가 다르면 시스템도 나뉜다. |
| **실무가 그렇다** | 현장에서 ERP와 MES는 **거의 항상 별도 제품**(벤더도, 운영팀도 다름). ERP는 SAP, MES는 다른 전문 벤더인 식. |
| **I/F 개발이 기간계 실무의 절반** | "다른 시스템과 데이터를 어떻게 안전하게 주고받는가"가 SI/기간계 개발의 핵심 역량. 이걸 직접 만들어 보는 게 Part 2의 목적. |
| **장애 격리** | 공장 MES가 멈춰도 본사 ERP의 영업·회계는 돌아가야 한다. 시스템이 분리돼 있어야 한쪽 장애가 다른 쪽을 죽이지 않는다. |

> DB를 일부러 `erp_db` / `mes_db`로 **분리**하는 이유도 같다. "두 시스템은 서로의 테이블을 직접 못 본다 → 반드시 정해진 인터페이스(REST/메시지)로만 대화한다"를 코드로 강제하기 위해서다.

## 3. 구성 요소별 역할 — "왜 필요한가"

이번 Part 2에서 새로 등장하는 기술들을, **각자 무슨 문제를 푸는지** 중심으로 정리한다.

### 3-1. hwlee-mes (두 번째 애플리케이션)
- **무엇**: 공장 현장을 다루는 별도 Spring Boot 앱(포트 8082, 전용 DB `mes_db`).
- **왜**: 위 2절의 이유. 현장 도메인(설비·작업자·작업지시·실적·품질)은 ERP와 관심사가 달라 따로 둔다.

### 3-2. REST 동기 호출 (Spring `RestClient`)
- **무엇**: ERP가 MES의 API를 **직접 호출**하고 **응답을 기다리는** 방식. HTTP 요청/응답.
- **언제 쓰나**: ERP가 생산지시를 발행할 때 → MES에 작업지시를 **즉시 등록**하고 "등록됐다"는 확인을 바로 받아야 하는 경우.
- **왜 동기인가**: "지시를 내렸는데 들어갔는지 모름"은 곤란하다. 결과를 즉시 알아야 하는 명령형 상호작용이라 동기가 맞다.

### 3-3. Circuit Breaker (`Resilience4j`)
- **무엇**: 외부 시스템(MES) 호출이 계속 실패하면, 한동안 **호출 자체를 차단**하고 빠르게 실패시키는 안전장치. "두꺼비집(차단기)"과 같다.
- **왜 필요한가**: MES가 죽었는데 ERP가 계속 호출하면 → 매 요청이 타임아웃까지 매달려 **ERP의 스레드가 고갈**되고 ERP까지 같이 죽는다(장애 전파). 차단기가 "MES 지금 맛이 갔으니 잠깐 호출 끊자"고 판단해 ERP를 보호한다.
- **곁들이는 것**: Retry(일시 오류 재시도), Timeout(무한 대기 방지).

### 3-4. Kafka (비동기 메시지 버스)
- **무엇**: 시스템 간 **메시지를 큐처럼 쌓아 두고** 받는 쪽이 자기 속도로 가져가는 구조. 보낸 쪽은 "메시지를 토픽에 넣었다"까지만 책임지고 응답을 기다리지 않는다.
- **언제 쓰나**: MES → ERP 생산 실적 전송. 현장에서 실적이 **수시로 대량 발생**하고, ERP가 잠깐 느리거나 재기동 중이어도 **유실 없이** 나중에 처리돼야 하는 경우.
- **왜 비동기인가**: 실적은 "통보"다. MES가 ERP의 응답을 기다릴 이유가 없고, ERP가 잠깐 안 떠 있어도 메시지는 Kafka에 남아 있다가 ERP가 살아나면 이어서 처리된다(**내구성·완충**).

### 3-5. Zookeeper
- **무엇**: Kafka 클러스터의 메타데이터·브로커 조정을 담당하는 보조 서비스.
- **왜**: (학습 단계에선) Kafka를 띄우기 위한 **동반 인프라**로 이해하면 충분. docker-compose에 Kafka와 함께 띄운다.

### 3-6. Transactional Outbox 패턴 ⭐ (Part 2의 백미)
- **무엇이 문제냐**: MES가 "① 실적을 DB에 저장"하고 "② Kafka에 메시지 발행" 두 가지를 해야 하는데, **이 둘은 하나의 트랜잭션으로 묶이지 않는다**(DB와 Kafka는 별개 시스템). ①은 됐는데 ② 직전에 죽으면 → 실적은 저장됐는데 ERP는 영영 모른다. 반대면 → 있지도 않은 실적이 ERP로 간다. **데이터 불일치**.
- **해법**: ② 대신, 같은 DB 트랜잭션 안에서 **`outbox` 테이블에 "보낼 메시지"를 한 줄 INSERT**한다. ①과 outbox INSERT는 한 트랜잭션이라 **둘 다 되거나 둘 다 안 된다**. 그 후 별도 발행기(Polling Publisher)가 outbox를 읽어 Kafka로 보내고 발송 표시를 한다.
- **왜 중요한가**: "DB 커밋과 메시지 발행의 원자성" — 분산 시스템에서 데이터 정합성을 지키는 **가장 대표적인 실무 패턴**이다.

### 3-7. 멱등성 (Idempotency)
- **무엇**: 같은 메시지가 **두 번 들어와도 결과가 한 번 처리한 것과 같게** 만드는 성질.
- **왜 필요한가**: 네트워크/재시도 특성상 메시지는 **중복 도착할 수 있다**(Kafka는 "최소 1회" 전달). 멱등 처리가 없으면 실적이 두 번 반영돼 재고가 이중 차감된다. 메시지마다 고유 키를 두고 "이미 처리한 키면 무시"로 막는다.

### 3-8. 분산 추적 (Micrometer Tracing + Zipkin)
- **무엇**: 하나의 비즈니스 요청이 **여러 시스템을 거치는 경로 전체**를 하나의 흐름으로 추적·시각화하는 도구.
- **왜 필요한가**: 단일 시스템에선 로그만 봐도 됐지만, "수주 → 작업지시 → 실적 → 재고반영"이 ERP·MES·Kafka를 넘나들면 **어디서 끊겼는지 추적이 어렵다**. 추적 ID를 요청에 실어 보내면 두 시스템 로그를 하나의 타임라인으로 묶어 볼 수 있다.

### 3-9. OpenAPI 계약 (`contracts/`)
- **무엇**: ERP와 MES가 주고받을 API/이벤트의 **명세를 한 곳에 미리 정의**해 둔 파일(`/contracts`).
- **왜 필요한가**: 두 시스템이 따로 개발되므로, **"무엇을 어떤 모양으로 주고받을지"를 먼저 합의**해야 양쪽이 어긋나지 않는다. 계약이 곧 두 팀의 약속.

## 4. 두 가지 연계 방식 — 언제 REST, 언제 Kafka?

Part 2에서 가장 자주 마주칠 판단이다.

| 구분 | 동기 REST | 비동기 Kafka |
| --- | --- | --- |
| 방향(이 프로젝트) | ERP → MES (작업지시 발행) | MES → ERP (실적 전송) |
| 응답 | 즉시 기다림 | 기다리지 않음 |
| 성격 | **명령** ("이거 해라") | **통보** ("이렇게 됐다") |
| 상대가 죽으면 | 즉시 실패 (Circuit Breaker로 보호) | 메시지가 쌓여 있다가 나중에 처리 |
| 적합한 상황 | 결과를 바로 알아야 함, 건당 처리 | 대량·수시 발생, 유실 불가, 완충 필요 |

> 한 줄 기준: **"결과를 지금 당장 알아야 하면 REST, 안전하게 전달만 되면 되면 Kafka."**

## 5. 전체 데이터 흐름 (목표 E2E 시나리오)

Part 2를 다 만들면 아래 한 줄기가 **두 시스템을 가로질러** 흐른다. (Phase 16에서 완성)

```
[ERP] 영업 수주(SD)
   └─▶ [ERP] 생산 계획/지시(PP)
          └─▶ ① REST 동기 ──▶ [MES] 작업지시 등록
                                   └─▶ [MES] 현장 작업 → 생산 실적 + 자재 투입 기록
                                          └─▶ ② Kafka 비동기(Outbox) ──▶ [ERP] 실적 수신
                                                 ├─▶ [ERP-MM] 자재 차감 / 완제품 입고
                                                 └─▶ [ERP-FI] 생산 원가 계산
   ◀────────────────────────────────────────────────────────────────────────────
   └─▶ [ERP] 출하(SD) → 인보이스 → 수금
```

즉, Part 1에서 만든 ERP의 PP·MM·FI가 **MES의 실제 실적과 연결되어** 비로소 "계획이 현장을 거쳐 다시 재무로 돌아오는" 제조 ERP의 완전한 고리가 닫힌다.

## 6. Phase 11에서 실제로 만들 것 (이번 단계의 범위)

이번 Phase 11은 **기능이 아니라 "연결 토대"**를 까는 단계다. 구체 산출물:

1. **새 프로젝트 `hwlee-mes`** 생성 (Spring Boot 3, Java 21, 포트 8082, 전용 `mes_db`).
2. **공유 인프라 docker-compose 확장**: 기존 MySQL에 더해 **Kafka + Zookeeper + Zipkin**(+ MES용 DB) 추가.
3. **양쪽 헬스체크 API** — 두 앱이 각자 잘 뜨는지 확인.
4. **ERP→MES RestClient + Circuit Breaker 기본 설정** (실제 작업지시 전송은 Phase 12).
5. **양쪽 Kafka Producer/Consumer 기본 설정** (실제 실적 메시지는 Phase 14).
6. **`contracts/` 디렉토리** 신설 — API/이벤트 계약을 둘 자리.
7. **분산 추적(Zipkin) 연결** 확인 — 두 시스템에 걸친 요청 추적 준비.

> 즉 Phase 11이 끝나면 "**두 시스템이 떠 있고, 서로 부를 수 있고, 메시지를 주고받을 통로가 열린** 상태"가 된다. 실제 업무 로직(작업지시·실적·품질)은 Phase 12~15에서 이 토대 위에 얹는다.
