## 1. 최종 통합 아키텍처

```
┌──────────────────────── hwlee-erp (8080, erp_db:3307) ────────────────────────┐
│  SD 영업   MM 재고   FI 회계   HR 인사   PP 생산   배치(Phase9)   리포트(Phase10)  │
│                                                                               │
│  PP.MesDispatchService ──┐                         ┌── KafkaListener(실적/불량) │
│  MesClient(RestClient    │                         │   ProductionPerformance-   │
│   +Retry+CircuitBreaker) │                         │   Handler / QualityDefect- │
│  ReconciliationService   │                         │   Listener (멱등)          │
└──────────────┬───────────┘                         └────────────▲──────────────┘
               │ ① 작업지시 발행 (동기 REST)                        │ ② 실적·불량 (비동기)
               ▼                                                   │
┌──────────────────────── hwlee-mes (8082, mes_db:3308) ───────────┼──────────────┐
│  설비/작업자 마스터   작업지시(수신·멱등)   생산실적·자재투입   품질검사   설비가동률 │
│  PerformanceService/QualityService ──▶ Outbox(같은 트랜잭션) ──▶ OutboxPublisher  │
└───────────────────────────────────────────────────────────────────┼────────────┘
                                                                      ▼
                         Kafka  topics: mes.production.performance / mes.quality.defect
                         (Zookeeper)         Zipkin(9411) ← 두 시스템 분산 추적
```

- **동기(REST)**: ERP→MES 작업지시 발행. 결과 즉시 필요 + Circuit Breaker 로 장애 격리.
- **비동기(Kafka+Outbox)**: MES→ERP 실적/불량. 유실 없는 통보 + 멱등 수신.

## 2. E2E 시나리오 (계획 → 현장 → 재무)

| # | 시스템 | 동작 | 방식 |
| --- | --- | --- | --- |
| 1 | ERP PP | 생산지시 생성(BOM 전개) → 착수(release) | 내부 |
| 2 | ERP→MES | 작업지시 발행 `POST /api/production-orders/{id}/dispatch` | **동기 REST**(멱등·CB) |
| 3 | MES | 작업 시작(설비·작업자) → 생산 실적(부분) → 자재 투입 자동 | 내부 |
| 4 | MES→ERP | 실적 이벤트 Outbox→Kafka(`mes.production.performance`) | **비동기** |
| 5 | ERP | 수신(멱등) → 부품 차감 + 완제품 입고 + 생산 원가 분개 | 내부 |
| 6 | MES | 품질 검사 → 불량 시 이벤트 Outbox→Kafka(`mes.quality.defect`) | **비동기** |
| 7 | ERP | 불량 통보 기록(통계, 재고/회계 영향 없음) | 내부 |
| 8 | 운영 | 정합성 검증 `GET /api/integration/reconciliation` | ERP→MES 조회 대조 |

> ERP 출하(SD)·인보이스는 Part 1(Phase 2)에서 이미 구현됨 — 완제품 입고 후 동일 OTC 흐름으로 이어진다.

### 실행 예 (검증 완료)
1. 생산지시(노트북) 생성→release→dispatch → MES `WO-...`(RECEIVED)
2. MES start→실적(양품 N) → Kafka → ERP 노트북 +N·부품 BOM 비례 차감·생산분개(차)제품/대)원재료)
3. 멱등: 같은 메시지 재수신 시 "이미 처리됨" skip(재고 불변)
4. 품질검사(불량) → ERP `quality_defect_log` 기록
5. 정합성: dispatched PO ↔ MES WO 대조 → 일치 시 `consistent:true`, 수량 틀어지면 `QTY_MISMATCH`

## 3. 운영 관점 — 분산 시스템의 3대 장치

### 3-1. 회복성 (Circuit Breaker)
- ERP `MesClient` = Retry(3회) + CircuitBreaker(name=mes) + Timeout(connect 2s/read 3s).
- **MES 다운 시**: dispatch 가 503 으로 **빠르게 실패**(타임아웃 대기·스레드 고갈 없음) → ERP 보호. 복구 시 회로가 닫혀 정상화.

### 3-2. 정합성 (사후 검증)
- 분산 환경은 한쪽만 반영될 수 있다. `ReconciliationService` 가 ERP dispatched 생산지시와 MES 작업지시를 대조해 **MISSING_IN_MES / QTY_MISMATCH** 를 찾아낸다.
- 멱등성(processed_event / outbox event_id)은 중복 반영을 원천 차단 — "최소 1회 전달 + 멱등 수신 = 결과적 일관성".

### 3-3. 추적 (분산 트레이싱)
- 두 앱 모두 Micrometer Tracing → **Zipkin(http://localhost:9411)**. ERP→MES REST 호출이 하나의 trace 로 묶여, "어디서 끊겼는지"를 타임라인으로 본다.

## 4. 한 줄 회고

Part 1 에서 만든 ERP(계획·재무)와 Part 2 의 MES(현장 실행)가, **동기 REST(명령)** 와 **비동기 Kafka+Outbox(통보)** 두 인터페이스로 연결되어 — 영업 수주가 현장을 거쳐 다시 재고·원가로 돌아오는 **제조 ERP+MES 의 완결된 고리**를 이룬다. 단일 시스템 사고에서 **분산 시스템 사고**(원자성·멱등성·정합성·회복성·추적)로의 전환이 이 프로젝트의 핵심 학습이다.
