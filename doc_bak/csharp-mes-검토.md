# C# MES 검토 메모

> 작성일: 2026-06-09 · 상태: **검토만 (구현 미착수)**
> 배경: 공장 MES 구인공고가 대체로 C#을 요구함. 현재 보유 스택은 Java/Spring(hwlee-erp, hwlee-mes). C#은 미경험.
> 핵심 질문: "집에서 C# MES를 실무와 비슷하게 구현·학습할 수 있는가?"

---

## 1. 결론

- **충분히 가능하다.** 실물 PLC 없이도 시뮬레이터로 실무 유사 환경 구성 가능.
- **핵심은 C# 문법이 아니라 도메인·설비연동이다.** Java를 아는 상태면 C# 문법은 1~2주면 적응.
- 취업 타겟이 C# MES라면, "C# 전환"보다 **"이미 아는 ERP/MES 도메인을 C#/WPF/OPC-UA로 한 번 더 구현"** 하는 전략이 효율적.
  - 면접 어필: "Java로 ERP를 만들고, 그 도메인을 C# MES로 현장 단말까지 구현했다."

---

## 2. 공장 MES에서 실제 쓰는 C# 스택

| 영역 | 실무에서 흔한 것 |
|------|-----------------|
| 언어/런타임 | C# + **.NET Framework 4.x** (아직 많음) 또는 .NET 6/8 |
| UI | **WinForms** / **WPF** 데스크톱 클라이언트 (현장 PC, 키오스크, 작업지시 단말) |
| DB | **MS SQL Server** (압도적), 저장 프로시저 多 |
| 설비 연동 | **OPC-UA / OPC-DA**, PLC(Siemens/Mitsubishi/LS), 시리얼/TCP, 바코드·라벨프린터 |
| 상위 연동 | ERP(SAP/더존 등)와 인터페이스 (REST/파일/DB링크) |
| 기타 | WCF(레거시), Windows Service 배경작업, Crystal Report |

> **ERP(웹 중심)와의 가장 큰 차이**: 공장 MES는 현장 단말용 **데스크톱 앱(WinForms/WPF)** 비중이 크다. 지금 Spring으로 만드는 웹 MES와 모양이 다르다.

---

## 3. "실무랑 비슷"의 진짜 핵심 (C# 문법이 아님)

1. **설비 연동(I/O)** — OPC-UA로 PLC 태그 읽기/쓰기, 설비 신호로 생산실적 자동 집계. **MES의 심장이고 ERP엔 없는 부분.**
2. **현장 UI 패턴** — 작업지시 조회 → 바코드 스캔 → 실적 입력 → 불량/LOT 추적. 마우스 최소, 터치/스캔 위주.
3. **추적성(Traceability)** — LOT/Serial 정·역추적, 4M 이력.
4. **ERP 인터페이스** — hwlee-erp의 생산오더 dispatch 구간과 MES가 주고받는 부분.
   - 관련 기존 작업: `pp/integration`, `V39__production_order_mes_dispatch.sql`

---

## 4. 집에서 만들 수 있는 환경

**OPC-UA 시뮬레이터 + SQL Server(도커) + WPF 클라이언트** 조합으로 실무 유사 미니 MES를 단계적으로 구현 가능.
- 구조: 기존 `hwlee-erp`를 ERP로 두고, C# MES가 그 ERP와 연동.
- 진행 방식: Java 때처럼 도메인 브리핑 · 코드 워크스루 · 시연 가이드 제공.

## 5. .NET / UI 계보 정리

### 런타임(.NET) 계보

| 구분 | 등장 | 상태 | 요약 |
|------|------|------|------|
| **.NET Framework** (1.0~4.8) | 2002~ | **레거시** (4.8에서 멈춤, 보안패치만) | Windows 전용. 옛날 사내 시스템 표준 |
| .NET Core 1~3.1 | 2016~ | 과도기(지원종료) | 크로스플랫폼으로 재작성 |
| **.NET 5/6/7/8/9** | 2020~ | **현역** (지금의 ".NET") | Core의 후속. "Framework" 떼고 그냥 ".NET" |

→ **.NET Framework 4.8이 막차.** 이후 .NET 5 → 6 → 8 로 갈라짐. **.NET Framework = 구세대 / .NET 8 = 신세대.**

### UI (WinForms vs WPF) — 런타임과 별개 축

| | 등장 | 성격 |
|---|------|------|
| **WinForms** | 2002 | 더 옛날. 단순·투박, 빠른 화면 생성. 현장 단말에 아직 많음 |
| **WPF** | 2006 | 더 현대적. XAML UI, 데이터바인딩/MVVM, 디자인 자유도 ↑ |

> 둘 다 .NET Framework에서도, 최신 .NET 8에서도 돌아간다. 그래서 조합 4가지:
> - `.NET Framework + WinForms` → **가장 올드.** 오래된 공장 시스템에 제일 흔함
> - `.NET Framework + WPF` → 올드 런타임 + 현대 UI
> - `.NET 8 + WinForms` → 신 런타임 + 올드 UI (레거시 화면 유지보수)
> - `.NET 8 + WPF` → **요즘 신규 개발 표준**

### 무엇을 배울 것인가

- **취업용 현실**: 공장 MES 구인은 아직 `.NET Framework 4.x + WinForms` 유지보수가 많음. (설비 엮인 시스템은 한번 깔면 10년+ 유지)
- **학습용 추천**: 새로 배운다면 **`.NET 8 + WPF`**. 문법·구조 최신이고, WinForms는 "더 단순한 옛날 방식"이라 나중에 금방 적응. (반대 순서는 옛날 습관이 남음)

## 6. 실제 시작 시 결정할 것 (미정)

- **범위**: 풀 MES 말고 "작업지시 → 실적 → 추적" 1개 라인 슬라이스부터.
- **환경**: `.NET 8 + WPF` vs `.NET Framework + WinForms` (구인공고 보고 결정).
- **연동**: hwlee-erp와 붙일지, 독립으로 갈지.
