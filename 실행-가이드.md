# 프로젝트 실행/종료 가이드

매일 학습을 시작할 때 / 끝낼 때 따라 하는 운영 절차. Phase에 상관없이 동일하다.

---

## 한눈에 보기

작업 디렉토리: `/Users/hwlee/IdeaProjects/my-app/erp`

### 시작 (Docker → 앱 순서)

```bash
docker compose up -d                  # ① 인프라(MySQL) 백그라운드 시작
cd hwlee-erp && ./gradlew bootRun     # ② Spring Boot 앱 시작 (이 터미널 점유됨)
```

동작 확인:
```bash
curl http://localhost:8080/api/health | jq    # {"status":"UP","db":"OK",...}
# Swagger UI: http://localhost:8080/swagger-ui.html
```

### 종료 (앱 → Docker 순서, 시작의 역순)

```bash
# 앱 실행 중인 터미널에서:  Ctrl+C       ← ① 앱 종료
docker compose stop                          # ② 인프라 종료 (데이터 보존)
```

### 자주 쓰는 보조 명령

```bash
docker compose ps                    # 컨테이너 상태 (healthy인지)
docker compose logs -f mysql         # MySQL 로그 실시간 (Ctrl+C로 빠져나옴)
cd hwlee-erp && ./gradlew test       # 통합 테스트만 (인프라 없어도 OK)
docker compose down -v               # ⚠️ 완전 초기화 (DB 데이터 삭제)
```

## 핵심 원칙

- **시작은 안에서 밖으로 켤 수 없다** — DB(컨테이너)가 떠 있어야 앱이 뜬다. 그래서 항상 **Docker 먼저, 앱 나중**.
- **종료는 반대 순서** — 앱 먼저 끄고 DB를 끈다. 반대로 하면 앱이 DB 연결 끊김 에러를 뱉는다.
- **데이터는 Docker 볼륨에 산다** — 컨테이너를 멈춰도 `flyway_schema_history`와 적용된 테이블은 그대로 남는다. 진짜로 비우려면 `-v` 옵션을 줘야 함 (아래 [초기화](#초기화-깨끗하게-다시-시작) 참고).

## 시작 절차

작업 디렉토리는 항상 ERP 루트.

```bash
cd /Users/hwlee/IdeaProjects/my-app/erp
```

### ① Docker 인프라 시작

```bash
docker compose up -d
```

- `-d` = detach, 백그라운드로 띄움. 터미널을 점유하지 않는다.
- 처음 한 번은 이미지(`mysql:8.0`)를 받느라 시간이 좀 걸린다. 두 번째부터는 빠르다.

**상태 확인:**

```bash
docker compose ps
```

기대 출력:
```
NAME                IMAGE       STATUS              PORTS
hwlee-erp-mysql     mysql:8.0   Up (healthy)        0.0.0.0:3307->3306/tcp
```

`(healthy)`가 떠야 다음 단계로. `(starting)`이면 5~10초 더 기다린다.

**로그로 부팅 진행을 보고 싶을 때:**

```bash
docker compose logs -f mysql
# Ctrl+C 로 빠져나옴 (컨테이너는 계속 떠 있음)
```

### ② Spring Boot 앱 시작

선택지 두 가지.

#### (A) 터미널에서

```bash
cd hwlee-erp
./gradlew bootRun
```

- 부팅 로그가 그 터미널에 실시간으로 뜬다.
- **이 터미널은 점유**된다. 다른 명령을 치려면 새 터미널을 열거나, 백그라운드(`./gradlew bootRun &`)로 돌린다. 학습 단계에서는 그냥 터미널을 새로 여는 게 직관적.

#### (B) IntelliJ에서

- `ErpApplication.java` 좌측의 ▶ 버튼 → Run.
- 콘솔 창에 부팅 로그가 뜬다. 정지는 빨간 ■ 버튼.

**부팅 성공 신호 (둘 다 공통):**

로그 마지막 부분에서 다음 두 줄이 보이면 성공.
```
Successfully applied 1 migration to schema `erp_db` ...   ← Flyway 적용 완료
Tomcat started on port 8080                                ← HTTP 서버 기동
```

### ③ 동작 확인

```bash
curl http://localhost:8080/api/health | jq
```

기대 응답:
```json
{ "status": "UP", "db": "OK", "timestamp": "..." }
```

브라우저로는: <http://localhost:8080/swagger-ui.html>

## 종료 절차

### ① Spring Boot 앱 종료

| 시작한 방법 | 종료 방법 |
| --- | --- |
| 터미널 `./gradlew bootRun` | 그 터미널에서 **`Ctrl+C`** |
| 백그라운드(`&`) | `lsof -i :8080`으로 PID 확인 → `kill <PID>` |
| IntelliJ | 콘솔의 **빨간 ■** 버튼 |

> **`Ctrl+C`를 두 번 누르지 말 것.** 한 번이면 graceful shutdown. 두 번 누르면 강제 종료라 마무리 작업(연결 닫기 등)이 안 된다. 학습 단계에서는 큰 영향 없지만 습관을 들여 두면 좋음.

### ② Docker 인프라 종료

세 가지 옵션이 있고, **고를 줄 알아야 한다**. 차이를 표로:

| 명령 | 컨테이너 | 데이터 (볼륨) | 다음 시작 속도 | 언제 쓰나 |
| --- | --- | --- | --- | --- |
| `docker compose stop` | 멈춤 (보존) | 보존 | 가장 빠름 | **일반적인 종료** — 잠시 자리 비울 때, 노트북 닫을 때 |
| `docker compose down` | 삭제 | 보존 | 보통 | 컨테이너 설정만 깨끗이 다시 만들고 싶을 때 |
| `docker compose down -v` | 삭제 | **삭제** | 가장 느림 (초기화) | DB를 진짜 비우고 처음부터 다시 시작하고 싶을 때 |

**기본은 `docker compose stop`** — 데이터 살리고 빠르게 재개.

```bash
docker compose stop
```

확인:
```bash
docker compose ps
# STATUS가 Exited(0) 으로 표시되면 정상 종료
```

> Docker Desktop을 그냥 종료해도 컨테이너는 다음 부팅 시 멈춰 있는 상태로 남는다. 하지만 `docker compose stop`으로 의도적으로 끄는 게 깔끔.

## 초기화 (깨끗하게 다시 시작)

DB를 완전히 비우고 Flyway가 V1부터 다시 돌게 하고 싶을 때.

```bash
# 1) 앱 끄기 (Ctrl+C)
# 2) 컨테이너 + 볼륨 삭제
docker compose down -v
# 3) 다시 시작
docker compose up -d
# 4) 앱 시작
cd hwlee-erp && ./gradlew bootRun
```

부팅 로그에서 다시 `Successfully applied 1 migration`이 보이면 처음부터 다시 적용된 것.

> **주의**: `-v`는 **삭제**다. 마스터/트랜잭션 데이터가 쌓여 있다면 다 날아간다. Phase 1 이후에는 신중하게.

## 자주 막히는 지점

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| `docker compose up`이 즉시 끝나고 다음 명령에서 컨테이너가 안 보임 | Docker Desktop이 꺼져 있음 | Docker Desktop 실행 |
| `Port 3307 is already in use` | 다른 프로세스가 점유 (다른 MySQL 인스턴스 등) | `lsof -i :3307` → 해당 PID `kill` |
| 앱 시작 시 `Connection refused: localhost:3307` | MySQL이 아직 `(starting)` 상태 | `docker compose ps`로 `(healthy)` 될 때까지 대기 |
| Flyway 에러 `FlywayValidateException` | `V1__init.sql`을 수정한 뒤 다시 시작 (체크섬 불일치) | `docker compose down -v`로 초기화 후 재시작 |
| `./gradlew bootRun`이 권한 없음 에러 | gradlew 실행 권한 없음 | `chmod +x hwlee-erp/gradlew` |
| 앱이 떴는데 헬스체크가 `db: ERROR` | 앱은 떴지만 DB 연결 실패 — 보통 3307 vs 3306 혼동 | `application.yml`의 포트 확인 |

## 상태 점검 명령 모음

```bash
# 컨테이너 상태
docker compose ps

# 컨테이너 로그 (실시간)
docker compose logs -f mysql

# MySQL에 접속해서 직접 쿼리
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db

# 마이그레이션 이력 보기
docker exec -it hwlee-erp-mysql mysql -uerp -perp -D erp_db \
  -e "SELECT version, description, success, installed_on FROM flyway_schema_history;"

# 포트 점유 확인
lsof -i :8080      # ERP 앱
lsof -i :3307      # MySQL (호스트)
lsof -i :8081      # MES 앱 (Phase 11~)

# 호스트에 떠 있는 자바 프로세스
jps -l
```

## 정리

- **시작**: Docker → 앱
- **종료**: 앱 → Docker (`docker compose stop`이 디폴트)
- **데이터 삭제는 `-v`**: 평소엔 절대 붙이지 말 것
- **테스트는 인프라와 독립**: Testcontainers가 자체 컨테이너를 띄움
