## ①-2 프론트엔드 실행 (SPA — 위 백엔드가 떠 있어야 함)

> ERP는 **React**, MES는 **Vue**. 둘 다 Vite dev 서버로 띄운다.
> dev 서버가 `/api` 요청을 각 백엔드(8080·8082)로 **프록시**하므로 CORS 설정 없이 실제 API를 그대로 쓴다.

```bash
# 4) ERP 프론트 (터미널 3 = 새 탭) → http://localhost:5174

# 5) MES 프론트 (터미널 4 = 새 탭) → http://localhost:5173
cd hwlee-mes/frontend
npm install          # 처음 한 번
npm run dev
```

- **Node 22 이상** 필요 (현재 확인된 버전: v22.16.0 / npm 10.9.2).
- 백엔드가 안 떠 있으면 화면은 뜨지만 API 호출이 전부 실패한다 → **①을 먼저**.
- 백엔드 주소를 바꾸고 싶으면 환경변수로:
  ```bash
  ERP_API_TARGET=https://erp.hyunwoo.pro npm run dev   # ERP 프론트
  MES_API_TARGET=http://localhost:8082    npm run dev   # MES 프론트
  ```

## 4. 자주 겪는 문제

### "Port 8080 (또는 8082) was already in use" — 포트가 이미 쓰임
이전에 띄운 앱이 안 꺼졌을 때. 그 포트를 쓰는 프로세스를 찾아 종료:
```bash
lsof -ti tcp:8080 -sTCP:LISTEN | xargs kill   # ERP
lsof -ti tcp:8082 -sTCP:LISTEN | xargs kill   # MES
lsof -ti tcp:5174 -sTCP:LISTEN | xargs kill   # ERP 프론트
lsof -ti tcp:5173 -sTCP:LISTEN | xargs kill   # MES 프론트
```

### 프론트는 떴는데 화면이 비어 있거나 API 가 401/500
- 백엔드(8080·8082)가 안 떠 있는 경우가 대부분 → `curl http://localhost:8080/actuator/health` 확인.
- ERP 는 토큰 만료일 수도 있다 → 로그아웃 후 재로그인(또는 브라우저 localStorage 비우기).

### `npm run dev` 가 안 되거나 모듈 에러
```bash
node -v                      # v22 이상인지
rm -rf node_modules && npm install
```

### 앱은 떴는데 DB 연결 오류
도커가 안 떠 있거나 아직 준비 안 됨. 상태 확인 후 잠깐 기다렸다 재시도:
```bash
docker compose ps          # mysql 들이 healthy 인지
docker compose up -d        # 안 떠 있으면 다시
```

### MES 실적이 ERP에 안 들어옴 (Kafka)
Kafka 컨테이너가 떠 있어야 한다. `docker compose ps` 로 `hwlee-kafka` 가 healthy 인지 확인.

# 🤖 AI 어시스턴트(에이전트) 실행

> ERP 화면 우측의 챗봇이 부르는 **로컬 LLM 에이전트**(`../erp-agent`, Python/FastAPI, 포트 8000).
> 자연어를 로컬 **Ollama `qwen2.5:7b`** 가 해석 → ERP 조회/작업. (외부 API·비용 없음)
> 흐름: **ERP 챗봇 → Spring 프록시(`/api/assistant/chat`) → 에이전트(`/chat`) → ERP API/DB**

## A. 실행 순서 (순서대로)

```bash
# 1) Ollama 켜져 있는지 확인 (UP 이어야 함). 안 떠 있으면 실행.
curl -s http://localhost:11434/api/tags >/dev/null && echo "ollama UP" || ollama serve
#    (모델 없으면 처음 한 번:  ollama pull qwen2.5:7b  ← 약 4.7GB)

# 2) 에이전트 웹 서버 실행 (터미널 새 탭) — ERP 챗봇이 이걸 부른다
cd /Users/hwlee/IdeaProjects/my-app/erp-agent
.venv/bin/uvicorn server:app --port 8000
```

- 기본 연결 대상은 `mock`(가짜 데이터) → **ERP 를 안 띄워도 챗봇 데모가 된다.**
- **첫 질문은 수십 초** 걸릴 수 있다(모델 로딩). 정상.
- 실제 ERP 데이터로 쓰려면 아래 C 처럼 `ERP_TARGET=local` 로 실행.

## B. 종료 / 확인

```bash
# 종료: 에이전트 터미널에서  Ctrl + C
# 살아있나 확인 (현재 연결 대상도 같이 보여줌)
curl -s http://localhost:8000/health      # → {"status":"ok","target":"mock",...}
```

## C. 어느 ERP 에 붙일지 — `ERP_TARGET` (mock / local / prod)

실행 한 줄로 대상만 바꾼다.

| 값 | 의미 | 쓰기(수주·상신) |
| --- | --- | --- |
| `mock` | 가짜 인메모리 데이터 (기본, ERP 없이 데모) | ✅ 가능(시연용) |
| `local` | 로컬 실 ERP (`http://localhost:8080`, 먼저 실행 필요) | 🔒 비활성(조회만) |
| `prod` | 운영 ERP (`https://erp.hyunwoo.pro`, 상시 가동) | 🔒 비활성(조회만) |

> `.env` 에 이미 대상별 계정이 채워져 있어 **`ERP_TARGET` 만 바꾸면 바로 조회**된다.

### 운영 데이터로 조회하기 (A 방식 — 이 맥의 에이전트 → 운영 ERP)

에이전트가 **`.env` 의 `PROD_USER` 계정으로 운영 ERP에 스스로 로그인**해 JWT 를 얻고,
그 토큰으로 조회한다. **운영 ERP 의 역할 권한(RBAC)이 챗봇에도 그대로 적용**된다.

```bash
# CLI(REPL)로 운영 데이터 조회 — 가장 간단
cd /Users/hwlee/IdeaProjects/my-app/erp-agent
ERP_TARGET=prod .venv/bin/python main.py
#  예)  "고객 목록 보여줘"  "올해 매출 얼마야?"  "수주 내역"
```

- **조회만 가능(읽기 전용)** — 수주 생성·결재 상신 같은 쓰기는 운영 데이터 보호를 위해 막혀 있다.
  (쓰기 시연이 필요하면 `ERP_TARGET=mock` 으로.)
- 조회 가능 범위: **결재·수주·고객·손익**. 그 외 자유 질문(Text-to-SQL)은 운영 DB 미연결이라 지원 안 함.
- **역할별로 보이는 게 다르다** — 예: `PROD_USER=kim@hyunwoo.com`(영업)이면 손익은 🔒 권한 없음.
  계정을 `admin@`/`lee@`(회계) 등으로 바꾸면 그 계정 권한대로 조회된다.

> 로컬 실 ERP 로 조회하려면 `ERP_TARGET=local`(위 ① 로 ERP 8080 을 띄워둔 상태에서).
> "이번 달 수주 합계" 같은 자유조회(Text-to-SQL)는 `.env` 의 `SQL_DB_URL` 을 채워야 하며 **로컬 전용**이다.
